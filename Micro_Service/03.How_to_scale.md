# মাইক্রোসার্ভিস স্কেলিং — আসল সত্যটা কী?

## তোমার প্রশ্নটি সঠিক

তুমি যে প্রশ্নটি করেছ সেটি অত্যন্ত যুক্তিসঙ্গত। যদি দুটো সার্ভিস একই মেশিনে, একই OS-এর নিচে, শুধু আলাদা পোর্টে চলে — তাহলে CPU-র উপর চাপ তো একই থাকে। একটি HTTP request যদি `Order Service` থেকে `Payment Service`-এ যায়, তবুও সেটা একই CPU-তেই প্রক্রিয়া হচ্ছে। এই পর্যবেক্ষণ সম্পূর্ণ সঠিক।

কিন্তু এখানেই লুকিয়ে আছে সেই ভুল ধারণাটি যা বেশিরভাগ টিউটোরিয়াল পরিষ্কার করে না। **মাইক্রোসার্ভিসের স্কেলিং সুবিধা একটি মেশিনের মধ্যে নয় — এটি একাধিক মেশিনে ছড়িয়ে দেওয়ার ক্ষমতার মধ্যে।**

---

## একই মেশিনে মাইক্রোসার্ভিস: কোনো স্কেলিং সুবিধা নেই

তোমার উদাহরণটি সৎভাবে বিশ্লেষণ করলে বলতে হয় — হ্যাঁ, একটিমাত্র ৩২GB RAM, ৮-কোর মেশিনে দুটো আর্কিটেকচার চালালে CPU-র চাপের দিক থেকে পার্থক্য নগণ্য। বরং মাইক্রোসার্ভিস একটু বেশি রিসোর্স খরচ করে, কারণ প্রতিটি সার্ভিসের নিজস্ব runtime overhead আছে।

```
একটি মেশিন — মনোলিথ বনাম মাইক্রোসার্ভিস

┌──────────────────────────────┐   ┌──────────────────────────────┐
│   Server (32GB, 8-core)      │   │   Server (32GB, 8-core)      │
│                              │   │                              │
│  ┌────────────────────────┐  │   │  ┌──────┐ ┌──────┐ ┌──────┐  │
│  │                        │  │   │  │ User │ │Order │ │Pay   │  │
│  │   Monolith (one proc)  │  │   │  │ :3001│ │ :3002│ │ :3003│  │
│  │                        │  │   │  └──────┘ └──────┘ └──────┘  │
│  └────────────────────────┘  │   │                              │
│                              │   │   একই CPU, একই RAM         │
│  CPU চাপ: ধরো ৭০%            │   │   CPU চাপ: ৭০% + overhead    │
└──────────────────────────────┘   └──────────────────────────────┘

     ↑                                        ↑
  সরল, কম overhead               একটু বেশি overhead, কিন্তু
                                  স্কেলিং সুবিধা এখনো নেই!
```

এই পর্যায়ে মাইক্রোসার্ভিস দিয়ে তুমি কোনো স্কেলিং সুবিধা পাওনি। তুমি শুধু আর্কিটেকচারাল বিচ্ছেদ পেয়েছ। **স্কেলিং সুবিধা আসে তখনই, যখন তুমি সেই সার্ভিসগুলোকে আলাদা মেশিনে নিয়ে যাও।**

## আসল স্কেলিং: হরিজন্টাল স্কেলিং (Horizontal Scaling)

স্কেলিং দুই ধরনের। **Vertical Scaling** মানে হলো একই মেশিনে আরো বেশি RAM, আরো শক্তিশালী CPU লাগানো। এটি ব্যয়বহুল এবং একটি নির্দিষ্ট সীমায় আটকে যায় — একটি মেশিনে তুমি কতটুকুই বা RAM লাগাবে? **Horizontal Scaling** মানে হলো একটি শক্তিশালী মেশিনের পরিবর্তে অনেকগুলো সাধারণ মেশিন ব্যবহার করা। এটিই আধুনিক ব্যাপক-মাপের সিস্টেমের মূল কৌশল।

মনোলিথিক সিস্টেমে Horizontal Scaling করা যায়, কিন্তু সমস্যা হলো পুরো মনোলিথটিকেই নকল করতে হয়। মানে, যদি শুধু পণ্য-সার্চে চাপ বেশি, তবুও User, Order, Payment সমেত পুরো সিস্টেমের ৫টি কপি চালু করতে হবে। প্রতিটি কপিতে সব মডিউল থাকবে, যার বেশিরভাগ সেই মুহূর্তে অলস বসে আছে।

মাইক্রোসার্ভিসে তুমি শুধুমাত্র সেই সার্ভিসটির কপি বাড়াতে পারো যেখানে চাপ বেশি।

## বাস্তব দুনিয়ায় কেমন দেখায়?

Netflix-এর উদাহরণ নাও। তারা যখন সন্ধ্যাবেলা ব্যবহারকারীরা সিনেমা দেখতে বসে, তখন **Video Streaming Service**-এ চাপ কয়েকশো গুণ বেড়ে যায়। কিন্তু **User Profile Service** বা **Billing Service**-এ চাপ প্রায় একই থাকে। মাইক্রোসার্ভিসের কারণে তারা শুধুমাত্র Video Streaming-এর জন্য শত শত নতুন instance চালু করতে পারে — Billing Service-কে স্পর্শ না করেই।

```
Netflix-এর রাত ৯টার চিত্র (কাল্পনিক):

Service               Instance Count
─────────────────────────────────────
Video Streaming   →   🟩🟩🟩🟩🟩🟩🟩🟩  (×500)  ← চাপ বেশি
Recommendation    →   🟩🟩🟩🟩           (×50)   ← মাঝারি চাপ
User Auth         →   🟩🟩               (×10)   ← কম চাপ
Billing           →   🟩                 (×2)    ← প্রায় নিষ্ক্রিয়
```

এটি Kubernetes বা AWS ECS-এর মতো Container Orchestration সিস্টেম দিয়ে স্বয়ংক্রিয়ভাবে পরিচালিত হয়। চাপ বাড়লে নতুন container উঠে, চাপ কমলে বন্ধ হয়ে যায়।

## তাহলে একই মেশিনে মাইক্রোসার্ভিসের সুবিধা কী?

এখানে একটি সৎ স্বীকারোক্তি দরকার। একটিমাত্র মেশিনে চললে মাইক্রোসার্ভিসের **স্কেলিং সুবিধা নেই** — বরং সামান্য বেশি overhead আছে। কিন্তু তোমার প্রশ্নের একটি গুরুত্বপূর্ণ উত্তর হলো: **মাইক্রোসার্ভিসের সুবিধা শুধু স্কেলিং নয়**। সুবিধাগুলো হলো:

প্রথমত, **Fault Isolation** — একটি সার্ভিস ক্র্যাশ করলে অন্যগুলো চলতে থাকে। একই মেশিনেও এটি সত্য। Payment Service-এর memory leak যদি সেই process-কে মেরে ফেলে, User Service তখনও সচল থাকবে।

দ্বিতীয়ত, **Independent Deployment** — তুমি শুধু Product Service ডেপ্লয় করতে পারবে, পুরো সিস্টেম নামাতে হবে না। এটি একই মেশিনেও কাজ করে।

তৃতীয়ত, **Technology Freedom** — Product Service Go-তে, Notification Service Node.js-এ লিখতে পারো।

কিন্তু **স্কেলিং সুবিধাটি সম্পূর্ণরূপে পাওয়া যায় শুধুমাত্র তখন, যখন সার্ভিসগুলো আলাদা মেশিনে বা আলাদা Container-এ চলতে পারে।** এই কারণেই Docker এবং Kubernetes মাইক্রোসার্ভিসের অপরিহার্য সঙ্গী হয়ে উঠেছে।
