# মাইক্রোসার্ভিস আর্কিটেকচার — কেন এবং কীভাবে

## প্রথম কথা: সফটওয়্যার তৈরির শুরুর গল্প

যখন কোনো সফটওয়্যার প্রকল্পের যাত্রা শুরু হয়, তখন সবকিছু খুবই সরল মনে হয়। একটি ওয়েবসাইট বানাতে হবে, একটি ডেটাবেজ থাকবে, একটি সার্ভার থাকবে — এবং সবকিছু মিলিয়ে একটিমাত্র কোডবেসে সব লেখা হবে। এই পদ্ধতিকে বলা হয় **মনোলিথিক আর্কিটেকচার** (Monolithic Architecture)। শুরুতে এটি দ্রুত কাজ করার সবচেয়ে স্বাভাবিক উপায় বলে মনে হয়। কিন্তু সময়ের সাথে সাথে, যখন ব্যবহারকারীর সংখ্যা বাড়ে, ফিচার যোগ হয়, এবং দলের আকার বৃদ্ধি পায় — তখন এই একক বিশাল সিস্টেমটি একটি ভারী বোঝায় পরিণত হয়।

---

## মনোলিথিক সিস্টেম কী?

একটি মনোলিথিক অ্যাপ্লিকেশন হলো এমন একটি সিস্টেম, যেখানে সমস্ত কার্যক্ষমতা — ব্যবহারকারীর লগইন থেকে শুরু করে পণ্যের তালিকা, পেমেন্ট প্রক্রিয়াকরণ, নোটিফিকেশন পাঠানো — সবকিছুই একই কোডবেসে, একই প্রক্রিয়ায়, এবং প্রায়শই একটিমাত্র ডেটাবেজে একত্রিত থাকে। নিচের চিত্রটি একটি মনোলিথিক ই-কমার্স সিস্টেমের সরলীকৃত রূপ।

```
┌─────────────────────────────────────────────────────────┐
│                  MONOLITHIC APPLICATION                 │
│                                                         │
│  ┌───────────┐  ┌───────────┐  ┌───────────────────┐    │
│  │  User     │  │  Product  │  │  Order Management │    │
│  │  Module   │  │  Module   │  │  Module           │    │
│  └─────┬─────┘  └─────┬─────┘  └────────┬──────────┘    │
│        │              │                  │              │
│  ┌─────┴──────────────┴──────────────────┴──────────┐   │
│  │            Shared Business Logic Layer           │   │
│  └─────────────────────────┬────────────────────────┘   │
│                            │                            │
│  ┌─────────────────────────▼─────────────────────────┐  │
│  │              Single Shared Database               │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
           ↑
   সবকিছু একসাথে, একটি বিশাল এককে
```

প্রথম দৃষ্টিতে এটি পরিষ্কার এবং সহজবোধ্য মনে হয়। কিন্তু যখন এই সিস্টেম বড় হতে থাকে, তখন বিভিন্ন গুরুতর সমস্যা দেখা দেয়।

## মনোলিথিক সিস্টেমের সমস্যাগুলো

### ১. স্কেলিং-এর সমস্যা (The Scaling Problem)

ধরা যাক, একটি ই-কমার্স সাইটে হঠাৎ বিশাল একটি সেল শুরু হলো। হাজার হাজার ব্যবহারকারী একসাথে পণ্য খুঁজছে এবং অর্ডার দিচ্ছে। এই চাপ সামলাতে গেলে পুরো অ্যাপ্লিকেশনটিকেই স্কেল করতে হবে — শুধু পণ্য-সার্চ মডিউলটিকে নয়। অথচ হয়তো লগইন সার্ভিসে তেমন চাপ নেই। মনোলিথিক সিস্টেমে এই বিচক্ষণতার কোনো সুযোগ নেই। পুরো সিস্টেমের জন্য বাড়তি সার্ভার কিনতে হবে, যা অত্যন্ত ব্যয়বহুল এবং অপচয়মূলক।

```
     চাহিদা বেশি শুধু এখানে
             ↓
┌────────────────────────────┐
│  Product Search  (🔥 HIGH) │
├────────────────────────────┤    ← পুরো সিস্টেম স্কেল করতে হবে
│  User Login      (😴 LOW)  │       যদিও বেশিরভাগ অংশে
├────────────────────────────┤       চাপ নেই
│  Notifications   (😴 LOW)  │
└────────────────────────────┘
         Monolith
```

### ২. একটি ত্রুটি পুরো সিস্টেম ভেঙে দেয় (Single Point of Failure)

মনোলিথিক সিস্টেমের সবচেয়ে ভয়ঙ্কর দিক হলো, একটি মডিউলে কোনো মারাত্মক বাগ বা মেমোরি লিক হলে সম্পূর্ণ অ্যাপ্লিকেশনটি ক্র্যাশ করতে পারে। যদি পেমেন্ট মডিউলে কোনো সমস্যা হয়, তাহলে কেবল পেমেন্ট বন্ধ থাকবে না — পুরো ওয়েবসাইটটাই অকেজো হয়ে পড়তে পারে।

```
         একটি মডিউলে বাগ
                ↓
  ┌─────────────────────────┐
  │  💥 Payment Bug         │
  │    ↓                    │
  │  পুরো অ্যাপ ক্র্যাশ!        │
  │                         │
  │  ✗ User Login           │
  │  ✗ Product Search       │
  │  ✗ Notifications        │
  └─────────────────────────┘
```

### ৩. ডেভেলপমেন্ট ধীর হয়ে পড়ে (Development Velocity)

যখন একটি প্রজেক্টে ৫০ জন ডেভেলপার কাজ করে, এবং সবাই একই কোডবেসে পরিবর্তন করে — তখন কোড মার্জ করা, কনফ্লিক্ট সমাধান করা এবং একটি পরিবর্তন টেস্ট করা প্রচণ্ড কষ্টকর হয়ে যায়। একজন ডেভেলপার পেমেন্ট ফিচারে কাজ করছে, আরেকজন একই ফাইলে প্রোডাক্ট ফিচার যোগ করছে — দুজনের কাজ একসাথে মার্জ করা প্রায়ই দুঃস্বপ্নে পরিণত হয়।

### ৪. প্রযুক্তি পরিবর্তনের বাধা (Technology Lock-in)

মনোলিথিক সিস্টেমে পুরো অ্যাপ্লিকেশন সাধারণত একটিমাত্র প্রযুক্তি স্ট্যাকে লেখা হয়। ধরা যাক সিস্টেমটি Node.js-এ লেখা। এখন যদি নতুন একটি মডিউলের জন্য Go-এর পারফরম্যান্স প্রয়োজন হয়, তবুও একই ভাষায় থাকতে বাধ্য হতে হয়। প্রযুক্তির বিবর্তনের সুযোগ থেকে দল বঞ্চিত থাকে।

### ৫. ডেপ্লয়মেন্টের ঝুঁকি (Deployment Risk)

মনোলিথিক সিস্টেমে একটি ছোট পরিবর্তনের জন্যও পুরো অ্যাপ্লিকেশনটিকে নতুন করে ডেপ্লয় করতে হয়। একটি বাটনের রঙ পরিবর্তন করলেও পুরো সার্ভার রিস্টার্ট দিতে হয়। প্রতিটি ডেপ্লয়মেন্টই একটি উচ্চ-ঝুঁকিপূর্ণ ঘটনায় পরিণত হয়।

## মাইক্রোসার্ভিস: সমাধানের পথ

মাইক্রোসার্ভিস আর্কিটেকচার এই সমস্যাগুলোর উত্তর হিসেবে জন্ম নিয়েছে। মূল ধারণাটি সহজ: একটি বড় সিস্টেমকে ভেঙে অনেকগুলো ছোট, স্বাধীন, এবং একক-দায়িত্বসম্পন্ন সার্ভিসে বিভক্ত করো। প্রতিটি সার্ভিস নিজের কাজ নিজে করবে, নিজের ডেটাবেজ রাখবে, এবং অন্য সার্ভিসের সাথে নেটওয়ার্কের মাধ্যমে কথা বলবে।

```
                        ┌──────────────────────────────────────────────┐
                        │           MICROSERVICES ARCHITECTURE         │
                        └──────────────────────────────────────────────┘

  Client Request
       │
       ▼
┌─────────────┐
│  API Gateway│  ← একটি প্রবেশদ্বার, সব অনুরোধ এখান দিয়ে আসে
└──────┬──────┘
       │
  ┌────┴──────────────────────────────────────┐
  │                                           │
  ▼                    ▼                      ▼
┌──────────┐     ┌──────────┐          ┌──────────────┐
│  User    │     │ Product  │          │    Order     │
│ Service  │     │ Service  │          │   Service    │
│          │     │          │          │              │
│ [DB: 👤] │     │ [DB: 📦] │          │  [DB: 🧾]   │
└──────────┘     └──────────┘          └──────┬───────┘
                                              │
                                  ┌───────────┴───────────┐
                                  ▼                       ▼
                           ┌────────────┐         ┌─────────────┐
                           │  Payment   │         │Notification │
                           │  Service   │         │  Service    │
                           │            │         │             │
                           │ [DB: 💳]   │         │ [DB: 🔔]    │
                           └────────────┘         └─────────────┘

   প্রতিটি সার্ভিস স্বাধীন, নিজের ডেটাবেজসহ, আলাদাভাবে ডেপ্লয়যোগ্য
```

এখন যদি পণ্য-সার্চে চাপ বাড়ে, শুধুমাত্র `Product Service`-এর কপি বাড়ানো যাবে। বাকি সার্ভিসগুলোকে স্পর্শ করতে হবে না। যদি `Notification Service`-এ বাগ থাকে, তবুও ব্যবহারকারী পণ্য কিনতে পারবে, কারণ অন্য সার্ভিসগুলো স্বাধীনভাবে চলছে।

## সমাপ্তি কথা

মাইক্রোসার্ভিস কোনো যাদুমন্ত্র নয়। এটি নতুন জটিলতা নিয়ে আসে — নেটওয়ার্ক কমিউনিকেশন, সার্ভিস ডিসকভারি, ডিস্ট্রিবিউটেড ট্রানজেকশন — এই সবই শিখতে হয়। কিন্তু বড় মাপের, দ্রুত বর্ধনশীল সিস্টেমের জন্য এটি এখন পর্যন্ত সবচেয়ে কার্যকর সমাধান। Amazon, Netflix, Uber — এই প্রতিষ্ঠানগুলো যখন মনোলিথের সীমাবদ্ধতায় আটকে গিয়েছিল, তখন মাইক্রোসার্ভিসই তাদের বিশ্বব্যাপী পরিষেবা দেওয়ার সক্ষমতা দিয়েছে।

পরবর্তী অধ্যায়ে আমরা দেখব কীভাবে এই স্বাধীন সার্ভিসগুলো একে অপরের সাথে কথা বলে — সেখানে আসবে **API Gateway**, **Message Queue**, **Service Discovery** এবং আরো অনেক ধারণা।
