# মাইক্রোসার্ভিস যোগাযোগ — নেটওয়ার্কের গভীরে

## মূল প্রশ্নটা কী?

gRPC বা Kafka সরিয়ে রাখো। প্রশ্নটা আরো নিচের স্তরের — আলাদা মেশিনে থাকা দুটো process আদৌ কীভাবে একে অপরের সাথে কথা বলতে পারে? gRPC বা Kafka নিজেরাও তো কোনো না কোনো মাধ্যমে কথা বলছে। সেই মাধ্যমটা কী?

উত্তর হলো: **TCP/IP**। ইন্টারনেটে যত কমিউনিকেশন হয় — HTTP, gRPC, Kafka, RabbitMQ, WebSocket — সবকিছু শেষ পর্যন্ত TCP/IP-এর উপর দাঁড়িয়ে আছে। এটাই হলো মাইক্রোসার্ভিসের যোগাযোগের ভিত্তি।

---

## নেটওয়ার্ক স্তরের পূর্ণ চিত্র

```
  service-A (Machine: 10.0.0.1)          service-B (Machine: 10.0.0.2)
  ─────────────────────────────          ─────────────────────────────

  ┌─────────────────────────┐            ┌─────────────────────────┐
  │   Application Layer     │            │   Application Layer     │
  │  (gRPC / HTTP / Kafka)  │            │  (gRPC / HTTP / Kafka)  │
  ├─────────────────────────┤            ├─────────────────────────┤
  │   Transport Layer       │            │   Transport Layer       │
  │        TCP              │            │        TCP              │
  ├─────────────────────────┤            ├─────────────────────────┤
  │   Network Layer         │            │   Network Layer         │
  │        IP               │            │        IP               │
  ├─────────────────────────┤            ├─────────────────────────┤
  │   Physical / Link Layer │            │   Physical / Link Layer │
  │  (Ethernet / WiFi)      │            │  (Ethernet / WiFi)      │
  └────────────┬────────────┘            └────────────▲────────────┘
               │                                      │
               │         Physical Network             │
               │    (Data Center Cable / Cloud VPC)   │
               └──────────────────────────────────────┘

  উপরের প্রতিটি layer নিচের layer-এর উপর নির্ভরশীল।
  gRPC বা Kafka যাই হোক, শেষ পর্যন্ত TCP packet হয়ে তারের মধ্য দিয়ে যায়।
```

gRPC, HTTP REST, Kafka — এগুলো সবই **Application Layer Protocol**। তারা TCP connection খুলে তার উপর দিয়ে নিজেদের format-এ data পাঠায়। কিন্তু data যাওয়ার রাস্তা একটাই — TCP/IP।

## TCP Connection — দুটো মেশিনের মধ্যে কথোপকথনের ভিত্তি

যখন `order-service` (10.0.0.1) থেকে `payment-service` (10.0.0.2)-এ কল যায়, তখন প্রথমে একটি TCP connection প্রতিষ্ঠা হয়। এটি তিনটি ধাপে হয় — যাকে বলে **Three-Way Handshake**।

```
  order-service                          payment-service
  (10.0.0.1:54321)                       (10.0.0.2:50051)

       │                                       │
       │──── SYN ────────────────────────────►│  "আমি কথা বলতে চাই"
       │                                       │
       │◄─── SYN-ACK ────────────────────────│  "ঠিক আছে, আমিও রেডি"
       │                                       │
       │──── ACK ────────────────────────────►│  "শুরু করি তাহলে"
       │                                       │
       │  ══════ TCP Connection প্রতিষ্ঠিত ══════ │
       │                                       │
       │──── [HTTP / gRPC data] ─────────────►│
       │◄─── [Response data] ─────────────────│
       │                                       │
```

এই connection-টি একটি **virtual pipe** — দুটো মেশিনের মধ্যে একটি নির্ভরযোগ্য, ordered, এবং error-checked যোগাযোগের চ্যানেল। এই pipe-এর উপর দিয়েই HTTP request, gRPC frame, বা যেকোনো data যায়।

## একটি HTTP Request-এর সম্পূর্ণ যাত্রা

এখন দেখা যাক একটি সাধারণ REST call — `order-service` থেকে `user-service`-এ — আলাদা মেশিনে হলে ঠিক কী কী ঘটে।

```
order-service কোড:
  fetch("http://user-service:3001/users/42")

         ↓  K8s DNS resolve করে
         ↓  "user-service" → 10.96.5.12 (K8s Service ClusterIP)

         ↓  K8s kube-proxy রুট করে
         ↓  10.96.5.12 → 10.0.0.3:3001 (actual Pod IP, Machine C)

  ┌──────────────────────────────────────────────────────────────┐
  │                    Data Center Network                       │
  │                                                              │
  │  Machine A (10.0.0.1)         Machine C (10.0.0.3)          │
  │  ┌──────────────────┐         ┌──────────────────┐          │
  │  │  order-service   │         │  user-service    │          │
  │  │                  │         │                  │          │
  │  │  TCP SYN ───────────────────────────────────► │          │
  │  │  TCP SYN-ACK ◄─────────────────────────────── │          │
  │  │  TCP ACK ───────────────────────────────────► │          │
  │  │                  │         │                  │          │
  │  │  GET /users/42   │         │                  │          │
  │  │  HTTP/1.1 ──────────────────────────────────► │          │
  │  │                  │         │                  │          │
  │  │  HTTP 200 OK  ◄──────────────────────────────  │          │
  │  │  {"id":42,...}   │         │                  │          │
  │  └──────────────────┘         └──────────────────┘          │
  └──────────────────────────────────────────────────────────────┘

  মোট সময়: ~0.5ms থেকে 5ms (একই data center-এ)
```

লক্ষ করো — এই পুরো যাত্রায় গুরুত্বপূর্ণ কাজটি করছে K8s-এর **kube-proxy**। সে প্রতিটি Node-এ চলে এবং iptables rules দিয়ে নিশ্চিত করে যে Service IP-তে যাওয়া traffic সঠিক Pod-এ পৌঁছায়।

## তাহলে gRPC আলাদা কী করে?

HTTP REST যদি TCP-র উপর চলে, gRPC-ও TCP-র উপর চলে — তাহলে gRPC আলাদা কোথায়? পার্থক্যটা হলো TCP connection ব্যবহারের দক্ষতায়।

```
  HTTP/1.1 (REST এর সাধারণ ব্যবহার):

  Request 1 ──► [TCP connection খুলল] ──► Response 1 ──► [বন্ধ]
  Request 2 ──► [TCP connection খুলল] ──► Response 2 ──► [বন্ধ]
  Request 3 ──► [TCP connection খুলল] ──► Response 3 ──► [বন্ধ]

  প্রতিটি request-এ নতুন TCP handshake — সময় এবং CPU অপচয়


  HTTP/2 (gRPC যেটার উপর চলে):

  একটি TCP connection ──────────────────────────────────────────►
                          │
              ┌───────────┼──────────────┐
              ▼           ▼              ▼
           Stream 1    Stream 2       Stream 3
           (req/res)   (req/res)      (req/res)

  একটিমাত্র TCP connection-এ একসাথে অনেক request — multiplexing
```

gRPC HTTP/2 ব্যবহার করে। এর মানে একটি TCP connection জীবিত রেখে তার উপর দিয়ে একই সাথে অনেকগুলো request-response চলতে পারে। এটি connection overhead অনেক কমিয়ে দেয়। তার উপর data binary (Protocol Buffer) format-এ পাঠায়, যা JSON-এর চেয়ে অনেক ছোট।

## Service Mesh — যখন সার্ভিস যোগাযোগ জটিল হয়

প্রোডাকশনে শত শত সার্ভিস হলে নতুন সমস্যা দেখা দেয়। কীভাবে জানব কোন সার্ভিস কোন সার্ভিসকে কল করছে? কোনো call fail হলে আবার চেষ্টা করবে কি? Timeout কত হবে? এই সব প্রশ্নের উত্তর দেয় **Service Mesh**।

```
  Service Mesh ছাড়া:                 Service Mesh সহ (Istio/Linkerd):

  order-svc ──HTTP──► user-svc        order-svc ──► [sidecar proxy]
                                                          │
  Retry logic কোথায়?                                      │ mTLS
  Timeout কে ঠিক করে?                                     │ Retry
  Tracing কীভাবে?                                          │ Timeout
                                                          │ Tracing
                                           [sidecar proxy] ──► user-svc

  প্রতিটি Pod-এর পাশে একটি Proxy (Envoy) বসে।
  সব network traffic এই proxy-র মধ্য দিয়ে যায়।
  Application code কিছুই জানে না — infrastructure সব করে।
```

Service Mesh-এ **Sidecar Pattern** ব্যবহার হয়। প্রতিটি Pod-এর পাশে একটি ছোট proxy container বসে — সাধারণত Envoy। সব outgoing এবং incoming traffic এই proxy দিয়ে যায়। Retry, timeout, circuit breaking, mutual TLS encryption — সব এই proxy করে। Application code শুধু business logic লেখে।

## Cloud-এ মেশিনগুলো কীভাবে একে অপরকে দেখতে পায়?

এই পুরো সংযোগ সম্ভব হয় cloud provider-এর **Virtual Private Cloud (VPC)** এর কারণে।

```
  AWS/GCP Data Center

  ┌──────────────────────────────────────────────────────────────┐
  │                    VPC (10.0.0.0/16)                         │
  │                                                              │
  │  Availability Zone A          Availability Zone B            │
  │  ┌────────────────────┐      ┌────────────────────┐          │
  │  │  Subnet 10.0.1.0   │      │  Subnet 10.0.2.0   │          │
  │  │                    │      │                    │          │
  │  │  Node1: 10.0.1.10  │      │  Node2: 10.0.2.10  │          │
  │  │  Node2: 10.0.1.11  │      │  Node3: 10.0.2.11  │          │
  │  └────────────────────┘      └────────────────────┘          │
  │                                                              │
  │  এই VPC-র ভেতরে সব machine একে অপরকে private IP দিয়ে       │
  │  সরাসরি দেখতে পায় — যেন একই LAN-এ আছে                      │
  │                                                              │
  │  বাইরে থেকে দেখা যায় না — firewall (Security Group) দিয়ে   │
  │  নিয়ন্ত্রিত                                                  │
  └──────────────────────────────────────────────────────────────┘
```

AWS-এ VPC, GCP-তে VPC Network — এগুলো software-defined network যা physical data center-এর ভেতরে একটি isolated private network তৈরি করে। K8s Node-রা এই VPC-র মধ্যেই থাকে, তাই তারা private IP দিয়ে সরাসরি কথা বলতে পারে। বাইরের internet থেকে এই IP-গুলো দেখা যায় না।

## সব একসাথে: একটি Request-এর পূর্ণ যাত্রা

```
  Browser → api.myshop.com/api/orders/new

  ১. DNS resolve: api.myshop.com → 34.120.50.5 (Load Balancer IP)

  ২. AWS Load Balancer (34.120.50.5)
     └─► K8s Node (10.0.1.10) এ পাঠাল

  ৩. Ingress Controller (nginx, Node 10.0.1.10)
     └─► /api/orders → order-service Pod (10.0.2.15:3002)

  ৪. order-service Pod (10.0.2.15)
     └─► user-service কে call করতে হবে
         DNS: user-service → 10.96.0.5 (K8s ClusterIP)
         kube-proxy: 10.96.0.5 → 10.0.1.22:3001 (user Pod)
         TCP connection → HTTP GET → Response পেল

  ৫. order-service আবার call করল payment-service
         DNS: payment-service → 10.96.0.8
         kube-proxy: 10.96.0.8 → 10.0.2.30:3003
         TCP → gRPC call → Response পেল

  ৬. order-service response পাঠাল browser-এ
     └─► Ingress → Load Balancer → Browser

  মোট নেটওয়ার্ক hop: ৬-৮ টি
  মোট সময়: ~20-50ms (একই region-এ)
```

## সারসংক্ষেপ: স্তর অনুযায়ী

```
┌─────────────────────────────────────────────────────────────┐
│  যা তুমি লেখো (Application Layer)                          │
│    fetch("http://user-service:3001/users/42")               │
├─────────────────────────────────────────────────────────────┤
│  K8s যা করে (Service Discovery + Routing)                   │
│    "user-service" → DNS → ClusterIP → kube-proxy → Pod IP  │
├─────────────────────────────────────────────────────────────┤
│  Protocol যা করে (Transport)                                │
│    TCP Three-Way Handshake → Connection প্রতিষ্ঠা          │
│    HTTP/1.1 বা HTTP/2 (gRPC) data পাঠানো                   │
├─────────────────────────────────────────────────────────────┤
│  Cloud যা করে (Physical Network)                            │
│    VPC-র মধ্যে private IP routing                           │
│    Data center-এর physical cable/fiber দিয়ে packet যাওয়া  │
└─────────────────────────────────────────────────────────────┘

gRPC, Kafka, RabbitMQ — এগুলো সবই এই একই TCP/IP-এর উপর বসা।
পার্থক্য শুধু কীভাবে data সাজানো হয় এবং connection ব্যবহার করা হয়।
```

পরবর্তী স্বাভাবিক প্রশ্ন হতে পারে — এই network call-গুলো fail করলে কী হয়? **Circuit Breaker**, **Retry**, **Timeout** — এই patterns কীভাবে distributed system-কে resilient রাখে। অথবা যেতে পারি সরাসরি — কখন synchronous (HTTP/gRPC) আর কখন asynchronous (Kafka/RabbitMQ) communication বেছে নেওয়া উচিত।
