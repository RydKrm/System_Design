# মাইক্রোসার্ভিস সংযোগ ও প্রোডাকশন ডেপ্লয়মেন্ট

## প্রশ্নের কেন্দ্রে যাওয়া

তুমি ঠিকই বলেছ — Docker Compose একটি চমৎকার সমাধান, কিন্তু সেটি শুধুমাত্র একটি মেশিনের মধ্যে। প্রোডাকশনে যখন সত্যিকারের স্কেলিং দরকার, তখন একাধিক মেশিনে সার্ভিস ছড়িয়ে দিতে হয়। এই পুরো গল্পটি তিনটি স্তরে বোঝা দরকার — প্রথমে একটি মেশিনে Docker, তারপর একাধিক মেশিনে নেটওয়ার্কিং, এবং সবশেষে প্রোডাকশনের বাস্তবতা।

---

## স্তর ১ — Docker Compose: একটি মেশিনে সার্ভিস সংযোগ

Docker Compose যখন চালু হয়, তখন সে একটি **Virtual Network** তৈরি করে। এই নেটওয়ার্কটি সম্পূর্ণ সফটওয়্যারে তৈরি — কোনো তার নেই, কোনো physical সুইচ নেই। প্রতিটি Container এই virtual network-এ যুক্ত হয় এবং তার **service name** দিয়েই একে অপরকে চিনতে পারে।

```
┌─────────────────────────────────────────────────────────────────┐
│                    Single Machine (Docker Host)                 │
│                                                                 │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │              Docker Virtual Network (bridge)             │  │
│   │          subnet: 172.18.0.0/16  (সফটওয়্যারে তৈরি)          │  │
│   │                                                          │  │
│   │  ┌─────────────┐    ┌─────────────┐   ┌─────────────┐    │  │
│   │  │user-service │    │order-service│   │  postgres   │    │  │
│   │  │172.18.0.2   │    │172.18.0.3   │   │172.18.0.4   │    │  │
│   │  │  :3001      │    │  :3002      │   │  :5432      │    │  │
│   │  └──────┬──────┘    └──────┬──────┘   └──────┬──────┘    │  │
│   │         │                  │                  │          │  │
│   │  order-service কল করতে:   │                  │          │  │
│   │  http://order-service:3002 │◄─────────────────┘          │  │
│   │  (IP না, নাম দিয়ে!)       │                               │  │
│   └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

Docker-এর ভেতরে একটি ছোট **DNS resolver** আছে। `order-service` নাম লিখলে সে নিজেই সেই Container-এর IP বের করে দেয়। তুমি যখন `docker-compose.yml`-এ service-এর নাম দাও, সেটাই হয়ে যায় সেই Container-এর hostname।

```yaml
# docker-compose.yml — service নামই হয় hostname

services:
  user-service:
    build: ./user-service
    ports:
      - "3001:3001"
    networks:
      - app-network

  order-service:
    build: ./order-service
    ports:
      - "3002:3002"
    networks:
      - app-network
    environment:
      # IP নয়, service নাম দিয়ে কথা বলে
      USER_SERVICE_URL: http://user-service:3001

  postgres:
    image: postgres:15
    networks:
      - app-network

networks:
  app-network:
    driver: bridge   # ← virtual network তৈরি হয় এই driver দিয়ে
```

এই ব্যবস্থায় `order-service` থেকে `user-service`-কে কল করতে গেলে URL হবে `http://user-service:3001` — কোনো IP মনে রাখতে হবে না, কারণ Container উঠলে Docker নিজেই IP assign করে এবং নাম দিয়ে resolve করে।

## স্তর ২ — একাধিক মেশিন: আসল সমস্যা শুরু হয়

এখন যদি `product-service`-এ হঠাৎ অনেক চাপ আসে এবং তুমি সেটি আলাদা একটি মেশিনে নিয়ে যেতে চাও — তখন Docker Compose আর কাজ করে না। কারণ Docker Compose জানে শুধু একটি মেশিনের কথা। দুটো মেশিনের মধ্যে কীভাবে Container-রা কথা বলবে?

```
মেশিন A (192.168.1.10)          মেশিন B (192.168.1.20)
┌──────────────────────┐        ┌──────────────────────┐
│  user-service        │        │  product-service     │
│  order-service       │        │  product-service     │  ← ২টি কপি
│                      │        │  product-service     │
│  Docker Network A    │        │  Docker Network B    │
│  172.18.0.0/16       │        │  172.18.0.0/16       │
└──────────┬───────────┘        └──────────┬───────────┘
           │                               │
           │   ??? কীভাবে কথা বলবে ???     │
           └───────────────────────────────┘
                  Physical Network
```

এখানেই দরকার হয় **Container Orchestration** — অনেক মেশিনে Container-দের পরিচালনার ব্যবস্থা। এই কাজের জন্য শিল্পের মানদণ্ড হলো **Kubernetes**।

## স্তর ৩ — Kubernetes: একাধিক মেশিনে Container পরিচালনা

Kubernetes (সংক্ষেপে K8s) মূলত একটি **Cluster Manager**। সে অনেকগুলো মেশিনকে একটি একক কম্পিউটিং পুলে পরিণত করে। তুমি বলো "আমার `product-service`-এর ৩টি কপি চাই" — K8s নিজেই ঠিক করে কোন মেশিনে কোনটি বসবে।

```
┌─────────────────────────────────────────────────────────────────────┐
│                       KUBERNETES CLUSTER                            │
│                                                                     │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                    Control Plane (Master)                      │ │
│  │   API Server  │  Scheduler  │  etcd (state DB)  │  Controller  │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                              │ নির্দেশ দেয়                         │
│        ┌─────────────────────┼─────────────────────┐               │
│        ▼                     ▼                     ▼               │
│  ┌───────────┐         ┌───────────┐         ┌───────────┐         │
│  │  Node 1   │         │  Node 2   │         │  Node 3   │         │
│  │(Machine A)│         │(Machine B)│         │(Machine C)│         │
│  │           │         │           │         │           │         │
│  │ [user-svc]│         │[product#1]│         │[product#2]│         │
│  │ [order   ]│         │[product#2]│         │[payment  ]│         │
│  │           │         │           │         │           │         │
│  └───────────┘         └───────────┘         └───────────┘         │
│                                                                     │
│   K8s Virtual Network (overlay) — সব Node একই network-এ আছে মনে হয় │
└─────────────────────────────────────────────────────────────────────┘
```

K8s একটি **Overlay Network** তৈরি করে যা সব মেশিনকে একটি flat network-এ পরিণত করে। Node 1-এর `user-service` Node 3-এর `payment-service`-কে ডাকতে পারে ঠিক যেন তারা একই মেশিনে আছে।

## K8s-এ সার্ভিস কীভাবে একে অপরকে খোঁজে?

Kubernetes-এ প্রতিটি সার্ভিসের জন্য একটি **Service** অবজেক্ট থাকে। এটি একটি stable DNS নাম এবং একটি Virtual IP দেয়। Pod-রা আসে-যায়, IP পরিবর্তন হয়, কিন্তু Service-এর নাম সবসময় একই থাকে।

```
                  Service Discovery — K8s-এ

  order-service Pod                         user-service
  ─────────────────                         ─────────────
  কল করে:                                  K8s Service Object
  http://user-service:3001                  ClusterIP: 10.96.0.5
         │                                  DNS: user-service
         │                                       │
         ▼                                       │ route করে
    K8s DNS (CoreDNS)   ─────────────────────►  Pod #1 (172.18.0.2)
    "user-service" কে                           Pod #2 (172.18.0.5)
    resolve করে                                 Pod #3 (172.18.0.8)
    10.96.0.5 তে
```

এখানে **CoreDNS** নামের একটি component আছে যা K8s cluster-এর ভেতরে DNS server হিসেবে কাজ করে। তুমি service-এর নাম দিলেই সে সঠিক Pod-এ পৌঁছে দেয়।

## প্রোডাকশন ডেপ্লয়মেন্টের পূর্ণ চিত্র

এখন দেখা যাক একটি request কীভাবে ব্যবহারকারীর browser থেকে শুরু হয়ে সার্ভিসের ভেতর দিয়ে যায়।

```
Internet
   │
   ▼
┌──────────────────────────────────────────────────────────────┐
│                    Cloud Provider (AWS / GCP)                │
│                                                              │
│  ┌─────────────┐                                            │
│  │    DNS      │  (api.myapp.com → Load Balancer IP)         │
│  └──────┬──────┘                                            │
│         │                                                    │
│  ┌──────▼──────┐                                            │
│  │   External  │  (AWS ALB / GCP Load Balancer)              │
│  │Load Balancer│  Traffic ভাগ করে K8s Nodes-এ পাঠায়         │
│  └──────┬──────┘                                            │
│         │                                                    │
│  ┌──────▼──────────────────────────────────────────────┐    │
│  │                  Kubernetes Cluster                 │    │
│  │                                                     │    │
│  │  ┌─────────────┐                                   │    │
│  │  │ Ingress     │  Path-based routing:               │    │
│  │  │ Controller  │  /api/users  → user-service        │    │
│  │  │ (nginx)     │  /api/orders → order-service       │    │
│  │  └──────┬──────┘  /api/products → product-service  │    │
│  │         │                                           │    │
│  │   ┌─────┴──────────────────────┐                   │    │
│  │   │                            │                   │    │
│  │   ▼                            ▼                   │    │
│  │ ┌──────────┐            ┌──────────────┐           │    │
│  │ │user-svc  │            │ product-svc  │           │    │
│  │ │ Pod ×2   │            │  Pod ×5      │           │    │
│  │ └──────────┘            └──────────────┘           │    │
│  │                                                     │    │
│  └─────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

এই চিত্রে কয়েকটি গুরুত্বপূর্ণ উপাদান আছে যা বোঝা দরকার।

**External Load Balancer** হলো cloud provider-এর দেওয়া একটি সার্ভিস যা বাইরের traffic গ্রহণ করে এবং K8s cluster-এর বিভিন্ন Node-এ পাঠিয়ে দেয়। এটি জানে না K8s-এর ভেতরে কে কোথায় আছে — সে শুধু traffic ভাগ করে।

**Ingress Controller** হলো K8s cluster-এর ভেতরের gateway। এটি URL path দেখে ঠিক করে কোন request কোন service-এ যাবে। `/api/products` দেখলে `product-service`-এ পাঠাবে, `/api/users` দেখলে `user-service`-এ।

## স্বয়ংক্রিয় স্কেলিং — HPA (Horizontal Pod Autoscaler)

K8s-এর সবচেয়ে শক্তিশালী সুবিধা হলো **Horizontal Pod Autoscaler (HPA)**। তুমি বলে দাও "CPU ৭০% পেরোলে নতুন Pod তৈরি করো" — K8s নিজেই সেটি করবে।

```
  সময়       product-service CPU    Pod Count
  ─────────────────────────────────────────────
  রাত ১০টা       20%              ×2 pods
  রাত ১০:৩০      45%              ×2 pods
  রাত ১১টা       72% ← HPA trigger  ×4 pods (নতুন Pod উঠল)
  রাত ১১:৩০      85% ← আরো চাপ    ×6 pods
  রাত ১২টা       30% ← চাপ কমল    ×3 pods (অতিরিক্ত Pod নামল)

  খরচ: শুধু যতক্ষণ Pod চালু, ততক্ষণের জন্য
```

## সম্পূর্ণ যাত্রা: Code থেকে Production পর্যন্ত

```
তুমি code লিখলে
      │
      ▼
git push → GitHub/GitLab
      │
      ▼
CI/CD Pipeline (GitHub Actions / GitLab CI)
      │  Docker image build হলো
      │  Tests চলল
      ▼
Container Registry (Docker Hub / AWS ECR)
      │  image: myapp/product-service:v1.2.3
      ▼
Kubernetes Cluster
      │  kubectl apply -f product-deployment.yaml
      │  নতুন Pod উঠল, পুরনো Pod নামল (zero downtime)
      ▼
Production Traffic পায় নতুন version
```

এই পুরো প্রক্রিয়াটিকে বলে **GitOps** — code push করলেই স্বয়ংক্রিয়ভাবে production আপডেট হয়ে যায়।

## সারসংক্ষেপ

মাইক্রোসার্ভিসের সংযোগ একটি বিবর্তনের গল্প। শুরুতে Docker Compose-এর virtual bridge network দিয়ে একটি মেশিনে সব সার্ভিস চলে এবং service name দিয়ে একে অপরকে চেনে। যখন scale করতে হয় এবং একাধিক মেশিন দরকার হয়, তখন Kubernetes-এর overlay network সব মেশিনকে একটি flat নেটওয়ার্কে জুড়ে দেয় এবং CoreDNS সার্ভিস আবিষ্কারের কাজ করে। বাইরের traffic আসে Load Balancer হয়ে, তারপর Ingress Controller সেটি সঠিক সার্ভিসে পাঠায়। আর HPA স্বয়ংক্রিয়ভাবে চাহিদা অনুযায়ী Pod সংখ্যা বাড়ায় ও কমায়।

পরবর্তী বিষয় হিসেবে আমরা দেখতে পারি — সার্ভিসগুলো একে অপরের সাথে কীভাবে কথা বলে: **Synchronous (REST/gRPC)** নাকি **Asynchronous (RabbitMQ/Kafka)** — এবং কোন পরিস্থিতিতে কোনটি বেছে নেওয়া উচিত।
