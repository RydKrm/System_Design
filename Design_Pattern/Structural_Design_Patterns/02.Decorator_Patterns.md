# Decorator Pattern in Go: A Complete Guide for Beginners

## The Spirit of Decoration

Picture a plain wooden chair, functional but unremarkable. You might first cover its seat with a soft cushion, transforming its comfort without altering its fundamental purpose as a seat. Later, you drape an elegant fabric over the back, adding visual warmth. Each layer enhances the chair’s behaviour or appearance, yet the core object—the chair—remains unchanged, and you can add or remove layers in any combination you desire. This is the essence of the Decorator pattern.

The Decorator pattern is a structural design pattern that allows you to attach additional responsibilities to an object dynamically, at runtime, without affecting other objects of the same class. Instead of creating a vast hierarchy of subclasses to cover every possible combination of features, you wrap an object with one or more decorators that implement the same interface as the object they envelop. The result is a flexible alternative to subclassing for extending functionality, keeping your code open for extension but closed for modification.

In Go, where we favour composition and interface satisfaction over inheritance, the Decorator pattern naturally matches idiomatic practice. A decorator is simply a struct that holds a reference to the component it decorates and itself satisfies the component’s interface. Because Go interfaces are satisfied implicitly, a decorator can be slipped into any code that expects the component, making the entire arrangement transparent to the client.

## The Core Participants

Every decorator‑based design revolves around four roles:

- **Component**: An interface that declares the operations that can be dynamically augmented. The client interacts solely with this abstraction, keeping it decoupled from concrete implementations and from the fact that a chain of decorators may exist.
- **ConcreteComponent**: A struct that implements the `Component` interface and provides the basic behaviour that decorators will later extend. This is the object we start with, the plain chair in our analogy.
- **Decorator**: A base struct that also implements the `Component` interface and contains a field of type `Component`. This base decorator’s primary job is to forward requests to the wrapped component. It provides the foundation upon which concrete decorators build.
- **ConcreteDecorator**: Structs that extend the base `Decorator`, overriding the interface methods to add extra behaviour before and/or after delegating to the wrapped component. Each concrete decorator adds one distinct responsibility.

The beauty of this arrangement is that decorators can be stacked. A concrete component can be wrapped by decorator A, then by decorator B, and so on. The resulting object still looks like an ordinary component to the client, but carries the accumulated behaviour of all the layers.

### Visualising the Relationship

An ASCII representation helps illuminate the structure. Notice how every decorator both holds a `Component` and implements it, forming a recursive composition.

```text
+------------------+        +---------------------+
|     Client       |------->|   «interface»       |
+------------------+        |    Component        |
                            +----------+----------+
                                       ^
                                       |
                       +---------------+---------------+
                       |                               |
            +----------+----------+         +----------+----------+
            | ConcreteComponent   |         |   Decorator    (base)|
            +---------------------+         | - wrapped: Component |
                                            +----------+----------+
                                                       ^
                                                       |
                                     +-----------------+-----------------+
                                     |                                   |
                           +---------+---------+              +----------+---------+
                           | ConcreteDecoratorA|              | ConcreteDecoratorB |
                           +-------------------+              +--------------------+
                           | added behaviour   |              | added behaviour    |
                           +-------------------+              +--------------------+
```

The client sees only the `Component` interface. The base decorator holds a reference to that same interface and forwards calls. Sub‑decorators override methods to insert their own logic around the delegation.

## A Gentle Introduction through Go Code

Let us translate these abstract concepts into concrete Go. Consider a simple coffee shop scenario where we sell beverages. A `Beverage` component provides a description and a cost. The plain coffee is our concrete component, and we want to add condiments—milk, mocha, whipped cream—dynamically, without creating a new subclass for every possible combination.

```go
// Component interface
type Beverage interface {
    Description() string
    Cost() float64
}

// ConcreteComponent
type Espresso struct{}

func (e Espresso) Description() string {
    return "Espresso"
}

func (e Espresso) Cost() float64 {
    return 1.99
}
```

Now we define the base decorator. In Go, we can make it a concrete struct that holds a `Beverage` and forwards the interface methods. While many languages make the decorator abstract, Go’s simplicity allows us to reuse it directly or embed it.

```go
// Base Decorator – it implements Beverage by embedding a Beverage.
// By itself it adds nothing, but it establishes the delegation model.
type CondimentDecorator struct {
    beverage Beverage
}

func (c CondimentDecorator) Description() string {
    return c.beverage.Description()
}

func (c CondimentDecorator) Cost() float64 {
    return c.beverage.Cost()
}
```

Concrete decorators will embed this base decorator or simply hold a `Beverage` field and implement the interface themselves. The second approach, direct implementation without a base struct, is more common in Go because it avoids unnecessary indirection. Let’s use that style for clarity.

A `Milk` decorator adds its own description and cost to the wrapped beverage.

```go
// ConcreteDecorator: Milk
type Milk struct {
    beverage Beverage
}

func (m Milk) Description() string {
    return m.beverage.Description() + ", Milk"
}

func (m Milk) Cost() float64 {
    return m.beverage.Cost() + 0.30
}
```

Similarly, a `Mocha` decorator can be built:

```go
// ConcreteDecorator: Mocha
type Mocha struct {
    beverage Beverage
}

func (m Mocha) Description() string {
    return m.beverage.Description() + ", Mocha"
}

func (m Mocha) Cost() float64 {
    return m.beverage.Cost() + 0.45
}
```

### Weaving the Decorators

The client code creates a core object and wraps it in any number of decorators. Because every decorator is itself a `Beverage`, they can be nested arbitrarily.

```go
func main() {
    // Start with a simple espresso
    var drink Beverage = Espresso{}

    // Dynamically decorate it
    drink = Milk{beverage: drink}
    drink = Mocha{beverage: drink}
    drink = Mocha{beverage: drink}   // double mocha for the brave

    fmt.Printf("Order: %s\n", drink.Description())
    fmt.Printf("Cost: $%.2f\n", drink.Cost())
    // Output:
    // Order: Espresso, Milk, Mocha, Mocha
    // Cost: $3.19
}
```

The client manipulates the object strictly through the `Beverage` interface. The decorators add behaviour transparently; the core `Espresso` struct never changes. You can add a new `Whip` decorator tomorrow without touching any existing code, perfectly adhering to the open/closed principle.

## Covering Every Detail

### How It Differs from Inheritance

In classical object‑oriented design, extending behaviour often means creating a subclass for each combination: `EspressoWithMilk`, `EspressoWithMocha`, `EspressoWithMilkAndMocha`, and so forth. This combinatorial explosion makes maintenance nightmarish. The Decorator pattern sidesteps that by using composition and delegation. Each feature becomes a small, self‑contained wrapper that can be mixed and matched. Because Go lacks classical inheritance and encourages composition, decorators feel natural. They are a prime example of “favour composition over inheritance.”

### The Transparency Principle

Decorators must obey the same interface as the components they wrap. This transparency is what allows clients to treat a decorated object identically to an undecorated one. It implies that decorators should not add public methods that are not part of the component interface, as doing so would break the uniform treatment. If a decorator needs extra configuration, it is typically passed through its constructor so that the interface remains clean.

### Nesting and Order

Decorators can be stacked in any order, and the order matters. In the coffee example, description concatenation and cost addition are commutative, but in many real‑world scenarios, wrapping order is significant. For instance, if one decorator compresses data and another encrypts it, applying encryption before compression yields a different (and usually better) result. The pattern gives the client full control over the sequence of wrapping.

### Intra‑Decoration State

Concrete decorators can hold their own state. A `LoggingDecorator` for an `io.Reader`, for example, could record the number of bytes read. This state is entirely private to the decorator and does not leak into the component. Because each decorator is a separate object, state is naturally compartmentalised.

### Decorators and the Go Standard Library

Go’s standard library uses the decorator idea extensively under the name “middleware” or “wrapper”. Consider:

- `io.Reader` is the component interface. `bufio.Reader` is a concrete decorator that adds buffering. It wraps another `io.Reader` and implements `io.Reader` itself. Similarly, `gzip.Reader` wraps a reader to provide decompression.
- `http.Handler` is the component interface for HTTP handlers. Middleware functions such as logging, authentication, or rate‑limiting are decorators: they accept an `http.Handler` and return a new `http.Handler` that adds the desired behaviour. For example:

```go
func LoggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        log.Println("Request received")
        next.ServeHTTP(w, r)
    })
}
```

Here the returned `http.HandlerFunc` is an adapter that turns a function into a handler, but the essence is decoration: a wrapped handler is returned, and the client sees only `http.Handler`.

### When to Employ the Decorator Pattern

You reach for the Decorator pattern when you need to add responsibilities to individual objects dynamically and transparently, without affecting other objects. It is especially useful when you have a core class that is closed for modification—perhaps from a third‑party library—but you must augment its behaviour. It also shines when many possible combinations of features exist, and a static class hierarchy would be too rigid or large.

### Benefits and Trade‑offs

**Benefits:**

- **High Flexibility:** Objects can be decorated at runtime, and decorations can be combined freely.
- **Single Responsibility:** Each decorator carries exactly one additional responsibility, making the code modular and easy to understand.
- **Open/Closed Principle:** New decorators can be introduced without altering existing component or decorator code.
- **Composition over Inheritance:** The pattern is a perfect embodiment of this Go‑friendly principle.

**Trade‑offs:**

- **Many Little Objects:** A system that heavily uses decorators can end up with numerous small, similar‑looking structs, which may confuse developers unfamiliar with the codebase.
- **Order Dependency:** Debugging a deeply nested chain of decorators can be tricky because the flow of control passes through many layers. It also becomes harder to inspect or test the concrete component at the core.
- **Interface Fidelity:** If the component interface is large, writing a decorator that forwards every method while adding behaviour to only a few can be tedious. In Go, this is mitigated somewhat by struct embedding (a concrete decorator can embed a `Component` and override only specific methods), but careful thought is still required.

### Decorator versus Adapter

While both are structural patterns that involve wrapping, their intentions diverge. The Adapter wraps an object to change its interface to one the client expects. The component and the adapter may have completely different interface signatures. The Decorator, on the other hand, enhances an object without altering its interface; the component and decorator share the same supertype. Adapter bridges incompatible interfaces; decorator adds responsibilities while keeping the interface identical.

### Practical Guidance for Go Developers

To implement a decorator in Go, start by defining a clean, focused interface for the component. Keep it small (the Interface Segregation Principle). Then, write concrete components. For each new behaviour, create a struct that holds a field of the component’s interface type, and implement the same interface. In each method, perform your additional logic before and/or after calling the wrapped component’s method. Always ensure that the decorator does not expose public methods beyond the component interface if you intend full transparency. Use constructor functions like `NewMilk(beverage Beverage) *Milk` to wire decorators in a readable way.

For complex chains, consider whether a base decorator struct is beneficial. In Go, you might embed the component interface inside the decorator and delegate methods explicitly. This technique, known as “embedding for delegation”, reduces boilerplate when the decorator only overrides a subset of methods. For example:

```go
type BaseDecorator struct {
    Beverage
}

func (bd BaseDecorator) Cost() float64 {
    return bd.Beverage.Cost()
}
```

Then `Milk` could embed `BaseDecorator` and override `Cost` and `Description`. However, the explicit delegation approach remains clearer for beginners and is recommended unless you have a broad interface.

## Conclusion

The Decorator pattern is like layering translucent sheets of coloured film over a bright lamp: each sheet adds a new hue while the light itself continues to shine through. In Go, decorators empower you to build flexible, composable systems that stay faithful to the language’s compositional soul. By wrapping objects in behaviour‐adding shells, you create open‑ended architectures that accept change with grace, all without rewriting the original source. Once you cultivate the decorator instinct, you will find that many feature requests can be satisfied by simply composing a new wrapper rather than disturbing existing, tested code.
