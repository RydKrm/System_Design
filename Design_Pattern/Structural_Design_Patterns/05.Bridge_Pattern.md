## The Bridge Pattern: Decoupling Abstraction from Implementation

### Understanding the Fundamental Challenge

Now let us turn our attention to a different kind of structural problem. Imagine you are building a reporting subsystem for your ERP. Your business users need different types of reports: sales reports, inventory reports, customer reports, and financial reports. Additionally, these reports must be generated from different data sources: some companies use PostgreSQL, others use MongoDB, and some integrate with external APIs. Finally, these reports need to be exported in multiple formats: PDF, Excel, CSV, and HTML.

If you approach this problem naively, you might create classes like `PDFSalesReportFromPostgres`, `ExcelSalesReportFromPostgres`, `PDFSalesReportFromMongoDB`, `ExcelSalesReportFromMongoDB`, and so on. With four report types, three data sources, and four output formats, you would need forty-eight different classes! This combinatorial explosion makes the codebase unmaintainable and adding new variations extremely painful.

The Bridge pattern solves this problem by recognizing that you have multiple orthogonal dimensions of variation. Report type is one dimension, data source is another, and output format is yet another. Instead of creating a single class hierarchy that tries to capture all combinations, you create separate hierarchies for each dimension and connect them through composition rather than inheritance.

### The Architectural Philosophy

The Bridge pattern works by splitting a large class or set of closely related classes into two separate hierarchies: abstraction and implementation. The abstraction contains high-level control logic and delegates the actual work to the implementation object. Crucially, the abstraction holds a reference to the implementation, and this reference can be changed at runtime.

In our reporting example, the report types (Sales, Inventory, Customer) form the abstraction hierarchy. They define what data to retrieve and how to present it. The data sources (PostgreSQL, MongoDB, API) form the implementation hierarchy. They define how to connect to the data store and retrieve information. The abstraction calls methods on the implementation interface without knowing which concrete implementation it is working with.

This separation provides enormous flexibility. You can develop and test each hierarchy independently. You can add new report types without touching data source code. You can add new data sources without modifying any report logic. You can even switch data sources at runtime based on configuration or user preference.

// Report is the abstraction interface that uses the bridge
type Report interface {
    Generate() (string, error)
    SetDataSource(source DataSource)
    GetTitle() string
    GetMetadata() map[string]string
}

// BaseReport provides common functionality for all report types
type BaseReport struct {
    dataSource DataSource
    title      string
    query      string
    metadata   map[string]string
}

func (br *BaseReport) SetDataSource(source DataSource) {
    br.dataSource = source
}

func (br *BaseReport) GetTitle() string {
    return br.title
}

func (br *BaseReport) GetMetadata() map[string]string {
    return br.metadata
}

// SalesReport generates comprehensive sales analysis
type SalesReport struct {
    BaseReport
    period      string
    region      string
    includeDetails bool
}

func NewSalesReport(period, region string, includeDetails bool) *SalesReport {
    return &SalesReport{
        BaseReport: BaseReport{
            title: "Sales Performance Report",
            query: "SELECT * FROM sales WHERE period = ? AND region = ?",
            metadata: make(map[string]string),
        },
        period:      period,
        region:      region,
        includeDetails: includeDetails,
    }
}

func (sr *SalesReport) Generate() (string, error) {
    if sr.dataSource == nil {
        return "", fmt.Errorf("data source not configured")
    }
    
    // Connect to data source
    if !sr.dataSource.IsConnected() {
        if err := sr.dataSource.Connect(); err != nil {
            return "", fmt.Errorf("failed to connect: %w", err)
        }
    }
    defer sr.dataSource.Close()
    
    // Fetch data
    fmt.Println("\n--- Fetching Sales Data ---")
    data, err := sr.dataSource.FetchData(sr.query)
    if err != nil {
        return "", fmt.Errorf("failed to fetch data: %w", err)
    }
    
    // Get aggregated totals
    totalRevenue, err := sr.dataSource.ExecuteAggregation("SUM(total_amount)")
    if err != nil {
        return "", fmt.Errorf("failed to calculate totals: %w", err)
    }
    
    // Generate report
    var report strings.Builder
    
    // Header
    report.WriteString("╔════════════════════════════════════════════════════════════╗\n")
    report.WriteString(fmt.Sprintf("║  %-56s  ║\n", sr.title))
    report.WriteString("╠════════════════════════════════════════════════════════════╣\n")
    report.WriteString(fmt.Sprintf("║  Period: %-48s  ║\n", sr.period))
    report.WriteString(fmt.Sprintf("║  Region: %-48s  ║\n", sr.region))
    report.WriteString(fmt.Sprintf("║  Data Source: %-43s  ║\n", sr.dataSource.GetSourceName()))
    report.WriteString(fmt.Sprintf("║  Generated: %-46s  ║\n", time.Now().Format("2006-01-02 15:04:05")))
    report.WriteString("╚════════════════════════════════════════════════════════════╝\n\n")
    
    // Executive Summary
    report.WriteString("EXECUTIVE SUMMARY\n")
    report.WriteString("─────────────────────────────────────────────────────────────\n")
    report.WriteString(fmt.Sprintf("Total Revenue: $%,.2f\n", totalRevenue))
    report.WriteString(fmt.Sprintf("Number of Transactions: %d\n", len(data)))
    if len(data) > 0 {
        avgTransaction := totalRevenue / float64(len(data))
        report.WriteString(fmt.Sprintf("Average Transaction Value: $%,.2f\n", avgTransaction))
    }
    report.WriteString("\n")
    
    // Detailed breakdown if requested
    if sr.includeDetails {
        report.WriteString("DETAILED TRANSACTIONS\n")
        report.WriteString("─────────────────────────────────────────────────────────────\n")
        
        for i, record := range data {
            report.WriteString(fmt.Sprintf("\nTransaction #%d:\n", i+1))
            
            // Handle different data structures from different sources
            customerName := sr.extractCustomerName(record)
            productName := sr.extractProductName(record)
            amount := sr.extractAmount(record)
            date := sr.extractDate(record)
            
            report.WriteString(fmt.Sprintf("  Customer: %s\n", customerName))
            report.WriteString(fmt.Sprintf("  Product: %s\n", productName))
            report.WriteString(fmt.Sprintf("  Amount: $%,.2f\n", amount))
            report.WriteString(fmt.Sprintf("  Date: %s\n", date))
        }
    }
    
    report.WriteString("\n")
    report.WriteString("═══════════════════════════════════════════════════════════\n")
    report.WriteString("              END OF REPORT\n")
    report.WriteString("═══════════════════════════════════════════════════════════\n")
    
    return report.String(), nil
}

func (sr *SalesReport) extractCustomerName(record map[string]interface{}) string {
    // PostgreSQL style
    if name, ok := record["customer_name"].(string); ok {
        return name
    }
    // MongoDB style
    if customer, ok := record["customer"].(map[string]interface{}); ok {
        if name, ok := customer["name"].(string); ok {
            return name
        }
    }
    // API style
    if client, ok := record["client"].(map[string]interface{}); ok {
        if name, ok := client["companyName"].(string); ok {
            return name
        }
    }
    return "Unknown"
}

func (sr *SalesReport) extractProductName(record map[string]interface{}) string {
    if name, ok := record["product_name"].(string); ok {
        return name
    }
    if name, ok := record["product"].(string); ok {
        return name
    }
    return "Unknown"
}

func (sr *SalesReport) extractAmount(record map[string]interface{}) float64 {
    if amt, ok := record["total_amount"].(float64); ok {
        return amt
    }
    if amt, ok := record["revenue"].(float64); ok {
        return amt
    }
    if amt, ok := record["totalRevenue"].(float64); ok {
        return amt
    }
    return 0.0
}

func (sr *SalesReport) extractDate(record map[string]interface{}) string {
    if date, ok := record["order_date"].(string); ok {
        return date
    }
    if date, ok := record["timestamp"].(string); ok {
        return date
    }
    if date, ok := record["fiscalPeriod"].(string); ok {
        return date
    }
    return "Unknown"
}

// ReportingEngine coordinates report generation across different data sources
type ReportingEngine struct {
    dataSources map[string]DataSource
}

func NewReportingEngine() *ReportingEngine {
    return &ReportingEngine{
        dataSources: make(map[string]DataSource),
    }
}

func (re *ReportingEngine) RegisterDataSource(name string, source DataSource) {
    re.dataSources[name] = source
    fmt.Printf("[Reporting Engine] Registered data source: %s\n", name)
}

func (re *ReportingEngine) GenerateReport(report Report, dataSourceName string) (string, error) {
    source, exists := re.dataSources[dataSourceName]
    if !exists {
        return "", fmt.Errorf("data source '%s' not found", dataSourceName)
    }
    
    fmt.Printf("\n[Reporting Engine] Generating '%s' using '%s'\n", 
        report.GetTitle(), source.GetSourceName())
    
    report.SetDataSource(source)
    return report.Generate()
}

func (re *ReportingEngine) ListAvailableDataSources() []string {
    sources := make([]string, 0, len(re.dataSources))
    for name := range re.dataSources {
        sources = append(sources, name)
    }
    return sources
}

    output, _ = engine.GenerateReport(newSalesReport, "mongodb")
    fmt.Println(output)
    
    fmt.Println("\n" + strings.Repeat("═", 60))
    fmt.Println("              END OF DEMONSTRATION")
    fmt.Println(strings.Repeat("═", 60))
}

### The Strategic Value in Enterprise Systems

The Bridge pattern delivers exceptional value in ERP environments where you face multiple orthogonal dimensions of variation. Consider these real-world scenarios where the pattern proves indispensable:

First, **multi-tenant systems with different deployment models**. Your ERP might need to support customers who use cloud databases, on-premise databases, or hybrid architectures. The Bridge pattern allows report logic to remain completely independent of the underlying storage infrastructure. You can onboard a new customer using a different database technology without touching any report code.

Second, **gradual migration scenarios**. When your organization decides to migrate from one database to another, the Bridge pattern enables a phased approach. You can run both systems in parallel, gradually moving reports from the old data source to the new one, testing thoroughly at each step. The abstraction layer protects your business logic from migration turbulence.

Third, **regulatory compliance across jurisdictions**. Different regions may require data to be stored in specific ways or accessed through particular channels. The Bridge pattern lets you plug in jurisdiction-specific data source implementations while keeping your reporting logic universal and compliant.

Fourth, **performance optimization strategies**. You might implement a caching data source that wraps the actual database, or a data source that aggregates information from multiple systems. These optimizations plug into the existing architecture without requiring changes to report implementations.

## Comparative Analysis: When to Use Which Pattern

Understanding when to apply the Composite pattern versus the Bridge pattern requires recognizing the fundamental problem each solves. These patterns address different architectural challenges, and choosing correctly depends on analyzing your domain structure and variation points.

**Use the Composite pattern when:**

- Your domain naturally exhibits hierarchical part-whole relationships
- You need to treat individual objects and groups uniformly
- Operations should propagate through tree structures
- The structure itself is the primary organizational principle
- Examples: organizational charts, bill of materials, file systems, menu structures

**Use the Bridge pattern when:**

- You have multiple independent dimensions of variation
- Different abstractions need to work with different implementations
- You want to avoid a combinatorial explosion of subclasses
- Implementation details should be hidden from abstractions
- Examples: cross-platform UI frameworks, reporting systems, payment processing, notification systems

**Can they work together?** Absolutely. In a sophisticated ERP system, you might have a Composite pattern managing your organizational hierarchy while using the Bridge pattern for report generation across that hierarchy. Each pattern plays to its strengths: Composite handles the structural complexity, while Bridge manages the implementation variety.

Consider this example: an organizational reporting system where each department (Composite node) can generate reports (Bridge abstraction) from different data sources (Bridge implementation). The Composite pattern lets you aggregate reports across the entire hierarchy, while the Bridge pattern ensures each report can access data from any source.

## Conclusion

The Composite and Bridge patterns represent essential tools in your structural design pattern toolkit, particularly when building large-scale ERP systems in Go. The Composite pattern elegantly handles hierarchical structures that pervade enterprise domains, from organizational charts to product categories to account hierarchies. It transforms potentially complex recursive operations into simple, uniform interfaces that make your code more maintainable and extensible.

The Bridge pattern addresses a different but equally critical challenge: managing multiple dimensions of variation without succumbing to class explosion. By separating abstraction from implementation, it provides the flexibility needed in modern ERP systems where business logic must remain independent of infrastructure choices, deployment models, and integration requirements.

Both patterns exemplify key principles of object-oriented design: favoring composition over inheritance, programming to interfaces rather than implementations, and keeping your code open for extension but closed for modification. As you build your ERP solution in Go, these patterns will help you create architectures that are both powerful and maintainable, growing gracefully as business requirements evolve.

Remember that patterns are not rigid recipes but flexible guidelines. Adapt them to your specific context, combine them when appropriate, and always focus on solving real problems rather than applying patterns for their own sake. The true mastery comes from recognizing the problems these patterns solve and understanding when their benefits outweigh their costs in your particular situation.
