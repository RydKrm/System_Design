# Structural Design Patterns: Composite and Bridge Patterns

## Introduction to Structural Design Patterns in ERP Systems

When you construct a large-scale Enterprise Resource Planning system, you are not merely writing code that solves individual problems—you are architecting a living, breathing digital ecosystem that must handle the complexity of real-world business operations. Structural design patterns guide you in composing objects and classes into larger structures while keeping these structures flexible and efficient. Among these patterns, the Composite and Bridge patterns stand out as particularly crucial for ERP systems, as they address fundamental challenges you will face: managing hierarchical data structures and handling multiple dimensions of variation. Let us embark on a detailed exploration of these patterns, understanding not just how they work, but why they matter in the context of enterprise software.

## The Composite Pattern: Building Tree Structures That Feel Natural

### Understanding the Core Problem

Imagine you are tasked with building the organizational module of your ERP system. Your company has a complex structure: departments contain teams, teams contain employees, and sometimes departments even contain other departments. Now, your manager asks you to implement a feature that calculates the total salary budget for any level of the organization. Perhaps they want to know the budget for the entire company, or just the engineering department, or even a specific team within that department.

Without the Composite pattern, you would face a dilemma. You could write separate logic for handling individual employees versus groups of employees, but this leads to awkward conditional statements everywhere in your code. Each time you want to perform an operation, you would need to check: "Is this an individual employee or a group? If it is a group, I need to iterate through it. But wait, what if the group contains other groups? I need nested iterations!" This approach quickly becomes unmaintainable as your hierarchy deepens.

The Composite pattern solves this problem elegantly by allowing you to treat individual objects and compositions of objects uniformly through a common interface. Whether you are working with a single employee or an entire department containing multiple teams and sub-departments, you use the same interface. The pattern handles the complexity of recursion internally, making your client code remarkably simple and clean.

### The Architectural Blueprint

The Composite pattern consists of three key players. First, you have the Component interface, which declares operations that are common to both simple and complex objects. In our organizational example, this might include operations like calculating budget, counting employees, or printing the structure. Second, you have Leaf objects, which represent the endpoints in your tree structure—objects that cannot contain other objects. In our case, individual employees are leaves. Third, you have Composite objects, which can contain both leaves and other composites. Departments and teams are composites because they can contain employees and other organizational units.

The magic happens because both leaves and composites implement the same Component interface. When you call an operation on a composite, it typically delegates that operation to all its children, aggregates the results, and returns them. The client code does not need to know whether it is dealing with a leaf or a composite—it simply calls the operation and receives the result.

package organization

import (
    "fmt"
    "strings"
    "time"
)

// OrganizationUnit is the Component interface
// Both individual employees and groups implement this interface
type OrganizationUnit interface {
    GetName() string
    GetEmployeeCount() int
    GetBudget() float64
    GetAverageSalary() float64
    PrintStructure(indent int)
    FindEmployee(name string) *Employee
    GetAllEmployees() []*Employee
}

// Employee represents a leaf node - an individual worker
type Employee struct {
    name       string
    position   string
    salary     float64
    hireDate   time.Time
    employeeID string
    department string
}

func NewEmployee(name, position, employeeID string, salary float64, hireDate time.Time) *Employee {
    return &Employee{
        name:       name,
        position:   position,
        salary:     salary,
        hireDate:   hireDate,
        employeeID: employeeID,
    }
}

func (e *Employee) GetName() string {
    return e.name
}

func (e *Employee) GetEmployeeCount() int {
    return 1
}

func (e *Employee) GetBudget() float64 {
    // Annual salary cost
    return e.salary
}

func (e *Employee) GetAverageSalary() float64 {
    return e.salary
}

func (e *Employee) PrintStructure(indent int) {
    prefix := strings.Repeat("  ", indent)
    yearsOfService := time.Since(e.hireDate).Hours() / (24 * 365)
    fmt.Printf("%s└─ %s (%s) | ID: %s | Salary: $%.2f/year | Service: %.1f years\n",
        prefix, e.name, e.position, e.employeeID, e.salary, yearsOfService)
}

func (e *Employee) FindEmployee(name string) *Employee {
    if strings.Contains(strings.ToLower(e.name), strings.ToLower(name)) {
        return e
    }
    return nil
}

func (e *Employee) GetAllEmployees() []*Employee {
    return []*Employee{e}
}

func (e *Employee) GetPosition() string {
    return e.position
}

func (e *Employee) GetSalary() float64 {
    return e.salary
}

func (e *Employee) GetHireDate() time.Time {
    return e.hireDate
}

// Department represents a composite node - can contain employees and subdepartments
type Department struct {
    name         string
    manager      *Employee
    units        []OrganizationUnit
    overhead     float64 // Fixed operational costs
    budget       float64 // Additional discretionary budget
    departmentID string
}

func NewDepartment(name, departmentID string, manager *Employee, overhead, budget float64) *Department {
    dept := &Department{
        name:         name,
        departmentID: departmentID,
        manager:      manager,
        overhead:     overhead,
        budget:       budget,
        units:        make([]OrganizationUnit, 0),
    }
    if manager != nil {
        manager.department = name
    }
    return dept
}

func (d *Department) Add(unit OrganizationUnit) {
    d.units = append(d.units, unit)
}

func (d *Department) Remove(unit OrganizationUnit) {
    for i, u := range d.units {
        if u == unit {
            d.units = append(d.units[:i], d.units[i+1:]...)
            return
        }
    }
}

func (d *Department) GetName() string {
    return d.name
}

func (d *Department) GetEmployeeCount() int {
    count := 0
    if d.manager != nil {
        count = 1
    }
    for _, unit := range d.units {
        count += unit.GetEmployeeCount()
    }
    return count
}

func (d *Department) GetBudget() float64 {
    totalBudget := d.overhead + d.budget
    
    // Add manager's salary if exists
    if d.manager != nil {
        totalBudget += d.manager.GetBudget()
    }
    
    // Recursively add all subordinate units' budgets
    for _, unit := range d.units {
        totalBudget += unit.GetBudget()
    }
    
    return totalBudget
}

func (d *Department) GetAverageSalary() float64 {
    employeeCount := d.GetEmployeeCount()
    if employeeCount == 0 {
        return 0
    }
    
    totalSalaries := 0.0
    if d.manager != nil {
        totalSalaries += d.manager.GetSalary()
    }
    
    for _, unit := range d.units {
        employees := unit.GetAllEmployees()
        for _, emp := range employees {
            totalSalaries += emp.GetSalary()
        }
    }
    
    return totalSalaries / float64(employeeCount)
}

func (d *Department) PrintStructure(indent int) {
    prefix := strings.Repeat("  ", indent)
    fmt.Printf("%s┌─ [%s] | ID: %s | Employees: %d | Budget: $%.2f | Avg Salary: $%.2f\n",
        prefix, d.name, d.departmentID, d.GetEmployeeCount(), d.GetBudget(), d.GetAverageSalary())
    
    if d.manager != nil {
        fmt.Printf("%s│  Manager:\n", prefix)
        d.manager.PrintStructure(indent + 1)
    }
    
    if len(d.units) > 0 {
        fmt.Printf("%s│  Units:\n", prefix)
        for _, unit := range d.units {
            unit.PrintStructure(indent + 1)
        }
    }
}

func (d *Department) FindEmployee(name string) *Employee {
    // Check manager first
    if d.manager != nil {
        if found := d.manager.FindEmployee(name); found != nil {
            return found
        }
    }
    
    // Search in all subordinate units
    for _, unit := range d.units {
        if found := unit.FindEmployee(name); found != nil {
            return found
        }
    }
    
    return nil
}

func (d *Department) GetAllEmployees() []*Employee {
    employees := make([]*Employee, 0)
    
    if d.manager != nil {
        employees = append(employees, d.manager)
    }
    
    for _, unit := range d.units {
        employees = append(employees, unit.GetAllEmployees()...)
    }
    
    return employees
}

func (d *Department) GetDepartmentID() string {
    return d.departmentID
}

func (d *Department) GetOverhead() float64 {
    return d.overhead
}

// Team represents a specialized composite for smaller groups
type Team struct {
    name       string
    teamLead   *Employee
    members    []OrganizationUnit
    teamBudget float64
}

func NewTeam(name string, teamLead *Employee, teamBudget float64) *Team {
    return &Team{
        name:       name,
        teamLead:   teamLead,
        members:    make([]OrganizationUnit, 0),
        teamBudget: teamBudget,
    }
}

func (t *Team) Add(member OrganizationUnit) {
    t.members = append(t.members, member)
}

func (t *Team) Remove(member OrganizationUnit) {
    for i, m := range t.members {
        if m == member {
            t.members = append(t.members[:i], t.members[i+1:]...)
            return
        }
    }
}

func (t *Team) GetName() string {
    return t.name
}

func (t *Team) GetEmployeeCount() int {
    count := 0
    if t.teamLead != nil {
        count = 1
    }
    for _, member := range t.members {
        count += member.GetEmployeeCount()
    }
    return count
}

func (t *Team) GetBudget() float64 {
    budget := t.teamBudget
    if t.teamLead != nil {
        budget += t.teamLead.GetBudget()
    }
    for _, member := range t.members {
        budget += member.GetBudget()
    }
    return budget
}

func (t *Team) GetAverageSalary() float64 {
    employeeCount := t.GetEmployeeCount()
    if employeeCount == 0 {
        return 0
    }
    
    totalSalaries := 0.0
    if t.teamLead != nil {
        totalSalaries += t.teamLead.GetSalary()
    }
    
    for _, member := range t.members {
        employees := member.GetAllEmployees()
        for _, emp := range employees {
            totalSalaries += emp.GetSalary()
        }
    }
    
    return totalSalaries / float64(employeeCount)
}

func (t *Team) PrintStructure(indent int) {
    prefix := strings.Repeat("  ", indent)
    fmt.Printf("%s┌─ 【Team: %s】 | Members: %d | Budget: $%.2f\n",
        prefix, t.name, t.GetEmployeeCount(), t.GetBudget())
    
    if t.teamLead != nil {
        fmt.Printf("%s│  Team Lead:\n", prefix)
        t.teamLead.PrintStructure(indent + 1)
    }
    
    if len(t.members) > 0 {
        fmt.Printf("%s│  Team Members:\n", prefix)
        for _, member := range t.members {
            member.PrintStructure(indent + 1)
        }
    }
}

func (t *Team) FindEmployee(name string) *Employee {
    if t.teamLead != nil {
        if found := t.teamLead.FindEmployee(name); found != nil {
            return found
        }
    }
    
    for _, member := range t.members {
        if found := member.FindEmployee(name); found != nil {
            return found
        }
    }
    
    return nil
}

func (t *Team) GetAllEmployees() []*Employee {
    employees := make([]*Employee, 0)
    
    if t.teamLead != nil {
        employees = append(employees, t.teamLead)
    }
    
    for _, member := range t.members {
        employees = append(employees, member.GetAllEmployees()...)
    }
    
    return employees
}

// OrganizationAnalyzer provides advanced analysis capabilities
type OrganizationAnalyzer struct {
    root OrganizationUnit
}

func NewOrganizationAnalyzer(root OrganizationUnit) *OrganizationAnalyzer {
    return &OrganizationAnalyzer{root: root}
}

func (oa *OrganizationAnalyzer) GetTotalHeadcount() int {
    return oa.root.GetEmployeeCount()
}

func (oa *OrganizationAnalyzer) GetTotalBudget() float64 {
    return oa.root.GetBudget()
}

func (oa *OrganizationAnalyzer) GetCompanyAverageSalary() float64 {
    return oa.root.GetAverageSalary()
}

func (oa *OrganizationAnalyzer) SearchEmployee(name string) *Employee {
    return oa.root.FindEmployee(name)
}

func (oa *OrganizationAnalyzer) GetSalaryDistribution() map[string]int {
    distribution := make(map[string]int)
    employees := oa.root.GetAllEmployees()
    
    for _, emp := range employees {
        salary := emp.GetSalary()
        switch {
        case salary < 50000:
            distribution["< $50k"]++
        case salary < 75000:
            distribution["$50k - $75k"]++
        case salary < 100000:
            distribution["$75k - $100k"]++
        case salary < 150000:
            distribution["$100k - $150k"]++
        default:
            distribution["> $150k"]++
        }
    }
    
    return distribution
}

func (oa *OrganizationAnalyzer) PrintFullHierarchy() {
    fmt.Println("\n========================================")
    fmt.Println("    ORGANIZATION STRUCTURE")
    fmt.Println("========================================")
    oa.root.PrintStructure(0)
    fmt.Println("========================================")
}

### Why This Pattern Transforms Your ERP Architecture

The Composite pattern brings transformative benefits to your ERP system. First and foremost, it provides uniform treatment of objects. Your client code does not need to distinguish between single objects and compositions—both are accessed through the same interface. This dramatically simplifies the code that uses these structures, eliminating conditional logic and making operations feel natural and intuitive.

Second, it enables recursive operations to be expressed cleanly. When you call `GetBudget()` on the company root, it automatically propagates through the entire hierarchy, aggregating results at each level. You do not write the recursion yourself—the pattern handles it. This makes complex calculations remarkably simple to implement and maintain.

Third, it makes the structure extensible. Adding new types of organizational units requires minimal changes to existing code. You simply implement the `OrganizationUnit` interface, and the new type integrates seamlessly into the hierarchy. Want to add project teams that span departments? Contractors that work across multiple teams? The pattern accommodates these additions gracefully.
