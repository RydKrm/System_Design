# Adapter Pattern in Go: A Thorough Guide for Beginners

## The Essence of the Adapter Pattern

Imagine you have just returned from a journey abroad, toting a sleek new laptop whose power plug has flat, two-pronged ends. You arrive home, eager to charge it, only to realize that your wall socket demands round, three-pronged plugs. The physical shapes simply do not match. You do not discard either device; instead, you reach for a small plastic accessory—a travel adapter—that sits between the socket and your charger, converting the incompatible interface without altering the original hardware.

In software engineering, the Adapter pattern (also known as the Wrapper pattern) performs exactly the same role. It is a structural design pattern that allows objects with mismatched interfaces to collaborate. It converts the interface of one class into another interface that the client code expects. The core triumph of this pattern is that it lets otherwise incompatible classes work together without modifying their source code. In statically typed languages like Go, where interfaces define precise contracts, the Adapter pattern becomes an indispensable tool for integrating legacy code, third‑party libraries, or disparate subsystems into a cohesive whole.

The pattern is formally categorised as a structural pattern from the Gang of Four catalogue. It encapsulates the logic required to map one interface onto another, thereby preserving the open/closed principle—classes should be open for extension but closed for modification. By introducing an adapter, you shield the client from the complexity and variety of the adaptee, presenting a unified view.

## The Core Participants

Every implementation of the Adapter pattern revolves around three key players:

- **Target**: This is the interface that the client code understands and interacts with. The client’s entire universe revolves around this contract; it knows nothing about the underlying implementations beyond what the target promises.
- **Adaptee**: The existing class or component that possesses the functionality we desperately need, but whose interface is incompatible with the target. The adaptee is often a legacy module, a third‑party library, or any piece of code we cannot (or should not) change.
- **Adapter**: The bridge between the two worlds. The adapter is a concrete struct that implements the target interface while internally holding a reference to an adaptee. When the client calls a method declared in the target, the adapter translates that call into one or more calls on the adaptee, performing any necessary parameter conversion or data restructuring along the way.

A fourth, implicit participant is the **Client**, which collaborates only with objects that satisfy the target interface, remaining blissfully unaware of the adapter’s existence.

### Visualising the Relationship

An ASCII‑art diagram helps fix these relationships in the mind. In the following sketch, the client sees only the target interface. The adapter, which implements that interface, wraps the adaptee and delegates work to it.

```text
+----------------+            +---------------------+
|    Client      |            |   «interface»       |
|                |----------->|     Target          |
+----------------+            +----------+----------+
                                         ^
                                         |
                               +---------+---------+
                               |      Adapter      |          +-----------------+
                               | - adaptee: Adaptee|--------->|     Adaptee     |
                               +-------------------+          +-----------------+
```

In a language like Go, where we favour composition over inheritance, the Object Adapter variant is the natural choice. The adapter holds an instance of the adaptee as a field (composed) and implements the target interface. There is also a Class Adapter variant that relies on multiple inheritance to simultaneously inherit from a target class and an adaptee class, but Go intentionally omits classical inheritance, so we concentrate on the object adapter.

## The Go Implementation: From Theory to Code

Let us ground these ideas with a concrete Go example. Suppose you are building a modern document‑processing system. The new system works with a clean `Document` struct and expects any printer to adhere to an elegant `ModernPrinter` interface:

```go
// Target interface
type ModernPrinter interface {
    PrintDocument(doc Document) error
}

type Document struct {
    Title string
    Body  string
}
```

Elsewhere in the company, a battle‑tested legacy subsystem exists—a `LegacyPrinter` class that can only accept plain strings. Its interface looks entirely different:

```go
// Adaptee: the legacy component we cannot modify
type LegacyPrinter struct{}

func (lp *LegacyPrinter) Print(s string) error {
    fmt.Println("Legacy Printer Output:", s)
    return nil
}
```

The client code that orchestrates document printing expects to use a `ModernPrinter`. Directly passing a `LegacyPrinter` would result in a compilation error because `LegacyPrinter` does not implement `ModernPrinter`. This is where we fashion an adapter.

### Crafting the Adapter

We create a struct called `LegacyPrinterAdapter` that holds a `*LegacyPrinter` and satisfies the `ModernPrinter` contract.

```go
// Adapter: implements ModernPrinter by wrapping LegacyPrinter
type LegacyPrinterAdapter struct {
    legacy *LegacyPrinter
}

func NewLegacyPrinterAdapter(lp *LegacyPrinter) *LegacyPrinterAdapter {
    return &LegacyPrinterAdapter{legacy: lp}
}

// The adapter translates the Document to a string and delegates.
func (a *LegacyPrinterAdapter) PrintDocument(doc Document) error {
    // Convert the Document into the plain string the adaptee expects.
    printableContent := fmt.Sprintf("Title: %s\nBody: %s", doc.Title, doc.Body)
    return a.legacy.Print(printableContent)
}
```

The adapter’s `PrintDocument` method converts the rich `Document` object into the simple string that the legacy system understands. Any additional logic—splitting, escaping, formatting—lives here, not in the client.

### Bringing It All Together

A client that expects a `ModernPrinter` can now seamlessly consume the legacy printer through the adapter:

```go
func main() {
    doc := Document{
        Title: "Adapter Pattern",
        Body:  "Converting interfaces since 1994.",
    }

    legacy := &LegacyPrinter{}
    adapted := NewLegacyPrinterAdapter(legacy)

    // Client code works only with the ModernPrinter interface.
    var printer ModernPrinter = adapted
    _ = printer.PrintDocument(doc)
}
```

When the client invokes `printer.PrintDocument(doc)`, the adapter’s implementation springs into action, formats the document, and forwards the call to `LegacyPrinter.Print`. The client remains oblivious to the archaic string‑based printer behind the curtain.

## Covering Every Nuance

### Object Adapter versus Class Adapter in Go

Though Go lacks classical inheritance, a discussion of the two adapter forms remains instructive. The **Object Adapter** uses composition, exactly as shown above. It is flexible because one adapter can work with any concrete adaptee that matches the required type, or even with an interface if we widen the adapter’s field appropriately. The **Class Adapter** in languages such as C++ or Java uses multiple inheritance to inherit both the target interface and the adaptee’s implementation. In Go, a pseudo class‑adapter can be achieved via struct embedding. If we want the adapter to expose the adaptee’s methods directly (perhaps because the target interface is broader), we can embed the adaptee and then override or add methods. For example:

```go
type EmbeddedAdapter struct {
    LegacyPrinter
}

func (ea *EmbeddedAdapter) PrintDocument(doc Document) error {
    // Uses the embedded LegacyPrinter.Print method.
    return ea.Print(doc.Body)
}
```

This approach is less common because it couples the adapter permanently to a concrete adaptee and may accidentally leak unwanted methods into the adapter’s public surface. The object adapter with an explicit field remains the idiomatic and safer choice.

### Two‑Way Adapters and Pluggable Adapters

Sometimes a single adapter must work in both directions, enabling a new system to call an old one and vice versa. This is a two‑way adapter. In Go, it simply means that the adapter implements both the target interface and the adaptee’s interface, or provides conversion methods that map calls bi‑directionally. The structure does not fundamentally change; it just adds more methods.

A **pluggable adapter** is a more dynamic idea. Instead of creating a separate adapter type for every adaptee, we employ higher‑order functions. For instance, Go’s `http.HandlerFunc` is an adapter that converts a plain function signature `func(ResponseWriter, *Request)` into the `http.Handler` interface. This adapter is “pluggable” because any matching function instantly becomes a handler. Similarly, you could write a generic adapter factory:

```go
type Converter func(string) Document

type FunctionalAdapter struct {
    printer  LegacyPrinter
    converter Converter
}

func (fa *FunctionalAdapter) PrintDocument(doc Document) error {
    return fa.printer.Print(fa.converter(doc))
}
```

By injecting the conversion logic, the same adapter can serve multiple adaptee variations without proliferating types.

### Adapting Interfaces to Interfaces

Adapters are not limited to wrapping concrete structs. Frequently, we must adapt one interface to another. Go’s standard library brims with such examples. The `io.Reader` interface (with `Read(p []byte) (n int, err error)`) is a universal abstraction, but sometimes a function expects an `io.ReadCloser`. The `ioutil.NopCloser` function takes an `io.Reader` and returns an `io.ReadCloser` by wrapping it in an adapter that adds a no‑op `Close` method. Here, the adaptee is itself an interface, and the adapter is a thin struct that delegates `Read` and implements `Close`.

Similarly, the `io.Writer` can be adapted to an `io.Reader` via `io.Pipe()`, though that involves more synchronisation internally. The pattern remains the same: a new struct holds a reference to the adaptee and conforms to the desired target.

### Recognising the Adapter in Go’s Ecosystem

Once you internalise the pattern, you will spot it everywhere:

- `sort.Reverse` adapts a `sort.Interface` into a reversed order.
- `context.WithTimeout` adapts a parent context into a child that carries a cancellation deadline.
- `http.StripPrefix` adapts an `http.Handler` by stripping a URL prefix before delegating.

All these functions return a new object that implements the same interface while adding or transforming behaviour—a classic adapter signature.

### When to Apply the Adapter Pattern

You reach for the Adapter pattern when you encounter a class, struct, or even a function that provides exactly the behaviour your system needs, but its method names or type signatures do not align with the interface your code expects. Rather than rewriting the existing component (which may be impossible or risky) or complicating your own code with conditional logic, you introduce an adapter. The effort of writing that small translation layer pays for itself many times over by preserving separation of concerns and enabling painless integration.

This pattern shines especially bright when working with third‑party libraries. You can wrap their concrete types in adapters that expose only the methods your application cares about, keeping your own interface clean and insulated from library churn. If the library ever changes, only the adapter needs to be updated—the rest of your system remains blissfully unaffected.

### Benefits and Trade‑offs

**Benefits:**

- **Single Responsibility Principle:** The translation logic is isolated within the adapter, so neither the client nor the adaptee must be modified.
- **Open/Closed Principle:** You can introduce new adapters without altering existing classes or client code.
- **Seamless Integration:** Enables reuse of existing functionality that would otherwise be discarded due to interface mismatch.
- **Flexible Design:** Because Go interfaces are satisfied implicitly, adapters are unobtrusive—you can slip them into any code that depends on the target interface.

**Trade‑offs:**

- **Increased Complexity:** Introducing an additional layer of indirection can make the codebase harder to navigate if overused or poorly documented.
- **Performance Overhead:** Each call through an adapter adds a layer of function invocation and possible data conversion. In performance‑critical paths, this must be carefully considered, though in most applications it is negligible.
- **Over‑adapting:** Creating adapters for every minor discrepancy can lead to a tangle of small wrapper types. Reserve the pattern for genuine incompatibilities that hinder collaboration.

## Conclusion

The Adapter pattern is the bridge that allows our software to speak the same language despite dialectical differences. In Go, it manifests as a humble struct that wraps an adaptee, fulfills a target interface, and gently translates one call into another. Mastering this pattern means recognising that when interfaces disagree, you need not renegotiate the contract—you can simply provide a translator. By embedding this understanding deeply, you will build systems that are more modular, more accepting of legacy components, and vastly easier to evolve.
