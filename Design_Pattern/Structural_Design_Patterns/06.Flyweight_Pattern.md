# Flyweight Pattern in Go: A Complete Guide for Beginners

## The Principle of Shared Lightness

Imagine a writer composing a novel in a word processor. Each character on the screen is drawn with a particular glyph shape, a font face, a size, and a colour. If the processor created a brand‑new object containing all that data for every single character—a document of ten thousand letters—the memory consumption would be staggering. Instead, the application separates the character’s intrinsic properties (the glyph itself, the font metrics), which are shared across all occurrences, from its extrinsic properties (the position on the page, the specific colour applied), which vary. The intrinsic part is stored once and referenced by every occurrence. This idea, of pulling out the shared invariant data and reusing it, is the soul of the Flyweight pattern.

The Flyweight pattern is a structural design pattern that lets you fit more objects into available memory by sharing common parts of state among multiple objects instead of keeping all of the data in each object. It divides an object’s state into two categories: **intrinsic state**, which is constant and can be shared across contexts, and **extrinsic state**, which is context‑dependent and passed in by the client at the time of use. The pattern manufactures a pool of flyweight objects—lightweight objects containing only intrinsic state—and reuses them mercilessly, thus drastically reducing the number of instances that need to exist at any moment.

In Go, the Flyweight pattern is most naturally expressed using a factory that manages a map of flyweight instances, returning an existing one when a similar request arrives. The flyweight struct itself is immutable (or treated as such), ensuring that sharing is safe. This pattern shines when an application must spawn a huge number of similar objects, such as character glyphs in a text editor, particle effects in a game, or product variants in an e‑commerce catalogue.

## The Core Participants

The Flyweight pattern orchestrates four roles:

- **Flyweight**: An interface through which flyweight objects can receive and act on extrinsic state. It declares methods that the client calls, supplying the varying, context‑specific information.
- **ConcreteFlyweight**: A struct that implements the `Flyweight` interface and stores the intrinsic state—the data that can be shared. Instances of this struct are reused across many objects.
- **FlyweightFactory**: A creator and keeper of flyweight objects. It maintains a pool (often a `map`) of existing flyweights and ensures that they are created only once. When a client asks for a flyweight, the factory returns an existing instance or creates a new one, registers it, and then returns it.
- **Client**: The code that maintains or computes the extrinsic state and calls flyweight methods, passing the extrinsic state as arguments. The client does not create flyweights directly; it always asks the factory.

Sometimes objects that cannot be shared are called **UnsharedConcreteFlyweight**. In practice, the pattern often permits a mixture: some leaf objects are flyweights, while others remain full‑fledged objects.

The following diagram illustrates how the factory guards the flyweight pool and how the client interacts with flyweights using extrinsic state.

The factory owns all flyweights. The client never creates them directly. For each unique intrinsic key, exactly one flyweight instance exists.

## A Gentle Introduction Through Go Code

Let us build a simple graphical forest renderer that places trees on a canvas. Every tree has a name, a colour, and a texture (think of the bark pattern). These three properties are the intrinsic state—they are identical for every tree of the same species. The tree’s location (x, y coordinates) and its size are extrinsic: they change for every individual tree.

We start with the flyweight interface and a concrete flyweight.

```go
// Flyweight interface
type TreeFlyweight interface {
    Render(x, y, size int)
}

// ConcreteFlyweight – holds intrinsic, sharable state
type TreeType struct {
    name    string
    color   string
    texture string
}

func (t *TreeType) Render(x, y, size int) {
    fmt.Printf("Rendering %s tree (color:%s, texture:%s) at (%d,%d) size %d\n",
        t.name, t.color, t.texture, x, y, size)
}
```

Now, the factory that manages and reuses `TreeType` instances.

```go
// FlyweightFactory
type TreeFactory struct {
    pool map[string]*TreeType
}

func NewTreeFactory() *TreeFactory {
    return &TreeFactory{pool: make(map[string]*TreeType)}
}

func (f *TreeFactory) GetTreeType(name, color, texture string) *TreeType {
    key := name + "_" + color + "_" + texture
    if existing, ok := f.pool[key]; ok {
        return existing
    }
    // Create, store, and return
    newType := &TreeType{name: name, color: color, texture: texture}
    f.pool[key] = newType
    return newType
}
```

The client maintains the extrinsic state in a separate struct, which holds a reference to the shared flyweight.

```go
// Client-side object containing extrinsic state
type Tree struct {
    x, y, size int
    treeType   *TreeType // reference to the flyweight
}

func NewTree(x, y, size int, treeType *TreeType) *Tree {
    return &Tree{x: x, y: y, size: size, treeType: treeType}
}

func (t *Tree) Render() {
    t.treeType.Render(t.x, t.y, t.size)
}
```

Now the orchestrating code demonstrates the memory saving. Even though we plant a forest of a thousand trees, only a handful of `TreeType` flyweights are created.

```go
func main() {
    factory := NewTreeFactory()
    forest := make([]*Tree, 0, 1000)

    // Plant 1000 trees of only 3 species
    for i := 0; i < 300; i++ {
        treeType := factory.GetTreeType("Oak", "Green", "Rough")
        forest = append(forest, NewTree(i*10, i*5, 15, treeType))
    }
    for i := 0; i < 400; i++ {
        treeType := factory.GetTreeType("Pine", "Dark Green", "Smooth")
        forest = append(forest, NewTree(i*8, i*3, 20, treeType))
    }
    for i := 0; i < 300; i++ {
        treeType := factory.GetTreeType("Cherry", "Pink", "Delicate")
        forest = append(forest, NewTree(i*12, i*7, 10, treeType))
    }

    fmt.Printf("Total trees: %d\n", len(forest))
    fmt.Printf("Flyweight TreeType instances: %d\n", len(factory.pool))
    // Renders the first few trees
    for _, tree := range forest[:3] {
        tree.Render()
    }
}
```

Output:
```
Total trees: 1000
Flyweight TreeType instances: 3
Rendering Oak tree (color:Green, texture:Rough) at (0,0) size 15
Rendering Oak tree (color:Green, texture:Rough) at (10,5) size 15
Rendering Oak tree (color:Green, texture:Rough) at (20,10) size 15
```

We have a thousand trees but only three `TreeType` flyweights. The memory saved is proportional to the number of trees minus the number of unique types, which in a real application with thousands of objects is enormous.

## Covering Every Detail

### Intrinsic versus Extrinsic State

The critical design step is separating the two kinds of state. Intrinsic state is context‑independent and can be shared without alteration. It is typically set when the flyweight is constructed and never changed (immutability). Extrinsic state depends on the object’s context and is supplied by the client at method invocation time. Identifying intrinsic state may require a careful analysis: ask yourself, “Would two objects always have the same value for this field if they represent the same kind of thing?” If yes, it is intrinsic. A classic example: in a text editor, the character ‘A’ in 12‑point Times New Roman has an intrinsic glyph shape and metrics; its row and column on the page are extrinsic.

### Immutability and Thread Safety

Because flyweights are shared, they must behave as if they are immutable. If multiple client objects reference the same flyweight and one modifies it, the change leaks to all other clients, often with disastrous consequences. In Go, you can enforce immutability by not exporting the fields of the flyweight struct and providing only read‑through methods. In concurrent programs, immutability also eliminates the need for synchronisation, making flyweights inherently goroutine‑safe.

### The Flyweight Factory in Go

The factory is typically a struct that owns a `sync.Mutex` along with the pool map if flyweights are created concurrently. Go’s `sync.Map` can also serve for read‑heavy pools. In many applications, the factory is a singleton or is held in a global context, though package‑level functions with a package‑private pool are equally idiomatic. For example:

```go
var (
    poolMutex sync.Mutex
    pool      = make(map[string]*TreeType)
)

func GetTreeType(name, color, texture string) *TreeType {
    key := name + "_" + color + "_" + texture
    poolMutex.Lock()
    defer poolMutex.Unlock()
    if tt, ok := pool[key]; ok {
        return tt
    }
    tt := &TreeType{name: name, color: color, texture: texture}
    pool[key] = tt
    return tt
}
```

This global approach removes the need to pass the factory around, though it may complicate testing. Dependency injection of the factory remains a cleaner, more testable pattern.

### Flyweight and the Go Standard Library

Go’s `sync.Pool` is a flyweight‑like construct that reuses temporary objects to relieve pressure on the garbage collector. While `sync.Pool` does not enforce intrinsic/extrinsic separation, its spirit—reuse instead of instantiate—is deeply aligned with the Flyweight pattern. Similarly, `strings.Builder` internally uses a buffer that might be pooled, and the `regexp` package caches compiled regular expressions to avoid re‑compilation. The `image` package provides concrete colour palettes where a single `color.RGBA` value is shared. Recognising these as flyweights helps you grasp the pattern’s practical utility.

### When to Utilise the Flyweight Pattern

The Flyweight pattern is most beneficial when:

- Your application must support a huge number of objects that would otherwise consume prohibitive amounts of memory.
- Most of the object’s state can be made extrinsic and the number of distinct intrinsic states is relatively small.
- The objects’ identities do not matter because they represent value types rather than entities with independent lifetimes. In other words, the objects are conceptually interchangeable once the extrinsic state is supplied.
- You are willing to trade a slightly more complex design (separated state, factory lookups) for a dramatic reduction in memory footprint.

### Benefits and Trade‑offs

**Benefits:**

- **Memory Efficiency:** By sharing intrinsic state, you can eliminate the memory cost of duplicate data across thousands or millions of objects.
- **Performance:** Fewer allocations mean reduced garbage‑collection pressure, which can improve throughput in high‑load systems.
- **Centralised Control:** The factory provides a single point where intrinsic state is managed, which can simplify configuration changes (e.g., switching a font or a texture).

**Trade‑offs:**

- **Complexity:** The separation of state into intrinsic and extrinsic can complicate the codebase. Developers must constantly track what belongs where.
- **Runtime Lookup Cost:** A map lookup is required each time a flyweight is retrieved. For performance‑critical paths, this overhead must be measured, though it is usually far outweighed by memory savings.
- **Difficult Debugging:** Because many logical objects share the same physical flyweight instance, inspecting state or tracing bugs can become trickier.
- **Not for Everything:** If the number of objects is small or the intrinsic state varies frequently, the flyweight offers no advantage and adds unnecessary layers.

### Flyweight versus Other Patterns

The Flyweight pattern often works in tandem with the **Composite** pattern to represent a hierarchical structure where many leaf nodes share flyweights. A **Factory Method** or **Abstract Factory** can be employed to create the flyweight factory. **Singleton** can be applied to the factory to ensure only one pool exists. The **State** pattern can use flyweights to represent shared state objects. Understanding these synergies allows you to combine patterns for robust solutions.

## Conclusion

The Flyweight pattern is the art of squeezing abundance into a small space by sharing what is common and supplying what is unique on demand. In Go, with its pragmatic approach to composition, immutability, and concurrency, flyweights are easily built using factories and maps. When you stand before a legion of objects all bearing the same core data, remember: separate the constant from the variable, pool the constant, and let the variable fly in as a parameter. What emerges is a design that is both memory‑wise and elegantly transparent.
