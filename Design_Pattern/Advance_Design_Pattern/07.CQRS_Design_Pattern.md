# CQRS Pattern: Command Query Responsibility Segregation for Enterprise ERP Systems

## Introduction: The Challenge of Complex Data Operations

When you build an Enterprise Resource Planning system, you quickly discover that reading data and writing data have fundamentally different characteristics and requirements. Consider an order management module: when a customer places an order, you execute a complex write operation that validates inventory, calculates pricing, reserves stock, processes payment, and triggers notifications. This write operation must maintain consistency, enforce business rules, and handle transactional boundaries carefully. Conversely, when a manager views a sales dashboard, they execute a read operation that aggregates data from multiple tables, applies filters, calculates summaries, and formats results for display. This read operation prioritizes speed, flexibility, and rich querying capabilities.

Yet in traditional architectures, we handle both concerns with the same model. The same entity classes serve both write operations and read operations. The same database schema supports both transactional updates and analytical queries. The same API endpoints handle both commands that change state and queries that retrieve state. This conflation of concerns creates tension and compromise. Write operations suffer from complex queries needed for reads. Read operations suffer from normalized structures optimized for writes. Performance suffers as a single database must excel at both transaction processing and analytical querying.

The CQRS pattern—Command Query Responsibility Segregation—proposes a radical solution: separate the write side from the read side entirely. Commands that modify state use one model optimized for writes. Queries that retrieve state use a different model optimized for reads. This separation eliminates the compromises inherent in trying to serve both concerns with a single model, allowing each side to be optimized independently for its specific requirements.

Let us explore how CQRS works in practice, why it matters for complex ERP systems, when to apply it, and how to implement it effectively in Go.

## Understanding CQRS: The Core Principle

### The Traditional CRUD Approach

In traditional CRUD (Create, Read, Update, Delete) architectures, a single model serves all operations. Let us examine this approach and understand its limitations.

### The CQRS Approach: Separation of Concerns

CQRS fundamentally separates the command side (writes) from the query side (reads), allowing each to use models and storage optimized for their specific needs.

```go
package cqrs

import (
    "fmt"
    "time"
)

// ✅ CQRS: Separate models for commands and queries

// ============================================================================
// COMMAND SIDE (Write Model) - Optimized for transactional integrity
// ============================================================================

// Commands represent intentions to change state
type CreateProductCommand struct {
    ID          string
    Name        string
    Description string
    SKU         string
    Price       float64
    CostPrice   float64
    Category    string
    InitialStock int
    SupplierID  string
}

type UpdateProductPriceCommand struct {
    ProductID string
    NewPrice  float64
}

type AdjustStockCommand struct {
    ProductID string
    Quantity  int    // Positive for increase, negative for decrease
    Reason    string // "sale", "restock", "adjustment"
}

// Write model - focused on business rules and transactions
type ProductWriteModel struct {
    ID           string
    Name         string
    SKU          string
    Price        float64
    CostPrice    float64
    Stock        int
    Category     string
    SupplierID   string
    ReorderLevel int
    Version      int // For optimistic locking
    CreatedAt    time.Time
    UpdatedAt    time.Time
} 

// Command handlers enforce business rules
type ProductCommandHandler struct {
    writeStore ProductWriteStore
    eventBus   EventBus
}

func NewProductCommandHandler(store ProductWriteStore, bus EventBus) *ProductCommandHandler {
    return &ProductCommandHandler{
        writeStore: store,
        eventBus:   bus,
    }
}

func (h *ProductCommandHandler) HandleCreateProduct(cmd CreateProductCommand) error {
    fmt.Printf("\n[Command Handler] Creating product: %s\n", cmd.Name)
    
    // Business rule: SKU must be unique
    if h.writeStore.ExistsBySKU(cmd.SKU) {
        return fmt.Errorf("SKU already exists: %s", cmd.SKU)
    }
    
    // Business rule: Price must be greater than cost
    if cmd.Price <= cmd.CostPrice {
        return fmt.Errorf("price must be greater than cost price")
    }
    
    // Create write model
    product := &ProductWriteModel{
        ID:         cmd.ID,
        Name:       cmd.Name,
        SKU:        cmd.SKU,
        Price:      cmd.Price,
        CostPrice:  cmd.CostPrice,
        Stock:      cmd.InitialStock,
        Category:   cmd.Category,
        SupplierID: cmd.SupplierID,
        Version:    1,
        CreatedAt:  time.Now(),
        UpdatedAt:  time.Now(),
    }
    
    // Persist to write store
    if err := h.writeStore.Save(product); err != nil {
        return fmt.Errorf("failed to save product: %w", err)
    }
    
    // Publish event for read model synchronization
    event := ProductCreatedEvent{
        ProductID:   cmd.ID,
        Name:        cmd.Name,
        SKU:         cmd.SKU,
        Price:       cmd.Price,
        Category:    cmd.Category,
        Stock:       cmd.InitialStock,
        OccurredAt:  time.Now(),
    }
    
    h.eventBus.Publish("ProductCreated", event)
    
    fmt.Printf("[Command Handler] Product created successfully: %s\n", cmd.ID)
    return nil
}

func (h *ProductCommandHandler) HandleUpdatePrice(cmd UpdateProductPriceCommand) error {
    fmt.Printf("\n[Command Handler] Updating price for product: %s\n", cmd.ProductID)
    
    product, err := h.writeStore.GetByID(cmd.ProductID)
    if err != nil {
        return err
    }
    
    oldPrice := product.Price
    
    // Business rule: Price must be greater than cost
    if cmd.NewPrice <= product.CostPrice {
        return fmt.Errorf("new price must be greater than cost price")
    }
    
    // Update write model
    product.Price = cmd.NewPrice
    product.UpdatedAt = time.Now()
    product.Version++
    
    if err := h.writeStore.Save(product); err != nil {
        return err
    }
    
    // Publish event
    event := ProductPriceChangedEvent{
        ProductID:  cmd.ProductID,
        OldPrice:   oldPrice,
        NewPrice:   cmd.NewPrice,
        OccurredAt: time.Now(),
    }
    
    h.eventBus.Publish("ProductPriceChanged", event)
    
    fmt.Printf("[Command Handler] Price updated: $%.2f -> $%.2f\n", oldPrice, cmd.NewPrice)
    return nil
}

func (h *ProductCommandHandler) HandleAdjustStock(cmd AdjustStockCommand) error {
    fmt.Printf("\n[Command Handler] Adjusting stock for product: %s by %d\n", 
        cmd.ProductID, cmd.Quantity)
    
    product, err := h.writeStore.GetByID(cmd.ProductID)
    if err != nil {
        return err
    }
    
    // Business rule: Stock cannot go negative
    newStock := product.Stock + cmd.Quantity
    if newStock < 0 {
        return fmt.Errorf("insufficient stock: current=%d, requested=%d", 
            product.Stock, -cmd.Quantity)
    }
    
    oldStock := product.Stock
    product.Stock = newStock
    product.UpdatedAt = time.Now()
    product.Version++
    
    if err := h.writeStore.Save(product); err != nil {
        return err
    }
    
    // Publish event
    event := StockAdjustedEvent{
        ProductID:  cmd.ProductID,
        OldStock:   oldStock,
        NewStock:   newStock,
        Quantity:   cmd.Quantity,
        Reason:     cmd.Reason,
        OccurredAt: time.Now(),
    }
    
    h.eventBus.Publish("StockAdjusted", event)
    
    fmt.Printf("[Command Handler] Stock adjusted: %d -> %d\n", oldStock, newStock)
    return nil
}

// Events represent things that happened
type ProductCreatedEvent struct {
    ProductID  string
    Name       string
    SKU        string
    Price      float64
    Category   string
    Stock      int
    OccurredAt time.Time
}

type ProductPriceChangedEvent struct {
    ProductID  string
    OldPrice   float64
    NewPrice   float64
    OccurredAt time.Time
}

type StockAdjustedEvent struct {
    ProductID  string
    OldStock   int
    NewStock   int
    Quantity   int
    Reason     string
    OccurredAt time.Time
}

// Event handler updates read model
type ProductReadModelSynchronizer struct {
    readStore ProductReadStore
}

func NewProductReadModelSynchronizer(store ProductReadStore) *ProductReadModelSynchronizer {
    return &ProductReadModelSynchronizer{
        readStore: store,
    }
}

func (s *ProductReadModelSynchronizer) OnProductCreated(event ProductCreatedEvent) {
    fmt.Printf("[Read Sync] Synchronizing ProductCreated event: %s\n", event.ProductID)
    
    readModel := &ProductReadModel{
        ID:          event.ProductID,
        Name:        event.Name,
        SKU:         event.SKU,
        Price:       event.Price,
        Category:    event.Category,
        Stock:       event.Stock,
        TotalSold:   0,
        Revenue:     0,
        IsLowStock:  event.Stock < 10,
        LastUpdated: time.Now(),
    }
    
    s.readStore.Upsert(readModel)
}

func (s *ProductReadModelSynchronizer) OnProductPriceChanged(event ProductPriceChangedEvent) {
    fmt.Printf("[Read Sync] Synchronizing ProductPriceChanged event: %s\n", event.ProductID)
    
    readModel, err := s.readStore.GetByID(event.ProductID)
    if err != nil {
        fmt.Printf("[Read Sync] Error: %v\n", err)
        return
    }
    
    readModel.Price = event.NewPrice
    readModel.LastUpdated = time.Now()
    
    s.readStore.Upsert(readModel)
}

func (s *ProductReadModelSynchronizer) OnStockAdjusted(event StockAdjustedEvent) {
    fmt.Printf("[Read Sync] Synchronizing StockAdjusted event: %s\n", event.ProductID)
    
    readModel, err := s.readStore.GetByID(event.ProductID)
    if err != nil {
        fmt.Printf("[Read Sync] Error: %v\n", err)
        return
    }
    
    readModel.Stock = event.NewStock
    readModel.IsLowStock = event.NewStock < 10
    
    // If this was a sale, update sales stats
    if event.Reason == "sale" {
        readModel.TotalSold += -event.Quantity // Quantity is negative for sales
        readModel.Revenue += readModel.Price * float64(-event.Quantity)
        now := time.Now()
        readModel.LastSoldAt = &now
    }
    
    readModel.LastUpdated = time.Now()
    
    s.readStore.Upsert(readModel)
}

type SimpleEventBus struct {
    handlers map[string][]interface{}
}

func NewSimpleEventBus() *SimpleEventBus {
    return &SimpleEventBus{
        handlers: make(map[string][]interface{}),
    }
}

func (bus *SimpleEventBus) Publish(eventType string, event interface{}) error {
    fmt.Printf("[Event Bus] Publishing %s event\n", eventType)
    
    handlers, exists := bus.handlers[eventType]
    if !exists {
        return nil
    }
    
    for _, handler := range handlers {
        switch eventType {
        case "ProductCreated":
            if h, ok := handler.(func(ProductCreatedEvent)); ok {
                h(event.(ProductCreatedEvent))
            }
        case "ProductPriceChanged":
            if h, ok := handler.(func(ProductPriceChangedEvent)); ok {
                h(event.(ProductPriceChangedEvent))
            }
        case "StockAdjusted":
            if h, ok := handler.(func(StockAdjustedEvent)); ok {
                h(event.(StockAdjustedEvent))
            }
        }
    }
    
    return nil
}

func (bus *SimpleEventBus) Subscribe(eventType string, handler interface{}) error {
    bus.handlers[eventType] = append(bus.handlers[eventType], handler)
    fmt.Printf("[Event Bus] Subscribed to %s events\n", eventType)
    return nil
}
```

## When to Use CQRS

### Scenarios Where CQRS Shines

CQRS is not a universal pattern—it adds complexity and should be applied judiciously. However, in certain scenarios, its benefits dramatically outweigh its costs.

**Complex Domain Logic**: When your write operations involve intricate business rules, multiple validations, and complex state transitions, CQRS allows you to model these concerns separately from read concerns.

**Performance Requirements Differ**: When writes need ACID transactions but reads need fast aggregations and denormalized data, CQRS allows each side to use appropriate storage and optimization strategies.

**Different Scalability Needs**: When your system has far more reads than writes (or vice versa), CQRS enables independent scaling. You can scale the read side horizontally without affecting the write side.

**Multiple Read Models**: When different users need different views of the same data—executives need dashboards, analysts need reports, mobile apps need simplified views—CQRS allows multiple read models optimized for each use case.

**Event Sourcing Integration**: When you want to maintain a complete audit trail of all changes, CQRS pairs naturally with Event Sourcing, where the write model appends events and read models are projections of those events.

### Scenarios Where CQRS Adds Unnecessary Complexity

**Simple CRUD Applications**: If your application is primarily simple create, read, update, delete operations without complex business logic, traditional architectures suffice.

**Small Data Volumes**: If your data fits comfortably in memory and queries are already fast, the complexity of CQRS provides no benefit.

**Immediate Consistency Required**: If your application cannot tolerate eventual consistency between write and read models, CQRS becomes problematic because synchronization takes time.

**Small Team, Limited Expertise**: CQRS requires understanding distributed systems concepts. Small teams without this expertise may find the complexity overwhelming.

### Pattern 2: Event Sourcing with CQRS

CQRS pairs naturally with Event Sourcing, where the write model consists of an event store and read models are projections.

```go
package eventsourcing

import (
    "fmt"
    "time"
)

// Event sourcing: Store events, not current state
type Event struct {
    ID            string
    AggregateID   string
    AggregateType string
    EventType     string
    Data          interface{}
    Version       int
    Timestamp     time.Time
}

// Account aggregate rebuilt from events
type AccountAggregate struct {
    ID             string
    AccountNumber  string
    Balance        float64
    Status         string
    Version        int
    UncommittedEvents []Event
}

func (a *AccountAggregate) Deposit(amount float64) {
    event := Event{
        ID:            generateID(),
        AggregateID:   a.ID,
        AggregateType: "Account",
        EventType:     "MoneyDeposited",
        Data: MoneyDepositedEvent{
            Amount: amount,
        },
        Version:   a.Version + 1,
        Timestamp: time.Now(),
    }
    
    a.Apply(event)
    a.UncommittedEvents = append(a.UncommittedEvents, event)
}

func (a *AccountAggregate) Withdraw(amount float64) error {
    if a.Balance < amount {
        return fmt.Errorf("insufficient funds")
    }
    
    event := Event{
        ID:            generateID(),
        AggregateID:   a.ID,
        AggregateType: "Account",
        EventType:     "MoneyWithdrawn",
        Data: MoneyWithdrawnEvent{
            Amount: amount,
        },
        Version:   a.Version + 1,
        Timestamp: time.Now(),
    }
    
    a.Apply(event)
    a.UncommittedEvents = append(a.UncommittedEvents, event)
    return nil
}

func (a *AccountAggregate) Apply(event Event) {
    switch event.EventType {
    case "AccountOpened":
        data := event.Data.(AccountOpenedEvent)
        a.AccountNumber = data.AccountNumber
        a.Balance = data.InitialDeposit
        a.Status = "Active"
        
    case "MoneyDeposited":
        data := event.Data.(MoneyDepositedEvent)
        a.Balance += data.Amount
        
    case "MoneyWithdrawn":
        data := event.Data.(MoneyWithdrawnEvent)
        a.Balance -= data.Amount
        
    case "AccountClosed":
        a.Status = "Closed"
    }
    
    a.Version = event.Version
}

// Event types
type AccountOpenedEvent struct {
    AccountNumber  string
    CustomerID     string
    InitialDeposit float64
}

type MoneyDepositedEvent struct {
    Amount float64
}

type MoneyWithdrawnEvent struct {
    Amount float64
}

// Event store
type EventStore interface {
    SaveEvents(aggregateID string, events []Event, expectedVersion int) error
    GetEvents(aggregateID string) ([]Event, error)
}

// Read model projection
type AccountBalanceProjection struct {
    AccountID      string
    AccountNumber  string
    CurrentBalance float64
    LastUpdated    time.Time
}

type AccountBalanceProjector struct {
    store AccountBalanceStore
}

func (p *AccountBalanceProjector) Project(event Event) {
    var balance *AccountBalanceProjection
    var err error
    
    balance, err = p.store.GetByAccountID(event.AggregateID)
    if err != nil {
        balance = &AccountBalanceProjection{
            AccountID: event.AggregateID,
        }
    }
    
    switch event.EventType {
    case "AccountOpened":
        data := event.Data.(AccountOpenedEvent)
        balance.AccountNumber = data.AccountNumber
        balance.CurrentBalance = data.InitialDeposit
        
    case "MoneyDeposited":
        data := event.Data.(MoneyDepositedEvent)
        balance.CurrentBalance += data.Amount
        
    case "MoneyWithdrawn":
        data := event.Data.(MoneyWithdrawnEvent)
        balance.CurrentBalance -= data.Amount
    }
    
    balance.LastUpdated = time.Now()
    p.store.Save(balance)
}

type AccountBalanceStore interface {
    Save(projection *AccountBalanceProjection) error
    GetByAccountID(accountID string) (*AccountBalanceProjection, error)
}

func generateID() string {
    return fmt.Sprintf("evt-%d", time.Now().UnixNano())
}

func DemonstrateEventSourcingWithCQRS() {
    fmt.Println("╔════════════════════════════════════════════════════════════╗")
    fmt.Println("║         EVENT SOURCING + CQRS                              ║")
    fmt.Println("╚════════════════════════════════════════════════════════════╝")
    
    fmt.Println("\nEvent Sourcing writes events, CQRS projects to read models:")
    fmt.Println("  ✓ Complete audit trail")
    fmt.Println("  ✓ Can rebuild state from events")
    fmt.Println("  ✓ Time travel (replay to any point)")
    fmt.Println("  ✓ Multiple projections from same events")
}
```

## Handling Eventual Consistency

### Understanding the Consistency Trade-off

CQRS typically involves eventual consistency between the write and read models. When a command executes, the write model updates immediately, but read models may take milliseconds (or longer) to synchronize.

```go
package consistency

import (
    "fmt"
    "time"
)

// Demonstrating eventual consistency

func ProcessOrder(commandHandler *OrderCommandHandler, queryHandler *OrderQueryHandler) {
    fmt.Println("╔════════════════════════════════════════════════════════════╗")
    fmt.Println("║         EVENTUAL CONSISTENCY IN CQRS                       ║")
    fmt.Println("╚════════════════════════════════════════════════════════════╝\n")
    
    orderID := "ORD-2024-001"
    
    // Execute command (write side)
    fmt.Println("=== Time T+0ms: Executing Command ===")
    fmt.Printf("Creating order: %s\n", orderID)
    // commandHandler.HandleCreateOrder(...)
    fmt.Println("✓ Command completed - order saved to write store")
    
    // Read immediately (before sync)
    fmt.Println("\n=== Time T+1ms: Immediate Read ===")
    fmt.Println("Querying order immediately after command...")
    // order, err := queryHandler.GetOrder(orderID)
    fmt.Println("✗ Order not found in read model yet (eventual consistency)")
    
    // Simulate event propagation delay
    fmt.Println("\n=== Time T+50ms: Event Processing ===")
    time.Sleep(50 * time.Millisecond)
    fmt.Println("Event bus delivering OrderCreated event to projections...")
    fmt.Println("✓ Read model synchronized")
    
    // Read after sync
    fmt.Println("\n=== Time T+51ms: Query After Sync ===")
    fmt.Println("Querying order again...")
    fmt.Println("✓ Order found in read model")
    
    fmt.Println("\n\nHandling Eventual Consistency:")
    fmt.Println("  1. Accept it: Most queries don't need immediate consistency")
    fmt.Println("  2. Redirect: Return command result directly, avoid immediate query")
    fmt.Println("  3. UI feedback: Show 'processing' state")
    fmt.Println("  4. Versioning: Include version numbers in responses")
    fmt.Println("  5. Compensate: Allow retry mechanisms")
}
```

## Best Practices for CQRS Implementation

### Practice 1: Start Simple, Evolve as Needed

Don't implement full CQRS with event sourcing and multiple read models on day one. Start with simple command/query separation and evolve.

```go

// - Simple CRUD operations

// - No separate storage yet

type ProductService struct {
    repo ProductRepository

    // Write logic with validation
    return nil

    // Read logic
    return nil, nil

// - Still same database

// Phase 4: Full CQRS (when needed)
```

Commands should represent business operations, not technical operations.

## Conclusion: CQRS as a Powerful Tool for Complex Systems

Command Query Responsibility Segregation is not a pattern to apply universally, but when your ERP system faces complex business logic, differing performance requirements for reads and writes, and needs for multiple specialized views of data, CQRS provides a powerful architectural foundation.

By separating the command side from the query side, you gain the freedom to optimize each independently. The write side can focus on enforcing business rules, maintaining transactional consistency, and protecting data integrity. The read side can denormalize data, cache aggressively, and provide specialized views optimized for each use case.

The trade-offs—increased complexity, eventual consistency, more moving parts—are real and must be carefully considered. Start simple, apply CQRS only where its benefits clearly outweigh its costs, and evolve your architecture as your system's complexity demands it.

In Go, with its excellent support for interfaces, concurrent programming, and clear separation of concerns, implementing CQRS can feel natural and idiomatic. Your command handlers become clear expressions of business intent. Your query handlers become efficient data retrieval mechanisms. And the separation between them keeps your codebase organized and maintainable as your ERP system grows in complexity.
