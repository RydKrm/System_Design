# Dependency Injection Pattern for Building Enterprise ERP Systems in Go

## Introduction: The Problem of Tight Coupling

When you begin building an Enterprise Resource Planning system, you will write components that depend on other components. Your `OrderService` needs a database to store orders. Your `InvoiceGenerator` needs an email service to send invoices. Your `PaymentProcessor` needs a logging system to track transactions. The natural, intuitive way to handle these dependencies is to create them directly where you need them. Inside `OrderService`, you instantiate a database connection. Inside `InvoiceGenerator`, you create an email client. Inside `PaymentProcessor`, you initialize a logger.

This approach feels simple and straightforward. You write `db := NewPostgresDatabase()` right where you need the database, and everything works. But this simplicity is deceptive. You have created tight coupling between your components and their dependencies. Your `OrderService` cannot function without PostgreSQL. Your `InvoiceGenerator` cannot be tested without sending real emails. Your `PaymentProcessor` cannot be used in any context without its specific logger implementation.

As your ERP system grows, this tight coupling becomes increasingly problematic. Testing becomes difficult because you cannot isolate components. Reusing code becomes impossible because components drag their dependencies with them. Changing implementations requires modifying multiple components. Your codebase becomes rigid, fragile, and resistant to change.

Dependency Injection is the antidote to tight coupling. Rather than components creating their own dependencies, dependencies are provided (injected) from the outside. This simple inversion of control transforms your architecture, making it testable, flexible, and maintainable. Let us explore how dependency injection works in Go, why it matters profoundly for ERP systems, and how to apply it effectively.

## Understanding Dependency Injection: The Core Concept

### What is a Dependency?

Before we can inject dependencies, we must understand what a dependency is. In software, a dependency is any object or service that a component needs to perform its function. If `OrderService` needs a database to store orders, the database is a dependency of `OrderService`. If `EmailSender` needs an SMTP client to send emails, the SMTP client is a dependency of `EmailSender`.

Dependencies typically fall into several categories:

**Data access dependencies** provide persistence capabilities—databases, file systems, caches, and external APIs that store and retrieve data.

**Communication dependencies** enable interaction with external systems—email services, message queues, HTTP clients, and notification systems.

**Cross-cutting dependencies** provide infrastructure concerns—logging, metrics, authentication, and configuration management.

**Business logic dependencies** represent other domain services that perform specific business functions needed by the component.

### The Traditional Approach: Direct Instantiation

Let us examine the traditional approach to handling dependencies and understand why it creates problems.

```go
package traditional

import (
    "fmt"
    "time"
)

// ❌ BAD: Tight coupling through direct instantiation
// Dependencies are created inside the component

type PostgresDatabase struct {
    connectionString string
}

func NewPostgresDatabase() *PostgresDatabase {
    return &PostgresDatabase{
        connectionString: "postgresql://localhost:5432/erp",
    }
}

func (db *PostgresDatabase) SaveOrder(orderID string, data map[string]interface{}) error {
    fmt.Printf("[PostgreSQL] Saving order %s\n", orderID)
    return nil
}

func (db *PostgresDatabase) GetOrder(orderID string) (map[string]interface{}, error) {
    fmt.Printf("[PostgreSQL] Retrieving order %s\n", orderID)
    return map[string]interface{}{"id": orderID}, nil
}

type SMTPEmailService struct {
    smtpServer string
}

func NewSMTPEmailService() *SMTPEmailService {
    return &SMTPEmailService{
        smtpServer: "smtp.company.com:587",
    }
}

func (es *SMTPEmailService) SendEmail(to, subject, body string) error {
    fmt.Printf("[SMTP] Sending email to %s: %s\n", to, subject)
    return nil
}

type FileLogger struct {
    logFile string
}

func NewFileLogger() *FileLogger {
    return &FileLogger{
        logFile: "/var/log/app.log",
    }
}

func (fl *FileLogger) Log(message string) {
    fmt.Printf("[File Logger] %s\n", message)
}

// OrderService with tight coupling
type TightlyCoupledOrderService struct {
    // Dependencies are concrete types, created internally
    database     *PostgresDatabase
    emailService *SMTPEmailService
    logger       *FileLogger
}

func NewTightlyCoupledOrderService() *TightlyCoupledOrderService {
    // Creating dependencies directly - TIGHT COUPLING!
    return &TightlyCoupledOrderService{
        database:     NewPostgresDatabase(),
        emailService: NewSMTPEmailService(),
        logger:       NewFileLogger(),
    }
}

func (os *TightlyCoupledOrderService) CreateOrder(customerID string, items []string, total float64) error {
    os.logger.Log("Creating order...")
    
    orderID := fmt.Sprintf("ORD-%d", time.Now().Unix())
    orderData := map[string]interface{}{
        "customer_id": customerID,
        "items":       items,
        "total":       total,
    }
    
    // Tightly coupled to PostgreSQL
    if err := os.database.SaveOrder(orderID, orderData); err != nil {
        os.logger.Log(fmt.Sprintf("Failed to save order: %v", err))
        return err
    }
    
    // Tightly coupled to SMTP
    if err := os.emailService.SendEmail(customerID, "Order Confirmation", "Your order has been placed"); err != nil {
        os.logger.Log(fmt.Sprintf("Failed to send email: %v", err))
        // Continue even if email fails
    }
    
    os.logger.Log(fmt.Sprintf("Order created successfully: %s", orderID))
    return nil
}

// Problems with tight coupling:
// 1. Cannot test without PostgreSQL, SMTP server, and file system
// 2. Cannot use different database in different environments
// 3. Cannot mock dependencies for unit testing
// 4. Cannot reuse service with different implementations
// 5. Hard to understand what dependencies the service needs
// 6. Changes to dependencies require changing OrderService

func DemonstrateTightCoupling() {
    fmt.Println("╔════════════════════════════════════════════════════════════╗")
    fmt.Println("║         TIGHT COUPLING: THE PROBLEM                        ║")
    fmt.Println("╚════════════════════════════════════════════════════════════╝")
    
    fmt.Println("\nProblems with tight coupling:")
    fmt.Println("  ✗ OrderService cannot be tested without real database")
    fmt.Println("  ✗ OrderService cannot be tested without real SMTP server")
    fmt.Println("  ✗ Cannot use different database for testing")
    fmt.Println("  ✗ Cannot mock dependencies")
    fmt.Println("  ✗ Service is not reusable in different contexts")
    fmt.Println("  ✗ Hard to understand dependencies from outside")
    
    service := NewTightlyCoupledOrderService()
    service.CreateOrder("CUST-001", []string{"Item1", "Item2"}, 299.99)
    
    fmt.Println("\nThis service can ONLY work with:")
    fmt.Println("  - PostgreSQL database")
    fmt.Println("  - SMTP email service")
    fmt.Println("  - File-based logger")
}
```

### The Dependency Injection Approach

Dependency Injection inverts the control flow. Instead of components creating their dependencies, dependencies are provided to components from the outside. This is typically accomplished in Go through constructor injection, where dependencies are passed as parameters to the constructor function.

```go
package injection

import (
    "fmt"
    "time"
)

// ✅ GOOD: Loose coupling through dependency injection
// Dependencies are defined as interfaces and injected from outside

// Define interfaces for dependencies
type Database interface {
    SaveOrder(orderID string, data map[string]interface{}) error
    GetOrder(orderID string) (map[string]interface{}, error)
}

type EmailService interface {
    SendEmail(to, subject, body string) error
}

type Logger interface {
    Log(message string)
}

// Concrete implementations
type PostgresDatabase struct {
    connectionString string
}

func NewPostgresDatabase(connString string) *PostgresDatabase {
    return &PostgresDatabase{
        connectionString: connString,
    }
}

func (db *PostgresDatabase) SaveOrder(orderID string, data map[string]interface{}) error {
    fmt.Printf("[PostgreSQL] Saving order %s to %s\n", orderID, db.connectionString)
    return nil
}

func (db *PostgresDatabase) GetOrder(orderID string) (map[string]interface{}, error) {
    fmt.Printf("[PostgreSQL] Retrieving order %s\n", orderID)
    return map[string]interface{}{"id": orderID}, nil
}

type SMTPEmailService struct {
    smtpServer string
}

func NewSMTPEmailService(server string) *SMTPEmailService {
    return &SMTPEmailService{
        smtpServer: server,
    }
}

func (es *SMTPEmailService) SendEmail(to, subject, body string) error {
    fmt.Printf("[SMTP] Sending email to %s via %s\n", to, es.smtpServer)
    return nil
}

type ConsoleLogger struct{}

func NewConsoleLogger() *ConsoleLogger {
    return &ConsoleLogger{}
}

func (cl *ConsoleLogger) Log(message string) {
    fmt.Printf("[Console Logger] %s\n", message)
}

// OrderService with dependency injection
type OrderService struct {
    // Dependencies are interfaces, injected from outside
    database     Database
    emailService EmailService
    logger       Logger
}

// Constructor accepts dependencies as parameters
func NewOrderService(db Database, email EmailService, logger Logger) *OrderService {
    return &OrderService{
        database:     db,
        emailService: email,
        logger:       logger,
    }
}

func (os *OrderService) CreateOrder(customerID string, items []string, total float64) error {
    os.logger.Log("Creating order...")
    
    orderID := fmt.Sprintf("ORD-%d", time.Now().Unix())
    orderData := map[string]interface{}{
        "customer_id": customerID,
        "items":       items,
        "total":       total,
    }
    
    // Works with ANY database implementation
    if err := os.database.SaveOrder(orderID, orderData); err != nil {
        os.logger.Log(fmt.Sprintf("Failed to save order: %v", err))
        return err
    }
    
    // Works with ANY email service implementation
    if err := os.emailService.SendEmail(customerID, "Order Confirmation", "Your order has been placed"); err != nil {
        os.logger.Log(fmt.Sprintf("Failed to send email: %v", err))
    }
    
    os.logger.Log(fmt.Sprintf("Order created successfully: %s", orderID))
    return nil
}

func DemonstrateDependencyInjection() {
    fmt.Println("╔════════════════════════════════════════════════════════════╗")
    fmt.Println("║         DEPENDENCY INJECTION: THE SOLUTION                 ║")
    fmt.Println("╚════════════════════════════════════════════════════════════╝")
    
    fmt.Println("\nBenefits of dependency injection:")
    fmt.Println("  ✓ OrderService can be tested with mock dependencies")
    fmt.Println("  ✓ Can use different implementations in different contexts")
    fmt.Println("  ✓ Dependencies are explicit in constructor signature")
    fmt.Println("  ✓ Service is reusable with any compatible implementations")
    fmt.Println("  ✓ Easy to swap implementations without changing service")
    
    // Create real implementations
    db := NewPostgresDatabase("postgresql://localhost:5432/production")
    email := NewSMTPEmailService("smtp.company.com:587")
    logger := NewConsoleLogger()
    
    // Inject dependencies
    service := NewOrderService(db, email, logger)
    
    fmt.Println("\n=== Using Production Dependencies ===")
    service.CreateOrder("CUST-001", []string{"Item1", "Item2"}, 299.99)
    
    // Easy to use different implementations!
    testDB := NewPostgresDatabase("postgresql://localhost:5432/test")
    testEmail := &MockEmailService{}
    testLogger := &MockLogger{}
    
    testService := NewOrderService(testDB, testEmail, testLogger)
    
    fmt.Println("\n=== Using Test Dependencies ===")
    testService.CreateOrder("CUST-TEST", []string{"TestItem"}, 50.00)
}

// Mock implementations for testing
type MockEmailService struct {
    sentEmails []string
}

func (mes *MockEmailService) SendEmail(to, subject, body string) error {
    fmt.Printf("[Mock Email] Captured email to %s (not actually sent)\n", to)
    mes.sentEmails = append(mes.sentEmails, to)
    return nil
}

type MockLogger struct {
    logs []string
}

func (ml *MockLogger) Log(message string) {
    fmt.Printf("[Mock Logger] Captured log: %s\n", message)
    ml.logs = append(ml.logs, message)
}
```

## Types of Dependency Injection in Go

### Constructor Injection (Most Common in Go)

Constructor injection is the most prevalent form of dependency injection in Go. Dependencies are passed as parameters to the constructor function (typically a `New*` function) and stored as fields in the struct.

```go
package constructor

import (
    "fmt"
    "time"
)

// Interfaces define contracts
type PaymentGateway interface {
    ProcessPayment(amount float64, cardToken string) (string, error)
}

type NotificationService interface {
    NotifyPaymentSuccess(customerID, transactionID string) error
    NotifyPaymentFailure(customerID string, reason string) error
}

type AuditLogger interface {
    LogPayment(customerID, transactionID string, amount float64, status string)
}

// PaymentProcessor with constructor injection
type PaymentProcessor struct {
    gateway      PaymentGateway
    notification NotificationService
    audit        AuditLogger
    maxRetries   int
}

// Constructor explicitly declares all dependencies
func NewPaymentProcessor(
    gateway PaymentGateway,
    notification NotificationService,
    audit AuditLogger,
) *PaymentProcessor {
    return &PaymentProcessor{
        gateway:      gateway,
        notification: notification,
        audit:        audit,
        maxRetries:   3,
    }
}

// With optional configuration
func NewPaymentProcessorWithConfig(
    gateway PaymentGateway,
    notification NotificationService,
    audit AuditLogger,
    maxRetries int,
) *PaymentProcessor {
    return &PaymentProcessor{
        gateway:      gateway,
        notification: notification,
        audit:        audit,
        maxRetries:   maxRetries,
    }
}

func (pp *PaymentProcessor) ProcessCustomerPayment(customerID string, amount float64, cardToken string) error {
    fmt.Printf("\n[Payment Processor] Processing payment of $%.2f for customer %s\n", amount, customerID)
    
    var transactionID string
    var err error
    
    // Attempt payment with retries
    for attempt := 1; attempt <= pp.maxRetries; attempt++ {
        transactionID, err = pp.gateway.ProcessPayment(amount, cardToken)
        
        if err == nil {
            // Payment successful
            pp.audit.LogPayment(customerID, transactionID, amount, "success")
            pp.notification.NotifyPaymentSuccess(customerID, transactionID)
            
            fmt.Printf("[Payment Processor] Payment successful: %s\n", transactionID)
            return nil
        }
        
        fmt.Printf("[Payment Processor] Attempt %d failed: %v\n", attempt, err)
        
        if attempt < pp.maxRetries {
            time.Sleep(time.Duration(attempt) * time.Second)
        }
    }
    
    // Payment failed after all retries
    pp.audit.LogPayment(customerID, "", amount, "failed")
    pp.notification.NotifyPaymentFailure(customerID, err.Error())
    
    return fmt.Errorf("payment failed after %d attempts: %w", pp.maxRetries, err)
}

// Concrete implementations
type StripeGateway struct {
    apiKey string
}

func NewStripeGateway(apiKey string) *StripeGateway {
    return &StripeGateway{apiKey: apiKey}
}

func (sg *StripeGateway) ProcessPayment(amount float64, cardToken string) (string, error) {
    fmt.Printf("[Stripe] Processing $%.2f with token %s\n", amount, cardToken)
    time.Sleep(100 * time.Millisecond) // Simulate API call
    
    transactionID := fmt.Sprintf("STRIPE-TXN-%d", time.Now().Unix())
    return transactionID, nil
}

type EmailNotificationService struct {
    fromAddress string
}

func NewEmailNotificationService(fromAddress string) *EmailNotificationService {
    return &EmailNotificationService{fromAddress: fromAddress}
}

func (ens *EmailNotificationService) NotifyPaymentSuccess(customerID, transactionID string) error {
    fmt.Printf("[Email] Sending success notification to %s for transaction %s\n", customerID, transactionID)
    return nil
}

func (ens *EmailNotificationService) NotifyPaymentFailure(customerID string, reason string) error {
    fmt.Printf("[Email] Sending failure notification to %s: %s\n", customerID, reason)
    return nil
}

type DatabaseAuditLogger struct {
    connectionString string
}

func NewDatabaseAuditLogger(connString string) *DatabaseAuditLogger {
    return &DatabaseAuditLogger{connectionString: connString}
}

func (dal *DatabaseAuditLogger) LogPayment(customerID, transactionID string, amount float64, status string) {
    fmt.Printf("[Audit DB] Logging payment: Customer=%s, Transaction=%s, Amount=$%.2f, Status=%s\n",
        customerID, transactionID, amount, status)
}

func DemonstrateConstructorInjection() {
    fmt.Println("╔════════════════════════════════════════════════════════════╗")
    fmt.Println("║         CONSTRUCTOR INJECTION                              ║")
    fmt.Println("╚════════════════════════════════════════════════════════════╝")
    
    // Create dependencies
    gateway := NewStripeGateway("sk_test_12345")
    notification := NewEmailNotificationService("noreply@company.com")
    audit := NewDatabaseAuditLogger("postgresql://localhost:5432/audit")
    
    // Inject dependencies via constructor
    processor := NewPaymentProcessor(gateway, notification, audit)
    
    // Use the processor
    processor.ProcessCustomerPayment("CUST-001", 199.99, "tok_visa_4242")
    
    fmt.Println("\n✓ Dependencies clearly visible in constructor")
    fmt.Println("✓ All dependencies required at creation time")
    fmt.Println("✓ Immutable once created (dependencies don't change)")
}
```

### Method Injection (Less Common)

Method injection passes dependencies as parameters to individual methods rather than storing them in the struct. This is useful when a dependency is only needed for specific operations.

```go
package method

import "fmt"

// Method injection - dependencies passed per method call

type Report struct {
    Title string
    Data  []string
}

type ReportGenerator struct {
    title string
}

func NewReportGenerator(title string) *ReportGenerator {
    return &ReportGenerator{title: title}
}

// Formatter is injected per method call
type ReportFormatter interface {
    Format(report *Report) string
}

func (rg *ReportGenerator) GenerateReport(data []string, formatter ReportFormatter) string {
    report := &Report{
        Title: rg.title,
        Data:  data,
    }
    
    // Use injected formatter
    return formatter.Format(report)
}

type PDFFormatter struct{}

func (pf *PDFFormatter) Format(report *Report) string {
    return fmt.Sprintf("[PDF] %s: %v", report.Title, report.Data)
}

type ExcelFormatter struct{}

func (ef *ExcelFormatter) Format(report *Report) string {
    return fmt.Sprintf("[Excel] %s: %v", report.Title, report.Data)
}

func DemonstrateMethodInjection() {
    fmt.Println("\n=== Method Injection ===")
    
    generator := NewReportGenerator("Sales Report")
    data := []string{"Q1: $100k", "Q2: $150k", "Q3: $200k"}
    
    // Different formatter per call
    pdfOutput := generator.GenerateReport(data, &PDFFormatter{})
    fmt.Println(pdfOutput)
    
    excelOutput := generator.GenerateReport(data, &ExcelFormatter{})
    fmt.Println(excelOutput)
    
    fmt.Println("\n✓ Useful when dependency varies per operation")
}
```

### Interface Injection (Rare in Go)

Interface injection involves the dependency providing an injector method that injects itself into clients. This is rare in Go because it requires the client to expose a setter interface.

```go
package interfaceinjection

import "fmt"

// Interface injection - dependency injects itself into client

type LoggerInjector interface {
    InjectLogger(logger Logger)
}

type Logger interface {
    Log(message string)
}

type Service struct {
    logger Logger
}

func (s *Service) InjectLogger(logger Logger) {
    s.logger = logger
    fmt.Println("[Service] Logger injected")
}

func (s *Service) DoWork() {
    if s.logger != nil {
        s.logger.Log("Service doing work")
    }
}

type SimpleLogger struct{}

func (sl *SimpleLogger) Log(message string) {
    fmt.Printf("[Logger] %s\n", message)
}

func DemonstrateInterfaceInjection() {
    fmt.Println("\n=== Interface Injection ===")
    
    service := &Service{}
    logger := &SimpleLogger{}
    
    // Logger injects itself
    service.InjectLogger(logger)
    service.DoWork()
    
    fmt.Println("\n✓ Rarely used in Go - constructor injection preferred")
}
```

## Building a Dependency Injection Container for Go

While Go does not have a built-in DI framework like some languages, we can create a simple, type-safe container to manage dependencies in large applications.

```go
package container

import (
    "fmt"
    "sync"
)

// Container manages dependencies for the application
type Container struct {
    services map[string]interface{}
    mu       sync.RWMutex
}

func NewContainer() *Container {
    return &Container{
        services: make(map[string]interface{}),
    }
}

// Register adds a service to the container
func (c *Container) Register(name string, service interface{}) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.services[name] = service
    fmt.Printf("[Container] Registered service: %s\n", name)
}

// Get retrieves a service from the container
func (c *Container) Get(name string) (interface{}, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    service, exists := c.services[name]
    return service, exists
}

// MustGet retrieves a service or panics if not found
func (c *Container) MustGet(name string) interface{} {
    service, exists := c.Get(name)
    if !exists {
        panic(fmt.Sprintf("service not found: %s", name))
    }
    return service
}

// Type-safe service registry using generics-like pattern
type ServiceRegistry struct {
    container *Container
}

func NewServiceRegistry(container *Container) *ServiceRegistry {
    return &ServiceRegistry{container: container}
}

// Database service
type DatabaseService interface {
    Query(sql string) ([]map[string]interface{}, error)
    Execute(sql string) error
}

type PostgresService struct {
    connectionString string
}

func NewPostgresService(connString string) *PostgresService {
    return &PostgresService{connectionString: connString}
}

func (ps *PostgresService) Query(sql string) ([]map[string]interface{}, error) {
    fmt.Printf("[Postgres] Executing query: %s\n", sql)
    return []map[string]interface{}{{"id": 1}}, nil
}

func (ps *PostgresService) Execute(sql string) error {
    fmt.Printf("[Postgres] Executing command: %s\n", sql)
    return nil
}

// Cache service
type CacheService interface {
    Get(key string) (interface{}, bool)
    Set(key string, value interface{})
}

type RedisCache struct {
    address string
}

func NewRedisCache(address string) *RedisCache {
    return &RedisCache{address: address}
}

func (rc *RedisCache) Get(key string) (interface{}, bool) {
    fmt.Printf("[Redis] Getting key: %s\n", key)
    return nil, false
}

func (rc *RedisCache) Set(key string, value interface{}) {
    fmt.Printf("[Redis] Setting key: %s\n", key)
}

// Logger service
type LoggerService interface {
    Info(message string)
    Error(message string)
    Debug(message string)
}

type StructuredLogger struct {
    level string
}

func NewStructuredLogger(level string) *StructuredLogger {
    return &StructuredLogger{level: level}
}

func (sl *StructuredLogger) Info(message string) {
    fmt.Printf("[INFO] %s\n", message)
}

func (sl *StructuredLogger) Error(message string) {
    fmt.Printf("[ERROR] %s\n", message)
}

func (sl *StructuredLogger) Debug(message string) {
    if sl.level == "debug" {
        fmt.Printf("[DEBUG] %s\n", message)
    }
}

// Application service using injected dependencies from container
type UserService struct {
    db     DatabaseService
    cache  CacheService
    logger LoggerService
}

func NewUserService(db DatabaseService, cache CacheService, logger LoggerService) *UserService {
    return &UserService{
        db:     db,
        cache:  cache,
        logger: logger,
    }
}

func (us *UserService) GetUser(userID string) (map[string]interface{}, error) {
    us.logger.Info(fmt.Sprintf("Fetching user: %s", userID))
    
    // Check cache first
    if cached, exists := us.cache.Get(userID); exists {
        us.logger.Debug("User found in cache")
        return cached.(map[string]interface{}), nil
    }
    
    // Query database
    results, err := us.db.Query(fmt.Sprintf("SELECT * FROM users WHERE id = '%s'", userID))
    if err != nil {
        us.logger.Error(fmt.Sprintf("Database query failed: %v", err))
        return nil, err
    }
    
    if len(results) == 0 {
        return nil, fmt.Errorf("user not found")
    }
    
    user := results[0]
    
    // Store in cache
    us.cache.Set(userID, user)
    
    return user, nil
}

// Bootstrap function to wire up all dependencies
func Bootstrap() *Container {
    container := NewContainer()
    
    // Register infrastructure services
    db := NewPostgresService("postgresql://localhost:5432/erp")
    container.Register("database", db)
    
    cache := NewRedisCache("localhost:6379")
    container.Register("cache", cache)
    
    logger := NewStructuredLogger("info")
    container.Register("logger", logger)
    
    // Register application services with dependencies
    userService := NewUserService(
        db,
        cache,
        logger,
    )
    container.Register("userService", userService)
    
    return container
}

func DemonstrateContainer() {
    fmt.Println("╔════════════════════════════════════════════════════════════╗")
    fmt.Println("║         DEPENDENCY INJECTION CONTAINER                     ║")
    fmt.Println("╚════════════════════════════════════════════════════════════╝\n")
    
    // Bootstrap application
    container := Bootstrap()
    
    // Retrieve and use services
    userService := container.MustGet("userService").(*UserService)
    
    user, err := userService.GetUser("USER-001")
    if err != nil {
        fmt.Printf("Error: %v\n", err)
    } else {
        fmt.Printf("Retrieved user: %v\n", user)
    }
    
    fmt.Println("\n✓ Container manages all dependencies centrally")
    fmt.Println("✓ Services wired up once at application startup")
    fmt.Println("✓ Easy to swap implementations for testing")
}
```

## Practical Patterns for Dependency Injection in ERP Systems

### Pattern 1: Service Layer with Injected Repositories

A common pattern in ERP systems involves service layers that orchestrate business logic using injected repository dependencies.

```go
package service

import (
    "fmt"
    "time"
)

// Domain models
type Customer struct {
    ID        string
    Name      string
    Email     string
    TierLevel string
    CreatedAt time.Time
}

type Order struct {
    ID         string
    CustomerID string
    Items      []OrderItem
    Total      float64
    Status     string
    CreatedAt  time.Time
}

type OrderItem struct {
    ProductID string
    Quantity  int
    Price     float64

func (os *OrderService) PlaceOrder(customerID string, items []OrderItem) (*Order, error) {
    fmt.Printf("\n[Order Service] Processing order for customer %s\n", customerID)
    
    // Get customer
    customer, err := os.customerRepo.FindByID(customerID)
    if err != nil {
        return nil, fmt.Errorf("customer not found: %w", err)
    }
    
    // Validate and get product details
    for i, item := range items {
        product, err := os.productRepo.FindByID(item.ProductID)
        if err != nil {
            return nil, fmt.Errorf("product not found: %s", item.ProductID)
        }
        
        // Set price from product
        items[i].Price = product["price"].(float64)
        
        // Check stock
        stock := product["stock"].(int)
        if stock < item.Quantity {
            return nil, fmt.Errorf("insufficient stock for product %s", item.ProductID)
        }
    }
    
    // Calculate total
    total := os.pricing.CalculateOrderTotal(items, customer.TierLevel)
    total = os.pricing.ApplyDiscount(total, customer.TierLevel)
    
    // Create order
    order := &Order{
        ID:         fmt.Sprintf("ORD-%d", time.Now().Unix()),
        CustomerID: customerID,
        Items:      items,
        Total:      total,
        Status:     "pending",
        CreatedAt:  time.Now(),
    }
    
    // Save order
    if err := os.orderRepo.Save(order); err != nil {
        return nil, fmt.Errorf("failed to save order: %w", err)
    }
    
    // Update stock
    for _, item := range items {
        if err := os.productRepo.UpdateStock(item.ProductID, -item.Quantity); err != nil {
            fmt.Printf("[Order Service] Warning: Failed to update stock for %s\n", item.ProductID)
        }
    }
    
    // Send confirmation
    if err := os.notification.SendOrderConfirmation(customer.Email, order.ID, total); err != nil {
        fmt.Printf("[Order Service] Warning: Failed to send confirmation: %v\n", err)
    }
    
    order.Status = "confirmed"
    os.orderRepo.Save(order)
    
    fmt.Printf("[Order Service] Order %s placed successfully: $%.2f\n", order.ID, total)
    
    return order, nil
}

func (os *OrderService) GetCustomerOrders(customerID string) ([]*Order, error) {
    return os.orderRepo.FindByCustomer(customerID)
}
```

## Best Practices for Dependency Injection in Go

### Practice 1: Define Minimal Interfaces

Keep interfaces small and focused. Define interfaces in terms of what the consumer needs, not what the provider offers.

```go
package interfaces

// ❌ BAD: Large interface with many methods
type DataStore interface {
    Connect() error
    Disconnect() error
    Query(sql string) ([]map[string]interface{}, error)
    Execute(sql string) error
    BeginTransaction() error
    CommitTransaction() error
    RollbackTransaction() error
    GetConnectionCount() int
    SetMaxConnections(max int)
    GetStatistics() map[string]interface{}
}

// ✅ GOOD: Small, focused interfaces
type OrderReader interface {
    GetOrder(id string) (*Order, error)
}

type OrderWriter interface {
    SaveOrder(order *Order) error
}

// Consumers only depend on what they need
type OrderReportGenerator struct {
    reader OrderReader // Only needs reading
}

type OrderProcessor struct {
    writer OrderWriter // Only needs writing
}
```

### Practice 2: Avoid Service Locator Anti-Pattern

Don't hide dependencies behind a global service locator. Make dependencies explicit through constructor injection.

```go
package antipattern

import "fmt"

// ❌ BAD: Service Locator hides dependencies
var globalRegistry = make(map[string]interface{})

func Register(name string, service interface{}) {
    globalRegistry[name] = service
}

func Get(name string) interface{} {
    return globalRegistry[name]
}

type BadService struct {
    // Dependencies are hidden!
}

func (bs *BadService) DoWork() {
    // Hidden dependency on global registry
    db := Get("database")
    logger := Get("logger")
    
    fmt.Printf("Using hidden dependencies: %v, %v\n", db, logger)
}

// ✅ GOOD: Explicit constructor injection
type GoodService struct {
    db     Database
    logger Logger
}

func NewGoodService(db Database, logger Logger) *GoodService {
    return &GoodService{
        db:     db,
        logger: logger,
    }
}

func (gs *GoodService) DoWork() {
    // Dependencies are clear and explicit
    gs.logger.Log("Working with database")
    gs.db.Query("SELECT *")
}
```

### Practice 3: Use Constructor Functions

Always use constructor functions (New* pattern) to create instances with injected dependencies.

```go
package constructors

// ✅ GOOD: Clear constructor with all dependencies
type InvoiceService struct {
    repository InvoiceRepository
    printer    InvoicePrinter
    sender     InvoiceSender
}

func NewInvoiceService(
    repository InvoiceRepository,
    printer InvoicePrinter,
    sender InvoiceSender,
) *InvoiceService {
    return &InvoiceService{
        repository: repository,
        printer:    printer,
        sender:     sender,
    }
}

// For optional dependencies, use functional options pattern
type InvoiceServiceConfig struct {
    RetryAttempts int
    Timeout       int
}

type InvoiceServiceOption func(*InvoiceServiceConfig)

func WithRetryAttempts(attempts int) InvoiceServiceOption {
    return func(c *InvoiceServiceConfig) {
        c.RetryAttempts = attempts
    }
}

func NewInvoiceServiceWithOptions(
    repository InvoiceRepository,
    printer InvoicePrinter,
    sender InvoiceSender,
    opts ...InvoiceServiceOption,
) *InvoiceService {
    config := &InvoiceServiceConfig{
        RetryAttempts: 3,
        Timeout:       30,
    }
    
    for _, opt := range opts {
        opt(config)
    }
    
    return &InvoiceService{
        repository: repository,
        printer:    printer,
        sender:     sender,
    }
}
```

## Conclusion: The Transformative Power of Dependency Injection

Dependency Injection is not merely a technical pattern—it is a fundamental shift in how you think about component relationships. By inverting control and injecting dependencies rather than creating them, you transform rigid, tightly-coupled code into flexible, testable, maintainable systems.

In ERP applications where business requirements constantly evolve, where testing is critical for correctness, and where components must be reusable across different contexts, Dependency Injection becomes indispensable. It allows you to swap database implementations for testing, replace email services without touching business logic, and evolve your architecture without massive refactoring.

Go's interface-based type system makes Dependency Injection natural and type-safe. By defining clear contracts through interfaces and injecting dependencies through constructors, you build systems that are both simple to understand and powerful in their flexibility.

As you build your Go-based ERP system, make Dependency Injection a core architectural principle. Your tests will be easier to write, your code will be easier to change, and your system will be more resilient to the inevitable evolution of business requirements.
