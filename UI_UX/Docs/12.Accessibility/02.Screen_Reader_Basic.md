## Screen Reader Basics

### How It Works — The Invisible Layer of the Web

A screen reader is a piece of assistive software that reads the contents of a screen aloud, allowing people who are blind or have severe visual impairments to navigate and use digital products through audio output and, often, a braille display. Popular screen readers include **NVDA** and **JAWS** on Windows, **VoiceOver** on macOS and iOS, and **TalkBack** on Android. When a screen reader encounters a webpage or mobile application, it does not see the visual design. It reads an underlying structure — a tree of elements, attributes, roles, and labels — and converts this structure into spoken language.

This underlying structure is governed by two interconnected concepts: the **Document Object Model (DOM)** and **ARIA (Accessible Rich Internet Applications)**. The DOM is the hierarchical representation of all elements in a web page. When a screen reader traverses the DOM, it listens for semantic signals — tags and attributes that carry meaning beyond mere appearance. A `<button>` element, for instance, tells the screen reader not just that there is something clickable, but that it is a button, and it communicates this role aloud. A `<nav>` element signals a navigation region. A `<h1>` tag signals the primary heading. A `<p>` tag signals a paragraph of text.

The problem arises when designers and developers build interfaces using non-semantic HTML — using generic `<div>` and `<span>` elements for everything, styled to look like buttons and checkboxes and menus, but carrying no semantic meaning whatsoever. To a sighted user, a `<div>` styled as a button looks and functions just like a button. To a screen reader, it is simply a container — anonymous, roleless, and meaningless. The screen reader cannot tell the user what it is, cannot automatically indicate that it is interactive, and cannot handle keyboard interaction unless the developer has manually recreated that behavior.

### How It Is Applied in UI/UX Design

From a UI/UX design perspective, screen reader accessibility begins at the design stage — long before a developer writes a single line of code. Designers must think in terms of what they call the **accessibility tree**: the logical, linear flow of information as it would be read aloud by a screen reader, traversed from top to bottom.

The first practice is the thoughtful use of **alt text for images**. Every image in a UI that carries meaning must have a text description. An image of a user's profile photo might have the alt text "Profile photo of Maria Gonzalez." A decorative divider line, which carries no meaning, should have an empty alt attribute (`alt=""`) to signal to the screen reader that it can be safely skipped. A complex chart or infographic requires a substantial description — either in the alt attribute or in adjacent body text — that conveys the same information the visual represents.

The second practice involves **focus management and keyboard navigation**. Screen reader users typically navigate by pressing the Tab key to move between interactive elements, and Enter or Space to activate them. A well-designed interface ensures that every interactive element is reachable by keyboard, that the focus order is logical and follows the visual layout, and that the currently focused element is visually indicated through a clearly visible **focus ring**. Designers who suppress the default browser focus ring without providing a custom alternative are inadvertently making their product unusable for keyboard-only and screen-reader users.

The third and perhaps most conceptually rich practice is the application of **ARIA labels and live regions**. ARIA (Accessible Rich Internet Applications) is a set of attributes that can be added to HTML elements to enrich their semantic meaning when native HTML tags are insufficient. For example, a custom dropdown menu built with `<div>` elements can be made accessible by adding `role="combobox"`, `aria-expanded="true/false"`, and `aria-activedescendant` to communicate its state to the screen reader. A loading spinner that appears when content is fetching can be announced through an `aria-live="polite"` region, so the screen reader automatically reads "Loading content" without the user having to navigate to it.

In product design practice, UX designers often create a dedicated **accessibility annotation** layer in their Figma or Sketch files — a set of notes that communicate to the development team exactly what each element's role, label, and state should be. This includes specifying the reading order for complex layouts, identifying which decorative images should be hidden from screen readers, and noting where ARIA live regions should be implemented for dynamic content updates.
