# Mobile Component Design — A Complete Guide

> _How every core UI component should be designed for mobile apps — with anatomy, states, measurements, behavior, and real examples._

---

## Introduction — Components Are the Vocabulary of an Interface

If a design system is a language, then components are its words. Every interaction a user has with a mobile app happens through a component — they tap a button, fill in an input field, read content in a card, navigate via a tab bar. These are not decorative elements. They are functional units that carry meaning, communicate affordance (what can be done), provide feedback (what happened), and establish the emotional character of the entire product.

Understanding how to design each component correctly — the right proportions, the right states, the right visual hierarchy, the right behavior — is one of the most important practical skills in mobile UI design. A beautifully conceived product concept, implemented with poorly designed components, will feel broken and unprofessional. A modest concept, executed with perfectly crafted components, will feel polished and trustworthy.

This guide covers every foundational mobile component: its purpose, its anatomy, its measurements, its states, its variants, its behavioral specifications, and the common mistakes designers make with it. Every decision described here is grounded in platform guidelines (Apple HIG, Material Design), accessibility standards (WCAG), and years of accumulated best practice across the industry.

## 1. Buttons — Primary, Secondary, Destructive

### What a Button Is and What It Does

A button is a tappable element that triggers a discrete action — it does something immediately when pressed. This is the most important thing to understand about buttons: they are action triggers, not navigation links (though the visual boundary sometimes blurs). "Save," "Submit," "Delete," "Add to Cart," "Sign In" are buttons. "View Profile," "See all results," "Learn more" are often links, though they may be styled as buttons.

The fundamental design challenge of a button is communicating **affordance** — making it immediately obvious that this element is tappable and that tapping it will do something. This requires the right combination of shape, color, size, label, and feedback state.

### The Button Hierarchy — Why Three Types Exist

Every app needs a clear hierarchy of actions. Not all actions are equal — some are the primary goal of a screen (save a record, complete a purchase), some are secondary alternatives (cancel, go back), and some are potentially harmful or irreversible (delete, remove, unsubscribe). The three-tier button system — primary, secondary, destructive — maps visual prominence directly to action importance. The most visually prominent button is the primary action. The least visually prominent button that is still clearly interactive is the secondary action. The visually distinct but controlled button is the destructive action.

This hierarchy does more than organize aesthetics. It protects users. When the most prominent button is always the safest, most expected action, accidental taps on destructive actions are dramatically reduced. The visual hierarchy itself is a safety mechanism.

### Primary Button — The Main CTA

The primary button is the most visually prominent interactive element on a screen. It represents the main action the designer intends the user to take. There should typically be only one primary button per screen (or per logical section), because having two equally prominent primary buttons creates confusion about which action is more important.

Anatomically, a primary button consists of a solid filled background in the brand's primary color, a label in a contrasting color (almost always white on a colored background), optional leading or trailing icons, and corner rounding. The filled solid background is what makes it "primary" — it has maximum visual weight among the button types.

The height of a mobile primary button should be 48px minimum, with 52–56px being the most comfortable size for thumb interaction. This height, combined with horizontal padding of 24px minimum, creates a touch target that is easy to tap accurately even in motion. The label font size should be 15–17px at weight 600 (SemiBold) — substantial enough to be clearly readable but not so heavy that it overwhelms the button. Letter spacing should be 0 to slightly positive.

### Secondary Button — The Alternative Action

The secondary button represents an alternative or less important action. On a form, it might be "Cancel" alongside the primary "Save." On a modal, it might be "Maybe Later" alongside the primary "Upgrade Now." Its visual design must clearly communicate that it is interactive (tappable) but subordinate to the primary button.

Secondary buttons are typically designed with an **outline style** (transparent background, colored border and label) or a **ghost style** (no border, just a colored label). The outline style is more common and more clearly communicates interactivity. The ghost style works when space is very tight or when the context makes the interactivity obvious.

The secondary button must maintain the same height and minimum touch target as the primary button. Many designers make secondary buttons too small — reducing their height or padding to visually distinguish them from the primary. This is wrong. The visual subordination should come from reduced visual weight (outline instead of fill), not from reduced size.

### Destructive Button — The Danger Zone

The destructive button is used for actions that are irreversible or potentially harmful: "Delete Account," "Remove Card," "Cancel Subscription," "Clear All Data." It must communicate clearly that this action carries risk, while still being clearly interactive.

The most common approach is red — specifically a controlled, accessible red (not neon red, which feels aggressive; a deep, readable red like #DC2626 or #B91C1C). However, a destructive button should almost never be the primary visual element on a screen. It should be secondary in visual weight, often styled as an outline button in red, or a tonal button with a light red background. The fully filled, solid red primary button should only appear when the entire screen is explicitly dedicated to a destructive confirmation — for example, a "Confirm Deletion" screen that appears after the user has already tapped a destructive action.

The most important design pattern for destructive actions is the **two-step confirmation**: the user taps a destructive action (which may be styled as a secondary/outline red button), which triggers a confirmation modal or sheet where a solid red primary button appears to confirm the action. This two-step pattern is standard practice precisely because it prevents accidental permanent actions.

## 2. Cards — The Container of Content

### What a Card Is

A card is a bounded, elevated surface that groups related pieces of content and actions into a single cohesive unit. Cards are the most universally used content pattern in mobile design — they appear in feeds, dashboards, lists, product grids, profile sections, and nearly everywhere content needs to be organized into scannable, tappable units.

The defining visual characteristic of a card is its **surface elevation** — it appears to float above the background, creating a sense that it is a distinct, discrete object. This elevation communicates that the card's contents belong together and that the card itself may be interactive as a unit. Elevation can be expressed through shadow, through a contrasting background color, or through a border.

### Card Anatomy

A complete card is assembled from several optional layers, each serving a specific function. Understanding which layers to include, and how to proportion them, is the core challenge of card design.

The **container** is the card's bounding shape — its border-radius, background color, shadow, and dimensions. Border-radius on mobile cards is typically 12–16px, which is large enough to feel clearly rounded and modern but not so large that it becomes pill-shaped. The background is usually white (#FFFFFF) on a light background or a slightly lighter dark surface on dark backgrounds. Shadow should be subtle — a crisp, small shadow creates the impression of floating without appearing overdone.

The **media area** is an optional image or video at the top of the card, spanning the full card width. The most important design decision for media is its **aspect ratio** — maintaining a consistent aspect ratio across all cards in a grid ensures perfect alignment. The most common mobile card media ratios are 16:9 (for video thumbnails), 3:2 (for article images), and 1:1 (for product or profile images).

The **content area** holds the title, supporting text, metadata, and any tags or badges. The content area has internal padding of 16px on all sides. The title should be at most two lines with a text overflow ellipsis on the third line — cards should never have unpredictable heights caused by long titles. Supporting text is limited to two lines maximum.

The **action area** is optional and contains buttons, links, or icon buttons that apply to the entire card. In many modern mobile designs, the entire card is tappable as a single interaction target, making a separate action area unnecessary.

## 3. Input Fields — Where Users Communicate with the App

### The Most Critical Component

Input fields are where users give information to the app — typing their name, password, address, search query, or message. They are the most interaction-intensive component in mobile UI. Every millisecond of friction in an input field costs conversions, registrations, or user satisfaction. Poor input field design is one of the leading causes of form abandonment on mobile.

The challenges of input fields on mobile are amplified compared to desktop: the software keyboard takes up approximately half the screen when active, the touch target for text entry is the entire field (not just a cursor), finger-typing is slower and more error-prone than keyboard typing, and autocorrect introduces its own unpredictable behaviors.

### Input Field Anatomy

An input field consists of a **label**, a **container**, **helper text**, **placeholder text**, an optional **leading element** (icon or prefix text), and an optional **trailing element** (clear button, show/hide password toggle, unit label). Each of these elements must be carefully designed for clarity and usability.

The **label** is the field's permanent identifier. It must always be visible, not hidden inside the field as placeholder text. The most common mistake in mobile form design is using placeholder text as labels — when the user starts typing, the placeholder disappears, and they can no longer see what the field was for. Labels should sit above the field, outside the container, at a consistent size (13–14px, weight 500 or 600) that clearly distinguishes them from the input text.

The **container** must have a minimum height of 48px (matching the minimum touch target recommendation). The background is typically white on light interfaces, with a subtle border (1–1.5px) in a neutral color for the default state. The border color changes for each interactive state, creating clear visual feedback about the field's current condition.

## 4. Bottom Navigation — The Primary Wayfinding System

### What Bottom Navigation Is and Why It Exists on Mobile

Bottom navigation is a persistent navigation component fixed to the bottom of the mobile screen, providing access to 3–5 top-level destinations in the app. It is the primary navigation pattern for mobile apps (as opposed to desktop, which typically uses top navigation or side navigation).

The reason for placing navigation at the bottom is purely ergonomic: the bottom of the screen is in the natural thumb reachability zone when holding a phone in one hand. Top navigation bars are difficult to reach with a thumb on modern large phones. Apple's HIG, Google's Material Design, and the overwhelming majority of successful mobile apps use bottom navigation as the primary wayfinding system.

### Bottom Navigation Design Rules

Bottom navigation should have exactly 3–5 items. Fewer than 3 items does not justify a persistent bottom bar (use something simpler). More than 5 items creates confusion and makes each item too small to be clearly identifiable.

Each item consists of an icon and a label. Both are required — icon-only navigation forces users to memorize what each icon means, creating cognitive overhead. Label-only navigation lacks the quick-scan visual differentiation that icons provide. Icon + label together creates the fastest, most reliable navigation system.

The active (currently selected) item should be visually differentiated from inactive items through both color and weight changes. The active icon uses the brand primary color. Inactive icons use a neutral gray. The label size should not change between active and inactive states — changing the label size causes the entire bottom bar to shift, which is visually distracting.

## 5. Tab Bars — Secondary In-Screen Navigation

### Tab Bars vs. Bottom Navigation

Tab bars are frequently confused with bottom navigation, but they serve different purposes. Bottom navigation is for switching between the top-level destinations of an app. Tab bars are for switching between views or categories within a single screen or section. Tab bars typically appear below a header (stuck to the top of the scrollable content area) or occasionally within a modal or bottom sheet.

For example: a banking app might use bottom navigation with tabs for "Home," "Accounts," "Transfers," "Cards," and "Profile." Within the "Accounts" screen, a tab bar might appear with tabs for "Checking," "Savings," and "Investments." The bottom navigation takes you between the main sections of the app; the tab bar filters or changes the view within one section.

## 6. Modals and Bottom Sheets — Focused Interruptions

### The Difference Between Modals and Sheets

A modal is a dialog that appears overlaid on top of the current screen, dimming the background and requiring the user to interact with it before they can return to the main content. A bottom sheet is a panel that slides up from the bottom edge of the screen, similarly overlaying the content but with a distinctly mobile interaction model (drag to dismiss, swipe up to expand).

Both modals and bottom sheets serve the same conceptual purpose: presenting focused, temporary content or actions that require attention without navigating away from the current screen. The choice between them is mostly about context: modals work better for short, symmetrical confirmations (yes/no, accept/decline). Bottom sheets work better for presenting lists of options, detailed content, forms, or any content that might need more vertical space or that benefits from the natural bottom-of-screen gesture model.

## 7. Lists — The Workhorse of Mobile Content

### What Lists Are and Their Universal Importance

Lists are the most fundamental content pattern in mobile design. A list is a vertically scrolling sequence of items that share the same structure and purpose. Contact lists, message threads, notification feeds, settings menus, search results, transaction histories — all of these are lists. Understanding list design means understanding mobile UI.

The power of lists comes from their predictability. When users understand the structure of one list item, they understand all list items. Scanning a list becomes effortless because the eye knows exactly where to look on each row. This is why consistency within a list is more important than creativity — the repetition is the feature.

## 8. Empty States — When Nothing Is There

### Why Empty States Are Not Afterthoughts

An empty state is what the user sees when a list, feed, or section contains no content. New users see empty states constantly — an empty inbox, an empty favorites list, an empty dashboard. Users who have cleared their content also see empty states — an empty cart, no search results, no notifications.

Empty states are one of the most neglected areas of mobile design and one of the most important opportunities. A poorly designed empty state communicates: "There is nothing here. Good luck." A well-designed empty state communicates: "Nothing is here yet, and here is exactly what to do about it."

The empty state should have three elements: an **illustration or icon** that is pleasant and contextually relevant (not generic), a **headline** that clearly states the situation, and a **supporting message** plus a **CTA** (call-to-action) that tells the user exactly what to do next. The empty state is the onboarding moment — the very first screen many users encounter in any feature. It sets the tone for the entire feature and determines whether users take the first action.

## 9. Error States — Communicating Problems Clearly

### Error States Are Trust Moments

When something goes wrong in a mobile app — a form submission fails, a network request times out, a payment is declined — the error state is the app's response. How the app communicates errors at this moment is a direct reflection of the app's quality, empathy, and trustworthiness. Vague, technical, or accusatory error messages destroy user trust instantly. Clear, specific, helpful error messages can actually _increase_ trust because they demonstrate that the app understands what went wrong and cares about helping the user fix it.

The golden rule of error messages: **never blame the user, never be vague, always tell them what to do next**. "An error occurred" is never acceptable. "Invalid input" is never acceptable. "Something went wrong" is never acceptable — unless the error is genuinely unknown, in which case even then a specific recovery action ("Try again") must be provided.

### The Three Levels of Error States

Error states exist at three levels of severity, each requiring a different visual treatment. Understanding these levels is critical for error design.

The **inline validation error** is the smallest and most focused — it applies to a single input field. "This email address is already in use." "Password must be at least 8 characters." These appear directly below the specific field that has the error, in small red text (12–13px) accompanied by a red border on the field. They should appear immediately when the field loses focus (on blur), not only when the form is submitted.

The **page-level banner error** is a strip that appears at the top of a page or form when the entire submission fails — for example, when a server returns a validation error. "Please fix the 2 errors below to continue." This banner draws attention to the fact that something went wrong before the user scrolls down to find the specific field errors.

The **full-screen error state** is used when an entire screen or feature cannot load or function — a network error, a 404, a permissions denial. This full-screen treatment is similar in structure to an empty state: illustration, headline, supporting message, recovery CTA.

## The Principle That Unifies All Components

Every component in this guide, despite its unique function and visual design, is governed by one overarching principle: **every component must communicate its state, its affordance, and its purpose — without any instruction from the user and without any explanation from the designer**.

A user should never need to wonder whether a button is tappable. They should never be unsure whether a form field has an error. They should never feel lost when a list is empty. They should never be confused about what a destructive action will do. Each component, designed correctly, answers all of these questions through its visual language alone — through shape, color, size, hierarchy, and motion working in concert.

This is not achieved by adding more design elements to a component. It is achieved by removing everything that does not serve these communicative purposes. The greatest compliment a component can receive is that users use it effortlessly and correctly the very first time they encounter it — without ever thinking about the design at all.
