  <section class="reveal">
    <div class="section-label">§ 01 — Foundation</div>
    <h2>What Is a Funnel, and Why Does the Shape Matter?</h2>

    <p>In the physical world, a funnel is a cone-shaped tool that channels liquid from a wide opening into a narrow spout. In product design, a funnel is a metaphor for a sequence of steps that users must pass through to accomplish a goal — signing up for a service, completing a purchase, onboarding into an application, or submitting a form. The metaphor is apt because, just like a physical funnel, the flow begins wide at the top and narrows at the bottom.</p>

    <p>At the very top of a funnel sits the largest possible audience — all the users who begin the journey. As users progress through each successive step, some of them exit. They get confused, distracted, frustrated, or simply lose interest. By the time the flow reaches the final step, only a fraction of the original audience remains. This universal narrowing is why the shape — a funnel — is so apt.</p>

    <div class="highlight">The critical insight of funnel analysis is not that users drop off — that is inevitable and expected. The insight is <strong>which step loses the most users</strong>, and <strong>why</strong>. That is where the design opportunity lives.</div>

    <p>Consider a simple real-world example: an e-commerce website. A user arrives on the homepage, browses products, adds something to their cart, begins checkout, enters their payment details, and finally receives an order confirmation. Each of these is a step in the purchase funnel. If 10,000 users start this journey but only 800 complete it, the overall conversion rate is 8%. But that headline number tells you almost nothing actionable. The funnel view, however, tells you that 4,000 users dropped off specifically at the payment step — and that is where you should focus your entire attention.</p>
  </section>

    <h3>The Three Numbers You Must Always Track</h3>
    <p>For every step in a funnel, three numbers tell the complete story.</p>

    <div class="math-grid">
      <div class="math-cell">
        <div class="math-cell-label">Step Volume</div>
        <div class="math-cell-formula">count(users at step N)</div>
        <div class="math-cell-example">Tells you the raw scale. A drop from 10,000 to 500 is more urgent than a drop from 50 to 25, even if both are 95%.</div>
      </div>
      <div class="math-cell">
        <div class="math-cell-label">Step-to-Step Rate</div>
        <div class="math-cell-formula">step_N / step_(N-1) × 100</div>
        <div class="math-cell-example">Tells you the efficiency of each individual step in isolation. This is your diagnostic signal per step.</div>
      </div>
      <div class="math-cell">
        <div class="math-cell-label">Overall Conv. Rate</div>
        <div class="math-cell-formula">final_step / entry_step × 100</div>
        <div class="math-cell-example">Tells you the total health of the entire flow. This is your North Star metric for the funnel.</div>
      </div>

    <div class="section-label">§ 05 — Diagnosis</div>
    <h2>Reading the Leaks — Why Users Drop Off at Each Step</h2>

    <p>Identifying that a leak exists is only the first half of the work. The second, harder half is diagnosing why users are leaving at a specific step. A high drop-off rate at Step 4 could be caused by a slow-loading page, a confusing form, a missing trust signal, an unexpected price reveal, a broken button, or a dozen other causes. Funnel analysis locates the problem; other tools — heatmaps, session recordings, user interviews — explain it.</p>

    <p>There is, however, a useful diagnostic framework that experienced product designers apply. Each step in a funnel serves a particular cognitive function for the user. When a step has an abnormally high drop-off, the cause is usually one of four categories of failure: a <strong>clarity failure</strong> (the user does not understand what to do or what they will get), a <strong>trust failure</strong> (the user is uncertain whether it is safe to proceed), a <strong>friction failure</strong> (the user encounters too many obstacles to complete the step), or a <strong>value failure</strong> (the user does not feel that proceeding is worth the effort).</p>

    <div class="section-label">§ 06 — Engineering Perspective</div>
    <h2>How Funnels Are Built in the Backend</h2>

    <p>As a backend developer, you are likely more interested in the technical side: how is funnel data actually structured, stored, and queried? The answer lies in event tables and window functions. Every user action — page view, button click, form submit — is stored as a row in an events table. The funnel query then counts, per user, whether they completed each step in the correct sequence within a given time window.</p>

    <p>The most fundamental query pattern for funnel analysis uses <strong>conditional counting</strong> — counting users who have a given event AND also had the preceding events before it. This is typically implemented with window functions or self-joins in SQL, or with pipeline operators in MongoDB's aggregation framework.</p>

    <div class="code-block">
<span class="cmt">-- events table structure (PostgreSQL)</span>
<span class="kw">CREATE TABLE</span> events (
  id          <span class="fn">BIGSERIAL</span> <span class="kw">PRIMARY KEY</span>,
  user_id     <span class="fn">VARCHAR</span>(<span class="num">36</span>)  <span class="kw">NOT NULL</span>,
  event_name  <span class="fn">VARCHAR</span>(<span class="num">100</span>) <span class="kw">NOT NULL</span>,
  occurred_at <span class="fn">TIMESTAMPTZ</span> <span class="kw">NOT NULL</span>,
  properties  <span class="fn">JSONB</span>
);

<span class="cmt">-- Create index for funnel queries (critical for performance)</span>
<span class="kw">CREATE INDEX</span> idx_events_user_event
  <span class="kw">ON</span> events (user_id, event_name, occurred_at);

<span class="cmt">-- ─────────────────────────────────────────────────────</span>
<span class="cmt">-- FUNNEL QUERY: E-commerce checkout (last 30 days)</span>
<span class="cmt">-- ─────────────────────────────────────────────────────</span>
<span class="kw">WITH</span> funnel_steps <span class="kw">AS</span> (
  <span class="kw">SELECT</span>
    user_id,
    <span class="fn">MAX</span>(<span class="kw">CASE WHEN</span> event_name = <span class="str">'product_viewed'</span>  <span class="kw">THEN</span> <span class="num">1</span> <span class="kw">ELSE</span> <span class="num">0</span> <span class="kw">END</span>) <span class="kw">AS</span> step1,
    <span class="fn">MAX</span>(<span class="kw">CASE WHEN</span> event_name = <span class="str">'add_to_cart'</span>     <span class="kw">THEN</span> <span class="num">1</span> <span class="kw">ELSE</span> <span class="num">0</span> <span class="kw">END</span>) <span class="kw">AS</span> step2,
    <span class="fn">MAX</span>(<span class="kw">CASE WHEN</span> event_name = <span class="str">'checkout_started'</span> <span class="kw">THEN</span> <span class="num">1</span> <span class="kw">ELSE</span> <span class="num">0</span> <span class="kw">END</span>) <span class="kw">AS</span> step3,
    <span class="fn">MAX</span>(<span class="kw">CASE WHEN</span> event_name = <span class="str">'payment_entered'</span>  <span class="kw">THEN</span> <span class="num">1</span> <span class="kw">ELSE</span> <span class="num">0</span> <span class="kw">END</span>) <span class="kw">AS</span> step4,
    <span class="fn">MAX</span>(<span class="kw">CASE WHEN</span> event_name = <span class="str">'order_completed'</span>  <span class="kw">THEN</span> <span class="num">1</span> <span class="kw">ELSE</span> <span class="num">0</span> <span class="kw">END</span>) <span class="kw">AS</span> step5
  <span class="kw">FROM</span> events
  <span class="kw">WHERE</span> occurred_at >= <span class="fn">NOW</span>() - <span class="str">'30 days'</span>::<span class="fn">INTERVAL</span>
  <span class="kw">GROUP BY</span> user_id
)
<span class="kw">SELECT</span>
  <span class="fn">COUNT</span>(*) <span class="kw">FILTER</span> (<span class="kw">WHERE</span> step1 = <span class="num">1</span>)                  <span class="kw">AS</span> s1_product_view,
  <span class="fn">COUNT</span>(*) <span class="kw">FILTER</span> (<span class="kw">WHERE</span> step1 = <span class="num">1</span> <span class="kw">AND</span> step2 = <span class="num">1</span>)     <span class="kw">AS</span> s2_add_to_cart,
  <span class="fn">COUNT</span>(*) <span class="kw">FILTER</span> (<span class="kw">WHERE</span> step2 = <span class="num">1</span> <span class="kw">AND</span> step3 = <span class="num">1</span>)     <span class="kw">AS</span> s3_checkout,
  <span class="fn">COUNT</span>(*) <span class="kw">FILTER</span> (<span class="kw">WHERE</span> step3 = <span class="num">1</span> <span class="kw">AND</span> step4 = <span class="num">1</span>)     <span class="kw">AS</span> s4_payment,
  <span class="fn">COUNT</span>(*) <span class="kw">FILTER</span> (<span class="kw">WHERE</span> step4 = <span class="num">1</span> <span class="kw">AND</span> step5 = <span class="num">1</span>)     <span class="kw">AS</span> s5_completed
<span class="kw">FROM</span> funnel_steps;
    </div>

    <p>In a Node.js / TypeScript microservice context, events are typically produced to a message queue (RabbitMQ or Kafka) by each service that owns the relevant action, and then consumed by an analytics ingestion service that writes to a data warehouse. Redis is often used as a write buffer to batch high-volume events before flushing them to PostgreSQL or a columnar store like ClickHouse, which is optimised for the kind of aggregate funnel queries shown above.</p>

    <div class="code-block">
<span class="cmt">// Node.js: Tracking an event (TypeScript)</span>
<span class="kw">interface</span> <span class="fn">AnalyticsEvent</span> {
  userId:    <span class="fn">string</span>;
  event:     <span class="fn">string</span>;
  timestamp: <span class="fn">Date</span>;
  props:     <span class="fn">Record</span>&lt;<span class="fn">string</span>, <span class="fn">unknown</span>&gt;;
}

<span class="kw">async function</span> <span class="fn">trackEvent</span>(event: <span class="fn">AnalyticsEvent</span>): <span class="kw">Promise</span>&lt;<span class="kw">void</span>&gt; {
  <span class="cmt">// 1. Push to Redis stream (non-blocking, fast)</span>
  <span class="kw">await</span> redis.<span class="fn">xadd</span>(
    <span class="str">'events:stream'</span>,
    <span class="str">'*'</span>,
    <span class="str">'payload'</span>, JSON.<span class="fn">stringify</span>(event)
  );

  <span class="cmt">// 2. A background worker reads the stream and</span>
  <span class="cmt">//    bulk-inserts into PostgreSQL every 500ms</span>
  <span class="cmt">//    This prevents hot-path latency for the user</span>
}

<span class="cmt">// Called in your checkout controller:</span>
<span class="kw">await</span> <span class="fn">trackEvent</span>({
  userId:    req.user.id,
  event:     <span class="str">'checkout_started'</span>,
  timestamp: <span class="kw">new</span> <span class="fn">Date</span>(),
  props: {
    cartValue:    cart.total,
    itemCount:    cart.items.<span class="fn">length</span>,
    device:       req.device.type,
    sessionId:    req.session.id
  }
});

  <section class="reveal" style="border-bottom: none;">
    <div class="section-label">§ 07 — Synthesis</div>
    <h2>The Designer–Engineer Collaboration</h2>

    <p>Funnel analysis sits at the precise intersection of engineering and design. Engineers build the instrumentation infrastructure that makes funnel data possible — the event tracking pipelines, the data schemas, the warehouse queries. Designers and product managers interpret this data and use it to make decisions about where to invest redesign effort.</p>

    <p>When both disciplines understand the language of funnels fluently — when an engineer knows what a "checkout funnel drop-off" means in terms of user experience, and when a designer understands that "firing an event" requires careful backend implementation — the product team as a whole becomes dramatically more effective. Data-driven design is not a tool used by one role; it is a shared language spoken across the entire product organisation.</p>

    <div class="highlight">The greatest products are not designed by the most talented designers. They are designed by teams who iterate the fastest — who find the leaks, fix them quickly, and measure the result. Funnel analysis is the instrument they use to navigate.</div>

function renderFunnel() {
  const scenario = SCENARIOS[currentScenario];
  const maxUsers = scenario.steps[0].users;
  const container = document.getElementById('funnelBars');
  container.innerHTML = '';

  scenario.steps.forEach((step, i) => {
    const pct = Math.round((step.users / maxUsers) * 100);
    const prevUsers = i > 0 ? scenario.steps[i-1].users : step.users;
    const dropPct = i > 0 ? Math.round(((prevUsers - step.users) / prevUsers) * 100) : 0;

    const div = document.createElement('div');
    div.className = 'funnel-step';
    div.onclick = () => selectStep(i);
    div.innerHTML = `
      <div class="funnel-step-label">
        <span class="funnel-step-name">${i+1}. ${step.name}</span>
        <span class="funnel-step-count">${step.users.toLocaleString()}</span>
      </div>
      <div class="funnel-bar-track">
        <div class="funnel-bar-fill" style="width:0%; background:${step.color};" data-pct="${pct}">
          <span class="funnel-bar-pct">${pct}%</span>
        </div>
        ${i > 0 ? `<div class="funnel-drop-indicator" style="left:${pct}%"><span class="drop-tag">-${dropPct}%</span></div>` : ''}
      </div>
    `;
    container.appendChild(div);

  // Animate bars in

}
