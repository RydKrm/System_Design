## Elevation & Shadows

### The Physics of Shadow — How Light Creates Depth

To design shadows well, one must first understand how shadows work in the physical world. A shadow is the absence of light. When an object is positioned between a light source and a surface, it blocks some of the light from reaching that surface, creating a region of relative darkness — a shadow — in the shape of the object. The characteristics of that shadow are determined by three variables: the distance between the object and the surface it casts its shadow upon, the distance between the object and the light source, and the character of the light source itself (whether it is a single hard point of light, or a diffuse ambient source spread across an area).

When an object is very close to a surface — almost resting upon it — its shadow is small, sharp, and dark. It hugs the object tightly because the object is nearly touching the surface and there is little room for the shadow to spread. As the object lifts away from the surface, the shadow grows larger, softer, and lighter. It spreads outward because the light now has more space to flow around the edges of the object before reaching the surface. The shadow is also more diffuse because the angular separation between the lit and unlit regions becomes smaller at greater distances.

This physical behavior is the foundation of every elevation system in modern UI design. When a card "floats" above the background of a page, its shadow should behave exactly as a physically elevated card would behave: it should be larger, softer, and more diffuse than the shadow of an element that rests flush against the surface. When a modal dialog appears above the page content, it should cast a shadow that reflects its significantly greater elevation — a broad, very soft shadow that communicates clearly: this object is far above the surface beneath it.

### Building an Elevation Scale in CSS

A professional elevation system is not a collection of ad hoc shadow values applied individually to each component. It is a structured scale — a set of named levels, each corresponding to a specific elevation in the conceptual space of the interface, each with a precisely defined shadow that reflects the physics of that elevation. Google's Material Design system defines five elevation levels; Atlassian's design system uses six; many custom design systems define four to eight, depending on the complexity of the product's layering requirements.

Each elevation level in a well-crafted system uses not one but typically two shadows simultaneously: an **ambient shadow** and a **key shadow**. The ambient shadow simulates the soft, diffuse fill light that surrounds an object from all directions in real-world lighting — it is large, very soft, and very low-opacity. The key shadow simulates the direct shadow cast by the primary light source — it is smaller, crisper, and slightly more opaque. Together, they produce a shadow that reads as physically convincing because it mimics the actual dual nature of light in the real world.

```
ELEVATION SYSTEM — CSS Implementation
══════════════════════════════════════════════════════════════════════

  DUAL-SHADOW TECHNIQUE:
  Key shadow   = direct light source shadow (crisper, smaller)
  Ambient shadow = environmental fill light (larger, softer, lower opacity)

  :root {
    /* Level 0 — Flat. Resting on surface. No elevation. */
    --shadow-0: none;

    /* Level 1 — Subtle lift. Cards, list items, chips. */
    --shadow-1:
      0px 1px 2px  rgba(0,0,0, 0.12),   /* key */
      0px 1px 3px  rgba(0,0,0, 0.08);   /* ambient */

    /* Level 2 — Clear elevation. Dropdowns, popovers. */
    --shadow-2:
      0px 2px 4px  rgba(0,0,0, 0.14),   /* key */
      0px 4px 8px  rgba(0,0,0, 0.08);   /* ambient */

    /* Level 3 — Medium float. Navigation drawers, sidebars. */
    --shadow-3:
      0px 4px 8px  rgba(0,0,0, 0.14),   /* key */
      0px 8px 16px rgba(0,0,0, 0.08);   /* ambient */

    /* Level 4 — High elevation. FABs, sticky headers. */
    --shadow-4:
      0px 6px 12px rgba(0,0,0, 0.16),   /* key */
      0px 12px 24px rgba(0,0,0, 0.08);  /* ambient */

    /* Level 5 — Maximum. Modals, dialogs, full-screen overlays. */
    --shadow-5:
      0px 8px 16px rgba(0,0,0, 0.18),   /* key */
      0px 24px 48px rgba(0,0,0, 0.10);  /* ambient */
  }

  ANIMATED ELEVATION (hover interaction):
  .card {
    box-shadow: var(--shadow-1);
    transform: translateY(0);
    transition:
      box-shadow 250ms cubic-bezier(0.4, 0, 0.2, 1),
      transform   250ms cubic-bezier(0.4, 0, 0.2, 1);
  }
  .card:hover {
    box-shadow: var(--shadow-3);
    transform: translateY(-2px);  /* physically lifts away from surface */
  }
  /* The card visually rises on hover — shadow and movement are synchronized */

  COLORED SHADOWS (brand-aware elevation):
  .primary-button {
    background: #5B4CF5;
    box-shadow:
      0px 4px 12px rgba(91, 76, 245, 0.40),  /* colored key shadow */
      0px 1px 3px  rgba(91, 76, 245, 0.20);  /* colored ambient */
  }
  /* Shadow tinted with button color — feels physically coherent */
```

### Shadow Anti-Patterns — What to Avoid

The most common mistake designers make with shadows is applying them without a coherent system — adding a dramatic shadow to a card on one screen, a flat design on another, a subtly different shadow on a third. The resulting product has no consistent spatial logic. The user's eye cannot establish which elements are elevated above others, because the shadows do not form a coherent vocabulary of depth.

A second common mistake is applying shadows that are too dark and too crisp — a harsh `0px 4px 8px rgba(0,0,0,0.5)` that reads as a dark outline rather than a real shadow. Real shadows, cast by real objects under real light, are almost never this dark or this hard at the scale of UI elements. They are subtle, diffuse, and gentle. The rule of thumb is that if you can immediately see your shadow, it is probably too dark. The shadow should be felt rather than seen.
