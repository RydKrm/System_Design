## Part 3: State Management in UI вҖ” The Life of a Component

### States Are Not Bugs; They Are Features

Every interactive component in a user interface lives through a series of **states** вҖ” conditions that reflect the relationship between the component and the user, or between the component and the system. States are not optional cosmetic additions. They are fundamental to usability and accessibility. Without clearly communicated states, users do not know if they clicked something, whether something is available, or why an action is not working.

The core interactive states that every design system must define are:

**The Default State** is the resting condition of a component. This is how the component looks when nothing is happening. It is the baseline from which all other states depart.

**The Hover State** communicates responsiveness. When a user moves their cursor over an interactive element, the element should signal that it is "alive" вҖ” that it will respond to a click. This is achieved through subtle visual feedback: a slight change in background color, a shadow deepening, an underline appearing on a link, a border brightening. The hover state tells the user "yes, I am clickable." This feedback loop is the foundation of what UX designers call _affordance_ вҖ” the visual cue that suggests how an object should be used.

**The Pressed (Active) State** occurs during the instant of a click вҖ” the moment the mouse button is held down before being released. This provides tactile-like feedback in a digital interface, mimicking the physical sensation of pressing a real button. The component typically appears to depress slightly: it gets darker, smaller (via a scale transform), or shifts by a pixel or two downward. This micro-interaction is brief but profoundly satisfying to users.

**The Focus State** is critically important for accessibility. When a user navigates an interface using only a keyboard (Tab key), the currently focused element must be visually distinct. Browsers apply a default blue outline for this purpose, and many designers override this with a more brand-consistent ring or glow effect. Removing focus indicators without providing an alternative is not just bad design вҖ” it is an accessibility failure that excludes users with motor disabilities.

**The Disabled State** communicates impossibility. When a component is disabled, it should not be invisible вҖ” users need to know the element exists but is currently unavailable. The convention is to reduce opacity (typically to 40вҖ“60%), remove interactive cursor behavior, and ensure that no hover or click events are registered. Critically, a disabled state should ideally be accompanied by a tooltip or nearby text explaining _why_ the action is unavailable.

**The Loading State** is a modern addition that has become essential in asynchronous interfaces. When a button triggers an API call, it should communicate that work is happening. A spinner, a pulsing animation, or text changing from "Submit" to "SubmittingвҖҰ" prevents users from clicking repeatedly in frustration.

Here is how these states are expressed in CSS, using the token system we already built:

```css
/* в”Җв”Җв”Җ Default в”Җв”Җв”Җ */
.btn-primary {
  background-color: var(--color-action-primary);   /* blue-500 */
  color:            white;
  padding:          var(--space-2) var(--space-4);
  border-radius:    4px;
  transition:       background 150ms ease, transform 100ms ease, box-shadow 150ms ease;
  cursor:           pointer;
}

/* в”Җв”Җв”Җ Hover: slightly darker, grows shadow в”Җв”Җв”Җ */
.btn-primary:hover {
  background-color: var(--color-action-hover);     /* blue-600 */
  box-shadow:       0 4px 12px rgba(59, 130, 246, 0.35);
}

/* в”Җв”Җв”Җ Pressed: depresses slightly в”Җв”Җв”Җ */
.btn-primary:active {
  background-color: var(--color-blue-700);
  transform:        scale(0.97) translateY(1px);
  box-shadow:       none;
}

/* в”Җв”Җв”Җ Focus: accessible keyboard ring в”Җв”Җв”Җ */
.btn-primary:focus-visible {
  outline:        3px solid var(--color-action-primary);
  outline-offset: 3px;
}

/* в”Җв”Җв”Җ Disabled: faded, inert в”Җв”Җв”Җ */
.btn-primary:disabled,
.btn-primary[aria-disabled="true"] {
  opacity:        0.45;
  cursor:         not-allowed;
  pointer-events: none;
}

/* в”Җв”Җв”Җ Loading: cursor communicates wait в”Җв”Җв”Җ */
.btn-primary[data-loading="true"] {
  cursor:         wait;
  pointer-events: none;
}
```

Every one of these states contributes to what UX designers call **feedback** вҖ” one of the most fundamental principles of human-computer interaction. Humans need to know that their actions have been received and processed. Without states, an interface is silent and unresponsive вҖ” and silence, in digital products, breeds anxiety and frustration.
