## Component Variants — Building with Intent

### The Philosophy of Variants

A component variant is a **deliberate, pre-defined variation of a component** that addresses a specific need or context. Think of a button. Buttons are not all equal. Some are for primary actions ("Save", "Submit", "Buy Now"). Some are for secondary actions ("Cancel", "Back"). Some are for dangerous actions ("Delete Account"). Some are small, sitting inside a table cell. Some are large, demanding attention at the center of a page.

Rather than creating entirely different components for each of these cases, or worse, allowing every developer to style their own version, a design system defines variants. These are the agreed-upon, intentional shapes a component can take.

### The Anatomy of a Variant System

Variants operate along multiple dimensions simultaneously. A button component, for example, might have:

**Intent Variants** — these reflect the semantic purpose of the action. A `primary` button signals the main action of a page. A `secondary` button offers an alternative. A `ghost` button is subtle, often used in toolbars. A `danger` button communicates destructive consequences and is typically styled in red to invoke caution.

**Size Variants** — these reflect the spatial context where the component appears. A `small` button fits inside a compact data table. A `medium` button is the default. A `large` button is used in prominent call-to-action sections.

**Shape Variants** — some design systems offer `rounded`, `square`, or `pill` shapes to accommodate brand preferences.

In modern component frameworks, variants are typically declared using a variant API. Here is how this looks in practice using a popular pattern:

typescript

```typescript
// Defining a Button component with variants (TypeScript + CSS-in-JS style)

type ButtonProps = {
  intent:  'primary' | 'secondary' | 'ghost' | 'danger';
  size:    'sm'      | 'md'        | 'lg';
  label:   string;
};

const variantStyles = {
  intent: {
    primary:   { background: 'var(--color-action-primary)', color: '#fff' },
    secondary: { background: 'transparent', border: '1px solid var(--color-action-primary)' },
    ghost:     { background: 'transparent', color: 'var(--color-text-muted)' },
    danger:    { background: 'var(--color-danger)', color: '#fff' },
  },
  size: {
    sm: { padding: '6px 12px', fontSize: '12px' },
    md: { padding: '8px 16px', fontSize: '14px' },
    lg: { padding: '12px 24px', fontSize: '16px' },
  }
};

// Usage: <Button intent="danger" size="lg" label="Delete Account" />
```

This is a critical insight for product design: by defining variants in advance and giving them **semantic names** (not visual names), you encode the _meaning_ of an action into the component system itself. You never say "make it red." You say "make it danger." The system knows that "danger" means red. This separation of meaning from appearance is what makes large-scale UI systems maintainable.
