# gRPC Request-Response Cycle — Layer by Layer

### What Each Layer Does, How It Does It, and How Data Transforms There

---

## A Note Before We Begin

The previous guide walked through this cycle as a single flowing story. This guide takes a different shape on purpose. Each of the eighteen stages is now treated as its own isolated layer, and for every single one we ask the exact same three questions in the exact same order: what is this layer's job, how does it actually perform that job mechanically, and what does the data look like the moment before versus the moment after passing through it. Think of this less as a story and more as eighteen separate inspection stops, where at each stop we open the box, look at what is inside, and watch it change shape before it moves to the next stop.

Throughout, we will follow one single piece of data on its journey: the Order Service asking the User Service for the user with ID `usr-4521`.

---

## Layer 1 — Client Calls a gRPC Method

**What this layer does.** This is the starting point of the entire cycle, and its only job is to let an application engineer express an intention — "I want the user with this ID" — without thinking about networking at all. This layer exists purely to preserve the illusion that a remote call is no different from a local function call.

**How it does it.** The engineer writes ordinary Go code, invoking a method on a client object that was generated automatically by the `protoc` compiler from the `.proto` file. Nothing about this line of code touches sockets, bytes, or connections; it is pure application logic sitting at the very top of the stack.

**How the data transforms here.** At this layer, the data exists in its most natural, native form: a plain in-memory Go struct, allocated on the stack, with named fields holding typed values. There is no encoding, no wrapping, nothing network-related yet.

```go
resp, err := userClient.GetUser(ctx, &pb.GetUserRequest{
    UserId: "usr-4521",
})
```

```
DATA AT THIS LAYER:
┌─────────────────────────────┐
│  GetUserRequest{            │
│    UserId: "usr-4521"       │
│  }                          │
│  (a native Go struct in RAM)│
└─────────────────────────────┘
```

## Layer 2 — gRPC Client Stub Converts Request

**What this layer does.** This layer's job is translation and packaging. The struct created in Layer 1 needs to be paired up with information about _which_ remote function is being called, because unlike a local function call, the receiving side has no compiler-enforced way of knowing which method you meant — that information has to travel alongside the data itself.

**How it does it.** The generated stub code — the `userClient` object — intercepts the call, reads the method signature it was generated from, and constructs an internal request descriptor containing both the fully-qualified method name (something like `/user.UserService/GetUser`) and a reference to the message object that needs to be serialized. This descriptor is then handed off to gRPC's internal transport machinery, which is the layer below.

**How the data transforms here.** The lone Go struct from Layer 1 does not change shape itself, but it stops being "just data" and becomes "data plus a destination address." It is now bundled together with routing metadata.

```
DATA AT THIS LAYER:
┌─────────────────────────────────────────┐
│  method: "/user.UserService/GetUser"    │
│  message: GetUserRequest{               │
│    UserId: "usr-4521"                   │
│  }                                      │
│  (struct + routing info, still in RAM)  │
└─────────────────────────────────────────┘
```

## Layer 3 — Protobuf Serialization

**What this layer does.** This layer's job is to shrink the in-memory struct down into the smallest, most efficient binary representation possible, so that it can be transmitted across a network where every byte costs time and bandwidth.

**How it does it.** The Protobuf runtime walks through the struct field by field, in the order they are populated. For each field, it writes a single **tag byte** that encodes both the field number (taken directly from the `.proto` definition) and the wire type (a small code indicating whether the value is a varint, a length-delimited string, a 32-bit number, and so on). Immediately after the tag, for length-delimited types like strings, it writes a **length byte**, followed by the **raw bytes** of the actual value. Critically, the field name `"user_id"` is never written anywhere in this output — only the numeric tag `1` is, because both client and server already agree, from the shared `.proto` file, that tag `1` always means `user_id` in this message.

**How the data transforms here.** The bundled struct-plus-metadata from Layer 2 becomes a flat sequence of raw bytes — no longer a structured object at all, just a string of binary digits with no field names anywhere inside it.

```go
data, _ := proto.Marshal(&pb.GetUserRequest{UserId: "usr-4521"})
```

```
BEFORE (Go struct):              AFTER (raw binary, hex):
GetUserRequest{                  0A 08 75 73 72 2D 34 35 32 31
  UserId: "usr-4521"             │  │  └──────┬──────┘
}                                │  │     "usr-4521" in ASCII
                                 │  └─ length = 8 bytes
                                 └─ tag = field 1, wire type 2

Total: 10 bytes (no field name "user_id" appears anywhere)
```

## Layer 4 — gRPC Creates HTTP/2 Frame

**What this layer does.** Raw Protobuf bytes cannot simply be dropped onto a network connection by themselves; the receiving side would have no way to know where one message ends and another begins, or which logical request these bytes even belong to. This layer's job is to wrap those bytes in a structured container that HTTP/2 understands.

**How it does it.** gRPC produces two distinct kinds of HTTP/2 frames for a single call. First, a **HEADERS frame** is created, carrying metadata such as the HTTP method, the request path (which encodes the RPC method name), and the content type identifying this as a gRPC-Protobuf payload. Second, a **DATA frame** is created, carrying the actual 10 bytes of Protobuf payload from Layer 3, prefixed internally by gRPC's own 5-byte framing header (1 byte indicating compression, 4 bytes indicating message length). Both frames are tagged with the same numeric **Stream ID**, which is how the two frames are known to belong to the same logical call.

**How the data transforms here.** The flat binary blob from Layer 3 gets wrapped inside a labeled envelope — it gains a Stream ID, a frame type, a length field, and flags, all sitting around the original bytes rather than altering them.

```
DATA AT THIS LAYER:

HEADERS Frame (Stream ID: 7)              DATA Frame (Stream ID: 7)
┌────────────────────────────────┐        ┌─────────────────────────────┐
│:method: POST                   │        │ Length: 10                  │
│:path: /user.UserService/GetUser│        │ Flags: END_STREAM           │
│content-type: application/grpc  │        │ Payload:                    │
└────────────────────────────────┘        │0A 08 75 73 72 2D 34 35 32 31│
                                          └─────────────────────────────┘
```

---

## Layer 5 — HTTP/2 Multiplexing

**What this layer does.** In a real microservice system, the Order Service is rarely talking to only one other service at a time — it might be calling the User Service and the Payment Service within the same few milliseconds. This layer's job is to let all of that traffic share a single underlying TCP connection rather than forcing a new, expensive connection to be opened for every single call.

**How it does it.** Because every frame from Layer 4 already carries a Stream ID, HTTP/2 is free to interleave frames belonging to completely different RPC calls on the same connection, sending a little bit of one call, then a little bit of another, back and forth, with no call ever required to wait for an unrelated call to finish first. The receiving side simply groups incoming frames by their Stream ID to reconstruct each logical request independently.

**How the data transforms here.** The frames themselves are not altered in content, but their position in time changes — they are no longer guaranteed to travel as a clean, isolated sequence; they may be physically interleaved on the wire with frames from unrelated calls.

```
DATA AT THIS LAYER (one shared TCP connection):

  [Stream 7: HEADERS] [Stream 9: HEADERS] [Stream 7: DATA] [Stream 9: DATA]
       (GetUser call)    (GetPayment call)   (GetUser call)  (GetPayment call)

  Both calls share the same pipe, distinguished only by Stream ID.
```

---

## Layer 6 — Data Goes Through TCP

**What this layer does.** HTTP/2 frames need to physically cross a network — possibly through switches, routers, and unreliable links — and arrive at the destination completely intact and in the correct order. This layer's job is to guarantee exactly that reliability.

**How it does it.** The operating system's TCP stack takes the byte stream handed to it by HTTP/2 and breaks it into **segments**, each one stamped with a sequence number indicating its position in the overall stream. As segments travel across the network, the receiving TCP stack acknowledges each one it receives; if an acknowledgment doesn't arrive in time, the sending side automatically retransmits the lost segment. This is the layer where "lost packets" get silently fixed before anything above it ever notices.

**How the data transforms here.** The HTTP/2 frame bytes are chopped into possibly multiple TCP segments, each one gaining a TCP header containing source port, destination port, sequence number, and acknowledgment number.

```
DATA AT THIS LAYER:

TCP Segment:
┌─────────────────────────────────────────┐
│ Src Port: 51022     Dst Port: 50051     │
│ Seq: 10293           Ack: 88210         │
│ Payload: [HTTP/2 frame bytes from above]│
└─────────────────────────────────────────┘

        travels across physical network or localhost loopback
```

---

## Layer 7 — Server Receives TCP Packets

**What this layer does.** On the receiving machine — the User Service — something has to catch these incoming segments off the network interface and begin the process of reconstructing the original byte stream.

**How it does it.** The kernel's network stack on the server receives each TCP segment, verifies its checksum to confirm no corruption occurred in transit, and buffers segments that arrive out of order until the missing pieces show up, so that the final byte stream handed upward is in perfectly correct sequence — exactly as it was before transmission, regardless of what order it physically arrived in.

**How the data transforms here.** Scattered, possibly out-of-order TCP segments are reassembled into one continuous, ordered byte stream, with the TCP headers stripped away, ready to be interpreted as HTTP/2 framing.

```
DATA AT THIS LAYER:

Incoming segments (possibly out of order):
  [Seq 10293] [Seq 10303] [Seq 10298]
        │           │           │
        └───────────┴───────────┘
                    ▼
   Reordered, continuous byte stream
   (TCP headers now stripped, just raw bytes)
```

---

## Layer 8 — HTTP/2 Frame Reassembled

**What this layer does.** The raw byte stream recovered from TCP needs to be reinterpreted back into the structured HEADERS and DATA frames that were originally created in Layer 4, so the server's gRPC implementation knows what method is being called and where the payload begins and ends.

**How it does it.** The server's HTTP/2 implementation reads the byte stream and parses out frame boundaries using the length field embedded at the start of each frame. It groups frames by Stream ID, recognizing that the HEADERS frame and the DATA frame both carrying Stream ID 7 belong to the same call, and reads the `:path` header to determine that the `GetUser` method specifically is being invoked.

**How the data transforms here.** The flat byte stream is parsed back into distinct, labeled HTTP/2 frame objects in memory — essentially the exact mirror image of what Layer 4 produced on the client side.

```
DATA AT THIS LAYER:

Reassembled:
  Stream ID: 7
  Method: /user.UserService/GetUser
  Payload (10 bytes): 0A 08 75 73 72 2D 34 35 32 31
```

---

## Layer 9 — Protobuf Deserialization

**What this layer does.** The 10 raw bytes recovered from the DATA frame are meaningless to application code as-is; this layer's job is to turn them back into a typed, structured object the handler function can actually work with.

**How it does it.** The Protobuf runtime reads the bytes from the beginning. It reads the first byte as a tag, decodes it into field number 1 and wire type 2 (length-delimited), reads the next byte as a length of 8, and then reads the following 8 bytes as the raw value, interpreting them as a UTF-8 string. Because the server's code was generated from the identical `.proto` file as the client's, it knows with complete certainty that field number 1 in a `GetUserRequest` message corresponds to `user_id` — this shared schema is what makes the decoding unambiguous.

**How the data transforms here.** The flat 10-byte binary blob is reconstructed into a typed, named Go struct — effectively the exact reverse of Layer 3.

```go
var req pb.GetUserRequest
proto.Unmarshal(payloadBytes, &req)
```

```
BEFORE (raw binary, hex):              AFTER (Go struct):
0A 08 75 73 72 2D 34 35 32 31    →    GetUserRequest{
                                          UserId: "usr-4521"
                                        }
```

## Layer 10 — Interceptors Execute

**What this layer does.** Before any business logic runs, the server typically needs to enforce cross-cutting rules that apply uniformly to every incoming call — verifying authentication, logging the request, enforcing rate limits, or starting a distributed tracing span. This layer's job is to apply those rules in one centralized place rather than scattering the same checks across every individual handler.

**How it does it.** gRPC organizes interceptors as a chain; each interceptor receives the deserialized request object along with a reference to "the next thing to call" and decides whether to pass control forward or short-circuit with an error. An authentication interceptor, for example, inspects metadata attached to the call, validates a token, and either calls `handler(ctx, req)` to continue the chain or returns an error immediately, in which case the real business logic is never even reached.

**How the data transforms here.** The request object itself is typically passed through unchanged in shape, but the surrounding execution context is enriched — values like an authenticated user identity or a trace span get attached to the `context.Context` that travels alongside the request.

```go
func AuthInterceptor(ctx context.Context, req interface{},
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {

    token := extractToken(ctx)
    if !isValid(token) {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }
    return handler(ctx, req) // pass onward, unchanged in shape
}
```

```
DATA AT THIS LAYER:
┌───────────────────────────────────────────┐
│  req:  GetUserRequest{UserId: "usr-4521"} │
│  ctx:  + authenticated_user: "svc-order"  │
│        + trace_id: "abc-123"              │
└───────────────────────────────────────────┘
```

## Layer 11 — Service Handler Invoked

**What this layer does.** This is the layer where the actual business purpose of the call finally gets fulfilled — everything before this has been transport and plumbing; this is where the engineer's real logic lives.

**How it does it.** Once every interceptor in the chain has approved the call, gRPC invokes the concrete Go function that was registered to handle `GetUser` requests, passing it the deserialized request object and the enriched context. This function is ordinary application code, completely unaware of Protobuf, HTTP/2, or TCP — from its point of view, it simply received a struct and must return one.

**How the data transforms here.** The request object is read by the handler to extract the values it needs to act on; no transport-level transformation happens here, but the request now triggers application logic, such as preparing a database query.

```go
func (s *userServer) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.UserResponse, error) {
    user, err := s.repo.FindByID(ctx, req.UserId) // req.UserId == "usr-4521"
    ...
}
```

```
DATA AT THIS LAYER:
  req.UserId  →  "usr-4521"  (read out of the struct, used as a query parameter)
```

## Layer 12 — Database Call

**What this layer does.** The handler needs to actually fetch the requested information from persistent storage, since the gRPC layer itself holds no data — it only transports requests and responses.

**How it does it.** The handler issues a query against PostgreSQL or MongoDB using the extracted `user_id` value, and the database engine looks up the matching row or document, returning it across the database driver's own wire protocol back into the Go process as a result set.

**How the data transforms here.** A string value (`"usr-4521"`) is transformed into a full database query, and the database's tabular or document-shaped result is mapped back into application-level Go values.

```sql
SELECT id, name, email FROM users WHERE id = 'usr-4521';
```

```
DATA AT THIS LAYER (row returned):
┌──────────┬───────────────┬──────────────────────┐
│ id       │ name          │ email                │
├──────────┼───────────────┼──────────────────────┤
│ usr-4521 │ Karim Ahmed   │ karim@example.com    │
└──────────┴───────────────┴──────────────────────┘
```

---

## Layer 13 — Response Object Created

**What this layer does.** The raw database row, in its database-shaped form, is not something that can be sent back over gRPC directly — it must be mapped into the exact response message type defined in the `.proto` contract.

**How it does it.** The handler reads the values out of the database row and constructs a new instance of the generated `UserResponse` struct, assigning each field explicitly. This is also the natural place where fields the client should never see — like an internally stored password hash — are simply left unset, since only what is explicitly assigned will ever be serialized and sent.

**How the data transforms here.** A database row (a tabular structure tied to SQL/Mongo's own type system) becomes a typed Protobuf message struct, tied to the `.proto` contract's type system instead.

```go
resp := &pb.UserResponse{
    UserId: user.ID,
    Name:   user.Name,
    Email:  user.Email,
}
```

```
DATA AT THIS LAYER:
UserResponse{
  UserId: "usr-4521",
  Name:   "Karim Ahmed",
  Email:  "karim@example.com",
}
```

---

## Layer 14 — Response Serialized

**What this layer does.** Exactly mirroring Layer 3, but now for the outgoing response — the structured object must be compressed down into compact binary before it can leave the server.

**How it does it.** The same Protobuf marshaling process runs again: each populated field in `UserResponse` is walked in order, and for each one, a tag byte (field number plus wire type) is written, followed by a length byte where applicable, followed by the raw value bytes. Just as before, no field names are written — only the numeric tags defined in the shared `.proto` schema.

**How the data transforms here.** The `UserResponse` struct becomes a flat sequence of raw binary bytes, ready for transport, with each field's data packed tightly with no padding or names.

```go
data, _ := proto.Marshal(resp)
```

```
BEFORE (Go struct):                 AFTER (raw binary, hex):
UserResponse{                       0A 08 75 73 72 2D 34 35 32 31
  UserId: "usr-4521",               12 0B 4B 61 72 69 6D 20 41 68 6D 65 64
  Name:   "Karim Ahmed",            1A 11 6B 61 72 69 6D 40 65 78 61 6D 70 ...
  Email:  "karim@example.com",
}
```

---

## Layer 15 — HTTP/2 Response Frame Created

**What this layer does.** Just as the original request needed to be wrapped for transport in Layer 4, the response binary now needs the same treatment before it can travel back across the network.

**How it does it.** The server's gRPC implementation wraps the serialized response bytes inside a new HTTP/2 DATA frame, reusing the exact same Stream ID (7) that the original request arrived on, so the client can correlate this incoming frame with the call it is still waiting on. A trailing HEADERS frame is also produced, carrying the gRPC status code that indicates whether the call succeeded or failed.

**How the data transforms here.** The flat response binary gains the same kind of framing envelope the request bytes received earlier — a Stream ID, length field, and flags wrapped around the payload.

```
DATA AT THIS LAYER:

DATA Frame (Stream ID: 7)              Trailing HEADERS (Stream ID: 7)
┌──────────────────────────┐           ┌──────────────────────────┐
│ Payload: [response bytes]│           │ grpc-status: 0 (OK)      │
│ Flags: END_STREAM        │           │ grpc-message: ""         │
└──────────────────────────┘           └──────────────────────────┘
```

---

## Layer 16 — Response Sent Through TCP

**What this layer does.** This layer's job is identical in nature to Layer 6, just running in the opposite direction — it must reliably carry the response frames back across the network to the original caller.

**How it does it.** The server's TCP stack breaks the HTTP/2 response frames into segments, stamps them with sequence numbers, and transmits them toward the Order Service, relying on the same acknowledgment-and-retransmission mechanism to guarantee nothing is lost along the way.

**How the data transforms here.** The response frame bytes are segmented and wrapped in TCP headers for the return trip, exactly mirroring what happened to the request in Layer 6.

```
DATA AT THIS LAYER:

TCP Segment (return trip):
┌─────────────────────────────────────────┐
│ Src Port: 50051     Dst Port: 51022     │
│ Seq: 88210           Ack: 10303         │
│ Payload: [HTTP/2 response frame bytes]  │
└─────────────────────────────────────────┘
```

---

## Layer 17 — Client Receives and Deserializes Response

**What this layer does.** This layer combines, on the client side, the same reassembly and decoding work that Layers 7 through 9 performed on the server side — but now for the incoming response rather than the original request.

**How it does it.** The Order Service's TCP stack reassembles the incoming segments into a continuous byte stream, the HTTP/2 layer parses that stream back into HEADERS and DATA frames grouped by Stream ID 7, and the Protobuf runtime then decodes the DATA frame's payload bytes back into a typed `UserResponse` struct, using the same tag-by-tag reading process described in Layer 9.

**How the data transforms here.** Raw response bytes, having traveled the entire round trip, are reconstructed step by step back into a structured, typed Go object — the mirror image of every encoding step that happened on the way out.

```go
var resp pb.UserResponse
proto.Unmarshal(responseBytes, &resp)
```

```
BEFORE (raw binary, hex):                    AFTER (Go struct):
0A 08 75 73 72 2D 34 35 32 31           →   UserResponse{
12 0B 4B 61 72 69 6D 20 41 68 6D 65 64        UserId: "usr-4521",
1A 11 6B 61 72 69 6D 40 65 78 61 6D 70 ...    Name:   "Karim Ahmed",
                                                Email:  "karim@example.com",
                                              }
```

## Layer 18 — Client Gets Final Object

**What this layer does.** This is the final destination of the entire cycle, the layer where all the machinery disappears again and the application engineer is simply handed the answer to the question they asked all the way back in Layer 1.

**How it does it.** The generated client stub returns the fully reconstructed `UserResponse` struct as the return value of the original method call, unblocking the line of code that has been waiting this whole time, and execution in the Order Service simply continues onward as if nothing more complicated than a function call had happened.

**How the data transforms here.** Nothing further transforms at this layer — the typed struct produced in Layer 17 is simply delivered into the variable the engineer declared, completing the round trip from struct, to bytes, to frames, to segments, and all the way back to struct again.

```go
resp, err := userClient.GetUser(ctx, &pb.GetUserRequest{UserId: "usr-4521"})

// resp now holds:
// &UserResponse{
//   UserId: "usr-4521",
//   Name:   "Karim Ahmed",
//   Email:  "karim@example.com",
// }
```

```
DATA AT THIS LAYER:
┌─────────────────────────────────────┐
│  UserResponse{                      │
│    UserId: "usr-4521",              │
│    Name:   "Karim Ahmed",           │
│    Email:  "karim@example.com",     │
│  }                                  │
│  (a native Go struct in RAM, again) │
└─────────────────────────────────────┘
```

## The Eighteen Layers, Side by Side

```
┌────┬──────────────────────────────┬────────────────────────────-───────┐
│ #  │ Layer                        │ Data Shape After This Layer        │
├────┼──────────────────────────────┼──────────────────────────-─────────┤
│ 1  │ Client Calls Method          │ Native Go struct                   │
│ 2  │ Stub Converts Request        │ Struct + method routing metadata   │
│ 3  │ Protobuf Serialization       │ Raw binary (tag-length-value)      │
│ 4  │ HTTP/2 Frame Created         │ Binary wrapped in HEADERS+DATA     │
│ 5  │ HTTP/2 Multiplexing          │ Frames interleaved on shared conn  │
│ 6  │ TCP Transmission             │ TCP segments w/ seq numbers        │
│ 7  │ Server Receives TCP          │ Reassembled, ordered byte stream   │
│ 8  │ HTTP/2 Frame Reassembled     │ Parsed HEADERS+DATA frame objects  │
│ 9  │ Protobuf Deserialization     │ Native Go struct (request)         │
│ 10 │ Interceptors Execute         │ Struct unchanged, context enriched │
│ 11 │ Handler Invoked              │ Struct read, business logic runs   │
│ 12 │ Database Call                │ SQL row / Mongo document           │
│ 13 │ Response Object Created      │ Native Go struct (response)        │
│ 14 │ Response Serialized          │ Raw binary (tag-length-value)      │
│ 15 │ HTTP/2 Response Frame Created│ Binary wrapped in DATA+HEADERS     │
│ 16 │ Response Sent via TCP        │ TCP segments w/ seq numbers        │
│ 17 │ Client Receives & Decodes    │ Native Go struct (response)        │
│ 18 │ Client Gets Final Object     │ Native Go struct, delivered to code│
└────┴──────────────────────────────┴───────────────────────────-────────┘
```

The pattern worth noticing across all eighteen stops is symmetry: every transformation applied on the way out has an exact mirror-image transformation undoing it on the way back. Serialization is undone by deserialization, framing is undone by frame reassembly, segmentation is undone by stream reassembly — and the only layer with no mirror at all is the database call in the middle, which is the one true point in the entire cycle where new information actually enters the system rather than simply changing shape.
