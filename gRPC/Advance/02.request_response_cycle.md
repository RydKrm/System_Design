# The Complete gRPC Request-Response Cycle

### Every Transformation, Layer by Layer — Explained with Data

---

## Part One — Why We Need `.proto` Files

Before a single byte travels across the network, two services must agree on something fundamental: what shape the data takes, and what functions can be called remotely. This agreement is the entire reason `.proto` files exist, and understanding why will make everything that follows make sense.

Picture two teams working on two different microservices — the Order Service team and the User Service team. They sit in different parts of the building, perhaps in different countries, and they almost certainly do not read each other's source code. If the User Service team decides one day to rename a field from `email` to `emailAddress`, and there is no shared contract enforcing the old name, the Order Service will keep sending requests expecting `email`, and something will quietly break — usually in production, usually at the worst possible time. This is precisely the failure mode that plagues loosely-typed REST APIs, where the only contract is a Wiki page or a Postman collection that nobody updates.

A `.proto` file solves this by becoming the single source of truth that both sides compile against. It is not documentation that can drift out of sync with reality — it is the actual blueprint from which both the client code and the server code are generated. If the contract changes, the code that depends on it literally fails to compile until it is updated. This shifts an entire category of bugs from "discovered at 3 AM in production" to "caught instantly on your laptop by the compiler," which is one of the most valuable trades you can make as a backend engineer.

Designing a good `.proto` file follows a few disciplined habits that experienced engineers develop over time. Each field is assigned a permanent number, and that number — not the field name — is what actually gets encoded onto the wire, so once a field number ships to production, it must never be reused or reassigned, even if the field itself is later removed. Field numbers between 1 and 15 take up only a single byte on the wire, so the most frequently used fields in a message should claim those low numbers, while rarely-used fields can comfortably sit higher up. Messages should be designed to evolve gracefully: new fields should always be added with new numbers rather than repurposing old ones, and fields you no longer use should be marked `reserved` rather than deleted outright, which prevents a future engineer from accidentally reusing that number and corrupting data for any client still running older code.

Here is what a thoughtfully designed `.proto` file looks like in practice, with the reasoning embedded as comments:

```protobuf
syntax = "proto3";

package order.v1;
option go_package = "myapp/pb/orderpb";

// Frequently accessed fields get the low numbers (1-15)
// because they encode in a single byte on the wire.
message Order {
  string order_id     = 1;   // hot field, low number
  string user_id      = 2;   // hot field, low number
  double total_amount = 3;
  OrderStatus status   = 4;
  int64  created_at    = 5;  // unix timestamp

  reserved 6, 7;              // old fields removed, numbers retired forever
  reserved "discount_code";   // old field name retired forever

  repeated OrderItem items = 8;
}

enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0; // proto3 requires a zero-value default
  ORDER_STATUS_PENDING     = 1;
  ORDER_STATUS_CONFIRMED   = 2;
  ORDER_STATUS_SHIPPED     = 3;
}

message OrderItem {
  string product_id = 1;
  int32  quantity   = 2;
  double unit_price = 3;
}

message GetOrderRequest {
  string order_id = 1;
}

// The service block is the actual remote-procedure contract:
// it lists which functions can be called across the network.
service OrderService {
  rpc GetOrder (GetOrderRequest) returns (Order);
}
```

Notice that the contract cleanly separates **messages** (the data shapes) from the **service** (the callable functions). This separation is intentional — the same `Order` message might be reused across several different RPC calls, and keeping data definitions independent from the functions that use them keeps the whole schema easier to evolve without breaking unrelated parts of the system.

## Part Two — What a Client Request Looks Like, In Data Form

Before walking through the full cycle step by step, it helps enormously to see the entire transformation in one condensed view — from the Go object your code touches, all the way down to raw bytes on the wire, and back up again on the other side.

```
┌──────────────────────────────────────────────────────────────────────┐
│                  THE DATA, AT EVERY LAYER                            │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  LAYER 1 — Application Object (Go struct)                            │
│  ┌─────────────────────────────────────────┐                         │
│  │  &GetOrderRequest{ OrderId: "ord-9981" }│                         │
│  └─────────────────────────────────────────┘                         │
│                      │                                               │
│                      ▼                                               │
│  LAYER 2 — Protobuf Binary (Tag-Length-Value encoding)               │
│  ┌─────────────────────────────────────────┐                         │
│  │  0A 08 6F 72 64 2D 39 39 38 31          │                         │
│  │  │  │  └──────────┬──────────┘          │                         │
│  │  │  │             "ord-9981" (ASCII)    │                         │
│  │  │  └─ length: 8 bytes                  │                         │
│  │  └─ tag: field #1, wire type 2 (string) │                         │
│  └─────────────────────────────────────────┘                         │
│                      │                                               │
│                      ▼                                               │
│  LAYER 3 — HTTP/2 DATA Frame                                         │
│  ┌─────────────────────────────────────────┐                         │
│  │  Length: 10 | Type: DATA | Stream ID: 5 │                         │
│  │  Flags: END_STREAM                      │                         │
│  │  Payload: [the 10 bytes from Layer 2]   │                         │
│  └─────────────────────────────────────────┘                         │
│                      │                                               │
│                      ▼                                               │
│  LAYER 4 — TCP Segments (raw bytes on the wire)                      │
│  ┌─────────────────────────────────────────┐                         │
│  │  SYN → ACK → [segment 1][segment 2]...  │                         │
│  └─────────────────────────────────────────┘                         │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

Each layer wraps the one beneath it like nested envelopes. The Go struct is the letter you write. Protobuf is folding that letter into a small, dense paper packet. HTTP/2 framing is putting that packet into an envelope with a stream number written on the outside, so the post office knows which conversation it belongs to. TCP is the delivery truck that physically carries the envelope to the destination, breaking it into smaller pieces if needed and guaranteeing every piece arrives, in order, intact.

## Part Three — The Full Eighteen-Step Cycle

Now we walk through the entire journey, step by step, the way a senior engineer would trace it on a whiteboard for a newcomer. Imagine the Order Service is asking the User Service: "give me the user with ID `usr-4521`."

```
   ORDER SERVICE (Client)                              USER SERVICE (Server)
 ╔═══════════════════════╗                            ╔═══════════════════════╗
 ║   1  Client Calls     ║                            ║                       ║
 ║   2  Stub Converts    ║                            ║                       ║
 ║   3  Protobuf Encode  ║                            ║                       ║
 ║   4  HTTP/2 Frame     ║                            ║                       ║
 ║   5  Multiplexed  ────║──── 6  TCP ──────────────► ║   7  TCP Received     ║
 ║                       ║                            ║   8  Frame Reassembled║
 ║                       ║                            ║   9  Protobuf Decode  ║
 ║                       ║                            ║  10  Interceptors     ║
 ║                       ║                            ║  11  Handler Runs     ║
 ║                       ║                            ║  12  Database Call    ║
 ║                       ║                            ║  13  Response Built   ║
 ║                       ║                            ║  14  Response Encode  ║
 ║  18  Final Object ◄───║◄── 16 TCP back ──15  Frame ║                       ║
 ║  17  Decode           ║                            ║                       ║
 ╚═══════════════════════╝                            ╚═══════════════════════╝
```

### Step 1 — Client Calls a gRPC Method

Everything begins with a single line of application code that looks deceptively ordinary. The engineer writing the Order Service never touches sockets, never thinks about bytes — they simply call a method, exactly as if the User Service lived in the same process.

```go
resp, err := userClient.GetUser(ctx, &pb.GetUserRequest{
    UserId: "usr-4521",
})
```

At this exact moment, in memory, you have nothing more exotic than a plain Go struct sitting on the stack:

```
GetUserRequest{
  UserId: "usr-4521"
}
```

This is the illusion gRPC is built to maintain — that calling across a network feels identical to calling a function down the hall. Everything from here on is the machinery working invisibly underneath that illusion.

### Step 2 — The gRPC Client Stub Converts the Request

The `userClient` object is not something you wrote by hand; it was generated by `protoc` from the `.proto` file. This generated code is called the **stub**, and its entire job is to be the translator standing between your clean Go code and the messy business of network transmission. The stub takes the `GetUserRequest` struct, attaches metadata about which method is being called (`/user.UserService/GetUser`), and hands the whole package off to gRPC's internal transport machinery.

```
┌────────────────────────────────────────────┐
│  STUB RECEIVES:                            │
│   method: "/user.UserService/GetUser"      │
│   message: GetUserRequest{UserId: "usr..."}│
│                                            │
│  STUB PREPARES:                            │
│   → marshaling instructions for Protobuf   │
│   → routing path for HTTP/2                │
└────────────────────────────────────────────┘
```

### Step 3 — Protobuf Serialization

Now the actual encoding happens. The Protobuf runtime walks through the `GetUserRequest` struct field by field, and for each populated field, it writes a **tag** (which packs together the field number and the wire type) followed by the **length** of the data, followed by the **raw bytes** of the value itself. Field names are never written — only their numeric tags — which is the single biggest reason Protobuf payloads are so much smaller than JSON.

```
Go struct:                Binary output (hex):
GetUserRequest{           0A 08 75 73 72 2D 34 35 32 31
  UserId: "usr-4521"      │  │  └──────┬──────┘
}                         │  │     "usr-4521" in ASCII
                          │  └─ length = 8 bytes
                          └─ tag = (field 1 << 3) | wiretype 2

Total size: 10 bytes (vs. ~28 bytes as JSON text)
```

### Step 4 — gRPC Creates an HTTP/2 Frame

Those 10 raw bytes cannot simply be thrown onto the network alone — HTTP/2 needs to wrap them inside a structured **frame** so the receiving end knows exactly how to interpret what is arriving. gRPC packages the Protobuf bytes inside an HTTP/2 **DATA frame**, and separately sends a **HEADERS frame** carrying metadata like the content type and the RPC method path.

```
HEADERS Frame (Stream ID: 7)
┌───────────────────────────────────────┐
│ :method: POST                         │
│ :path: /user.UserService/GetUser      │
│ content-type: application/grpc+proto  │
└───────────────────────────────────────┘

DATA Frame (Stream ID: 7)
┌───────────────────────────────────────┐
│ Length: 10                            │
│ Flags: END_STREAM                     │
│ Payload: 0A 08 75 73 72 2D 34 35 32 31│
└───────────────────────────────────────┘
```

The **Stream ID** here is critical — it is a numeric label that ties the HEADERS frame and the DATA frame together as belonging to the same logical request, even though, as the next step explains, many unrelated frames from other requests might be interleaved on the wire around them.

### Step 5 — HTTP/2 Multiplexing

If the Order Service happens to be calling the User Service and the Payment Service at the same moment, all of that traffic can share a single TCP connection rather than opening a new connection for every call. HTTP/2 achieves this through multiplexing: every frame carries its Stream ID, so frames from entirely different requests can be interleaved on the wire and still be correctly reassembled on the other end, with no request ever blocking another.

```
Single TCP Connection, multiple streams interleaved:

  [Stream 7: HEADERS][Stream 9: HEADERS][Stream 7: DATA][Stream 9: DATA]
       (GetUser)         (GetPayment)      (GetUser)       (GetPayment)
```

### Step 6 — Data Goes Through TCP

The HTTP/2 frames are now handed to the operating system's TCP stack, which breaks them into TCP segments if they are larger than the network's maximum transmission unit, attaches sequence numbers so the receiver can reorder anything that arrives out of order, and guarantees — through acknowledgments and retransmission — that every byte eventually arrives intact. This is the layer where reliability is actually enforced; HTTP/2 trusts TCP completely to never lose or corrupt data silently.

```
TCP Segment:
┌─────────────────────────────────────────┐
│ Src Port: 51022   Dst Port: 50051       │
│ Seq: 10293        Ack: 88210            │
│ Payload: [HTTP/2 frame bytes]           │
└─────────────────────────────────────────┘
        │
        ▼  travels across the network (or localhost loopback)
```

### Step 7 — Server Receives TCP Packets

On the User Service machine, the kernel's network stack receives the incoming TCP segments, verifies their checksums, places them in the correct order based on sequence numbers, and hands the reassembled byte stream up to the application layer — in this case, the gRPC server's HTTP/2 listener, which has been sitting idle on port 50051 waiting for exactly this.

### Step 8 — HTTP/2 Frame Reassembled

The gRPC server's HTTP/2 layer reads the incoming byte stream and reconstructs the original frames: it recognizes the HEADERS frame on Stream 7, reads the `:path` to know that `GetUser` is being invoked, and recognizes the accompanying DATA frame on the same Stream ID as the Protobuf payload belonging to that call. At this point, the server has recovered the exact same 10 raw bytes that were serialized back on Step 3.

```
Reassembled:
  Stream ID: 7
  Method: /user.UserService/GetUser
  Payload (10 bytes): 0A 08 75 73 72 2D 34 35 32 31
```

### Step 9 — Protobuf Deserialization

The server now runs the reverse of Step 3. It reads the tag byte, determines this is field number 1 with wire type 2 (a length-delimited string), reads the length byte to know 8 bytes follow, and reads those 8 bytes as the string `"usr-4521"`. The flat binary stream is reconstructed back into a typed, structured Go object — and crucially, this reconstruction is only possible because both sides compiled from the identical `.proto` file, so the server knows with certainty that field 1 means `user_id`.

```
Binary in:                          Go struct out:
0A 08 75 73 72 2D 34 35 32 31  →   GetUserRequest{
                                      UserId: "usr-4521"
                                    }
```

### Step 10 — Interceptors Execute

Before the actual business logic ever runs, gRPC passes the request through a chain of **interceptors** — middleware functions that wrap every incoming call. This is exactly where you would place cross-cutting concerns like authentication token validation, request logging, rate limiting, or distributed tracing span creation, so that individual handler functions stay focused purely on business logic rather than repeating the same boilerplate on every method.

```go
func AuthInterceptor(ctx context.Context, req interface{},
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {

    token := extractToken(ctx)
    if !isValid(token) {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }
    log.Printf("Incoming call: %s", info.FullMethod)

    // Pass control onward to the next interceptor, or finally the handler
    return handler(ctx, req)
}
```

### Step 11 — Service Handler Invoked

Only now, after passing through every interceptor, does gRPC finally call the actual function the engineering team wrote — the handler that contains the real business logic for fetching a user.

```go
func (s *userServer) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.UserResponse, error) {
    user, err := s.repo.FindByID(ctx, req.UserId)
    if err != nil {
        return nil, status.Error(codes.NotFound, "user not found")
    }
    return &pb.UserResponse{
        UserId: user.ID,
        Name:   user.Name,
        Email:  user.Email,
    }, nil
}
```

### Step 12 — Database Call

Inside the handler, the actual work happens — a query is issued to PostgreSQL or MongoDB to retrieve the requested user's row or document, and the raw database record is mapped into application-level structures.

```
SQL:  SELECT id, name, email FROM users WHERE id = 'usr-4521';

Row returned:
┌──────────┬───────────────┬──────────────────────┐
│ id       │ name          │ email                │
├──────────┼───────────────┼──────────────────────┤
│ usr-4521 │ Karim Ahmed   │ karim@example.com    │
└──────────┴───────────────┴──────────────────────┘
```

### Step 13 — Response Object Created

The database row is mapped into the `UserResponse` Protobuf message type that was defined in the `.proto` contract, becoming a structured Go object ready to travel back across the wire.

```
UserResponse{
  UserId: "usr-4521",
  Name:   "Karim Ahmed",
  Email:  "karim@example.com",
}
```

### Step 14 — Response Serialized

Exactly as in Step 3, but now in the reverse direction, the Protobuf runtime walks through this `UserResponse` struct and encodes each populated field into the same compact Tag-Length-Value binary format.

```
UserResponse{...}  →  binary:
0A 08 75 73 72 2D 34 35 32 31
12 0B 4B 61 72 69 6D 20 41 68 6D 65 64
1A 11 6B 61 72 69 6D 40 65 78 61 6D 70 6C 65 2E 63 6F 6D
```

### Step 15 — HTTP/2 Response Frame Created

These response bytes are packaged into a new HTTP/2 DATA frame, tagged with the very same Stream ID (7) that the original request used, so the client can correlate this response with the call it made. A trailing HEADERS frame also carries the gRPC status code, confirming success or describing failure.

```
DATA Frame (Stream ID: 7)        Trailing HEADERS (Stream ID: 7)
┌─────────────────────────┐      ┌─────────────────────────┐
│Payload: [response bytes]│      │ grpc-status: 0 (OK)     │
└─────────────────────────┘      └─────────────────────────┘
```

### Step 16 — Response Sent Through TCP

Just as the request did in Step 6, this response now travels back down through the TCP stack, gets segmented if necessary, and is transmitted reliably back across the network to the Order Service.

### Step 17 — Client Receives and Deserializes Response

The Order Service's gRPC client reassembles the incoming HTTP/2 frames on Stream 7, recognizes them as the long-awaited reply to the original call, and runs Protobuf deserialization on the payload bytes — reconstructing the binary back into a typed `UserResponse` Go struct, mirroring exactly what the server did back in Step 9.

### Step 18 — Client Gets Final Object

The generated stub hands the fully reconstructed object back to the line of code that has been waiting, blocked, this entire time:

```go
resp, err := userClient.GetUser(ctx, &pb.GetUserRequest{UserId: "usr-4521"})

// resp now equals:
// &UserResponse{
//   UserId: "usr-4521",
//   Name:   "Karim Ahmed",
//   Email:  "karim@example.com",
// }
```

From the perspective of the engineer who wrote this one line, none of the eighteen steps in between were ever visible. They called a function and got an object back — exactly the illusion gRPC promises. But underneath that single line, a Go struct was reduced to ten raw bytes, wrapped in framed envelopes, multiplexed across a shared connection, carried reliably over TCP, unwrapped, reconstructed, passed through a chain of middleware, handed to business logic, turned into a database query, mapped back into a struct, re-encoded, and shipped all the way back — typically completing the entire round trip in under a single millisecond on a local network.

## Part Four — The Whole Cycle, End to End in One View

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  CLIENT SIDE                          │  SERVER SIDE                │
│  ───────────                          │  ───────────                │
│                                       │                             │
│  1.  Call GetUser(req)                │                             │
│  2.  Stub wraps method + message      │                             │
│  3.  Protobuf encode  → binary bytes  │                             │
│  4.  Wrap in HTTP/2 frame             │                             │
│  5.  Multiplex onto shared connection │                             │
│  6.  TCP transmit         ────────────┼──►  7.  TCP receive         │
│                                       │     8.  Reassemble frame    │
│                                       │     9.  Protobuf decode     │
│                                       │    10.  Run interceptors    │
│                                       │    11.  Call handler        │
│                                       │    12.  Query database      │
│                                       │    13.  Build response obj  │
│                                       │    14.  Protobuf encode     │
│                                       │    15.  Wrap in HTTP/2 frame│
│ 18.  Return final object  ◄───────────┼─── 16.  TCP transmit back   │
│ 17.  Protobuf decode                  │                             │
│                                       │                             │
└─────────────────────────────────────────────────────────────────────┘
```

Every layer in this diagram exists for a specific reason: Protobuf exists to make the data small and unambiguous, HTTP/2 exists to move that data efficiently and concurrently, TCP exists to guarantee it arrives intact, and the generated stub exists to hide all of that complexity behind what looks, to the application developer, like nothing more than an ordinary function call.
