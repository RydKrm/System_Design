### **SOLID Principles in JavaScript (Brief Explanation with Examples)**  

SOLID is a set of **5 key design principles** in object-oriented programming (OOP) that make code **flexible, maintainable, and scalable**.  

---

## **1. Single Responsibility Principle (SRP)**  
**"A class should have only one reason to change"** ‚Üí Do **one thing** well.  

### ‚ùå **Bad (Violates SRP)**  
```javascript
class User {
  constructor(name) {
    this.name = name;
  }

  saveToDB() {  // Handles DB logic
    console.log(`Saving ${this.name} to database...`);
  }

  sendEmail() {  // Handles email logic
    console.log(`Sending email to ${this.name}...`);
  }
}
```
**Problem:**  
- `User` class manages **database + email** ‚Üí **Two responsibilities!**  

### ‚úÖ **Good (Follows SRP)**  
```javascript
class User {
  constructor(name) {
    this.name = name;
  }
}

class UserDB {
  save(user) {
    console.log(`Saving ${user.name} to database...`);
  }
}

class EmailService {
  sendEmail(user) {
    console.log(`Sending email to ${user.name}...`);
  }
}
```
**Benefit:**  
- Each class has **one job** ‚Üí Easier to modify.

## **2. Open/Closed Principle (OCP)**  
**"Open for extension, closed for modification"** ‚Üí Add new features **without changing existing code**.  

### ‚ùå **Bad (Violates OCP)**  
```javascript
class Payment {
  process(method) {
    if (method === "credit") {
      console.log("Processing credit card...");
    } else if (method === "paypal") {
      console.log("Processing PayPal...");
    }
    // Adding a new payment requires modifying this class!
  }
}
```
**Problem:**  
- Need to **modify `Payment`** for every new payment type.  

### ‚úÖ **Good (Follows OCP)**  
```javascript
class PaymentMethod {
  process() {
    throw new Error("Method 'process()' must be implemented.");
  }
}

class CreditCard extends PaymentMethod {
  process() {
    console.log("Processing credit card...");
  }
}

class PayPal extends PaymentMethod {
  process() {
    console.log("Processing PayPal...");
  }
}

class Payment {
  process(method) {
    method.process();  // Works for any new payment type!
  }
}
```
**Benefit:**  
- **Add new payments** (e.g., Bitcoin) **without touching `Payment` class.**

## **3. Liskov Substitution Principle (LSP)**  
**"Subclasses should replace parent classes without breaking code"** ‚Üí Follow **inheritance properly**.  

### ‚ùå **Bad (Violates LSP)**  
```javascript
class Bird {
  fly() {
    console.log("Flying...");
  }
}

class Ostrich extends Bird {  // Ostrich can't fly!
  fly() {
    throw new Error("Can't fly!");  // Breaks parent behavior!
  }
}
```
**Problem:**  
- `Ostrich` breaks `Bird`‚Äôs contract ‚Üí **Unsafe substitution.**  

### ‚úÖ **Good (Follows LSP)**  
```javascript
class Bird {}

class FlyingBird extends Bird {
  fly() {
    console.log("Flying...");
  }
}

class Ostrich extends Bird {  // Doesn't inherit fly()
  run() {
    console.log("Running...");
  }
}
```
**Benefit:**  
- Subclasses **don‚Äôt break** parent behavior.

## **4. Interface Segregation Principle (ISP)**  
**"Clients shouldn‚Äôt depend on interfaces they don‚Äôt use"** ‚Üí **Avoid fat interfaces.**  

### ‚ùå **Bad (Violates ISP)**  
```javascript
class Machine {
  print() {}
  scan() {}
  fax() {}
}

class OldPrinter extends Machine {
  print() {
    console.log("Printing...");
  }
  scan() {  // Not needed but forced!
    throw new Error("Can't scan!");
  }
}
```
**Problem:**  
- `OldPrinter` forced to implement **unused methods.**  

### ‚úÖ **Good (Follows ISP)**  
```javascript
class Printer {
  print() {}
}

class Scanner {
  scan() {}
}

class OldPrinter extends Printer {  // Only implements print()
  print() {
    console.log("Printing...");
  }
}
```
**Benefit:**  
- **No forced unused methods** ‚Üí Cleaner code.

## **5. Dependency Inversion Principle (DIP)**  
**"Depend on abstractions, not concrete classes"** ‚Üí **Decouple code.**  

### ‚ùå **Bad (Violates DIP)**  
```javascript
class LightBulb {
  turnOn() {
    console.log("Bulb on!");
  }
}

class Switch {
  constructor() {
    this.bulb = new LightBulb();  // Direct dependency!
  }

  operate() {
    this.bulb.turnOn();
  }
}
```
**Problem:**  
- `Switch` **hardcoded** to `LightBulb` ‚Üí Not flexible.  

### ‚úÖ **Good (Follows DIP)**  
```javascript
class Switchable {
  turnOn() {
    throw new Error("Method 'turnOn()' must be implemented.");
  }
}

class LightBulb extends Switchable {
  turnOn() {
    console.log("Bulb on!");
  }
}

class Fan extends Switchable {
  turnOn() {
    console.log("Fan on!");
  }
}

class Switch {
  constructor(device) {  // Depends on abstraction
    this.device = device;
  }

  operate() {
    this.device.turnOn();
  }
}

const bulb = new LightBulb();
const switch = new Switch(bulb);  // Works with any Switchable device!
```
**Benefit:**  
- **Switch works with any device** (Bulb, Fan, etc.) ‚Üí **Loose coupling.**

### **Summary (SOLID Principles in JavaScript)**  
| Principle | Key Idea | Example Fix |
|-----------|---------|-------------|
| **SRP** | One class = One job | Split `User` into `User`, `UserDB`, `EmailService` |
| **OCP** | Extend without modifying | Use `PaymentMethod` abstraction |
| **LSP** | Safe inheritance | Don‚Äôt let `Ostrich` break `Bird` |
| **ISP** | Avoid fat interfaces | Split `Machine` into `Printer`, `Scanner` |
| **DIP** | Depend on abstractions | `Switch` depends on `Switchable`, not `LightBulb` |
