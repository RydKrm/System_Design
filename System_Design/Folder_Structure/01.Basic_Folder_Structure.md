## 1. Philosophy and Design Principles

Before the folder structure, understand the **why** behind every directory decision.

At 10M users and 1M products, a flat monolithic folder structure fails you in four specific ways: **independent deployability** becomes impossible (a bug in the recommendations engine deploys with a payment fix), **team autonomy** is impossible (12 teams all editing the same service folder), **observability** is impossible (one log stream for every operation type), and **scaling** is impossible (you cannot scale product search without scaling the entire checkout too).

The structure below is a **modular monolith that's pre-wired for microservice extraction**. Every "service" is a self-contained Go module today, deployable as a single binary or as independent containers tomorrow вЂ” without restructuring the code. This is the most pragmatic path for a team growing from 10 to 100 engineers: you get the simplicity of a monolith at the start, and the escape hatch to microservices when teams and traffic demand it.

## 2. Repository Strategy

```
Strategy: Monorepo (single Git repository for all services and infrastructure)

Why monorepo at this scale:
  вњ… Atomic commits across service + infrastructure + docs
  вњ… Single CI/CD configuration with shared tooling
  вњ… Shared pkg/ libraries with no versioning ceremony
  вњ… Easier cross-service refactoring (rename a struct field everywhere at once)
  вњ… Consistent observability field names enforced by shared packages
  вњ… One place for security scanning, dependency auditing, policy enforcement

Tooling that makes monorepo work at scale:
  - Bazel / Nx for incremental builds (only rebuild what changed)
  - CODEOWNERS files for per-directory ownership
  - Path-based CI triggers (only run order-service pipeline when order/ changes)
  - buf.build for Protobuf schema management across services
```

## 14. Why Each Directory Exists

```
DIRECTORY                    WHY IT EXISTS AT THIS SCALE
в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ

services/*/domain/           Pure Go structs with zero framework dependencies.
                             These are the types that travel across the entire
                             system. At 10M users, correctness matters more
                             than convenience вЂ” domain types enforce invariants.

services/*/migrations/       Each service owns its own database schema.
                             No shared tables, no cross-service foreign keys.
                             This is the only way to let teams deploy independently.

services/*/saga/             At checkout, you must coordinate inventory + payment
                             + notification across services atomically. Sagas
                             replace distributed transactions with compensating
                             actions and event-driven coordination.

pkg/outbox/                  The outbox pattern guarantees that DB writes and
                             event publications are atomic. Without it, a crash
                             between "save order to DB" and "publish OrderCreated
                             to Kafka" leaves your system in an inconsistent state.
                             At 10M users, this happens every hour.

pkg/featureflags/            Enables gradual rollouts and A/B testing without
                             deploys. Critical for a system at this scale where
                             a bad deploy affecting 10M users is catastrophic.

pkg/idempotency/             Payment operations must be idempotent. A network
                             timeout after "charge card" must not result in a
                             double charge. The idempotency store ensures that
                             retried requests return the same result.

pkg/cache/multilevel.go      L1 (in-process LRU) + L2 (Redis) cache reduces
                             Redis load by 80%+ for hot product pages. At 1M
                             products and 10M users, product page reads would
                             saturate a single Redis cluster without L1 caching.

ml/features/                 A feature store ensures that training and serving
                             use the same feature computation logic. Without it,
                             training-serving skew causes model accuracy to degrade
                             silently in production вЂ” the hardest bug to diagnose.

data/migrations/             Per-service migrations stored together for visibility.
                             Engineers can see the full schema history of every
                             service in one place, making cross-service impact
                             analysis tractable.

observability/slo/           SLO definitions are code, not dashboard clicks.
                             Version-controlled SLOs enable PR review for
                             reliability changes ("you're proposing to lower the
                             checkout SLO from 99.9% to 99.5% вЂ” why?").

security/vault/              At 10M users, secrets rotation is not optional.
                             A leaked DB password must be rotated in minutes,
                             not days. Vault + K8s integration enables automatic
                             rotation without application restarts.

.github/CODEOWNERS           At 100+ engineers, without ownership rules, anyone
                             can approve anything. CODEOWNERS enforces that only
                             the payments team can merge to payment-service/, only
                             the platform team can merge to infrastructure/.

docs/adr/                    Architecture Decision Records are the institutional
                             memory of the engineering organisation. When an
                             engineer asks "why did we choose Kafka over RabbitMQ?",
                             the answer is in ADR-002, written 3 years ago.
```

_This structure is designed to grow with you. Start by deploying a single binary containing all services. As teams grow and traffic demands it, extract services one at a time вЂ” the folder boundaries already define the extraction points. The observability, messaging, and infrastructure layers are designed to work identically whether you have 1 service or 50._
