# RabbitMQ: A Complete Beginner's Guide

## Understanding Why We Need RabbitMQ

Imagine you are building a modern web application where users can upload profile pictures. When a user uploads an image, your application needs to perform several tasks: resize the image to multiple dimensions for different devices, generate thumbnails, scan for inappropriate content, update the user's profile in the database, send a notification email, and perhaps update a cache. If you tried to do all of these tasks synchronously in a single request-response cycle, your user would be waiting for a very long time, and if any one of these tasks failed, the entire operation might fail.

This is where message brokers like RabbitMQ become invaluable. RabbitMQ acts as a middleman or intermediary that sits between different parts of your application, allowing them to communicate asynchronously. When your application receives an image upload, it can quickly acknowledge the upload to the user, then place messages onto RabbitMQ describing the work that needs to be done. Other services, running independently, can pick up these messages and process them at their own pace.

RabbitMQ solves several critical problems in distributed systems. First, it provides decoupling, meaning your services don't need to know about each other directly. The image upload service doesn't need to know how many image processing services exist or where they are located; it simply sends a message to RabbitMQ. Second, it provides reliability through message persistence and acknowledgments, ensuring that even if a service crashes while processing a message, the message won't be lost. Third, it enables load balancing and scalability. If you have multiple instances of an image processing service, RabbitMQ can distribute work among them automatically. Fourth, it handles peak loads gracefully. During high traffic periods, messages can queue up and be processed when services have capacity, rather than overwhelming your servers.

Beyond these technical benefits, RabbitMQ enables better software architecture. It allows different teams to work on different services using different programming languages and technologies. Your image processing service could be written in Python for its excellent image libraries, while your notification service is in Node.js for its efficient I/O handling, and your main application is in Golang for performance. They all communicate through RabbitMQ using a standard messaging protocol called AMQP (Advanced Message Queuing Protocol).

## How RabbitMQ Works: The Fundamental Architecture

To understand how RabbitMQ works, think of it as a sophisticated post office system for your applications. Just as a post office receives letters, sorts them, stores them temporarily, and delivers them to recipients, RabbitMQ receives messages from producers, routes them according to rules, stores them in queues, and delivers them to consumers.

At its core, RabbitMQ is a message broker that implements the AMQP protocol. It runs as a standalone server (or cluster of servers for high availability) that your applications connect to. When RabbitMQ starts, it creates a management interface and opens ports for applications to connect. The default port for AMQP connections is 5672, and the management web interface typically runs on port 15672.

The RabbitMQ server maintains several internal components. The connection manager handles network connections from client applications. Each application that wants to use RabbitMQ must first establish a TCP connection to the server. Within a single connection, applications can create multiple channels, which are virtual connections that share the physical TCP connection. This is efficient because creating TCP connections is expensive, but channels are lightweight.

Inside RabbitMQ, there are exchanges, queues, and bindings. Exchanges are the entry points where messages arrive from producers. Queues are the storage locations where messages wait to be consumed. Bindings are the rules that tell RabbitMQ how to route messages from exchanges to queues. This three-tier architecture is what makes RabbitMQ so flexible and powerful.

Exchange -> Queues -> Binding

RabbitMQ also maintains metadata about all these components in its internal database. When you declare a queue or an exchange, this information is stored so that if RabbitMQ restarts, it can recreate the same structure. If you make queues and messages persistent, they survive restarts, providing durability.

The broker uses Erlang's actor model internally, where each queue is an independent process. This means queues can process messages concurrently, and if one queue has a problem, it doesn't affect others. RabbitMQ's use of Erlang also gives it excellent concurrency handling and fault tolerance capabilities.

## Understanding Message Flow in RabbitMQ

The journey of a message through RabbitMQ follows a well-defined path from creation to consumption. Let's trace this journey step by step to understand how data flows through the system.

```
┌──────────────     ┌──────────────┐    ┌────────────┐   ┌──────────────┐
│              │    │              │    │            │   │              │
│   PRODUCER   │───▶│   EXCHANGE   │───▶│   QUEUE    │──▶│   CONSUMER   │          │              │    │              │    │            │   └──────────────┘         └──────────────┘    └──────────────┘    └────────────┘
   Step 1            Step 2                   Step 3           Step 4
  Publish msg     Route msg based        Store msg until     Receive & process
  to exchange     on routing rules        consumer ready         message
```

The complete flow begins when a producer application creates a message. This message typically consists of two parts: the message body, which contains the actual data (often JSON, XML, or binary data), and message properties, which include metadata like routing key, delivery mode (persistent or transient), content type, headers, and more.

The producer then publishes this message to an exchange. It doesn't send the message directly to a queue; this is a crucial distinction. The producer specifies which exchange to use and provides a routing key, which is a string that the exchange will use to determine where to route the message. For example, if you're sending an image processing task, you might use a routing key like "image.resize" or "image.thumbnail".

When the exchange receives the message, it examines the routing key and compares it against the bindings that have been configured. Bindings are connections between exchanges and queues that include a pattern or routing key. Based on the exchange type (which we'll explore in detail shortly), the exchange decides which queue or queues should receive this message. An exchange might send the message to zero, one, or multiple queues depending on its type and the binding patterns.

Once the exchange determines the target queue or queues, it copies the message and places it into each queue. The queue acts as a buffer, storing messages in the order they arrive (though this can be modified with priorities). Messages wait in the queue until a consumer is ready to process them. If multiple consumers are connected to the same queue, RabbitMQ distributes messages among them in a round-robin fashion by default, providing automatic load balancing.

When a consumer is ready to process messages, it pulls a message from the queue (or RabbitMQ pushes it to the consumer, depending on the consumption mode). The consumer then processes the message according to its business logic. After successful processing, the consumer sends an acknowledgment back to RabbitMQ. Only when RabbitMQ receives this acknowledgment does it remove the message from the queue. If the consumer crashes before acknowledging, RabbitMQ will redeliver the message to another consumer, ensuring no work is lost.

Let's visualize a more detailed flow:

```
Producer Application                    RabbitMQ Broker                    Consumer Application
        │                                      │                                    │
        │  1. Create message                   │                                    │
        │     {user_id: 123,                   │                                    │
        │      action: "resize"}               │                                    │
        │                                      │                                    │
        │  2. Publish to exchange              │                                    │
        │     exchange: "image_tasks"          │                                    │
        │     routing_key: "image.resize"      │                                    │
        ├─────────────────────────────────────▶│                                    │
        │                                      │                                    │
        │                                      │  3. Exchange receives message      │
        │                                      │     Checks routing rules           │
        │                                      │     Finds matching queue           │
        │                                      │                                    │
        │                                      │  4. Message stored in queue        │
        │                                      │     [image.resize.queue]           │
        │                                      │                                    │
        │                                      │  5. Consumer subscribes            │
        │                                      │◀───────────────────────────────────┤
        │                                      │                                    │

```

This flow provides several powerful guarantees. If the consumer crashes after step 6 but before step 8, RabbitMQ will redeliver the message. If RabbitMQ itself crashes, persistent messages in durable queues survive. If no queue matches the routing rules, the message is either dropped or sent to an alternate exchange if configured.

## Deep Dive into the Producer → Exchange → Queue → Consumer Flow

Now let's examine each component in this flow with greater detail to understand their roles and responsibilities.

**The Producer's Responsibility**

A producer is any application that sends messages to RabbitMQ. The producer's job is remarkably simple yet important. It must establish a connection to RabbitMQ, create a channel within that connection, and publish messages to an exchange. The producer needs to make several decisions when publishing: which exchange to use, what routing key to provide, whether the message should be persistent (survive broker restarts), and what properties to attach.

The producer doesn't need to know anything about queues or consumers. It simply sends messages to exchanges and trusts RabbitMQ to handle the rest. This decoupling is powerful because you can add or remove consumers without changing producer code. You can add new routing rules without modifying producers.

A producer can also request confirmations from RabbitMQ, where the broker acknowledges that it has successfully received and routed the message. This provides an extra layer of reliability for critical messages. Additionally, producers can publish messages with the "mandatory" flag, which tells RabbitMQ to return the message if it cannot be routed to any queue, allowing the producer to handle unroutable messages.

**The Exchange's Intelligence**

The exchange is the routing brain of RabbitMQ. When a message arrives at an exchange, it must decide where to send it. This decision is based on the exchange type, the message's routing key, and the bindings that connect the exchange to queues.

Exchanges have names and types, and they can be durable (survive restarts) or temporary. They can be auto-deleted when no longer in use. An exchange doesn't store messages; it only routes them. If a message arrives at an exchange and the exchange determines that no queues match the routing criteria, the message is silently dropped unless the producer set the mandatory flag.

One powerful feature is that exchanges can be bound to other exchanges, creating complex routing topologies. For example, you might have a main exchange that routes messages to specialized exchanges, which then route to specific queues. This allows for sophisticated messaging patterns.

**The Queue's Patience**

A queue is a buffer that stores messages until consumers are ready to process them. Each queue has a name, and like exchanges, queues can be durable (survive restarts) or temporary, and they can be auto-deleted when all consumers disconnect.

Queues maintain the order of messages (FIFO by default), though you can configure priority queues where high-priority messages jump ahead. A queue can have multiple consumers connected to it, and RabbitMQ will distribute messages among them. A queue can also have a maximum length or size limit, and you can configure what happens when these limits are reached (reject new messages, drop old messages, etc.).

Queues are also where message TTL (time-to-live) is enforced. You can set messages to expire if they aren't consumed within a certain time. Dead letter exchanges can be configured to receive messages that are rejected, expire, or exceed queue length limits, providing sophisticated error handling.

**The Consumer's Diligence**

A consumer is an application that receives messages from queues and processes them. Consumers connect to RabbitMQ, subscribe to one or more queues, and receive messages either by pulling them (basic.get) or by having them pushed (basic.consume, which is more efficient).

The consumer's most important responsibility is acknowledgment. When a consumer receives a message, it must tell RabbitMQ whether the message was processed successfully. A positive acknowledgment (ACK) tells RabbitMQ to delete the message. A negative acknowledgment (NACK) or rejection tells RabbitMQ that processing failed, and RabbitMQ can requeue the message or send it to a dead letter exchange.

Consumers can also implement prefetch limits, which control how many unacknowledged messages they can receive at once. This prevents a fast consumer from grabbing all messages while slow consumers sit idle. Setting prefetch to 1, for example, ensures fair distribution among consumers.

This architecture allows for remarkable flexibility. You can have multiple producers sending to the same exchange, multiple exchanges routing to the same queue, and multiple consumers processing from the same queue. Each component can scale independently based on your needs.

## Understanding All Exchange Types with Code Examples

RabbitMQ provides four main types of exchanges, each with different routing behaviors. Understanding these types is crucial because they determine how your messages flow through the system. Let's explore each type with detailed explanations and Golang code examples.

### Direct Exchange: Exact Matching

A direct exchange routes messages based on exact matching between the routing key in the message and the routing key in the binding. Think of it like a postal sorting facility where each letter must have an exact address match to be delivered to the right mailbox.

When a message arrives at a direct exchange with a routing key of `image.resize`, the exchange looks for queues bound to it with exactly that binding key `image.resize`. If a queue is bound with `image.thumbnail`, this message won't go there. This makes direct exchanges perfect for task distribution where you want specific types of work to go to specific workers.

```go
package main

import (
	"log"
	"github.com/streadway/amqp"
)

// DirectExchangeExample demonstrates direct exchange routing
func DirectExchangeExample() {
	// Connect to RabbitMQ server
	conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
	if err != nil {
		log.Fatalf("Failed to connect: %v", err)
	}
	defer conn.Close()

	// Create a channel
	ch, err := conn.Channel()
	if err != nil {
		log.Fatalf("Failed to open channel: %v", err)
	}
	defer ch.Close()

	// Declare a direct exchange
	// The exchange name is "logs_direct"
	// Exchange type is "direct"
	// Durable means the exchange survives broker restarts
	err = ch.ExchangeDeclare(
		"logs_direct", // exchange name
		"direct",      // exchange type
		true,          // durable
		false,         // auto-deleted
		false,         // internal
		false,         // no-wait
		nil,           // arguments
	)
	if err != nil {
		log.Fatalf("Failed to declare exchange: %v", err)
	}

	// Declare queues for different log severities
	queueInfo, _ := ch.QueueDeclare("info_queue", true, false, false, false, nil)
	queueError, _ := ch.QueueDeclare("error_queue", true, false, false, false, nil)
	queueWarning, _ := ch.QueueDeclare("warning_queue", true, false, false, false, nil)

	// Bind queues to exchange with specific routing keys
	// Info queue receives messages with routing key "info"
	ch.QueueBind(queueInfo.Name, "info", "logs_direct", false, nil)
	
	// Error queue receives messages with routing key "error"
	ch.QueueBind(queueError.Name, "error", "logs_direct", false, nil)
	
	// Warning queue receives messages with routing key "warning"
	ch.QueueBind(queueWarning.Name, "warning", "logs_direct", false, nil)

	// Publish messages with different routing keys
	// This message goes only to error_queue
	ch.Publish(
		"logs_direct", // exchange
		"error",       // routing key
		false,         // mandatory
		false,         // immediate
		amqp.Publishing{
			ContentType: "text/plain",
			Body:        []byte("Database connection failed"),
		},
	)

	// This message goes only to info_queue
	ch.Publish(
		"logs_direct",
		"info",
		false,
		false,
		amqp.Publishing{
			ContentType: "text/plain",
			Body:        []byte("User logged in successfully"),
		},
	)

	log.Println("Direct exchange messages published")
}

/*
Flow Visualization:

Producer                   Direct Exchange                    Queues
   │                      (logs_direct)                         │
   │                            │                               │
   │ Publish("error", msg)      │                               │
   ├───────────────────────────▶│                               │
   │                            │  Match "error"                │
   │                            ├──────────────────────────────▶│ error_queue
   │                            │                               │
   │ Publish("info", msg)       │                               │
   ├───────────────────────────▶│                               │
   │                            │  Match "info"                 │
   │                            ├──────────────────────────────▶│ info_queue
   │                            │                               │
   │ Publish("warning", msg)    │                               │
   ├───────────────────────────▶│                               │
   │                            │  Match "warning"              │
   │                            ├──────────────────────────────▶│ warning_queue
*/
```

### Topic Exchange: Pattern Matching

A topic exchange is more sophisticated than a direct exchange because it routes messages based on pattern matching. The routing key is a list of words separated by dots (like "usa.news.weather"), and bindings can use wildcards. The asterisk (*) matches exactly one word, and the hash (#) matches zero or more words.

This is incredibly powerful for complex routing scenarios. Imagine you're building a news application where different subscribers want different categories of news. Some want all sports news, some want only football news, and some want all news from Europe regardless of category.

```go
package main

import (
	"log"
	"github.com/streadway/amqp"
)

// TopicExchangeExample demonstrates topic exchange with pattern matching
func TopicExchangeExample() {
	conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
	if err != nil {
		log.Fatalf("Failed to connect: %v", err)
	}
	defer conn.Close()

	ch, err := conn.Channel()
	if err != nil {
		log.Fatalf("Failed to open channel: %v", err)
	}
	defer ch.Close()

	// Declare topic exchange
	err = ch.ExchangeDeclare(
		"news_topics", // exchange name
		"topic",       // exchange type - THIS IS KEY
		true,          // durable
		false,         // auto-deleted
		false,         // internal
		false,         // no-wait
		nil,           // arguments
	)
	if err != nil {
		log.Fatalf("Failed to declare exchange: %v", err)
	}

	// Create queues for different subscribers
	allSportsQueue, _ := ch.QueueDeclare("all_sports_queue", true, false, false, false, nil)
	footballQueue, _ := ch.QueueDeclare("football_only_queue", true, false, false, false, nil)
	europeQueue, _ := ch.QueueDeclare("europe_news_queue", true, false, false, false, nil)
	urgentQueue, _ := ch.QueueDeclare("urgent_all_queue", true, false, false, false, nil)

	// Bind with patterns
	// *.sports.* matches "usa.sports.football", "uk.sports.cricket", etc.
	ch.QueueBind(allSportsQueue.Name, "*.sports.*", "news_topics", false, nil)
	
	// *.sports.football matches only football sports news from any region
	ch.QueueBind(footballQueue.Name, "*.sports.football", "news_topics", false, nil)
	
	// europe.# matches "europe.sports.football", "europe.politics", "europe.weather.sunny", etc.
	ch.QueueBind(europeQueue.Name, "europe.#", "news_topics", false, nil)
	
	// #.urgent matches any message ending with "urgent"
	ch.QueueBind(urgentQueue.Name, "#.urgent", "news_topics", false, nil)

	// Publish messages with different routing keys
	
	// Goes to: all_sports_queue, football_only_queue
	ch.Publish("news_topics", "usa.sports.football", false, false, amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("NFL playoffs update"),
	})

	// Goes to: all_sports_queue, europeQueue
	ch.Publish("news_topics", "europe.sports.cricket", false, false, amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("Cricket match results"),
	})

	// Goes to: europeQueue only
	ch.Publish("news_topics", "europe.politics", false, false, amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("Election results announced"),
	})

	// Goes to: all_sports_queue, football_only_queue, europeQueue, urgentQueue
	ch.Publish("news_topics", "europe.sports.football.urgent", false, false, amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("Major injury to star player"),
	})

	log.Println("Topic exchange messages published")
}

/*
Pattern Matching Rules:

Routing Key: "usa.sports.football"
Pattern: "*.sports.*"        → MATCH (one word, "sports", one word)
Pattern: "*.sports.football"  → MATCH (one word, "sports", "football")
Pattern: "europe.#"           → NO MATCH (doesn't start with "europe")
Pattern: "#.urgent"           → NO MATCH (doesn't end with "urgent")

Routing Key: "europe.sports.football.urgent"
Pattern: "*.sports.*"         → NO MATCH (has 4 words, not 3)
Pattern: "europe.#"           → MATCH (starts with "europe", # matches rest)
Pattern: "#.urgent"           → MATCH (# matches beginning, ends with "urgent")
*/
```

### Fanout Exchange: Broadcasting

A fanout exchange is the simplest type. It completely ignores routing keys and simply broadcasts every message to all queues bound to it. Think of it like a radio tower broadcasting a signal to all receivers within range.

This is perfect for scenarios where multiple independent systems need to react to the same event. For example, when a user makes a purchase, you might want to update inventory, send a confirmation email, trigger analytics, update a data warehouse, and notify a shipping system. All these actions are independent and should all happen in response to the same purchase event.

```go
package main

import (
	"encoding/json"
	"log"
	"github.com/streadway/amqp"
)

// FanoutExchangeExample demonstrates broadcasting to all queues
func FanoutExchangeExample() {
	conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
	if err != nil {
		log.Fatalf("Failed to connect: %v", err)
	}
	defer conn.Close()

	ch, err := conn.Channel()
	if err != nil {
		log.Fatalf("Failed to open channel: %v", err)
	}
	defer ch.Close()

	// Declare fanout exchange
	err = ch.ExchangeDeclare(
		"user_events",  // exchange name
		"fanout",       // exchange type - broadcasts to all
		true,           // durable
		false,          // auto-deleted
		false,          // internal
		false,          // no-wait
		nil,            // arguments
	)
	if err != nil {
		log.Fatalf("Failed to declare exchange: %v", err)
	}

	// Create queues for different services
	emailQueue, _ := ch.QueueDeclare("email_service_queue", true, false, false, false, nil)
	analyticsQueue, _ := ch.QueueDeclare("analytics_queue", true, false, false, false, nil)
	auditQueue, _ := ch.QueueDeclare("audit_log_queue", true, false, false, false, nil)
	cacheQueue, _ := ch.QueueDeclare("cache_invalidation_queue", true, false, false, false, nil)

	// Bind all queues to fanout exchange
	// Note: routing key is empty string "" because fanout ignores it
	ch.QueueBind(emailQueue.Name, "", "user_events", false, nil)
	ch.QueueBind(analyticsQueue.Name, "", "user_events", false, nil)
	ch.QueueBind(auditQueue.Name, "", "user_events", false, nil)
	ch.QueueBind(cacheQueue.Name, "", "user_events", false, nil)

	// Create an event
	event := map[string]interface{}{
		"event_type": "user_registered",
		"user_id":    12345,
		"email":      "user@example.com",
		"timestamp":  "2024-01-15T10:30:00Z",
	}

	eventJSON, _ := json.Marshal(event)

	// Publish to fanout exchange
	// The routing key can be anything (or empty), it's ignored
	err = ch.Publish(
		"user_events", // exchange
		"",            // routing key (ignored in fanout)
		false,         // mandatory
		false,         // immediate
		amqp.Publishing{
			ContentType: "application/json",
			Body:        eventJSON,
		},
	)

	if err != nil {
		log.Fatalf("Failed to publish: %v", err)
	}

	log.Println("Event broadcasted to all services")
}

/*
Fanout Flow:

                            Fanout Exchange
                            (user_events)
                                  │
                                  │ (Ignores routing key)
                                  │
        ┌─────────────────────────┼─────────────────────────┐
        │                         │                         │
        ▼                         ▼                         ▼
  email_service_queue     analytics_queue          audit_log_queue
        │                         │                         │
        ▼                         ▼                         ▼
  Send welcome email     Track registration       Log for compliance
        
        
All queues receive the SAME message simultaneously
*/
```

### Headers Exchange: Attribute-Based Routing

A headers exchange routes messages based on message header attributes rather than routing keys. When binding a queue to a headers exchange, you specify a set of headers that must match. You can also specify whether all headers must match (x-match: all) or any header can match (x-match: any).

This is useful for complex routing logic where the routing decision depends on multiple criteria. For example, routing image processing tasks based on image format, size, and priority.

```go
package main

import (
	"log"
	"github.com/streadway/amqp"
)

// HeadersExchangeExample demonstrates routing based on message headers
func HeadersExchangeExample() {
	conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
	if err != nil {
		log.Fatalf("Failed to connect: %v", err)
	}
	defer conn.Close()

	ch, err := conn.Channel()
	if err != nil {
		log.Fatalf("Failed to open channel: %v", err)
	}
	defer ch.Close()

	// Declare headers exchange
	err = ch.ExchangeDeclare(
		"image_processing", // exchange name
		"headers",          // exchange type
		true,               // durable
		false,              // auto-deleted
		false,              // internal
		false,              // no-wait
		nil,                // arguments
	)
	if err != nil {
		log.Fatalf("Failed to declare exchange: %v", err)
	}

	// Create queues for different processing types
	jpegQueue, _ := ch.QueueDeclare("jpeg_processor_queue", true, false, false, false, nil)
	pngQueue, _ := ch.QueueDeclare("png_processor_queue", true, false, false, false, nil)
	priorityQueue, _ := ch.QueueDeclare("priority_processor_queue", true, false, false, false, nil)

	// Bind jpeg queue - must match format=jpeg
	ch.QueueBind(
		jpegQueue.Name,
		"",
		"image_processing",
		false,
		amqp.Table{
			"x-match": "all",         // all headers must match
			"format":  "jpeg",
		},
	)

	// Bind png queue - must match format=png
	ch.QueueBind(
		pngQueue.Name,
		"",
		"image_processing",
		false,
		amqp.Table{
			"x-match": "all",
			"format":  "png",
		},
	)

	// Bind priority queue - must match priority=high (any format)
	ch.QueueBind(
		priorityQueue.Name,
		"",
		"image_processing",
		false,
		amqp.Table{
			"x-match":  "all",
			"priority": "high",
		},
	)

	// Publish JPEG image processing task
	ch.Publish(
		"image_processing",
		"",    // routing key ignored in headers exchange
		false,
		false,
		amqp.Publishing{
			ContentType: "application/json",
			Headers: amqp.Table{
				"format": "jpeg",
				"size":   "large",
			},
			Body: []byte(`{"image_id": "img_001", "operation": "resize"}`),
		},
	)

	// Publish high priority PNG task
	// This goes to BOTH png_processor_queue AND priority_processor_queue
	ch.Publish(
		"image_processing",
		"",
		false,
		false,
		amqp.Publishing{
			ContentType: "application/json",
			Headers: amqp.Table{
				"format":   "png",
				"priority": "high",
			},
			Body: []byte(`{"image_id": "img_002", "operation": "thumbnail"}`),
		},
	)

	log.Println("Headers exchange messages published")
}

/*
Headers Matching Logic:

Message Headers: {format: "jpeg", size: "large"}
Binding: {x-match: "all", format: "jpeg"} → MATCH (format matches)
Binding: {x-match: "all", format: "png"} → NO MATCH (format doesn't match)
Binding: {x-match: "all", priority: "high"} → NO MATCH (priority not in message)

Message Headers: {format: "png", priority: "high"}
Binding: {x-match: "all", format: "png"} → MATCH
Binding: {x-match: "all", priority: "high"} → MATCH
(This message goes to TWO queues)

With x-match: "any", matching ANY header would route the message
*/
```

### Setting Up the RabbitMQ Connection Manager

First, we need a robust connection manager that handles connections, reconnections, and provides channels for our producers and consumers.

```go
// rabbitmq/connection.go
package rabbitmq

import (
	"fmt"
	"log"
	"time"

	"github.com/streadway/amqp"
)

// RabbitMQ holds our connection and channel
type RabbitMQ struct {
	Connection *amqp.Connection
	Channel    *amqp.Channel
	URI        string
}

// NewRabbitMQ creates a new RabbitMQ connection
// This function establishes a connection to RabbitMQ and creates a channel
// The URI format is: amqp://username:password@hostname:port/
func NewRabbitMQ(uri string) (*RabbitMQ, error) {
	// Connect to RabbitMQ server
	// This creates a TCP connection to the RabbitMQ broker
	conn, err := amqp.Dial(uri)
	if err != nil {
		return nil, fmt.Errorf("failed to connect to RabbitMQ: %v", err)
	}

	// Create a channel
	// Channels are virtual connections inside the TCP connection
	// They allow multiple operations without creating new TCP connections
	ch, err := conn.Channel()
	if err != nil {
		conn.Close()
		return nil, fmt.Errorf("failed to open channel: %v", err)
	}

	// Set quality of service (QoS)
	// Prefetch count of 1 means: give me only 1 message at a time
	// This ensures fair distribution among multiple consumers
	// A consumer must acknowledge a message before getting the next one
	err = ch.Qos(
		1,     // prefetch count - number of unacknowledged messages
		0,     // prefetch size - in bytes, 0 means no limit
		false, // global - false means apply to this channel only
	)
	if err != nil {
		ch.Close()
		conn.Close()
		return nil, fmt.Errorf("failed to set QoS: %v", err)
	}

	rmq := &RabbitMQ{
		Connection: conn,
		Channel:    ch,
		URI:        uri,
	}

	log.Println("Successfully connected to RabbitMQ")
	return rmq, nil
}

// Close cleanly shuts down the connection
func (r *RabbitMQ) Close() {
	if r.Channel != nil {
		r.Channel.Close()
	}
	if r.Connection != nil {
		r.Connection.Close()
	}
	log.Println("RabbitMQ connection closed")
}

// DeclareImageProcessingInfrastructure sets up exchanges and queues
// This creates the entire messaging topology for our application
func (r *RabbitMQ) DeclareImageProcessingInfrastructure() error {
	// Declare the main topic exchange
	// Topic exchange allows flexible routing with pattern matching
	err := r.Channel.ExchangeDeclare(
		"image_tasks",  // exchange name
		"topic",        // exchange type - allows pattern-based routing
		true,           // durable - survives broker restart
		false,          // auto-deleted - no, we want it to persist
		false,          // internal - no, external producers can publish
		false,          // no-wait - wait for server confirmation
		nil,            // arguments - no additional arguments
	)
	if err != nil {
		return fmt.Errorf("failed to declare exchange: %v", err)
	}

	// Declare queues for different image processing tasks
	queues := []struct {
		name       string
		routingKey string
	}{
		{"resize_queue", "image.resize"},
		{"thumbnail_queue", "image.thumbnail"},
		{"convert_queue", "image.convert"},
	}

	for _, q := range queues {
		// Declare the queue
		// Durable queues survive broker restarts
		queue, err := r.Channel.QueueDeclare(
			q.name, // queue name
			true,   // durable - queue survives restart
			false,  // auto-delete - no
			false,  // exclusive - no, multiple connections can use it
			false,  // no-wait
			nil,    // arguments
		)
		if err != nil {
			return fmt.Errorf("failed to declare queue %s: %v", q.name, err)
		}

		// Bind the queue to the exchange with a routing key
		// This creates the routing rule: messages with routing key "image.resize"
		// go to "resize_queue"
		err = r.Channel.QueueBind(
			queue.Name,     // queue name
			q.routingKey,   // routing key - pattern to match
			"image_tasks",  // exchange name
			false,          // no-wait
			nil,            // arguments
		)
		if err != nil {
			return fmt.Errorf("failed to bind queue %s: %v", q.name, err)
		}

		log.Printf("Queue '%s' declared and bound with routing key '%s'", q.name, q.routingKey)
	}

	return nil
}
```

```go
type Producer struct {
	channel *amqp.Channel
}

// NewProducer creates a new producer
func NewProducer(channel *amqp.Channel) *Producer {
	return &Producer{channel: channel}
}

// PublishTask publishes an image processing task to RabbitMQ
// The routing key determines which queue receives the task
func (p *Producer) PublishTask(task ImageTask) error {
	// Convert the task to JSON
	// This serializes our Go struct into a JSON byte array
	taskJSON, err := json.Marshal(task)
	if err != nil {
		return fmt.Errorf("failed to marshal task: %v", err)
	}

	// Determine routing key based on operation
	// This maps our business logic (operation type) to routing keys
	routingKey := fmt.Sprintf("image.%s", task.Operation)

	// Publish the message
	err = p.channel.Publish(
		"image_tasks", // exchange name - where to send the message
		routingKey,    // routing key - determines which queue gets it
		false,         // mandatory - if true, returns unroutable messages
		false,         // immediate - deprecated, keep false
		amqp.Publishing{
			DeliveryMode: amqp.Persistent,    // 2 = persistent, survives restart
			ContentType:  "application/json", // MIME type
			Body:         taskJSON,           // the actual message content
		},
	)

	if err != nil {
		return fmt.Errorf("failed to publish message: %v", err)
	}

	log.Printf("Published task: %s for image %s with routing key %s", 
		task.Operation, task.ImageID, routingKey)
	return nil
}
```

```go
// Consumer processes image tasks from a queue
type Consumer struct {

	queueName string
}

// NewConsumer creates a new consumer for a specific queue
func NewConsumer(channel *amqp.Channel, queueName string) *Consumer {
	return &Consumer{
		channel:   channel,
		queueName: queueName,
	}
}

// Start begins consuming messages from the queue
func (c *Consumer) Start() error {
	// Register as a consumer on the queue
	// This tells RabbitMQ to start sending messages to us
	messages, err := c.channel.Consume(
		c.queueName, // queue name
		"",          // consumer tag - empty means auto-generate
		false,       // auto-ack - FALSE means we manually acknowledge
		false,       // exclusive - allow multiple consumers
		false,       // no-local - not supported by RabbitMQ
		false,       // no-wait
		nil,         // arguments
	)
	if err != nil {
		return fmt.Errorf("failed to register consumer: %v", err)
	}

	log.Printf("Consumer started, waiting for messages on queue: %s", c.queueName)

	// Process messages in a loop
	// This runs forever, processing each message as it arrives
	for msg := range messages {
		c.handleMessage(msg)
	}

	return nil
}

// handleMessage processes a single message
func (c *Consumer) handleMessage(msg amqp.Delivery) {
	log.Printf("Received message: %s", string(msg.Body))

	// Parse the message into our ImageTask struct
	var task ImageTask
	err := json.Unmarshal(msg.Body, &task)
	if err != nil {
		log.Printf("Error parsing message: %v", err)
		// Negative acknowledgment without requeue
		// This tells RabbitMQ the message is bad and should be discarded
		msg.Nack(false, false)
		return
	}

	// Process the task based on operation type
	err = c.processTask(task)
	if err != nil {
		log.Printf("Error processing task: %v", err)
```

### How This All Works Together

This application demonstrates a complete RabbitMQ workflow in a real-world scenario. Here's how the pieces fit together:

When you run the application as a producer, it creates three different types of image processing tasks and publishes them to the "image_tasks" exchange with different routing keys. The resize task gets routing key "image.resize", the thumbnail task gets "image.thumbnail", and the convert task gets "image.convert". Because we configured our queues with these exact routing keys as bindings, each message routes to its appropriate queue.

When you run the application as a consumer, it starts three concurrent goroutines, each listening to a different queue. These consumers wait for messages, and when a message arrives, they parse it, simulate processing (in a real application, this would involve actual image manipulation libraries), and then acknowledge the message back to RabbitMQ.

The power of this architecture becomes clear when you consider scalability. You could run multiple instances of the consumer on different servers, and RabbitMQ would automatically distribute work among them. If one consumer crashes while processing a message, RabbitMQ will redeliver that message to another consumer because we're using manual acknowledgments. If you need to add a new type of image processing operation, you simply create a new queue, bind it with a new routing key, and deploy a consumer for it—without touching any existing code.

The persistent messages and durable queues ensure that even if RabbitMQ restarts, your tasks aren't lost. The QoS setting of prefetch count 1 ensures fair distribution, preventing one fast consumer from grabbing all messages while slower consumers sit idle.

This is a basic but production-ready setup. To make it more robust, you would add connection retry logic, implement dead letter queues for messages that fail repeatedly, add monitoring and metrics, implement circuit breakers for external dependencies, and possibly use RabbitMQ's built-in clustering for high availability.
