# RabbitMQ: Acknowledgements, Durability, and Prefetch - Complete Guide

## Understanding Message Acknowledgements (ACK)

Imagine you are running a pizza delivery service. When you hand a pizza to a customer, you don't just leave it at their door and walk away. You wait for them to confirm they received it, perhaps by signing a receipt or saying "thank you." Only then do you mark that delivery as complete in your system. If you simply dropped pizzas and left without confirmation, you would have no idea if they were actually received. The customer might have been away, or their door might have been locked, or they might have rejected the order. Without acknowledgment, you cannot know the true state of the delivery.

This same principle applies to message processing in RabbitMQ through acknowledgements. When RabbitMQ delivers a message to a consumer, it needs to know whether that message was successfully processed. Without this feedback mechanism, RabbitMQ cannot make intelligent decisions about what to do with messages. Should it remove the message from the queue? Should it redeliver it? Should it consider the consumer healthy or problematic? Acknowledgements provide this critical feedback loop that makes RabbitMQ reliable and fault-tolerant.

In RabbitMQ, there are three fundamental acknowledgement modes that govern how messages are handled between the broker and consumers. Understanding these modes deeply is essential for building reliable messaging systems, as choosing the wrong mode can lead to message loss, duplicate processing, or system inefficiency.

### Auto-Acknowledgement Mode: Speed Over Safety

Auto-acknowledgement mode is the simplest but most dangerous acknowledgement strategy. When you enable auto-acknowledgement, RabbitMQ considers a message successfully processed the instant it delivers that message to a consumer. The broker does not wait for any confirmation from the consumer. It simply sends the message over the network and immediately removes it from the queue, assuming everything went perfectly.

This approach is incredibly fast because there is no acknowledgement overhead. The consumer does not need to send any response back to RabbitMQ, and RabbitMQ does not need to track which messages are awaiting acknowledgement. The message delivery becomes a simple one-way operation, minimizing network traffic and broker bookkeeping.

However, this speed comes with severe reliability risks. Consider what happens if the consumer receives a message but then crashes while processing it. Maybe the consumer was resizing an image and the server lost power. Maybe it was updating a database and the connection failed. Maybe there was a bug in the processing code that caused a panic. In any of these scenarios, the message is lost forever because RabbitMQ already deleted it from the queue. There is no recovery mechanism, no retry logic, and no safety net.

Auto-acknowledgement should only be used in scenarios where message loss is acceptable. Perhaps you are processing non-critical telemetry data where losing occasional readings does not matter. Perhaps you are displaying real-time stock prices where stale data is useless anyway and losing a message just means showing the next price update instead. But for any business-critical workflow—order processing, financial transactions, user notifications, data synchronization—auto-acknowledgement is inappropriate and dangerous.

### Manual Acknowledgement Mode: Reliability Through Confirmation

Manual acknowledgement mode inverts the safety and performance tradeoff. In this mode, RabbitMQ delivers a message to a consumer and then waits. The message remains in the queue, marked as "unacknowledged" but assigned to that specific consumer. RabbitMQ will not deliver this message to any other consumer, and it will not delete the message from the queue. The broker simply waits for the consumer to send an explicit acknowledgement signal.

The consumer, meanwhile, receives the message and begins processing it. It might decode the JSON payload, query a database, call external APIs, transform data, write files, or perform any other business logic. Only after all of this processing completes successfully does the consumer send an acknowledgement back to RabbitMQ. This acknowledgement is a network message that says, "Yes, I received this message, I processed it correctly, and you can now safely delete it from the queue."

When RabbitMQ receives this acknowledgement, it removes the message from the queue permanently. The message has been successfully processed and no longer needs to exist. The system can move on to the next message.

But what happens if the consumer never sends an acknowledgement? Perhaps the consumer crashed, perhaps the network connection dropped, perhaps the processing took too long and the consumer gave up. In these cases, RabbitMQ eventually detects that the consumer has disconnected (through TCP connection monitoring). When this happens, RabbitMQ automatically requeues the unacknowledged message, making it available for delivery to another consumer. This automatic redelivery ensures that no work is lost due to consumer failures.

This creates a powerful guarantee: at-least-once delivery. Every message will be processed at least one time, even if consumers crash or networks fail. If a consumer receives a message, processes half of it, and then crashes, another consumer will receive that same message and process it again. Your application will never lose messages due to infrastructure failures.

Of course, this guarantee comes with an important implication: your message processing must be idempotent. Idempotency means that processing the same message multiple times produces the same result as processing it once. If a message says "charge the customer $50," you cannot simply charge $50 every time you receive the message, or a crash and retry would result in double-charging. Instead, you might include a unique transaction ID in the message and check whether that transaction has already been processed before charging the customer.

### Negative Acknowledgements: Handling Failures Explicitly

Manual acknowledgement mode also supports negative acknowledgements, which allow consumers to explicitly report processing failures. When a consumer encounters a message it cannot process—perhaps the data is malformed, perhaps a required external service is unavailable, perhaps the message references a resource that no longer exists—it can send a negative acknowledgement (NACK) back to RabbitMQ.

A negative acknowledgement tells RabbitMQ, "I received this message, but I could not process it successfully." The consumer can specify whether RabbitMQ should requeue the message (giving it another chance to be processed, perhaps by a different consumer or after transient issues resolve) or discard it permanently.

There is also a reject operation, which is similar to NACK but only applies to a single message, while NACK can reject multiple messages at once. Both operations serve the same fundamental purpose: giving consumers explicit control over failure handling.

These negative acknowledgements are crucial for sophisticated error handling. You might reject and requeue a message if you encounter a temporary database connection error, hoping that retrying will succeed. You might reject without requeuing if the message data is fundamentally invalid and will never be processable. You might send the rejected message to a dead letter exchange for later analysis and manual intervention.

### The Acknowledgement Workflow in Detail

Let us trace the complete lifecycle of a message under manual acknowledgement to see how all these pieces fit together. The journey begins when a producer publishes a message to an exchange. RabbitMQ routes this message to one or more queues based on bindings and routing keys. The message sits in the queue, waiting for a consumer to become available.

When a consumer connects to RabbitMQ and subscribes to the queue, RabbitMQ begins delivering messages. It selects the next undelivered message from the queue and sends it to the consumer over the network. At this moment, RabbitMQ marks the message as "unacknowledged" in its internal state. The message is still in the queue, but it is flagged as being actively processed by a specific consumer on a specific channel. RabbitMQ will not deliver this message to any other consumer unless the current consumer fails.

```
Message Lifecycle with Manual ACK:

Time  →  RabbitMQ Queue State          Consumer State              Action
─────────────────────────────────────────────────────────────────────────────
T0      [Msg1: Ready]                 Idle                        Message waiting
        [Msg2: Ready]

T1      [Msg1: Unacked→Consumer1]     Processing Msg1             RabbitMQ delivers
        [Msg2: Ready]                                             message to consumer

T2      [Msg1: Unacked→Consumer1]     Processing...               Consumer working
        [Msg2: Ready]                 (decoding, validating,      on business logic
                                       database operations)

T3      [Msg2: Ready]                 Msg1 complete              Consumer sends ACK
                                      Ready for next              RabbitMQ deletes Msg1

T4      [Msg2: Unacked→Consumer1]     Processing Msg2             Cycle repeats
```

The consumer receives the message and begins processing. It might spend milliseconds or minutes on this work, depending on the complexity of the task. RabbitMQ simply waits, tracking that this message is assigned to this consumer. During this time, if RabbitMQ detects that the consumer's TCP connection has closed (indicating a crash, network failure, or graceful shutdown), it will automatically requeue the message even if no explicit acknowledgement was sent.

After the consumer completes processing successfully, it sends an ACK message back to RabbitMQ. This is a small protocol message that includes the delivery tag, which is a unique identifier for this specific message delivery. When RabbitMQ receives the ACK, it looks up the message by its delivery tag, verifies that it is still marked as unacknowledged to this consumer, and then permanently deletes the message from the queue. The message has been successfully processed and is no longer needed.

If instead the consumer encounters an error during processing, it might send a NACK or reject message. This tells RabbitMQ that processing failed. If the NACK includes the "requeue" flag set to true, RabbitMQ will put the message back at the front of the queue (or back in its original position, depending on configuration) so another consumer can try processing it. If the requeue flag is false, RabbitMQ will either discard the message or route it to a dead letter exchange if one is configured.

This workflow provides remarkable fault tolerance. Consumer crashes do not lose messages because unacknowledged messages are automatically requeued. Network failures do not lose messages because RabbitMQ tracks unacknowledged messages until it receives confirmation. Processing errors can be handled gracefully through NACK and dead letter exchanges. The system is self-healing and resilient.

## Understanding Message Durability

While acknowledgements protect messages from being lost due to consumer failures, durability protects messages from being lost due to broker failures. Imagine you have a RabbitMQ server that crashes or loses power. When the server restarts, what happens to all the messages that were sitting in queues? By default, they are gone forever. RabbitMQ stores messages in memory for performance, and memory is erased when a server crashes or restarts. This is where message durability becomes critical.

Durability in RabbitMQ operates at multiple levels, creating a comprehensive safety system that ensures messages survive broker restarts. Each level addresses a different component of the messaging infrastructure, and all levels must work together to provide true end-to-end durability.

### Exchange Durability: Preserving Routing Rules

The first level of durability is exchange durability. An exchange is durable if it survives a broker restart. When you declare an exchange as durable, RabbitMQ writes the exchange definition to disk. This definition includes the exchange name, type (direct, topic, fanout, or headers), and any configuration parameters.

When RabbitMQ starts up after a crash or restart, it reads these exchange definitions from disk and recreates all durable exchanges exactly as they were configured before the shutdown. Any bindings between durable exchanges and durable queues are also restored from disk, ensuring that routing rules remain intact.

Without exchange durability, a broker restart would delete all exchanges except the default built-in exchanges. Any producer trying to publish messages to custom exchanges would receive errors because those exchanges no longer exist. By making exchanges durable, you ensure that your routing topology survives restarts and producers can continue operating normally.

However, exchange durability alone does not save messages. Exchanges do not store messages; they only route them to queues. Even if an exchange is durable, messages passing through that exchange will be lost on restart unless the queues themselves are also durable.

### Queue Durability: Preserving Message Storage

Queue durability is the second level of protection. A durable queue is one whose definition survives a broker restart. When you declare a queue as durable, RabbitMQ writes the queue definition to disk, including the queue name, configuration parameters, and any bindings to exchanges.

When RabbitMQ restarts, it recreates all durable queues from these disk definitions. Consumers can reconnect to these queues and continue consuming messages as if nothing happened. The queue infrastructure is preserved.

However, queue durability alone still does not save messages. A durable queue remembers that it exists and how it is configured, but by default, the messages inside the queue are still stored only in memory. If RabbitMQ crashes, those in-memory messages are lost even though the queue itself is recreated on restart.

This is a crucial distinction that often confuses newcomers to RabbitMQ. Queue durability preserves the queue structure, not the queue contents. It is like having a durable filing cabinet that survives a fire, but the papers inside the cabinet are still flammable. You need an additional layer of protection for the messages themselves.

### Message Persistence: Saving Message Content

Message persistence is the third and final level of durability. When you publish a message with the delivery mode set to persistent, RabbitMQ writes that message to disk. The message is stored in RabbitMQ's message store, which is a sophisticated on-disk structure optimized for fast sequential writes and reads.

Persistent messages in durable queues survive broker restarts. When RabbitMQ crashes and then restarts, it reads the message store from disk and reloads all persistent messages back into their respective durable queues. Consumers can then receive and process these messages as normal, even though the broker was completely restarted in the interim.

For a message to truly survive a broker restart, all three levels must be configured correctly. The exchange must be durable, the queue must be durable, and the message must be persistent. If any of these three is missing, the message can be lost.

Consider what happens with different combinations:

If the exchange is non-durable but the queue is durable and the message is persistent, the exchange will disappear on restart. The queue and message survive, but producers cannot publish new messages to the exchange because it no longer exists.

If the exchange is durable and the queue is durable but the message is non-persistent, the routing topology survives but the messages are lost. The queue exists but is empty after restart.

If the exchange and message are durable but the queue is non-durable, the queue disappears on restart. The message was written to disk, but it was written to a queue that no longer exists, so the message is effectively orphaned and lost.

Only when all three are configured for durability does the complete message survive from producer to consumer even through broker restarts.

### The Performance Cost of Durability

Durability provides safety, but it comes with a performance cost. Writing messages to disk is orders of magnitude slower than keeping them in memory. Disk I/O becomes the bottleneck in a durable messaging system, potentially reducing throughput from hundreds of thousands of messages per second to a few thousand.

RabbitMQ mitigates this cost through several optimizations. It batches disk writes, grouping multiple messages together and writing them in a single disk operation. It uses operating system page cache to reduce actual physical disk writes. It performs asynchronous writes, acknowledging messages before they have been fully fsynced to disk in some configurations.

However, these optimizations create tradeoffs. Asynchronous writes improve performance but create a small window where a message might be acknowledged to the producer before it is safely on disk. If the server crashes in this narrow window, the message could be lost even though the producer believes it was safely stored. For maximum safety, you can configure RabbitMQ to wait for fsync before acknowledging, but this significantly reduces throughput.

The durability decision ultimately depends on your application requirements. If you are processing financial transactions or critical business data, durability is non-negotiable despite the performance cost. If you are processing high-volume telemetry or real-time analytics where occasional message loss is acceptable, you might choose speed over durability. Most applications fall somewhere in between, using durability for critical messages and non-durability for less important ones.

## Understanding Prefetch and Quality of Service (QoS)

Prefetch, also known as Quality of Service or QoS, is a mechanism that controls how many unacknowledged messages a consumer can have at any given time. It is a flow control system that prevents consumers from being overwhelmed with too many messages while also ensuring efficient utilization of consumer capacity.

To understand why prefetch is necessary, imagine a scenario without any prefetch limit. RabbitMQ would deliver every message in a queue to a consumer as fast as the network allows. If a queue contains ten thousand messages and a consumer connects, RabbitMQ might push all ten thousand messages to that consumer immediately, flooding its network buffer and memory. The consumer would struggle to process this avalanche of messages, potentially running out of memory or becoming unresponsive.

Even worse, consider a scenario with multiple consumers of different speeds connected to the same queue. Without prefetch limits, when RabbitMQ distributes messages, it uses a simple round-robin algorithm. If you have three consumers—one very fast, one medium speed, and one very slow—RabbitMQ would give each consumer one-third of the messages. The fast consumer would finish its third quickly and then sit idle while the slow consumer struggles with its third. This is inefficient load balancing because fast consumers are underutilized while slow consumers are overloaded.

### How Prefetch Works

Prefetch count is a number you set on a channel that tells RabbitMQ, "Do not deliver more than N unacknowledged messages to this consumer at one time." When you set a prefetch count of 1, for example, RabbitMQ will deliver one message to the consumer and then wait. It will not deliver a second message until the consumer acknowledges the first one. When you set a prefetch count of 10, RabbitMQ can deliver up to ten messages before pausing and waiting for acknowledgements.

This creates intelligent flow control. If a consumer is processing messages quickly, it will acknowledge them quickly, and RabbitMQ will keep delivering new messages to maintain the prefetch count. The fast consumer stays busy. If a consumer is processing messages slowly, it will acknowledge them slowly, and RabbitMQ will deliver new messages less frequently. The slow consumer is not overwhelmed.

When multiple consumers are connected to the same queue with prefetch limits, RabbitMQ achieves fair load balancing. It continuously delivers messages to whichever consumers have room in their prefetch window. Fast consumers that acknowledge quickly will receive more messages. Slow consumers that acknowledge slowly will receive fewer messages. Each consumer works at its own optimal pace, and no consumer sits idle while others are overloaded.

```
Prefetch Behavior with Multiple Consumers:

Without Prefetch (Round-Robin):
─────────────────────────────────
Consumer 1 (Fast):    [1][2][3] → Done → Idle → Idle → Idle
Consumer 2 (Medium):  [4][5][6] → Processing → Done → Idle
Consumer 3 (Slow):    [7][8][9] → Processing → Processing → Processing

Result: Consumer 1 finishes quickly and sits idle while Consumer 3 struggles


With Prefetch Count = 1:
─────────────────────────────────
Consumer 1 (Fast):    [1] ACK → [4] ACK → [7] ACK → [10] ACK
Consumer 2 (Medium):  [2] ACK → [5] ACK → [8] ACK
Consumer 3 (Slow):    [3] .... still processing [3]

Result: Fast consumers process more messages, achieving better overall throughput
```

### Choosing the Right Prefetch Count

Selecting an appropriate prefetch count requires understanding your consumer's processing characteristics and your reliability requirements. A prefetch count of 1 provides maximum fairness and the fastest redelivery on consumer failure, but it may not provide maximum throughput because the consumer might have to wait for network round-trips between messages.

A higher prefetch count like 10 or 20 can improve throughput by reducing network round-trips and keeping consumers continuously busy, but it means that if a consumer crashes, up to 10 or 20 messages might need to be redelivered. There is also more memory usage because the consumer and RabbitMQ both need to track more unacknowledged messages.

For CPU-intensive processing where each message takes significant time (seconds or minutes), a prefetch count of 1 is often ideal. The processing time dominates the network round-trip time, so there is no benefit to buffering multiple messages, and you get the fastest failure recovery.

For I/O-intensive processing where each message involves waiting for external services or databases, a higher prefetch count might be beneficial. The consumer can process multiple messages concurrently or pipeline them efficiently, reducing idle time waiting for I/O.

For high-throughput, low-latency processing where each message is processed in milliseconds, a moderate prefetch count balances throughput and fairness. You want enough buffering to avoid network delays but not so much that slow consumers get overloaded.

### Global QoS vs Channel QoS

RabbitMQ allows you to set QoS at two different scopes: per-channel or globally per connection. Channel-level QoS (which is the default and most common) applies the prefetch limit to each consumer channel independently. If you have two channels in your application each consuming from different queues, each channel can have its own prefetch limit.

Global QoS applies the prefetch limit across all consumers on the entire connection, regardless of how many channels they use. This is less commonly used but can be important in applications that create many channels and want to limit total outstanding messages across all of them.

		ExchangeDurable: true,
		QueueDurable:    true,
		MessagePersist:  true,
		
		// Manual acknowledgement for reliability
		AutoAck:     false,
		Exclusive:   false,
		ConsumerTag: "reliable-consumer",
	}
}

// Validate checks if configuration is valid
func (c *RabbitMQConfig) Validate() error {
	if c.URI == "" {
		return fmt.Errorf("RabbitMQ URI cannot be empty")
	}
	if c.ExchangeName == "" {
		return fmt.Errorf("exchange name cannot be empty")
	}
	if c.QueueName == "" {
		return fmt.Errorf("queue name cannot be empty")
	}
	if c.PrefetchCount < 0 {
		return fmt.Errorf("prefetch count must be non-negative")
	}
	return nil
}

### Connection Management with Auto-Recovery

```go
// rabbitmq/connection.go
package rabbitmq

import (
	"fmt"
	"log"
	"sync"
	"time"

	"rabbitmq-reliable-service/config"
	"github.com/streadway/amqp"
)

// Connection manages RabbitMQ connection with automatic recovery
// This is the foundation of our reliable messaging system
type Connection struct {
	config      *config.RabbitMQConfig
	conn        *amqp.Connection
	channel     *amqp.Channel
	
	// Channels for monitoring connection health
	connErr     chan *amqp.Error
	channelErr  chan *amqp.Error
	
	// Control channels
	done        chan bool
	
	// Mutex for thread-safe operations
	mu          sync.RWMutex
	
	// State tracking
	isConnected bool
}

// NewConnection creates a new connection manager
func NewConnection(cfg *config.RabbitMQConfig) (*Connection, error) {
	if err := cfg.Validate(); err != nil {
		return nil, fmt.Errorf("invalid config: %w", err)
	}

	c := &Connection{
		config: cfg,
		done:   make(chan bool),
	}

	// Establish initial connection
	if err := c.connect(); err != nil {
		return nil, err
	}

	// Start connection monitor in background
	go c.monitorConnection()

	return c, nil
}

// connect establishes a connection to RabbitMQ
// This is called initially and after any disconnection
func (c *Connection) connect() error {
	c.mu.Lock()
	defer c.mu.Unlock()

	var err error
	
	// Close existing connection if any
	if c.conn != nil && !c.conn.IsClosed() {
		c.conn.Close()
	}

	log.Printf("Connecting to RabbitMQ at %s...", c.config.URI)
	
	// Establish TCP connection to RabbitMQ
	c.conn, err = amqp.Dial(c.config.URI)
	if err != nil {
		return fmt.Errorf("failed to connect: %w", err)
	}

	// Create a channel for operations
	c.channel, err = c.conn.Channel()
	if err != nil {
		c.conn.Close()
		return fmt.Errorf("failed to open channel: %w", err)
	}

	// Set up Quality of Service (Prefetch)
	// This is CRITICAL for proper load balancing and preventing consumer overload
	// prefetchCount: maximum number of unacknowledged messages
	// prefetchSize: maximum total size in bytes (0 = no limit)
	// global: false = apply per consumer, true = apply per connection
	err = c.channel.Qos(
		c.config.PrefetchCount, // prefetch count
		0,                      // prefetch size
		false,                  // global
	)
	if err != nil {
		c.channel.Close()
		c.conn.Close()
		return fmt.Errorf("failed to set QoS: %w", err)
	}

	// Declare exchange with durability settings
	// Durable exchange survives broker restart
	err = c.channel.ExchangeDeclare(
		c.config.ExchangeName,   // name
		c.config.ExchangeType,   // type
		c.config.ExchangeDurable, // durable - survives restart
		false,                   // auto-deleted - no
		false,                   // internal - no
		false,                   // no-wait
		nil,                     // arguments
	)
	if err != nil {
		c.channel.Close()
		c.conn.Close()
		return fmt.Errorf("failed to declare exchange: %w", err)
	}

	// Declare queue with durability settings
	// Durable queue survives broker restart
	queue, err := c.channel.QueueDeclare(
		c.config.QueueName,    // name
		c.config.QueueDurable, // durable - survives restart
		false,                 // auto-delete - no
		c.config.Exclusive,    // exclusive - no
		false,                 // no-wait
		nil,                   // arguments
	)
	if err != nil {
		c.channel.Close()
		c.conn.Close()
		return fmt.Errorf("failed to declare queue: %w", err)
	}

	// Bind queue to exchange
	err = c.channel.QueueBind(
		queue.Name,              // queue name
		c.config.RoutingKey,     // routing key
		c.config.ExchangeName,   // exchange
		false,                   // no-wait
		nil,                     // arguments
	)
	if err != nil {
		c.channel.Close()
		c.conn.Close()
		return fmt.Errorf("failed to bind queue: %w", err)
	}

	// Set up error notification channels
	// These channels receive errors when connection/channel close
	c.connErr = make(chan *amqp.Error)
	c.channelErr = make(chan *amqp.Error)
	c.conn.NotifyClose(c.connErr)
	c.channel.NotifyClose(c.channelErr)

	c.isConnected = true
	log.Printf("Successfully connected to RabbitMQ")
	log.Printf("Exchange: %s (durable: %v)", c.config.ExchangeName, c.config.ExchangeDurable)
	log.Printf("Queue: %s (durable: %v)", c.config.QueueName, c.config.QueueDurable)
	log.Printf("Prefetch: %d messages", c.config.PrefetchCount)

	return nil
}

// monitorConnection watches for connection failures and reconnects
// This runs in a background goroutine throughout the application lifetime
func (c *Connection) monitorConnection() {
	for {
		select {
		case <-c.done:
			log.Println("Connection monitor shutting down")
			return
			
		case err := <-c.connErr:
			if err != nil {
				log.Printf("Connection error: %v", err)
				c.reconnect()
			}
			
		case err := <-c.channelErr:
			if err != nil {
				log.Printf("Channel error: %v", err)
				c.reconnect()
			}
		}
	}
}

// reconnect attempts to reconnect with exponential backoff
func (c *Connection) reconnect() {
	c.mu.Lock()
	c.isConnected = false
	c.mu.Unlock()

	for attempt := 1; attempt <= c.config.MaxReconnectAttempts; attempt++ {
		log.Printf("Reconnection attempt %d/%d", attempt, c.config.MaxReconnectAttempts)
		
		err := c.connect()
		if err == nil {
			log.Printf("Reconnection successful")
			return
		}

		log.Printf("Reconnection failed: %v", err)
		
		// Exponential backoff: wait longer between each attempt
		backoff := c.config.ReconnectDelay * time.Duration(attempt)
		log.Printf("Waiting %v before next attempt...", backoff)
		time.Sleep(backoff)
	}

	log.Printf("Failed to reconnect after %d attempts", c.config.MaxReconnectAttempts)
}

// IsConnected returns current connection state
func (c *Connection) IsConnected() bool {
	c.mu.RLock()
	defer c.mu.RUnlock()
	return c.isConnected
}

// GetChannel returns the current channel
// Caller should check IsConnected before using
func (c *Connection) GetChannel() *amqp.Channel {
	c.mu.RLock()
	defer c.mu.RUnlock()
	return c.channel
}

// Close gracefully shuts down the connection
func (c *Connection) Close() {
	close(c.done)
	
	c.mu.Lock()
	defer c.mu.Unlock()
	
	if c.channel != nil {
		c.channel.Close()
	}
	if c.conn != nil {
		c.conn.Close()
	}
	
	c.isConnected = false
	log.Println("RabbitMQ connection closed")
}
```

// NewProducer creates a new reliable producer
func NewProducer(conn *Connection, cfg *config.RabbitMQConfig) (*Producer, error) {
	// Enable publisher confirms for reliability
	// This makes RabbitMQ acknowledge each published message
	channel := conn.GetChannel()
	if err := channel.Confirm(false); err != nil {
		return nil, fmt.Errorf("failed to enable publisher confirms: %w", err)
	}

	return &Producer{
		conn:   conn,
		config: cfg,
	}, nil
}

// PublishWithConfirm publishes a message and waits for confirmation
// This ensures the message was received and routed by RabbitMQ
func (p *Producer) PublishWithConfirm(ctx context.Context, message interface{}) error {
	// Check connection health
	if !p.conn.IsConnected() {
		return fmt.Errorf("not connected to RabbitMQ")
	}

	// Serialize message to JSON
	body, err := json.Marshal(message)
	if err != nil {
		return fmt.Errorf("failed to marshal message: %w", err)
	}

	// Get the channel
	channel := p.conn.GetChannel()

	// Create a channel to receive publisher confirmation
	confirms := channel.NotifyPublish(make(chan amqp.Confirmation, 1))

	// Determine delivery mode based on persistence setting
	// DeliveryMode 1 = non-persistent (memory only)
	// DeliveryMode 2 = persistent (written to disk)
	deliveryMode := uint8(1)
	if p.config.MessagePersist {
		deliveryMode = 2
	}

	// Publish the message
	err = channel.PublishWithContext(
		ctx,
		p.config.ExchangeName, // exchange
		p.config.RoutingKey,   // routing key
		false,                 // mandatory - return if unroutable
		false,                 // immediate - deprecated, always false
		amqp.Publishing{
			DeliveryMode: deliveryMode,
			ContentType:  "application/json",
			Body:         body,
			Timestamp:    time.Now(),
			// Add additional metadata
			MessageId:    generateMessageID(),
			AppId:        "reliable-service",
		},
	)
	if err != nil {
		return fmt.Errorf("failed to publish: %w", err)
	}

	// Wait for confirmation from RabbitMQ
	// This confirms the message was received and routed
	select {
	case confirm := <-confirms:
		if !confirm.Ack {
			return fmt.Errorf("message was not confirmed by broker")
		}
		log.Printf("Message confirmed by broker (delivery tag: %d)", confirm.DeliveryTag)
		return nil
		
	case <-ctx.Done():
		return fmt.Errorf("publish confirmation timeout: %w", ctx.Err())
		
	case <-time.After(5 * time.Second):
		return fmt.Errorf("publish confirmation timeout after 5 seconds")
	}
}

// PublishBatch publishes multiple messages efficiently
// Uses a single channel and batches confirms
func (p *Producer) PublishBatch(ctx context.Context, messages []interface{}) error {
	if !p.conn.IsConnected() {
		return fmt.Errorf("not connected to RabbitMQ")
	}

	if len(messages) == 0 {
		return nil
	}

	channel := p.conn.GetChannel()
	confirms := channel.NotifyPublish(make(chan amqp.Confirmation, len(messages)))

	deliveryMode := uint8(1)
	if p.config.MessagePersist {
		deliveryMode = 2
	}

	// Publish all messages
	for _, message := range messages {
		body, err := json.Marshal(message)
		if err != nil {
			return fmt.Errorf("failed to marshal message: %w", err)
		}

		err = channel.PublishWithContext(
			ctx,
			p.config.ExchangeName,
			p.config.RoutingKey,
			false,
			false,
			amqp.Publishing{
				DeliveryMode: deliveryMode,
				ContentType:  "application/json",
				Body:         body,
				Timestamp:    time.Now(),
				MessageId:    generateMessageID(),
			},
		)
		if err != nil {
			return fmt.Errorf("failed to publish message: %w", err)
		}
	}

	// Wait for all confirmations
	confirmed := 0
	for confirmed < len(messages) {
		select {
		case confirm := <-confirms:
			if !confirm.Ack {
				return fmt.Errorf("message %d was not confirmed", confirmed)
			}
			confirmed++
			
		case <-ctx.Done():
			return fmt.Errorf("batch publish timeout: %w", ctx.Err())
			
		case <-time.After(10 * time.Second):
			return fmt.Errorf("batch publish timeout after 10 seconds")
		}
	}

	log.Printf("Successfully published and confirmed %d messages", len(messages))
	return nil
}

// generateMessageID creates a unique message identifier
func generateMessageID() string {
	return fmt.Sprintf("%d", time.Now().UnixNano())
}

import (

// MessageHandler is a function that processes messages
// It returns an error if processing failed
type MessageHandler func(ctx context.Context, body []byte) error

// Consumer consumes messages with proper acknowledgement handling
type Consumer struct {
	conn    *Connection
	config  *config.RabbitMQConfig
	handler MessageHandler
}

// NewConsumer creates a new reliable consumer
func NewConsumer(conn *Connection, cfg *config.RabbitMQConfig, handler MessageHandler) *Consumer {
	return &Consumer{
		conn:    conn,
		config:  cfg,
		handler: handler,
	}
}

// Start begins consuming messages
// This runs until the context is cancelled or an unrecoverable error occurs
func (c *Consumer) Start(ctx context.Context) error {
	for {
		// Check if context is cancelled
		select {
		case <-ctx.Done():
			log.Println("Consumer stopping due to context cancellation")
			return ctx.Err()
		default:
		}

		// Check connection health
		if !c.conn.IsConnected() {
			log.Println("Not connected, waiting for reconnection...")
			time.Sleep(2 * time.Second)
			continue
		}

		// Start consuming
		if err := c.consume(ctx); err != nil {
			log.Printf("Consume error: %v, will retry...", err)
			time.Sleep(2 * time.Second)
		}
	}
}

// consume is the main consumption loop
func (c *Consumer) consume(ctx context.Context) error {
	channel := c.conn.GetChannel()

	// Register as a consumer
	// auto-ack is FALSE - we will manually acknowledge messages
	// This is CRITICAL for reliability
	messages, err := channel.Consume(
		c.config.QueueName,    // queue
		c.config.ConsumerTag,  // consumer tag
		c.config.AutoAck,      // auto-ack (should be false for reliability)
		c.config.Exclusive,    // exclusive
		false,                 // no-local (not supported)
		false,                 // no-wait
		nil,                   // args
	)
	if err != nil {
		return fmt.Errorf("failed to register consumer: %w", err)
	}

	log.Printf("Consumer started, waiting for messages...")

	// Process messages
	for {
		select {
		case <-ctx.Done():
			return ctx.Err()

		case msg, ok := <-messages:
			if !ok {
				// Channel closed, need to reconnect
				return fmt.Errorf("message channel closed")
			}
			
			// Process the message
			c.processMessage(ctx, msg)
		}
	}
}

// processMessage handles a single message with proper ACK/NACK
func (c *Consumer) processMessage(ctx context.Context, msg amqp.Delivery) {
	startTime := time.Now()
	
	log.Printf("Processing message: delivery_tag=%d, redelivered=%v",
		msg.DeliveryTag, msg.Redelivered)

	// Create a timeout context for message processing
	// This prevents a single message from blocking forever
	processCtx, cancel := context.WithTimeout(ctx, 30*time.Second)
	defer cancel()

	// Call the handler function
	err := c.handler(processCtx, msg.Body)
	
	processingTime := time.Since(startTime)

	if err != nil {
		// Processing failed
		c.handleProcessingError(msg, err, processingTime)
	} else {
		// Processing succeeded
		c.handleProcessingSuccess(msg, processingTime)
	}
}

// handleProcessingSuccess acknowledges successful processing
func (c *Consumer) handleProcessingSuccess(msg amqp.Delivery, duration time.Duration) {
	// Send positive acknowledgement to RabbitMQ
	// multiple = false means acknowledge only this message
	// multiple = true would acknowledge this and all prior messages
	err := msg.Ack(false)
	if err != nil {
		log.Printf("Failed to ACK message %d: %v", msg.DeliveryTag, err)
		// This is serious - we processed the message but couldn't acknowledge
		// The message will be redelivered, leading to duplicate processing
		// In production, you might want to log this to a monitoring system
		return
	}

	log.Printf("Message %d processed successfully in %v", msg.DeliveryTag, duration)
}

// handleProcessingError handles processing failures with intelligent retry logic
func (c *Consumer) handleProcessingError(msg amqp.Delivery, err error, duration time.Duration) {
	log.Printf("Failed to process message %d: %v (took %v)", msg.DeliveryTag, err, duration)

	// Check if this message has been redelivered multiple times
	// x-death header contains redelivery information
	redeliveryCount := c.getRedeliveryCount(msg)
	
	// Decide whether to requeue based on error type and redelivery count
	shouldRequeue := c.shouldRequeue(err, redeliveryCount, msg.Redelivered)

	if shouldRequeue {
		// Send negative acknowledgement with requeue
		// RabbitMQ will put the message back in the queue
		// Another consumer (or this one) will try again
		log.Printf("Requeuing message %d (redelivery count: %d)", msg.DeliveryTag, redeliveryCount)
		
		if err := msg.Nack(false, true); err != nil {
			log.Printf("Failed to NACK message %d: %v", msg.DeliveryTag, err)
		}
	} else {
		// Send negative acknowledgement without requeue
		// This discards the message (or sends to dead letter exchange if configured)
		log.Printf("Rejecting message %d without requeue (redelivery count: %d)",
			msg.DeliveryTag, redeliveryCount)
		
		if err := msg.Nack(false, false); err != nil {
			log.Printf("Failed to NACK message %d: %v", msg.DeliveryTag, err)
		}
		
		// In production, you would log this to a dead letter queue or error tracking system
		c.logToDeadLetterQueue(msg, err)
	}
}

// shouldRequeue determines if a message should be requeued
// This implements a retry policy with maximum attempts
func (c *Consumer) shouldRequeue(err error, redeliveryCount int, redelivered bool) bool {
	// Maximum retry attempts
	maxRetries := 3

	// Don't retry if we've exceeded max attempts
	if redeliveryCount >= maxRetries {
		log.Printf("Message exceeded max retries (%d), will not requeue", maxRetries)
		return false
	}

	// Check error type - some errors should not be retried
	// For example, JSON parsing errors won't be fixed by retrying
	if isUnrecoverableError(err) {
		log.Printf("Unrecoverable error detected, will not requeue")
		return false
	}

	// For recoverable errors (DB connection, network issues), requeue
	return true
}

// getRedeliveryCount extracts redelivery count from message headers
func (c *Consumer) getRedeliveryCount(msg amqp.Delivery) int {
	if msg.Headers == nil {
		return 0
	}

	// Check x-death header which RabbitMQ adds when messages are dead-lettered
	if xDeath, ok := msg.Headers["x-death"].([]interface{}); ok && len(xDeath) > 0 {
		if death, ok := xDeath[0].(amqp.Table); ok {
			if count, ok := death["count"].(int64); ok {
				return int(count)
			}
		}
	}

	// Fallback: if message is redelivered at least once, count is at least 1
	if msg.Redelivered {
		return 1
	}

	return 0
}

// isUnrecoverableError determines if an error is worth retrying
func isUnrecoverableError(err error) bool {
	// In a real application, you would check specific error types
	// For example:
	// - JSON unmarshal errors
	// - Validation errors
	// - Business logic errors that won't change on retry
	
	// This is a simplified example
	errorStr := err.Error()
	
	// Don't retry JSON errors
	if contains(errorStr, "json") || contains(errorStr, "unmarshal") {
		return true
	}
	
	// Don't retry validation errors
	if contains(errorStr, "validation") || contains(errorStr, "invalid") {
		return true
	}
	
	return false
}

// logToDeadLetterQueue logs failed messages for later analysis
func (c *Consumer) logToDeadLetterQueue(msg amqp.Delivery, err error) {
	// In production, you would:
	// 1. Publish to a dead letter exchange
	// 2. Log to a monitoring system (Sentry, DataDog, etc.)
	// 3. Store in a database for later retry
	// 4. Send alerts if failure rate is high
	
	log.Printf("DEAD LETTER: delivery_tag=%d, error=%v, body=%s",
		msg.DeliveryTag, err, string(msg.Body))
}

// Helper function
func contains(s, substr string) bool {
	return len(s) >= len(substr) && (s == substr || 
		(len(s) > len(substr) && 
			(s[:len(substr)] == substr || s[len(s)-len(substr):] == substr ||
				containsInner(s, substr))))
}

func containsInner(s, substr string) bool {
	for i := 0; i <= len(s)-len(substr); i++ {
		if s[i:i+len(substr)] == substr {
			return true
		}
	}
	return false
}

## Summary: How It All Works Together

This production-grade implementation demonstrates all three critical concepts working in harmony.

**Acknowledgements** are handled through manual ACK mode. When a consumer receives a message, it processes it completely and only then sends an acknowledgement. If processing fails, the consumer sends a NACK with intelligent retry logic based on error type and redelivery count. If the consumer crashes before acknowledging, RabbitMQ automatically requeues the message for another consumer. This provides at-least-once delivery guarantees.

**Durability** is configured at all three levels. Exchanges are declared as durable, so they survive broker restarts. Queues are declared as durable, so their definitions survive restarts. Messages are published with DeliveryMode 2 (persistent), so their content is written to disk. Together, these three settings ensure that messages survive broker crashes and restarts, providing true persistence.

**Prefetch** is set to 1, meaning each consumer receives only one unacknowledged message at a time. This provides maximum fairness and fastest failure recovery. When a consumer finishes processing and acknowledges a message, RabbitMQ immediately delivers the next message. Fast consumers process more messages, slow consumers process fewer, achieving automatic load balancing without any consumer sitting idle.

The connection manager provides automatic reconnection with exponential backoff, ensuring the application recovers from network failures. The producer uses publisher confirms to ensure messages are safely received by RabbitMQ. The consumer implements sophisticated error handling with retry logic and dead letter logging.

This is a production-ready foundation that you can build upon for real-world applications, providing reliability, fault tolerance, and graceful failure handling throughout the entire message lifecycle.
