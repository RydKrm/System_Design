# Publish/Subscribe Pattern with RabbitMQ - Complete Guide

## Understanding the Publish/Subscribe Pattern

Imagine you are standing in a bustling newsroom of a major newspaper. Throughout the day, journalists write articles about different topics: sports, politics, business, entertainment, and weather. Each article is published to the newsroom's central board. Now, imagine there are various departments in the building who are interested in different types of news. The sports section editor wants to see all sports articles, the business section editor wants all business articles, and the general news editor wants to see everything. Some departments might want to see multiple categories—perhaps the weekend supplement editor wants both entertainment and sports articles.

In this scenario, when a journalist publishes an article, they simply post it to the central board with a category tag. They do not need to know who will read it, how many people will read it, or what those readers will do with the information. The journalist's job is simply to publish the article. Meanwhile, each editor has subscribed to specific categories they care about. When an article matching their subscription appears on the board, they automatically receive a copy. Multiple editors might receive the same article if they have overlapping subscriptions, or an article might reach no one if nobody has subscribed to that category. The key insight is that publishers and subscribers are completely decoupled from each other—they do not know about each other's existence.

This is the essence of the Publish/Subscribe pattern, often abbreviated as Pub/Sub. It is a messaging pattern where senders of messages, called publishers, do not send messages directly to specific receivers. Instead, publishers categorize messages and send them to a message broker. The broker then distributes these messages to all interested parties, called subscribers, who have expressed interest in receiving messages of particular categories. The critical characteristic that defines Pub/Sub is that a single published message can be received by multiple subscribers simultaneously. This is fundamentally different from a queue-based system where each message is consumed by exactly one consumer.

### The Philosophy Behind Publish/Subscribe

To truly understand Publish/Subscribe, we need to understand the problem it solves and the philosophy that drives its design. In traditional point-to-point communication, if you want to send a message to three different recipients, you must explicitly send three separate messages, knowing the address or identity of each recipient. This creates tight coupling—the sender must know who the receivers are, must handle each receiver individually, and must deal with the complexity of managing multiple destinations. If you want to add a fourth receiver, you must modify the sender's code. If you want to remove a receiver, you must modify the sender's code again. The sender and receivers are inextricably linked.

Publish/Subscribe inverts this relationship entirely. The sender becomes a publisher who broadcasts messages into the void without knowledge of who might be listening. The receivers become subscribers who express interest in certain types of messages without knowledge of who might be publishing them. The message broker sits in the middle, acting as the matchmaker between publishers and subscribers. This inversion creates powerful decoupling that fundamentally changes how we can architect systems.

Consider a real-world example: a stock market data feed. Thousands of companies are publishing stock price updates every second. Millions of different systems and applications need to consume this data: trading platforms, analysis tools, portfolio managers, news services, regulatory systems, historical archives, and individual investor applications. In a point-to-point model, each company publishing stock data would need to maintain connections to millions of subscribers and send each update to each subscriber individually. This would be impossibly complex and would create a massive coupling nightmare. When a new trading platform launches and wants to receive stock data, every single stock data publisher would need to be modified to add this new subscriber.

With Publish/Subscribe, stock data publishers simply broadcast price updates to a message broker, categorized by stock symbol, exchange, or other relevant criteria. Subscribers connect to the broker and subscribe to the specific stocks or categories they care about. A day trader's application might subscribe to updates for fifty specific stocks. A regulatory system might subscribe to all stocks on a particular exchange. A news service might subscribe to updates for stocks in the technology sector that show significant price movements. The publishers never know about any of these subscribers, and the subscribers never coordinate with the publishers. The broker handles all the distribution logic.

When a new subscriber appears, nothing changes for publishers—they continue publishing data exactly as before. When a publisher goes offline, existing subscribers simply stop receiving updates from that source but continue receiving updates from other publishers. This loose coupling enables systems to evolve, scale, and change independently without cascading modifications throughout the entire system.

### How Publish/Subscribe Works in RabbitMQ

RabbitMQ implements the Publish/Subscribe pattern through its exchange and binding mechanism, which provides sophisticated routing capabilities beyond simple topic-based subscription. Understanding how RabbitMQ achieves Pub/Sub requires understanding the flow of a message through the system and the role each component plays.

When a publisher wants to broadcast a message, it does not send the message to a queue directly. Instead, it sends the message to an exchange. The exchange is RabbitMQ's routing mechanism—it receives messages from publishers and distributes them to queues based on routing rules. For Publish/Subscribe scenarios, we typically use either a fanout exchange or a topic exchange. The fanout exchange is the simplest form of Pub/Sub: every message sent to the exchange is copied and delivered to every queue bound to that exchange. The topic exchange provides more sophisticated filtering, allowing subscribers to subscribe to specific patterns of messages using routing keys with wildcards.

Each subscriber creates its own queue and binds that queue to the exchange. This binding represents the subscriber's subscription—it tells RabbitMQ "I want to receive messages from this exchange." With a fanout exchange, the binding has no additional parameters because all messages go to all bound queues. With a topic exchange, the binding includes a routing key pattern that specifies which messages the subscriber wants to receive, such as "stock.NASDAQ._" to receive updates for all NASDAQ stocks, or "stock._.AAPL" to receive Apple stock updates from any exchange.

When a message arrives at the exchange, RabbitMQ evaluates all the bindings to that exchange. For a fanout exchange, RabbitMQ copies the message to every bound queue. For a topic exchange, RabbitMQ checks the message's routing key against each binding's pattern and copies the message to queues whose patterns match. This copying behavior is what makes Pub/Sub different from traditional queuing—the same message content is duplicated and delivered to multiple queues, and therefore to multiple subscribers.

```
Publish/Subscribe Message Flow:

                                Publisher
                                    │
                                    │ Publish message
                                    │ "AAPL stock price: $150"
                                    ▼
                            ┌───────────────┐
                            │   Exchange    │
                            │   (Fanout)    │
                            └───────────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    │               │               │
                    │ Copy msg      │ Copy msg      │ Copy msg
                    ▼               ▼               ▼
            ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
            │   Queue A    │ │   Queue B    │ │   Queue C    │
            │  (Trader)    │ │ (Analytics)  │ │  (Archive)   │
            └──────────────┘ └──────────────┘ └──────────────┘
                    │               │               │
                    ▼               ▼               ▼
            ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
            │ Subscriber A │ │ Subscriber B │ │ Subscriber C │
            │  (Trader)    │ │ (Analytics)  │ │  (Archive)   │
            └──────────────┘ └──────────────┘ └──────────────┘

All three subscribers receive the SAME message simultaneously
```

Each subscriber consumes messages from its own queue at its own pace. If the trading application processes messages quickly, it consumes them rapidly from Queue A. If the analytics system is slower, messages accumulate in Queue B, but this does not affect the other subscribers. If the archive system goes offline, messages accumulate in Queue C (or are discarded if the queue is not durable), but the trader and analytics systems continue receiving messages normally. This independence is another key characteristic of Pub/Sub—subscribers operate completely independently without affecting each other.

### Fanout Exchange: The Purest Form of Broadcast

The fanout exchange is the simplest and purest implementation of Publish/Subscribe. When you send a message to a fanout exchange, RabbitMQ ignores the routing key entirely and broadcasts the message to every queue bound to the exchange. This is true broadcasting—every subscriber receives every message, no filtering, no routing logic, just pure dissemination.

Fanout exchanges are perfect for scenarios where you genuinely want every subscriber to receive every message. Imagine a system monitoring server that publishes system metrics every second: CPU usage, memory usage, disk I/O, network traffic. Multiple different systems might want to consume these metrics. A real-time dashboard wants to display current values. A time-series database wants to store historical data. An alerting system wants to check if any metrics exceed thresholds. A capacity planning system wants to analyze trends. All these systems need the exact same metrics data, so a fanout exchange is ideal—publish once, deliver to all.

Another classic use case for fanout is cache invalidation in distributed systems. When you update data in a database, you need to invalidate corresponding entries in all cache servers. You publish a cache invalidation message to a fanout exchange, and every cache server subscribes to that exchange. All cache servers receive the invalidation message and clear their cached entries. You do not need to know how many cache servers exist or where they are located—they self-register by subscribing to the exchange.

```
Fanout Exchange Pattern:

Publisher                    Fanout Exchange                    Subscribers
    │                              │                                 │
    │  Publish("metrics", {...})   │                                 │
    ├─────────────────────────────▶│                                 │
    │                              │                                 │
    │                              │  Broadcast to ALL queues        │
    │                              │                                 │
    │                              ├────────────────────────────────▶│ Dashboard
    │                              │  (receives ALL messages)        │
    │                              │                                 │
    │                              ├────────────────────────────────▶│ Database
    │                              │  (receives ALL messages)        │
    │                              │                                 │
    │                              ├────────────────────────────────▶│ Alerts
    │                              │  (receives ALL messages)        │
    │                              │                                 │
    │                              └────────────────────────────────▶│ Analytics
    │                                 (receives ALL messages)        │

Every subscriber receives every message - no filtering
```

The simplicity of fanout exchanges makes them extremely fast and efficient. RabbitMQ does not need to evaluate routing rules or match patterns—it simply copies messages to all bound queues. This makes fanout the highest-throughput exchange type in RabbitMQ. However, this simplicity is also a limitation. If you have subscribers who only want certain messages, they still receive all messages and must filter them at the application level. This wastes network bandwidth and processing power. For scenarios requiring selective subscription, topic exchanges provide a better solution.

### Topic Exchange: Filtered Publish/Subscribe

Topic exchanges extend the Pub/Sub pattern with sophisticated filtering capabilities. Instead of broadcasting every message to every subscriber, topic exchanges allow subscribers to specify patterns of messages they want to receive. Publishers attach routing keys to messages describing what the message is about, and subscribers bind to the exchange with patterns describing what they want to receive.

Routing keys in topic exchanges are hierarchical strings with words separated by dots, such as "stock.NASDAQ.AAPL" or "log.error.database" or "sensor.temperature.warehouse.north". Subscribers use wildcard patterns in their bindings to match routing keys. The asterisk (*) wildcard matches exactly one word, and the hash (#) wildcard matches zero or more words. This creates powerful filtering capabilities.

Consider a logging system where applications publish log messages with routing keys structured as "log.LEVEL.COMPONENT". A log message might be "log.error.database" or "log.info.authentication" or "log.warning.payment". Different systems have different logging interests. The operations team might subscribe to "log.error._" to receive all error logs regardless of component. The database team might subscribe to "log._.database" to receive all logs from the database component regardless of severity. The security team might subscribe to "log.#.authentication" to receive all authentication-related logs. A comprehensive logging archive might subscribe to "log.#" to receive absolutely everything.

```
Topic Exchange Routing:

Publisher                        Topic Exchange                    Subscribers
    │                                   │                               │
    │  Publish                          │                               │
    │  key: "log.error.database"        │                               │
    ├──────────────────────────────────▶│                               │
    │                                   │                               │
    │                                   │  Match patterns               │
    │                                   │                               │
    │                                   │  Pattern: "log.error.*"       │
    │                                   ├──────────────────────────────▶│ Ops Team
    │                                   │  MATCH ✓                      │ (gets it)
    │                                   │                               │
    │                                   │  Pattern: "log.*.database"    │
    │                                   ├──────────────────────────────▶│ DB Team
    │                                   │  MATCH ✓                      │ (gets it)
    │                                   │                               │
    │                                   │  Pattern: "log.warning.*"     │
    │                                   │  NO MATCH ✗                   │ DevOps
    │                                   │  (message not sent)           │ (doesn't get it)
    │                                   │                               │
    │                                   │  Pattern: "log.#"             │
    │                                   ├──────────────────────────────▶│ Archive
    │                                   │  MATCH ✓                      │ (gets it)

Same message goes to Ops, DB Team, and Archive (3 subscribers)
but not to DevOps (pattern didn't match)
```

This selective broadcasting is extremely powerful for building flexible, evolvable systems. When you add a new subscriber with new filtering requirements, you simply create a new binding with the appropriate pattern—no changes to publishers or existing subscribers. When requirements change, you modify bindings without touching code. When you want temporary monitoring, you create a temporary queue with a specific binding and delete it when done.

Topic exchanges also enable multi-tenancy and data segregation. In a SaaS application serving multiple customers, you might publish messages with routing keys like "customer.123.order.created" or "customer.456.payment.processed". Each customer's event processor subscribes to "customer.CUSTOMER_ID.#" to receive only events for that customer. This provides isolation without requiring separate infrastructure for each customer.

### The Independence of Subscribers

One of the most profound aspects of Publish/Subscribe is the complete independence and isolation of subscribers. This independence manifests in several critical ways that fundamentally change how we design distributed systems.

First, subscribers are temporally independent. A subscriber does not need to be online when a message is published. If a subscriber is offline, messages destined for its queue accumulate waiting for it to reconnect. When the subscriber comes back online, it receives all the messages published during its downtime. This is radically different from traditional synchronous communication where both parties must be available simultaneously. With Pub/Sub, a publisher can broadcast messages even if no subscribers currently exist. When subscribers eventually connect, they begin receiving messages from that point forward.

Second, subscribers are spatially independent. Publishers do not know where subscribers are located, how many there are, or even if any exist. A publisher broadcasts messages to an exchange and its work is done. Subscribers could be running on the same server, across the datacenter, in different geographic regions, or not exist at all. The publisher's behavior does not change based on subscriber presence or location. This enables transparent scaling—you can add subscribers anywhere without reconfiguring publishers.

Third, subscribers are performance-independent. If one subscriber is slow, it does not affect other subscribers or the publisher. Each subscriber has its own queue and processes messages at its own pace. A fast subscriber quickly drains its queue and receives new messages immediately. A slow subscriber accumulates messages in its queue and processes them when it can. A crashed subscriber's queue fills up (or messages are discarded based on queue settings) but other subscribers continue operating normally. The publisher is never blocked waiting for slow subscribers.

Fourth, subscribers are failure-independent. If a subscriber crashes, fails, or is terminated, other subscribers are completely unaffected. The failed subscriber's queue might fill up or messages might be discarded, but the publisher continues publishing and other subscribers continue receiving messages. There is no cascading failure, no retry logic needed in the publisher, no error handling complexity. Failures are isolated and contained.

### Durability and Reliability in Publish/Subscribe

While Publish/Subscribe provides powerful decoupling, it also introduces challenges around durability and reliability. When a message is broadcast to multiple subscribers, you want to ensure that each subscriber reliably receives and processes the message even in the face of failures. RabbitMQ provides several mechanisms to ensure reliability in Pub/Sub scenarios.

Message persistence ensures that messages survive broker restarts. When you publish a message with delivery mode set to persistent and send it to a durable exchange that routes to durable queues, the message is written to disk. If RabbitMQ crashes and restarts, durable exchanges, durable queues, and persistent messages are all restored from disk. This means that even if the broker goes down after a message is published but before all subscribers have consumed it, those subscribers will still receive the message after the broker recovers.

Publisher confirms provide acknowledgment that RabbitMQ has successfully received and routed a message. In Pub/Sub scenarios, this is particularly important because a message might be routed to zero, one, or many queues. Publisher confirms tell you that the message was successfully routed. The mandatory flag can be used to detect when a message cannot be routed to any queue, allowing publishers to handle scenarios where no subscribers currently exist.

Consumer acknowledgments ensure that messages are not lost if a subscriber crashes while processing. When a subscriber receives a message, it must explicitly acknowledge that message after successfully processing it. If the subscriber crashes before acknowledging, RabbitMQ automatically redelivers the message. In Pub/Sub systems, each queue has independent acknowledgments—one subscriber acknowledging a message does not affect other subscribers' copies.

These reliability mechanisms work together to provide strong guarantees in Pub/Sub systems. A message published persistently to a durable exchange that routes to durable queues will be delivered to all current and future subscribers until explicitly acknowledged by each subscriber. This creates at-least-once delivery semantics where every subscriber is guaranteed to receive every relevant message at least once, though subscribers might receive duplicates if they crash and restart.

```go
// rabbitmq/infrastructure.go
package rabbitmq

import (
	"fmt"
	"log"

	"pubsub-system/config"
	"github.com/streadway/amqp"
)

// InfrastructureManager manages RabbitMQ exchanges and queues
type InfrastructureManager struct {
	channel *amqp.Channel
	config  *config.PubSubConfig
}

// NewInfrastructureManager creates infrastructure manager
func NewInfrastructureManager(channel *amqp.Channel, cfg *config.PubSubConfig) *InfrastructureManager {
	return &InfrastructureManager{
		channel: channel,
		config:  cfg,
	}
}

// SetupFanoutInfrastructure creates fanout exchange for broadcasting
func (im *InfrastructureManager) SetupFanoutInfrastructure() error {
	// Declare fanout exchange
	// Fanout broadcasts to ALL bound queues, ignoring routing keys
	err := im.channel.ExchangeDeclare(
		im.config.FanoutExchangeName, // name
		"fanout",                      // type - broadcasts to all
		im.config.ExchangeDurable,     // durable
		false,                         // auto-deleted
		false,                         // internal
		false,                         // no-wait
		nil,                           // arguments
	)
	if err != nil {
		return fmt.Errorf("failed to declare fanout exchange: %w", err)
	}

	log.Printf("Fanout exchange created: %s", im.config.FanoutExchangeName)
	return nil
}

// SetupTopicInfrastructure creates topic exchange for filtered pub/sub
func (im *InfrastructureManager) SetupTopicInfrastructure() error {
	// Declare topic exchange
	// Topic allows pattern-based routing with wildcards
	err := im.channel.ExchangeDeclare(
		im.config.TopicExchangeName, // name
		"topic",                      // type - pattern matching
		im.config.ExchangeDurable,    // durable
		false,                        // auto-deleted
		false,                        // internal
		false,                        // no-wait
		nil,                          // arguments
	)
	if err != nil {
		return fmt.Errorf("failed to declare topic exchange: %w", err)
	}

	log.Printf("Topic exchange created: %s", im.config.TopicExchangeName)
	return nil
}

// CreateSubscriberQueue creates a queue for a subscriber
// For fanout: binds queue with empty routing key (broadcasts all)
// For topic: binds queue with specific routing key pattern
func (im *InfrastructureManager) CreateSubscriberQueue(
	subscriberName string,
	exchangeName string,
	exchangeType string,
	routingKeyPattern string,
) (string, error) {
	queueName := fmt.Sprintf("%s_%s_queue", exchangeName, subscriberName)

	// Declare queue for this subscriber
	// Each subscriber gets its own queue for message independence
	_, err := im.channel.QueueDeclare(
		queueName,                 // name
		im.config.QueueDurable,    // durable
		false,                     // auto-delete
		false,                     // exclusive
		false,                     // no-wait
		nil,                       // arguments
	)
	if err != nil {
		return "", fmt.Errorf("failed to declare queue: %w", err)
	}

	// Bind queue to exchange
	// For fanout: routing key is ignored (empty string is fine)
	// For topic: routing key pattern determines which messages arrive
	err = im.channel.QueueBind(
		queueName,          // queue
		routingKeyPattern,  // routing key pattern
		exchangeName,       // exchange
		false,              // no-wait
		nil,                // arguments
	)
	if err != nil {
		return "", fmt.Errorf("failed to bind queue: %w", err)
	}

	log.Printf("Subscriber queue created and bound: %s -> %s (pattern: %s)",
		queueName, exchangeName, routingKeyPattern)
	
	return queueName, nil
}

// SetupCompleteInfrastructure sets up all exchanges
func (im *InfrastructureManager) SetupCompleteInfrastructure() error {
	if err := im.SetupFanoutInfrastructure(); err != nil {
		return err
	}
	if err := im.SetupTopicInfrastructure(); err != nil {
		return err
	}
	log.Println("Complete pub/sub infrastructure setup successful")
	return nil
}
```

### Generic Publisher

```go
// publisher/publisher.go
package publisher

import (
	"context"
	"fmt"
	"log"
	"time"

	"pubsub-system/config"
	"github.com/streadway/amqp"
)

// Publisher handles publishing messages to exchanges
type Publisher struct {
	channel      *amqp.Channel
	config       *config.PubSubConfig
	exchangeName string
	confirms     chan amqp.Confirmation
}

// NewPublisher creates a new publisher for an exchange
func NewPublisher(channel *amqp.Channel, cfg *config.PubSubConfig, exchangeName string) (*Publisher, error) {
	p := &Publisher{
		channel:      channel,
		config:       cfg,
		exchangeName: exchangeName,
	}

	// Enable publisher confirms if configured
	if cfg.EnableConfirms {
		if err := channel.Confirm(false); err != nil {
			return nil, fmt.Errorf("failed to enable confirms: %w", err)
		}
		p.confirms = channel.NotifyPublish(make(chan amqp.Confirmation, 1))
		log.Println("Publisher confirms enabled")
	}

	return p, nil
}

// Publish publishes a message to the exchange
// routingKey is used for topic exchanges, ignored for fanout
func (p *Publisher) Publish(ctx context.Context, routingKey string, message []byte) error {
	// Determine delivery mode
	deliveryMode := uint8(1) // non-persistent
	if p.config.Persistent {
		deliveryMode = 2 // persistent
	}

	// Publish the message
	err := p.channel.PublishWithContext(
		ctx,
		p.exchangeName, // exchange
		routingKey,     // routing key (ignored by fanout, used by topic)
		false,          // mandatory
		false,          // immediate
		amqp.Publishing{
			DeliveryMode: deliveryMode,
			ContentType:  "application/json",
			Body:         message,
			Timestamp:    time.Now(),
		},
	)

	if err != nil {
		return fmt.Errorf("failed to publish: %w", err)
	}

	// Wait for confirmation if enabled
	if p.config.EnableConfirms {
		select {
		case confirm := <-p.confirms:
			if !confirm.Ack {
				return fmt.Errorf("message not confirmed by broker")
			}
			log.Printf("Message confirmed: exchange=%s, routing_key=%s, size=%d bytes",
				p.exchangeName, routingKey, len(message))
		case <-time.After(p.config.PublishTimeout):
			return fmt.Errorf("publish confirmation timeout")
		case <-ctx.Done():
			return ctx.Err()
		}
	} else {
		log.Printf("Message published: exchange=%s, routing_key=%s, size=%d bytes",
			p.exchangeName, routingKey, len(message))
	}

	return nil
}

// PublishBatch publishes multiple messages efficiently
func (p *Publisher) PublishBatch(ctx context.Context, messages []MessageWithKey) error {
	for _, msg := range messages {
		if err := p.Publish(ctx, msg.RoutingKey, msg.Body); err != nil {
			return fmt.Errorf("failed to publish message in batch: %w", err)
		}
	}
```

```go
// subscriber/subscriber.go
package subscriber

import (
	"context"
	"fmt"
	"log"

	"pubsub-system/config"
	"github.com/streadway/amqp"
)

// MessageHandler processes received messages
type MessageHandler func(ctx context.Context, body []byte) error

// Subscriber handles subscribing to messages from a queue
type Subscriber struct {
	channel     *amqp.Channel
	config      *config.PubSubConfig
	queueName   string
	consumerTag string
	handler     MessageHandler
}

// NewSubscriber creates a new subscriber
func NewSubscriber(
	channel *amqp.Channel,
	cfg *config.PubSubConfig,
	queueName string,
	consumerTag string,
	handler MessageHandler,
) *Subscriber {
	return &Subscriber{
		channel:     channel,
		config:      cfg,
		queueName:   queueName,
		consumerTag: consumerTag,
		handler:     handler,
	}
}

// Start begins consuming messages
func (s *Subscriber) Start(ctx context.Context) error {
	// Register consumer
	messages, err := s.channel.Consume(
		s.queueName,    // queue
		s.consumerTag,  // consumer tag
		s.config.AutoAck, // auto-ack
		false,          // exclusive
		false,          // no-local
		false,          // no-wait
		nil,            // args
	)
	if err != nil {
		return fmt.Errorf("failed to register consumer: %w", err)
	}

	log.Printf("Subscriber started: queue=%s, consumer=%s", s.queueName, s.consumerTag)

	// Process messages
	for {
		select {
		case <-ctx.Done():
			log.Printf("Subscriber %s stopping", s.consumerTag)
			return ctx.Err()

		case msg, ok := <-messages:
			if !ok {
				return fmt.Errorf("message channel closed")
			}
			s.handleMessage(ctx, msg)
		}
	}
}

// handleMessage processes a single message
func (s *Subscriber) handleMessage(ctx context.Context, msg amqp.Delivery) {
	log.Printf("[%s] Received message: size=%d bytes", s.consumerTag, len(msg.Body))

	// Call handler
	err := s.handler(ctx, msg.Body)

	if !s.config.AutoAck {
		if err != nil {
			log.Printf("[%s] Error processing message: %v", s.consumerTag, err)
			// NACK and requeue on error
			msg.Nack(false, true)
		} else {
			// ACK on success
			if err := msg.Ack(false); err != nil {
				log.Printf("[%s] Failed to ACK: %v", s.consumerTag, err)
			} else {
				log.Printf("[%s] Message processed successfully", s.consumerTag)
			}
		}
	}
}
```

## Summary: The Power of Publish/Subscribe

This comprehensive implementation demonstrates the Publish/Subscribe pattern in both its pure broadcast form (fanout) and filtered form (topic). The fanout example shows how a single metrics message is simultaneously delivered to dashboard, database, alerting, and analytics subscribers—all receiving the identical message without any knowledge of each other. The topic example shows how sophisticated filtering enables subscribers to receive only the specific events they care about based on routing key patterns.

The key insight of Pub/Sub is the complete decoupling between publishers and subscribers. Publishers broadcast messages without knowing who receives them. Subscribers receive messages without knowing who published them. This decoupling enables independent scaling, independent deployment, independent development, and graceful handling of subscriber failures. When you add a new subscriber, no publishers need modification. When you remove a subscriber, the system continues operating normally. This is the power of the Publish/Subscribe pattern with RabbitMQ.
