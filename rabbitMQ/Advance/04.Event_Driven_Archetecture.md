# Event-Driven Architecture with RabbitMQ - Complete Guide

## Understanding Event-Driven Architecture

Imagine you are standing in a bustling city square where a town crier announces important news. When the crier shouts "The market is now open!" various people respond in different ways. Merchants rush to set up their stalls, customers begin shopping, street performers start their acts, and food vendors prepare their carts. The town crier does not need to know who these people are or what they will do. The crier simply announces the event, and everyone who cares about that event takes appropriate action independently. This is the essence of Event-Driven Architecture.

Event-Driven Architecture (EDA) is a software design paradigm where the flow of the program is determined by events rather than by sequential procedure calls. An event represents something significant that happened in the system: a user registered, an order was placed, a payment was processed, an inventory level dropped below a threshold, a file was uploaded. When these events occur, they are published to an event bus or message broker like RabbitMQ. Other components that care about these events subscribe to receive them and react accordingly. The publishers and subscribers do not know about each other directly; they only know about the events themselves.

This architectural pattern stands in stark contrast to traditional request-response architectures where components communicate directly through synchronous API calls. In a traditional e-commerce system, when a user places an order, the order service might directly call the payment service, which then calls the inventory service, which then calls the shipping service, and finally calls the notification service. Each service must wait for the previous one to complete, creating a tightly coupled chain of dependencies. If any service in this chain fails or becomes slow, the entire operation fails or slows down. The order service must know about all downstream services and must handle all their errors.

In an event-driven architecture, the order service simply publishes an "OrderPlaced" event to RabbitMQ and immediately returns success to the user. The payment service, inventory service, shipping service, and notification service all independently subscribe to this event. When they receive it, they each perform their respective tasks asynchronously and in parallel. If the notification service is down, the order still gets processed, paid for, and shipped. The notification can be sent later when the service recovers. Each service operates independently, at its own pace, without creating blocking dependencies.

### The Core Benefits of Event-Driven Architecture

Event-Driven Architecture provides several transformative benefits that fundamentally change how we build and scale distributed systems. The most immediate benefit is decoupling. Services become truly independent, knowing nothing about each other beyond the structure of the events they exchange. You can add new services that react to existing events without modifying any existing code. You can remove services without breaking the system. You can deploy services independently, scale them independently, and develop them in different programming languages using different technologies. This decoupling accelerates development velocity because teams can work independently without coordinating changes.

The second major benefit is scalability and resilience. Because services communicate asynchronously through a message broker, they naturally handle load spikes gracefully. If your payment service receives a thousand order events in one second, it processes them at its own pace, working through the queue without being overwhelmed. If a service crashes, events accumulate in the queue waiting for it to recover, rather than causing cascading failures throughout the system. You can scale individual services based on their specific load patterns. If the notification service is slow, you can scale it up without touching any other service.

The third benefit is temporal decoupling. Services do not need to be available at the same time. A service can publish an event even if no subscribers are currently online. When subscribers come online later, they receive and process the events. This is crucial for maintenance, deployments, and handling services with different availability requirements. It also enables better fault tolerance—a temporary outage of a subscriber does not cause data loss because events are safely stored in the message broker.

The fourth benefit is enabling real-time reactions and complex event processing. Multiple services can react to the same event simultaneously, each performing different actions. You can build sophisticated workflows where one event triggers multiple other events, creating event chains. You can correlate multiple events to detect patterns and trigger actions based on combinations of events. This makes it easy to implement features like "send an email if a user abandons their cart three times" or "alert if inventory drops below threshold and supplier is on holiday."

### The Conceptual Model of Events

To understand Event-Driven Architecture deeply, we need to understand what events actually are and how they differ from commands and queries. An event is a statement of fact about something that already happened in the past. Events are always named in past tense: "OrderPlaced", "PaymentProcessed", "UserRegistered", "InventoryUpdated". The key characteristic of an event is that it is immutable and represents historical truth. Once an "OrderPlaced" event has occurred, it cannot be undone or changed—it is a permanent record that an order was placed at a specific time.

Events carry data describing what happened. An "OrderPlaced" event might contain the order ID, customer ID, items ordered, total amount, timestamp, and any other relevant information about the order. This data should be complete enough that subscribers can react to the event without needing to call back to the publisher for more information. This is called event enrichment—including all necessary context in the event itself to enable autonomous processing by subscribers.

Events differ fundamentally from commands. A command is an instruction to do something in the future: "PlaceOrder", "ProcessPayment", "SendEmail". Commands are imperative and can be rejected. If you send a "PlaceOrder" command and the inventory is insufficient, the command fails. Events, being historical facts, cannot be rejected—they represent something that already happened. You publish an "OrderPlaced" event only after the order has been successfully placed in your system.

Events also differ from queries. A query asks for information: "GetOrderDetails", "FetchUserProfile". Queries expect synchronous responses with data. Events are fire-and-forget—the publisher sends the event and continues immediately without waiting for any response. Subscribers process events asynchronously at their own pace.

Understanding these distinctions is crucial for designing clean event-driven systems. Use commands when you need synchronous validation and immediate feedback. Use queries when you need to retrieve current state. Use events when you need to broadcast that something significant happened, enabling multiple independent reactions.

### Event-Driven Architecture Patterns

Event-Driven Architecture encompasses several distinct patterns, each suited to different use cases. Understanding these patterns helps you choose the right approach for your specific requirements.

The **Event Notification** pattern is the simplest and most common. When something significant happens, a service publishes a lightweight event notification. The event contains minimal information—perhaps just an identifier and a timestamp. Subscribers who care about this event receive the notification and then query the source system for full details if needed. This pattern keeps events small and avoids duplication of data across events. However, it creates coupling because subscribers must call back to the publisher, and it requires the publisher to maintain queryable state.

The **Event-Carried State Transfer** pattern includes complete, self-contained information in the event. When an order is placed, the "OrderPlaced" event contains all order details: items, quantities, prices, customer information, shipping address, everything. Subscribers have all the data they need to process the event without making any additional calls. This creates autonomous subscribers but results in larger events and data duplication. Each subscriber effectively maintains its own denormalized copy of relevant data from the events it receives. This pattern is powerful for building highly decoupled, independently scalable services but requires careful management of data consistency.

The **Event Sourcing** pattern treats events as the primary source of truth. Instead of storing current state in a database, you store the sequence of events that led to that state. The current state is derived by replaying events. For example, instead of storing a bank account balance, you store "AccountOpened", "MoneyDeposited", "MoneyWithdrawn" events. The current balance is calculated by summing all deposits and subtracting all withdrawals. This provides complete audit history, enables time travel (you can reconstruct state at any point in history), and makes it easy to build new views of data by replaying events. However, it adds complexity in querying current state and managing event schema evolution.

The **CQRS** (Command Query Responsibility Segregation) pattern separates the write model from the read model. Commands modify state by generating events. These events update the write model and also propagate to build optimized read models. For example, when a user updates their profile, this generates a "ProfileUpdated" event. This event updates the normalized database table (write model) and also updates denormalized search indexes, caches, and analytics databases (read models). Each read model is optimized for specific query patterns. CQRS works beautifully with event-driven architecture because events naturally flow from the write side to update multiple read models.

The **Saga** pattern coordinates multi-step distributed transactions using events. Instead of a single transaction spanning multiple services, each step is a local transaction that publishes an event. The next step reacts to that event with its own transaction and event. If any step fails, compensating events are published to undo previous steps. For example, booking a trip might involve reserving a flight, hotel, and car. Each reservation publishes a success or failure event. If the car reservation fails, compensating events cancel the flight and hotel reservations. Sagas maintain eventual consistency across services without requiring distributed transactions.

### RabbitMQ as the Event Backbone

RabbitMQ serves as the central nervous system in an event-driven architecture, providing the infrastructure for reliable event distribution. Its features align perfectly with event-driven requirements. The exchange system provides flexible routing so events can reach the right subscribers. Topic exchanges enable pattern-based subscriptions where services subscribe to categories of events using wildcards. Fanout exchanges enable broadcasting events to all interested parties. Headers exchanges enable content-based routing where subscription depends on event metadata.

RabbitMQ's durability features ensure events are never lost. Persistent messages and durable queues mean that events survive broker restarts. Publisher confirms ensure that events are safely stored before acknowledging the publisher. Consumer acknowledgments ensure events are not lost if a subscriber crashes while processing. These reliability features are essential in event-driven systems where events represent critical business facts that cannot be lost.

RabbitMQ's queuing naturally handles the asynchronous nature of event-driven architecture. Publishers send events and immediately continue without waiting. Subscribers process events at their own pace, limited only by their capacity. If a subscriber is temporarily offline, events queue up waiting for it. If a subscriber is slow, events accumulate in its queue but do not affect other subscribers or the publisher. This temporal decoupling is fundamental to event-driven architecture's resilience.

Dead letter queues and retry mechanisms integrate seamlessly with event-driven patterns. If a subscriber cannot process an event due to a temporary error, the event can be retried automatically. If the error is permanent, the event goes to a dead letter queue for investigation. This ensures no event is silently lost, maintaining the integrity of your event-driven system.

### System Architecture Overview

Our e-commerce system will consist of several independent services, each owning its domain and reacting to events from other services. The services communicate exclusively through events published to RabbitMQ—there are no direct service-to-service API calls. This creates a highly decoupled system where services can be developed, deployed, and scaled independently.

The **Order Service** is responsible for receiving customer orders and initiating the order fulfillment workflow. When a customer places an order through the user interface, the Order Service validates the order data, assigns an order ID, stores the order in its database with status "pending", and publishes an "OrderPlaced" event to RabbitMQ. The service then immediately returns success to the customer. The customer sees "Your order has been received" without waiting for payment processing, inventory reservation, or shipping. The Order Service also subscribes to events like "PaymentFailed" and "InventoryOutOfStock" to update order status and potentially trigger cancellation workflows.

The **Payment Service** subscribes to "OrderPlaced" events. When it receives such an event, it processes payment using the payment information included in the event. If payment succeeds, it publishes a "PaymentProcessed" event containing the order ID, payment confirmation number, and amount charged. If payment fails, it publishes a "PaymentFailed" event with error details. The Payment Service maintains its own database of payment transactions for reconciliation and reporting. It knows nothing about inventory, shipping, or notifications—it simply reacts to order events and publishes payment results.

The **Inventory Service** also subscribes to "OrderPlaced" events. When it receives an order event, it checks stock levels for the requested items. If sufficient stock is available, it reserves the inventory (decrements available stock but does not commit the decrement until shipping confirms), updates its database, and publishes an "InventoryReserved" event. If stock is insufficient, it publishes an "InventoryOutOfStock" event. The Inventory Service maintains authoritative data about product stock levels and handles all inventory management operations based on events.

The **Shipping Service** subscribes to both "PaymentProcessed" and "InventoryReserved" events. It needs both to occur before creating a shipment—you should not ship an unpaid order, and you cannot ship without reserved inventory. The Shipping Service uses event correlation to wait for both events for the same order ID. Once both are received, it creates a shipment record in its database, assigns a tracking number, and publishes a "ShipmentCreated" event. If it receives "PaymentFailed" or "InventoryOutOfStock", it does nothing—no shipment is created.

The **Notification Service** subscribes to all events using a wildcard subscription pattern. It reacts to different events by sending appropriate notifications to customers. "OrderPlaced" triggers an order confirmation email. "PaymentProcessed" might trigger a payment receipt email. "ShipmentCreated" triggers a shipping notification with tracking information. The Notification Service is completely isolated—it can be down for hours and when it recovers, it processes all queued events and sends delayed notifications. Other services are completely unaffected by notification service outages.

### Event Schema Design

Designing good event schemas is crucial for maintainable event-driven systems. Events are contracts between services and must be carefully designed to balance completeness, size, and evolvability. Let us explore best practices through our e-commerce example.

Each event should be self-describing, containing all context needed to understand what happened. An event should include a unique event ID for idempotency checking and deduplication, a timestamp indicating when the event occurred, a version number for schema evolution, and the aggregate or entity ID that the event relates to. The event payload should include all relevant data about what changed, not just an identifier.

Events should be immutable. Once published, an event represents historical fact and should never be modified. If you discover an error in an event, publish a compensating event rather than trying to modify the original. For example, if you published an "OrderPlaced" event with the wrong total amount, publish an "OrderAmountCorrected" event rather than trying to change the original event.

Events should be as small as practical while still being complete enough for autonomous processing. This is a tradeoff. Including all data creates larger events but enables subscribers to process events without calling back to the publisher. Including only IDs creates smaller events but requires subscribers to make additional calls for details. Generally, favor including complete data for frequently accessed information and IDs for rarely accessed details or very large data.

Events should use consistent naming conventions. Event names should be past tense verbs describing what happened: "OrderPlaced", not "PlaceOrder". Use domain language that business stakeholders understand. Organize events into bounded contexts matching your domain model. Use hierarchical naming with context prefixes if you have many events: "order.placed", "order.cancelled", "payment.processed", "inventory.reserved".

Events must be versioned to support evolution. As your system grows, you will need to add fields, change types, or restructure events. Use semantic versioning in a version field within the event. Design subscribers to be tolerant of unknown fields and missing optional fields. When making breaking changes, publish events with a new version number and maintain subscribers that can handle both old and new versions during a transition period.

```go
// events/events.go
package events

import (
	"encoding/json"
	"time"
)

// EventType represents the type of event
type EventType string

const (
	// Order events
	EventOrderPlaced    EventType = "order.placed"
	EventOrderCancelled EventType = "order.cancelled"
	EventOrderCompleted EventType = "order.completed"
	
	// Payment events
	EventPaymentProcessed EventType = "payment.processed"
	EventPaymentFailed    EventType = "payment.failed"
	EventPaymentRefunded  EventType = "payment.refunded"
	
	// Inventory events
	EventInventoryReserved   EventType = "inventory.reserved"
	EventInventoryOutOfStock EventType = "inventory.out_of_stock"
	EventInventoryReleased   EventType = "inventory.released"
	
	// Shipping events
	EventShipmentCreated   EventType = "shipment.created"
	EventShipmentDispatched EventType = "shipment.dispatched"
	EventShipmentDelivered EventType = "shipment.delivered"
	
	// Notification events
	EventEmailSent EventType = "notification.email_sent"
	EventSMSSent   EventType = "notification.sms_sent"
)

// BaseEvent contains common fields for all events
// This is embedded in all specific event types
type BaseEvent struct {
	EventID     string    `json:"event_id"`      // Unique identifier for this event
	EventType   EventType `json:"event_type"`    // Type of event
	AggregateID string    `json:"aggregate_id"`  // ID of the entity this event relates to
	OccurredAt  time.Time `json:"occurred_at"`   // When the event occurred
	Version     int       `json:"version"`       // Schema version for evolution
	CorrelationID string  `json:"correlation_id,omitempty"` // For tracing related events
	CausationID   string  `json:"causation_id,omitempty"`   // ID of event that caused this event
}

// OrderPlacedEvent represents an order being placed
type OrderPlacedEvent struct {
	BaseEvent
	OrderID      string                 `json:"order_id"`
	CustomerID   string                 `json:"customer_id"`
	Items        []OrderItem            `json:"items"`
	TotalAmount  float64                `json:"total_amount"`
	Currency     string                 `json:"currency"`
	PaymentMethod string                `json:"payment_method"`
	ShippingAddress Address             `json:"shipping_address"`
	Metadata     map[string]interface{} `json:"metadata,omitempty"`
}

// OrderItem represents an item in an order

	ProductID   string  `json:"product_id"`
	ProductName string  `json:"product_name"`
	Quantity    int     `json:"quantity"`
	Price       float64 `json:"price"`
	SKU         string  `json:"sku"`
}

// Address represents a shipping or billing address
type Address struct {
	Street  string `json:"street"`
	City    string `json:"city"`
	State   string `json:"state"`
	Zip     string `json:"zip"`
	Country string `json:"country"`
}

// PaymentProcessedEvent represents successful payment

	BaseEvent
	OrderID            string    `json:"order_id"`

	Amount             float64   `json:"amount"`
	Currency           string    `json:"currency"`
	PaymentMethod      string    `json:"payment_method"`
	TransactionID      string    `json:"transaction_id"`
	ProcessedAt        time.Time `json:"processed_at"`
	PaymentGateway     string    `json:"payment_gateway"`
}

// PaymentFailedEvent represents failed payment
type PaymentFailedEvent struct {
	BaseEvent
	OrderID       string    `json:"order_id"`
	Amount        float64   `json:"amount"`
	Currency      string    `json:"currency"`
	PaymentMethod string    `json:"payment_method"`
	FailureReason string    `json:"failure_reason"`
	FailureCode   string    `json:"failure_code"`
	Retryable     bool      `json:"retryable"`
	FailedAt      time.Time `json:"failed_at"`
}

// InventoryReservedEvent represents successful inventory reservation
type InventoryReservedEvent struct {
	BaseEvent
	OrderID       string               `json:"order_id"`
	ReservationID string               `json:"reservation_id"`
	Items         []InventoryReservation `json:"items"`
	ReservedAt    time.Time            `json:"reserved_at"`
	ExpiresAt     time.Time            `json:"expires_at"` // Reservation expiry
}

	Quantity        int    `json:"quantity"`

}

// InventoryOutOfStockEvent represents insufficient inventory
type InventoryOutOfStockEvent struct {
	BaseEvent
	OrderID      string              `json:"order_id"`

	CheckedAt    time.Time          `json:"checked_at"`
}

// OutOfStockItem represents an item that is out of stock
type OutOfStockItem struct {
	ProductID        string `json:"product_id"`
	SKU              string `json:"sku"`
	RequestedQty     int    `json:"requested_quantity"`
	AvailableQty     int    `json:"available_quantity"`
}

// ShipmentCreatedEvent represents shipment creation
type ShipmentCreatedEvent struct {
	BaseEvent
	OrderID         string    `json:"order_id"`
	ShipmentID      string    `json:"shipment_id"`
	TrackingNumber  string    `json:"tracking_number"`
	Carrier         string    `json:"carrier"`
	EstimatedDelivery time.Time `json:"estimated_delivery"`
	Items           []OrderItem `json:"items"`
	ShippingAddress Address   `json:"shipping_address"`
	CreatedAt       time.Time `json:"created_at"`
}

// Event is a generic container for any event type
type Event struct {
	Type    EventType       `json:"type"`
	Payload json.RawMessage `json:"payload"`
}

// NewBaseEvent creates a new base event with common fields populated
func NewBaseEvent(eventType EventType, aggregateID string) BaseEvent {
	return BaseEvent{
		EventID:     generateEventID(),
		EventType:   eventType,
		AggregateID: aggregateID,
		OccurredAt:  time.Now().UTC(),
		Version:     1,
	}
}

// WithCorrelation adds correlation ID to event
func (e *BaseEvent) WithCorrelation(correlationID string) *BaseEvent {
	e.CorrelationID = correlationID
	return e
}

// WithCausation adds causation ID to event
func (e *BaseEvent) WithCausation(causationID string) *BaseEvent {
	e.CausationID = causationID
	return e
}

// generateEventID creates a unique event identifier
func generateEventID() string {
	return time.Now().Format("20060102150405") + "-" + randomString(8)
}

func randomString(length int) string {
	const charset = "abcdefghijklmnopqrstuvwxyz0123456789"
	b := make([]byte, length)
	for i := range b {
		b[i] = charset[time.Now().UnixNano()%int64(len(charset))]
	}
	return string(b)
}
```

### Event Publisher

```go
// events/publisher.go
package events

import (
	"context"
	"encoding/json"
	"fmt"
	"log"
	"time"

	"github.com/streadway/amqp"
)

// Publisher handles publishing events to RabbitMQ
type Publisher struct {
	channel      *amqp.Channel
	exchangeName string
}

// NewPublisher creates a new event publisher
func NewPublisher(channel *amqp.Channel, exchangeName string) *Publisher {
	return &Publisher{
		channel:      channel,
		exchangeName: exchangeName,
	}
}

// Publish publishes an event to the event exchange
// The routing key is derived from the event type
// This enables topic-based subscription patterns
func (p *Publisher) Publish(ctx context.Context, event interface{}) error {
	// Marshal event to JSON
	payload, err := json.Marshal(event)
	if err != nil {
		return fmt.Errorf("failed to marshal event: %w", err)
	}

	// Extract event type for routing
	// We use reflection to get the EventType field
	routingKey := p.extractRoutingKey(event)
	
	log.Printf("Publishing event: type=%s, size=%d bytes", routingKey, len(payload))

	// Publish with persistence and confirmation
	err = p.channel.PublishWithContext(
		ctx,
		p.exchangeName, // exchange
		routingKey,     // routing key (event type)
		false,          // mandatory
		false,          // immediate
		amqp.Publishing{
			DeliveryMode: amqp.Persistent,    // Persist to disk
			ContentType:  "application/json",
			Body:         payload,
			Timestamp:    time.Now(),
			MessageId:    p.extractEventID(event),
			Type:         routingKey,
			Headers: amqp.Table{
				"event_type": routingKey,
			},
		},
	)

	if err != nil {
		return fmt.Errorf("failed to publish event: %w", err)
	}

	log.Printf("Event published successfully: type=%s", routingKey)
	return nil
}

// PublishBatch publishes multiple events in a batch
```

```go
package events

import (
	"context"
	"encoding/json"
	"fmt"
	"log"

	"github.com/streadway/amqp"
)

// EventHandler is a function that processes an event
type EventHandler func(ctx context.Context, eventType EventType, payload []byte) error

// Subscriber handles subscribing to events from RabbitMQ
type Subscriber struct {
	channel     *amqp.Channel
	queueName   string
	handlers    map[EventType]EventHandler
	consumerTag string
}

// NewSubscriber creates a new event subscriber
func NewSubscriber(channel *amqp.Channel, queueName, consumerTag string) *Subscriber {
	return &Subscriber{
		channel:     channel,
		queueName:   queueName,
		handlers:    make(map[EventType]EventHandler),
		consumerTag: consumerTag,
	}
}

// On registers a handler for a specific event type
// Multiple handlers can be registered for different event types
func (s *Subscriber) On(eventType EventType, handler EventHandler) {
	s.handlers[eventType] = handler
	log.Printf("Registered handler for event type: %s", eventType)
}

// Start begins consuming events from the queue
func (s *Subscriber) Start(ctx context.Context) error {
	// Register as consumer
	messages, err := s.channel.Consume(
		s.queueName,   // queue
		s.consumerTag, // consumer tag
		false,         // auto-ack (false for manual ack)
		false,         // exclusive
		false,         // no-local
		false,         // no-wait
		nil,           // args
	)
	if err != nil {
		return fmt.Errorf("failed to register consumer: %w", err)
	}

	log.Printf("Subscriber started: queue=%s, consumer=%s", s.queueName, s.consumerTag)

	// Process messages
	for {
		select {
		case <-ctx.Done():
			log.Println("Subscriber stopping due to context cancellation")
			return ctx.Err()

		case msg, ok := <-messages:
			if !ok {
				log.Println("Message channel closed")
				return fmt.Errorf("message channel closed")
			}

			s.handleMessage(ctx, msg)
		}
	}
}

// handleMessage processes a single message
func (s *Subscriber) handleMessage(ctx context.Context, msg amqp.Delivery) {
	// Extract event type from message headers
	eventTypeStr := msg.Type
	if eventTypeStr == "" {
		// Fallback to header
		if et, ok := msg.Headers["event_type"].(string); ok {
			eventTypeStr = et
		}
	}

	eventType := EventType(eventTypeStr)
	
	log.Printf("Received event: type=%s, delivery_tag=%d, size=%d bytes",
		eventType, msg.DeliveryTag, len(msg.Body))

	// Find handler for this event type
	handler, exists := s.handlers[eventType]
	if !exists {
		log.Printf("No handler registered for event type: %s, skipping", eventType)
		// Acknowledge message even though we're not handling it
		// This prevents unhandled events from blocking the queue
		msg.Ack(false)
		return
	}

	// Call the handler
	err := handler(ctx, eventType, msg.Body)
	if err != nil {
		log.Printf("Error handling event %s: %v", eventType, err)
		
		// Check if this is a retriable error
		// For now, we'll NACK and requeue transient errors
		// In production, you'd integrate with retry mechanisms
		msg.Nack(false, true) // NACK with requeue
		return
	}

	// Success - acknowledge the message
	err = msg.Ack(false)
	if err != nil {
		log.Printf("Failed to ACK message: %v", err)
	} else {
		log.Printf("Event processed successfully: type=%s", eventType)
	}
}

// UnmarshalEvent is a helper to unmarshal event payload
func UnmarshalEvent(payload []byte, target interface{}) error {
	return json.Unmarshal(payload, target)
}
```

```go
// SetupEventExchange creates the main event exchange
// This is a topic exchange for pattern-based routing
func (ei *EventInfrastructure) SetupEventExchange() error {
	// Declare topic exchange for events
	// Topic exchange allows wildcard subscriptions like "order.*" or "payment.#"
	err := ei.channel.ExchangeDeclare(
		ei.exchangeName, // name
		"topic",         // type - topic for pattern matching
		true,            // durable - survive restart
		false,           // auto-deleted
		false,           // internal
		false,           // no-wait
		nil,             // arguments
	)
	if err != nil {
		return fmt.Errorf("failed to declare event exchange: %w", err)
	}

	log.Printf("Event exchange created: %s (type: topic)", ei.exchangeName)
	return nil
}

// SetupServiceQueue creates a queue for a service with event bindings
func (ei *EventInfrastructure) SetupServiceQueue(
	serviceName string,
	eventPatterns []string,
) error {
	queueName := fmt.Sprintf("%s_events_queue", serviceName)

	// Declare durable queue for this service
	_, err := ei.channel.QueueDeclare(
		queueName, // name
		true,      // durable
		false,     // auto-delete
		false,     // exclusive
		false,     // no-wait
		nil,       // arguments
	)
	if err != nil {
		return fmt.Errorf("failed to declare queue for %s: %w", serviceName, err)
	}

	// Bind queue to exchange with event patterns
	// Each pattern allows the service to subscribe to specific events
	for _, pattern := range eventPatterns {
		err = ei.channel.QueueBind(
			queueName,       // queue
			pattern,         // routing key pattern
			ei.exchangeName, // exchange
			false,           // no-wait
			nil,             // arguments
		)
		if err != nil {
			return fmt.Errorf("failed to bind queue %s with pattern %s: %w",
				queueName, pattern, err)
		}
		log.Printf("Bound queue %s to pattern: %s", queueName, pattern)
	}

	log.Printf("Service queue created: %s with %d bindings", queueName, len(eventPatterns))
	return nil
}
```

```go
		return err
	}

	log.Printf("Shipping service received PaymentProcessed: %s", event.OrderID)

	state := h.service.correlator.RecordPayment(event)
	return h.service.CreateShipment(ctx, state)
}
```

## Summary: The Power of Event-Driven Architecture

This comprehensive implementation demonstrates how Event-Driven Architecture with RabbitMQ creates highly decoupled, scalable, and resilient distributed systems. The Order Service publishes an "OrderPlaced" event without knowing or caring which services will react to it. Payment, Inventory, Shipping, and Notification services all independently subscribe to relevant events and process them asynchronously at their own pace.

Each service operates autonomously, maintaining its own data and publishing events about significant state changes in its domain. Services communicate only through events, never through direct API calls. This creates true decoupling where services can be developed by different teams, deployed independently, scaled independently, and even replaced entirely without affecting other services.

The event correlation pattern in the Shipping Service demonstrates how to build sophisticated workflows where multiple conditions must be satisfied before taking action. The service waits for both payment confirmation and inventory reservation before creating a shipment, coordinating these asynchronous events elegantly.

This architecture naturally provides fault tolerance, scalability, and maintainability. Failed services simply lag behind in processing events until they recover. High-load services can be scaled up independently. New features can be added by creating new services that subscribe to existing events without modifying any existing code. This is the power of Event-Driven Architecture with RabbitMQ.
