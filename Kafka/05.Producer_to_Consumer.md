# Kafka Data Flow — Partition, group_id & Consumer Groups

### Explained Simply, from the Ground Up

---

## The One Analogy That Explains Everything

Imagine Kafka is a **post office**. The **topic** is the post office building. The **partitions** are the sorting lanes inside it — Lane 0, Lane 1, Lane 2. Every letter (message) that arrives gets dropped into one of those lanes. Messages go in one end and pile up in order. That is all a partition is. It is a lane.

Now, your entire question — every consumer gets all messages, or some consumers get some messages — comes down to **one single thing**: the `group_id`. It is the company name written on the mailbox. And the rule is exactly this:

```
  ┌────────────────────────────────────────────────────────────────┐
  │                    THE ONE RULE                                │
  │                                                                │
  │   Different group_id  =  every service gets ALL messages       │
  │      Same group_id    =  instances SHARE the messages          │
  └────────────────────────────────────────────────────────────────┘
```

Everything else in this document is just an explanation of why that rule works the way it does.

---

## What a Message Actually Is

Before following a message on its journey, you need to know what it contains. A Kafka message is not just raw data — it is an envelope with several fields.

```
  ┌─────────────────────────────────────────────────────────────┐
  │                   A KAFKA MESSAGE                           │
  │                                                             │
  │  KEY       →  "order-8821"                                  │
  │               (routing instruction — picks the partition)   │
  │                                                             │
  │  VALUE     →  { "order_id": "8821", "total": 129.99 }       │
  │               (the actual event data — your payload)        │
  │                                                             │
  │  OFFSET    →  42                                            │
  │               (position in the partition, assigned by       │
  │                Kafka, immutable, never changes)             │
  │                                                             │
  │  TIMESTAMP →  2024-11-15T09:23:41Z                          │
  │                                                             │
  │  HEADERS   →  source-service: "order-service"               │
  │               trace-id: "abc-123"                           │
  └─────────────────────────────────────────────────────────────┘
```

The **key** is the most important field for understanding data flow. It is not part of your event data — it is a routing instruction. When you set the key to `"order-8821"`, Kafka runs a simple formula: `hash("order-8821") % number_of_partitions` and the result is the partition number this message goes to. Same key always = same partition. This is how Kafka guarantees ordering per entity.

## What a Partition Actually Is

A partition is a numbered, ordered, append-only queue that lives inside a topic. Think of the topic as a post office building and the partition as a sorting lane inside it.

```
  Topic: "order.placed"  (3 partitions)

  ┌────────────────────────────────────────────────────────────────┐
  │  POST OFFICE BUILDING — "order.placed"                         │
  │                                                                │
  │  Lane 0 (P0):  [msg0] [msg3] [msg6] [msg9]  → new msgs...      │
  │                off:0   off:1  off:2   off:3                    │
  │                                                                │
  │  Lane 1 (P1):  [msg1] [msg4] [msg7] [msg10] → new msgs...      │
  │                off:0   off:1  off:2   off:3                    │
  │                                                                │
  │  Lane 2 (P2):  [msg2] [msg5] [msg8] [msg11] → new msgs...      │
  │                off:0   off:1  off:2   off:3                    │
  └────────────────────────────────────────────────────────────────┘

  IMPORTANT:
  ✅  Within ONE partition — messages are strictly ordered
  ✅  Each partition has its OWN offset counter starting from 0
  ✅  Offset 0 in P0 is a completely different message than offset 0 in P1
  ⚠️  Across partitions — there is NO ordering guarantee
```

Messages never move between partitions. Once a message lands in Lane 2 with offset 5, it stays in Lane 2 at offset 5 forever — until the retention period expires and Kafka deletes it.

The **purpose of having multiple partitions** is parallelism. A single partition can only be read by one consumer instance at a time (within a group). Three partitions means three consumer instances can work simultaneously, each reading one lane, tripling your throughput.

## How a Producer Decides Which Partition to Use

The producer makes the partitioning decision before the message ever reaches Kafka. The logic is simple:

```
  PRODUCER PARTITIONING DECISION

  Case A — Message has a KEY:
  ───────────────────────────────────────────────────────────────
  key = "order-8821"
  hash("order-8821") = 2,847,392,019
  2,847,392,019 % 3 partitions = partition 1

  → msg goes to P1, every time, forever
  → All future events for order-8821 also go to P1
  → Guarantees order-8821's events are always read in sequence ✅

  Case B — Message has NO KEY:
  ───────────────────────────────────────────────────────────────
  Kafka distributes messages round-robin across partitions:
  msg1 → P0,  msg2 → P1,  msg3 → P2,  msg4 → P0 ...

  → Load is perfectly balanced ✅
  → But ordering is NOT guaranteed ⚠️
  → Use this when the order of individual messages does not matter
```

The key insight is: **if the order of events matters** (e.g. you must process "order placed" before "order shipped"), always use a key. If you only care about throughput and each message is independent, skip the key and let Kafka balance the load automatically.

## Your Case 1 — Every Consumer Gets ALL Messages

This is when three completely different services all want to react to every single order event. The Inventory Service needs all orders. The Notification Service needs all orders. The Analytics Service needs all orders. None of them should miss a single message.

```
  YOUR CASE 1: Different group_id → Full broadcast to each service

  ┌──────────────┐
  │ Order Service│
  │  (Producer)  │
  │  sends:      │
  │  msg1 msg2   │
  │  msg3 msg4   │
  └──────┬───────┘
         │
         ▼
  ┌───────────────────────────────┐
  │  Topic: "order.placed"        │
  │                               │
  │  P0: [msg1][msg4]             │
  │  P1: [msg2]                   │
  │  P2: [msg3]                   │
  └────────┬──────────────────────┘
           │
           │  Kafka delivers independently to EACH group_id
           │
     ┌─────┴────────────────────────────────────────────┐
     │                   │                              │
     ▼                   ▼                              ▼
  ┌──────────────┐  ┌──────────────┐         ┌──────────────────┐
  │  Inventory   │  │ Notification │         │   Analytics      │
  │  Service     │  │  Service     │         │   Service        │
  │              │  │              │         │                  │
  │ group_id:    │  │ group_id:    │         │ group_id:        │
  │  "inv-svc"   │  │ "notif-svc"  │         │  "analytics-svc" │
  │              │  │              │         │                  │
  │ Gets:        │  │ Gets:        │         │ Gets:            │
  │ msg1,2,3,4 ✅│  │ msg1,2,3,4 ✅│        │ msg1,2,3,4 ✅    │
  └──────────────┘  └──────────────┘         └──────────────────┘

  WHY THIS WORKS:
  ─────────────────────────────────────────────────────────────────
  Kafka keeps a separate bookmark (called a committed offset) for
  every group_id. "inv-svc" has its own bookmark. "notif-svc" has
  its own. "analytics-svc" has its own. They never interfere with
  each other. Reading a message does not remove it from the topic.
  It stays there until the retention period expires.
  ─────────────────────────────────────────────────────────────────

  REAL WORLD EXAMPLE:
  A user registers. The "user.registered" event lands in the topic.
  → Notification Service reads it → sends welcome email
  → CRM Service reads it          → creates customer record
  → Analytics Service reads it    → updates signup dashboard
  → Risk Service reads it         → scores fraud risk

  The User Service that produced the event has no idea any of
  these consumers exist. Adding a 5th consumer requires ZERO
  changes to any existing service.
```

## Your Case 2 — Each Consumer Gets SOME Messages

This is when you have one service — the Inventory Service — running as three separate instances for performance. You do not want all three instances to process the same order. You want them to split the work: instance 1 handles some orders, instance 2 handles others, instance 3 handles the rest.

```
  YOUR CASE 2: Same group_id → Work is split between instances

  ┌──────────────┐
  │ Order Service│
  │  (Producer)  │
  │  sends:      │
  │  msg1 – msg6 │
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────────────┐
  │  Topic: "order.placed"  (3 partitions)       │
  │                                              │
  │  P0: [msg1] [msg4]                           │
  │  P1: [msg2] [msg5]                           │
  │  P2: [msg3] [msg6]                           │
  └───────┬──────────────────────────────────────┘
          │
          │  Kafka sees all three have group_id = "inv-svc"
          │  It treats them as ONE group and ASSIGNS partitions
          │
  ┌───────┴──────────────────────────────────────────────────────┐
  │                                                              │
  │  P0 ──────────────▶  Inventory Instance 1 (group: "inv-svc") │
  │                       reads: msg1, msg4                      │
  │                                                              │
  │  P1 ──────────────▶  Inventory Instance 2 (group: "inv-svc") │
  │                       reads: msg2, msg5                      │
  │                                                              │
  │  P2 ──────────────▶  Inventory Instance 3 (group: "inv-svc") │
  │                       reads: msg3, msg6                      │
  └──────────────────────────────────────────────────────────────┘

  WHY THIS WORKS:
  ─────────────────────────────────────────────────────────────────
  Within a consumer group, Kafka enforces a strict rule:
  each partition is assigned to EXACTLY ONE instance at a time.

  Instance 1 OWNS P0. Only it reads from P0.
  Instance 2 OWNS P1. Only it reads from P1.
  Instance 3 OWNS P2. Only it reads from P2.

  No message is ever processed twice.
  All three instances run in parallel → 3× the throughput.
  ─────────────────────────────────────────────────────────────────

  WHAT HAPPENS WHEN AN INSTANCE CRASHES:
  ─────────────────────────────────────────────────────────────────
  Instance 2 dies.

  Kafka detects the failure (no heartbeat within 45 seconds).
  Triggers a REBALANCE — reassigns partitions automatically.

  New assignment:
    Instance 1 → P0 + P1   (absorbs the dead instance's partition)
    Instance 3 → P2

  Instance 1 resumes P1 from the last COMMITTED offset.
  Nothing is lost. No manual intervention needed.
  ─────────────────────────────────────────────────────────────────
```

## Both Cases Running Simultaneously

In a real system, you use both patterns at the same time on the same topic. This is not a contradiction — it is the design.

```
  REAL SYSTEM: Both patterns on the same topic

  Topic: "order.placed"  (6 partitions)

  ┌──────────────────────────────────────────────────────────────┐
  │  P0  P1  P2  P3  P4  P5  ← all 6 partitions hold messages    │
  └──────────────────────────────────────────────────────────────┘
                    │
       ┌────────────┼─────────────────────────────────────┐
       │            │                                     │
       ▼            ▼                                     ▼
  ┌────────────────────────┐            ┌─────────────────────────┐
  │  Consumer Group:       │            │  Consumer Group:        │
  │  "inventory-svc"       │            │  "analytics-svc"        │
  │                        │            │                         │
  │  I1: P0, P1            │            │  A1: P0, P1, P2         │
  │  I2: P2, P3            │            │  A2: P3, P4, P5         │
  │  I3: P4, P5            │            │                         │
  │                        │            │  Both analytics         │
  │  3 workers split       │            │  instances get ALL      │
  │  the work. Together    │            │  messages between       │
  │  they process ALL 6    │            │  them (same result      │
  │  partitions.           │            │  as one, but faster).   │
  └────────────────────────┘            └─────────────────────────┘

  The two groups are completely invisible to each other.
  Inventory consuming P0 has zero effect on Analytics reading P0.
  They each maintain separate offset bookmarks per partition.
```

## The Consumer Group Lifecycle

When instances start, crash, or scale, Kafka automatically redistributes partitions. This process is called a **rebalance**.

```
  CONSUMER GROUP LIFECYCLE

  ═════════════════════════════════════════════════════
  STEP 1: One instance starts
  ═════════════════════════════════════════════════════

  Topic has 3 partitions.  Group: "inventory-svc"

  Instance 1 joins the group.
  Kafka assigns ALL partitions to the only instance.

  Instance 1 → P0, P1, P2

  ═════════════════════════════════════════════════════
  STEP 2: Two more instances start (scale up)
  ═════════════════════════════════════════════════════

  Instance 2 joins → REBALANCE triggered
  Instance 3 joins → REBALANCE triggered

  New assignment:
    Instance 1 → P0
    Instance 2 → P1
    Instance 3 → P2

  Each reads one partition. Maximum parallelism. ✅

  ═════════════════════════════════════════════════════
  STEP 3: Instance 2 crashes
  ═════════════════════════════════════════════════════

  Kafka detects missing heartbeat → REBALANCE triggered.

  New assignment:
    Instance 1 → P0, P1   ← absorbs P1 from dead Instance 2
    Instance 3 → P2

  Resumes from last committed offset. Nothing lost. ✅

  ═════════════════════════════════════════════════════
  CEILING: What if instances > partitions?
  ═════════════════════════════════════════════════════

  3 partitions, 5 instances, same group_id:

  Instance 1 → P0  ✅  (reading)
  Instance 2 → P1  ✅  (reading)
  Instance 3 → P2  ✅  (reading)
  Instance 4 → ---  ⏸  (idle — no partition available)
  Instance 5 → ---  ⏸  (idle — no partition available)

  Instances 4 and 5 sit as warm standbys.
  If Instance 1 crashes, Instance 4 immediately picks up P0.
  Maximum active parallelism = number of partitions.
  To increase it, increase the partition count first.
```

## The Offset — Your Bookmark

The offset is how Kafka knows where each consumer group has read up to. It is just a number — the position of the last successfully processed message within a partition.

```
  OFFSET IN ACTION

  Partition 0 of topic "order.placed":
  ┌──────┬──────┬──────┬──────┬──────┬──────┐
  │ msg0 │ msg1 │ msg2 │ msg3 │ msg4 │ msg5 │
  └──────┴──────┴──────┴──────┴──────┴──────┘
    0      1      2      3      4      5

  Group "inventory-svc" has processed up to offset 3.
  Their bookmark says: committed offset = 3

  Group "analytics-svc" has processed up to offset 1.
  Their bookmark says: committed offset = 1

  Both groups read from the SAME partition.
  Both keep SEPARATE bookmarks.
  Reading msg0 in group "inv" does NOT remove it for group "analytics".

  ─────────────────────────────────────────────────────────────
  What happens when inventory-svc instance crashes:

  Last committed offset for P0 = 3
  → On restart, instance reads from offset 4
  → msg4 and msg5 are redelivered and processed ✅
  → msg0, msg1, msg2, msg3 are NOT reprocessed (already committed)
  ─────────────────────────────────────────────────────────────

  Kafka stores all committed offsets in an internal topic:
  __consumer_offsets

  group_id: "inventory-svc", topic: "order.placed", P0 → offset 3
  group_id: "inventory-svc", topic: "order.placed", P1 → offset 7
  group_id: "analytics-svc", topic: "order.placed", P0 → offset 1
  (and so on for every group × every partition)
```

```
  │            │             │ P0: [A][D][G]...   │◀──────── group "inv-svc"
  │  key:      │   key:      │ P1: [B][E][H]...   │◀──────── group "inv-svc"
  │  "ord-B" ──┼────────────▶│ P2: [C][F][I]...   │◀──────── group "inv-svc"
  │            │   key:      │                    │
  │  key:      │  "ord-C" ──▶│                    │◀──────── group "notif-svc"
  │  "ord-C"   │             │   (also reads      │◀──────── group "notif-svc"
  └────────────┘             │    ALL messages,   │◀──────── group "notif-svc"
                             │    independently)  │
                             └────────────────────┘

  ─────────────────────────────────────────────────────────────────────────
  Summary:

  group "inv-svc" has 3 instances → each owns 1 partition → splits work
  group "notif-svc" has 3 instances → each owns 1 partition → splits work

  Both groups get ALL messages, but each group splits the processing
  work internally across its instances.

  ─────────────────────────────────────────────────────────────────────────
  The two questions answered:

  Q: How do I make ALL services get ALL messages?
  A: Give each service a DIFFERENT group_id.

  Q: How do I make one service split the work across its instances?
  A: Give all instances of that service the SAME group_id.
  ─────────────────────────────────────────────────────────────────────────
```

## Summary — Three Concepts, Plain English

**What is a partition?** It is a numbered lane inside a topic. Messages go in one end and pile up in order. It is the unit of parallelism — more lanes means more workers can read simultaneously.

**What is a `group_id`?** It is the name written on your service's mailbox. Kafka uses it to decide whether to broadcast a full copy of all messages or to split messages between instances. Same name = share the work. Different name = each gets everything.

**What is a consumer group?** It is all the instances of one service that share the same `group_id`. Kafka coordinates them automatically — assigning one partition to each instance, rebalancing when instances join or leave, tracking each group's reading progress independently from every other group.
