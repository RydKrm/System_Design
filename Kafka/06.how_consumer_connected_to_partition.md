# Four Questions Before Writing Kafka in Go

---

## Question 1 — When I Create a Topic, Can I Declare the Partition Count?

Yes, absolutely. The partition count is something you declare at the moment you create the topic, and it is one of the most consequential decisions you make because, as discussed in the previous guide, partitions cannot be easily changed later without disrupting the key-to-partition routing.

There are three ways to create a topic with a specific partition count.

The first way is the **Kafka command-line tool**, which ships with every Kafka installation. You run it directly on the server or from any machine that can reach the broker.

```
  CREATING A TOPIC WITH THE CLI

  kafka-topics.sh \
    --bootstrap-server localhost:9092 \
    --create \
    --topic order.placed \
    --partitions 6 \
    --replication-factor 3

  This creates:
  ┌─────────────────────────────────────────────────────────┐
  │  Topic: "order.placed"                                  │
  │                                                         │
  │  P0 ──── will be distributed across 3 brokers           │
  │  P1 ──── each partition has 3 copies (RF=3)             │
  │  P2 ──── one copy is the leader (serves reads/writes)   │
  │  P3 ──── two copies are followers (silent backups)      │
  │  P4                                                     │
  │  P5                                                     │
  └─────────────────────────────────────────────────────────┘

  --partitions 6       → 6 independent queues inside this topic
  --replication-factor 3 → 3 copies of every partition

  To inspect what you created:
  kafka-topics.sh --bootstrap-server localhost:9092 \
    --describe --topic order.placed

  Output:
  Topic: order.placed   Partitions: 6   Replication factor: 3
  Partition: 0  Leader: Broker2  Replicas: 2,3,1  Isr: 2,3,1
  Partition: 1  Leader: Broker1  Replicas: 1,2,3  Isr: 1,2,3
  Partition: 2  Leader: Broker3  Replicas: 3,1,2  Isr: 3,1,2
  ... and so on
```

The second way is through **Kafka's Admin API in Go**, which you will use directly in your microservices code. This is the right approach for production: your service creates the topics it needs at startup, which means your infrastructure is defined in code, is version-controlled, and does not require manual CLI commands on the server.

```
  CREATING A TOPIC PROGRAMMATICALLY (Go — preview before the full guide)

  adminClient, _ := kafka.NewAdminClient(&kafka.ConfigMap{
      "bootstrap.servers": "localhost:9092",
  })

  adminClient.CreateTopics(ctx, []kafka.TopicSpecification{
      {
          Topic:             "order.placed",
          NumPartitions:     6,      ← you declare this
          ReplicationFactor: 3,      ← and this
      },
  })

  Kafka receives this request, creates the topic on the broker(s),
  and from this moment forward the topic exists with exactly
  6 partitions. Producers can now write to it. Consumers can subscribe.
```

The third way is **auto-creation**, where Kafka creates the topic automatically the first time a producer or consumer references a name that does not exist yet. This is controlled by the broker config `auto.create.topics.enable=true` (the default). When auto-creation happens, Kafka uses the broker's default partition count (`num.partitions`, default 1) and default replication factor — which are almost never the values you actually want. Auto-creation is convenient for local development but is universally disabled in production, because a typo in a topic name would silently create a new single-partition topic instead of raising an error.

```
  WHAT HAPPENS WHEN auto.create.topics.enable = true (dangerous)

  Your producer tries to write to "order.plaaced" (typo).

  Kafka thinks: "I have never seen this topic."
  Kafka creates it automatically with:
    partitions = 1  (broker default)
    replication = 1  (broker default)

  Your producer happily writes to the wrong topic with 1 partition.
  Your consumer reads from "order.placed" (correct spelling).
  Consumer receives nothing. Producer sees no error.
  You spend 45 minutes wondering what is wrong. 😔

  Production rule: always disable auto-creation.
  Always create topics explicitly with the partition count you intend.
```

## Question 2 — Can a Consumer Read from a Specific Partition, or Does Kafka Decide?

Both are possible, and understanding the difference between them is important before you write any consumer code, because they are used for completely different purposes.

The first and by far the most common approach is **letting Kafka decide** — you subscribe to a topic by name, Kafka assigns you partitions based on your group membership and the rebalancing algorithm, and you read whatever partitions Kafka gives you. This is what the `Subscribe()` API does. You say "I want messages from the order.placed topic" and Kafka decides which partitions this consumer instance owns.

```
  APPROACH 1: Subscribe — Kafka decides (normal usage)

  consumer.Subscribe(["order.placed"], nil)
  ↑ You name the topic. Kafka assigns partitions.

  Group "inv-svc" has 3 instances. Topic has 6 partitions.

  Kafka's Group Coordinator assigns:
    Instance 1 → P0, P1
    Instance 2 → P2, P3
    Instance 3 → P4, P5

  When Instance 2 dies:
    Kafka reassigns P2, P3 to Instance 1 and Instance 3.
    You do nothing. It is automatic.

  When you add Instance 4:
    Kafka rebalances. All four get roughly equal partitions.
    You do nothing. It is automatic.

  ✅ Use Subscribe when:
     - You want to scale horizontally
     - You want automatic failure recovery
     - You do not care which specific partition you read
     - This is 99% of real-world consumer code
```

The second approach is **manual partition assignment** — you tell Kafka exactly which partition you want to read, bypassing the group coordination system entirely. This is what the `Assign()` API does. When you use Assign, Kafka does not consider your group_id for partition assignment at all. You are directly claiming a partition.

```
  APPROACH 2: Assign — You decide (advanced / special cases)

  consumer.Assign([]kafka.TopicPartition{
      {Topic: &"order.placed", Partition: 2},  ← exactly partition 2
  })

  This consumer reads ONLY from Partition 2.
  No group coordination. No rebalancing. No automatic recovery.

  If this consumer crashes, nobody takes over Partition 2.
  You must handle failover manually in your code.

  ✅ Use Assign when:
     - You are building a tool to replay specific partitions
     - You are reading from a specific offset for debugging
     - You are building Kafka Streams (internal, managed by framework)
     - You need to start from a precise offset for data migration
     - This is rare — maybe 1% of real-world consumer code

  ⚠️  Do NOT use Assign for normal microservice consumers.
      You lose all the automatic scaling and failover benefits.
```

There is also a third hybrid: **reading from a specific offset within a subscribed partition**. After subscribing normally and receiving a partition assignment, you can tell Kafka "for partition 2, start reading from offset 1000 rather than from where my group left off." This is done with `Seek()`. This is useful for replaying messages, for testing, or for disaster recovery scenarios where you need to re-process a specific window of history.

```
  THE THREE READING MODES COMPARED

  ┌──────────────┬──────────────────────────────┬──────────────────┐
  │   Method     │    Who assigns partition?    │   Use case       │
  ├──────────────┼──────────────────────────────┼──────────────────┤
  │  Subscribe() │  Kafka (automatic, balanced) │  Normal services │
  │  Assign()    │  You  (manual, no rebalance) │  Tools, replay   │
  │  Seek()      │  Kafka assigns, you set      │  Replay from a   │
  │              │  the starting offset manually│  specific point  │
  └──────────────┴──────────────────────────────┴──────────────────┘
```

For all the Go code you are about to write in the next guide — microservice producers and consumers — you will use `Subscribe()`. Kafka decides. You scale. It recovers. You do not think about individual partitions.

## Question 3 — When a New Consumer Joins, Does It Start from the Beginning or from Now?

This is one of the most important configuration decisions you will make, and it is controlled by a single setting on the consumer: `auto.offset.reset`.

First, let us understand the situation precisely. A consumer group has an **offset** — a bookmark for each partition it reads. This bookmark is stored in Kafka. When a consumer reconnects after a crash, it reads its bookmark and resumes from exactly where it left off. This part is unambiguous.

The interesting question is: what happens when the offset bookmark **does not exist yet**? This happens in two situations: a brand new group that has never read this topic before, or an existing group reading a topic it has never subscribed to before.

```
  WHEN DOES "no existing offset" HAPPEN?

  Situation A: Brand new consumer group
  ──────────────────────────────────────────────────────────────
  You deploy a new service. It uses group_id = "new-analytics".
  It subscribes to "order.placed".
  Kafka looks in __consumer_offsets for:
    group "new-analytics" + topic "order.placed" + P0 → offset?
  NOTHING FOUND. This group has never read this topic.
  → auto.offset.reset decides what happens next.

  Situation B: Existing group, new topic
  ──────────────────────────────────────────────────────────────
  Your "inv-svc" group has been running for weeks.
  You add a new topic "shipment.sent" for it to consume.
  Kafka looks for:
    group "inv-svc" + topic "shipment.sent" + P0 → offset?
  NOTHING FOUND for this topic. The group is new to this topic.
  → auto.offset.reset decides what happens next.

  Situation C: Consumer reconnects after crash
  ──────────────────────────────────────────────────────────────
  "inv-svc" was reading P0 and committed offset 452.
  Instance 1 crashes. Comes back online.
  Kafka finds: group "inv-svc" + P0 → offset 452.
  → Resumes from offset 453. auto.offset.reset is NOT consulted.
  → The setting only matters when NO bookmark exists.
```

The `auto.offset.reset` setting has two meaningful values:

```
  auto.offset.reset = "earliest"
  ──────────────────────────────────────────────────────────────
  Start from the VERY FIRST message still in the topic.
  (Kafka keeps messages for the retention period — 7 days by default)

  Topic "order.placed" P0:
  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
  │ m0  │ m1  │ m2  │ m3  │ m4  │ m5  │ m6  │ m7  │ m8  │
  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
   ↑ new consumer starts HERE
   (reads all 9 messages from the beginning)

  ✅ Use "earliest" when:
     - The service needs to process historical events
     - Rebuilding a database from event history
     - First time deploying an analytics service that needs all data
     - You want to replay everything Kafka still has

  ──────────────────────────────────────────────────────────────
  auto.offset.reset = "latest" (the other common choice)
  ──────────────────────────────────────────────────────────────
  Start from the NEXT new message that arrives AFTER you subscribe.
  Ignore everything that already exists in the topic.

  Topic "order.placed" P0:
  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
  │ m0  │ m1  │ m2  │ m3  │ m4  │ m5  │ m6  │ m7  │ m8  │
  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
                                                           ↑ new consumer
                                                    starts HERE
  (reads only m9, m10, m11 ... when they arrive in the future)

  ✅ Use "latest" when:
     - Old messages are irrelevant (e.g. a metrics dashboard)
     - You only care about events from the moment this service launched
     - Processing old messages would cause incorrect side effects
       (e.g. sending emails for orders placed months ago)
```

Now, your specific question: the group already has 5 consumers running, and a 6th consumer joins. What does the 6th consumer read?

```
  YOUR EXACT SCENARIO: 6th consumer joins an active group

  Group "inv-svc" has been running for 2 weeks.
  5 consumers. Topic "order.placed" has 6 partitions.
  Current state:

  Consumer 1 → P0  (committed offset: 88,421)
  Consumer 2 → P1  (committed offset: 91,003)
  Consumer 3 → P2  (committed offset: 87,644)
  Consumer 4 → P3  (committed offset: 90,117)
  Consumer 5 → P4, P5  (both behind — committed: 79,200, 81,900)

  Consumer 6 starts up. group_id = "inv-svc".
  Kafka triggers a REBALANCE.

  New assignment:
  Consumer 1 → P0  (committed offset: 88,421)
  Consumer 2 → P1  (committed offset: 91,003)
  Consumer 3 → P2  (committed offset: 87,644)
  Consumer 4 → P3  (committed offset: 90,117)
  Consumer 5 → P4  (committed offset: 79,200)
  Consumer 6 → P5  ← takes P5 from Consumer 5
                      committed offset for P5: 81,900

  Consumer 6 does NOT start from the beginning.
  Consumer 6 does NOT start from "now" (latest).
  Consumer 6 starts from offset 81,901 — exactly where the
  GROUP LEFT OFF on P5. The bookmark belongs to the GROUP,
  not to the individual instance.

  ─────────────────────────────────────────────────────────────
  KEY INSIGHT: auto.offset.reset is ONLY consulted when
  the group has NO bookmark at all for that partition.
  If the group already has a committed offset, the new
  consumer resumes from that committed offset — always.
  The new instance inherits the group's progress.
  ─────────────────────────────────────────────────────────────

  VISUAL TIMELINE for P5:

  Offset: 0 ─────────────── 81,900 ──────── 81,901 ──────────▶ future
               (old messages)   │         (Consumer 6 starts HERE)
                          last commit by
                          Consumer 5 before
                          rebalance
```

This is actually a beautiful property of Kafka's design. The offset is owned by the **group**, not by any individual consumer instance. An instance is just a temporary worker that holds a partition assignment for a while. When it leaves — by crashing, being scaled down, or being replaced by a new instance — the group's progress is preserved. The next instance to take that partition picks up exactly where the previous one left off. No messages are lost, and no messages are reprocessed simply because an instance was replaced.

## Question 4 — How Does a Consumer Connect to a Topic? How Does Kafka Register It and Start Serving Data?

This is the most mechanical question of the four, and the answer reveals the full sequence of handshakes that happen between a consumer process and the Kafka cluster from the first line of connection code to the first message being read. There are eight distinct steps.

```
  THE FULL CONNECTION AND REGISTRATION SEQUENCE

  ═══════════════════════════════════════════════════════════════
  STEP 1: BOOTSTRAP — find the cluster
  ═══════════════════════════════════════════════════════════════

  Your consumer code has "bootstrap.servers" = "kafka1:9092"

  Consumer process                    Kafka Broker (kafka1)
       │                                      │
       │ ── TCP connection ──────────────────▶│
       │ ── "Who is in this cluster?"────────▶│
       │ ◀── "Brokers: kafka1, kafka2,kafka3" │
       │     "Topic 'order.placed' has:       │
       │      P0 leader: kafka2               │
       │      P1 leader: kafka3               │
       │      P2 leader: kafka1"              │
       │                                      │

  The consumer now has the full cluster metadata map.
  It knows where every partition leader lives.
  It does NOT need kafka1 anymore — it was just a doorman.

  ═══════════════════════════════════════════════════════════════
  STEP 2: FIND GROUP COORDINATOR — locate your group's manager
  ═══════════════════════════════════════════════════════════════

  group_id = "inv-svc"
  Kafka computes: hash("inv-svc") % number_of_brokers = kafka2

  Consumer                            kafka2 (Group Coordinator)
       │                                      │
       │ ── "FindCoordinator for inv-svc" ───▶│
       │ ◀── "I am the coordinator" ──────────│
       │                                      │

  kafka2 is now responsible for managing the "inv-svc" group.
  All membership decisions go through kafka2.

  ═══════════════════════════════════════════════════════════════
  STEP 3: JOIN GROUP — register as a member
  ═══════════════════════════════════════════════════════════════

  Consumer                            kafka2 (Group Coordinator)
       │                                      │
       │ ── JoinGroup request: ──────────────▶│
       │    group_id: "inv-svc"               │
       │    member_id: "" (first time)        │ ← Kafka assigns
       │    subscribed_topics:["order.placed"]│   a member_id
       │    session_timeout: 45000ms          │
       │                                      │
       │    (WAITS — coordinator collects     │
       │     all members before responding)   │
       │                                      │
  Other consumers also send JoinGroup requests.
  Coordinator waits for all members to join.

  ═══════════════════════════════════════════════════════════════
  STEP 4: ELECT LEADER — one member computes the assignment
  ═══════════════════════════════════════════════════════════════

  Coordinator picks one consumer as "Group Leader".
  (Usually the first one to join. Not related to partition leader.)

                                      kafka2 (Group Coordinator)
  Consumer 1 (Group Leader) ◀── JoinGroup response:
                                  "you are the leader"
                                  "members: [C1, C2, C3, C4, C5]"
                                  "available partitions: P0–P5"

  Consumers 2–5 ◀── JoinGroup response:
                     "you are a follower, wait for SyncGroup"

  ═══════════════════════════════════════════════════════════════
  STEP 5: COMPUTE ASSIGNMENT — leader divides the partitions
  ═══════════════════════════════════════════════════════════════

  Consumer 1 (Group Leader) — runs assignment algorithm locally:
    C1 → P0, P1
    C2 → P2, P3
    C3 → P4, P5
    C4 → (no partition — only 3 members? adjust as needed)

  This assignment computation happens IN THE CONSUMER PROCESS,
  not on the broker. The broker does not know about business logic.

  ═══════════════════════════════════════════════════════════════
  STEP 6: SYNC GROUP — distribute the assignment
  ═══════════════════════════════════════════════════════════════

  Consumer 1 (Leader) ── SyncGroup (with full assignment) ──▶ kafka2
  Consumers 2–5       ── SyncGroup (empty — just waiting)  ──▶ kafka2

  kafka2 ◀── receives full assignment from leader
  kafka2 ──▶ sends each consumer ONLY its own assignment:
    C1 ◀── "your partitions: P0, P1"
    C2 ◀── "your partitions: P2, P3"
    C3 ◀── "your partitions: P4, P5"

  ═══════════════════════════════════════════════════════════════
  STEP 7: FETCH OFFSETS — where to start reading
  ═══════════════════════════════════════════════════════════════

  Each consumer now knows which partitions it owns.
  It asks the coordinator: "What is the last committed
  offset for my partitions in group 'inv-svc'?"

  C1 ── OffsetFetch(P0, P1) ──▶ kafka2
  C1 ◀── "P0: offset 88421, P1: offset 91003"

  If no committed offset exists (new group):
  C1 ◀── "P0: no offset (use auto.offset.reset)"

  ═══════════════════════════════════════════════════════════════
  STEP 8: FETCH — start reading messages
  ═══════════════════════════════════════════════════════════════

  C1 knows:
    - It owns P0 and P1
    - P0 leader is on kafka2
    - P1 leader is on kafka3
    - P0 resume from offset 88422
    - P1 resume from offset 91004

  C1 ── FetchRequest(P0, from: 88422) ──▶ kafka2 (P0 leader)
  C1 ── FetchRequest(P1, from: 91004) ──▶ kafka3 (P1 leader)

  kafka2 ◀── returns [msg88422, msg88423, msg88424, ...]
  kafka3 ◀── returns [msg91004, msg91005, ...]

  Your handler function receives messages.
  The poll loop begins. ✅
```

After Step 8, two things run continuously in parallel:

```
  WHAT RUNS CONTINUOUSLY AFTER CONNECTION

  ┌────────────────────────────────────────────────────────────┐
  │  THREAD 1: Poll Loop (your code drives this)               │
  │                                                            │
  │  loop:                                                     │
  │    msg = consumer.ReadMessage(300ms timeout)               │
  │    handler(msg)                                            │
  │    consumer.CommitMessage(msg) ← advances the bookmark     │
  │                                                            │
  └────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────┐
  │  THREAD 2: Heartbeat (Kafka client library handles this)   │
  │                                                            │
  │  Every heartbeat.interval.ms (default: 3 seconds):         │
  │  consumer ── Heartbeat("I am alive") ──▶ Group Coordinator │
  │  coordinator ◀── "ok, your assignment still valid"         │
  │                                                            │
  │  If heartbeat stops for session.timeout.ms (default: 45s): │
  │  coordinator declares consumer dead → triggers REBALANCE   │
  │                                                            │
  └────────────────────────────────────────────────────────────┘

  The heartbeat is why your consumer must not block the poll loop
  for too long. If your handler takes 50 seconds on one message,
  the heartbeat thread cannot run, the coordinator thinks you died,
  and triggers a rebalance — kicking you out of the group even
  though you are still alive. This is a common production bug.
```

Here is what the topic "sees" from its perspective (topics do not actually "see" anything — Kafka brokers do — but this mental model helps):

```
  HOW KAFKA TRACKS A CONSUMER GROUP FROM THE BROKER'S PERSPECTIVE

  Before any consumer exists:
  ─────────────────────────────────────────────────────
  Topic "order.placed":
    P0: messages exist, no reader
    P1: messages exist, no reader
    P2: messages exist, no reader

  Group "inv-svc" does not exist in __consumer_offsets.

  ─────────────────────────────────────────────────────
  After 3 consumers connect (Steps 1–8 above):
  ─────────────────────────────────────────────────────
  Group coordinator (kafka2) state:
    group: "inv-svc"
    status: STABLE
    members:
      C1 [last heartbeat: 0.3s ago] → P0, P1
      C2 [last heartbeat: 0.1s ago] → P2, P3
      C3 [last heartbeat: 0.8s ago] → P4, P5

  __consumer_offsets topic:
    inv-svc / order.placed / P0 → 88422
    inv-svc / order.placed / P1 → 91004
    inv-svc / order.placed / P2 → 87645
    ... (one entry per group × partition)

  ─────────────────────────────────────────────────────
  When C3 stops sending heartbeats (crashes):
  ─────────────────────────────────────────────────────
  Coordinator: "C3 heartbeat not received for 45 seconds."
  → Removes C3 from group
  → group status: REBALANCING
  → Sends JoinGroup request to C1 and C2 (forces rejoin)
  → New assignment: C1 → P0, P1, P4   C2 → P2, P3, P5
  → group status: STABLE

  C4's old offsets (P4: 87200, P5: 81900) survive in
  __consumer_offsets. C1 picks up P4 from offset 87201.
  Nothing is lost.
```

## One Final Thing Before You Write the Code

There is one behaviour that catches almost every developer the first time they write a Kafka consumer in Go, so it is worth flagging now.

The confluent-kafka-go library's `ReadMessage()` call does two things simultaneously: it polls for new messages AND it sends heartbeats to the coordinator. If you stop calling `ReadMessage()` — for example, because your handler is taking a long time to process a message — the heartbeat stops, the coordinator thinks you are dead, and triggers a rebalance that kicks you out of the group.

```
  THE MOST COMMON BEGINNER BUG

  ❌ WRONG — blocking the poll loop:

  for {
      msg, _ := consumer.ReadMessage(-1)  // blocks until a message arrives
      slowDatabaseWrite(msg)  // takes 60 seconds
      // Heartbeat never gets sent during those 60 seconds
      // Coordinator thinks you died → REBALANCE → you lose your partition
  }

  ✅ RIGHT — keep the poll loop moving:

  for {
      msg, err := consumer.ReadMessage(300 * time.Millisecond)
      // 300ms timeout: even if no message, returns control here
      // Heartbeat is sent during the timeout wait ✅
      if err != nil { continue }  // timeout, loop again
      go slowDatabaseWrite(msg)   // OR use a worker pool
      consumer.CommitMessage(msg)
  }

  The rule: ReadMessage() must be called frequently.
  Never block the thread that calls it for more than
  max.poll.interval.ms (default: 5 minutes) or you lose your assignment.
```

With these four questions answered and this warning in mind, you are fully prepared to write clean, correct Kafka producer and consumer code in Go.
