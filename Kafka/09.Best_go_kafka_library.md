# franz-go vs confluent-kafka-go

### A Deep Comparison for Large-Scale Distributed Systems

---

## Preface — Why This Choice Matters More Than You Think

Choosing a Kafka client library feels like a small technical decision. It is not. In a large-scale distributed system, your Kafka client is the connective tissue between every service. It touches every message that flows through your platform. Its performance characteristics determine your throughput ceiling. Its rebalancing behaviour determines how much downtime your consumers experience during deployments. Its CGO dependency (or lack of one) determines how complicated your Docker builds and CI pipelines become. Its error handling model determines whether your on-call engineer can understand what went wrong at 3 AM.

This guide compares the two most serious Go Kafka libraries in exhaustive detail, across every dimension that matters for production systems at scale. By the end, you will know which one to choose and, more importantly, why.

---

## Chapter 1 — The Fundamental Architecture Difference

The most important difference between the two libraries is not a feature list item. It is an architectural philosophy that shapes everything else.

`confluent-kafka-go` is a **thin Go wrapper around `librdkafka`**, a mature, battle-tested Kafka client written in C. When you import confluent-kafka-go, you are importing a Go API surface, but the actual Kafka protocol work — TCP connections, batching, compression, retry logic — is all done by C code running inside your Go process via CGO. Go's goroutine scheduler, garbage collector, and network poller cannot see or manage this C code. They share the same process but operate in separate worlds.

`franz-go` is a **ground-up pure Go implementation** of the Kafka client protocol. There is not a single line of C. Every connection, every protocol message, every retry decision is written in Go and runs on Go's scheduler. The Go runtime has full visibility into and control over everything the library does.

```
  ARCHITECTURE COMPARISON

  confluent-kafka-go                    franz-go
  ─────────────────────────────         ─────────────────────────────
  ┌─────────────────────────┐           ┌─────────────────────────┐
  │      Your Go Code       │           │      Your Go Code       │
  └───────────┬─────────────┘           └───────────┬─────────────┘
              │  Go API calls                        │  Go API calls
  ┌───────────▼─────────────┐           ┌───────────▼─────────────┐
  │   confluent Go wrapper  │           │   kgo / kadm packages   │
  │   (thin translation     │           │   (full implementation) │
  │    layer)               │           │                         │
  └───────────┬─────────────┘           │   Go scheduler: ✅ full │
              │  CGO calls              │   control of goroutines │
  ┌───────────▼─────────────┐           │                         │
  │      librdkafka (C)     │           │   GC: ✅ can see all    │
  │                         │           │   memory                │
  │   Go scheduler: ❌      │           │                         │
  │   cannot manage         │           │   Profiling: ✅ pprof  │
  │   C threads             │           │   shows everything      │
  │                         │           └───────────┬─────────────┘
  │   GC: ❌ cannot see     │                       │  native Go
  │   C memory              │                       │  net/tls
  │                         │                       ▼
  │   Profiling: ❌ pprof   │           ┌─────────────────────────┐
  │   misses C activity     │           │  Kafka Brokers          │
  └───────────┬─────────────┘           └─────────────────────────┘
              │  C sockets
              ▼
  ┌─────────────────────────┐
  │  Kafka Brokers          │
  └─────────────────────────┘
```

This architectural difference has cascading consequences across performance, deployability, debugging, and operational behaviour — which the following chapters explore one by one.

## Chapter 2 — Build and Deployment: The CGO Problem at Scale

In a large distributed system, you might have 20 microservices, each with its own Docker image, built by a CI/CD pipeline that runs hundreds of times per day. Every second of build time multiplies across all those builds. Every extra system dependency multiplies across all those environments. This is where the CGO dependency of confluent-kafka-go becomes a real operational tax.

When you use confluent-kafka-go, every machine that builds your code must have `librdkafka-dev` installed. This means your local developer machine, your CI server, your Docker build container, and your production container image all need the C library. In a Docker multi-stage build — the industry standard for producing small Go binaries — you must install the library in both the build stage AND the runtime stage. The runtime container is no longer a minimal scratch or distroless image; it must be a full Linux distribution with a C runtime. Alpine Linux users must use `apk add librdkafka-dev pkgconf` and set `CGO_ENABLED=1`. The binary cannot be cross-compiled from macOS to Linux with a simple `GOOS=linux go build` — you need a Linux host or a Docker build container.

```
  DOCKER BUILD COMPARISON

  confluent-kafka-go Dockerfile:
  ─────────────────────────────────────────────────────────────────
  FROM golang:1.22-alpine AS builder
  # Must install C toolchain AND librdkafka headers
  RUN apk add --no-cache gcc musl-dev librdkafka-dev pkgconf
  WORKDIR /app
  COPY . .
  # CGO must be enabled — cannot build without it
  RUN CGO_ENABLED=1 GOOS=linux go build -o /service .

  # Runtime image CANNOT be scratch — needs C runtime
  FROM alpine:3.19
  # Must install the C shared library in the runtime container too
  RUN apk add --no-cache librdkafka ca-certificates
  COPY --from=builder /service /service
  CMD ["/service"]

  Build time added by C library installation:  ~45–90 seconds
  Runtime image size:  ~25–40 MB (vs ~5 MB for scratch)
  ─────────────────────────────────────────────────────────────────

  franz-go Dockerfile:
  ─────────────────────────────────────────────────────────────────
  FROM golang:1.22-alpine AS builder
  # No extra dependencies. Pure Go builds with nothing extra.
  WORKDIR /app
  COPY . .
  # CGO disabled — fully static binary, cross-compilable anywhere
  RUN CGO_ENABLED=0 GOOS=linux go build -o /service .

  # Runtime image CAN be scratch — no C runtime needed
  FROM scratch
  COPY --from=builder /service /service
  COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
  CMD ["/service"]

  Build time added:     ~0 seconds (standard Go build)
  Runtime image size:   ~8–12 MB
  Cross-compilation:    works from any OS
  ─────────────────────────────────────────────────────────────────

  In a CI/CD pipeline running 200 builds/day across 20 services:
  confluent-kafka-go adds ~90s × 200 = 5 hours of CI time per day.
  franz-go adds ~0s.
```

For a large-scale system where deployment velocity is a competitive advantage, this build time difference compounds significantly over months and years.

## Chapter 3 — Performance: Throughput and Latency at Scale

Both libraries are capable of very high throughput — neither is going to be the bottleneck in a typical microservice. But at extreme scale (millions of messages per second across many partitions), the differences in how they handle batching, compression, and network I/O become measurable.

`librdkafka` (confluent's underlying C library) has been tuned over many years by the Confluent engineering team and has enormous real-world mileage. Its C implementation uses OS-level polling efficiently and has very low overhead per message at the metal level. For raw single-client throughput benchmarks, it tends to perform extremely well.

`franz-go` achieves competitive throughput through a different approach: it uses Go's native `net` package for TCP (which integrates with the Go runtime's netpoller and goroutine scheduler), and it implements batching and protocol encoding more aggressively than confluent. Its `PollFetches` batch-at-a-time model reduces overhead per message when consuming at high rates, because you pay the function call overhead once per batch rather than once per message.

```
  THROUGHPUT CHARACTERISTICS

  Producer throughput (approximate, single client, 1KB messages):
  ─────────────────────────────────────────────────────────────────
  confluent-kafka-go:  ~600,000–900,000 msg/sec  (librdkafka C engine)
  franz-go:            ~500,000–800,000 msg/sec  (pure Go implementation)

  Both are fast enough for virtually all production workloads.
  The difference matters only at extreme scale (>500k msg/sec).

  Consumer latency (end-to-end, 99th percentile):
  ─────────────────────────────────────────────────────────────────
  confluent-kafka-go:  ~2–5ms    (ReadMessage one-at-a-time model)
  franz-go:            ~1–3ms    (PollFetches batch model, less overhead)

  franz-go's batch model reduces per-message overhead significantly
  at high consumer throughput. Fewer syscalls, fewer allocations.

  Memory allocation (GC pressure):
  ─────────────────────────────────────────────────────────────────
  confluent-kafka-go:  C heap invisible to Go GC. Can cause GC pauses
                       when Go GC runs and C heap is large.
  franz-go:            All allocations in Go heap. GC has full
                       visibility. Predictable pause times.
                       More amenable to GOGC tuning.

  CPU profiling:
  ─────────────────────────────────────────────────────────────────
  confluent-kafka-go:  pprof shows gaps — C activity is invisible.
                       Hard to diagnose where CPU is being spent.
  franz-go:            pprof shows complete picture. Every goroutine,
                       every function call visible. Much easier to
                       diagnose and optimise under load.
```

For most real-world services — even at tens of thousands of messages per second — neither library will be the bottleneck. Your database, your business logic, and your network latency will limit throughput long before the Kafka client does. The performance difference only becomes meaningful at hyperscale — think Kafka clusters processing millions of messages per second across thousands of partitions.

## Chapter 4 — Consumer Group Rebalancing: The Most Critical Operational Difference

This is where the two libraries diverge most sharply in a way that directly impacts your system's availability during deployments and scaling operations.

Every time you deploy a new version of a service, Kubernetes performs a rolling update: it starts a new pod before stopping the old one. Each pod joining and leaving the consumer group triggers a rebalance. In a system with frequent deployments — which is the norm for healthy engineering teams — this means frequent rebalances. How a library handles rebalancing determines how much message processing delay your users experience during every deployment.

The traditional rebalancing approach (used by confluent-kafka-go by default) is called **eager rebalancing** or a **stop-the-world rebalance**. When any member joins or leaves the group, ALL members must stop consuming, revoke ALL their partition assignments, re-join the group, wait for a new assignment to be computed, and then resume consuming. During this period — which can last anywhere from 2 to 60 seconds depending on session timeout configuration — no messages are processed by any consumer in the group. Consumer lag grows.

```
  EAGER REBALANCING (confluent-kafka-go default)

  Group "inventory-svc": 3 instances, 6 partitions
  State: I1→P0,P1  I2→P2,P3  I3→P4,P5  (all consuming normally)

  Event: New deployment. I4 starts up.

  ──────────────────────────────────────────────────────────────────
  t=0s    I4 sends JoinGroup request.
  t=0s    COORDINATOR: "REBALANCE TRIGGERED. Everyone stop."

  t=0s    I1: STOPS consuming P0, P1. Revokes all partitions.
  t=0s    I2: STOPS consuming P2, P3. Revokes all partitions.
  t=0s    I3: STOPS consuming P4, P5. Revokes all partitions.

  ← ALL FOUR INSTANCES ARE NOW PROCESSING ZERO MESSAGES ←
  ← CONSUMER LAG IS GROWING ←

  t=2s    All members re-join. New assignment computed:
          I1→P0  I2→P1  I3→P2  I4→P3  (I3 and I2 also share P4,P5)

  t=2s    All instances resume consuming.
  ──────────────────────────────────────────────────────────────────
  Dead window: ~2 seconds (can be up to 45s with slow members)
  Messages accumulated during dead window: potentially thousands
  Deploy a service 10 times/day: up to 20s of dead time daily
  ──────────────────────────────────────────────────────────────────
```

franz-go defaults to **cooperative incremental rebalancing** (KIP-429), which eliminates this stop-the-world pause. When a member joins or leaves, only the partitions that need to physically move are briefly paused. All other partitions keep flowing without interruption.

```
  COOPERATIVE REBALANCING (franz-go default)

  Group "inventory-svc": 3 instances, 6 partitions
  State: I1→P0,P1  I2→P2,P3  I3→P4,P5  (all consuming normally)

  Event: New deployment. I4 starts up.

  ──────────────────────────────────────────────────────────────────
  t=0s    I4 sends JoinGroup. Rebalance triggered.

  Phase 1: Coordinator identifies which partitions need to MOVE.
           Decision: P1 moves from I1 to I4.
                     P3 moves from I2 to I4.
           (I1 keeps P0. I2 keeps P2. I3 keeps P4, P5 unchanged.)

  t=0s    I1: KEEPS consuming P0. ✅ Only revokes P1.
  t=0s    I2: KEEPS consuming P2. ✅ Only revokes P3.
  t=0s    I3: KEEPS consuming P4, P5. ✅ NOT AFFECTED AT ALL.
  t=0s    I4: Takes P1 and P3. Starts consuming.

  ──────────────────────────────────────────────────────────────────
  Dead window: near-zero for P0, P2, P4, P5 (never paused)
               ~100ms for P1, P3 (only these two were revoked)
  Messages accumulated: negligible
  Deploy 10 times/day: effectively zero downtime
  ──────────────────────────────────────────────────────────────────

  THE DIFFERENCE IN PRODUCTION:

  confluent (eager):
  ┌────────────────────────────────────────────────┐
  │ DEPLOY  ←2s pause→ ALL partitions stopped      │ × 10 deploys/day
  └────────────────────────────────────────────────┘
  Cumulative daily dead window: ~20–600 seconds

  franz-go (cooperative):
  ┌────────────────────────────────────────────────┐
  │ DEPLOY  ←~100ms pause on 2 partitions only→    │ × 10 deploys/day
  └────────────────────────────────────────────────┘
  Cumulative daily dead window: ~1 second
```

It is possible to configure confluent-kafka-go to use cooperative rebalancing by setting `partition.assignment.strategy=cooperative-sticky`. But this is not the default — you must know to set it. With franz-go, cooperative rebalancing is the default. You have to actively opt out of it (which you should never do on a new project).

## Chapter 5 — API Design: Config Strings vs. Typed Options

The two libraries represent opposite philosophies in Go API design, and this affects both development experience and reliability in production.

confluent-kafka-go uses a `ConfigMap` — essentially a `map[string]interface{}` where every configuration property is a string key and the value can be any type. This mirrors librdkafka's own configuration system, which is string-based by nature. The consequence is that configuration mistakes are runtime errors, not compile errors.

```
  CONFLUENT CONFIG: Runtime errors from string typos

  // This compiles perfectly. The error only appears at runtime.
  p, err := kafka.NewProducer(&kafka.ConfigMap{
      "bootstrap.servers":  "localhost:9092",
      "aks":                "all",     // typo: "aks" not "acks"
      "enable.idempotance": true,      // typo: "idempotance" not "idempotence"
      "linger.ms":          "five",    // wrong type: string instead of int
  })
  // err is nil — Kafka accepts the config but "aks" and "idempotance"
  // are silently ignored. Your messages have no acks and no idempotence.
  // You discover this during a broker failure, not during code review.
```

franz-go uses typed option functions. Every configuration option is a Go function that accepts the correct type. A typo in a function name is a compile error. Passing the wrong type to an option is a compile error.

```
  FRANZ-GO CONFIG: Compile errors catch mistakes immediately

  // These would NOT compile — caught by the Go compiler:
  client, err := kgo.NewClient(
      kgo.SeedBrokers("localhost:9092"),
      kgo.RequiredAks(kgo.AllISRAcks()),  // compile error: function doesn't exist
      kgo.ProducerLinger("five"),          // compile error: expects time.Duration
  )

  // The correct code:
  client, err := kgo.NewClient(
      kgo.SeedBrokers("localhost:9092"),
      kgo.RequiredAcks(kgo.AllISRAcks()),  // typed: cannot misspell
      kgo.ProducerLinger(5 * time.Millisecond), // typed: must be Duration
  )

  // Compile-time safety means configuration errors are found during
  // development, not during a 2 AM production incident.
```

```
  API DESIGN COMPARISON

  ┌────────────────────────────┬──────────────────────────┬──────────────────────────┐
  │  Operation                 │  confluent-kafka-go      │  franz-go                │
  ├────────────────────────────┼──────────────────────────┼──────────────────────────┤
  │  Create client             │  kafka.NewProducer(map)  │  kgo.NewClient(opts...)  │
  │  Set acks                  │  "acks": "all"           │  kgo.RequiredAcks(...)   │
  │  Set linger                │  "linger.ms": 5          │  kgo.ProducerLinger(5ms) │
  │  Set group ID              │  "group.id": "svc"       │  kgo.ConsumerGroup("svc")│
  │  Set offset reset          │  "auto.offset.reset":    │  kgo.ConsumeResetOffset( │
  │                            │  "earliest"              │  kgo.NewOffset().AtStart)│
  │  Config typo detection     │  Runtime (silent)        │  Compile time (loud)     │
  │  Type safety               │  None (interface{})      │  Full (typed functions)  │
  │  Autocomplete in IDE       │  Poor (string keys)      │  Excellent (functions)   │
  └────────────────────────────┴──────────────────────────┴──────────────────────────┘
```

## Chapter 6 — Delivery Guarantees and Error Handling

Both libraries support idempotent production and at-least-once delivery, but they expose these guarantees through very different APIs.

In confluent-kafka-go, idempotent production is OFF by default. You must explicitly set `"enable.idempotence": true`. If you forget, retries can produce duplicate messages during network failures. Delivery reports arrive through a separate `Events()` channel that you must drain in a background goroutine — if you forget to drain it, the channel fills up and the producer stalls silently.

In franz-go, idempotent production is ON by default. You have to actively call `kgo.DisableIdempotentWrite()` to turn it off (which you almost never want). Delivery reports arrive through a per-record callback you register at the point of production — there is no separate channel to manage, no goroutine to start, no risk of forgetting.

```
  DELIVERY REPORT COMPARISON

  confluent-kafka-go — requires a separate goroutine:
  ─────────────────────────────────────────────────────────────────
  // Forget to start this goroutine → producer stalls silently
  go func() {
      for e := range p.Events() {
          switch ev := e.(type) {
          case *kafka.Message:
              if ev.TopicPartition.Error != nil {
                  // message failed
              }
          }
      }
  }()
  p.Produce(&kafka.Message{...}, nil) // delivery report on Events()
  ─────────────────────────────────────────────────────────────────

  franz-go — callback per record, no goroutine to manage:
  ─────────────────────────────────────────────────────────────────
  // Delivery report is right here, inline. Cannot forget it.
  client.Produce(ctx, record, func(r *kgo.Record, err error) {
      if err != nil {
          // message failed
      } else {
          // message delivered to partition r.Partition offset r.Offset
      }
  })
  ─────────────────────────────────────────────────────────────────
```

## Chapter 7 — Observability: Debugging in Production

When something goes wrong in a large-scale system — consumer lag spikes, a partition stops flowing, a batch of messages is delayed — your ability to diagnose the issue quickly depends on how observable your Kafka client is.

confluent-kafka-go surfaces metrics through `GetMetadata()` and the `Stats` event type on the Events() channel. These metrics come from librdkafka's internal statistics — comprehensive, but expressed in C library terms. More critically, Go's `pprof` profiler and execution tracer see only the Go side of the client. The C threads in librdkafka are invisible to pprof. If 80% of your CPU is being spent inside librdkafka, pprof shows 80% of CPU as idle in the Go goroutines waiting on CGO calls — deeply misleading.

franz-go is entirely visible to Go's tooling. Every goroutine the library creates appears in `pprof` profiles. Every allocation appears in heap profiles. Every network operation appears in execution traces. When you run `go tool pprof` on a production service using franz-go, you get a complete and accurate picture of exactly where time and memory are being spent.

```
  OBSERVABILITY COMPARISON

  Scenario: Consumer is slow. CPU is high. What is taking time?

  confluent-kafka-go pprof output:
  ─────────────────────────────────────────────────────────────────
  Total: 100%
    45%  runtime.cgocall          ← "something in C is expensive"
    30%  your handler code
    15%  json.Unmarshal
    10%  other Go code

  "Something in C is expensive." That is all pprof can tell you.
  You cannot drill further. You need librdkafka-level tools.
  ─────────────────────────────────────────────────────────────────

  franz-go pprof output:
  ─────────────────────────────────────────────────────────────────
  Total: 100%
    30%  your handler code
    25%  kgo.(*fetchRequest).encode  ← specific function visible
    20%  json.Unmarshal
    15%  snappy.Decode               ← decompression is expensive
    10%  other

  Every function is visible. You can see exactly what franz-go
  is doing and optimise it. In this case: switching to lz4
  compression would halve the decompression overhead.
  ─────────────────────────────────────────────────────────────────
```

## Chapter 8 — Ecosystem, Maturity, and Support

This is where confluent-kafka-go holds a clear advantage that no feature list can overcome.

librdkafka, which powers confluent-kafka-go, was created in 2012. It has been used in production by thousands of companies including LinkedIn, Netflix, Uber, and Airbnb. The Confluent company employs engineers specifically to maintain and improve it. When Kafka releases a new protocol version, Confluent updates librdkafka quickly — it is a strategic priority for them. The bug tracker has been battle-hardened by over a decade of production use across every conceivable environment.

franz-go was created in 2020 by Travis Bischel, an individual developer. It is younger, has fewer production deployments, and has a smaller team behind it. However, it is widely considered the most correct pure Go Kafka implementation — its protocol test suite is exceptionally thorough, it passes Kafka's official compatibility test suite, and its GitHub issue tracker reflects a library that takes correctness extremely seriously.

```
  MATURITY AND ECOSYSTEM COMPARISON

  ┌────────────────────────────┬──────────────────────────┬──────────────────────────┐
  │  Dimension                 │  confluent-kafka-go      │  franz-go                │
  ├────────────────────────────┼──────────────────────────┼──────────────────────────┤
  │  Underlying library age    │  2012 (librdkafka)       │  2020 (pure Go)          │
  │  Go wrapper age            │  2016                    │  2020                    │
  │  Maintained by             │  Confluent (company)     │  Individual + community  │
  │  GitHub stars (2024)       │  ~4,200                  │  ~2,900                  │
  │  Known production users    │  Thousands (all sizes)   │  Growing (many startups) │
  │  Kafka protocol coverage   │  Complete (librdkafka)   │  Complete (pure Go)      │
  │  Schema Registry support   │  Official client exists  │  Community package       │
  │  Kafka Streams equivalent  │  N/A (use Kafka Streams) │  N/A                     │
  │  Cloud provider support    │  Official on Confluent   │  Community maintained    │
  │  Long-term commitment risk │  Low (company-backed)    │  Moderate (individual)   │
  └────────────────────────────┴──────────────────────────┴──────────────────────────┘
```

The maturity difference is real and should not be dismissed. If your company has a strict vendor support policy, if you work in a regulated industry that requires software supply chain guarantees, or if you need Schema Registry integration with an official client, confluent-kafka-go is the safer choice for that reason alone.

## Chapter 9 — The Decision Framework

After understanding all the dimensions of comparison, here is a structured framework for making the choice in your specific situation.

```
  DECISION FRAMEWORK

  ══════════════════════════════════════════════════════════════
  Choose confluent-kafka-go IF:
  ══════════════════════════════════════════════════════════════

  ✅  Your organisation has a formal vendor support requirement
      and needs Confluent's commercial support contract.

  ✅  You need the official Confluent Schema Registry Go client
      (confluent's schema registry and confluent-kafka-go share
      the same serialisation format out of the box).

  ✅  You are running on a platform where CGO is already the norm
      and you have no containerisation constraints.

  ✅  You need the very highest single-client throughput and your
      use case is in the extreme range (>800k msg/sec per process).

  ✅  Your team is already deeply familiar with librdkafka's
      configuration properties and does not want to re-learn
      an API surface.

  ✅  You need the absolute maximum ecosystem certainty and are
      willing to pay the CGO tax in exchange for the longest
      production track record available.

  ══════════════════════════════════════════════════════════════
  Choose franz-go IF:
  ══════════════════════════════════════════════════════════════

  ✅  You are building a new system from scratch and want the
      best modern Go-native experience.

  ✅  You run on Docker/Kubernetes and want simple, fast builds
      with static binaries and minimal container images.

  ✅  You deploy frequently (>5 times/day) and cannot afford
      stop-the-world rebalances on every deployment.

  ✅  You want compile-time safety for all Kafka configuration
      properties — no more silent string typos.

  ✅  Your engineering team cares about pprof-visible profiling
      and complete observability into the Kafka client.

  ✅  You want cooperative incremental rebalancing ON by default
      without having to remember to configure it.

  ✅  You want a fully static Go binary with CGO_ENABLED=0 and
      scratch-based container images.

  ✅  Your throughput is under ~500k msg/sec per service instance
      (which covers the vast majority of real-world services).
```

The most important thing to understand is that both libraries will carry you to large-scale production successfully. There is no wrong choice between these two — only the wrong choice for your specific constraints. A team that chooses confluent-kafka-go and configures it correctly with `cooperative-sticky` rebalancing, explicit idempotence, and a proper delivery report goroutine will build a perfectly reliable system. A team that chooses franz-go and accepts that it is maintained by a smaller community will also build a perfectly reliable system — with simpler builds, better Go tooling integration, and zero-downtime rebalancing by default.

The decision is less about which is technically superior and more about which set of trade-offs fits your team's context, your deployment environment, and your operational constraints.
