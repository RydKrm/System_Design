## Preface — What We Are Building

This guide builds Kafka's producer and consumer from scratch in Go, explaining every single configuration key and every single line of code as if you are reading it for the first time. Nothing is assumed. Nothing is glossed over.

By the end you will have four things: a topic creator, a producer, a consumer using `Subscribe()` (Kafka assigns partitions), and a consumer using `Assign()` (you pick the partition manually). You will understand not just what each line does, but why it exists — connecting every piece of code back to the theory you have already learned.

The library used throughout is `confluent-kafka-go/v2`, which is a Go wrapper around `librdkafka` — the most battle-tested Kafka client library in existence. Every configuration key you set in Go maps directly to a `librdkafka` configuration property.

## Part 2 — Shared Configuration (`config/kafka.go`)

```go
package config

// KafkaConfig holds every connection and behaviour setting
// that ALL parts of the system — producer, consumer, admin —
// need to share. Centralising this means you change the broker
// address in one place, not scattered across five files.
type KafkaConfig struct {
    // Brokers is a comma-separated list of host:port pairs.
    // You only need ONE broker here — Kafka will tell your
    // client about all the other brokers automatically through
    // the bootstrap metadata exchange (Step 1 from Question 4).
    // In production, list 2–3 so a single broker restart does
    // not prevent startup.
    Brokers string

    // GroupID is the consumer group identity discussed at length
    // in the previous guides. Same GroupID across instances =
    // they share the work. Different GroupID = each gets all messages.
    GroupID string

    // ClientID is a human-readable name for this specific process.
    // It appears in Kafka broker logs and monitoring dashboards,
    // making it easy to see which service is producing/consuming
    // what. It does NOT affect partitioning or group behaviour —
    // it is purely for observability.
    ClientID string
}

// NewKafkaConfig returns a config populated with the values
// most commonly used in development. In production you would
// read these from environment variables or a config file.
func NewKafkaConfig(brokers, groupID, clientID string) *KafkaConfig {
    return &KafkaConfig{
        Brokers:  brokers,
        GroupID:  groupID,
        ClientID: clientID,
    }
}

// Topic name constants — always use constants, never raw strings.
// A typo in a topic name string creates a silent bug that is very
// hard to find. A typo in a constant name is a compile error that
// Go catches immediately.
const (
    TopicOrderPlaced       = "order.placed"
    TopicInventoryReserved = "inventory.reserved"
    TopicInventoryFailed   = "inventory.failed"
    TopicNotification      = "notification.send"
    TopicUserRegistered    = "user.registered"
)
```

## Part 3 — Creating Topics Programmatically (`topic/admin.go`)

This answers your first question directly: yes, you declare the partition count when you create the topic, and here is exactly how you do it in Go.

```go
package topic

import (
    "context"
    "fmt"
    "log"
    "time"

    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
)

// TopicSpec defines everything needed to create one Kafka topic.
// We use our own struct rather than the library's directly so
// callers do not need to import the kafka library just to declare
// a topic — only the admin package needs that dependency.
type TopicSpec struct {
    Name              string            // the topic name, e.g. "order.placed"
    NumPartitions     int               // how many partitions to create
    ReplicationFactor int               // how many copies of each partition
    Config           map[string]string  // optional per-topic overrides
}

// AdminClient wraps confluent's AdminClient to keep our code
// clean and add structured logging around every operation.
type AdminClient struct {
    client *kafka.AdminClient
}

// NewAdminClient creates a connection to the Kafka cluster
// for administrative operations — creating and inspecting topics.
// This is separate from the producer and consumer clients because
// administrative operations use a different API surface.
func NewAdminClient(brokers string) (*AdminClient, error) {
    client, err := kafka.NewAdminClient(&kafka.ConfigMap{
        // "bootstrap.servers" is the single most important config key
        // in any Kafka client. It tells the library where to find at
        // least one broker to start the metadata exchange with.
        // After the first connection, the broker returns a complete
        // map of the cluster — all other broker addresses, which
        // broker is the leader for each partition, etc.
        // You never need to list ALL brokers here, just enough to
        // guarantee at least one is reachable on startup.
        "bootstrap.servers": brokers,
    })
    if err != nil {
        // If we cannot connect to Kafka at all, there is no point
        // continuing. This is a fatal startup error.
        return nil, fmt.Errorf("failed to create admin client: %w", err)
    }
    log.Printf("[Admin] Connected to Kafka at %s", brokers)
    return &AdminClient{client: client}, nil
}

// EnsureTopics creates each topic in the list if it does not
// already exist. It is safe to call this every time your service
// starts up — if the topic already exists, Kafka simply returns
// a "topic already exists" error which we ignore intentionally.
// This pattern is called "ensure" or "idempotent create" and is
// the standard production approach.
func (a *AdminClient) EnsureTopics(specs []TopicSpec) error {
    // Convert our spec structs into the library's TopicSpecification.
    kafkaSpecs := make([]kafka.TopicSpecification, len(specs))
    for i, spec := range specs {
        kafkaSpecs[i] = kafka.TopicSpecification{
            // Topic is the name of the topic to create.
            // Kafka topic names can contain letters, numbers,
            // dots, underscores, and hyphens.
            // Convention: use dots to separate domain.event
            // e.g. "order.placed", "user.registered"
            Topic: spec.Name,

            // NumPartitions is the answer to your first question.
            // This is where you declare how many parallel lanes
            // this topic will have. Choose this based on:
            //   - Your target throughput
            //   - The maximum number of consumer instances you
            //     ever plan to run for this topic
            // You cannot easily reduce partitions later.
            // It is safer to start higher than you think you need.
            NumPartitions: spec.NumPartitions,

            // ReplicationFactor controls how many brokers keep
            // a copy of each partition. Must be <= broker count.
            // Development: 1 (you only have one broker)
            // Production:  3 (survive loss of 1 broker)
            ReplicationFactor: spec.ReplicationFactor,

            // Config lets you override broker defaults for this
            // specific topic. Examples:
            //   "retention.ms":     "604800000"  (7 days in ms)
            //   "cleanup.policy":   "compact"    (log compaction)
            //   "min.insync.replicas": "2"       (durability)
            Config: spec.Config,
        }
    }

    // CreateTopics sends the creation request to the cluster.
    // We use a context with a timeout so we do not hang forever
    // if Kafka is temporarily unreachable at startup.
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    results, err := a.client.CreateTopics(ctx, kafkaSpecs)
    if err != nil {
        return fmt.Errorf("CreateTopics request failed: %w", err)
    }

    // CreateTopics returns one result per topic — each result
    // has its own error (or nil if success). We must check each
    // one individually because some may succeed and some fail.
    for _, result := range results {
        if result.Error.Code() == kafka.ErrTopicAlreadyExists {
            // This is not a real error — the topic exists, which
            // is exactly what we want. Log it and move on.
            log.Printf("[Admin] Topic already exists: %s", result.Topic)
            continue
        }
        if result.Error.Code() != kafka.ErrNoError {
            return fmt.Errorf("failed to create topic %s: %v",
                result.Topic, result.Error)
        }
        log.Printf("[Admin] ✅ Topic created: %s (partitions=%d)",
            result.Topic, 0) // Kafka doesn't return partition count here
    }
    return nil
}

// Close releases the admin client's connection to the cluster.
// Always defer this after creating an AdminClient.
func (a *AdminClient) Close() {
    a.client.Close()
    log.Println("[Admin] Admin client closed")
}

// DescribeTopic prints detailed partition and replica information
// for a topic. Useful for debugging and verification after creation.
func (a *AdminClient) DescribeTopic(topicName string) error {
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    // We use GetMetadata to inspect what Kafka knows about a topic.
    // The second argument (true) means "only fetch metadata for the
    // named topic", not for the entire cluster.
    metadata, err := a.client.GetMetadata(&topicName, false, 10000)
    if err != nil {
        return fmt.Errorf("failed to get metadata for %s: %w", topicName, err)
    }
    _ = ctx

    topic, ok := metadata.Topics[topicName]
    if !ok {
        return fmt.Errorf("topic %s not found", topicName)
    }

    log.Printf("[Admin] Topic: %s  Partitions: %d",
        topicName, len(topic.Partitions))

    for _, p := range topic.Partitions {
        log.Printf("[Admin]   Partition %d | Leader: Broker %d | "+
            "Replicas: %v | ISR: %v",
            p.ID, p.Leader, p.Replicas, p.Isrs)
    }
    return nil
}
```

## Part 4 — The Producer (`producer/producer.go`)

```go
package producer

import (
    "encoding/json"
    "fmt"
    "log"
    "time"

    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
    "github.com/google/uuid"
)

// Producer wraps confluent's producer and adds structured
// logging, serialisation, and a clean publish API so that
// service code never touches the kafka library directly.
type Producer struct {
    p           *kafka.Producer // the underlying confluent producer
    serviceName string          // used in log lines for traceability
}

// NewProducer creates and configures a Kafka producer.
// Read every config key below carefully — each one controls a
// specific durability, performance, or reliability behaviour.
func NewProducer(brokers, serviceName string) (*Producer, error) {
    p, err := kafka.NewProducer(&kafka.ConfigMap{

        // ── CONNECTION ────────────────────────────────────────────
        // "bootstrap.servers": same as admin client — the entry point
        // into the cluster. The producer fetches cluster metadata
        // (all brokers, all partition leaders) from here on startup.
        "bootstrap.servers": brokers,

        // "client.id": a human-readable label for this producer
        // instance. Appears in broker logs and Kafka UI.
        // Helps you answer "which service is producing to this topic?"
        // during an incident.
        "client.id": serviceName + "-producer",

        // ── DURABILITY (acks) ─────────────────────────────────────
        // "acks" is the most important producer durability setting.
        // It controls how many brokers must confirm receipt of a
        // message before the producer considers it "sent".
        //
        // acks=0  → fire and forget. Producer does not wait for any
        //           confirmation. Fastest possible. If the broker
        //           crashes the moment the message arrives, it is
        //           gone. Use for metrics, logs where loss is OK.
        //
        // acks=1  → leader confirms. Producer waits for the partition
        //           LEADER to write the message to its local log.
        //           If the leader crashes BEFORE followers replicate,
        //           the message is lost. Common middle ground.
        //
        // acks=all → all in-sync replicas confirm. The leader waits
        //            until every broker in the ISR has written the
        //            message before sending the producer an ack.
        //            Combined with min.insync.replicas=2, this gives
        //            you zero data loss guarantee. Slowest but safest.
        //            Use for financial transactions, order events,
        //            anything where losing a message is unacceptable.
        "acks": "all",

        // ── RELIABILITY (retries) ─────────────────────────────────
        // "retries": how many times to retry a failed send before
        // giving up and returning an error. Network blips, leader
        // elections, and brief broker restarts all cause transient
        // send failures. Retrying handles these transparently.
        // Setting this to a large number (or math.MaxInt32) means
        // the producer will keep trying for the duration set by
        // "message.timeout.ms" below.
        "retries": 5,

        // "retry.backoff.ms": how long to wait between retry attempts.
        // Without backoff, retries hammer the broker immediately and
        // can make the situation worse. 100ms is a sensible default.
        "retry.backoff.ms": 100,

        // "message.timeout.ms": the total time (in milliseconds) a
        // message can spend waiting to be delivered — including all
        // retry attempts. If a message is not successfully delivered
        // within this window, it is failed. 30 seconds is generous
        // enough for most transient failures to resolve.
        "message.timeout.ms": 30000,

        // ── IDEMPOTENCE ──────────────────────────────────────────
        // "enable.idempotence": when true, the producer assigns each
        // message a sequence number. If a retry sends the same
        // message twice (because the first ack was lost in transit),
        // the broker uses the sequence number to detect the duplicate
        // and silently discard it. Without this, retries can produce
        // duplicate messages. With this, retries are safe.
        //
        // IMPORTANT: enable.idempotence=true REQUIRES acks=all.
        // It also automatically sets retries to a large value if
        // you have not set it yourself. It is the right default
        // for any production producer.
        "enable.idempotence": true,

        // ── PERFORMANCE (batching) ────────────────────────────────
        // "linger.ms": how long the producer waits before sending a
        // batch, even if the batch is not full.
        //
        // linger.ms=0 → send immediately. One message = one request.
        //               Lowest latency. Lowest throughput.
        //
        // linger.ms=5 → wait up to 5ms for more messages to accumulate.
        //               In 5ms, a busy producer can accumulate hundreds
        //               of messages and send them all in one network
        //               round trip. 5–20ms is the production sweet spot.
        //               A small latency cost for a large throughput gain.
        "linger.ms": 5,

        // "batch.size": maximum size of a single batch in bytes.
        // The producer sends a batch when EITHER linger.ms expires
        // OR the batch reaches batch.size — whichever comes first.
        // 65536 bytes (64 KB) is the common production default.
        // Increase to 512KB–1MB for very high throughput scenarios.
        "batch.size": 65536,

        // ── COMPRESSION ──────────────────────────────────────────
        // "compression.type": compress batches before sending.
        // Reduces network bandwidth and broker disk usage.
        //
        // none    → no compression. Fastest CPU. Highest bandwidth.
        // gzip    → best ratio, slowest CPU. Good for low-throughput.
        // snappy  → good ratio, fast CPU. Google's algorithm.
        // lz4     → excellent ratio, very fast CPU. Recommended default.
        // zstd    → best ratio with reasonable CPU. Modern choice.
        //
        // For JSON payloads (text-heavy), snappy or lz4 typically
        // achieves 3–5x compression with negligible CPU overhead.
        "compression.type": "snappy",

        // ── NETWORK BUFFERS ──────────────────────────────────────
        // "socket.keepalive.enable": send TCP keepalive probes on
        // idle connections. Without this, a firewall or NAT device
        // between your service and Kafka might silently close idle
        // TCP connections, causing the next send to fail with a
        // connection error. Enabling keepalive prevents this.
        "socket.keepalive.enable": true,
    })
    if err != nil {
        return nil, fmt.Errorf("[%s] failed to create producer: %w",
            serviceName, err)
    }

    prod := &Producer{p: p, serviceName: serviceName}

    // Start the delivery report listener in a background goroutine.
    // This is NOT optional. Confluent's producer uses an internal
    // channel called Events() to report the outcome of every send.
    // If you do not drain this channel, it fills up and the
    // producer's internal buffer backs up — eventually causing
    // Produce() calls to block or fail silently.
    // See drainDeliveryReports() below for the full explanation.
    go prod.drainDeliveryReports()

    log.Printf("[%s] Producer ready. Brokers: %s", serviceName, brokers)
    return prod, nil
}

// BaseEvent is embedded in every message we send to Kafka.
// It carries envelope fields that every consumer needs before
// deserialising the specific event payload.
type BaseEvent struct {
    EventID   string    `json:"event_id"`   // unique ID for this event
    EventType string    `json:"event_type"` // what kind of event this is
    Timestamp time.Time `json:"timestamp"`  // when it was produced
    Version   string    `json:"version"`    // schema version for evolution
}

// Publish sends ONE message to a Kafka topic.
//
// Parameters:
//   topic   — the topic name (use the constants from config package)
//   key     — the routing key. This determines which partition the
//             message goes to via hash(key) % numPartitions.
//             Use an entity ID (order_id, user_id) as the key so
//             all events for the same entity land in the same
//             partition and stay in order.
//             Pass "" for no key — messages are distributed
//             round-robin, which balances load but loses ordering.
//   payload — any Go value that can be JSON-marshalled.
//             This becomes the message's value (the actual data).
func (p *Producer) Publish(topic, key string, payload interface{}) error {
    // Step 1: Serialise the payload to JSON bytes.
    // Kafka only deals in bytes — it does not know or care that
    // the content is JSON. The producer and consumer agree on the
    // format out-of-band (via Schema Registry in production).
    data, err := json.Marshal(payload)
    if err != nil {
        return fmt.Errorf("[%s] marshal failed for topic %s: %w",
            p.serviceName, topic, err)
    }

    // Step 2: Call Produce() — this places the message in the
    // producer's internal memory buffer. It does NOT send the
    // message over the network immediately. The library batches
    // messages from the buffer according to linger.ms and
    // batch.size, then sends them in the background.
    //
    // This means Produce() returns almost instantly — it is
    // an in-memory operation. The actual network send happens
    // asynchronously in the library's background threads.
    err = p.p.Produce(&kafka.Message{
        // TopicPartition tells Kafka which topic to write to.
        // Partition: kafka.PartitionAny means "you decide, Kafka".
        // The library hashes the key and picks the partition.
        // If you pass kafka.PartitionAny with no key, it uses
        // round-robin distribution across all partitions.
        TopicPartition: kafka.TopicPartition{
            Topic:     &topic,
            Partition: kafka.PartitionAny,
        },

        // Key is the routing key in bytes.
        // []byte(key) converts the string to bytes.
        // If key is "", Key is empty and Kafka uses round-robin.
        Key: []byte(key),

        // Value is the actual message content — the JSON bytes
        // we serialised from the payload above.
        Value: data,

        // Timestamp: setting this explicitly means the message
        // carries the producer-side time, not the broker-arrival
        // time. This is important for event ordering analysis —
        // you want to know when the event OCCURRED, not when it
        // arrived at the broker (which can be later due to batching).
        Timestamp: time.Now(),

        // Headers are optional key-value metadata attached to
        // the message envelope. They are readable by consumers
        // WITHOUT deserialising the Value — useful for routing,
        // tracing, and filtering without parsing the full payload.
        Headers: []kafka.Header{
            // Which service produced this message? Useful in Kafka UI
            // and for debugging when multiple services write to the
            // same topic.
            {Key: "source-service", Value: []byte(p.serviceName)},

            // A unique trace ID per message for distributed tracing.
            // In production this would be the OpenTelemetry trace ID
            // propagated from the incoming HTTP request.
            {Key: "trace-id", Value: []byte(uuid.New().String())},
        },
    }, nil) // nil = use the producer's own Events() channel for delivery reports
    //       ^ If you pass a channel here instead of nil, delivery
    //         reports for THIS message go to that channel.
    //         nil is correct — we handle all reports in drainDeliveryReports().

    if err != nil {
        // Produce() itself fails only if the internal buffer is full
        // (queue.buffering.max.messages exceeded). This means the
        // producer is producing faster than it can send. Back off
        // or increase the buffer limit.
        return fmt.Errorf("[%s] produce failed for topic %s: %w",
            p.serviceName, topic, err)
    }

    log.Printf("[%s] ✉️  Queued → topic=%s key=%s payload=%d bytes",
        p.serviceName, topic, key, len(data))
    return nil
}

// PublishOrderPlaced is a typed convenience wrapper around Publish.
// Instead of having callers construct the payload map themselves,
// we define typed event structs and typed publisher methods.
// This makes it impossible to publish an order event without an
// order_id, because the struct field is required.
type OrderPlacedEvent struct {
    BaseEvent
    OrderID    string      `json:"order_id"`
    UserID     string      `json:"user_id"`
    Items      []OrderItem `json:"items"`
    TotalPrice float64     `json:"total_price"`
}

type OrderItem struct {
    ProductID string  `json:"product_id"`
    Name      string  `json:"name"`
    Quantity  int     `json:"quantity"`
    Price     float64 `json:"price"`
}

func (p *Producer) PublishOrderPlaced(orderID, userID string,
    items []OrderItem, total float64) error {

    event := OrderPlacedEvent{
        BaseEvent: BaseEvent{
            // uuid.New().String() generates a random UUID v4.
            // Each event gets a globally unique ID — critical for
            // idempotent consumers that need to detect duplicates.
            EventID:   uuid.New().String(),
            EventType: "ORDER_PLACED",
            Timestamp: time.Now(),
            Version:   "1.0",
        },
        OrderID:    orderID,
        UserID:     userID,
        Items:      items,
        TotalPrice: total,
    }

    // We use orderID as the key so ALL events related to this order
    // (placed, confirmed, shipped, delivered) land in the same
    // partition. This guarantees a consumer sees them in order.
    return p.Publish("order.placed", orderID, event)
}

// drainDeliveryReports runs in a background goroutine and processes
// every delivery report that the producer generates.
//
// When does a delivery report arrive?
// After the broker acknowledges a batch (or after all retries are
// exhausted and the message is failed), the library puts a *kafka.Message
// on the producer's Events() channel. This report tells you whether
// the message was actually stored on the broker or failed permanently.
//
// Why do we need a separate goroutine for this?
// The producer has a finite-size internal events channel. If nobody
// reads from it, it fills up. When it is full, Produce() calls start
// blocking, and eventually your entire service stalls. The draining
// goroutine must always be running as long as the producer is alive.
func (p *Producer) drainDeliveryReports() {
    // range over the Events() channel blocks until the channel has
    // an item, then processes it, then blocks again. This is the
    // idiomatic Go way to drain a channel continuously.
    for e := range p.p.Events() {
        switch ev := e.(type) {
        case *kafka.Message:
            // This is a delivery report for a specific message.
            if ev.TopicPartition.Error != nil {
                // The message failed after all retries. This means
                // the message is LOST — it will not be retried again.
                // In production, you would increment an error counter,
                // write to a dead-letter queue, or trigger an alert.
                log.Printf("[%s] ❌ DELIVERY FAILED topic=%s key=%s: %v",
                    p.serviceName,
                    *ev.TopicPartition.Topic,
                    string(ev.Key),
                    ev.TopicPartition.Error,
                )
            } else {
                // The message was successfully written to the broker
                // and all required replicas confirmed (since acks=all).
                // Partition and Offset tell you exactly WHERE in the
                // Kafka log this message landed — useful for debugging.
                log.Printf("[%s] ✅ DELIVERED topic=%s key=%s "+
                    "partition=%d offset=%d",
                    p.serviceName,
                    *ev.TopicPartition.Topic,
                    string(ev.Key),
                    ev.TopicPartition.Partition,
                    ev.TopicPartition.Offset,
                )
            }

        case kafka.Error:
            // These are producer-level errors (not per-message errors).
            // Examples: lost connection to broker, authentication failure.
            // The library handles most of these by retrying internally,
            // but logging them is important for observability.
            log.Printf("[%s] ⚠️  Producer error: %v", p.serviceName, ev)
        }
    }
    // When we exit this loop it means p.p.Events() was closed,
    // which only happens after p.p.Close() is called.
    log.Printf("[%s] Delivery report goroutine exited", p.serviceName)
}

// Flush waits up to timeoutMs milliseconds for all buffered
// messages to be delivered. Call this before Close() to ensure
// no messages are lost when the service shuts down.
//
// Why do we need Flush?
// When you call Close() without Flush(), any messages still in
// the producer's internal buffer are discarded silently. Flush
// gives those messages time to be sent and acknowledged before
// the buffer is torn down.
func (p *Producer) Flush(timeoutMs int) {
    remaining := p.p.Flush(timeoutMs)
    if remaining > 0 {
        log.Printf("[%s] ⚠️  %d messages were NOT flushed before shutdown",
            p.serviceName, remaining)
    } else {
        log.Printf("[%s] All messages flushed successfully", p.serviceName)
    }
}

// Close flushes pending messages and closes the producer.
// Always call this (via defer) in main() after creating a producer.
func (p *Producer) Close() {
    p.Flush(5000) // wait up to 5 seconds for pending messages
    p.p.Close()   // closes the underlying connection and events channel
    log.Printf("[%s] Producer closed", p.serviceName)
}
```

## Part 5 — Consumer with Subscribe() (`consumer/subscribe.go`)

This is the consumer you will use in almost every real service. You subscribe to a topic by name, Kafka assigns partitions to you automatically, and Kafka rebalances when you scale up or recover from failures.

```go
package consumer

import (
    "context"
    "encoding/json"
    "fmt"
    "log"
    "time"

    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
)

// MessageHandler is the function type your service code must implement.
// When a message arrives from Kafka, the consumer calls your handler
// with the topic name, the message key, and the raw JSON bytes.
// Return nil to commit the offset. Return an error to retry.
type MessageHandler func(topic, key string, value []byte) error

// SubscribeConsumer reads messages from topics using Subscribe(),
// which means Kafka's Group Coordinator assigns partitions to it.
// This is the right choice for 99% of microservice consumers.
type SubscribeConsumer struct {
    c           *kafka.Consumer
    serviceName string
    handlers    map[string]MessageHandler // topic name → handler function
}

// NewSubscribeConsumer creates and configures a Kafka consumer
// that uses Subscribe() — Kafka assigns partitions based on the
// group_id and the current group membership.
func NewSubscribeConsumer(brokers, groupID,
    serviceName string) (*SubscribeConsumer, error) {

    c, err := kafka.NewConsumer(&kafka.ConfigMap{

        // ── CONNECTION ────────────────────────────────────────────
        // Same as the producer — the entry point into the cluster.
        "bootstrap.servers": brokers,

        // "client.id": human-readable label for this consumer
        // instance. In a deployment with 5 instances, you would
        // append the pod name or instance number:
        //   "inventory-service-pod-3"
        // This makes it easy to see which specific instance is
        // lagging in Kafka UI or Prometheus dashboards.
        "client.id": serviceName + "-consumer",

        // ── GROUP IDENTITY ────────────────────────────────────────
        // "group.id" is THE most important consumer configuration.
        // This is the answer to the question from the previous guides:
        //
        //   Same group.id across instances → they SHARE the work
        //     (each partition goes to only one instance)
        //
        //   Different group.id per service → each gets ALL messages
        //     (every service has its own independent offset bookmark)
        //
        // The group coordinator uses this string to determine which
        // consumers are "teammates" (should share partitions) and
        // which are "independent services" (should each get everything).
        "group.id": groupID,

        // ── OFFSET RESET ──────────────────────────────────────────
        // "auto.offset.reset" answers your third question:
        // what happens when this group has NO committed offset yet
        // for a partition? (First time reading a topic, or after
        // the committed offset has expired past the retention window)
        //
        // "earliest" → start from the oldest message still in the topic.
        //              Good when you need to process historical events.
        //              Risk: processing old events can have side effects
        //              (sending old emails, decrementing old stock).
        //
        // "latest"   → start from the next NEW message after subscribing.
        //              Ignores all messages that existed before this
        //              consumer group was created. Good for services
        //              that only care about events from now onwards.
        //
        // "none"     → throw an error if no committed offset exists.
        //              Forces you to handle the "first start" case
        //              explicitly in code. Useful in strict pipelines.
        //
        // NOTE: This setting is ONLY consulted when no committed offset
        // exists. Once your group has committed an offset, new instances
        // joining the group inherit that committed offset — this setting
        // is NOT consulted again. (This answers your Q3 directly.)
        "auto.offset.reset": "earliest",

        // ── OFFSET COMMIT MODE ────────────────────────────────────
        // "enable.auto.commit": false means we commit offsets MANUALLY
        // after our handler returns successfully.
        //
        // WHY disable auto-commit?
        // Auto-commit runs on a timer (every auto.commit.interval.ms).
        // This means Kafka might mark a message as "processed" even
        // if your handler crashed halfway through processing it.
        // The message would then be skipped on restart — data loss.
        //
        // Manual commit means: the offset only advances AFTER your
        // handler returns nil. If the handler returns an error, or
        // the process crashes, the offset is NOT committed, and the
        // message is redelivered on the next restart. This gives you
        // at-least-once delivery — the safe default for most services.
        //
        // Your handlers must be IDEMPOTENT: processing the same message
        // twice should produce the same result as processing it once.
        // (Check "if already processed → skip" before acting.)
        "enable.auto.commit": false,

        // ── HEARTBEAT ─────────────────────────────────────────────
        // "heartbeat.interval.ms": how often to send a heartbeat
        // to the Group Coordinator to prove this consumer is alive.
        // The library sends heartbeats automatically in a background
        // thread — you do not write heartbeat code yourself.
        //
        // Must be less than session.timeout.ms / 3.
        // 3000ms (3 seconds) is the standard production value.
        "heartbeat.interval.ms": 3000,

        // "session.timeout.ms": if the coordinator does not receive
        // a heartbeat within this window, it declares the consumer
        // dead and triggers a rebalance to reassign its partitions.
        //
        // Too small → healthy consumers get kicked out during GC pauses
        //             or brief network hiccups. Too many false rebalances.
        // Too large → a truly dead consumer holds its partitions for
        //             a long time before recovery begins.
        // 45000ms (45 seconds) is the standard balanced value.
        "session.timeout.ms": 45000,

        // "max.poll.interval.ms": the maximum time between two
        // consecutive ReadMessage() calls before the coordinator
        // considers this consumer dead — even if heartbeats are fine.
        //
        // This exists because a consumer can send heartbeats from a
        // background thread while its main thread is stuck processing
        // a slow message. The coordinator cannot tell the difference
        // between "processing slowly" and "deadlocked". This setting
        // gives the consumer a maximum processing time per batch.
        //
        // If your handler regularly takes more than 5 minutes per
        // message, increase this value. But also consider: why is
        // a message handler taking 5+ minutes? That is a design smell.
        "max.poll.interval.ms": 300000, // 5 minutes

        // ── FETCH PERFORMANCE ─────────────────────────────────────
        // "fetch.min.bytes": the broker will not respond to a fetch
        // request until it has at least this many bytes available.
        // Increasing this reduces the number of fetch requests but
        // increases latency. 1 byte = respond as soon as ANY message
        // is available (lowest latency, most requests).
        "fetch.min.bytes": 1,

        // "fetch.wait.max.ms": if fetch.min.bytes is not satisfied,
        // wait at most this long before responding anyway (with
        // whatever is available, even if empty). 500ms means: even
        // if the topic is idle, your ReadMessage() call returns at
        // least every 500ms so you can check for cancellation.
        "fetch.wait.max.ms": 500,

        // "max.partition.fetch.bytes": maximum bytes to fetch PER
        // PARTITION per request. 1MB per partition is the default.
        // With 6 partitions, one fetch response can be up to 6MB.
        "max.partition.fetch.bytes": 1048576, // 1 MB
    })
    if err != nil {
        return nil, fmt.Errorf("[%s] failed to create consumer: %w",
            serviceName, err)
    }

    log.Printf("[%s] Consumer ready. group=%s brokers=%s",
        serviceName, groupID, brokers)

    return &SubscribeConsumer{
        c:           c,
        serviceName: serviceName,
        handlers:    make(map[string]MessageHandler),
    }, nil
}

// RegisterHandler registers a function to call when a message
// arrives on the given topic. You call this once per topic
// before calling Start(). Having per-topic handlers keeps the
// routing logic clean — each handler is focused on one event type.
func (sc *SubscribeConsumer) RegisterHandler(
    topic string, handler MessageHandler) {
    sc.handlers[topic] = handler
    log.Printf("[%s] Handler registered for topic: %s",
        sc.serviceName, topic)
}

// Start subscribes to all registered topics and begins the poll loop.
// It blocks until the context is cancelled, making it correct to run
// as a goroutine from main().
//
// The poll loop is the heart of every Kafka consumer. It:
//   1. Asks Kafka for the next batch of messages (ReadMessage)
//   2. Routes each message to the correct handler
//   3. Commits the offset if the handler succeeds
//   4. Repeats until the context is cancelled
func (sc *SubscribeConsumer) Start(ctx context.Context) error {
    // Collect all topic names from the registered handlers.
    topics := make([]string, 0, len(sc.handlers))
    for topic := range sc.handlers {
        topics = append(topics, topic)
    }

    // SubscribeTopics is the KEY API call that answers your Q2.
    // By calling Subscribe instead of Assign, you are telling Kafka:
    // "I want to read from these topics, but YOU decide which
    // partitions I get based on my group membership."
    //
    // Internally this triggers the 8-step Join/Sync handshake
    // described in Question 4's answer:
    //   FindCoordinator → JoinGroup → Leader computes assignment
    //   → SyncGroup → FetchOffsets → Start fetching
    //
    // The second argument (nil) is a rebalance callback. When nil,
    // the library handles rebalances automatically. You can provide
    // a callback here to run custom logic (e.g. flush a cache) when
    // partitions are revoked or assigned.
    if err := sc.c.SubscribeTopics(topics, nil); err != nil {
        return fmt.Errorf("[%s] subscribe failed: %w",
            sc.serviceName, err)
    }

    log.Printf("[%s] 👂 Subscribed to: %v", sc.serviceName, topics)

    // ── THE POLL LOOP ─────────────────────────────────────────────
    // This loop runs for the entire lifetime of the service.
    // Each iteration does one poll: ask Kafka for messages, process
    // whatever arrives, commit if successful, loop again.
    for {
        // Check if we have been asked to stop. select{} with a
        // default case makes this non-blocking — if the context
        // is not cancelled yet, we fall through immediately to
        // ReadMessage.
        select {
        case <-ctx.Done():
            log.Printf("[%s] Context cancelled, stopping consumer",
                sc.serviceName)
            return nil
        default:
        }

        // ReadMessage is the poll call. It asks Kafka: "give me the
        // next message from any of my assigned partitions, starting
        // from my committed offsets."
        //
        // The duration argument is the POLL TIMEOUT — how long to
        // wait if no messages are currently available.
        //
        // 300ms means: if no message is ready within 300ms, return
        // with a timeout error (IsTimeout() == true). We then loop
        // back to check ctx.Done() and poll again.
        //
        // WHY 300ms specifically?
        // It controls how quickly the consumer reacts to a shutdown
        // signal. With 300ms, the worst case is a 300ms delay between
        // ctx.Done() firing and the consumer stopping. Shorter =
        // more responsive, more CPU. Longer = less CPU, slower stop.
        //
        // IMPORTANT: ReadMessage also sends heartbeats internally.
        // The library's background thread sends heartbeats to the
        // coordinator during the wait period. This is why you must
        // keep calling ReadMessage regularly — it keeps the consumer
        // alive in the group's eyes.
        msg, err := sc.c.ReadMessage(300 * time.Millisecond)
        if err != nil {
            kafkaErr, ok := err.(kafka.Error)
            if ok && kafkaErr.IsTimeout() {
                // No messages arrived in the 300ms window. This is
                // completely normal — not an error. Just loop again.
                continue
            }
            // A real error (broker disconnect, auth failure, etc.)
            log.Printf("[%s] Consumer error: %v", sc.serviceName, err)
            continue
        }

        // A message arrived. Extract its routing fields.
        topic := *msg.TopicPartition.Topic
        key   := string(msg.Key)

        log.Printf("[%s] 📨 Received topic=%s key=%s "+
            "partition=%d offset=%d",
            sc.serviceName, topic, key,
            msg.TopicPartition.Partition,
            msg.TopicPartition.Offset,
        )

        // Route to the registered handler for this topic.
        handler, exists := sc.handlers[topic]
        if !exists {
            log.Printf("[%s] No handler for topic %s — skipping",
                sc.serviceName, topic)
            // Still commit — we do not want to be stuck replaying
            // messages for a topic we have no handler for.
            sc.c.CommitMessage(msg)
            continue
        }

        // Call the handler with the raw JSON bytes.
        // The handler is responsible for deserialising the value
        // into its specific event struct.
        if err := handler(topic, key, msg.Value); err != nil {
            // Handler returned an error. We do NOT commit the offset.
            // On the next poll (or after a restart), this message
            // will be redelivered and the handler will be called again.
            //
            // This is at-least-once delivery in action.
            // Your handler must be idempotent — safe to run twice.
            //
            // In production, after N failures you would move the
            // message to a Dead Letter Queue topic instead of
            // retrying forever, to prevent a bad message from
            // blocking all subsequent messages in that partition.
            log.Printf("[%s] ❌ Handler error topic=%s key=%s: %v — "+
                "offset NOT committed, will be redelivered",
                sc.serviceName, topic, key, err)
            continue
        }

        // Handler succeeded. Now commit the offset.
        //
        // CommitMessage tells Kafka's __consumer_offsets system:
        // "this consumer group has successfully processed the message
        // at partition=X, offset=Y. Next time we restart or rebalance,
        // start from offset Y+1."
        //
        // This is the bookmark that answers your Q3:
        // when a new instance joins the group and takes over this
        // partition, it reads THIS committed offset and resumes from
        // Y+1 — not from the beginning, not from now.
        if _, err := sc.c.CommitMessage(msg); err != nil {
            log.Printf("[%s] ⚠️  Failed to commit offset: %v",
                sc.serviceName, err)
            // Even if commit fails, we continue. The message was
            // processed successfully. If the commit failure persists,
            // the message will be reprocessed on the next restart —
            // which is acceptable with idempotent handlers.
        } else {
            log.Printf("[%s] ✅ Committed offset topic=%s "+
                "partition=%d offset=%d",
                sc.serviceName, topic,
                msg.TopicPartition.Partition,
                msg.TopicPartition.Offset,
            )
        }
    }
}

// Close cleanly shuts down the consumer.
// Call this via defer in main() after creating the consumer.
func (sc *SubscribeConsumer) Close() {
    // Unsubscribes from all topics and triggers a final rebalance
    // so the coordinator can reassign this consumer's partitions
    // to other instances before the connection closes.
    // Without this, the coordinator waits for session.timeout.ms
    // to expire before rebalancing — up to 45 seconds of delay.
    sc.c.Close()
    log.Printf("[%s] Consumer closed", sc.serviceName)
}

// Unmarshal is a helper for handler functions to deserialise
// the raw JSON bytes into a typed event struct.
// Usage in a handler:
//   var evt OrderPlacedEvent
//   if err := consumer.Unmarshal(value, &evt); err != nil { ... }
func Unmarshal(data []byte, target interface{}) error {
    return json.Unmarshal(data, target)
}
```

## Part 6 — Consumer with Assign() (`consumer/assign.go`)

This consumer answers the second part of your Q2: yes, you can read from a specific partition manually. This is used for debugging, data migration, and replay — not for normal service consumption.

```go
package consumer

import (
    "context"
    "fmt"
    "log"
    "time"

    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
)

// AssignConsumer reads from SPECIFIC partitions at SPECIFIC offsets.
// Unlike SubscribeConsumer, it bypasses the group coordination
// system entirely. No rebalancing. No automatic failover.
// Use this for: replaying history, debugging, one-off migration jobs.
type AssignConsumer struct {
    c           *kafka.Consumer
    serviceName string
}

// NewAssignConsumer creates a consumer configured for manual
// partition assignment. The configuration is simpler than Subscribe
// because we do not need group coordination machinery.
func NewAssignConsumer(brokers, serviceName string) (*AssignConsumer, error) {
    c, err := kafka.NewConsumer(&kafka.ConfigMap{
        "bootstrap.servers": brokers,
        "client.id":         serviceName + "-assign-consumer",

        // group.id is still required by the library even when using
        // Assign(). Set it to something descriptive but unique — it
        // will NOT participate in group coordination. Use a timestamp
        // or UUID to ensure it never collides with a real service group.
        "group.id": serviceName + "-manual-" + time.Now().Format("20060102"),

        // With Assign(), auto.offset.reset is rarely relevant because
        // you specify the exact offset in the assignment itself (via Seek).
        // We set "earliest" as a fallback safety net.
        "auto.offset.reset":  "earliest",
        "enable.auto.commit": false,

        // No heartbeat tuning needed — this consumer is not a group
        // member so the coordinator does not track it.
        "session.timeout.ms": 10000,
    })
    if err != nil {
        return nil, fmt.Errorf("[%s] failed to create assign consumer: %w",
            serviceName, err)
    }

    return &AssignConsumer{c: c, serviceName: serviceName}, nil
}

// ReadPartitionFromOffset reads messages from ONE specific partition
// starting at a specific offset, stopping when it reaches endOffset
// or when the context is cancelled.
//
// This is the direct answer to: "can I read from a specific partition?"
// Yes — this is exactly how.
//
// Parameters:
//   topicName   — the topic to read from
//   partition   — the EXACT partition number (0, 1, 2, ...)
//   startOffset — the offset to start reading from. Use:
//                   kafka.OffsetBeginning for the oldest message
//                   kafka.OffsetEnd       for the next new message
//                   kafka.Offset(42)      for a specific offset number
//   endOffset   — stop reading when you reach this offset (-1 = no limit)
//   handler     — function called for each message
func (ac *AssignConsumer) ReadPartitionFromOffset(
    ctx context.Context,
    topicName string,
    partition int32,
    startOffset kafka.Offset,
    endOffset int64,
    handler MessageHandler,
) error {

    // Assign() directly assigns a specific partition+topic to this
    // consumer. This bypasses the Group Coordinator entirely — no
    // JoinGroup, no SyncGroup, no heartbeat election. The consumer
    // immediately owns this partition without negotiating.
    //
    // Compare to Subscribe(): Kafka distributes partitions among all
    // group members. Here, YOU declare exactly which partition you want.
    err := ac.c.Assign([]kafka.TopicPartition{
        {
            Topic:     &topicName,
            Partition: partition,
            // Offset here tells Kafka WHERE to start reading.
            // kafka.OffsetBeginning = read from offset 0 (or earliest
            //   available if old messages have been deleted by retention)
            // kafka.OffsetEnd = read only new messages from now onwards
            // kafka.Offset(N) = read from a specific offset number
            Offset: startOffset,
        },
    })
    if err != nil {
        return fmt.Errorf("assign failed for %s[%d]: %w",
            topicName, partition, err)
    }

    log.Printf("[%s] Assigned to %s partition %d from offset %v",
        ac.serviceName, topicName, partition, startOffset)

    messagesRead := 0
    for {
        select {
        case <-ctx.Done():
            log.Printf("[%s] Stopped after reading %d messages",
                ac.serviceName, messagesRead)
            return nil
        default:
        }

        msg, err := ac.c.ReadMessage(300 * time.Millisecond)
        if err != nil {
            if kafkaErr, ok := err.(kafka.Error); ok && kafkaErr.IsTimeout() {
                continue // normal — no messages in the window
            }
            return fmt.Errorf("read error: %w", err)
        }

        // Check if we have read past the requested end offset.
        if endOffset > 0 &&
            int64(msg.TopicPartition.Offset) >= endOffset {
            log.Printf("[%s] Reached end offset %d — stopping",
                ac.serviceName, endOffset)
            return nil
        }

        if err := handler(
            *msg.TopicPartition.Topic,
            string(msg.Key),
            msg.Value,
        ); err != nil {
            log.Printf("[%s] Handler error at offset %d: %v",
                ac.serviceName,
                msg.TopicPartition.Offset, err)
        }

        messagesRead++
    }
}

// SeekAndRead demonstrates Seek() — the third reading mode.
// You first Subscribe() normally, get assigned a partition through
// group coordination, then Seek() to override the starting offset
// for that specific partition.
//
// Use case: your service is normally running, but you need to
// re-process messages from 2 hours ago in partition 3 specifically.
func (ac *AssignConsumer) SeekAndRead(
    ctx context.Context,
    topicName string,
    partition int32,
    seekToOffset kafka.Offset,
    handler MessageHandler,
) error {
    // Assign the partition first (we need ownership before seeking)
    err := ac.c.Assign([]kafka.TopicPartition{
        {Topic: &topicName, Partition: partition,
            Offset: kafka.OffsetBeginning},
    })
    if err != nil {
        return fmt.Errorf("assign failed: %w", err)
    }

    // Seek() repositions the consumer's read cursor for a specific
    // partition to a specific offset. After this call, the next
    // ReadMessage() will return the message at seekToOffset, not
    // whatever the committed offset was.
    //
    // Seek does NOT commit anything — it is a local operation that
    // only affects where THIS consumer reads next.
    err = ac.c.Seek(kafka.TopicPartition{
        Topic:     &topicName,
        Partition: partition,
        Offset:    seekToOffset, // jump to this exact position
    }, 5000) // 5 second timeout for the seek operation
    if err != nil {
        return fmt.Errorf("seek failed: %w", err)
    }

    log.Printf("[%s] Seeked to offset %v on %s[%d]",
        ac.serviceName, seekToOffset, topicName, partition)

    // Now read normally from the sought position
    for {
        select {
        case <-ctx.Done():
            return nil
        default:
        }
        msg, err := ac.c.ReadMessage(300 * time.Millisecond)
        if err != nil {
            if kafkaErr, ok := err.(kafka.Error); ok && kafkaErr.IsTimeout() {
                continue
            }
            return err
        }
        if err := handler(
            *msg.TopicPartition.Topic,
            string(msg.Key),
            msg.Value,
        ); err != nil {
            log.Printf("handler error: %v", err)
        }
    }
}

func (ac *AssignConsumer) Close() {
    ac.c.Close()
    log.Printf("[%s] Assign consumer closed", ac.serviceName)
}
```

## Part 7 — Wiring It All Together (`main.go`)

```go
package main

import (
    "context"
    "encoding/json"
    "log"
    "os"
    "os/signal"
    "syscall"
    "time"

    "github.com/riyad/kafka-go-deep-dive/consumer"
    "github.com/riyad/kafka-go-deep-dive/producer"
    "github.com/riyad/kafka-go-deep-dive/topic"
)

func main() {
    brokers := getEnv("KAFKA_BROKERS", "localhost:9092")
    groupID := getEnv("KAFKA_GROUP_ID", "order-service")

    // ── STEP 1: CREATE TOPICS ─────────────────────────────────────
    // Before any producer or consumer starts, ensure all topics
    // exist with the right partition counts. This is safe to call
    // on every startup — existing topics are left unchanged.
    admin, err := topic.NewAdminClient(brokers)
    if err != nil {
        log.Fatalf("Admin client failed: %v", err)
    }
    defer admin.Close()

    err = admin.EnsureTopics([]topic.TopicSpec{
        {
            Name: "order.placed",
            // 6 partitions: we plan to run up to 6 consumer instances
            // of the inventory service in parallel for max throughput.
            NumPartitions: 6,
            // 1 for development (single broker). Use 3 in production.
            ReplicationFactor: 1,
            Config: map[string]string{
                // Keep messages for 7 days (in milliseconds)
                "retention.ms": "604800000",
                // Do NOT use log compaction — we want full history
                "cleanup.policy": "delete",
            },
        },
        {
            Name:              "inventory.reserved",
            NumPartitions:     6,
            ReplicationFactor: 1,
        },
        {
            Name:              "notification.send",
            NumPartitions:     3, // fewer partitions, lower throughput topic
            ReplicationFactor: 1,
        },
    })
    if err != nil {
        log.Fatalf("Topic creation failed: %v", err)
    }

    // Inspect what was created
    admin.DescribeTopic("order.placed")

    // ── STEP 2: CREATE PRODUCER ───────────────────────────────────
    prod, err := producer.NewProducer(brokers, "order-service")
    if err != nil {
        log.Fatalf("Producer creation failed: %v", err)
    }
    defer prod.Close() // flush + close on shutdown

    // ── STEP 3: CREATE SUBSCRIBE CONSUMER ────────────────────────
    // This consumer uses Subscribe() — Kafka assigns partitions.
    subConsumer, err := consumer.NewSubscribeConsumer(
        brokers, groupID, "order-service")
    if err != nil {
        log.Fatalf("Subscribe consumer failed: %v", err)
    }
    defer subConsumer.Close()

    // Register handlers — one function per topic.
    subConsumer.RegisterHandler("inventory.reserved",
        handleInventoryReserved)
    subConsumer.RegisterHandler("notification.send",
        handleNotification)

    // ── STEP 4: RUN EVERYTHING ────────────────────────────────────
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    // Consumer runs in a background goroutine so the main goroutine
    // is free to produce messages and wait for shutdown signals.
    go func() {
        if err := subConsumer.Start(ctx); err != nil {
            log.Printf("Consumer stopped: %v", err)
        }
    }()

    // Produce some test messages to see the full flow.
    go func() {
        for i := 0; i < 5; i++ {
            orderID := fmt.Sprintf("ord-%04d", i+1)
            err := prod.PublishOrderPlaced(
                orderID,
                "user-99",
                []producer.OrderItem{
                    {ProductID: "prod-001", Name: "Keyboard",
                        Quantity: 1, Price: 129.99},
                },
                129.99,
            )
            if err != nil {
                log.Printf("Publish failed: %v", err)
            }
            time.Sleep(500 * time.Millisecond)
        }
    }()

    // Wait for SIGINT or SIGTERM (Ctrl+C or docker stop)
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit

    log.Println("Shutdown signal received")
    cancel() // stops the consumer poll loop
    // defer prod.Close() runs now, flushing pending messages
}

// handleInventoryReserved is called by the consumer whenever a
// message arrives on the "inventory.reserved" topic.
func handleInventoryReserved(topic, key string, value []byte) error {
    var event struct {
        OrderID string `json:"order_id"`
        UserID  string `json:"user_id"`
    }
    if err := consumer.Unmarshal(value, &event); err != nil {
        return fmt.Errorf("unmarshal failed: %w", err)
    }
```
