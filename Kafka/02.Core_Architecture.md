# Apache Kafka — The Core Architecture

### Understanding the Engine Room, Piece by Piece

---

## Chapter 1 — Apache Kafka: What It Actually Is

Before diving into the individual parts, it helps to have a mental picture of Kafka as a whole. Apache Kafka is, at its heart, a **distributed commit log**. Unlike a traditional message queue that holds a message until someone picks it up and then discards it, Kafka treats every message as an entry in an ordered, append-only log — much like the transaction log of a database. Messages are written to the end of the log, and they stay there for a configurable period of time, regardless of whether anyone has read them yet.

This one design decision — keeping messages in a persistent, ordered log — is what gives Kafka its extraordinary power. Consumers can re-read old messages. New services can join and replay the entire history. The system can handle millions of messages per second without losing a single one. Everything else about Kafka — partitions, consumer groups, offsets — is built on top of this foundational idea.

```
  The Traditional Queue vs. The Kafka Log

  TRADITIONAL QUEUE
  ┌─────┬─────┬─────┬─────┐
  │ M1  │ M2  │ M3  │ M4  │  ← Consumer picks M1, it is deleted. Gone forever.
  └─────┴─────┴─────┴─────┘

  KAFKA COMMIT LOG
  ┌─────┬─────┬─────┬─────┬─────┬─────┐
  │ M1  │ M2  │ M3  │ M4  │ M5  │ M6  │  ← Messages stay. Multiple consumers
  └─────┴─────┴─────┴─────┴─────┴─────┘    can read at their own pace.
     0     1     2     3     4     5
  (offset positions — never change)
```

Now, let us walk through every building block of this system, one by one.

## Chapter 2 — The Topic: A Named Channel for Your Events

A **Topic** is the most fundamental organizational unit in Kafka. It is simply a named, durable log where messages of a particular category are stored. When you produce an event — say, "an order was placed" — you send it to a specific topic, perhaps named `order.placed`. When a service wants to consume order events, it subscribes to that same topic.

Think of a topic as a dedicated newspaper. The `order.placed` topic is the Orders Newspaper. Anyone who wants to know about orders reads this newspaper. Anyone with news about orders publishes to it. The newspaper does not care who reads it, and readers do not care who wrote it. They are entirely decoupled through this shared, named channel.

Topics in Kafka can have any name you choose, and you can have as many topics as your system needs. In a real microservices architecture, you might see topics like `user.registered`, `payment.processed`, `inventory.updated`, `email.requested` — each a dedicated stream of events for a specific business domain.

```
  Multiple Topics in a Kafka Cluster

  ┌──────────────────────────────────────────────────────────┐
  │                    KAFKA CLUSTER                         │
  │                                                          │
  │  Topic: "order.placed"                                   │
  │  ┌──────┬──────┬──────┬──────┬──────┐                    │
  │  │ M1   │ M2   │ M3   │ M4   │ M5   │──▶ (new messages)  │
  │  └──────┴──────┴──────┴──────┴──────┘                    │
  │                                                          │
  │  Topic: "payment.processed"                              │
  │  ┌──────┬──────┬──────┐                                  │
  │  │ M1   │ M2   │ M3   │──▶ (new messages)                │
  │  └──────┴──────┴──────┘                                  │
  │                                                          │
  │  Topic: "user.registered"                                │
  │  ┌──────┬──────┬──────┬──────┐                           │
  │  │ M1   │ M2   │ M3   │ M4   │──▶ (new messages)         │
  │  └──────┴──────┴──────┴──────┘                           │
  └──────────────────────────────────────────────────────────┘
```

## Chapter 3 — The Partition: How Kafka Achieves Scale

If a topic is the newspaper, then a **partition** is the printing press. A single printing press can only work so fast, but if you have four presses running in parallel, each printing a quarter of the newspaper, your throughput multiplies fourfold. Partitions work exactly like this.

When you create a topic, you decide how many partitions it will have. Kafka then splits the topic's log into that many independent sub-logs. Each partition is an ordered, append-only log in its own right. Messages written to the topic are distributed across partitions — either in round-robin fashion (if the message has no key) or deterministically by hashing the message key (if a key is provided). The critical rule is: **within a single partition, order is strictly guaranteed**. Across partitions, there are no ordering guarantees.

This partitioning is the secret to Kafka's scalability. Each partition can live on a different physical machine, and each can be read by a different consumer simultaneously. More partitions means more parallelism, more throughput, and more capacity — at the cost of added complexity in ordering guarantees.

```
  A Topic Split Into 3 Partitions

  Topic: "order.placed"  (3 partitions)

  Partition 0:  ┌──────┬──────┬──────┬──────┐
                │ M0   │ M3   │ M6   │ M9   │──▶
                └──────┴──────┴──────┴──────┘
                  off:0  off:1  off:2  off:3

  Partition 1:  ┌──────┬──────┬──────┬──────┐
                │ M1   │ M4   │ M7   │ M10  │──▶
                └──────┴──────┴──────┴──────┘
                  off:0  off:1  off:2  off:3

  Partition 2:  ┌──────┬──────┬──────┐
                │ M2   │ M5   │ M8   │──▶
                └──────┴──────┴──────┘
                  off:0  off:1  off:2

  Each partition has its OWN independent offset sequence starting from 0.
  Messages within each partition are strictly ordered.
```

There is one especially important behavior to understand about message keys and partitioning. If you always send orders for a particular customer with their `customer_id` as the message key, Kafka guarantees that all messages for that customer always land in the same partition. This means that for a given customer, you always get strict ordering — a tremendously useful property when sequencing matters (you do not want to process a cancellation before the original order).

## Chapter 4 — The Producer: Writing Messages Into Kafka

A **Producer** is any application or service that writes messages into a Kafka topic. In your Go microservice, the Order Service would be a producer: when a customer places an order, the Order Service creates a message and sends it to the `order.placed` topic. That is, fundamentally, all a producer does — it writes.

But the producer is more sophisticated than it might appear. When sending a message, the producer must decide which partition to write to. If no message key is specified, Kafka distributes messages across partitions in a balanced way. If a key is provided, the producer hashes the key to always arrive at the same partition number, ensuring ordering for that key.

The producer also has configurable **acknowledgement modes**, which govern how confident you need to be that a message was actually stored before the producer continues. This is a reliability dial you can tune depending on your use case.

```
  Producer Sending Messages to a Topic

  ┌──────────────────┐
  │   Order Service  │
  │   (Producer)     │
  │                  │
  │  msg = {         │
  │   key: "user-99" │
  │   value: {...}   │      hash("user-99") % 3 = Partition 1
  │  }               │──────────────────────────────────────────┐
  └──────────────────┘                                          │
                                                                ▼
                             Topic: "order.placed"
                             ┌────────────────────────────────────────┐
  Partition 0:               │  ┌──────┬──────┬──────┐                │
                             │  │ M0   │ M3   │ M6   │──▶             │
                             │  └──────┴──────┴──────┘                │
                             │                                        │
  Partition 1: ◀── writes ─▶ │  ┌──────┬──────┬──────┬──────┐         │
                             │  │ M1   │ M4   │ M7   │ NEW  │──▶      │
                             │  └──────┴──────┴──────┴──────┘         │
                             │                                        │
  Partition 2:               │  ┌──────┬──────┬──────┐                │
                             │  │ M2   │ M5   │ M8   │──▶             │
                             │  └──────┴──────┴──────┘                │
                             └────────────────────────────────────────┘

  Acknowledgement Modes (acks):
  acks=0  →  Fire and forget. No confirmation. Fastest, least safe.
  acks=1  →  Leader confirms. Fast and reasonably safe.
  acks=all → All replicas confirm. Slowest, maximum durability.
```

## Chapter 5 — The Consumer: Reading Messages From Kafka

A **Consumer** is any application or service that reads messages from a Kafka topic. The Inventory Service reading from `order.placed` is a consumer. The Shipping Service reading from the same topic is also a consumer. Each is completely independent.

The most important thing to understand about Kafka consumers is that they **pull** messages rather than having messages pushed to them. The consumer asks Kafka: "Give me any new messages from partition 1 of topic `order.placed` starting at offset 7." Kafka responds with those messages, and the consumer processes them. This pull-based model gives consumers complete control over their own pace — a slow consumer cannot be overwhelmed by a fast broker.

When a consumer reads messages, it tracks its position in each partition using an **offset** (discussed in the next chapter). It commits this offset back to Kafka periodically, so that if the consumer crashes and restarts, it knows exactly where to resume without reprocessing or missing messages.

```
  A Consumer Reading From a Partition

  Partition 0 of "order.placed":
  ┌──────┬──────┬──────┬──────┬──────┬──────┐
  │ M0   │ M1   │ M2   │ M3   │ M4   │ M5   │
  └──────┴──────┴──────┴──────┴──────┴──────┘
    0      1      2      3      4      5

                               ▲
                               │
                      Consumer is HERE
                      (has read up to offset 3,
                       next read starts at 4)

  Consumer says to Kafka:
  "Give me messages from offset 4 onwards, partition 0."

  Kafka responds:
  "Here are M4, M5. Next offset to read: 6."
```

## Chapter 6 — The Offset: Your Bookmark in the Log

The **offset** is one of the most elegant ideas in Kafka's design. It is simply a monotonically increasing integer that uniquely identifies the position of every message within a partition. The first message in a partition has offset 0, the next has offset 1, and so on, forever. Offsets never repeat, never change, and never get recycled.

Think of the offset as a bookmark in a very long book. Each reader (consumer) has their own bookmark, and they carry it independently. One reader might be on page 5 while another is still on page 2. Neither affects the other. And crucially, the book itself does not change based on who has read it — the pages remain there until the library (Kafka) decides to archive old volumes after the retention period expires.

```
  Offsets Belong to the Consumer, Not the Message

  Partition 0:
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ M0   │ M1   │ M2   │ M3   │ M4   │ M5   │ M6   │
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┘
    0      1      2      3      4      5      6
                                ▲             ▲
                                │             │
               Inventory Svc    │             │  Analytics Svc
               (committed: 3)   │             │  (committed: 6)
               Next read: 4     │             │  Next read: 7


  If Inventory Service crashes → restarts → reads its committed
  offset (3) from Kafka → resumes at M4. Nothing is lost.

  Offset Storage:
  ┌─────────────────────────────────────────┐
  │ Kafka internal topic: __consumer_offsets│
  │                                         │
  │  Group: inventory-svc, P0 → offset 3    │
  │  Group: analytics-svc,  P0 → offset 6   │
  └─────────────────────────────────────────┘
```

Offsets are stored in a special internal Kafka topic called `__consumer_offsets`. Each consumer group maintains its own committed offset per partition, entirely independently of every other consumer group. This is what allows ten different services to all read from the same topic and each maintain their own independent reading progress.

## Chapter 7 — The Consumer Group: Scaling Your Readers

A **Consumer Group** is a set of consumer instances that cooperate to read from a topic together. Every consumer in a group shares the same `group.id`, and Kafka coordinates them so that each partition is assigned to exactly one consumer in the group at any given time. This is the mechanism that allows you to scale your consumers horizontally.

Imagine the `order.placed` topic has 3 partitions and you have 3 instances of the Inventory Service running. Kafka assigns one partition to each instance. All three instances work in parallel, each processing a third of the incoming messages, and together they get through three times as much work as a single instance could. If one instance dies, Kafka detects it and redistributes its partition to the remaining instances — this process is called a **rebalance**.

```
  Consumer Groups — Parallelism and Isolation

  Topic: "order.placed" — 3 Partitions (P0, P1, P2)

  ┌──────────────────────────────────────────────────┐
  │  Consumer Group: "inventory-service"             │
  │  ┌────────────┐ ┌────────────┐ ┌────────────┐    │
  │  │ Instance 1 │ │ Instance 2 │ │ Instance 3 │    │
  │  │  reads P0  │ │  reads P1  │ │  reads P2  │    │
  │  └────────────┘ └────────────┘ └────────────┘    │
  └──────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────┐
  │  Consumer Group: "shipping-service"              │
  │  ┌───────────────────────────────────────────┐   │
  │  │           Instance 1                      │   │
  │  │   reads P0 + P1 + P2 (all partitions)     │   │
  │  └───────────────────────────────────────────┘   │
  └──────────────────────────────────────────────────┘

  KEY RULES:
  ✅  Each partition → at most ONE consumer per group (no duplicate reads)
  ✅  Each group → gets ALL messages independently (full Pub/Sub)
  ⚠️  More consumers than partitions → some consumers sit idle
```

The golden rule of consumer groups is worth memorising: **the number of active consumers in a group is bounded by the number of partitions**. If you have 3 partitions and 5 consumer instances in the same group, only 3 will be actively reading; the other 2 will sit idle, waiting to take over if a running instance fails. This means that when you design a Kafka topic, choosing the right number of partitions is really a decision about how much parallelism you want to allow in the future.

## Chapter 8 — The Broker: A Node in the Kafka Cluster

A **Broker** is simply a single Kafka server — one machine (or container) running the Kafka process. Brokers are the workhorses of the Kafka cluster. They receive messages from producers, store them durably on disk, and serve them to consumers on demand. A Kafka cluster is made up of multiple brokers working together.

Each broker is given a unique numeric ID within the cluster. Partitions are distributed across brokers so that the storage and network load is spread evenly. When a producer sends a message to a topic, it is the responsible broker for that partition that receives and stores it. When a consumer wants to read, it connects to the broker that holds the partition it needs.

```
  A Kafka Cluster With 3 Brokers

  ┌────────────────────────────────────────────────────────────┐
  │                      KAFKA CLUSTER                         │
  │                                                            │
  │  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐   │
  │  │   Broker 1    │  │   Broker 2    │  │   Broker 3    │   │
  │  │   (id: 1)     │  │   (id: 2)     │  │   (id: 3)     │   │
  │  │               │  │               │  │               │   │
  │  │ order.P0      │  │ order.P1      │  │ order.P2      │   │
  │  │ payment.P0    │  │ payment.P1    │  │ user.P0       │   │
  │  └───────────────┘  └───────────────┘  └───────────────┘   │
  │                                                            │
  └────────────────────────────────────────────────────────────┘
         ▲                    ▲                    ▲
         │                    │                    │
  Producers and consumers connect to whichever broker
  holds the partition they need. The cluster coordinates
  this automatically via metadata that every broker stores.
```

The fact that Kafka is a cluster of brokers, rather than a single server, is what gives it fault tolerance and horizontal scalability. If Broker 2 goes down, the partitions it was hosting need to be covered by another broker — which is where replication enters the picture.

## Chapter 9 — The Replication Factor: Your Safety Net

A **Replication Factor** is the number of copies Kafka maintains of each partition across different brokers. A replication factor of 3 means that every partition has 3 identical copies living on 3 different brokers. If one broker catches fire — literally or figuratively — two other brokers still hold a complete copy of your data, and the system continues without interruption.

Setting the replication factor is one of the most important operational decisions you make when creating a topic. A replication factor of 1 means no redundancy at all — if the broker holding that partition dies, your data is inaccessible until it recovers. A factor of 3 is the standard recommended value for production systems, balancing storage cost against fault tolerance.

```
  Replication Factor = 3 for "order.placed" Partition 0

  Broker 1               Broker 2               Broker 3
  ┌──────────────┐       ┌──────────────┐       ┌──────────────┐
  │ order.P0     │       │ order.P0     │       │ order.P0     │
  │ (LEADER)     │──────▶│ (FOLLOWER)   │       │ (FOLLOWER)   │
  │              │       │              │       │              │
  │ M0,M1,M2,M3  │       │ M0,M1,M2,M3  │       │ M0,M1,M2,M3  │
  └──────────────┘       └──────────────┘       └──────────────┘
         │                      ▲                      ▲
         │                      │                      │
         └──────── replicates ──┴──────────────────────┘
                  (leader pushes writes to all followers)

  If Broker 1 dies:
  ┌──────────────┐       ┌──────────────┐       ┌──────────────┐
  │ Broker 1     │       │ order.P0     │       │ order.P0     │
  │ ☠️  DEAD     │       │ (NEW LEADER) │       │ (FOLLOWER)   │
  └──────────────┘       └──────────────┘       └──────────────┘
                                ▲
                         Kafka automatically
                         elects a new leader.
                         Producers & consumers
                         reroute seamlessly.
```

It is worth noting that replication in Kafka is not symmetric — there is always one designated leader for each partition, and the rest are followers. All reads and writes go through the leader. Followers exist purely to maintain copies in case the leader fails. This brings us naturally to the final concept.

## Chapter 10 — Leader & Follower Partitions: Who Does the Work

For every partition in a Kafka topic, one broker is designated as the **Leader** and the remaining replicas are designated as **Followers**. The leader is the single authoritative copy of that partition. All producer writes go to the leader. All consumer reads come from the leader. The followers exist silently in the background, continuously replicating every message the leader receives, ready to step up if the leader ever goes away.

This leader-follower model ensures that Kafka always has a single, consistent source of truth for each partition at any point in time, while still providing the redundancy and fault tolerance of having multiple copies. Kafka distributes the leadership of different partitions across different brokers — so Broker 1 might be the leader for `order.P0` and a follower for `order.P1`, while Broker 2 is the leader for `order.P1` and a follower for `order.P0`. This spreading of leadership ensures that no single broker carries all the write traffic.

```
  Leadership Distribution Across Brokers (Balanced)

  Topic: "order.placed" — 3 Partitions, Replication Factor 3

                  Broker 1        Broker 2        Broker 3
                ┌────────────┐  ┌────────────┐  ┌────────────┐
  Partition 0   │  LEADER ★  │  │  follower  │  │  follower  │
                ├────────────┤  ├────────────┤  ├────────────┤
  Partition 1   │  follower  │  │  LEADER ★  │  │  follower  │
                ├────────────┤  ├────────────┤  ├────────────┤
  Partition 2   │  follower  │  │  follower  │  │  LEADER ★  │
                └────────────┘  └────────────┘  └────────────┘

  Each broker carries a fair share of the write traffic.
  Each broker is a leader for some partitions, a follower for others.


  What Happens When Broker 2 Dies:

                  Broker 1        Broker 2        Broker 3
                ┌────────────┐  ┌────────────┐  ┌────────────┐
  Partition 0   │  LEADER ★  │  │  ☠️  DEAD │  │  follower  │
                ├────────────┤  ├────────────┤  ├────────────┤
  Partition 1   │  NEW LDR ★ │  │  ☠️  DEAD │  │  follower  │  ← Broker 1 promoted
                ├────────────┤  ├────────────┤  ├────────────┤
  Partition 2   │  follower  │  │  ☠️  DEAD  │  │  LEADER ★  │
                └────────────┘  └────────────┘  └────────────┘

  Kafka's Controller (a special broker role) detects the failure
  within seconds and triggers leader election automatically.
  Producers and consumers reroute to the new leaders transparently.
```

The speed of this election is what makes Kafka resilient in practice. A well-configured Kafka cluster can detect a broker failure and elect new leaders in a matter of seconds, with producers and consumers automatically reconnecting without any manual intervention. To the rest of your microservice architecture, the failure is invisible.

## The Complete Picture

Now that each piece is understood individually, here is how they all fit together into a single coherent system:

```
  Kafka Architecture — Full View

  PRODUCERS                    KAFKA CLUSTER                  CONSUMERS
  ─────────                    ─────────────                  ─────────

  ┌──────────┐                ┌──────────────────────────────────────────────────┐
  │  Order   │  write to      │  Topic: "order.placed"  (RF=3, 3 partitions)     │
  │ Service  │──────────────▶ │                                                  │
  └──────────┘  P1 (keyed)    │  Broker 1        Broker 2        Broker 3        │
                              │  ┌──────────┐   ┌──────────┐   ┌──────────┐      │
  ┌──────────┐                │  │ P0 LEAD★ │  │ P1 LEAD★ │   │ P2 LEAD★ │      │
  │ Payment  │  write to      │  │ P1 follow│   │ P2 follow│   │ P0 follow│      │
  │ Service  │──────────────▶ │  │ P2 follow│   │ P0 follow│   │ P1 follow│      │
  └──────────┘  P0 (keyed)    │  └──────────┘   └──────────┘   └──────────┘      │
                              └────────┬───────────────┬───────────────┬─────────┘
                                       │               │               │
                            ┌──────────▼───────────────▼───────────────▼──────────┐
                            │   Consumer Group: "inventory-service"               │
                            │   ┌──────────┐       ┌──────────┐  ┌──────────┐     │
                            │   │Instance 1│       │Instance 2│  │Instance 3│     │
                            │   │ reads P0 │       │ reads P1 │  │ reads P2 │     │
                            │   │offset: 42│       │offset: 38│  │offset: 51│     │
                            │   └──────────┘       └──────────┘  └──────────┘     │
                            └─────────────────────────────────────────────────────┘
                            ┌──────────────────────────────────────────────────────┐
                            │   Consumer Group: "analytics-service"                │
                            │   ┌────────────────────────────────────────────┐     │
                            │   │ Instance 1 (reads P0 + P1 + P2)            │     │
                            │   │ offsets: P0→12, P1→9, P2→17                │     │
                            │   └────────────────────────────────────────────┘     │
                            └──────────────────────────────────────────────────────┘
```

Every concept reinforces the others. Topics give your events a home. Partitions give your system scale. Producers write the events. Consumers read them at their own pace, tracked by offsets. Consumer groups allow parallel reading and independent progress. Brokers are the physical machines that store and serve everything. The replication factor ensures no broker failure loses your data. And the leader-follower model ensures every read and write has a clear, authoritative destination — with silent standbys ready to take over at any moment.
