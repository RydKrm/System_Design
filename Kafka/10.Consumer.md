# Everything About Kafka Consumers

### Settings, Message Flow, Registration, Design Principles, and a Production-Grade franz-go Implementation

---

## Preface — Why the Consumer Deserves Its Own Deep Study

The producer's job in Kafka is comparatively simple: serialise, route, send. The consumer's job is where almost all of the real complexity in a Kafka-based system lives. The consumer must coordinate with other instances of itself, track its own progress reliably, survive crashes without losing or duplicating work, recover gracefully when partitions move, and do all of this while still processing messages fast enough to keep up with production traffic. A producer that is slightly misconfigured loses you some performance. A consumer that is slightly misconfigured loses you data, duplicates work, or silently stops processing — these are the failures that wake people up at night.

This guide is organised in five parts, exactly following your five questions: every consumer setting and what it controls, the complete journey of a message from topic to service, the exact registration mechanics of how a consumer joins and starts receiving data, the design principles that separate a toy consumer from a production-grade one, and finally a complete franz-go implementation that embodies every principle discussed.

# PART 1 — Every Kafka Consumer Setting, Explained

## Chapter 1.1 — Identity Settings

These settings establish who this consumer is and what group it belongs to. They are the foundation everything else builds on.

```
  ┌────────────────────────────────────────────────────────────────────┐
  │  group.id                                                          │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: identifies which "team" this consumer belongs to.   │
  │  Same group.id = instances SHARE partitions (split the work).      │
  │  Different group.id = each gets an INDEPENDENT full copy of        │
  │  every message, with its own offset bookmark.                      │
  │                                                                    │
  │  This is the single most consequential setting in the entire       │
  │  consumer configuration. Every other behaviour — rebalancing,      │
  │  offset tracking, partition assignment — happens WITHIN the        │
  │  scope of a group.id.                                              │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  client.id                                                         │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: a human-readable label for THIS SPECIFIC PROCESS,   │
  │  distinct from group.id which identifies the whole team.           │
  │  Appears in broker logs, Kafka UI, and JMX metrics.                │
  │  Does NOT affect partition assignment or routing — purely for      │
  │  observability. In Kubernetes, set this to the pod name so you     │
  │  can immediately identify which physical instance is lagging.      │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  group.instance.id                                                 │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: enables STATIC GROUP MEMBERSHIP. Normally, every    │
  │  time a consumer restarts, it gets a brand new member identity     │
  │  inside the group, which triggers a rebalance. If you set a        │
  │  STABLE group.instance.id (e.g. "inventory-svc-pod-3"), Kafka      │
  │  recognises this as the SAME member returning after a restart      │
  │  and skips the rebalance entirely — as long as it returns within   │
  │  session.timeout.ms. This is hugely valuable for rolling restarts  │
  │  in Kubernetes, where pods restart frequently but predictably.     │
  └────────────────────────────────────────────────────────────────────┘
```

## Chapter 1.2 — Offset Management Settings

These settings control where a consumer starts reading and how it tracks progress — the single most important category for correctness.

```
  ┌────────────────────────────────────────────────────────────────────┐
  │  auto.offset.reset                                                 │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: decides where to start reading when NO committed    │
  │  offset exists yet for this group on a given partition.            │
  │                                                                    │
  │  earliest → start from the oldest message still retained.          │
  │             Use when you need full history (rebuilding state,      │
  │             first-time analytics deployment).                      │
  │                                                                    │
  │  latest   → start from the next NEW message after subscribing.     │
  │             Use when historical messages are irrelevant or         │
  │             would cause incorrect side effects if reprocessed.     │
  │                                                                    │
  │  none     → throw an exception instead of silently picking one.    │
  │             Forces you to handle the "no offset" case explicitly.  │
  │                                                                    │
  │  CRITICAL: this setting is ONLY consulted when no offset exists.   │
  │  Once a group has a committed offset, every future read — even     │
  │  by a brand new instance — resumes from that committed position.   │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  enable.auto.commit                                                │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: when true, the client automatically commits         │
  │  offsets on a timer (auto.commit.interval.ms), regardless of       │
  │  whether your handler actually finished processing.                │
  │                                                                    │
  │  true  → convenient, but DANGEROUS. If the timer fires between     │
  │          "message received" and "message processed", a crash       │
  │          causes that message to be silently skipped on restart.    │
  │                                                                    │
  │  false → you control exactly when an offset is committed —         │
  │          only after your business logic confirms success.          │
  │          This is the production-grade choice for any service       │
  │          where correctness matters.                                │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  auto.commit.interval.ms                                           │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: only relevant when enable.auto.commit=true.         │
  │  How often (in milliseconds) the automatic commit timer fires.     │
  │  Default: 5000 (5 seconds). Irrelevant in production-grade         │
  │  systems that disable auto-commit in favour of manual commits.     │
  └────────────────────────────────────────────────────────────────────┘
```

## Chapter 1.3 — Liveness and Failure Detection Settings

These settings determine how Kafka decides a consumer instance is dead and needs its partitions reassigned — a delicate balance between fast failure recovery and avoiding false-positive rebalances.

```
  ┌────────────────────────────────────────────────────────────────────┐
  │  session.timeout.ms                                                │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the maximum time the Group Coordinator will wait    │
  │  without receiving a heartbeat before declaring this consumer      │
  │  dead and triggering a rebalance to redistribute its partitions.   │
  │                                                                    │
  │  Too LOW  → healthy consumers get falsely evicted during brief     │
  │             GC pauses or network blips. Causes unnecessary         │
  │             rebalance churn.                                       │
  │  Too HIGH → a genuinely dead consumer holds onto its partitions    │
  │             for a long time before another instance takes over,    │
  │             during which those partitions are not being processed. │
  │  Standard production value: 45000 (45 seconds).                    │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  heartbeat.interval.ms                                             │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: how often the client sends a heartbeat to the       │
  │  coordinator to prove it is alive. Sent from a background thread   │
  │  managed entirely by the client library — you never write          │
  │  heartbeat code yourself.                                          │
  │  Rule of thumb: must be less than session.timeout.ms / 3, so       │
  │  that at least 2-3 heartbeats can be missed before timeout fires.  │
  │  Standard production value: 3000 (3 seconds).                      │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  max.poll.interval.ms                                              │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: a SEPARATE liveness check from heartbeats. This is  │
  │  the maximum time allowed between two consecutive calls to fetch   │
  │  the next batch of messages. It exists because heartbeats can      │
  │  keep flowing from a background thread even while your main        │
  │  processing thread is stuck or deadlocked on a slow message.       │
  │  If you do not poll for new messages within this window, the       │
  │  coordinator evicts you EVEN IF your heartbeats look healthy.      │
  │  Standard production value: 300000 (5 minutes) — generous enough   │
  │  for normal processing spikes, but will catch genuine deadlocks.   │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  rebalance.timeout.ms                                              │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: during a rebalance, the maximum time all group      │
  │  members are given to rejoin before being removed from the group.  │
  │  Standard production value: 60000 (60 seconds).                    │
  └────────────────────────────────────────────────────────────────────┘
```

## Chapter 1.4 — Fetch Performance Settings

These settings control the trade-off between latency and network efficiency when the consumer requests data from the broker.

```
  ┌────────────────────────────────────────────────────────────────────┐
  │  fetch.min.bytes                                                   │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the broker will NOT respond to a fetch request      │
  │  until at least this many bytes of data are available across all   │
  │  requested partitions — UNLESS fetch.max.wait.ms expires first.    │
  │  1 (default) → respond as soon as any data is available.           │
  │                Lowest latency, most fetch requests.                │
  │  Higher values (e.g. 50000) → wait for more data to accumulate,    │
  │                reducing request count at the cost of latency.      │
  │                Useful for very high-throughput batch-style         │
  │                consumers where latency matters less than           │
  │                network efficiency.                                 │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  fetch.max.wait.ms (aka fetch.wait.max.ms)                         │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the maximum time the broker will wait to satisfy    │
  │  fetch.min.bytes before responding anyway with whatever is         │
  │  available (even if it is nothing). This caps the worst-case       │
  │  latency when fetch.min.bytes is set high but the topic is quiet.  │
  │  Standard value: 500ms.                                            │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  max.partition.fetch.bytes                                         │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the maximum number of bytes the broker will return  │
  │  PER PARTITION in a single fetch response. With 6 assigned         │
  │  partitions and 1MB per partition, a single fetch response could   │
  │  be up to 6MB. Must be at least as large as the maximum message    │
  │  size your producers send, or messages larger than this limit      │
  │  will never be fetchable.                                          │
  │  Standard value: 1048576 (1 MB).                                   │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  fetch.max.bytes                                                   │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the maximum total bytes across ALL partitions in    │
  │  a single fetch response, regardless of how many partitions are    │
  │  assigned. This is a ceiling on top of max.partition.fetch.bytes.  │
  │  Standard value: 52428800 (50 MB).                                 │
  └────────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────┐
  │  max.poll.records                                                  │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the maximum number of records returned in a single  │
  │  poll/fetch call to your application code, even if more are        │
  │  available in the broker's response. Controls how large a "batch"  │
  │  your handler code processes at once — useful for bounding memory  │
  │  usage and processing latency per poll cycle.                      │
  │  Standard value: 500.                                              │
  └────────────────────────────────────────────────────────────────────┘
```

## Chapter 1.5 — Partition Assignment Strategy Settings

```
  ┌────────────────────────────────────────────────────────────────────┐
  │  partition.assignment.strategy                                     │
  ├────────────────────────────────────────────────────────────────────┤
  │  What it does: the algorithm used to decide which partitions go    │
  │  to which group member during a rebalance.                         │
  │                                                                    │
  │  RangeAssignor (legacy default in some clients):                   │
  │    Assigns consecutive partition ranges per topic. Can create      │
  │    imbalance with multiple topics. Always EAGER (stop-the-world).  │
  │                                                                    │
  │  RoundRobinAssignor:                                               │
  │    Distributes partitions evenly across consumers in a single      │
  │    pass. Better balance than Range. Still EAGER.                   │
  │                                                                    │
  │  CooperativeStickyAssignor (recommended):                          │
  │    Minimises partition movement during rebalances — keeps as       │
  │    many existing assignments unchanged as possible. Uses           │
  │    COOPERATIVE (incremental) rebalancing — only revokes the        │
  │    specific partitions that need to move, leaving all other        │
  │    partitions actively consuming throughout the rebalance.         │
  │                                                                    │
  │  franz-go uses cooperative-sticky-style balancing BY DEFAULT.      │
  │  You do not need to configure this — it is the built-in behaviour. │
  └────────────────────────────────────────────────────────────────────┘
```

# PART 2 — The Complete Message Journey: Topic to Service

## Chapter 2.1 — Where the Message Lives Before Anyone Reads It

Before a consumer can read anything, a message must already exist durably on a broker's disk, written there by a producer and replicated to follower brokers. This durable, ordered, on-disk log is the foundation everything else sits on top of.

```
  THE MESSAGE'S RESTING STATE — BEFORE ANY CONSUMER TOUCHES IT

  Broker 2 (leader for Partition 3 of "order.placed")
  ┌────────────────────────────────────────────────────────────┐
  │  Disk: /var/kafka-data/order.placed-3/                     │
  │                                                            │
  │  00000000000000000000.log                                  │
  │  ┌──────┬──────┬──────┬──────┬──────┬──────┐               │
  │  │ off0 │ off1 │ off2 │ off3 │ off4 │ off5 │ ← appended    │
  │  └──────┴──────┴──────┴──────┴──────┴──────┘  sequentially │
  │                                                            │
  │  High Watermark: offset 4                                  │
  │  (offset 5 written but not yet replicated to all ISR       │
  │   members — not yet visible to any consumer)               │
  └────────────────────────────────────────────────────────────┘

  This message sits here, durable and waiting, completely
  indifferent to whether zero, one, or a hundred consumers
  will eventually read it. Reading does not consume or remove
  the message — unlike a traditional queue, Kafka's read
  operation is non-destructive.
```

## Chapter 2.2 — The Eight-Step Connection Sequence

This is the full mechanical sequence — covered briefly in an earlier guide — expanded here in complete detail because it directly answers "how does a consumer register to a topic and how does data start flowing."

```
  STEP 1 — BOOTSTRAP: Discover the cluster
  ═══════════════════════════════════════════════════════════════
  Your consumer's only hardcoded knowledge is ONE broker address.
  It opens a TCP connection to that broker and requests metadata.

  Consumer                              Seed Broker
       │── TCP connect  ─────────────────────▶│
       │── "Send me cluster metadata" ───────▶│
       │◀─ "Brokers: [B1,B2,B3]               │
       │    Topic order.placed has 6 P's:     │
       │    P0 leader=B2  P1 leader=B3        │
       │    P2 leader=B1  P3 leader=B2 ..."   │

  The consumer now has a complete map of the cluster. The seed
  broker's job is done — it was just a doorman, not necessarily
  involved in any further communication.

  STEP 2 — FIND THE GROUP COORDINATOR
  ═══════════════════════════════════════════════════════════════
  Every consumer group has ONE broker responsible for managing
  its membership — the Group Coordinator. Determined by:
  hash(group.id) % number_of_brokers

  Consumer                              Computed Coordinator (e.g. B2)
       │── "Who coordinates group             │
       │    'inventory-service'?" ──────────▶ │
       │◀─ "I do." ───────────────────────────│

  STEP 3 — JOIN GROUP
  ═══════════════════════════════════════════════════════════════
  The consumer registers itself as a group member.

       │── JoinGroup request: ───────────────▶│
       │   group.id: "inventory-service"      │
       │   member.id: "" (empty on first join)│
       │   client.id: "inventory-svc-pod-3"   │
       │   subscribed topics: ["order.placed"]│
       │   session.timeout.ms: 45000          │
       │                                      │
       │   (consumer WAITS — coordinator      │
       │    gathers ALL members before        │
       │    responding to any of them)        │

  Other instances send their own JoinGroup requests around the
  same time. The coordinator waits a brief window for stragglers,
  then proceeds once it believes it has the full membership.

  STEP 4 — GROUP LEADER ELECTION
  ═══════════════════════════════════════════════════════════════
  The coordinator designates ONE member as the "Group Leader" —
  unrelated to partition leadership, this is purely about who
  computes the partition assignment for this rebalance.

       Coordinator ──▶ Member-1: "You are the leader. Here is the
                        full member list and topic metadata.
                        Compute the assignment."
       Coordinator ──▶ Member-2,3,4: "You are followers. Wait for
                        SyncGroup."

  STEP 5 — COMPUTE ASSIGNMENT (happens IN the consumer, not broker)
  ═══════════════════════════════════════════════════════════════
  This is a crucial and often misunderstood detail: Kafka brokers
  do NOT decide partition assignment. The Group Leader — one of
  YOUR consumer processes — runs the assignment algorithm locally
  using the chosen strategy (e.g. CooperativeStickyAssignor) and
  produces the full mapping.

         Member-1 (Leader) computes:
         Member-1 → P0, P1
         Member-2 → P2, P3
         Member-3 → P4, P5
         Member-4 → (none — fewer partitions than members)

  STEP 6 — SYNC GROUP: Distribute the assignment
  ═══════════════════════════════════════════════════════════════
       Member-1     ── SyncGroup(full assignment) ──▶ Coordinator
       Member-2,3,4 ── SyncGroup(empty, waiting)  ──▶ Coordinator

       Coordinator ──▶ Member-1: "Your partitions: P0, P1"
       Coordinator ──▶ Member-2: "Your partitions: P2, P3"
       Coordinator ──▶ Member-3: "Your partitions: P4, P5"
       Coordinator ──▶ Member-4: "Your partitions: (none)"

  STEP 7 — FETCH COMMITTED OFFSETS
  ═══════════════════════════════════════════════════════════════
  Each member now knows WHICH partitions it owns, but not WHERE
  to start reading. It asks the coordinator for the last committed
  offset for each owned partition.

       Member-1 ── OffsetFetch(P0, P1) ──▶ Coordinator
       Member-1 ◀── "P0: offset 88421, P1: offset 91003" ──

       If this group has NEVER read this partition before:
       Member-1 ◀── "P0: no committed offset" ──
       → auto.offset.reset decides: earliest or latest

  STEP 8 — FETCH: Begin reading actual message data
  ═══════════════════════════════════════════════════════════════
  Now the consumer knows exactly which partitions it owns AND
  exactly which offset to resume from for each. It sends Fetch
  requests DIRECTLY to the broker that LEADS each partition
  (which may be a different broker than the Group Coordinator).

       Member-1 ── Fetch(P0, from offset 88422) ──▶ Broker (P0 leader)
       Member-1 ◀── [msg88422, msg88423, msg88424, ...] ──

       Member-1 ── Fetch(P1, from offset 91004) ──▶ Broker (P1 leader)
       Member-1 ◀── [msg91004, msg91005, ...] ──

  Your handler function finally receives these messages.
  This is the moment data "arrives" in your service.
```

## Chapter 2.3 — What Runs Continuously After Connection

Once the eight-step handshake completes, two independent processes run for the entire lifetime of the consumer, and understanding both is essential.

```
  CONTINUOUS PROCESS 1: The Poll/Fetch Loop (driven by your code)

  ┌──────────────────────────────────────────────────────────────┐
  │  loop forever:                                               │
  │    records = client.PollFetches(ctx)  ← ask for next batch   │
  │    for each record in records:                               │
  │        handler(record)               ← your business logic   │
  │        markForCommit(record)         ← mark as "done"        │
  │    commitMarkedOffsets()             ← persist the bookmark  │
  └──────────────────────────────────────────────────────────────┘

  CONTINUOUS PROCESS 2: The Heartbeat (managed by the client library)

  ┌──────────────────────────────────────────────────────────────┐
  │  Every heartbeat.interval.ms (e.g. every 3 seconds):         │
  │    client ── Heartbeat("I am alive, group X, gen Y") ──▶Coord│
  │    Coord ◀── "OK, your assignment is still valid"            │
  │                                                              │
  │  This runs on a SEPARATE background goroutine/thread from    │
  │  your poll loop. You never write this code — the library     │
  │  handles it. But its health depends on your poll loop NOT    │
  │  blocking for longer than max.poll.interval.ms.              │
  └──────────────────────────────────────────────────────────────┘

  Why both matter together:
  ─────────────────────────────────────────────────────────────────
  If heartbeats stop (process dies, network partition):
    → session.timeout.ms expires → coordinator declares dead
    → rebalance triggered → partitions reassigned

  If heartbeats keep flowing but PollFetches is never called again
  (deadlocked handler, infinite loop in business logic):
    → max.poll.interval.ms expires → coordinator declares dead
    → rebalance triggered → partitions reassigned
    → even though heartbeats looked perfectly healthy!

  This is why max.poll.interval.ms exists as a SEPARATE check
  from session.timeout.ms — they catch different failure modes.
```

## Chapter 2.4 — The Full Picture: One Diagram From Topic to Handler

```
  COMPLETE DATA FLOW: TOPIC → BROKER → CONSUMER → YOUR HANDLER

  ┌─────────────────────────────────────────────────────────────────┐
  │  Topic "order.placed" — Partition 3 — Broker 2 (leader)         │
  │                                                                 │
  │  Disk: [off0][off1][off2][off3][off4] ← High Watermark at 4     │
  └────────────────────────┬────────────────────────────────────────┘
                           │
                           │ Step 8: Fetch(P3, from offset 89)
                           ▼
              ┌────────────────────────────┐
              │ Broker 2 reads from disk   │
              │ (often served from OS page │
              │  cache — very fast, RAM    │
              │  speed for recent data)    │
              └───────────┬────────────────┘
                          │ FetchResponse: [msg89, msg90, msg91]
                          ▼
              ┌────────────────────────────────────────┐
              │  Consumer Process (inventory-svc-pod-3)│
              │                                        │
              │  Network layer receives bytes          │
              │  ↓                                     │
              │  Deserialise into kgo.Record structs   │
              │  ↓                                     │
              │PollFetches() returns batch to your code│
              │  ↓                                     │
              │  fetches.EachRecord(func(r) {          │
              │     yourHandler(r.Topic, r.Key,r.Value)│
              │  })                                    │
              │  ↓                                     │
              │  Your business logic executes:         │
              │    - check inventory stock             │
              │    - deduct quantity                   │

              │  ↓                                     │
              │  MarkCommitRecords(r) — mark as done   │
              │  ↓                                     │
              │  CommitMarkedOffsets() — persist offset│
              │    to __consumer_offsets topic         │
              └────────────────────────────────────────┘
```

# PART 3 — Design Principles for a Large-Scale Consumer

## Chapter 3.1 — Idempotency Is Not Optional

In any consumer that uses manual offset commits (the production-correct choice), at-least-once delivery is the guarantee you get. This means every single handler you write WILL be called more than once for some messages, at some point, under some failure scenario. This is not a rare edge case to defend against — it is a guaranteed eventual occurrence in any long-running system. Your handler must produce the same end state whether it runs once or five times for the same message.

```
  IDEMPOTENCY PATTERNS

  ┌────────────────────────────────────────────────────────────────┐
  │  PATTERN: Idempotency key with a database unique constraint    │
  │                                                                │
  │  BEGIN TRANSACTION                                             │
  │    INSERT INTO processed_events (event_id, processed_at)       │
  │    VALUES ($1, NOW())                                          │
  │    ON CONFLICT (event_id) DO NOTHING                           │
  │    RETURNING event_id;                                         │
  │                                                                │
  │    -- if no row was returned, this event was already processed │
  │    -- skip the business logic entirely, commit offset, move on │
  │                                                                │
  │    -- if a row WAS returned, proceed with business logic       │
  │    UPDATE inventory SET quantity = quantity - 1 WHERE ...      │
  │  COMMIT TRANSACTION                                            │
  └────────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────┐
  │  PATTERN: Natural idempotency through "set" semantics          │
  │                                                                │
  │  Instead of: "increment counter by 1" (NOT idempotent —        │
  │              running twice increments by 2)                    │
  │                                                                │
  │  Use:        "set status = CONFIRMED" (IS idempotent —         │
  │              running twice still leaves status = CONFIRMED)    │
  │                                                                │
  │  Wherever your domain allows it, prefer absolute state         │
  │  transitions over relative increments/decrements.              │
  └────────────────────────────────────────────────────────────────┘
```

## Chapter 3.2 — Handler Speed and the Poll Loop Contract

Every consumer has an implicit contract with Kafka's group coordination system: keep calling PollFetches (or ReadMessage) regularly, or be evicted. This means slow handlers are not just a performance problem — they are a correctness and availability problem, because a sufficiently slow handler triggers unwanted rebalances.

```
  THE THREE STRATEGIES FOR HANDLING SLOW WORK

  Strategy 1: Increase max.poll.interval.ms
  ─────────────────────────────────────────────────────────────────
  Simple but limited. Only buys you time; does not solve the
  underlying problem of slow, serial processing.

  Strategy 2: Offload to a worker pool, poll loop stays fast
  ─────────────────────────────────────────────────────────────────
  for record := range incomingRecords {
      workerPool.Submit(record)  ← non-blocking handoff
  }
  // Poll loop keeps calling PollFetches() rapidly.
  // Workers process in parallel in the background.
  // CAUTION: you must track completion before committing offsets,
  // or you risk committing before processing actually finishes.

  Strategy 3: Pause partitions during slow processing
  ─────────────────────────────────────────────────────────────────
  client.PauseFetchPartitions(...)  ← stop fetching new data
  // process the current backlog
  client.ResumeFetchPartitions(...) ← resume fetching
  // Heartbeats keep flowing (separate thread) even while paused,
  // so you stay in the group without triggering eviction.
  // This is the cleanest way to apply backpressure deliberately.
```

## Chapter 3.3 — Dead Letter Queues: What Happens to Poison Messages

Without a dead-letter strategy, a single malformed or perpetually-failing message will block its entire partition forever — because the consumer refuses to commit past it, and refuses to skip it, so it retries the exact same message infinitely.

```
  THE POISON MESSAGE PROBLEM AND ITS SOLUTION

  WITHOUT a Dead Letter Queue:
  ─────────────────────────────────────────────────────────────────
  Partition 3: [...][msg47: malformed JSON][msg48][msg49]...
                      ▲
                      │ handler fails every time
                      │ offset never advances past 47
                      │ msg48, msg49... NEVER GET PROCESSED
                      │ even though they are perfectly fine
  This ONE bad message blocks the ENTIRE partition forever.

  WITH a Dead Letter Queue pattern:
  ─────────────────────────────────────────────────────────────────
  attempt 1: handler(msg47) fails → retry count = 1
  attempt 2: handler(msg47) fails → retry count = 2
  attempt 3: handler(msg47) fails → retry count = 3
             retry count >= maxRetries (3) →
             publish msg47 to "order.placed.dlq" topic
             commit offset past msg47 (give up retrying it)
             continue processing msg48, msg49... normally

  The DLQ topic accumulates problem messages for manual inspection,
  alerting, or a separate slow-path reprocessing job — without
  ever blocking the healthy message flow.
```

## Chapter 3.4 — Graceful Shutdown

A consumer that is killed abruptly (SIGKILL, OOM kill, hard pod eviction) cannot do anything about offset commits or partition handoff. But a consumer that receives a graceful termination signal (SIGTERM, which Kubernetes sends before SIGKILL) has a window of time to shut down cleanly — and using that window correctly significantly reduces both duplicate processing and rebalance-induced delays for the rest of the group.

```
  GRACEFUL SHUTDOWN SEQUENCE

  1. Receive SIGTERM (Kubernetes sends this before SIGKILL,
     typically giving you 30 seconds via terminationGracePeriodSeconds)

  2. Stop accepting NEW poll cycles
     (let the current in-flight batch finish processing)

  3. Commit all marked offsets for completed work
     client.CommitUncommittedOffsets(ctx) ← blocking, waits for ack

  4. Send LeaveGroup to the coordinator
     client.Close() does this automatically in franz-go
     → triggers an IMMEDIATE rebalance instead of making the
       coordinator wait the full session.timeout.ms (45s) to
       notice this member is gone

  5. Close the connection

  Without step 4, the rest of the group experiences a 45-second
  delay before this instance's partitions are reassigned — even
  though the instance shut down cleanly and immediately. Calling
  Close() properly is the difference between near-zero handoff
  delay and a 45-second gap in processing for those partitions.
```

## Chapter 3.5 — Monitoring: Consumer Lag Is Your Vital Sign

Of all the metrics in a Kafka-based system, **consumer lag** — the difference between the latest offset in a partition and the consumer group's committed offset — is the single most important one to watch continuously. It tells you, in real time, whether your consumers are keeping up with production.

```
  CONSUMER LAG IN PRACTICE

  Partition 3 latest offset: 1,204,558
  Group "inventory-svc" committed offset: 1,204,510
  Lag = 1,204,558 − 1,204,510 = 48 messages behind

  ┌────────────────────────────────────────────────────────────┐
  │  Healthy lag pattern: fluctuates near zero, occasional     │
  │  small spikes during traffic bursts, recovers quickly.     │
  │                                                            │
  │  Unhealthy lag pattern: monotonically increasing over time.│
  │  This means your consumers process slower than producers   │
  │  publish. Without intervention, lag grows forever.         │
  │                                                            │
  │  Causes of growing lag:                                    │
  │    - Not enough consumer instances for the partition count │
  │    - Slow downstream dependency (database, external API)   │
  │    - Handler doing unnecessary synchronous work            │
  │    - A poison message causing repeated retry loops         │
  │                                                            │
  │  Fix: scale consumer instances (up to partition count),    │
  │  optimise the slow path, or add a dead-letter mechanism.   │
  └────────────────────────────────────────────────────────────┘

  Alerting rule of thumb: alert when lag exceeds a threshold AND
  continues growing for more than 5 minutes (a brief spike during
  a deploy or traffic burst is normal and should not page anyone).
```

## Chapter 3.6 — The Full Design Checklist

```
  PRODUCTION CONSUMER DESIGN CHECKLIST

  Correctness
  ──────────────────────────────────────────────────────────────────
  ☐  Manual offset commits, never auto-commit
  ☐  Idempotent handlers (safe to process the same message twice)
  ☐  Offsets committed only AFTER successful processing
  ☐  Dead-letter queue for messages that fail repeatedly

  Liveness
  ──────────────────────────────────────────────────────────────────
  ☐  Handler never blocks longer than max.poll.interval.ms
  ☐  Slow work offloaded to worker pools, not done inline in poll loop
  ☐  Graceful shutdown on SIGTERM: commit + LeaveGroup before exit

  Scalability
  ──────────────────────────────────────────────────────────────────
  ☐  Partition count chosen for target peak parallelism
  ☐  Consumer instance count does not exceed partition count
  ☐  Cooperative rebalancing enabled (default in franz-go)

  Observability
  ──────────────────────────────────────────────────────────────────
  ☐  Consumer lag tracked and alerted on per partition per group
  ☐  Structured logging includes topic, partition, offset, key
  ☐  Distributed tracing propagated through message headers

  Resilience
  ──────────────────────────────────────────────────────────────────
  ☐  Retries with backoff for transient handler failures
  ☐  Circuit breaker for downstream dependency failures
  ☐  Static group membership (group.instance.id) for stable
     rolling restarts in Kubernetes
```

# PART 4 — Production-Grade Consumer in Go with franz-go

This implementation embodies every principle from Part 3: manual commits, idempotency-friendly hooks, dead-letter handling, worker-pool offloading, graceful shutdown, and structured observability.

```go
package consumer

import (
    "context"
    "encoding/json"
    "errors"
    "fmt"
    "log"
    "sync"
    "sync/atomic"
    "time"

    "github.com/twmb/franz-go/pkg/kgo"
)

// ─────────────────────────────────────────────────────────────────
// TYPES
// ─────────────────────────────────────────────────────────────────

// Handler is the function signature every business-logic handler
// implements. Returning nil marks the message for offset commit.
// Returning an error means the message will be retried (and
// eventually dead-lettered after maxRetries).
type Handler func(ctx context.Context, record *kgo.Record) error

// retryState tracks how many times we have attempted a specific
// record, keyed by a composite of topic+partition+offset since
// that uniquely identifies a message's position forever.
type retryState struct {
    mu     sync.Mutex
    counts map[string]int
}

func newRetryState() *retryState {
    return &retryState{counts: make(map[string]int)}
}

func recordKey(r *kgo.Record) string {
    return fmt.Sprintf("%s-%d-%d", r.Topic, r.Partition, r.Offset)
}

func (rs *retryState) increment(r *kgo.Record) int {
    rs.mu.Lock()
    defer rs.mu.Unlock()
    k := recordKey(r)
    rs.counts[k]++
    return rs.counts[k]
}

func (rs *retryState) clear(r *kgo.Record) {
    rs.mu.Lock()
    defer rs.mu.Unlock()
    delete(rs.counts, recordKey(r))
}

// ─────────────────────────────────────────────────────────────────
// CONFIG
// ─────────────────────────────────────────────────────────────────

// Config holds every tunable for the production consumer.
// Every field maps directly to a setting explained in Part 1.
type Config struct {
    Brokers            []string
    GroupID            string
    ClientID           string
    GroupInstanceID    string // empty = disabled (no static membership)
    Topics             []string
    SessionTimeout     time.Duration
    HeartbeatInterval  time.Duration
    MaxPollInterval    time.Duration
    RebalanceTimeout   time.Duration
    StartFromBeginning bool   // true = AtStart, false = AtEnd
    WorkerPoolSize     int    // number of goroutines processing in parallel
    MaxRetries         int    // attempts before sending to DLQ
    DLQTopic           string // empty = log and drop instead of DLQ
}

// DefaultConfig returns production-sensible defaults.
// Override only what your service specifically needs to differ.
func DefaultConfig(brokers []string, groupID, clientID string,
    topics ...string) Config {
    return Config{
        Brokers:            brokers,
        GroupID:            groupID,
        ClientID:           clientID,
        Topics:             topics,
        SessionTimeout:     45 * time.Second,
        HeartbeatInterval:  3 * time.Second,
        MaxPollInterval:    5 * time.Minute,
        RebalanceTimeout:   60 * time.Second,
        StartFromBeginning: true,
        WorkerPoolSize:     8,
        MaxRetries:         3,
        DLQTopic:           "",
    }
}

// ─────────────────────────────────────────────────────────────────
// THE CONSUMER
// ─────────────────────────────────────────────────────────────────

// Consumer is a production-grade Kafka consumer built on franz-go.
// It implements every design principle from Part 3:
//   - manual offset commits after successful processing
//   - worker pool for parallel handler execution without blocking
//     the poll loop's heartbeat-adjacent fetch cycle
//   - retry tracking with automatic dead-letter publishing
//   - graceful shutdown that commits and leaves the group cleanly
//   - structured logging for every lifecycle event
type Consumer struct {
    client      *kgo.Client
    dlqProducer *kgo.Client // separate client for publishing to DLQ
    cfg         Config
    handlers    map[string]Handler
    handlersMu  sync.RWMutex
    retries     *retryState

    // workQueue is the handoff channel between the poll loop and
    // the worker pool. The poll loop NEVER calls handlers directly —
    // it only enqueues work, keeping PollFetches calls frequent
    // and protecting against max.poll.interval.ms eviction even
    // when individual handlers are slow.
    workQueue chan *kgo.Record

    // inFlight tracks how many records are currently being processed
    // by the worker pool but not yet committed. Used during shutdown
    // to wait for in-flight work to drain before the final commit.
    inFlight atomic.Int64

    wg sync.WaitGroup
}

// New creates a fully configured production consumer.
func New(cfg Config) (*Consumer, error) {
    c := &Consumer{
        cfg:       cfg,
        handlers:  make(map[string]Handler),
        retries:   newRetryState(),
        workQueue: make(chan *kgo.Record, cfg.WorkerPoolSize*4),
    }

    opts := []kgo.Opt{
        // ── IDENTITY ──────────────────────────────────────────────
        kgo.SeedBrokers(cfg.Brokers...),
        kgo.ClientID(cfg.ClientID),
        kgo.ConsumerGroup(cfg.GroupID),
        kgo.ConsumeTopics(cfg.Topics...),

        // ── OFFSET MANAGEMENT ─────────────────────────────────────
        // Manual commits only. This is non-negotiable for a
        // production-grade consumer per the design principles
        // in Part 3 — auto-commit risks committing offsets for
        // work that has not actually finished processing.
        kgo.DisableAutoCommit(),

        // ── LIVENESS TUNING ───────────────────────────────────────
        kgo.SessionTimeout(cfg.SessionTimeout),
        kgo.HeartbeatInterval(cfg.HeartbeatInterval),
        kgo.RebalanceTimeout(cfg.RebalanceTimeout),

        // ── REBALANCE CALLBACKS ───────────────────────────────────
        // OnPartitionsRevoked fires BEFORE partitions are taken away.
        // We use this moment to commit everything we have successfully
        // processed so far, minimising duplicate work for whichever
        // instance picks up these partitions next.
        kgo.OnPartitionsRevoked(c.onPartitionsRevoked),

        // OnPartitionsAssigned fires when we receive new partitions
        // (startup, rebalance, or taking over from a failed instance).
        kgo.OnPartitionsAssigned(c.onPartitionsAssigned),

        // OnPartitionsLost fires specifically when partitions are
        // lost WITHOUT a clean revoke — e.g. this instance was too
        // slow to respond and the coordinator already reassigned
        // them elsewhere. We log this distinctly because it signals
        // a tuning problem (max.poll.interval.ms likely too low for
        // your actual processing time, or a genuine instance failure).
        kgo.OnPartitionsLost(c.onPartitionsLost),
    }

    // Static group membership: if configured, this consumer keeps
    // its identity across restarts (e.g. pod restarts in Kubernetes),
    // avoiding a full rebalance for routine, brief restarts.
    if cfg.GroupInstanceID != "" {
        opts = append(opts, kgo.InstanceID(cfg.GroupInstanceID))
    }

    if cfg.StartFromBeginning {
        opts = append(opts, kgo.ConsumeResetOffset(kgo.NewOffset().AtStart()))
    } else {
        opts = append(opts, kgo.ConsumeResetOffset(kgo.NewOffset().AtEnd()))
    }

    client, err := kgo.NewClient(opts...)
    if err != nil {
        return nil, fmt.Errorf("failed to create consumer client: %w", err)
    }

    pingCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    if err := client.Ping(pingCtx); err != nil {
        client.Close()
        return nil, fmt.Errorf("kafka unreachable: %w", err)
    }
    c.client = client

    // The DLQ producer is a SEPARATE, lightweight client used only
    // for publishing failed messages. Kept separate from the consumer
    // client because producing and consuming have different tuning
    // needs (acks=all for the DLQ to guarantee it is never lost).
    if cfg.DLQTopic != "" {
        dlqClient, err := kgo.NewClient(
            kgo.SeedBrokers(cfg.Brokers...),
            kgo.ClientID(cfg.ClientID+"-dlq-producer"),
            kgo.RequiredAcks(kgo.AllISRAcks()),
        )
        if err != nil {
            client.Close()
            return nil, fmt.Errorf("failed to create DLQ producer: %w", err)
        }
        c.dlqProducer = dlqClient
    }

    log.Printf("[%s] Consumer ready. group=%s topics=%v workers=%d",
        cfg.ClientID, cfg.GroupID, cfg.Topics, cfg.WorkerPoolSize)
    return c, nil
}

// RegisterHandler maps a topic to its business-logic handler.
func (c *Consumer) RegisterHandler(topic string, h Handler) {
    c.handlersMu.Lock()
    defer c.handlersMu.Unlock()
    c.handlers[topic] = h
    log.Printf("[%s] Handler registered: %s", c.cfg.ClientID, topic)
}

// onPartitionsRevoked commits everything we have processed so far
// for the partitions being taken away. This is the single most
// important callback for minimising duplicate processing during
// rebalances — without it, in-flight progress is lost and the next
// owner re-reads from the LAST COMMITTED offset, redoing work we
// had already completed but not yet persisted.
func (c *Consumer) onPartitionsRevoked(ctx context.Context,
    cl *kgo.Client, revoked map[string][]int32) {
    log.Printf("[%s] Partitions revoked: %v — draining and committing",
        c.cfg.ClientID, revoked)

    // Wait briefly for in-flight worker pool items to finish, so we
    // can mark and commit as much real progress as possible before
    // losing ownership. Bounded wait — we cannot hold up the
    // rebalance indefinitely.
    deadline := time.Now().Add(5 * time.Second)
    for c.inFlight.Load() > 0 && time.Now().Before(deadline) {
        time.Sleep(50 * time.Millisecond)
    }

    if err := cl.CommitUncommittedOffsets(ctx); err != nil {
        log.Printf("[%s] Commit on revoke failed: %v", c.cfg.ClientID, err)
    }
}

func (c *Consumer) onPartitionsAssigned(ctx context.Context,
    cl *kgo.Client, assigned map[string][]int32) {
    log.Printf("[%s] Partitions assigned: %v", c.cfg.ClientID, assigned)
}

// onPartitionsLost fires when we lose partitions WITHOUT a graceful
// revoke — this is a signal worth alerting on, since it usually
// means this instance was too slow (max.poll.interval.ms exceeded)
// or experienced a network partition.
func (c *Consumer) onPartitionsLost(ctx context.Context,
    cl *kgo.Client, lost map[string][]int32) {
    log.Printf("[%s] ⚠️  Partitions LOST (ungracefully): %v — "+
        "check for slow handlers or network issues",
        c.cfg.ClientID, lost)
}

// Start launches the worker pool and begins the poll loop.
// Run this as a goroutine from main(): go consumer.Start(ctx)
func (c *Consumer) Start(ctx context.Context) error {
    // Launch the worker pool BEFORE the poll loop, so workers are
    // ready to receive from workQueue the moment records arrive.
    for i := 0; i < c.cfg.WorkerPoolSize; i++ {
        c.wg.Add(1)
        go c.worker(ctx, i)
    }

    log.Printf("[%s] 👂 Poll loop + %d workers started",
        c.cfg.ClientID, c.cfg.WorkerPoolSize)

    for {
        // PollFetches must be called FREQUENTLY and must NOT block
        // on slow handler execution — that work happens in the
        // worker pool via workQueue. This is precisely the design
        // principle from Part 3.2: keep the poll loop fast, offload
        // slow work elsewhere.
        fetches := c.client.PollFetches(ctx)

        if fetches.IsClientClosed() {
            log.Printf("[%s] Client closed, draining work queue",
                c.cfg.ClientID)
            close(c.workQueue)
            c.wg.Wait() // wait for all workers to finish in-flight work
            return nil
        }

        if errs := fetches.Errors(); len(errs) > 0 {
            for _, e := range errs {
                log.Printf("[%s] ⚠️  Fetch error topic=%s p=%d: %v",
                    c.cfg.ClientID, e.Topic, e.Partition, e.Err)
            }
        }

        // Enqueue every record for worker processing. This is a
        // fast, non-blocking-ish handoff (channel send) — NOT the
        // slow business logic itself. The poll loop returns to
        // PollFetches again almost immediately.
        fetches.EachRecord(func(record *kgo.Record) {
            select {
            case c.workQueue <- record:
            case <-ctx.Done():
            }
        })
    }
}

// worker is one of cfg.WorkerPoolSize goroutines that pull records
// off workQueue and execute the registered handler, with retry
// tracking and dead-letter escalation built in.
func (c *Consumer) worker(ctx context.Context, id int) {
    defer c.wg.Done()
    for record := range c.workQueue {
        c.inFlight.Add(1)
        c.processRecord(ctx, record)
        c.inFlight.Add(-1)
    }
    log.Printf("[%s] Worker %d exiting", c.cfg.ClientID, id)
}

// processRecord runs the registered handler for one record, with
// retry counting and DLQ escalation on repeated failure.
func (c *Consumer) processRecord(ctx context.Context, record *kgo.Record) {
    c.handlersMu.RLock()
    handler, exists := c.handlers[record.Topic]
    c.handlersMu.RUnlock()

    if !exists {
        log.Printf("[%s] No handler for topic %s — committing anyway",
            c.cfg.ClientID, record.Topic)
        c.client.MarkCommitRecords(record)
        return
    }

    log.Printf("[%s] 📨 topic=%s key=%s partition=%d offset=%d",
        c.cfg.ClientID, record.Topic, string(record.Key),
        record.Partition, record.Offset)

    err := handler(ctx, record)
    if err == nil {
        // Success. Clear any retry history for this record and
        // mark its offset ready for commit.
        c.retries.clear(record)
        c.client.MarkCommitRecords(record)
        log.Printf("[%s] ✅ topic=%s partition=%d offset=%d",
            c.cfg.ClientID, record.Topic, record.Partition, record.Offset)
        return
    }

    // Handler failed. Track the attempt count for this exact record.
    attempts := c.retries.increment(record)
    log.Printf("[%s] ❌ attempt=%d/%d topic=%s key=%s offset=%d err=%v",
        c.cfg.ClientID, attempts, c.cfg.MaxRetries,
        record.Topic, string(record.Key), record.Offset, err)

    if attempts < c.cfg.MaxRetries {
        // Not yet exhausted retries. Do NOT mark for commit —
        // re-enqueue for another attempt. In a real system you would
        // add a short delay here (exponential backoff) rather than
        // retrying immediately.
        time.Sleep(backoffDuration(attempts))
        select {
        case c.workQueue <- record:
        case <-ctx.Done():
        }
        return
    }

    // Retries exhausted. This is a poison message — escalate to
    // the dead-letter queue (if configured) so it does not block
    // this partition forever, then mark it committed so processing
    // moves forward past it.
    if c.cfg.DLQTopic != "" && c.dlqProducer != nil {
        c.sendToDLQ(ctx, record, err)
    } else {
        log.Printf("[%s] 🪦 No DLQ configured — DROPPING after %d "+
            "failed attempts: topic=%s offset=%d",
            c.cfg.ClientID, attempts, record.Topic, record.Offset)
    }

    c.retries.clear(record)
    c.client.MarkCommitRecords(record)
}

// dlqEnvelope wraps a failed record with failure metadata for
// debugging — the original topic, the error, and the attempt count.
type dlqEnvelope struct {
    OriginalTopic     string          `json:"original_topic"`
    OriginalPartition int32           `json:"original_partition"`
    OriginalOffset    int64           `json:"original_offset"`
    Key               string          `json:"key"`
    Value             json.RawMessage `json:"value"`
    Error             string          `json:"error"`
    FailedAt          time.Time       `json:"failed_at"`
}

func (c *Consumer) sendToDLQ(ctx context.Context, record *kgo.Record,
    cause error) {
    envelope := dlqEnvelope{
        OriginalTopic:     record.Topic,
        OriginalPartition: record.Partition,
        OriginalOffset:    record.Offset,
        Key:               string(record.Key),
        Value:             record.Value,
        Error:             cause.Error(),
        FailedAt:          time.Now(),
    }
    data, err := json.Marshal(envelope)
    if err != nil {
        log.Printf("[%s] Failed to marshal DLQ envelope: %v",
            c.cfg.ClientID, err)
        return
    }

    dlqRecord := &kgo.Record{
        Topic: c.cfg.DLQTopic,
        Key:   record.Key, // preserve original key for partition affinity
        Value: data,
    }

    // Synchronous produce to the DLQ — we want to KNOW it succeeded
    // before giving up on the original message. If the DLQ publish
    // itself fails, we do not commit the original offset, forcing
    // a retry on restart rather than silently losing the message.
    results := c.dlqProducer.ProduceSync(ctx, dlqRecord)
    if err := results.FirstErr(); err != nil {
        log.Printf("[%s] ⚠️  DLQ publish FAILED for topic=%s offset=%d: %v",
            c.cfg.ClientID, record.Topic, record.Offset, err)
        return
    }
    log.Printf("[%s] 🪦 Sent to DLQ: original_topic=%s offset=%d",
        c.cfg.ClientID, record.Topic, record.Offset)
}

func backoffDuration(attempt int) time.Duration {
    d := 200 * time.Millisecond * time.Duration(1<<attempt)
    if d > 5*time.Second {
        d = 5 * time.Second
    }
    return d
}

// Close performs a graceful shutdown:
//   1. The poll loop sees the cancelled context and stops.
//   2. The work queue is closed and drained — workers finish
//      in-flight records.
//   3. OnPartitionsRevoked fires automatically as part of client.Close(),
//      committing all successfully-marked offsets.
//   4. A LeaveGroup request is sent, triggering an immediate
//      rebalance for the rest of the group instead of making them
//      wait out the full session.timeout.ms.
func (c *Consumer) Close() {
    c.client.Close()
    if c.dlqProducer != nil {
        c.dlqProducer.Close()
    }
    log.Printf("[%s] Consumer closed gracefully", c.cfg.ClientID)
}

// MustUnmarshal is a convenience helper for handlers.
func MustUnmarshal(record *kgo.Record, target interface{}) error {
    if err := json.Unmarshal(record.Value, target); err != nil {
        return fmt.Errorf("unmarshal failed for offset %d: %w",
            record.Offset, err)
    }
    return nil
}

// ErrPermanent can be wrapped by handlers to signal "do not retry,
// send straight to DLQ" for errors that are definitely not transient
// (e.g. malformed JSON that will never successfully parse).
var ErrPermanent = errors.New("permanent failure, do not retry")
```
