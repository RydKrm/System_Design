# Kafka Data Flow — The Complete Theory

### How a Message Travels from One Service to the Entire World

---

## Preface — Why Data Flow Is the Heart of Kafka

Most developers, when they first encounter Kafka, focus on the API — how do I create a producer, how do I write a consumer, what configuration do I need. This is understandable, but it puts the cart before the horse. The API is trivial to learn once you truly understand _what is actually happening_ when a message travels through the system. Until you understand the flow, you are writing code whose behaviour in failure scenarios, scaling scenarios, and ordering scenarios will surprise you. Once you understand the flow deeply, Kafka becomes entirely predictable — and predictability is the foundation of a reliable distributed system.

This guide takes you through every stage of a message's journey: from the moment a service decides to produce it, through the partitioning decision, across the broker's disk, through the replication machinery, and finally into the hands of one or many consumers — in every configuration that large-scale systems actually use in production.

---

## Chapter 1 — The Anatomy of a Kafka Message

Before following a message on its journey, you need to understand what a message actually is. In Kafka, a message (also called a **record**) is not just a blob of data. It is a structured envelope containing several distinct fields, each of which plays a role in how the message is routed, stored, and read.

The **key** is an optional byte array that the producer attaches to the message. The key is not the message's content — it is a routing instruction. When a key is present, Kafka hashes it and uses the result to determine which partition the message goes to. All messages with the same key always land in the same partition, which guarantees that they are stored in the order they were written and will be read in that same order. When no key is provided, Kafka distributes messages across partitions in a round-robin fashion, balancing load at the expense of ordering.

The **value** is the actual payload — the event data. This is the JSON, Avro, or Protobuf-encoded body of your event: the order details, the user registration, the payment confirmation. The value can be up to 1 MB by default (configurable up to a practical ceiling of a few MB — Kafka is not designed for very large messages).

The **timestamp** records when the message was produced, either set by the producer (the default) or overridden by the broker when the message arrives. The **headers** are an optional collection of key-value metadata pairs — useful for tracing IDs, schema version numbers, source service names, and similar envelope metadata that consumers need without having to deserialise the full value. The **offset** is assigned by the broker after the message is written and identifies the message's position within its partition permanently.

```
  Anatomy of a Single Kafka Message (Record)

  ┌─────────────────────────────────────────────────────────────────┐
  │                      KAFKA MESSAGE                              │
  │                                                                 │
  │  ┌───────────────┬─────────────────────────────────────────┐   │
  │  │  OFFSET       │  42                                     │   │
  │  │               │  (assigned by broker, immutable forever)│   │
  │  ├───────────────┼─────────────────────────────────────────┤   │
  │  │  TIMESTAMP    │  2024-11-15T09:23:41.882Z               │   │
  │  │               │  (set by producer or broker)            │   │
  │  ├───────────────┼─────────────────────────────────────────┤   │
  │  │  KEY          │  "order-8821"                           │   │
  │  │               │  (routing key → determines partition)   │   │
  │  ├───────────────┼─────────────────────────────────────────┤   │
  │  │  VALUE        │  {"order_id":"8821","user_id":"u-99",   │   │
  │  │               │   "items":[...],"total":129.99}         │   │
  │  │               │  (the actual event payload)             │   │
  │  ├───────────────┼─────────────────────────────────────────┤   │
  │  │  HEADERS      │  source-service: "order-service"        │   │
  │  │               │  trace-id: "abc-123-xyz"                │   │
  │  │               │  schema-version: "2"                    │   │
  │  └───────────────┴─────────────────────────────────────────┘   │
  └─────────────────────────────────────────────────────────────────┘
```

Every decision Kafka makes — which partition to write to, whether to allow a read, how to sequence messages for a consumer — is based on these fields. Understanding what each field does is understanding how Kafka thinks.

## Chapter 2 — Stage One: The Producer's Decision

The journey of every message begins not in Kafka, but in the producer — the service that decides something worth recording has happened. Before a single byte reaches a broker, the producer makes a sequence of decisions that permanently shape how the message will behave in the system.

The first decision is **serialisation**: converting the Go struct, Python object, or Java POJO into bytes. Kafka is byte-agnostic — it does not care whether the value is JSON, Avro, Protobuf, or raw binary. But the choice matters enormously for consumers: they must know the format to deserialise it. In production, the Schema Registry enforces this contract so both sides agree without ambiguity.

The second decision is **partitioning**: determining which partition the message will be written to. This is where the message key earns its role. The default partitioner computes `hash(key) % number_of_partitions` to arrive at a partition number. This is a deterministic function — the same key will always produce the same partition number for a given topic, as long as the partition count does not change. This determinism is what gives you the ordering guarantee: if the Order Service always uses `order-8821` as the key for every event related to order 8821, then all events for that order — placed, confirmed, shipped, delivered — will live in the same partition, in the exact sequence they were written.

```
  The Producer's Partitioning Decision

  Message key: "order-8821"
  Topic "order.events" has 6 partitions

  Step 1: Hash the key
  ──────────────────────────────────────
  hash("order-8821") = 2,847,392,019

  Step 2: Modulo by partition count
  ──────────────────────────────────────
  2,847,392,019 % 6 = 3

  Step 3: Write to Partition 3
  ──────────────────────────────────────

  ┌──────────────┐
  │ order-service│   All events for order-8821:
  │  (Producer)  │
  │              │   "order-8821 PLACED"   ──▶ Partition 3, offset 0
  │  key:        │   "order-8821 CONFIRMED"──▶ Partition 3, offset 1
  │  "order-8821"│   "order-8821 SHIPPED"  ──▶ Partition 3, offset 2
  └──────────────┘   "order-8821 DELIVERED"──▶ Partition 3, offset 3

  A consumer reading Partition 3 sees these in exact order.
  No other order's events can appear between them in that partition.


  What happens WITHOUT a key (no ordering guarantee):

  "order-8821 PLACED"   ──▶ Partition 0  (round-robin)
  "order-8821 CONFIRMED"──▶ Partition 1  (round-robin)
  "order-8821 SHIPPED"  ──▶ Partition 2  (round-robin)

  A consumer of any single partition sees only ONE of these events.
  No consumer sees them in sequence. Ordering is lost entirely.
```

The third producer decision is **batching**. Kafka producers do not send one message at a time — that would be catastrophically inefficient. Instead, they accumulate messages in an internal buffer and send them in batches. Two configuration properties govern this: `linger.ms` (how long the producer waits before sending a batch, even if the batch is not full — giving more messages time to accumulate) and `batch.size` (the maximum size of a batch in bytes). In high-throughput production systems, tuning these two values is one of the most effective ways to increase overall throughput at the cost of a small, predictable increase in latency. A `linger.ms` of 5 means the producer waits up to 5 milliseconds for more messages before sending — often enough time to accumulate hundreds of messages into a single network round trip.

```
  Producer Batching: How Messages Are Grouped Before Sending

  Without batching (naïve, low throughput):
  ┌─────────┐   msg1   ┌──────┐
  │Producer │─────────▶│Kafka │   1 network round trip per message
  │         │   msg2   │      │   Latency: low  Throughput: LOW
  │         │─────────▶│      │
  │         │   msg3   │      │
  │         │─────────▶│      │
  └─────────┘          └──────┘

  With batching (production default):
  ┌─────────┐                    ┌──────┐
  │Producer │                    │Kafka │
  │ buffer: │                    │      │
  │  msg1   │                    │      │
  │  msg2   │  [msg1+msg2+msg3]  │      │
  │  msg3   │───────────────────▶│      │   1 network round trip
  │         │  (linger: 5ms)     │      │   for 3 messages
  └─────────┘                    └──────┘   Throughput: HIGH

  Production values:
  linger.ms  = 5–20ms  (wait up to this long for more messages)
  batch.size = 65536   (64 KB max batch per partition)
  compression.type = snappy  (compress batches before sending)
```

The fourth decision is **compression**. Producers can compress batches before sending them to the broker. Kafka supports GZIP, Snappy, LZ4, and ZSTD. Compression reduces network bandwidth and disk usage at the cost of CPU on the producer and consumer. For text-heavy payloads like JSON, Snappy or LZ4 typically achieve 3–5x compression ratios with negligible CPU overhead. ZSTD achieves slightly better ratios with more CPU. GZIP achieves the best ratios but is slowest. In most production systems, LZ4 or Snappy is the right choice.

## Chapter 3 — Stage Two: The Broker Receives and Stores

When the producer's batch arrives at the broker, it does something that surprises many developers: it writes the messages to a file on disk almost immediately. There is no complex in-memory processing, no indexing, no query planning. The broker appends the bytes to the end of a **segment file** — a regular file on the Linux filesystem — and that is essentially it. This sequential append-only write is what makes Kafka so extraordinarily fast, even on spinning disks.

Each partition is stored on disk as a directory containing a set of **segment files**. The active segment is the file currently being written to. When it reaches the configured size (`log.segment.bytes`, default 1 GB), it is closed and a new active segment is opened. Closed segments become eligible for retention cleanup — if a message's timestamp is older than `log.retention.hours`, the segment containing it will eventually be deleted.

```
  How a Partition Lives on Disk

  Topic: "order.events", Partition 3
  Broker's disk directory: /var/kafka-data/order.events-3/

  ┌─────────────────────────────────────────────────────────────────┐
  │  /var/kafka-data/order.events-3/                                │
  │                                                                 │
  │  00000000000000000000.log   ← segment 1 (offsets 0–999,999)     │
  │  00000000000000000000.index ← offset index for fast seeks       │
  │  00000000000001000000.log   ← segment 2 (offsets 1M–1,999,999)  │
  │  00000000000001000000.index                                     │
  │  00000000000002000000.log   ← ACTIVE segment (being written to) │
  │  00000000000002000000.index                                     │
  └─────────────────────────────────────────────────────────────────┘

  When a consumer asks for offset 1,500,042:
  1. Kafka looks at segment filenames to find segment 2 (starts at 1M)
  2. Uses the .index file to jump to the approximate position in the .log
  3. Scans forward to offset 1,500,042
  4. Returns the message

  This seek takes microseconds even for terabytes of data.

  Retention cleanup (happens in background):
  If retention = 7 days and segment 1 is older than 7 days:
  → delete 00000000000000000000.log and its .index
  The messages are gone. Consumers that have not read them yet miss them.
```

The broker also maintains a crucial data structure called the **ISR list** — the In-Sync Replica list. For each partition, the leader broker tracks which follower brokers are sufficiently caught up to be considered "in sync." A follower falls out of the ISR if it falls more than `replica.lag.time.max.ms` (default 30 seconds) behind the leader. When the producer sends a message with `acks=all`, the leader waits until every member of the ISR has written the message before sending the acknowledgement. This is the mechanism that makes Kafka's durability guarantee precise and real, rather than aspirational.

## Chapter 4 — Stage Three: Replication, the Silent Safety Net

Replication happens in the background, invisibly to both producers and consumers, and yet it is the most important thing Kafka does. Without replication, a single disk failure would destroy your event history. With replication factor 3, three independent copies of every message exist simultaneously on three different machines. The cluster can lose any one of them — to hardware failure, power loss, or network partition — and continue operating without losing a single message.

The replication protocol is beautifully simple. The leader for a partition receives all writes. After writing each batch to its own log, it makes that batch available for followers to fetch. Followers behave exactly like consumers — they maintain a fetch connection to the leader and pull batches as fast as the network allows. The leader tracks the **High Watermark**: the offset up to which all ISR members have confirmed receipt. Only messages at or below the High Watermark are visible to consumers. This prevents a consumer from reading a message that has not yet been replicated — ensuring that if the leader dies and a follower takes over, the consumer's view of the log remains consistent.

```
  Replication Flow: Leader → Followers → High Watermark

  Partition 3 of "order.events", Replication Factor = 3

  BROKER 1 (LEADER for P3)          BROKER 2 (FOLLOWER)       BROKER 3 (FOLLOWER)
  ┌─────────────────────────┐        ┌───────────────────┐      ┌───────────────────┐
  │ offset 0: order-placed  │        │ offset 0: ✅copied│      │offset 0: ✅ copied│
  │ offset 1: order-confirm │        │ offset 1: ✅copied│      │offset 1: ✅ copied│
  │ offset 2: order-shipped │        │ offset 2: ✅copied│      │offset 2: fetching │
  │ offset 3: order-deliver │◀──────▶│                   │      │                   │
  │           ← just written│  fetch │                   │      │                   │
  └─────────────────────────┘        └───────────────────┘      └───────────────────┘

  ISR = [Broker1, Broker2]  (Broker3 is lagging, not yet in ISR)

  HIGH WATERMARK = offset 2
  (offset 3 is written but NOT yet replicated to all ISR members)

  Consumer asks: "Give me messages from offset 0"
  Kafka returns: offsets 0, 1, 2  ← up to and including High Watermark
  Kafka withholds: offset 3       ← not yet safe (not replicated)

  Two seconds later, Broker3 catches up:
  ISR = [Broker1, Broker2, Broker3]
  HIGH WATERMARK = offset 3  ← advances
  Consumer now receives: offset 3 ✅


  If Broker1 (leader) dies NOW:
  ──────────────────────────────────────────────────────────
  Kafka controller elects Broker2 as new leader.
  Broker2 has offsets 0, 1, 2 — all confirmed.
  offset 3 is SAFE because High Watermark confirmed it.
  NOT A SINGLE MESSAGE IS LOST. ✅
```

This interplay between the ISR, the High Watermark, and the `min.insync.replicas` setting is the engine of Kafka's durability. The `min.insync.replicas=2` setting says: "the ISR must contain at least 2 members for the leader to accept writes." If one follower falls out of sync and the ISR shrinks to just the leader, Kafka starts refusing new writes with a `NotEnoughReplicas` error — preferring unavailability to the risk of data loss. This is a deliberate design choice: Kafka chooses consistency and durability over availability in the face of replication failures.

## Chapter 5 — Stage Four: The Consumer Reads

With the message safely written and replicated, the consumer side of the story begins. Kafka's consumption model is the opposite of what you might expect if you are coming from traditional message queues. In RabbitMQ, the broker pushes messages to consumers. In Kafka, **consumers pull messages from the broker**. This is a foundational design decision with profound consequences.

The pull model means that a consumer controls its own pace entirely. It asks the broker: "give me up to 500 messages from partition 3 starting at offset 42." The broker responds with whatever is available, up to the requested limit. The consumer processes those messages — however long that takes — and then makes another request. There is no way for the broker to overwhelm a slow consumer, because the consumer decides when and how much to ask for. A consumer that takes 10 seconds to process each batch simply makes fewer requests per minute. It falls further behind the producers (its consumer lag grows), but it does not fail, does not drop messages, and does not need back-pressure signalling from the broker.

```
  The Consumer Pull Loop — What Actually Happens

  ┌──────────────────────────────────────────────────────────────────┐
  │  Consumer Process (e.g., inventory-service instance 1)           │
  │                                                                  │
  │  Loop forever:                                                   │
  │                                                                  │
  │  1. POLL  ─────────────────────────────────────────────────────▶ │
  │           "Give me messages from P0, P1 starting at my offsets"  │
  │                                                                  │
  │  2. RECEIVE ◀───────────────────────────────────────────────────│
  │           Broker returns: [msg@P0:off44, msg@P0:off45,           │
  │                            msg@P1:off12, msg@P1:off13]          │
  │                                                                  │
  │  3. PROCESS (your business logic runs here)                      │
  │           handleOrderPlaced(msg@P0:off44) → checks stock → ok   │
  │           handleOrderPlaced(msg@P0:off45) → checks stock → ok   │
  │           handleOrderPlaced(msg@P1:off12) → checks stock → fail  │
  │           handleOrderPlaced(msg@P1:off13) → checks stock → ok   │
  │                                                                  │
  │  4. COMMIT OFFSETS                                               │
  │           "I have successfully processed up to P0:off45"         │
  │           "I have successfully processed up to P1:off13"         │
  │           → Broker stores this in __consumer_offsets topic       │
  │                                                                  │
  │  5. REPEAT from step 1                                           │
  └──────────────────────────────────────────────────────────────────┘

  If the consumer crashes after step 3 but before step 4:
  → On restart, Kafka returns messages from the last COMMITTED offset
  → P0:off44 and P0:off45 are redelivered
  → Your handler must be idempotent (safe to run twice)
```

The offset commit is the consumer's bookmark. Kafka stores committed offsets in an internal topic called `__consumer_offsets`. This topic is itself replicated across the cluster, so the offsets survive broker failures. When a consumer restarts after a crash, it reads its last committed offset from this topic and resumes from exactly that position.

## Chapter 6 — The Five Fundamental Data Flow Patterns

Now that the machinery is understood, it is time to examine the different shapes that data flow can take in a real system. There are five fundamental patterns, and a large-scale distributed system will use all of them simultaneously for different topics and different use cases.

---

### Pattern 1 — One Producer, One Consumer (Point-to-Point)

This is the simplest possible flow. A single service produces messages to a topic, and a single other service consumes them. There is no broadcast, no fan-out — just a direct pipeline between two services, mediated by Kafka for durability and decoupling.

```
  Pattern 1: One Producer → One Consumer

  ┌───────────────┐     Topic: "payment.initiated"      ┌───────────────────┐
  │ Order Service │     ┌──────────────────────────┐    │  Payment Service  │
  │  (Producer)   │────▶│  P0: [m0][m1][m2][m3]    │───▶│  (Consumer)       │
  │               │     │  P1: [m0][m1][m2]        │    │                   │
  │               │     │  P2: [m0][m1][m2][m3][m4]│    │  group-id:        │
  └───────────────┘     └──────────────────────────┘    │  "payment-svc"    │
                                                        └───────────────────┘

  Use case:  Order Service tells Payment Service to charge the customer.
             Only the Payment Service should handle this — it is a command,
             not a broadcast announcement.

  Characteristics:
  - Simple and easy to reason about
  - The topic still provides durability and decoupling
  - Payment Service can be down and messages accumulate safely
  - Works well for command-style events ("do this specific thing")
```

Even in point-to-point, using Kafka instead of a direct HTTP call gives you resilience. The Payment Service can be redeployed, scaled, or temporarily unavailable without the Order Service caring — the messages wait patiently in the topic until the Payment Service comes back online and processes them in order.

---

### Pattern 2 — One Producer, Many Consumers (Fan-Out / Broadcast)

This pattern is where Kafka's design truly diverges from traditional message queues and becomes something categorically more powerful. A single producer publishes one message, and multiple completely independent services each receive their own copy and react in their own way.

The mechanism that makes this possible is the **consumer group**. Each subscribing service uses its own `group.id`. Kafka treats each group independently — it maintains a separate offset for each group per partition. When a message arrives in the topic, Kafka effectively makes a copy for each consumer group. Group A reads it, Group B reads it, Group C reads it — all independently, at their own pace, without any awareness of each other.

```
  Pattern 2: One Producer → Many Independent Consumers (Fan-Out)

  ┌──────────────┐
  │ User Service │
  │  (Producer)  │
  │              │
  │ Publishes:   │
  │ "user.       │
  │  registered" │
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────────┐
  │   Topic: "user.registered"               │
  │   (3 partitions, RF=3)                   │
  │                                          │
  │  P0: [user-A][user-D][user-G]...         │
  │  P1: [user-B][user-E][user-H]...         │
  │  P2: [user-C][user-F][user-I]...         │
  └──────────┬───────────────────────────────┘
             │
             │  Kafka delivers INDEPENDENTLY to each group
             │
     ┌───────┴────────────────────────────────────────────┐
     │                │                │                  │
     ▼                ▼                ▼                  ▼
  ┌──────────┐  ┌──────────┐  ┌──────────────┐  ┌──────────────┐
  │ Notif.   │  │ Analytics│  │  CRM Service │  │  Risk Service│
  │ Service  │  │ Service  │  │              │  │              │
  │          │  │          │  │              │  │              │
  │ group:   │  │ group:   │  │ group:       │  │ group:       │
  │ "notif"  │  │"analytics│  │ "crm-svc"    │  │ "risk-svc"   │
  │          │  │  -svc"   │  │              │  │              │
  │ Sends    │  │ Updates  │  │ Creates      │  │ Scores the   │
  │ welcome  │  │ signup   │  │ customer     │  │ new user for │
  │ email    │  │ dashboard│  │ record       │  │ fraud risk   │
  └──────────┘  └──────────┘  └──────────────┘  └──────────────┘

  ONE message published.
  FOUR independent reactions.
  User Service has no idea any of these services exist.
  Adding a fifth consumer requires ZERO changes to any existing service.
```

This is the Open/Closed principle of distributed systems: the producing service is closed for modification but open for extension. When a new business requirement arrives — "we need to score new users for fraud risk" — a new consumer group is added to the existing topic. The User Service is not touched. The existing consumers are not touched. The new service simply subscribes and starts receiving all messages from the beginning of the retention window.

### Pattern 3 — Many Producers, One Topic, Many Consumers

In a real system, multiple services often produce to the same topic. The `notification.send` topic from the previous chapter is a good example: the Order Service, the Shipping Service, the User Service, and the Marketing Service all produce notification requests to the same topic, and a single Notification Service consumes them all.

```
  Pattern 3: Many Producers → One Topic → One Consumer Group

  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ Order Service│  │Shipping Svc  │  │ User Service │  │Marketing Svc │
  │  (Producer)  │  │  (Producer)  │  │  (Producer)  │  │  (Producer)  │
  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
         │                 │                  │                  │
         │  "Order         │  "Shipment       │  "Welcome        │  "Promo
         │   confirmed"    │   dispatched"    │   email"         │   offer"
         │                 │                  │                  │
         └─────────────────┴──────────────────┴──────────────────┘
                                    │
                                    ▼
         ┌──────────────────────────────────────────────────────┐
         │           Topic: "notification.send"                 │
         │                                                      │
         │  P0: [order-confirm][promo-offer][shipment-dispatch] │
         │  P1: [welcome-email][order-confirm][promo-offer]     │
         │  P2: [shipment-dispatch][welcome-email]              │
         └──────────────────────────┬───────────────────────────┘
                                    │
                                    ▼
                         ┌─────────────────────┐
                         │ Notification Service│
                         │ Consumer Group:     │
                         │ "notification-svc"  │
                         │                     │
                         │  Instance 1 → P0    │
                         │  Instance 2 → P1    │
                         │  Instance 3 → P2    │
                         └─────────────────────┘

  Key insight: The Notification Service does not care WHO produced
  each message. It only cares about WHAT to send. Multiple producers
  can write to the same topic freely, as long as they agree on the
  message schema. The topic is the universal inbox.
```

This pattern centralises cross-cutting concerns. Notifications, audit logging, metrics collection — all of these are cross-cutting concerns that every service needs but no service should implement independently. By routing them through a shared topic, you get a single implementation, a single point of monitoring, and a single place to enforce rate limiting, templating, or compliance requirements.

### Pattern 4 — The Chain (Event-Driven Saga)

This is the most architecturally interesting pattern and the one that underpins the microservices system built in the previous chapter. Instead of a single hop from producer to consumer, messages trigger a chain of reactions that flow through multiple services in sequence, each service publishing a new event that the next service consumes.

```
  Pattern 4: The Chain — An Order Fulfilment Saga

  Step 1                Step 2                Step 3               Step 4
  ──────                ──────                ──────               ──────

  Order Svc             Inventory Svc         Order Svc            Notif. Svc
  PRODUCES              CONSUMES              CONSUMES             CONSUMES
  ─────────             ────────────          ────────────         ────────────
  "order.placed"  ───▶  "order.placed"        "inv.reserved" ───▶  "notif.send"
                        checks stock
                        PRODUCES         ─▶   "inv.reserved"
                        "inv.reserved"        PRODUCES
                                              "order.confirmed"
                                              PRODUCES         ─▶  "notif.send"
                                              "notif.send"

  Timeline view:

  t=0ms   Order Service: POST /orders received
          → saves order PENDING to DB
          → PRODUCES to "order.placed"   ─────────────────────────────────▶
                                                                          t=12ms
  t=12ms  Inventory Service: polls "order.placed"
          → checks stock (all available)
          → deducts stock
          → PRODUCES to "inventory.reserved" ───────────────────────────────▶
                                                                          t=25ms
  t=25ms  Order Service: polls "inventory.reserved"
          → updates order status to CONFIRMED
          → PRODUCES to "order.confirmed"    ─────────────────────────────▶
          → PRODUCES to "notification.send"  ─────────────────────────────▶
                                                                          t=38ms
  t=38ms  Notification Service: polls "notification.send"
          → sends confirmation email to customer  ✅

  Total wall-clock time: ~38ms for the entire saga.
  HTTP request returned at t=0ms with status 202 Accepted.
  Customer saw the response before the saga even completed.
```

The chain pattern is the implementation of what is called a **choreography-based saga**. No single service orchestrates the workflow — instead, each service knows only about its own inputs (what topics it listens to) and outputs (what topics it publishes to). The overall workflow emerges from the composition of individual reactions. This is extremely loosely coupled: you can replace the Inventory Service entirely without touching the Order Service, as long as the new implementation respects the same input and output topic contracts.

The alternative — **orchestration-based saga** — uses a central Saga Orchestrator service that explicitly calls each step in sequence and handles failures. Orchestration is easier to reason about but creates a central point of coordination. Both approaches are valid; large systems often use both, choosing based on the complexity of the workflow and the team's preference.

### Pattern 5 — The Scatter/Gather (Request-Reply over Kafka)

This is the most advanced pattern and one that is often misunderstood. Kafka is fundamentally a broadcast system — producers do not expect replies. But sometimes a service genuinely needs a response: "I asked the Fraud Service to evaluate this transaction, and I need its verdict before I can proceed." The Scatter/Gather pattern implements request-reply semantics over Kafka using a **reply topic** and a **correlation ID**.

```
  Pattern 5: Scatter/Gather — Request-Reply over Kafka

  ┌───────────────────────────────────────────────────────────────────┐
  │  SCATTER PHASE: Order Service asks multiple services for input    │
  └───────────────────────────────────────────────────────────────────┘

  Order Service (Producer):
  → Generates correlationId = "corr-7821"
  → Publishes to "fraud.check.request":
    { correlationId: "corr-7821",
      replyTopic: "fraud.check.reply",
      orderId: "ord-8821",
      userId: "u-99",
      amount: 1299.99 }

  → Publishes to "inventory.check.request":
    { correlationId: "corr-7821",
      replyTopic: "inventory.check.reply",
      items: [...] }

               ┌─────────────────────────────────────────────────┐
               │  Topic: "fraud.check.request"                   │
               └─────────────────────┬───────────────────────────┘
                                     │
                                     ▼
               Fraud Service consumes, evaluates, PRODUCES to:
               ┌─────────────────────────────────────────────────┐
               │  Topic: "fraud.check.reply"                     │
               │  { correlationId: "corr-7821", result: "PASS" } │
               └─────────────────────────────────────────────────┘

               ┌─────────────────────────────────────────────────┐
               │  Topic: "inventory.check.request"               │
               └─────────────────────┬───────────────────────────┘
                                     │
                                     ▼
               Inventory Service consumes, evaluates, PRODUCES to:
               ┌─────────────────────────────────────────────────┐
               │  Topic: "inventory.check.reply"                 │
               │  { correlationId: "corr-7821", result: "OK" }   │
               └─────────────────────────────────────────────────┘

  ┌───────────────────────────────────────────────────────────────────┐
  │  GATHER PHASE: Order Service collects replies                     │
  └───────────────────────────────────────────────────────────────────┘

  Order Service (Consumer of reply topics):
  → Reads fraud.check.reply   → finds corr-7821 → PASS ✅
  → Reads inventory.check.reply → finds corr-7821 → OK  ✅
  → Both replies received → proceed to place order

  Correlation table in Order Service memory:
  ┌──────────────┬───────────────────────┬──────────────────────┐
  │ CorrelationId│ fraud result          │ inventory result     │
  ├──────────────┼───────────────────────┼──────────────────────┤
  │ corr-7821    │ PASS ✅               │ OK ✅  → proceed     │
  │ corr-7899    │ FAIL ❌               │ -      → reject      │
  │ corr-7901    │ PASS ✅               │ waiting...           │
  └──────────────┴───────────────────────┴──────────────────────┘
```

Scatter/Gather over Kafka is more complex than the previous patterns but achieves something remarkable: it parallelises multiple checks or calls that would traditionally be sequential HTTP calls. Instead of calling Fraud, waiting for the response, then calling Inventory, waiting for that response — both requests are fired simultaneously and the results are gathered as they arrive. The total latency is bounded by the slowest responder, not the sum of all responders.

## Chapter 7 — Consumer Group Internals: The Full Picture

Consumer groups are the mechanism by which Kafka scales consumption horizontally, and their internal behaviour is sophisticated enough to warrant a dedicated chapter. Understanding exactly how groups work, how partitions are assigned, and what happens during a rebalance is essential for building systems that scale and recover gracefully.

When a consumer instance joins a group, it sends a `JoinGroup` request to the **Group Coordinator** — a specific broker that is responsible for managing that group (determined by `hash(group.id) % number_of_brokers`). The coordinator collects all the instances that want to join and designates one of them as the **Group Leader**. The Group Leader — not the broker — is responsible for computing the partition assignment. It receives the list of all members and all available partitions, runs a partition assignment algorithm, and sends the result back to the coordinator. The coordinator then distributes these assignments to each member in a `SyncGroup` response.

```
  Consumer Group Lifecycle: Join → Assign → Consume → Rebalance

  ═══════════════════════════════════════════════════════════════
  PHASE 1: INITIAL JOIN (3 instances starting up)
  ═══════════════════════════════════════════════════════════════

  Topic "order.placed": 6 partitions (P0–P5)
  Group: "inventory-service"

  Instance 1 starts  → sends JoinGroup → waits
  Instance 2 starts  → sends JoinGroup → waits
  Instance 3 starts  → sends JoinGroup → waits

  Coordinator designates Instance 1 as Group Leader.
  Instance 1 computes assignment:
    Instance 1 → P0, P1
    Instance 2 → P2, P3
    Instance 3 → P4, P5

  Coordinator distributes assignments.

  ┌─────────────────────────────────────────────────────────────┐
  │  Group: inventory-service                                   │
  │                                                             │
  │  ┌────────────┐    ┌────────────┐    ┌────────────┐         │
  │  │ Instance 1 │    │ Instance 2 │    │ Instance 3 │         │
  │  │  P0, P1    │    │  P2, P3    │    │  P4, P5    │         │
  │  └────────────┘    └────────────┘    └────────────┘         │
  │  Actively consuming. Consumer lag per partition tracked.    │
  └─────────────────────────────────────────────────────────────┘

  ═══════════════════════════════════════════════════════════════
  PHASE 2: SCALING UP (4th instance joins)
  ═══════════════════════════════════════════════════════════════

  Instance 4 sends JoinGroup to coordinator.
  Coordinator triggers a REBALANCE — all members must rejoin.

  Brief pause: all instances stop consuming during rebalance.
  (This is called a "stop-the-world" rebalance — a known Kafka limitation.)

  New assignment:
    Instance 1 → P0, P1
    Instance 2 → P2, P3
    Instance 3 → P4
    Instance 4 → P5       ← new instance gets one partition

  ┌─────────────────────────────────────────────────────────────┐
  │  ┌────────────┐  ┌────────────┐  ┌────────────┐ ┌────────┐  │
  │  │ Instance 1 │  │ Instance 2 │  │ Instance 3 │ │Inst. 4 │  │
  │  │  P0, P1    │  │  P2, P3    │  │   P4       │ │  P5    │  │
  │  └────────────┘  └────────────┘  └────────────┘ └────────┘  │
  └─────────────────────────────────────────────────────────────┘

  ═══════════════════════════════════════════════════════════════
  PHASE 3: INSTANCE FAILURE (Instance 2 crashes)
  ═══════════════════════════════════════════════════════════════

  Instance 2 stops sending heartbeats to coordinator.
  Coordinator waits session.timeout.ms (default: 45 seconds).
  Coordinator declares Instance 2 dead → triggers rebalance.

  New assignment:
    Instance 1 → P0, P1, P2    ← absorbs P2 from dead Instance 2
    Instance 3 → P3, P4        ← absorbs P3 from dead Instance 2
    Instance 4 → P5

  The dead instance's partitions are redistributed automatically.
  P2 and P3 resume from their last COMMITTED offset. No messages lost.

  ═══════════════════════════════════════════════════════════════
  CEILING: What if you add MORE instances than partitions?
  ═══════════════════════════════════════════════════════════════

  Topic has 6 partitions. Group has 8 instances.

  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐
  │ I1 │ │ I2 │ │ I3 │ │ I4 │ │ I5 │ │ I6 │ │ I7 │ │ I8 │
  │ P0 │ │ P1 │ │ P2 │ │ P3 │ │ P4 │ │ P5 │ │IDLE│ │IDLE│
  └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘

  I7 and I8 receive NO partitions and consume NO messages.
  They sit idle as warm standbys — ready to take over if
  any of I1–I6 fails, without triggering a full cold start.

  Maximum parallelism = number of partitions.
  To increase parallelism, increase partition count FIRST.
```

## Chapter 8 — Consumer Group Strategies: Three Assignment Algorithms

When the Group Leader computes the partition assignment, it uses a pluggable **partition assignment strategy**. Kafka ships with three built-in strategies, and each produces meaningfully different behaviour during rebalances. Choosing the right one has significant performance implications in large systems.

The **RangeAssignor** is the oldest strategy and the default for most Kafka versions. It assigns consecutive ranges of partitions to consumers, sorted alphabetically by topic. It is predictable and deterministic, but it has a flaw: when a consumer group subscribes to multiple topics, the first consumer in the sorted list ends up with a disproportionately large share of partitions. If topics A, B, and C each have 7 partitions and there are 3 consumers, the first consumer gets partition 0 and 1 from each topic — six partitions — while the third consumer might only get one.

The **RoundRobinAssignor** distributes partitions evenly across consumers by cycling through them in order. It produces a more balanced distribution than Range but, like Range, triggers a **full rebalance** whenever any consumer joins or leaves. During a full rebalance, every consumer in the group stops consuming, all assignments are revoked, and new assignments are computed from scratch. In a large consumer group with hundreds of instances, this pause can last tens of seconds.

The **CooperativeStickyAssignor** is the modern choice and what production systems should use. It solves two problems simultaneously. First, it is "sticky" — when a rebalance occurs, it tries to keep as many existing partition assignments unchanged as possible, only moving the minimum number of partitions necessary to achieve balance. Second, it implements **cooperative (incremental) rebalancing** — instead of revoking all assignments and stopping all consumers during a rebalance, it only revokes the specific partitions that need to move, allowing all other consumers to continue consuming throughout the process.

```
  Rebalance Comparison: Full vs. Cooperative

  SCENARIO: 4 consumers, 8 partitions. Consumer 3 leaves.

  ─────────────────────────────────────────────────────────────────
  FULL REBALANCE (RangeAssignor / RoundRobinAssignor)
  ─────────────────────────────────────────────────────────────────

  BEFORE:
  I1: P0, P1     I2: P2, P3     I3: P4, P5     I4: P6, P7
  (all consuming normally)

  I3 leaves → REBALANCE TRIGGERED

  ALL consumers stop consuming immediately. ← performance impact!
  All assignments revoked.
  New assignments computed.
  All consumers restart from committed offsets.

  AFTER:
  I1: P0, P1, P4    I2: P2, P3, P5    I4: P6, P7
  (consuming resumes)

  Downtime during rebalance: 2–30 seconds depending on group size.
  Messages accumulate in partitions during this window.
  Consumer lag spikes during every deployment, every crash.

  ─────────────────────────────────────────────────────────────────
  COOPERATIVE REBALANCE (CooperativeStickyAssignor)
  ─────────────────────────────────────────────────────────────────

  BEFORE:
  I1: P0, P1     I2: P2, P3     I3: P4, P5     I4: P6, P7
  (all consuming normally)

  I3 leaves → REBALANCE TRIGGERED

  Phase 1: Only P4 and P5 are revoked (the partitions that need to move).
           I1, I2, and I4 KEEP consuming their current partitions.
           No stop-the-world pause.

  Phase 2: P4 assigned to I1, P5 assigned to I2.

  AFTER:
  I1: P0, P1, P4    I2: P2, P3, P5    I4: P6, P7
  (consuming was never fully stopped — only 2 partitions paused briefly)

  Downtime during rebalance: near-zero for unaffected partitions.
  The recommended strategy for all production deployments.
```

## Chapter 9 — Offset Management: Every Failure Mode Explained

How a consumer manages its offsets determines exactly what guarantees it gets about message delivery. There are three possible delivery semantics in Kafka, and each corresponds to a specific offset commit strategy.

**At-most-once delivery** occurs when the consumer commits the offset _before_ processing the message. If the consumer crashes after committing but before finishing processing, the message is gone — it will not be redelivered. The offset has already moved past it. This approach gives you no duplicate processing, but at the cost of potential data loss. It is acceptable for use cases where losing an occasional message is tolerable and duplicate processing would be harmful — perhaps logging or analytics where approximation is fine.

**At-least-once delivery** occurs when the consumer commits the offset _after_ processing. If the consumer crashes after processing but before committing, the message will be redelivered on restart. You may process it twice. This is the most common semantics in production and is achieved with `enable.auto.commit=false` and manual commit after successful handling. The application must be **idempotent** — processing the same message twice should produce the same result as processing it once.

**Exactly-once delivery** is the ideal but most complex semantics. It is achieved through Kafka's **transactional API**, which coordinates atomic writes across both the consumer's offset commit and the producer's output messages. Used in conjunction with idempotent producers, it guarantees that each message is processed precisely once with no duplicates and no losses. It requires that both the source and destination of a message are Kafka topics (making it ideal for Kafka Streams applications).

```
  The Three Delivery Semantics — What Actually Happens

  AT-MOST-ONCE (commit before process)
  ──────────────────────────────────────────────────────────────
  msg arrives at offset 42
  → commit offset 43  ← "I will process 42"
  → CRASH HERE        ← message 42 never processed
  → restart → reads from offset 43
  → offset 42 is LOST FOREVER 😢
  Use when: losing messages is acceptable. Rare in practice.

  AT-LEAST-ONCE (commit after process) ← most common in production
  ──────────────────────────────────────────────────────────────
  msg arrives at offset 42
  → process message 42 ← business logic runs ✅
  → CRASH HERE         ← commit never sent
  → restart → reads from last committed offset 41
  → message 42 is REDELIVERED
  → process message 42 again ← runs twice ⚠️
  Your handler must be IDEMPOTENT:
    "reserve inventory for order-8821" → check if already reserved
    "send email for order-8821"        → check if already sent
    "decrement stock for order-8821"   → check if already decremented

  EXACTLY-ONCE (Kafka Transactions) ← complex, for Kafka→Kafka flows
  ──────────────────────────────────────────────────────────────
  BEGIN TRANSACTION
    read message at offset 42 from source topic
    write output to destination topic
    commit offset 42
  COMMIT TRANSACTION  ← all three happen atomically

  If crash before COMMIT: all three are rolled back. No partial state.
  If crash after COMMIT:  all three are committed. No duplicates.
  Requires: transactional.id on producer, Kafka Streams or manual TX API.
```
