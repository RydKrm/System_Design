# React Native Screen Navigation: A Complete Deep Dive

> A thorough, beginner-friendly guide to every major navigation pattern in React Native — Stack, Tab, Drawer, and beyond — explained from first principles with production-grade examples.

---

## Preface — What Is Navigation and Why Does It Matter?

A mobile application is never a single screen. It is a collection of screens stitched together into a journey — a user taps a product card and travels to a product detail page, swipes between tabs to move from a feed to a profile, or pulls out a side drawer to access settings. The system that manages this journey — that knows which screen the user is on, how they got there, and how to get back — is called the **navigation system**.

On the web, navigation is handled by the browser: the URL changes, the browser fetches a new page, and the back button takes you to the previous URL. On mobile, there is no browser. Your application is entirely responsible for managing its own navigation history, transitions, and deep linking. This responsibility is both a burden and an enormous creative opportunity — because you have complete control over how screens appear, disappear, and transform.

In React Native, the ecosystem-standard navigation library is **React Navigation** (`@react-navigation/native`). It is maintained by the Expo team, battle-tested in thousands of production applications, and provides a declarative API for building virtually any navigation structure imaginable. This guide covers every major navigator type — Stack, Native Stack, Tab, Material Top Tab, Drawer — and the advanced concepts that connect them: passing parameters, deep linking, authentication flows, and nested navigators.

# Part I — Foundation: How React Navigation Works

## Chapter 1: The Navigation Architecture

Before writing a single line of code, you need to understand the three-layer architecture that React Navigation is built on. Every navigation system in React Navigation consists of three fundamental concepts: a **Navigator**, **Screens**, and a **Navigation Container**.

The **NavigationContainer** is the root of your entire navigation tree. It holds the navigation state — the complete history of which screens have been visited — and provides it to all child navigators. You must have exactly one `NavigationContainer` at the top of your application, wrapping everything. Think of it as the browser itself — the environment within which all navigation occurs.

A **Navigator** defines a specific navigation pattern. The Stack Navigator manages a stack of screens (last in, first out — like a pile of cards). The Tab Navigator manages a row of tabs at the bottom or top of the screen. The Drawer Navigator manages a slide-out menu. Each navigator type handles its own transitions, gestures, and header styles.

A **Screen** is a registered route within a navigator. It maps a string name (like `"ProductDetail"`) to a React component. When you navigate to `"ProductDetail"`, the navigator renders that component with the navigation props injected.

```
React Navigation Architecture

┌──────────────────────────────────────────────────────────────────────┐
│                      <NavigationContainer>                           │
│              Holds all navigation state (history, params)            │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │                    Root Navigator                              │  │
│  │            (e.g., Tab Navigator or Stack Navigator)            │  │
│  │                                                                │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │  │
│  │  │   Screen A   │  │   Screen B   │  │      Screen C        │  │  │
│  │  │  (Component) │  │  (Component) │  │  (Nested Navigator)  │  │  │
│  │  │              │  │              │  │  ┌────────┬────────┐ │  │  │
│  │  │ receives:    │  │              │  │  │ Sub    │ Sub    │ │  │  │
│  │  │ navigation   │  │              │  │  │Screen1 │Screen2 │ │  │  │
│  │  │ route        │  │              │  │  └────────┴────────┘ │  │  │
│  │  └──────────────┘  └──────────────┘  └──────────────────────┘  │  │
│  └────────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────────┘
```

Every screen component that is registered with a navigator automatically receives two special props: `navigation` and `route`. The `navigation` prop gives you methods to move between screens (`navigate`, `goBack`, `push`, `replace`, `reset`). The `route` prop gives you the current screen's name and any parameters that were passed to it.

```
The navigation and route Props

┌──────────────────────────────────────────────────────────────────┐
│  navigation prop (methods to MOVE between screens)               │
│  ─────────────────────────────────────────────────               │
│  navigation.navigate('ScreenName', { params })                   │
│    → Go to a screen. If already in stack, may go back to it.    │
│                                                                  │
│  navigation.push('ScreenName', { params })                       │
│    → ALWAYS push a new copy onto the stack (even if exists).    │
│                                                                  │
│  navigation.goBack()                                             │
│    → Go to the previous screen (pop from stack).                │
│                                                                  │
│  navigation.replace('ScreenName', { params })                    │
│    → Replace current screen with a new one (no back button).    │
│                                                                  │
│  navigation.reset({ index: 0, routes: [{ name: 'Home' }] })      │
│    → Clear history and set a completely new navigation state.    │
│    → Used after login — you don't want "back" to go to Login.   │
│                                                                  │
│  navigation.setOptions({ title: 'New Title' })                   │
│    → Dynamically change header options from inside the screen.  │
│                                                                  │
│  route prop (info ABOUT the current screen)                      │
│  ──────────────────────────────────────────                      │
│  route.name     → "ProductDetail"                                │
│  route.params   → { productId: 42, fromCart: true }             │
│  route.key      → unique identifier for this screen instance    │
└──────────────────────────────────────────────────────────────────┘
```

# Part II — Stack Navigation

## Chapter 2: What Is a Stack Navigator?

The Stack Navigator is the most fundamental navigator in mobile applications. It works exactly like a physical stack of cards — when you navigate to a new screen, a new card is placed on top of the stack. When you go back, the top card is removed, revealing the one beneath it. On iOS, this produces the familiar right-to-left slide-in transition. On Android, it produces an upward slide or a fade transition.

The stack maintains a complete navigation **history**, which is what powers the back button. Every `navigate()` call adds a new entry to this history. `goBack()` removes the most recent entry. This is simple but powerful — it means you can have deep navigation hierarchies with the back button always taking you to exactly where you came from.

```
Stack Navigator — Visual Model

   navigate('B')         navigate('C')          goBack()
       │                      │                     │
       ▼                      ▼                     ▼
  ┌─────────┐           ┌─────────┐           ┌─────────┐
  │    B    │  ◄─ top   │    C    │ ◄─ top    │    B    │ ◄─ top
  ├─────────┤           ├─────────┤           ├─────────┤
  │    A    │           │    B    │           │    A    │
  └─────────┘           ├─────────┤           └─────────┘
                        │    A    │
                        └─────────┘

  Stack grows UP.         Stack at 3         Popped C, back to B.
  A is always at          screens deep.      B is now on top.
  the bottom.
```

React Navigation offers two stack implementations. The first is `@react-navigation/stack` — the JavaScript-based implementation that gives you maximum customization over transitions and header appearance. The second is `@react-navigation/native-stack` — a wrapper around the platform's native navigation primitives (UINavigationController on iOS, Fragment transactions on Android). The native stack is dramatically more performant and produces transitions that are completely indistinguishable from fully native apps. For new projects, always prefer the native stack unless you need custom transitions that only the JS stack can provide.

## Chapter 3: Setting Up the Stack Navigator — Complete Example

The following example builds a complete e-commerce browsing flow: a product list screen that navigates to a product detail screen, which can navigate further to a checkout screen. This demonstrates the essential patterns of stack navigation in a real-world context.

```javascript
// ── Installation ─────────────────────────────────────────────────────────────
// npm install @react-navigation/native @react-navigation/native-stack
// expo install react-native-screens react-native-safe-area-context

import React, { useLayoutEffect } from 'react';
import {
  View, Text, FlatList, TouchableOpacity,
  StyleSheet, Image, ScrollView, Alert,
} from 'react-native';
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';

const Stack = createNativeStackNavigator();
// createNativeStackNavigator() returns an object with two components:
// Stack.Navigator → the navigator wrapper
// Stack.Screen    → registers a route name → component mapping

// ─────────────────────────────────────────────────────────────────────────────
// SCREEN 1: Product List
// ─────────────────────────────────────────────────────────────────────────────
const PRODUCTS = [
  { id: 1, name: 'Wireless Headphones', price: 129.99, rating: 4.8, emoji: '🎧', category: 'Electronics' },
  { id: 2, name: 'Running Shoes',       price: 89.99,  rating: 4.6, emoji: '👟', category: 'Footwear'    },
  { id: 3, name: 'Coffee Maker',        price: 64.99,  rating: 4.7, emoji: '☕', category: 'Kitchen'     },
  { id: 4, name: 'Backpack 40L',        price: 49.99,  rating: 4.5, emoji: '🎒', category: 'Travel'      },
  { id: 5, name: 'Mechanical Keyboard', price: 159.99, rating: 4.9, emoji: '⌨️', category: 'Electronics' },
];

const ProductListScreen = ({ navigation }) => {
  // useLayoutEffect runs synchronously before the browser paints,
  // making it ideal for setting header options without a flash of old content.
  useLayoutEffect(() => {
    navigation.setOptions({
      title: '🛍 Products',
      headerRight: () => (
        <TouchableOpacity onPress={() => navigation.navigate('Cart')}>
          <Text style={{ fontSize: 22 }}>🛒</Text>
        </TouchableOpacity>
      ),
    });
  }, [navigation]);

  const handleProductPress = (product) => {
    // navigation.navigate() accepts a screen name and an optional params object.
    // Params can be any serializable JavaScript value.
    // The destination screen reads them via route.params.
    navigation.navigate('ProductDetail', {
      productId: product.id,
      productName: product.name,  // Pass name for instant header update
    });
  };

  return (
    <View style={styles.screen}>
      <FlatList
        data={PRODUCTS}
        keyExtractor={(item) => item.id.toString()}
        contentContainerStyle={{ padding: 16 }}
        renderItem={({ item }) => (
          <TouchableOpacity
            style={styles.productCard}
            onPress={() => handleProductPress(item)}
            activeOpacity={0.75}
          >
            <Text style={styles.productEmoji}>{item.emoji}</Text>
            <View style={styles.productInfo}>
              <Text style={styles.productName}>{item.name}</Text>
              <Text style={styles.productCategory}>{item.category}</Text>
              <View style={styles.productMeta}>
                <Text style={styles.productPrice}>${item.price}</Text>
                <Text style={styles.productRating}>⭐ {item.rating}</Text>
              </View>
            </View>
            <Text style={styles.chevron}>›</Text>
          </TouchableOpacity>
        )}
        ItemSeparatorComponent={() => <View style={{ height: 12 }} />}
      />
    </View>
  );
};

// ─────────────────────────────────────────────────────────────────────────────
// SCREEN 2: Product Detail
// ─────────────────────────────────────────────────────────────────────────────
const ProductDetailScreen = ({ navigation, route }) => {
  // route.params contains everything passed from the previous screen.
  // Destructure with a default in case params are undefined (defensive coding).
  const { productId, productName } = route.params ?? {};

  const product = PRODUCTS.find(p => p.id === productId);

  // Dynamically set the header title to the product name BEFORE the screen renders.
  useLayoutEffect(() => {
    navigation.setOptions({
      title: productName ?? 'Product',
      // Custom back button label (iOS only)
      headerBackTitle: 'Back',
    });
  }, [navigation, productName]);

  if (!product) {
    return (
      <View style={styles.centered}>
        <Text style={styles.errorText}>Product not found</Text>
        <TouchableOpacity onPress={() => navigation.goBack()}>
          <Text style={styles.link}>← Go Back</Text>
        </TouchableOpacity>
      </View>
    );
  }

  const handleAddToCart = () => {
    // navigate() can also navigate to sibling or ancestor screens.
    // Here we navigate to 'Checkout' and pass the product.
    navigation.navigate('Checkout', {
      product,
      quantity: 1,
    });
  };

  return (
    <ScrollView style={styles.screen} contentContainerStyle={{ padding: 24 }}>
      <View style={styles.detailHero}>
        <Text style={styles.detailEmoji}>{product.emoji}</Text>
      </View>
      <Text style={styles.detailName}>{product.name}</Text>
      <Text style={styles.detailCategory}>{product.category}</Text>
      <Text style={styles.detailPrice}>${product.price}</Text>
      <Text style={styles.detailRating}>⭐ {product.rating} / 5.0</Text>
      <Text style={styles.detailDescription}>
        This is a premium quality {product.name.toLowerCase()} carefully crafted for
        performance and longevity. Loved by thousands of customers worldwide.
      </Text>
      <TouchableOpacity style={styles.cta} onPress={handleAddToCart}>
        <Text style={styles.ctaText}>Add to Cart & Checkout →</Text>
      </TouchableOpacity>
      <TouchableOpacity
        style={[styles.cta, styles.ctaSecondary]}
        onPress={() => navigation.goBack()}
      >
        <Text style={[styles.ctaText, { color: '#6C63FF' }]}>← Back to Products</Text>
      </TouchableOpacity>
    </ScrollView>
  );
};

// ─────────────────────────────────────────────────────────────────────────────
// SCREEN 3: Checkout
// ─────────────────────────────────────────────────────────────────────────────
const CheckoutScreen = ({ navigation, route }) => {
  const { product, quantity } = route.params ?? {};
  const total = (product?.price * quantity).toFixed(2);

  const handleConfirmOrder = () => {
    Alert.alert('Order Placed! 🎉', `Your ${product.name} will arrive in 3-5 days.`, [
      {
        text: 'Back to Products',
        onPress: () => {
          // navigation.reset() clears the entire navigation stack and sets a
          // fresh history. After ordering, we don't want the user to be able
          // to hit "back" and end up on the checkout screen again.
          navigation.reset({
            index: 0,
            routes: [{ name: 'ProductList' }],
          });
        },
      },
    ]);
  };

  return (
    <View style={[styles.screen, styles.centered]}>
      <Text style={styles.checkoutEmoji}>{product?.emoji}</Text>
      <Text style={styles.checkoutTitle}>Order Summary</Text>
      <Text style={styles.checkoutItem}>{product?.name}</Text>
      <Text style={styles.checkoutQty}>Qty: {quantity}</Text>
      <Text style={styles.checkoutTotal}>Total: ${total}</Text>
      <TouchableOpacity style={styles.cta} onPress={handleConfirmOrder}>
        <Text style={styles.ctaText}>Confirm Order ✓</Text>
      </TouchableOpacity>
    </View>
  );
};

// ─────────────────────────────────────────────────────────────────────────────
// ROOT: The Navigator — wires screen names to components
// ─────────────────────────────────────────────────────────────────────────────
const ShopNavigator = () => (
  <NavigationContainer>
    <Stack.Navigator
      initialRouteName="ProductList"
      screenOptions={{
        // screenOptions applies to ALL screens in this navigator.
        // Individual Screen options prop can override these.
        headerStyle:      { backgroundColor: '#0D0D2B' },
        headerTintColor:  '#FFF',  // Back arrow and title color
        headerTitleStyle: { fontWeight: '700', fontSize: 17 },
        contentStyle:     { backgroundColor: '#080818' },
        animation: 'slide_from_right',  // iOS-style slide for all screens
      }}
    >
      <Stack.Screen
        name="ProductList"
        component={ProductListScreen}
        // options on the Screen level overrides screenOptions for this screen only
        options={{ title: '🛍 Products' }}
      />
      <Stack.Screen
        name="ProductDetail"
        component={ProductDetailScreen}
        // options can be a function that receives { navigation, route }
        // so you can set the title from initial params before the component mounts
        options={({ route }) => ({ title: route.params?.productName ?? 'Detail' })}
      />
      <Stack.Screen
        name="Checkout"
        component={CheckoutScreen}
        options={{
          title: '🛒 Checkout',
          // headerBackVisible: false hides the back button.
          // Use this for screens where going back would be inappropriate.
          headerBackVisible: true,
          // presentation: 'modal' slides the screen up from the bottom (iOS)
          // instead of the default right-to-left slide.
          presentation: 'modal',
        }}
      />
      <Stack.Screen
        name="Cart"
        component={() => (
          <View style={[styles.screen, styles.centered]}>
            <Text style={{ color: '#FFF', fontSize: 24 }}>🛒 Cart is empty</Text>
          </View>
        )}
        options={{ title: 'My Cart' }}
      />
    </Stack.Navigator>
  </NavigationContainer>
);

export default ShopNavigator;
```

## Chapter 4: Stack Transition Animations

One of the most powerful aspects of the JavaScript-based `@react-navigation/stack` (as opposed to native-stack) is the ability to define completely custom transition animations for each screen. You do this by providing a `cardStyleInterpolator` function that maps the animation progress value (0 to 1) to style properties. This is how you create effects like a fade-in, a scale-up, a slide-from-bottom, or any other creative entrance animation.

```javascript
import { CardStyleInterpolators } from '@react-navigation/stack';

// ── Built-in Interpolators ────────────────────────────────────────────────────
// React Navigation ships with several ready-made transition styles:

const transitionExamples = {

  // iOS-style horizontal slide (default on iOS)
  forHorizontalIOS: CardStyleInterpolators.forHorizontalIOS,

  // Android-style fade-through
  forFadeFromBottomAndroid: CardStyleInterpolators.forFadeFromBottomAndroid,

  // Modal slide from bottom
  forModalPresentationIOS: CardStyleInterpolators.forModalPresentationIOS,

  // ── Custom: Scale + Fade (like Material Design shared element) ─────────────
  forScaleFromCenterAndroid: ({ current, next, layouts }) => {
    // current.progress: animation progress of the incoming screen (0 → 1)
    // next?.progress: animation progress of the outgoing screen (1 → 0)

    return {
      cardStyle: {
        opacity: current.progress,
        transform: [
          {
            scale: current.progress.interpolate({
              inputRange: [0, 1],
              outputRange: [0.85, 1],  // Grows from 85% to 100%
            }),
          },
        ],
      },
    };
  },
};

// Usage in navigator:
<Stack.Screen
  name="Modal"
  component={ModalScreen}
  options={{
    cardStyleInterpolator: transitionExamples.forScaleFromCenterAndroid,
    transitionSpec: {
      open:  { animation: 'spring', config: { stiffness: 200, damping: 20 } },
      close: { animation: 'timing', config: { duration: 200 } },
    },
  }}
/>
```

# Part III — Tab Navigation

## Chapter 5: What Is a Tab Navigator?

Tab navigation is the dominant navigation pattern in consumer mobile applications — Instagram's bottom tabs (Feed, Search, Reels, Shop, Profile), WhatsApp's tabs (Chats, Status, Calls), Twitter's tabs (Home, Search, Notifications, Messages) are all examples of this pattern. A tab bar is always visible at the bottom (or occasionally the top) of the screen, and tapping a tab switches to that section's screen. Critically, each tab maintains its own independent navigation history — navigating deep into a stack in the "Search" tab does not affect the "Home" tab's stack.

```
Tab Navigator — State Model

                Bottom Tab Bar
  ┌───────────┬───────────┬───────────┬───────────┐
  │   Home    │  Search   │  Profile  │ Settings  │
  │    🏠     │    🔍    │    👤     │    ⚙️     │
  └───────────┴───────────┴───────────┴───────────┘
       │            │           │           │
       ▼            ▼           ▼           ▼
  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Home    │  │ Search  │  │ Profile │  │Settings │
  │ Feed    │  │ Results │  │ Detail  │  │ Screen  │
  │  ┌───┐  │  │         │  │  ┌───┐  │  │         │
  │  │ A │  │  │         │  │  │ P │  │  │         │
  │  │ B │  │  │  empty  │  │  │ Q │  │  │  empty  │
  │  │ C │  │  │         │  │  │ R │  │  │         │
  │  └───┘  │  │         │  │  └───┘  │  │         │
  └─────────┘  └─────────┘  └─────────┘  └─────────┘
  Own stack    Own stack     Own stack     Own stack
  (3 screens)  (1 screen)    (3 screens)   (1 screen)

  Each tab has a completely independent navigation stack.
  Switching tabs preserves the stack in each tab.
```

### Example 2: Complete Bottom Tab Navigator with Badges and Custom Icons

```javascript
// npm install @react-navigation/bottom-tabs

import React, { useState } from 'react';
import { View, Text, TouchableOpacity, StyleSheet } from 'react-native';
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';

const Tab  = createBottomTabNavigator();
const Stack = createNativeStackNavigator();

// ── Placeholder screens for the example ──────────────────────────────────────

const SearchStack = () => (

const NotificationsStack = () => (
  <Stack.Navigator screenOptions={stackTheme}>
    <Stack.Screen name="NotifHome" component={() => <PlaceholderScreen title="Notifications" color="🔔" />} options={{ title: 'Notifications' }} />

const ProfileStack = () => (
  <Stack.Navigator screenOptions={stackTheme}>

// ── Custom Tab Bar Icon with Badge ────────────────────────────────────────────
```

# Part IV — Drawer Navigation

## Chapter 7: What Is a Drawer Navigator?

The Drawer Navigator implements the familiar "hamburger menu" pattern — a panel that slides in from the left (or right) edge of the screen, containing a navigation menu. The user opens it by tapping the hamburger icon in the header or by swiping from the screen edge. This pattern is ideal for applications with many top-level sections that do not fit neatly into a tab bar (which works best with 2–5 items).

```
Drawer Navigator — Open/Closed States

  CLOSED STATE:                  OPEN STATE:
  ┌────────────────────┐         ┌──────────┬─────────────────┐
  │ ☰  My App         │         │  🅐 App   │                 │
  ├────────────────────┤         │ ──────── │   Main          │
  │                    │  ──►    │ 🏠 Home  │   Content       │
  │   Main Content     │  swipe  │ 📊 Dash  │   Area          │
  │   Area             │  right  │ 👥 Users │   (dimmed)      │
  │                    │         │ ⚙️ Sett. │                 │
  │                    │         │ 🚪 Logout│                 │
  └────────────────────┘         └──────────┴─────────────────┘
         Full screen                  Drawer slides in from left

  Closing: tap dimmed area OR swipe left OR call navigation.closeDrawer()
```

# Part V — Advanced Navigation Patterns

## Chapter 8: Nested Navigators — The Real-World Architecture

Real applications are never a single navigator. They compose multiple navigators together in a tree — a tab navigator where each tab contains a stack navigator, a drawer navigator that wraps a tab navigator, or a root stack navigator that contains both an authentication flow and the main app. This composition is called **nested navigation**, and it is the most important architectural concept in React Navigation.

```
Nested Navigator Tree — Production App Example

NavigationContainer
└── Root Stack Navigator            ← Handles Auth vs Main split
    ├── Auth Stack                  ← Not logged in
    │   ├── Splash Screen
    │   ├── Login Screen
    │   └── Register Screen
    │
    └── Main (Drawer Navigator)     ← Logged in
        ├── Home (Tab Navigator)
        │   ├── Feed (Stack)
        │   │   ├── FeedHome
        │   │   ├── PostDetail
        │   │   └── UserProfile
        │   ├── Search (Stack)
        │   │   ├── SearchHome
        │   │   └── SearchResults
        │   └── Profile (Stack)
        │       ├── ProfileHome
        │       └── EditProfile
        │
        ├── Analytics Screen
        ├── Settings Screen
        └── Help Screen

Navigating across levels:
  navigation.navigate('Main', {
    screen: 'Home',           ← Enter the Tab navigator
    params: {
      screen: 'Search',       ← Enter the Search tab
      params: {
        screen: 'SearchResults',  ← Go to this screen
        params: { query: 'shoes' }
      }
    }
  })
```

## Chapter 9: Authentication Flow — The Most Important Navigation Pattern

Every application with user accounts needs to handle the transition between the "unauthenticated" state (showing Login/Register) and the "authenticated" state (showing the main app). The correct way to implement this in React Navigation is with **conditional rendering** of navigators — not with `navigation.navigate()`. When the auth state changes, React Navigation automatically animates between the two navigator trees.

```javascript
import React, { useState, useEffect, createContext, useContext } from 'react';
import { View, Text, TouchableOpacity, StyleSheet, ActivityIndicator } from 'react-native';
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';
import * as SecureStore from 'expo-secure-store';

const Stack = createNativeStackNavigator();

// ─────────────────────────────────────────────────────────────────────────────
// AUTH CONTEXT
// Provides auth state and methods to all screens without prop drilling.
// ─────────────────────────────────────────────────────────────────────────────
const AuthContext = createContext(null);
export const useAuth = () => useContext(AuthContext);

// ── Auth Screens ──────────────────────────────────────────────────────────────
const LoginScreen = ({ navigation }) => {
  const { signIn } = useAuth();
  return (
    <View style={authStyles.screen}>
      <Text style={authStyles.title}>Welcome Back</Text>
      <Text style={authStyles.subtitle}>Sign in to continue</Text>
      <TouchableOpacity style={authStyles.btn} onPress={() => signIn('fake_token_xyz')}>
        <Text style={authStyles.btnText}>Sign In</Text>
      </TouchableOpacity>
      <TouchableOpacity onPress={() => navigation.navigate('Register')}>
        <Text style={authStyles.link}>No account? Register →</Text>
      </TouchableOpacity>
    </View>
  );
};

const RegisterScreen = () => {
  const { signIn } = useAuth();
  return (
    <View style={authStyles.screen}>
      <Text style={authStyles.title}>Create Account</Text>
      <TouchableOpacity style={authStyles.btn} onPress={() => signIn('new_user_token')}>
        <Text style={authStyles.btnText}>Register & Sign In</Text>
      </TouchableOpacity>
    </View>
  );
};

const HomeScreen = ({ navigation }) => {
  const { signOut } = useAuth();
  return (
    <View style={authStyles.screen}>
      <Text style={authStyles.title}>🎉 Welcome!</Text>
      <Text style={authStyles.subtitle}>You are logged in.</Text>
      <TouchableOpacity style={[authStyles.btn, { backgroundColor: '#FF4444' }]} onPress={signOut}>
        <Text style={authStyles.btnText}>Sign Out</Text>
      </TouchableOpacity>
    </View>
  );
};

// ─────────────────────────────────────────────────────────────────────────────
// ROOT APP — Conditional Navigator Rendering
// ─────────────────────────────────────────────────────────────────────────────
export default function App() {
  const [userToken, setUserToken] = useState(undefined); // undefined = still loading
  const [isLoading, setIsLoading] = useState(true);

  // Check for existing token on app start
  useEffect(() => {
    const bootstrapAuth = async () => {
      try {
        const token = await SecureStore.getItemAsync('auth_token');
        setUserToken(token ?? null); // null = definitely not logged in
      } catch {
        setUserToken(null);
      } finally {
        setIsLoading(false);
      }
    };
    bootstrapAuth();
  }, []);

  // Auth context methods
  const authContext = {
    userToken,
    signIn: async (token) => {
      await SecureStore.setItemAsync('auth_token', token);
      setUserToken(token);
      // No navigation.navigate() needed — changing userToken causes React
      // to re-render the navigator tree, showing the auth'd screens automatically.
    },
    signOut: async () => {
      await SecureStore.deleteItemAsync('auth_token');
      setUserToken(null);
      // Same — the navigator tree switches back to auth screens automatically.
    },
  };

  if (isLoading) {
    return (
      <View style={authStyles.screen}>
        <ActivityIndicator size="large" color="#6C63FF" />
        <Text style={authStyles.subtitle}>Loading...</Text>
      </View>
    );
  }

  return (
    <AuthContext.Provider value={authContext}>
      <NavigationContainer>
        <Stack.Navigator screenOptions={{ headerShown: false }}>

          {userToken == null ? (
            // ── AUTH FLOW (not logged in) ──────────────────────────────────
            // React Navigation automatically animates the transition when
            // userToken changes from null to a value (or vice versa).
            <>
              <Stack.Screen name="Login"    component={LoginScreen}    />
              <Stack.Screen name="Register" component={RegisterScreen} />
            </>
          ) : (
            // ── APP FLOW (logged in) ───────────────────────────────────────
            <>
              <Stack.Screen name="Home" component={HomeScreen} />
              {/* Add more authenticated screens here */}
            </>
          )}

        </Stack.Navigator>
      </NavigationContainer>
    </AuthContext.Provider>
  );
}

const authStyles = StyleSheet.create({
  screen: { flex: 1, backgroundColor: '#080818', justifyContent: 'center', alignItems: 'center', padding: 32 },
  title: { color: '#FFF', fontSize: 28, fontWeight: '800', marginBottom: 8 },
  subtitle: { color: '#555', fontSize: 15, marginBottom: 40 },
  btn: { backgroundColor: '#6C63FF', borderRadius: 14, padding: 16, width: '100%', alignItems: 'center', marginBottom: 16 },
  btnText: { color: '#FFF', fontWeight: '700', fontSize: 16 },
  link: { color: '#6C63FF', fontSize: 14 },
});
```

## Chapter 10: Passing Parameters — Deep Dive

Passing data between screens is one of the most frequently performed operations in any navigation system, and doing it correctly is essential for building maintainable applications. React Navigation provides `route.params` for receiving data and the second argument to `navigate()` for sending it. However, there are several important rules and patterns that beginners commonly get wrong.

```javascript
// ─────────────────────────────────────────────────────────────────────────────
// SENDING PARAMS — from the source screen
// ─────────────────────────────────────────────────────────────────────────────

// ✅ Correct: Pass primitive values and plain objects
navigation.navigate('UserProfile', {
  userId:    42,
  userName:  'Alice',
  fromTab:   'Search',
});

// ❌ Wrong: Never pass class instances, functions, or non-serializable objects
// navigation.navigate('Detail', { user: new User(...) }); // Will cause warnings
// navigation.navigate('Detail', { onDone: () => {} });    // Functions not allowed

// ─────────────────────────────────────────────────────────────────────────────
// RECEIVING PARAMS — in the destination screen
// ─────────────────────────────────────────────────────────────────────────────

const UserProfileScreen = ({ navigation, route }) => {
  // Always destructure with defaults to handle edge cases
  // (e.g., screen opened from a deep link without params)
  const {
    userId   = null,
    userName = 'Unknown',
    fromTab  = 'Home',
  } = route.params ?? {};

  // IMPORTANT: Params are not reactive in the same way as state.
  // If Screen A passes params to Screen B, and then Screen A updates those
  // params, Screen B will NOT automatically re-render. To send data BACK
  // from B to A, use navigation.navigate('ScreenA', { updatedData: ... })
  // and read it in A with a useFocusEffect hook.

  return (
    <View>
      <Text>User ID: {userId}</Text>
      <Text>User: {userName}</Text>
    </View>
  );
};

// ─────────────────────────────────────────────────────────────────────────────
// SENDING DATA BACK — from child screen to parent
// ─────────────────────────────────────────────────────────────────────────────
import { useFocusEffect } from '@react-navigation/native';

// Parent Screen: reads updated data when it comes back into focus
const ParentScreen = ({ navigation, route }) => {
  const [selectedColor, setSelectedColor] = useState('Red');

  // useFocusEffect runs every time this screen gains focus —
  // including when the user navigates back to it from a child.
  useFocusEffect(
    React.useCallback(() => {
      if (route.params?.selectedColor) {
        setSelectedColor(route.params.selectedColor);
      }
    }, [route.params?.selectedColor])
  );

  return (
    <TouchableOpacity
      onPress={() => navigation.navigate('ColorPicker', { current: selectedColor })}
    >
      <Text>Current Color: {selectedColor} — Tap to change</Text>
    </TouchableOpacity>
  );
};

// Child Screen: sends data back by navigating to parent with params
const ColorPickerScreen = ({ navigation }) => {
  const COLORS = ['Red', 'Blue', 'Green', 'Purple', 'Orange'];

  return (
    <View>
      {COLORS.map(color => (
        <TouchableOpacity
          key={color}
          onPress={() => {
            // Navigate back to parent AND pass data as params
            navigation.navigate('Parent', { selectedColor: color });
          }}
        >
          <Text>{color}</Text>
        </TouchableOpacity>
      ))}
    </View>
  );
};
```

## Chapter 11: useNavigation and useRoute Hooks

React Navigation provides two powerful hooks — `useNavigation` and `useRoute` — that give any component access to the navigation context without needing to receive `navigation` and `route` as props. This is transformative for component architecture: a deeply nested button component can trigger navigation without the navigation prop being drilled through every layer of intermediary components.

```javascript
import { useNavigation, useRoute, useFocusEffect, useIsFocused } from '@react-navigation/native';
import React, { useCallback, useEffect } from 'react';
import { View, Text, TouchableOpacity } from 'react-native';

// ── useNavigation: Access navigation methods from ANY component ───────────────
// This component is NOT a registered screen — it is a reusable UI component.
// Without useNavigation, it would need navigation passed as a prop through
// potentially many layers of parents.
const ProductCard = ({ product }) => {
  // useNavigation() finds the nearest navigation context automatically.
  const navigation = useNavigation();

  return (
    <TouchableOpacity
      onPress={() => navigation.navigate('ProductDetail', { productId: product.id })}
    >
      <Text>{product.name}</Text>
    </TouchableOpacity>
  );
};

// ── useRoute: Access current route params from ANY component ──────────────────
const ProductBreadcrumb = () => {
  const route = useRoute();
  const { productName } = route.params ?? {};
  return <Text>Home › Products › {productName}</Text>;
};

// ── useFocusEffect: Run code when screen gains focus ──────────────────────────
// This is essential for refreshing data when a user navigates back to a screen.
// It fires every time the screen comes into focus, unlike useEffect which
// only fires on mount.
const RefreshableScreen = () => {
  const [data, setData] = React.useState([]);

  useFocusEffect(
    // IMPORTANT: Always wrap the callback in useCallback.
    // Without it, a new function is created on every render,
    // causing an infinite loop of focus events.
    useCallback(() => {
      // This runs every time this screen gains focus
      const loadData = async () => {
        const fresh = await fetchLatestData();
        setData(fresh);
      };
      loadData();

      // The return function runs when the screen LOSES focus (cleanup)
      return () => {
        console.log('Screen lost focus — cleanup if needed');
      };
    }, [])
  );

  return <Text>{data.length} items</Text>;
};

// ── useIsFocused: Conditional rendering based on focus state ──────────────────
// Returns true when the screen is focused, false otherwise.
// Useful for stopping expensive operations (e.g., camera, video) when unfocused.
const CameraScreen = () => {
  const isFocused = useIsFocused();
  return (
    <View>
      {isFocused
        ? <Text>📷 Camera is ACTIVE</Text>   // Render camera only when focused
        : <Text>⏸ Camera paused</Text>       // Stop rendering when navigated away
      }
    </View>
  );
};
```

## Chapter 12: Deep Linking — URLs That Open Your App

Deep linking is the ability for external URLs to open a specific screen within your application. When a user clicks `myapp://product/42` in a browser, an email, or a QR code, your application should open and navigate directly to the product detail screen for product 42. This is an essential feature for marketing campaigns, push notification CTAs, and web-to-app transitions.

```javascript
import { NavigationContainer, getStateFromPath } from '@react-navigation/native';

// ─────────────────────────────────────────────────────────────────────────────
// DEEP LINK CONFIGURATION
// This config object tells React Navigation how to map URLs to screen routes.
// ─────────────────────────────────────────────────────────────────────────────

const linking = {
  // The URI schemes your app responds to:
  // myapp:// for when opened from another app
  // https://myapp.com for Universal Links (iOS) / App Links (Android)
  prefixes: ['myapp://', 'https://myapp.com'],

  config: {
    screens: {
      // Simple mapping: myapp://home → HomeScreen
      Home: 'home',

      // Parameterized route: myapp://product/42 → ProductDetail with { productId: '42' }
      ProductDetail: {
        path: 'product/:productId',
        parse: {
          // parse converts the string from the URL to the correct JS type
          productId: (id) => parseInt(id, 10),  // '42' → 42 (number)
        },
        stringify: {
          // stringify converts params back to URL-friendly strings (for linking)
          productId: (id) => id.toString(),
        },
      },

      // Nested navigators: myapp://app/search/results?query=shoes
      Main: {
        screens: {
          Home: {
            screens: {
              Search: {
                screens: {
                  SearchResults: {
                    path: 'search/results',
                    parse: {
                      query: (q) => decodeURIComponent(q),
                    },
                  },
                },
              },
            },
          },
        },
      },
    },
  },
};

// Usage:
export default function App() {
  return (
    // Pass the linking config to NavigationContainer.
    // React Navigation handles everything else automatically:
    // It listens for incoming URLs, parses them using your config,
    // and navigates to the correct screen with the correct params.
    <NavigationContainer linking={linking}>
      <AppNavigator />
    </NavigationContainer>
  );
}

// ── Testing Deep Links ────────────────────────────────────────────────────────
// To test on iOS simulator:
//   xcrun simctl openurl booted "myapp://product/42"
//
// To test on Android emulator:
//   adb shell am start -a android.intent.action.VIEW -d "myapp://product/42"
//
// In Expo Go:
//   npx uri-scheme open "myapp://product/42" --ios
```

## Conclusion

Navigation is the nervous system of a mobile application. The `NavigationContainer` provides the environment. Navigators define the movement patterns — stacks for depth, tabs for sibling sections, drawers for large menus. Screens are the destinations. Parameters are the messages passed between them. Hooks like `useNavigation`, `useRoute`, and `useFocusEffect` make these primitives accessible anywhere in the component tree. The authentication flow pattern — conditional navigator rendering — is the foundation of every app with user accounts. And deep linking ties your app to the broader ecosystem of URLs and push notifications.

Mastering React Navigation means internalising not just the API but the _why_ behind each design decision — why `reset()` instead of `navigate()` after login, why each tab gets its own stack, why `useFocusEffect` instead of `useEffect` for data refreshing. These are the decisions that separate a polished, production-grade application from one that merely functions.
