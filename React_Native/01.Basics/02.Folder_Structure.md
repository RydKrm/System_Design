## 🔍 Detailed File & Folder Explanations

### 📄 **Root Configuration Files**

#### **package.json**

- **Purpose**: Defines project metadata, dependencies, and npm scripts
- **Contains**:
    - Production dependencies (react, react-native, redux, axios)
    - Development dependencies (TypeScript, ESLint, Jest)
    - Scripts for running, building, and testing the app
- **Usage**: Run `npm install` to install all dependencies

#### **tsconfig.json**

- **Purpose**: TypeScript compiler configuration
- **Contains**:
    - Compiler options (target ES version, JSX settings)
    - Path aliases (@components, @screens, etc.)
    - Include/exclude patterns
- **Usage**: Enables TypeScript support and intellisense
- **Key Features**:
    - Absolute imports: `import Button from '@components/Button'`
    - Strict type checking enabled

#### **babel.config.js**

- **Purpose**: Configures Babel JavaScript compiler
- **Contains**:
    - Presets for React Native
    - Module resolver plugin for path aliases
    - Reanimated plugin configuration
- **Usage**: Transforms modern JavaScript to compatible code
- **Why Important**: Enables ES6+ features and absolute imports

#### **metro.config.js**

- **Purpose**: Metro bundler configuration
- **Contains**: Configuration for the JavaScript bundler
- **Usage**: Used during development and production builds
- **Why Important**: Metro bundles all JS files into a single file

#### **app.json**

- **Purpose**: React Native app metadata
- **Contains**: App name and display name
- **Usage**: Used by React Native CLI and build tools

#### **.gitignore**

- **Purpose**: Specifies files Git should ignore
- **Contains**:
    - node_modules/
    - Build artifacts
    - OS-specific files
    - Environment variables
- **Why Important**: Keeps repository clean and secure

#### **App.tsx**

- **Purpose**: Root application component
- **Contains**:
    - Redux Provider wrapper
    - Navigation Container
    - Theme Provider
    - Status Bar configuration
- **Usage**: Entry point for the React component tree
- **Key Responsibilities**:
    - Initializes global providers
    - Sets up navigation
    - Configures app-wide settings

#### **index.js**

- **Purpose**: Application entry point (JavaScript)
- **Contains**: App registration with React Native
- **Usage**: First file loaded when app starts
- **Why Important**: Bridges native code with React components

### 📂 **src/components/** - Reusable UI Components

#### **Button.tsx**

- **Purpose**: Reusable, customizable button component
- **Props**:
    - `title`: Button text
    - `onPress`: Click handler
    - `style`: Custom styles
    - `disabled`: Disabled state
- **Usage**: `<Button title="Submit" onPress={handleSubmit} />`
- **Why Use**: Consistent button styling across the app

#### **Card.tsx**

- **Purpose**: Container component with elevation/shadow
- **Props**:
    - `children`: Child components
    - `style`: Custom styles
- **Usage**: `<Card><Text>Content</Text></Card>`
- **Why Use**: Creates uniform card-based layouts

**Future Components**:

- **Input.tsx**: Form input fields with validation
- **Modal.tsx**: Popup/dialog component
- **Loader.tsx**: Loading spinner/skeleton screens

---

### 📂 **src/screens/** - Full Screen Components

#### **HomeScreen.tsx**

- **Purpose**: Main landing screen of the app
- **Contains**:
    - Welcome message
    - Navigation buttons to other screens
- **Navigation Props**: Receives navigation prop for screen transitions
- **Usage**: First screen users see when app loads

#### **DetailsScreen.tsx**

- **Purpose**: Displays detailed information
- **Receives**: Route params (itemId)
- **Usage**: Navigated to from HomeScreen with data
- **Example**: Showing product details, article content

#### **ProfileScreen.tsx**

- **Purpose**: User profile display screen
- **Receives**: Route params (userId)
- **Usage**: Shows user information and settings
- **Example**: User avatar, name, bio, settings

**Best Practices**:

- Keep screens focused on layout and composition
- Business logic should be in services or hooks
- Use Redux for global state, local state for UI

---

### 📂 **src/navigation/** - Navigation Setup

#### **AppNavigator.tsx**

- **Purpose**: Main navigation stack configuration
- **Contains**:
    - Stack Navigator setup
    - Screen definitions
    - Navigation options (headers, transitions)
- **Type Safety**: Defines `RootStackParamList` for type-safe navigation
- **Usage**: Manages all app navigation flow

**Navigation Flow**:

```typescript
Home → Details (with itemId)
Home → Profile (with userId)
```

**Key Features**:

- Type-safe navigation params
- Customizable headers
- Screen transitions
- Deep linking support (when configured)

---

### 📂 **src/redux/** - State Management

#### **store/store.ts**

- **Purpose**: Redux store configuration
- **Contains**:
    - Store creation with configureStore
    - Middleware setup
    - Root reducer combination
- **Exports**: `RootState` and `AppDispatch` types
- **Usage**: Centralized app state management

#### **reducers/userSlice.ts**

- **Purpose**: User state management
- **State Shape**:
    
    ```typescript
    {  id: string | null,  name: string | null,  email: string | null,  isAuthenticated: boolean}
    ```
    
- **Actions**:
    - `setUser`: Login user
    - `clearUser`: Logout user
    - `updateUserName`: Update username
- **Usage**: `dispatch(setUser({ id, name, email }))`

#### **reducers/appSlice.ts**

- **Purpose**: App-level state (loading, errors, theme)
- **State Shape**:
    
    ```typescript
    {  isLoading: boolean,  error: string | null,  theme: 'light' | 'dark'}
    ```
    
- **Actions**:
    - `setLoading`: Toggle loading state
    - `setError`: Set error messages
    - `toggleTheme`: Switch between light/dark mode
- **Usage**: Global UI state management

#### **hooks.ts**

- **Purpose**: Typed Redux hooks
- **Exports**:
    - `useAppDispatch`: Typed dispatch hook
    - `useAppSelector`: Typed selector hook
- **Usage**: `const dispatch = useAppDispatch()`
- **Why Important**: Type safety in components

---

### 📂 **src/services/** - API Layer

#### **api.service.ts**

- **Purpose**: Base API service with Axios
- **Features**:
    - Axios instance with base URL
    - Request/response interceptors
    - Authentication token injection
    - Error handling (401, 500, etc.)
- **Methods**:
    - `get<T>(url)`: GET requests
    - `post<T>(url, data)`: POST requests
    - `put<T>(url, data)`: PUT requests
    - `delete<T>(url)`: DELETE requests
- **Usage**: Singleton service for all API calls

#### **user.service.ts**

- **Purpose**: User-related API operations
- **Methods**:
    - `login(credentials)`: User login
    - `logout()`: User logout
    - `getUserProfile(userId)`: Get user data
    - `updateUserProfile(userId, data)`: Update user
- **Usage**: `UserService.login({ email, password })`
- **Why Separate**: Organizes API calls by domain

**Architecture Benefits**:

- Centralized API logic
- Easy to mock for testing
- Consistent error handling
- Token management in one place

---

### 📂 **src/hooks/** - Custom React Hooks

#### **useFetch.ts**

- **Purpose**: Data fetching with loading and error states
- **Returns**:
    
    ```typescript
    {  data: T | null,  loading: boolean,  error: Error | null,  refetch: () => void}
    ```
    
- **Usage**:
    
    ```typescript
    const { data, loading, error } = useFetch<User>('/api/user/1');
    ```
    
- **Why Use**: Simplifies API calls in components

#### **useDebounce.ts**

- **Purpose**: Debounces rapid value changes
- **Usage**:
    
    ```typescript
    const debouncedSearch = useDebounce(searchTerm, 500);
    ```
    
- **Use Cases**:
    - Search input fields
    - API calls on user typing
    - Performance optimization

**Hook Benefits**:

- Reusable logic
- Cleaner components
- Separation of concerns
- Easy testing

---

### 📂 **src/utils/** - Utility Functions

#### **validation.ts**

- **Purpose**: Form validation helpers
- **Functions**:
    - `validateEmail(email)`: Email format validation
    - `validatePassword(password)`: Password strength check
    - `validatePhoneNumber(phone)`: Phone format validation
    - `validateURL(url)`: URL validation
    - `isEmpty(value)`: Empty check
- **Usage**:
    
    ```typescript
    const isValid = validateEmail('user@example.com');const { isValid, errors } = validatePassword('pass123');
    ```
    

#### **dateFormatter.ts**

- **Purpose**: Date formatting and manipulation
- **Functions**:
    - `formatDate(date, format)`: Custom date formatting
    - `getRelativeTime(date)`: "2 hours ago" format
    - `isToday(date)`: Check if date is today
    - `isYesterday(date)`: Check if date is yesterday
- **Usage**:
    
    ```typescript
    formatDate(new Date(), 'MM/DD/YYYY');getRelativeTime('2024-01-15'); // "2 days ago"
    ```
    

#### **stringUtils.ts**

- **Purpose**: String manipulation utilities
- **Functions**:
    - `capitalize(str)`: Capitalize first letter
    - `capitalizeWords(str)`: Title case
    - `truncate(str, maxLength)`: Shorten strings
    - `slugify(str)`: URL-friendly strings
    - `generateRandomString(length)`: Random IDs
- **Usage**:
    
    ```typescript
    capitalize('hello'); // "Hello"truncate('Long text...', 10); // "Long te..."
    ```

### 📂 **src/context/** - React Context

#### **ThemeContext.tsx**

- **Purpose**: Global theme management (light/dark mode)
- **Provides**:
    - `isDarkMode`: Boolean theme state
    - `colors`: Theme color values
    - `toggleTheme()`: Switch themes
- **Usage**:
    
    ```typescript
    const { colors, toggleTheme } = useTheme();
    ```
    
- **Benefits**:
    - App-wide theme consistency
    - No prop drilling
    - Easy theme switching

**When to Use Context**:

- Global state needed in many components
- Avoid prop drilling
- Alternative to Redux for simple state

## 🔄 **Data Flow Example**

```
User Action (Button Press)
    ↓
Screen Component (HomeScreen)
    ↓
Dispatch Redux Action (setLoading)
    ↓
Call Service (UserService.getUserProfile)
    ↓
API Service (HTTP Request with Axios)
    ↓
Backend API
    ↓
Response Returns
    ↓
Update Redux State (setUser)
    ↓
Component Re-renders (with new data)
```

1. **Keep components small and focused**
2. **Use TypeScript for type safety**
3. **Write reusable utilities**
4. **Separate business logic from UI**
5. **Use absolute imports with aliases**
6. **Handle errors gracefully**
7. **Write meaningful commit messages**
8. **Document complex logic**
9. **Test critical paths**
10. **Keep dependencies updated**
