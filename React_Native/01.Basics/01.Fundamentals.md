	# React Native Fundamentals

### A Beginner's Guide to How React Native Works Under the Hood

## Chapter 1: How React Native Works on Android

Imagine you are writing a letter in English, but the recipient only understands French. You would need a translator sitting in the middle, reading your English letter and converting every sentence into French before handing it over. React Native works in a very similar fashion.

When you write a React Native application, you write your code in **JavaScript** — a language that mobile phones do not natively understand. Android, on the other hand, is built on **Java and Kotlin**, and its graphical interface is rendered using native Android components. So React Native's primary job is to act as that translator between your JavaScript world and the Android native world.

When your application launches on an Android device, two separate "worlds" come to life simultaneously. The first world is the **JavaScript thread**, where all your application logic, state management, and React component trees live. The second world is the **Native thread (Main/UI thread)**, where the actual Android Views are drawn on the screen — the buttons, scrollable lists, text inputs, and images that the user physically sees and touches.

```
┌────────────────────────────────────────────────────────────┐
│                      Android Device                        │
│                                                            │
│   ┌──────────────────┐         ┌──────────────────────┐    │
│   │ JavaScript World │         │  Native Android World│    │
│   │                  │         │                      │    │
│   │  - Your JS Code  │◄───────►│ - Android Views      │    │
│   │  - React Logic   │  Bridge │ - Buttons, Text      │    │
│   │  - State/Props   │         │ - Camera, GPS        │    │
│   │  - Business Logic│         │ - Actual UI on screen│    │
│   └──────────────────┘         └──────────────────────┘    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

These two worlds do not share memory. They cannot directly call each other's functions. This is a fundamental and important concept to grasp. Everything you do in JavaScript — pressing a button, scrolling a list, fetching data — must be communicated across to the native side through a carefully managed communication channel. In the old architecture, this channel was called the **Bridge**, which we shall explore shortly.

## Chapter 2: The Metro Bundler — Your Application's Personal Librarian

Before your JavaScript code ever reaches the phone, it must be prepared and packaged. This is the responsibility of the **Metro Bundler**. Think of Metro as a very diligent librarian who, before handing you a book, first collects every chapter from different shelves, arranges them in order, removes unnecessary pages, and staples everything into a single neat document for you to carry.

When you run your React Native project, Metro starts up as a local development server on your computer. It reads your entry file (usually `index.js`), then follows every `import` and `require` statement like a trail of breadcrumbs, collecting every single JavaScript file your application depends upon — including files from third-party libraries inside `node_modules`.

```
                          ┌──────────────────┐
   Your Source Files      │                  │
                          │                  │
   index.js ─────────────►│                  │
   App.js ───────────────►│   Metro Bundler  │────► bundle.js
   components/ ──────────►│                  │      (Single file,
   node_modules/ ────────►│   - Resolve      │       ready for
   assets/ ──────────────►│   - Transform    │       the device)
                          │   - Bundle       │
                          │                  │
                          └──────────────────┘
                                  │
                          ┌───────▼────────┐
                          │  Dev Server    │
                          │  localhost:8081│
                          │                │
                          │  Hot Reload ✓  │
                          │  Fast Refresh ✓│
                          └─────────-──────┘
```

Once all files are gathered, Metro **transforms** them. Since phones do not understand the latest modern JavaScript syntax (like JSX, which looks like `<View>` inside JavaScript), Metro uses **Babel** underneath to convert that modern code into older, universally compatible JavaScript. It also converts your JSX into plain function calls.

After transformation, Metro **bundles** everything — stitching all those individual files into a single JavaScript bundle file. This bundle is then served over a local network to your phone or emulator during development. When you save a file on your computer, Metro reprocesses only the changed portions and sends the update to your device almost instantly, enabling the beloved **Fast Refresh** feature that React Native developers rely upon every day.

In production (when you build your app for release on the Play Store), Metro produces a final, minified, and optimized bundle that gets embedded directly inside the APK file.

## Chapter 3: The Bridge Architecture — The Old Messenger System

For many years, the Bridge was the heart of React Native's communication model. To understand it deeply, picture a postal service operating between two countries that speak different languages. Every message must be written down, placed in an envelope, sent through the postal system, received on the other side, translated, and then acted upon. The Bridge worked in exactly this manner.

Technically, the Bridge serialized (converted) messages into **JSON strings** before sending them across, and deserialized (decoded) them on the other side. This meant that every interaction — a button press on the native side sending an event to JavaScript, or JavaScript sending an instruction to render a new View — had to go through this expensive process of converting data to text and back again.

```
┌─────────────────────────────────────────────────────────────────────┐
│                     OLD Bridge Architecture                         │
│                                                                     │
│  ┌─────────────┐          ┌──────────────┐        ┌──────────────┐  │
│  │  JS Thread  │          │    Bridge    │        │ Native Thread│  │
│  │             │          │              │        │              │  │
│  │ "Render a   │──JSON───►│  Serialize   │──────► │Create Android│  │
│  │  red button"│          │  Deserialize │        │ Button View  │  │
│  │             │◄──JSON───│              │◄─────  │              │  │
│  │ "Button was │          │  (Async      │        │ "User tapped │  │
│  │  pressed"   │          │   Queue)     │        │  the button" │  │
│  └─────────────┘          └──────────────┘        └──────────────┘  │
│                                                                     │
│  ⚠️  Problems:                                                      │
│     - Messages are ASYNCHRONOUS (no direct calls)                   │
│     - JSON serialization is SLOW for large data                     │
│     - Cannot share memory between threads                           │
│     - Animations could feel janky under heavy load                  │
└─────────────────────────────────────────────────────────────────────┘
```

The Bridge had three significant limitations that eventually prompted React Native's engineers to rethink the entire system.

Firstly, the Bridge was **asynchronous by nature**. JavaScript could not directly call a native function and wait for a result. It had to send a message and wait for a reply to come back later through the queue. This made certain operations feel sluggish, particularly animations and gesture-driven interactions where every frame matters.

Secondly, the Bridge imposed a **serialization overhead**. Whenever you passed a large object — say, a list of a thousand items — across the Bridge, React Native had to convert that entire object into a JSON string, send it across, and then parse it back into an object on the other side. For large datasets or rapid interactions, this created a measurable performance bottleneck.

Thirdly, the Bridge was a **single communication lane**. All messages from all parts of your application had to travel through the same queue, which could become congested during heavy rendering.

Despite these limitations, the Bridge architecture powered thousands of production applications and was genuinely remarkable in what it achieved. It was the foundation upon which React Native proved that writing mobile applications in JavaScript was not only possible but practical.

## Chapter 4: The New Architecture — JSI and Fabric

Recognizing the limitations of the Bridge, the React Native team embarked on a fundamental reimagining of the entire system, resulting in what is now called the **New Architecture**. The New Architecture is built upon two primary pillars: **JSI (JavaScript Interface)** and **Fabric**.

### JSI — Tearing Down the Bridge

JSI, which stands for **JavaScript Interface**, is perhaps the most transformative change in React Native's history. Where the old Bridge required messages to be serialized into JSON and passed asynchronously through a queue, JSI allows JavaScript to hold **direct references to native C++ objects** and call their methods synchronously, as if they were plain JavaScript functions.

```
┌────────────────────────────────────────────────────────────────────┐
│                      NEW Architecture with JSI                     │
│                                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    Shared C++ Layer (JSI)                   │   │
│  │                                                             │   │
│  │  JavaScript Runtime ◄──── Direct Memory Reference ────► Native  │ 
│  │                                                             │   │
│  │  ✅ No JSON serialization                                   │   │
│  │  ✅ Synchronous calls possible                              │   │
│  │  ✅ Shared memory — no copying of data                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                    │
│   JS: nativeModule.getValue()  ──────► Directly calls C++ method   │
│                                        Returns result instantly    │
│                                        No queue. No JSON. No wait  │
└────────────────────────────────────────────────────────────────────┘
```

Think of the old Bridge as two people communicating by writing letters. JSI, by contrast, is like those two people now sitting in the same room, speaking directly to one another. The communication is immediate, direct, and far more efficient.

JSI also enables a feature called **Host Objects** — JavaScript objects that actually live in native memory. When you access a property on such an object from JavaScript, you are directly reaching into native memory, without any copying or translation taking place.

### Fabric — The New Rendering Engine

While JSI modernized how JavaScript communicates with native code, **Fabric** is the reimagining of React Native's rendering system. In the old architecture, when React produced a virtual tree of components (what it wanted to display), this tree had to be sent across the Bridge to the native side, which would then create actual Android Views. This process was slow and one-directional.

```
┌─────────────────────────────────────────────────────────────────────┐
│                     Fabric Rendering System                         │
│                                                                     │
│  React JS Layer                                                     │
│  ┌─────────────────┐                                                │
│  │  React Element  │                                                │
│  │  Tree (Virtual) │                                                │
│  └────────┬────────┘                                                │
│           │  (via JSI — synchronous, no Bridge)                     │
│           ▼                                                         │
│  C++ Layer                                                          │
│  ┌─────────────────────────────────────────────────┐                │
│  │          Shadow Tree (C++ Yoga Layout)          │                │
│  │  Calculates positions, sizes, layout — in C++   │                │
│  └────────────────────┬────────────────────────────┘                │
│                       │                                             │
│                       ▼                                             │
│  Native Layer                                                       │
│  ┌─────────────────────────────────────────────────┐                │
│  │        Actual Android Views on Screen           │                │
│  │   (Rendered with full knowledge of layout)      │                │
│  └─────────────────────────────────────────────────┘                │
│                                                                     │
│  ✅ Synchronous rendering for critical interactions                 │
│  ✅ Layout calculated in C++ — faster than before                   │
│  ✅ Better support for React Concurrent Features                    │
└─────────────────────────────────────────────────────────────────────┘
```

Fabric introduces a **C++ Shadow Tree** that sits between your React components and the actual Android Views. This Shadow Tree uses a layout engine called **Yoga** (also written in C++) to calculate the positions and dimensions of every component on screen. Because this calculation now happens in C++ rather than being shuffled across the Bridge, it is significantly faster.

Furthermore, because Fabric communicates with JavaScript through JSI rather than the old Bridge, the rendering pipeline can now be **synchronous when necessary**. This is crucial for interactions like gestures, where you need the UI to respond to the user's finger within a single frame (16 milliseconds) to feel fluid and natural.

### Putting It All Together

```
┌─────────────────────────────────────────────────────────────────────┐
│               Complete New Architecture Overview                    │
│                                                                     │
│  ┌──────────────────┐                                               │
│  │  Your JavaScript │                                               │
│  │  Code & React    │                                               │
│  └────────┬─────────┘                                               │
│           │                                                         │
│           ▼  (JSI — Direct, Synchronous)                            │
│  ┌──────────────────────────────────────────┐                       │
│  │           C++ Core (Shared Layer)        │                       │
│  │                                          │                       │
│  │  ┌──────────────┐   ┌──────────────────┐ │                       │
│  │  │   Fabric     │   │  TurboModules    │ │                       │
│  │  │  (Rendering) │   │  (Native Modules)│ │                       │
│  │  └──────┬───────┘   └────────┬─────────┘ │                       │
│  └─────────┼────────────────────┼───────────┘                       │
│            │                    │                                   │
│            ▼                    ▼                                   │
│  ┌─────────────┐      ┌──────────────────────┐                      │
│  │ Android UI  │      │  Native APIs         │                      │
│  │ (Views)     │      │  (Camera, GPS, etc.) │                      │
│  └─────────────┘      └──────────────────────┘                      │
└─────────────────────────────────────────────────────────────────────┘
```

The New Architecture also introduces **TurboModules**, which are the modern replacement for the old Native Modules system. In the old world, all native modules — whether your app used them or not — were initialized at startup, which slowed down app launch times considerably. TurboModules are loaded **lazily**, meaning they are only initialized the moment your JavaScript code actually needs them, resulting in faster startup performance.

Together, JSI, Fabric, and TurboModules represent a complete rethinking of React Native from the ground up — one that removes the fundamental bottleneck of the old Bridge and positions React Native as a genuinely high-performance option for building mobile applications.

## Summary

|Concept|What It Does|Analogy|
|---|---|---|
|**React Native on Android**|Translates JS to Native Android Views|A translator between two people speaking different languages|
|**Metro Bundler**|Packages all JS files into one bundle|A librarian who assembles all chapters into one book|
|**Bridge (Old)**|Async JSON message passing between JS and Native|Sending letters between two countries via postal service|
|**JSI (New)**|Direct memory access between JS and Native C++|Two people in the same room speaking directly|
|**Fabric (New)**|Modern rendering engine using C++ Shadow Tree|A faster, smarter architect who calculates the blueprint before building|
|**TurboModules (New)**|Lazy-loaded native modules|A waiter who only fetches the dish when you actually order it|

---

_The journey from the Bridge to JSI and Fabric represents one of the most significant engineering transformations in mobile development tooling. Understanding these layers deeply will make you a far more capable React Native developer, for you will know not just how to write components, but precisely what happens beneath them._
