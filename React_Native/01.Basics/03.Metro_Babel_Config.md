# Metro Bundler Overview

## What is Metro?

**Metro** is the JavaScript bundler specifically designed for React Native by Facebook/Meta. It serves the same purpose as Webpack does for web applications, but it's optimized for React Native's unique requirements.

### Key Responsibilities:

1. **Bundles JavaScript files** - Combines all your JS/TS files into a single bundle
2. **Transforms modern JavaScript** - Converts ES6+ to compatible JavaScript
3. **Handles dependencies** - Resolves `import` and `require` statements
4. **Provides fast refresh** - Enables hot reloading during development
5. **Optimizes for mobile** - Creates efficient bundles for iOS and Android

### Why Metro Instead of Webpack?

- **Faster for React Native** - Optimized for mobile development workflow
- **Incremental bundling** - Only rebuilds changed files
- **Better dev experience** - Faster hot reload and refresh
- **Native platform support** - Works seamlessly with iOS and Android build tools

Let's break down every part:

---

## 1. **Import Statement**

```javascript
const { getDefaultConfig, mergeConfig } = require('@react-native/metro-config');
```

### What This Does:

- **`getDefaultConfig`**: A function that returns Metro's default configuration
- **`mergeConfig`**: A function that safely merges custom config with defaults
- **`@react-native/metro-config`**: Official React Native Metro config package

### Why It's Important:

- Gets you sensible defaults that work out-of-the-box
- Ensures your custom settings don't break default behavior
- Follows React Native's recommended configuration pattern

---

## 2. **JSDoc Comment**

```javascript
/**
 * Metro configuration
 * https://facebook.github.io/metro/docs/configuration
 *
 * @type {import('metro-config').MetroConfig}
 */
```

### What This Does:

- **Documentation comment** using JSDoc syntax
- **Type annotation** for TypeScript/IDE intellisense
- **Reference link** to official Metro documentation

### Benefits:

- Provides autocomplete in VS Code and other editors
- Type checking even in JavaScript files
- Helps developers understand configuration structure

---

## 3. **Configuration Object**

```javascript
const config = {};
```

### What This Does:

- Creates an **empty object** for custom Metro configuration
- This is where you add custom settings to override/extend defaults

### Currently Empty Because:

- Default configuration is sufficient for most projects
- You only add custom config when you need specific features
- Keeps configuration minimal and maintainable

---

## 4. **Export with Merge**

```javascript
module.exports = mergeConfig(getDefaultConfig(__dirname), config);
```

### What This Does:

- **`getDefaultConfig(__dirname)`**: Gets default Metro config for this project directory
- **`mergeConfig(..., config)`**: Safely merges your custom config with defaults
- **`module.exports`**: Exports the final merged configuration

### Why Use `mergeConfig`:

- **Safe merging** - Prevents accidentally overwriting critical defaults
- **Deep merge** - Correctly combines nested objects
- **Validation** - Ensures final config is valid

### What is `__dirname`:

- **Node.js global variable** containing the absolute path of the current directory
- **Used by Metro** to resolve file paths relative to project root
- **Example**: `/Users/yourname/Projects/ReactNativeProject`

# Advanced Metro Configuration Options

Here's what you can customize in the `config` object:

## Common Configuration Properties

```javascript
const config = {
  // 1. TRANSFORMER OPTIONS
  transformer: {
    // Enable inline requires for better performance
    getTransformOptions: async () => ({
      transform: {
        experimentalImportSupport: false,
        inlineRequires: true,
      },
    }),
    
    // Configure Babel transformer
    babelTransformerPath: require.resolve('react-native-svg-transformer'),
    
    // Asset plugins
    assetPlugins: ['asset-plugin-name'],
  },

  // 2. RESOLVER OPTIONS
  resolver: {
    // File extensions Metro should handle
    sourceExts: ['jsx', 'js', 'ts', 'tsx', 'json'],
    
    // Asset file extensions
    assetExts: ['png', 'jpg', 'jpeg', 'gif', 'webp', 'svg'],
    
    // Extra node modules to include
    extraNodeModules: {
      'custom-module': '/path/to/module',
    },
    
    // Blacklist/blocklist certain paths
    blockList: [/node_modules\/.*\/node_modules\/react-native\/.*/],
  },

  // 3. SERIALIZER OPTIONS
  serializer: {
    // Modules to exclude from bundle
    getModulesRunBeforeMainModule: () => [],
    
    // Polyfills to include
    getPolyfills: () => [],
  },

  // 4. SERVER OPTIONS
  server: {
    // Port for Metro bundler server
    port: 8081,
  },

  // 5. WATCHER OPTIONS
  watchFolders: [
    // Additional folders to watch for changes
    '/path/to/shared/components',
  ],

  // 6. PROJECT ROOT
  projectRoot: __dirname,
  watchFolders: [__dirname],
};
```

Let's explain each section:

---

## Transformer Options

### `transformer.getTransformOptions`

```javascript
transformer: {
  getTransformOptions: async () => ({
    transform: {
      experimentalImportSupport: false,
      inlineRequires: true,
    },
  }),
}
```

**Purpose**: Controls how code is transformed

**`experimentalImportSupport`**:

- **What**: Enables/disables import/export syntax transformations
- **Default**: `false`
- **When to enable**: If using advanced import features
- **Impact**: May affect bundle size and performance

**`inlineRequires`**:

- **What**: Converts imports to inline require() calls
- **Default**: `false`
- **When to enable**: To improve app startup performance
- **How it works**: Lazy loads modules only when needed
- **Example**:
    
    ```javascript
    // Beforeimport Button from './Button';// After (with inlineRequires: true)const Button = require('./Button');// Only loaded when Button is actually used
    ```
    

### `transformer.babelTransformerPath`

```javascript
babelTransformerPath: require.resolve('react-native-svg-transformer')
```

**Purpose**: Specifies custom Babel transformer

**Use Cases**:

- **SVG files**: Use `react-native-svg-transformer` to import SVG as components
- **Custom transformers**: Process files in special ways
- **Example with SVG**:
    
    ```javascript
    // Allows you to do:import Logo from './logo.svg';<Logo width={120} height={120} />
    ```
    

### `transformer.assetPlugins`

```javascript
assetPlugins: ['asset-plugin-name']
```

**Purpose**: Plugins to process assets during bundling

**Use Cases**:

- Image optimization
- Font processing
- Custom asset transformations

---

## Resolver Options

### `resolver.sourceExts`

```javascript
resolver: {
  sourceExts: ['jsx', 'js', 'ts', 'tsx', 'json']
}
```

**Purpose**: File extensions Metro treats as source code

**Default**: `['js', 'jsx', 'json', 'ts', 'tsx']`

**When to modify**:

- Adding new file types: `.mjs`, `.cjs`
- Custom extensions: `.graphql`, `.gql`

**Example with GraphQL**:

```javascript
sourceExts: ['jsx', 'js', 'ts', 'tsx', 'json', 'graphql', 'gql']
```

### `resolver.assetExts`

```javascript
resolver: {
  assetExts: ['png', 'jpg', 'jpeg', 'gif', 'webp', 'svg']
}
```

**Purpose**: File extensions Metro treats as static assets

**Default**: Includes common image, font, and media formats

**When to modify**:

- **Adding SVG as assets** (not components)
- **Custom file types**: `.obj`, `.fbx` (for 3D)
- **Fonts**: `.ttf`, `.otf`, `.woff`

**Example**:

```javascript
assetExts: ['png', 'jpg', 'ttf', 'otf', 'mp4', 'mp3']
```

### `resolver.extraNodeModules`

```javascript
resolver: {
  extraNodeModules: {
    'stream': require.resolve('readable-stream'),
  }
}
```

**Purpose**: Resolve custom module paths or polyfills

**Use Cases**:

- **Polyfills**: Add Node.js modules to React Native
- **Monorepos**: Link to shared packages
- **Path aliases**: Create shortcuts (though Babel is better for this)

**Example for crypto polyfill**:

```javascript
extraNodeModules: {
  'crypto': require.resolve('react-native-crypto'),
}
```

### `resolver.blockList` (formerly blacklist)

```javascript
resolver: {
  blockList: [
    /node_modules\/.*\/node_modules\/react-native\/.*/,
    /my-package\/dist\/.*/,
  ]
}
```

**Purpose**: Exclude certain files/folders from bundling

**Use Cases**:

- **Prevent duplicate dependencies**: Multiple React Native copies
- **Exclude build artifacts**: `dist/`, `build/`
- **Skip large files**: Documentation, tests in node_modules

**How it works**:

- Takes array of **regular expressions**
- Files matching patterns are **ignored**

# Babel Overview

## What is Babel?

**Babel** is a JavaScript **transpiler** (transformer/compiler) that converts modern JavaScript into backward-compatible versions.

### Key Responsibilities:

1. **Transforms modern syntax** - ES6+ → ES5
2. **JSX transformation** - JSX → React.createElement
3. **TypeScript compilation** - TS → JS
4. **Code optimization** - Minification, tree shaking
5. **Polyfill injection** - Add missing features

### Why Babel in React Native?

- **Modern JavaScript**: Use latest ES features
- **JSX Support**: Transform JSX to JavaScript
- **Cross-platform**: Same code works on iOS and Android
- **Plugin ecosystem**: Extend with custom transformations

          '@config': './src/config',

          '@context': './src/context',

        },
      },
    ],

Let's break down every part:

---

## 1. **Presets**

```javascript
presets: ['module:metro-react-native-babel-preset']
```

### What is a Preset?

A **preset** is a collection of Babel plugins bundled together. Instead of configuring dozens of individual plugins, you use a preset that includes everything you need.

### `metro-react-native-babel-preset`

**What it includes**:

1. **@babel/preset-env** - Transforms ES6+ syntax
2. **@babel/preset-react** - Transforms JSX
3. **@babel/preset-typescript** - Transforms TypeScript
4. **@babel/preset-flow** - Transforms Flow types (if used)
5. **react-native specific plugins**:
    - Transform arrow functions
    - Transform classes
    - Transform destructuring
    - Transform spread operators
    - Transform template literals
    - And many more...

**Example of what it does**:

```javascript
// Your code (ES6+)
const MyComponent = ({ name }) => {
  return <Text>Hello {name}</Text>;
};

// After Babel transformation
var MyComponent = function MyComponent(_ref) {
  var name = _ref.name;
  return React.createElement(Text, null, "Hello ", name);
};
```

### Why Use `module:` Prefix?

```javascript
'module:metro-react-native-babel-preset'
```

**Explanation**:

- **`module:`** tells Babel to resolve this as a Node module
- **Without it**: Babel would look for a file
- **With it**: Babel looks in `node_modules`

**Equivalent to**:

```javascript
require.resolve('metro-react-native-babel-preset')
```

### What This Preset Transforms:

#### 1. **JSX Syntax**

```javascript
// Before
<View style={styles.container}>
  <Text>Hello</Text>
</View>

// After
React.createElement(View, { style: styles.container },
  React.createElement(Text, null, "Hello")
)
```

#### 2. **Arrow Functions**

```javascript
// Before
const add = (a, b) => a + b;

// After
var add = function add(a, b) {
  return a + b;
};
```

#### 3. **Template Literals**

```javascript
// Before
const message = `Hello ${name}`;

// After
var message = "Hello " + name;
```

#### 4. **Destructuring**

```javascript
// Before
const { name, age } = user;

// After
var name = user.name;
var age = user.age;
```

#### 5. **Spread Operators**

```javascript
// Before
const newArray = [...oldArray, newItem];

// After
var newArray = oldArray.concat([newItem]);
```

#### 6. **Async/Await**

```javascript
// Before
const fetchData = async () => {
  const response = await fetch(url);
  return response.json();
};

// After (using regenerator-runtime)
var fetchData = function() {
  // Complex transformation with state machine
  // for async/await support
};
```

---

## 2. **Plugins**

```javascript
plugins: [
  'react-native-reanimated/plugin',
  [
    'module-resolver',
    { /* config */ }
  ],
]
```

Plugins are **individual transformations** applied to your code. They run in the order specified.

---

### Plugin 1: **react-native-reanimated/plugin**

```javascript
'react-native-reanimated/plugin'
```

**Purpose**: Enables React Native Reanimated library to work properly

**What is Reanimated?**

- Advanced animation library for React Native
- Runs animations on native thread (not JavaScript thread)
- Provides 60fps animations

**What This Plugin Does**:

1. **Transforms worklets** - Special functions that run on UI thread
    
    ```javascript
    // Before
    const animatedStyle = useAnimatedStyle(() => {
      return {
        transform: [{ translateX: offset.value }]
      };
    });
    
    // After: Plugin wraps this in special code
    // that allows it to run on native thread
    ```
    
2. **Auto-detects worklets** - Functions marked with 'worklet' directive
    
3. **Serializes functions** - Converts JS functions to run natively
    
4. **Optimizes performance** - Removes unnecessary code
    

**Critical Note**:

- **Must be listed LAST** in plugins array
- Won't work correctly if other plugins run after it
- Documentation specifically states this requirement

**Why It's Important**: Without this plugin, Reanimated animations will:

- Run on JavaScript thread (slower)
- Drop frames (janky animations)
- Not work at all in many cases

### Plugin 2: **babel-plugin-module-resolver**

```javascript
[
  'module-resolver',
  {
    root: ['./src'],
    extensions: ['.ios.js', '.android.js', '.js', '.ts', '.tsx', '.json'],
    alias: {
      '@components': './src/components',
      '@screens': './src/screens',
      // ... more aliases
    },
  },
]
```

**Purpose**: Create path aliases for cleaner imports

#### **Plugin Configuration Array Format**

```javascript
[
  'plugin-name',    // Plugin name
  { /* config */ }  // Plugin configuration object
]
```

When a plugin needs configuration, it's specified as an array with two elements:

1. Plugin name (string)
2. Configuration object

---

#### **Property: `root`**

```javascript
root: ['./src']
```

**Purpose**: Sets the base directory for module resolution

**What it does**:

- Tells Babel where to start looking for modules
- Enables relative imports from this root
- **Example**:
    
    ```javascript
    // Without rootimport Button from '../../../components/Button';// With root: ['./src']import Button from 'components/Button';
    ```
    

**Why array?**

- Can specify **multiple roots**
- Useful for monorepos or multiple source directories
- **Example**:
    
    ```javascript
    root: ['./src', './shared', './lib']
    ```
    

---

#### **Property: `extensions`**

```javascript
extensions: ['.ios.js', '.android.js', '.js', '.ts', '.tsx', '.json']
```

**Purpose**: File extensions the resolver should handle

**Order Matters!**

- Babel tries extensions **in order**
- First match wins

**Platform-Specific Extensions**:

```javascript
'.ios.js'     // iOS-specific file
'.android.js' // Android-specific file
'.js'         // Cross-platform file
```

**How it works**:

```javascript
import Button from './Button';

// Babel looks for files in this order:
// 1. Button.ios.js (on iOS)
// 2. Button.android.js (on Android)
// 3. Button.js (fallback)
// 4. Button.ts
// 5. Button.tsx
// 6. Button.json
```

**Real-World Example**:

```
components/
├── Button.ios.js      → Used on iOS
├── Button.android.js  → Used on Android
└── Button.js          → Fallback for both (if platform files missing)
```

**Why This Order?**

1. **Platform-specific first** - Use optimized code for each platform
2. **TypeScript next** - Support TS files
3. **JSON last** - Import JSON config files

---

#### **Property: `alias`**

```javascript
alias: {
  '@components': './src/components',
  '@screens': './src/screens',
  '@navigation': './src/navigation',
  // ... more
}
```

**Purpose**: Create shortcuts (aliases) for import paths

**What Each Alias Does**:

|Alias|Resolves To|Example Usage|
|---|---|---|
|`@components`|`./src/components`|`import Button from '@components/Button'`|
|`@screens`|`./src/screens`|`import HomeScreen from '@screens/HomeScreen'`|
|`@utils`|`./src/utils`|`import { validateEmail } from '@utils/validation'`|
|`@hooks`|`./src/hooks`|`import useFetch from '@hooks/useFetch'`|
|`@redux`|`./src/redux`|`import { store } from '@redux/store/store'`|

**Before Aliases**:

```javascript
// Deep nesting nightmare
import Button from '../../../components/Button';
import { validateEmail } from '../../../../utils/validation';
import HomeScreen from '../../screens/HomeScreen';
```

**After Aliases**:

```javascript
// Clean, readable imports
import Button from '@components/Button';
import { validateEmail } from '@utils/validation';
import HomeScreen from '@screens/HomeScreen';
```

**Benefits**:

1. **Readability**
    
    - Clear what you're importing
    - No counting `../../../`
2. **Maintainability**
    
    - Move files without updating imports
    - Refactor folder structure easily
3. **Consistency**
    
    - Same import path regardless of file location
    - Team members use same patterns
4. **Autocomplete**
    
    - IDEs can suggest imports
    - TypeScript knows where types are

**How It Works Under the Hood**:

```javascript
// Your code
import Button from '@components/Button';

// Babel transforms to
import Button from '/absolute/path/to/project/src/components/Button';
```

**TypeScript Integration**: For TypeScript to understand these aliases, `tsconfig.json` must match:

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@components/*": ["src/components/*"],
      "@screens/*": ["src/screens/*"]
    }
  }
}
```

**Important**: Babel and TypeScript configurations must match!

## 3. **Plugin Execution Order**

**Critical Concept**: Plugins run in specific order!

```javascript
plugins: [
  'react-native-reanimated/plugin',  // Runs FIRST (wrong!)
  'module-resolver',                  // Runs SECOND
]
```

**Correct Order** (in our config):

```javascript
plugins: [
  ['module-resolver', { /* config */ }],  // Runs FIRST
  'react-native-reanimated/plugin',       // Runs LAST ✓
]
```

**Why Order Matters**:

1. **Each plugin transforms code**
2. **Next plugin sees transformed code**
3. **Reanimated needs final code form**

**Example**:

```javascript
// Original code
import Button from '@components/Button';
const animated = useAnimatedStyle(() => {});

// After module-resolver
import Button from './src/components/Button';
const animated = useAnimatedStyle(() => {});

// After reanimated plugin
import Button from './src/components/Button';
const animated = __REANIMATED_WORKLET(/* optimized code */);
```

# How Metro & Babel Work Together

## The Build Pipeline

```
1. Metro finds all your files
        ↓
2. Metro reads file contents
        ↓
3. Metro sends to Babel for transformation
        ↓
4. Babel applies presets
        ↓
5. Babel applies plugins (in order)
        ↓
6. Babel returns transformed code
        ↓
7. Metro bundles transformed code
        ↓
8. Metro creates final bundle.js
        ↓
9. App loads bundle
```

## Example Flow

**Your Source File** (`src/screens/HomeScreen.tsx`):

```typescript
import Button from '@components/Button';

const HomeScreen = () => {
  return <Button title="Click me" />;
};
```

**Step 1: Metro finds file**

- Metro scans `src/` directory
- Finds `HomeScreen.tsx`
- Reads file contents

**Step 2: Metro sends to Babel**

- File content sent to Babel
- Babel receives TypeScript code

**Step 3: Babel preset transformation**

- `metro-react-native-babel-preset` runs
- Transforms TypeScript → JavaScript
- Transforms JSX → React.createElement

**Step 4: Module resolver plugin**

```javascript
// Before
import Button from '@components/Button';

// After
import Button from './src/components/Button';
```

**Step 5: Reanimated plugin**

- No changes (no animations in this file)

**Final Output**:

```javascript
var Button = require('./src/components/Button');

var HomeScreen = function HomeScreen() {
  return React.createElement(Button, { title: "Click me" });
};
```

**Step 6: Metro bundles**

- Combines with all other transformed files
- Creates single bundle

# Advanced Configuration Examples

## Example 1: Adding SVG Support

```javascript
// metro.config.js
const { getDefaultConfig } = require('@react-native/metro-config');

const config = {
  transformer: {
    babelTransformerPath: require.resolve('react-native-svg-transformer'),
  },
  resolver: {
    assetExts: getDefaultConfig(__dirname).resolver.assetExts.filter(
      ext => ext !== 'svg'
    ),
    sourceExts: [...getDefaultConfig(__dirname).resolver.sourceExts, 'svg'],
  },
};

module.exports = mergeConfig(getDefaultConfig(__dirname), config);
```

**What this does**:

1. Uses SVG transformer instead of default
2. Removes SVG from assets (not images anymore)
3. Adds SVG to source extensions (now components)

**Result**:

```javascript
import Logo from './logo.svg';
<Logo width={120} height={120} />
```

# Troubleshooting Common Issues

## Issue 1: Module Not Found

**Error**:

```
Unable to resolve module '@components/Button'
```

**Solutions**:

1. Check `babel.config.js` has module-resolver
2. Check `tsconfig.json` paths match
3. Restart Metro bundler: `npm start -- --reset-cache`
4. Clear Babel cache: `rm -rf node_modules/.cache`

## Issue 2: Reanimated Not Working

**Error**:

```
Reanimated 2 failed to create a worklet
```

**Solutions**:

1. Move reanimated/plugin to LAST in plugins array
2. Clear cache and rebuild
3. Check plugin is installed: `npm list babel-plugin-react-native-reanimated`

## Issue 3: SVG Not Working

**Error**:

```
SyntaxError: Unexpected token '<'
```

**Solutions**:

1. Install transformer: `npm install react-native-svg-transformer`
2. Update metro.config.js (see Example 1)
3. Clear cache

## Issue 4: Slow Build Times

**Solutions**:

1. Enable inline requires
2. Reduce watched folders
3. Use `blockList` to exclude unnecessary files
4. Update Metro: `npm install @react-native/metro-config@latest`
