# React Component Lifecycle

### Understanding the Complete Life of a Component — From Birth to Death

---

## What Is a Lifecycle?

Every living thing in this world passes through a predictable sequence of stages — it is born, it grows and changes, and eventually it ceases to exist. A React component follows a remarkably similar journey. It is created and appears on the screen, it updates in response to changing data or user interaction, and it is eventually removed from the screen when it is no longer needed. This journey — from creation to removal — is called the **component lifecycle**.

Understanding the lifecycle is one of the most important conceptual leaps a React developer must make. It answers fundamental questions that every beginner eventually asks: _"Why does my API call fire twice?"_, _"Why does my timer keep running after I navigate away?"_, _"How do I run code only once when the screen loads?"_, and _"How do I clean up resources when a component disappears?"_. All of these questions are, at their core, lifecycle questions.

```
┌─────────────────────────────────────────────────────────────────────┐
│                   The Component Lifecycle Overview                  │
│                                                                     │
│                         ┌─────────────┐                            │
│                         │   MOUNTING  │  ← Component appears       │
│                         │   (Birth)   │    on screen for           │
│                         └──────┬──────┘    the first time          │
│                                │                                   │
│                                ▼                                   │
│                         ┌─────────────┐                            │
│                         │  UPDATING   │  ← Component re-renders    │
│                         │  (Growth)   │    due to state/prop       │
│                         └──────┬──────┘    changes                 │
│                                │           (0 to many times)       │
│                                ▼                                   │
│                         ┌─────────────┐                            │
│                         │ UNMOUNTING  │  ← Component disappears    │
│                         │   (Death)   │    from screen             │
│                         └─────────────┘                            │
│                                                                     │
│  Every component goes through Mounting and Unmounting exactly once. │
│  Updating may happen zero times (static component) or thousands.    │
└─────────────────────────────────────────────────────────────────────┘
```

In the old class-based world of React, these lifecycle stages were managed through explicitly named methods: `componentDidMount`, `componentDidUpdate`, and `componentWillUnmount`. In the modern world of function components, all three stages are handled through a single, unified hook: **`useEffect`**. The behavior of `useEffect` changes depending on how you configure it, and understanding that configuration is the key to mastering the lifecycle.

## Phase 1: Mounting — The Component Is Born

### What Happens During Mounting

Mounting is the process by which a component is created and inserted into the screen for the very first time. When React encounters a component in the render tree that has not existed before, it goes through the following sequence of steps.

First, React calls your component function, executing all of its code from top to bottom. All `useState` calls are initialized with their initial values, all `useRef` calls create their ref objects, and the component returns its JSX — a description of what should appear on screen.

Second, React takes that JSX description and creates the actual native views. On Android, this means creating real `android.view.View` objects, `TextView` objects, and so on — the physical elements that the user will see.

Third, React commits these views to the screen, and the user can now see the component.

Fourth — and this is crucial — React runs the effects. All `useEffect` callbacks are executed **after** the screen has been painted. This sequencing is intentional: effects should never block or delay the initial render.

```
┌────────────────────────────────────────────────────────────────────┐
│                     Mounting Sequence                              │
│                                                                    │
│   Step 1: Component function executes                              │
│           useState(initialValue) → state initialized              │
│           useRef(null) → ref created                               │
│           JSX returned → virtual tree built                        │
│                │                                                   │
│                ▼                                                   │
│   Step 2: React creates native views from JSX                      │
│           <View> → android.view.View                               │
│           <Text> → TextView                                        │
│           <TextInput> → EditText                                   │
│                │                                                   │
│                ▼                                                   │
│   Step 3: Screen is painted — user sees the component             │
│                │                                                   │
│                ▼                                                   │
│   Step 4: useEffect callbacks run (AFTER paint)                    │
│           useEffect(() => { fetchData() }, [])  ← runs now        │
│           useEffect(() => { startTimer() }, []) ← runs now        │
└────────────────────────────────────────────────────────────────────┘
```

### Mounting with useEffect

The mounting phase is captured in `useEffect` by passing an **empty dependency array** `[]`. This tells React: "Run this effect once, and only once — right after the component first appears on screen. Never run it again, no matter how many times this component re-renders."

```jsx
useEffect(() => {
  // This code runs ONCE after the component mounts.
  // Perfect for:
  //   - Initial API calls
  //   - Setting up subscriptions
  //   - Starting timers
  //   - Logging analytics events
  //   - Auto-focusing an input
  console.log('Component mounted!');
}, []); // ← Empty array = run once on mount
```

## Phase 2: Updating — The Component Changes

### What Triggers an Update

Once a component is mounted, it lives on screen and may update many times before it eventually disappears. An update is triggered by one of three things: a change in the component's own **state** (via a `setState` call), a change in the **props** passed to the component by its parent, or a re-render of the **parent component** itself (which causes children to re-render by default, even if their props did not change).

When an update is triggered, React re-executes the component function from top to bottom. The `useState` hooks now return the latest state values rather than the initial ones, the component returns a new JSX description, React compares the new description against the previous one (a process called **reconciliation**), and updates only the native views that actually changed.

```
┌────────────────────────────────────────────────────────────────────┐
│                       Update Sequence                              │
│                                                                    │
│  Trigger: setState called / props changed / parent re-rendered    │
│                │                                                   │
│                ▼                                                   │
│  Component function re-executes                                    │
│  useState returns NEW values                                       │
│  New JSX tree returned                                             │
│                │                                                   │
│                ▼                                                   │
│  Reconciliation: React DIFFS old tree vs new tree                 │
│  Only changed views are updated on screen (efficient!)            │
│                │                                                   │
│                ▼                                                   │
│  Screen updated — user sees the change                            │
│                │                                                   │
│                ▼                                                   │
│  useEffect with changed deps runs its cleanup, then re-runs       │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────┐     │
│  │  What runs after each update?                            │     │
│  │                                                          │     │
│  │  useEffect(() => {...})          → runs after EVERY update│     │
│  │  useEffect(() => {...}, [a, b])  → runs only if a or b   │     │
│  │                                    changed               │     │
│  │  useEffect(() => {...}, [])      → does NOT run on update │     │
│  └──────────────────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────────────────┘
```

### Updating with useEffect

The update phase is captured in `useEffect` by providing **specific dependencies** in the array. React checks each dependency using strict equality (`===`). If any value in the array has changed since the last render, React runs the cleanup function from the previous effect (if one exists), and then runs the effect again.

```jsx
useEffect(() => {
  // This runs on mount AND whenever userId changes.
  // Perfect for:
  //   - Re-fetching data when a filter or ID changes
  //   - Reacting to prop changes
  //   - Syncing external systems with state
  fetchUserProfile(userId);
}, [userId]); // ← Runs when userId changes
```

### The Cleanup Within an Update

When a `useEffect` with dependencies runs again because a dependency changed, React first runs the **cleanup function** from the previous execution before running the effect fresh. This is an important detail that is often misunderstood.

```
┌────────────────────────────────────────────────────────────────────┐
│           Cleanup During Updates — Step by Step                    │
│                                                                    │
│  Initial mount:                                                    │
│  userId = 'user_1'                                                 │
│  → Effect runs: subscribe to user_1's feed                        │
│  → Cleanup stored: () => unsubscribe from user_1's feed           │
│                                                                    │
│  Update: userId changes to 'user_2'                                │
│  → Step 1: OLD cleanup runs first → unsubscribe from user_1       │
│  → Step 2: NEW effect runs → subscribe to user_2's feed           │
│  → New cleanup stored: () => unsubscribe from user_2's feed       │
│                                                                    │
│  Without this cleanup sequence, you would be subscribed to        │
│  BOTH user_1 and user_2 simultaneously — a resource leak!         │
└────────────────────────────────────────────────────────────────────┘
```

## Phase 3: Unmounting — The Component Is Removed

### What Happens During Unmounting

Unmounting occurs when a component is removed from the screen entirely. This happens when the user navigates away from a screen, when a conditional render evaluates to false (`{isVisible && <MyComponent />}`), when a list item is scrolled off screen and virtualized away, or when the parent component itself unmounts.

During unmounting, React removes the native views from the screen, releases the memory they occupied, and — crucially — runs the **cleanup functions** of all active `useEffect` hooks one final time.

This final cleanup run is the most important moment in a component's entire lifecycle. If you have started a timer, opened a WebSocket connection, set up an event listener, or begun any operation that persists beyond the component, this is your one and only opportunity to stop it. Failure to clean up properly leads to **memory leaks** — resources that continue consuming memory and CPU long after the component that created them has disappeared.

### The Unmounting Cleanup

The cleanup function is the function you return from a `useEffect` callback. React calls it when the component unmounts, and also before re-running the effect due to dependency changes (as described in the update section above).

```jsx
useEffect(() => {
  const subscription = eventEmitter.addListener('dataUpdate', handleData);
  const timer = setInterval(pollServer, 5000);

  // This function is called when the component UNMOUNTS.
  // It is also called before re-running this effect.
  return () => {
    subscription.remove();       // stop the event listener
    clearInterval(timer);        // stop the timer
    console.log('Cleaned up!');
  };
}, []);
```

## Class Component Lifecycle vs Function Component Lifecycle

Before hooks existed, React used **class components** with dedicated lifecycle methods. Even though you will write function components in all modern code, you will encounter class component code in older React Native projects, open-source libraries, and legacy codebases. Understanding the mapping between the two paradigms is therefore essential.

```
┌────────────────────────────────────────────────────────────────────┐
│          Class Lifecycle Methods → Function Hook Equivalents       │
├──────────────────────────────┬─────────────────────────────────────┤
│  CLASS COMPONENT             │  FUNCTION COMPONENT                 │
├──────────────────────────────┼─────────────────────────────────────┤
│  constructor(props)          │  useState(initialValue)             │
│  (initializing state)        │  (initial value argument)           │
├──────────────────────────────┼─────────────────────────────────────┤
│  componentDidMount()         │  useEffect(() => {}, [])            │
│  (runs once after mount)     │  (empty dependency array)           │
├──────────────────────────────┼─────────────────────────────────────┤
│  componentDidUpdate(         │  useEffect(() => {}, [dep])         │
│    prevProps, prevState)     │  (specific dependency array)        │
│  (runs after every update)   │                                     │
├──────────────────────────────┼─────────────────────────────────────┤
│  componentWillUnmount()      │  useEffect(() => {             │
│  (cleanup before removal)    │    return () => cleanup()           │
│                              │  }, [])                             │
├──────────────────────────────┼─────────────────────────────────────┤
│  shouldComponentUpdate()     │  React.memo()                       │
│  (prevent unnecessary        │  useMemo() / useCallback()          │
│   re-renders)                │                                     │
├──────────────────────────────┼─────────────────────────────────────┤
│  render()                    │  The function body itself           │
│  (returns JSX)               │  (what the component returns)       │
└──────────────────────────────┴─────────────────────────────────────┘
```

## A Complete Lifecycle Example

The following example demonstrates all three lifecycle phases — mounting, updating, and unmounting — in a single, cohesive component. It also shows what happens when child components are conditionally rendered, giving you direct visibility into when each phase occurs.

```jsx
import { useState, useEffect, useRef } from 'react';
import {
  View, Text, TouchableOpacity, ScrollView, StyleSheet
} from 'react-native';

// ─────────────────────────────────────────────
// A child component that clearly shows its lifecycle
// ─────────────────────────────────────────────
function LifecycleChild({ userId, color }) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const mountTimeRef = useRef(Date.now());

  // ── MOUNTING ──
  // Runs once when this component first appears on screen.
  useEffect(() => {
    const log = `[${color}] MOUNTED for userId: ${userId}`;
    console.log(log);

    // Simulate an API call that starts on mount
    const timer = setTimeout(() => {
      setData({ name: `User ${userId}`, role: 'Developer' });
      setLoading(false);
    }, 1000);

    // ── UNMOUNTING ──
    // This cleanup runs when the component is removed from screen.
    // It cancels the in-flight API simulation to prevent setting
    // state on an unmounted component.
    return () => {
      clearTimeout(timer);
      const aliveFor = ((Date.now() - mountTimeRef.current) / 1000).toFixed(1);
      console.log(`[${color}] UNMOUNTED. Was alive for ${aliveFor}s`);
    };
  }, []); // Empty array = runs once on mount, cleanup on unmount

  // ── UPDATING ──
  // Runs whenever userId changes. Cleans up the previous
  // fetch before starting a new one for the new userId.
  useEffect(() => {
    if (!userId) return;

    setLoading(true);
    setData(null);

    console.log(`[${color}] UPDATING — new userId: ${userId}`);

    const timer = setTimeout(() => {
      setData({ name: `User ${userId}`, role: 'Engineer' });
      setLoading(false);
    }, 800);

    // Cleanup: cancel previous fetch if userId changes again quickly
    return () => {
      clearTimeout(timer);
      console.log(`[${color}] UPDATE CLEANUP — cancelled fetch for userId: ${userId}`);
    };
  }, [userId]); // Runs when userId changes

  return (
    <View style={[styles.child, { borderColor: color }]}>
      <View style={[styles.colorBadge, { backgroundColor: color }]}>
        <Text style={styles.colorBadgeText}>{color} Child</Text>
      </View>
      {loading ? (
        <Text style={styles.loadingText}>⏳ Fetching user {userId}...</Text>
      ) : (
        <View>
          <Text style={styles.dataText}>👤 {data?.name}</Text>
          <Text style={styles.dataSubText}>Role: {data?.role}</Text>
        </View>
      )}
    </View>
  );
}

// ─────────────────────────────────────────────
// The parent component that controls the children
// ─────────────────────────────────────────────
export default function LifecycleDemo() {
  const [showBlue, setShowBlue] = useState(false);
  const [showGreen, setShowGreen] = useState(false);
  const [userId, setUserId] = useState(1);
  const [log, setLog] = useState([]);

  // Override console.log to also show in the on-screen log
  const addLog = (message) => {
    setLog(prev => [
      { id: Date.now() + Math.random(), text: message, time: new Date().toLocaleTimeString() },
      ...prev.slice(0, 14), // keep last 15 entries
    ]);
  };

  // Intercept console.log for display purposes
  useEffect(() => {
    const original = console.log;
    console.log = (...args) => {
      original(...args);
      addLog(args.join(' '));
    };
    return () => { console.log = original; };
  }, []);

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Component Lifecycle Demo</Text>
      <Text style={styles.subtitle}>
        Watch the lifecycle log as components mount, update, and unmount.
      </Text>

      {/* ── Controls ── */}
      <View style={styles.controlsCard}>
        <Text style={styles.controlsTitle}>Controls</Text>

        {/* Mount / Unmount buttons */}
        <View style={styles.controlRow}>
          <Text style={styles.controlLabel}>Blue Child:</Text>
          <TouchableOpacity
            style={[styles.controlBtn, showBlue ? styles.btnDanger : styles.btnSuccess]}
            onPress={() => setShowBlue(v => !v)}
          >
            <Text style={styles.controlBtnText}>
              {showBlue ? 'Unmount ✕' : 'Mount ✓'}
            </Text>
          </TouchableOpacity>
        </View>

        <View style={styles.controlRow}>
          <Text style={styles.controlLabel}>Green Child:</Text>
          <TouchableOpacity
            style={[styles.controlBtn, showGreen ? styles.btnDanger : styles.btnSuccess]}
            onPress={() => setShowGreen(v => !v)}
          >
            <Text style={styles.controlBtnText}>
              {showGreen ? 'Unmount ✕' : 'Mount ✓'}
            </Text>
          </TouchableOpacity>
        </View>

        {/* userId update buttons */}
        <View style={styles.controlRow}>
          <Text style={styles.controlLabel}>User ID ({userId}):</Text>
          <View style={styles.idBtns}>
            {[1, 2, 3, 4].map(id => (
              <TouchableOpacity
                key={id}
                style={[styles.idBtn, userId === id && styles.idBtnActive]}
                onPress={() => setUserId(id)}
              >
                <Text style={[styles.idBtnText, userId === id && styles.idBtnTextActive]}>
                  {id}
                </Text>
              </TouchableOpacity>
            ))}
          </View>
        </View>
      </View>

      {/* ── Component Stage ── */}
      <View style={styles.stage}>
        {/* Blue child — conditional render demonstrates mount/unmount */}
        {showBlue && (
          <LifecycleChild userId={userId} color="steelblue" />
        )}

        {/* Green child — mounts/unmounts independently */}
        {showGreen && (
          <LifecycleChild userId={userId} color="seagreen" />
        )}

        {!showBlue && !showGreen && (
          <Text style={styles.stagePlaceholder}>
            No components mounted yet.{'\n'}Tap "Mount ✓" above to begin.
          </Text>
        )}
      </View>

      {/* ── Lifecycle Log ── */}
      <View style={styles.logCard}>
        <View style={styles.logHeader}>
          <Text style={styles.logTitle}>📋 Lifecycle Log</Text>
          <TouchableOpacity onPress={() => setLog([])}>
            <Text style={styles.clearBtn}>Clear</Text>
          </TouchableOpacity>
        </View>
        <ScrollView style={styles.logScroll} nestedScrollEnabled>
          {log.length === 0 ? (
            <Text style={styles.logEmpty}>No events yet. Mount a component above.</Text>
          ) : (
            log.map(entry => (
              <View key={entry.id} style={styles.logEntry}>
                <Text style={styles.logTime}>{entry.time}</Text>
                <Text style={styles.logText}>{entry.text}</Text>
              </View>
            ))
          )}
        </ScrollView>
      </View>

    </View>
  );
}
```

## The Most Common Lifecycle Mistakes

### Mistake 1: Missing the Cleanup Function

The most dangerous mistake a React developer can make is starting a persistent operation — a timer, a subscription, an event listener — inside `useEffect` and forgetting to stop it in the cleanup function. When the component unmounts, that operation continues running in the background. If the callback tries to update state, React will throw a warning: _"Can't perform a React state update on an unmounted component."_

```jsx
// ❌ WRONG — timer runs forever after component unmounts
useEffect(() => {
  const timer = setInterval(() => {
    setCount(c => c + 1);
  }, 1000);
  // No cleanup! Timer leaks after unmount.
}, []);

// ✅ CORRECT — timer is cleared when component unmounts
useEffect(() => {
  const timer = setInterval(() => {
    setCount(c => c + 1);
  }, 1000);
  return () => clearInterval(timer); // cleanup
}, []);
```

### Mistake 2: Incorrect Dependency Array

Specifying the wrong dependencies — or omitting dependencies — is the second most common lifecycle mistake. If you use a value inside `useEffect` but do not list it in the dependency array, your effect will run with a stale (outdated) copy of that value. This is called a **stale closure** and produces bugs that are notoriously difficult to diagnose.

```jsx
// ❌ WRONG — userId is used but not listed as a dependency
// Effect will always run with the initial userId value, even after it changes
useEffect(() => {
  fetchUser(userId); // this will use the stale, initial userId
}, []); // userId missing from deps!

// ✅ CORRECT — userId is listed, so effect re-runs when it changes
useEffect(() => {
  fetchUser(userId);
}, [userId]);
```

### Mistake 3: Putting Everything in One useEffect

A single component can have as many `useEffect` calls as needed. It is much cleaner and more maintainable to separate unrelated concerns into separate effects than to bundle them all into one.

```jsx
// ❌ WRONG — one effect doing three unrelated things
useEffect(() => {
  fetchUserData();
  startHeartbeatTimer();
  logPageView();
}, []);

// ✅ CORRECT — each concern is isolated
useEffect(() => { fetchUserData(); }, []);
useEffect(() => {
  const timer = setInterval(ping, 30000);
  return () => clearInterval(timer);
}, []);
useEffect(() => { logPageView(screenName); }, [screenName]);
```

## Lifecycle Summary

```
┌────────────────────────────────────────────────────────────────────┐
│                    useEffect Cheat Sheet                           │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  MOUNT only (run once):                                            │
│  useEffect(() => { setup() }, [])                                  │
│                                                                    │
│  MOUNT + specific UPDATE (run when dep changes):                   │
│  useEffect(() => { doWork(dep) }, [dep])                           │
│                                                                    │
│  EVERY render (use sparingly):                                     │
│  useEffect(() => { doWork() })                                     │
│                                                                    │
│  UNMOUNT cleanup (always pair with setup):                         │
│  useEffect(() => {                                                 │
│    const resource = startSomething()                               │
│    return () => stopSomething(resource)  ← cleanup                │
│  }, [])                                                            │
│                                                                    │
│  UPDATE cleanup (cleanup before re-running):                       │
│  useEffect(() => {                                                 │
│    const sub = subscribe(channel)                                  │
│    return () => unsubscribe(sub)  ← runs before next effect       │
│  }, [channel])                                                     │
│                                                                    │
├────────────────────────────────────────────────────────────────────┤
│  Lifecycle Phase  │  Trigger             │  useEffect config       │
├───────────────────┼──────────────────────┼─────────────────────────┤
│  Mounting         │  First render        │  deps = []              │
│  Updating         │  State/prop change   │  deps = [value]         │
│  Unmounting       │  Component removed   │  return () => cleanup() │
└───────────────────┴──────────────────────┴─────────────────────────┘
```

---

_The lifecycle is not merely an academic concept — it is the operating manual for every component you will ever write. When your app behaves unexpectedly, the answer almost always lies somewhere in the lifecycle: an effect that runs too often, a cleanup that was forgotten, or a dependency that was missed. Master the lifecycle, and you master React._
