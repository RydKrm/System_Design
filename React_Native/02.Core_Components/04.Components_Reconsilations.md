## The Problem React Had to Solve

Whenever anything changes in your application — a button is pressed, data arrives from an API, a timer fires — React needs to update what is displayed on screen. The most naive approach would be to tear down the entire screen and rebuild it from scratch on every change. This would be catastrophically slow. Destroying and recreating thousands of native views sixty times per second would make any application completely unusable.

At the other extreme, React could try to update every single property of every single element on every render. This too would be unnecessarily slow and complex.

React's solution is a middle path — an intelligent algorithm that **computes the minimum set of changes** necessary to bring the screen from its current state to its desired state. This algorithm is called **Reconciliation**.

---

## What Is Reconciliation?

Reconciliation is the process by which React compares the **previous virtual tree** of components with the **new virtual tree** produced by the latest render, and determines precisely which parts of the real native UI need to be updated. The word "reconcile" means to bring two things into agreement — and that is exactly what React does: it reconciles the current state of the screen with what the screen should look like after a change.

The central tool React uses for this comparison is a technique called **diffing**. To understand diffing, you must first understand the virtual tree.

```
┌──────────────────────────────────────────────────────────────────────┐
│                    The Virtual Tree Concept                          │
│                                                                      │
│  Your JSX:                     Virtual Tree React builds:            │
│                                                                      │
│  <View>                        View                                  │
│    <Text>Hello</Text>            └── Text ("Hello")                  │
│    <View>                        └── View                            │
│      <Text>World</Text>               └── Text ("World")             │
│    </View>                                                           │
│  </View>                                                             │
│                                                                      │
│  This virtual tree is a plain JavaScript object — not real native    │
│  views. It is lightweight, fast to create, and fast to compare.      │
│                                                                      │
│  React keeps TWO copies of this tree:                                │
│  ┌────────────────────┐     ┌────────────────────┐                   │
│  │  Previous Tree     │     │  New Tree           │                  │
│  │  (what's on screen)│     │  (what should be)   │                  │
│  └─────────┬──────────┘     └──────────┬──────────┘                  │
│            └──────────┬─────────────────┘                            │
│                       │                                              │
│                       ▼                                              │
│              DIFFING ALGORITHM                                       │
│         Find minimum set of changes                                  │
│                       │                                              │
│                       ▼                                              │
│              Apply only those changes                                │
│              to the real native views                                │
└──────────────────────────────────────────────────────────────────────┘
```

## The Diffing Algorithm — Three Key Rules

React's diffing algorithm operates on a set of assumptions and rules that make it efficient enough to run in real time, even for complex UIs.

### Rule 1: Different Element Types Produce Different Trees

If the **type** of an element changes between renders — say, a `View` becomes a `ScrollView`, or a `Text` becomes a `TouchableOpacity` — React does not attempt to update the element in place. It **destroys the old element and all of its children entirely**, and creates the new element and all of its children from scratch. This is because React assumes that if the type changed, the structure below it is fundamentally different and not worth trying to reconcile.

This rule has an important implication: avoid placing conditionally-rendered elements that change type at the same position in the tree, as it will cause unnecessary destruction and recreation of their entire subtrees.

```
┌──────────────────────────────────────────────────────────────────────┐
│              Rule 1: Type Change = Destroy and Rebuild               │
│                                                                      │
│  Before update:              After update:                           │
│                                                                      │
│  <View>              ───►   <ScrollView>                             │
│    <Text>Hello</Text>  ✕      <Text>Hello</Text>    ← recreated      │
│    <Text>World</Text>  ✕      <Text>World</Text>    ← recreated      │
│  </View>                    </ScrollView>                            │
│                                                                      │
│  React destroys the View and ALL children, then builds fresh.        │
│  All child state is LOST. All child effects are cleaned up.          │
└──────────────────────────────────────────────────────────────────────┘
```

### Rule 2: Same Element Type = Update in Place

If the element type remains the same between renders, React keeps the existing native view and only **updates the properties that changed**. If a `View` had a background color of red and now has a background color of blue, React sends a single update to change only the background color. The view itself, all its children, and all their state are preserved.

```
┌──────────────────────────────────────────────────────────────────────┐
│             Rule 2: Same Type = Update Properties In Place           │
│                                                                      │
│  Before:                         After:                              │
│  <View                           <View                               │
│    backgroundColor="red"   ───►    backgroundColor="blue"  ← update │
│    padding={10}             ───►    padding={10}            ← same   │
│    opacity={1}              ───►    opacity={0.5}           ← update │
│  >                                >                                  │
│                                                                      │
│  React sends 2 property updates to the native layer.               │
│  The View is NOT destroyed. Children are preserved.                  │
└──────────────────────────────────────────────────────────────────────┘
```

### Rule 3: Keys Help React Track List Items

When rendering lists, React needs a way to identify which items have been added, removed, or reordered between renders. Without any additional information, React compares list items by their position — the first item of the new list with the first item of the old list, and so on. This positional comparison fails catastrophically when items are inserted at the beginning or reordered.

The **`key` prop** is React's solution. When you provide a stable, unique `key` on each item, React uses those keys to match old items with new items regardless of their position in the list. If an item's key appears in both the old and new tree, React knows it is the same item and updates it in place. If a key disappears, React knows that item was removed. If a new key appears, React knows a new item was added.

```
┌──────────────────────────────────────────────────────────────────────┐
│                  Rule 3: Keys in List Reconciliation                 │
│                                                                      │
│  WITHOUT KEYS — positional comparison:                               │
│                                                                      │
│  Old list:  [Apple, Banana, Cherry]                                  │
│  New list:  [Mango, Apple, Banana, Cherry]  ← item inserted at start│
│                                                                      │
│  React compares by position:                                         │
│  Position 0: Apple vs Mango   → UPDATE (unnecessary)                │
│  Position 1: Banana vs Apple  → UPDATE (unnecessary)                │
│  Position 2: Cherry vs Banana → UPDATE (unnecessary)                │
│  Position 3: (nothing) vs Cherry → CREATE                           │
│  Result: 3 unnecessary updates + 1 create = INEFFICIENT ❌          │
│                                                                      │
│  WITH KEYS — identity-based comparison:                              │
│                                                                      │
│  React matches by key:                                               │
│  key="mango"  → not in old list → CREATE (1 new item) ✅            │
│  key="apple"  → exists in old list → UPDATE in place ✅             │
│  key="banana" → exists in old list → UPDATE in place ✅             │
│  key="cherry" → exists in old list → UPDATE in place ✅             │
│  Result: 1 create, 0 unnecessary updates = EFFICIENT ✅             │
└──────────────────────────────────────────────────────────────────────┘
```

## The Fiber Architecture — React's Reconciliation Engine

In React 16, the internal reconciliation engine was completely rewritten and named **Fiber**. Understanding Fiber is not required for day-to-day React development, but it explains some important behaviors you will encounter.

Before Fiber, reconciliation was synchronous and uninterruptible. Once React started diffing a large tree, it had to finish the entire process before yielding control back to the browser or the native thread. If reconciliation took 200 milliseconds, the UI would freeze for that entire duration.

Fiber reimagined reconciliation as a series of small, interruptible units of work. React can now pause reconciliation in the middle, yield control to handle a more urgent task (like responding to a user gesture), and then resume where it left off. This is what enables React's **Concurrent Features** — features like `Suspense`, `useTransition`, and `useDeferredValue` that allow React to prioritize urgent updates over non-urgent ones.

```
┌───────────────────────────────────────────────────────────────────────┐
│                Old vs New Reconciliation (Fiber)                     │
│                                                                       │
│  OLD (Stack Reconciler — before React 16):                           │
│                                                                       │
│  Start ──────────────────────────────────────────► Done             │
│         [can't stop, can't pause, must finish]                       │
│         UI freezes during long reconciliations ❌                    │
│                                                                       │
│  NEW (Fiber Reconciler — React 16+):                                 │
│                                                                       │
│  Start ──► Work ──► Pause ──► Handle gesture ──► Resume ──► Done    │
│                    (yield)    (higher priority)   (continue)         │
│                                                                       │
│  React breaks reconciliation into small "fibers" of work            │
│  and processes them using priority scheduling.                       │
│                                                                       │
│  HIGH PRIORITY:  User input, animations, gestures                   │
│  NORMAL PRIORITY: Data fetching results, state updates              │
│  LOW PRIORITY:  Off-screen content, deferred updates                │
└───────────────────────────────────────────────────────────────────────┘
```

## Reconciliation in Practice — What Developers Must Know

Understanding reconciliation directly informs several important development practices. The following principles emerge directly from how the reconciliation algorithm works.

**Never generate keys from array indices for dynamic lists.** Using an array index as a key is a common but damaging mistake. If items are reordered or inserted, the indices of existing items change, and React incorrectly treats them as new items. Always use a stable, unique identifier — a database ID, a UUID, or any property that does not change when the list changes.

jsx

```jsx
// ❌ WRONG — index as key
{items.map((item, index) => (
  <ItemCard key={index} item={item} />  // breaks when items reorder
))}

// ✅ CORRECT — stable unique ID as key
{items.map((item) => (
  <ItemCard key={item.id} item={item} />  // React tracks identity correctly
))}
```

**Never define components inside other components.** If you define a component function inside a parent component's render function, React sees it as a new component type on every render (because it is literally a new function reference each time). This causes React to destroy and recreate the child on every single parent render, losing all child state and running all child effects from scratch.

jsx

```jsx
// ❌ WRONG — new component type every render → full destroy/recreate
function ParentComponent() {
  function InlineCard({ item }) {  // new function every render!
    return <View>...</View>;
  }
  return <InlineCard item={data} />;
}

// ✅ CORRECT — stable component type → React updates in place
function ItemCard({ item }) {  // defined outside — stable reference
  return <View>...</View>;
}
function ParentComponent() {
  return <ItemCard item={data} />;
}
```

**Use keys to deliberately reset component state.** Since changing a component's key causes React to destroy it and create a fresh one, you can intentionally use this behavior to reset a component's state. If you want a form to reset completely when a user selects a different record, simply change the key — React will unmount the old form and mount a fresh one with clean state.

jsx

```jsx
// Changing the key forces a full remount — all state resets to initial
<ProfileForm key={selectedUserId} userId={selectedUserId} />
```

## A Visual Reconciliation Example

jsx

```jsx
import { useState } from 'react';
import { View, Text, TouchableOpacity, FlatList, StyleSheet } from 'react-native';

const INITIAL_ITEMS = [
  { id: 'a', label: 'Item Alpha', color: '#FF6B6B' },
  { id: 'b', label: 'Item Beta', color: '#4ECDC4' },
  { id: 'c', label: 'Item Gamma', color: '#45B7D1' },
  { id: 'd', label: 'Item Delta', color: '#96CEB4' },
];

function ListCard({ item }) {
  // This counter persists as long as React considers this the SAME component
  // (i.e., same key). If the key changes, this state resets to 0.
  const [taps, setTaps] = useState(0);

  return (
    <TouchableOpacity
      style={[styles.card, { borderLeftColor: item.color }]}
      onPress={() => setTaps(t => t + 1)}
    >
      <View style={{ flex: 1 }}>
        <Text style={styles.cardLabel}>{item.label}</Text>
        <Text style={styles.cardId}>key="{item.id}"</Text>
      </View>
      <View style={[styles.tapBadge, { backgroundColor: item.color }]}>
        <Text style={styles.tapCount}>{taps}</Text>
        <Text style={styles.tapLabel}>taps</Text>
      </View>
    </TouchableOpacity>
  );
}

export default function ReconciliationDemo() {
  const [items, setItems] = useState(INITIAL_ITEMS);
  const [useIndex, setUseIndex] = useState(false);

  const addToFront = () => {
    const newItem = {
      id: `new_${Date.now()}`,
      label: `New Item ${Math.floor(Math.random() * 100)}`,
      color: '#9B59B6',
    };
    setItems(prev => [newItem, ...prev]);
  };

  const shuffle = () => {
    setItems(prev => [...prev].sort(() => Math.random() - 0.5));
  };

  const removeFirst = () => {
    setItems(prev => prev.slice(1));
  };

  const reset = () => setItems(INITIAL_ITEMS);

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Reconciliation & Keys Demo</Text>
      <Text style={styles.subtitle}>
        Tap cards to increment their counters. Then add/shuffle/remove items.
        With stable keys, tap counts survive reorders. With index keys, they scramble.
      </Text>

      {/* Key strategy toggle */}
      <View style={styles.toggleRow}>
        <TouchableOpacity
          style={[styles.toggleBtn, !useIndex && styles.toggleActive]}
          onPress={() => setUseIndex(false)}
        >
          <Text style={[styles.toggleText, !useIndex && styles.toggleTextActive]}>
            ✅ Stable ID Keys
          </Text>
        </TouchableOpacity>
        <TouchableOpacity
          style={[styles.toggleBtn, useIndex && styles.toggleActive]}
          onPress={() => setUseIndex(true)}
        >
          <Text style={[styles.toggleText, useIndex && styles.toggleTextActive]}>
            ❌ Index Keys
          </Text>
        </TouchableOpacity>
      </View>

      {useIndex && (
        <View style={styles.warningBanner}>
          <Text style={styles.warningText}>
            ⚠ Using index as key. Reorder or add items and watch tap counts mismatch!
          </Text>
        </View>
      )}

      {/* Action buttons */}
      <View style={styles.actions}>
        <TouchableOpacity style={styles.actionBtn} onPress={addToFront}>
          <Text style={styles.actionText}>+ Front</Text>
        </TouchableOpacity>
        <TouchableOpacity style={styles.actionBtn} onPress={shuffle}>
          <Text style={styles.actionText}>⇄ Shuffle</Text>
        </TouchableOpacity>
        <TouchableOpacity style={styles.actionBtn} onPress={removeFirst}>
          <Text style={styles.actionText}>✕ Remove</Text>
        </TouchableOpacity>
        <TouchableOpacity style={[styles.actionBtn, styles.resetBtn]} onPress={reset}>
          <Text style={[styles.actionText, { color: '#fff' }]}>↺ Reset</Text>
        </TouchableOpacity>
      </View>

      {/* The list — key strategy controlled by toggle */}
      <FlatList
        data={items}
        keyExtractor={(item, index) =>
          useIndex ? String(index) : item.id  // switch between strategies
        }
        renderItem={({ item }) => <ListCard item={item} />}
      />

      {/* Explanation */}
      <View style={styles.explanationCard}>
        <Text style={styles.explanationTitle}>What React sees:</Text>
        <Text style={styles.explanationText}>
          {useIndex
            ? 'With index keys: when you add to front, item at index 0 was "Alpha" (3 taps), now it\'s "New Item". React thinks it\'s the SAME component — tap count transfers incorrectly!'
            : 'With ID keys: each item has identity "a", "b", "c". Reordering doesn\'t confuse React — it tracks each card by its ID, not its position. Tap counts stay correct!'}
        </Text>
      </View>
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, backgroundColor: '#f2f2f7', padding: 14 },
  title: { fontSize: 20, fontWeight: '800', color: '#1a1a2e', marginBottom: 6 },
  subtitle: { fontSize: 12, color: '#777', lineHeight: 17, marginBottom: 14 },
  toggleRow: { flexDirection: 'row', gap: 8, marginBottom: 10 },
  toggleBtn: {
    flex: 1, paddingVertical: 9, borderRadius: 8, alignItems: 'center',
    backgroundColor: '#fff', borderWidth: 1.5, borderColor: '#ddd',
  },
  toggleActive: { backgroundColor: '#1a1a2e', borderColor: '#1a1a2e' },
  toggleText: { fontSize: 13, color: '#555', fontWeight: '600' },
  toggleTextActive: { color: '#fff' },
  warningBanner: {
    backgroundColor: '#fff3cd', borderRadius: 8, padding: 10, marginBottom: 10,
  },
  warningText: { fontSize: 12, color: '#856404', lineHeight: 17 },
  actions: { flexDirection: 'row', gap: 6, marginBottom: 12 },
  actionBtn: {
    flex: 1, paddingVertical: 8, borderRadius: 8, alignItems: 'center',
    backgroundColor: '#fff', borderWidth: 1.5, borderColor: '#ddd',
  },
  resetBtn: { backgroundColor: '#6C63FF', borderColor: '#6C63FF' },
  actionText: { fontSize: 12, fontWeight: '700', color: '#444' },
  card: {
    backgroundColor: '#fff', borderRadius: 10, padding: 14,
    marginBottom: 8, flexDirection: 'row', alignItems: 'center',
    elevation: 1, borderLeftWidth: 4,
  },
  cardLabel: { fontSize: 15, fontWeight: '600', color: '#1a1a1a' },
  cardId: { fontSize: 11, color: '#aaa', marginTop: 3, fontFamily: 'monospace' },
  tapBadge: {
    width: 48, height: 48, borderRadius: 24,
    alignItems: 'center', justifyContent: 'center',
  },
  tapCount: { fontSize: 18, fontWeight: '800', color: '#fff' },

  explanationCard: {
    backgroundColor: '#e8f4fd', borderRadius: 10, padding: 14, marginTop: 10,
  },
  explanationTitle: { fontSize: 13, fontWeight: '700', color: '#2980b9', marginBottom: 6 },
  explanationText: { fontSize: 12, color: '#444', lineHeight: 18 },
});
```

## Summary

```
┌────────────────────────────────────────────────────────────────────────┐
│                CONTROLLED vs UNCONTROLLED — Quick Rules               │
├────────────────────────────────────────────────────────────────────────┤
│  Use CONTROLLED when:                                                  │
│  • You need real-time validation as the user types                    │
│  • The input's value drives other UI decisions                        │
│  • You need to transform the input (format phone, auto-capitalize)    │
│  • You want to disable/enable the submit button dynamically           │
│  • Multiple inputs must stay in sync                                  │
│  • This is the default. Use it for everything in production.          │
│                                                                        │
│  Use UNCONTROLLED when:                                               │
│  • You only need the value at submission, not in between             │
│  • Integrating with a library that manages its own state             │
│  • A critical performance optimization is needed (rare)               │
└────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────┐
│                  RECONCILIATION — Core Rules to Remember              │
├────────────────────────────────────────────────────────────────────────┤
│  1. Same element type → React updates in place (fast)                 │
│  2. Different element type → React destroys and rebuilds (slow)       │
│  3. Keys give list items stable identity across reorders              │
│  4. NEVER use array index as key for dynamic/reorderable lists        │
│  5. NEVER define components inside other components                   │
│  6. Changing a key deliberately forces a full remount (useful trick)  │
│  7. Fiber makes reconciliation interruptible — high-priority updates  │
│     (gestures, input) are never blocked by low-priority work          │
└────────────────────────────────────────────────────────────────────────┘
```
