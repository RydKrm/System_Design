## The Problem — Prop Drilling

Imagine a large office building. The CEO on the top floor needs to send a memo to an employee on the ground floor. In a well-organized building, the CEO hands it to a manager, who hands it to a supervisor, who hands it to a team lead, who finally hands it to the employee. Every intermediate person had to handle the memo even though it was never meant for them.

In React, this exact phenomenon exists and is called **prop drilling** — the practice of passing props down through multiple layers of components simply to get data from a parent at the top of the tree to a deeply nested child at the bottom, even though all the intermediate components have no use for that data themselves.

```
┌────────────────────────────────────────────────────────────────────────┐
│                    The Prop Drilling Problem                           │
│                                                                        │
│  App (has user data)                                                   │
│   └── passes user={user} ↓                                            │
│       Dashboard (doesn't need user)                                   │
│        └── passes user={user} ↓                                       │
│            Layout (doesn't need user)                                 │
│             └── passes user={user} ↓                                  │
│                 Sidebar (doesn't need user)                           │
│                  └── passes user={user} ↓                             │
│                      UserAvatar (FINALLY needs user) ✓                │
│                                                                        │
│  Dashboard, Layout, and Sidebar are all POLLUTED with a prop          │
│  they do not care about, just to pass it through.                     │
│                                                                        │
│  Problems:                                                             │
│  ─ Every intermediate component must be modified                     │
│  ─ Deeply nested components are tightly coupled to the top           │
│  ─ Renaming a prop means changing every file in the chain            │
│  ─ Code becomes increasingly hard to follow and maintain             │
└────────────────────────────────────────────────────────────────────────┘
```

As applications grow in complexity, prop drilling becomes one of the most painful maintenance burdens in React development. Adding a new piece of global data — a theme preference, user authentication status, language setting — means threading a new prop through potentially dozens of components.

---

## The Solution — React Context

The Context API is React's built-in solution to prop drilling. It creates a **global data channel** — a kind of broadcast system — that any component in the tree can subscribe to, regardless of how deeply nested it is. The top-level component broadcasts data into the channel, and any descendant can tune into that channel and receive the data directly, without any intermediate components needing to participate.

```
┌────────────────────────────────────────────────────────────────────────┐
│                    Context API — The Solution                          │
│                                                                        │
│  App (creates Context and provides user data)                         │
│   │                                                                   │
│   │  ┌─────────────────────────────────────────┐                     │
│   │  │           Context Channel               │                     │
│   │  │         { user, theme, lang }           │                     │
│   │  └─────────────────────────────────────────┘                     │
│   │            ↑ broadcast                │ any descendant            │
│   │                                       │ can tune in              │
│   └── Dashboard (ignores context)         │                           │
│        └── Layout (ignores context)       │                           │
│             └── Sidebar (ignores context) │                           │
│                  └── UserAvatar ◄─────────┘                           │
│                      (reads user directly from context)               │
│                                                                        │
│  Dashboard, Layout, Sidebar: completely unaware of user data ✅       │
│  UserAvatar: reads user from context directly ✅                      │
└────────────────────────────────────────────────────────────────────────┘
```

## How Context Works — Three Essential Parts

The Context API is composed of three distinct pieces that work together. Understanding each piece separately is the key to using Context confidently.

### Part 1: Creating a Context

You create a context with `React.createContext()`. This function takes an optional **default value** — the value that will be provided to any consumer that is not wrapped in a Provider. In practice, the default value is rarely used directly, because you almost always wrap your app in a Provider. However, it serves as a useful fallback and as documentation of what shape the context data takes.

jsx

```jsx
import { createContext } from 'react';

// Creates a context with a default value of null
// The default only applies to consumers outside any Provider
const ThemeContext = createContext(null);

// Convention: export the context so components can import it
export default ThemeContext;
```

### Part 2: The Provider — Broadcasting Data

The **Provider** is a special component that React gives you on every context object (`ThemeContext.Provider`). You wrap the section of your component tree that should have access to the context data inside this Provider. The Provider accepts a `value` prop — whatever you pass as `value` is what every consumer below it in the tree will receive.

Any component nested inside the Provider — no matter how deeply — can access this value. Components outside the Provider cannot.

```
┌────────────────────────────────────────────────────────────────────────┐
│                    Provider Scope                                      │
│                                                                        │
│  <ThemeContext.Provider value={theme}>                                 │
│                                                                        │
│      All components here ───► CAN read theme from context             │
│                                                                        │
│  </ThemeContext.Provider>                                              │
│                                                                        │
│  Components outside ────────► CANNOT read theme from this Provider    │
│  (they get the default value from createContext)                      │
└────────────────────────────────────────────────────────────────────────┘
```

### Part 3: The Consumer — `useContext`

Any component that wants to read from a context uses the `useContext` hook. You pass the context object to `useContext`, and it returns the current value provided by the nearest Provider above it in the tree.

An important and critical performance note: **whenever the Provider's `value` changes, every component that calls `useContext` with that context will re-render**. This is by design — it ensures all consumers stay in sync. But it also means you must be thoughtful about what you put in context and how you structure the `value` object, which we will address in the advanced patterns section.

jsx

```jsx
import { useContext } from 'react';
import ThemeContext from './ThemeContext';

function UserAvatar() {
  // Reads directly from context — no props needed from parent!
  const theme = useContext(ThemeContext);

  return (
    <View style={{ backgroundColor: theme.colors.card }}>
      <Text style={{ color: theme.colors.text }}>Alice</Text>
    </View>
  );
}
```

## A Complete Context Example — Theme + Auth System

The following example demonstrates a production-quality pattern for managing two independent contexts — a theme context and an authentication context — that many components need without prop drilling.

jsx

```jsx
import { createContext, useContext, useState, useCallback, useMemo } from 'react';
import {
  View, Text, TouchableOpacity, ScrollView, Switch, StyleSheet
} from 'react-native';

// ═══════════════════════════════════════════════
// 1. CREATE CONTEXTS
// ═══════════════════════════════════════════════

const ThemeContext = createContext(null);
const AuthContext = createContext(null);

// ═══════════════════════════════════════════════
// 2. CREATE PROVIDERS WITH ALL RELATED LOGIC
// ═══════════════════════════════════════════════

const THEMES = {
  light: {
    name: 'Light',
    bg: '#f2f2f7',
    card: '#ffffff',
    text: '#1a1a2e',
    subtext: '#666',
    accent: '#6C63FF',
    border: '#e0e0e0',
    navBg: '#ffffff',
  },
  dark: {
    name: 'Dark',
    bg: '#0d0d1a',
    card: '#1a1a2e',
    text: '#ffffff',
    subtext: '#aaa',
    accent: '#8B80FF',
    border: '#2a2a3e',
    navBg: '#111120',
  },
  ocean: {
    name: 'Ocean',
    bg: '#0a2a3a',
    card: '#0d3347',
    text: '#e0f4ff',
    subtext: '#7ab8d4',
    accent: '#00BCD4',
    border: '#1a4a60',
    navBg: '#081f2e',
  },
};

function ThemeProvider({ children }) {
  const [themeName, setThemeName] = useState('light');

  // useMemo prevents creating a new object on every render
  // which would cause ALL consumers to re-render unnecessarily
  const value = useMemo(() => ({
    theme: THEMES[themeName],
    themeName,
    setTheme: setThemeName,
    availableThemes: Object.keys(THEMES),
  }), [themeName]);

  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  const login = useCallback((credentials) => {
    // Simulate login — in a real app this would call an API
    setUser({
      id: 'usr_001',
      name: credentials.name || 'Jane Developer',
      email: credentials.email || 'jane@example.com',
      role: 'Senior Engineer',
      avatar: '👩‍💻',
      permissions: ['read', 'write', 'admin'],
    });
  }, []);

  const logout = useCallback(() => {
    setUser(null);
  }, []);

  const hasPermission = useCallback((permission) => {
    return user?.permissions?.includes(permission) ?? false;
  }, [user]);

  const value = useMemo(() => ({
    user,
    isAuthenticated: !!user,
    login,
    logout,
    hasPermission,
  }), [user, login, logout, hasPermission]);

  return (
    <AuthContext.Provider value={value}>
      {children}
    </AuthContext.Provider>
  );
}

// ═══════════════════════════════════════════════
// 3. CUSTOM HOOKS — Clean, readable consumer API
// ═══════════════════════════════════════════════

// Instead of calling useContext(ThemeContext) everywhere,
// we create a named hook that reads more clearly
function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error('useTheme must be used inside ThemeProvider');
  }
  return context;
}

function useAuth() {
  const context = useContext(AuthContext);
  if (!context) {
    throw new Error('useAuth must be used inside AuthProvider');
  }
  return context;
}

// ═══════════════════════════════════════════════
// 4. DEEPLY NESTED COMPONENTS — no prop drilling!
// ═══════════════════════════════════════════════

// This component is deeply nested (App → Screen → Content → Card → UserBadge)
// but reads from BOTH contexts directly. No parent passed anything to it.
function UserBadge() {
  const { theme } = useTheme();
  const { user, isAuthenticated } = useAuth();

  if (!isAuthenticated) {
    return (
      <View style={[styles.badge, { backgroundColor: theme.border }]}>
        <Text style={[styles.badgeText, { color: theme.subtext }]}>
          🔒 Not signed in
        </Text>
      </View>
    );
  }

  return (
    <View style={[styles.badge, { backgroundColor: theme.accent + '20', borderColor: theme.accent }]}>
      <Text style={styles.badgeAvatar}>{user.avatar}</Text>
      <View>
        <Text style={[styles.badgeName, { color: theme.text }]}>{user.name}</Text>
        <Text style={[styles.badgeRole, { color: theme.subtext }]}>{user.role}</Text>
      </View>
    </View>
  );
}

// Permission-gated button — reads auth context to show/hide based on permissions
function AdminPanel() {
  const { theme } = useTheme();
  const { hasPermission, isAuthenticated } = useAuth();

  if (!isAuthenticated) return null;

  return (
    <View style={[styles.adminPanel, { backgroundColor: theme.card, borderColor: theme.border }]}>
      <Text style={[styles.adminTitle, { color: theme.text }]}>Admin Panel</Text>
      {['read', 'write', 'admin'].map(perm => (
        <View key={perm} style={styles.permRow}>
          <Text style={[styles.permName, { color: theme.subtext }]}>{perm}</Text>
          <Text style={{ fontSize: 16 }}>
            {hasPermission(perm) ? '✅' : '❌'}
          </Text>
        </View>
      ))}
    </View>
  );
}

// Theme switcher — reads and writes to theme context
function ThemeSwitcher() {
  const { theme, themeName, setTheme, availableThemes } = useTheme();

  return (
    <View style={[styles.card, { backgroundColor: theme.card, borderColor: theme.border }]}>
      <Text style={[styles.cardTitle, { color: theme.text }]}>Appearance</Text>
      <View style={styles.themeRow}>
        {availableThemes.map(name => (
          <TouchableOpacity
            key={name}
            style={[
              styles.themeChip,
              { borderColor: THEMES[name].accent },
              themeName === name && { backgroundColor: THEMES[name].accent },
            ]}
            onPress={() => setTheme(name)}
          >
            <Text style={[
              styles.themeChipText,
              { color: themeName === name ? '#fff' : THEMES[name].accent },
            ]}>
              {THEMES[name].name}
            </Text>
          </TouchableOpacity>
        ))}
      </View>
    </View>
  );
}

// Login form — writes to auth context
function LoginCard() {
  const { theme } = useTheme();
  const { login, logout, isAuthenticated, user } = useAuth();

  if (isAuthenticated) {
    return (
      <View style={[styles.card, { backgroundColor: theme.card, borderColor: theme.border }]}>
        <Text style={[styles.cardTitle, { color: theme.text }]}>Account</Text>
        <UserBadge />
        <TouchableOpacity
          style={[styles.btn, { backgroundColor: '#e74c3c' }]}
          onPress={logout}
        >
          <Text style={styles.btnText}>Sign Out</Text>
        </TouchableOpacity>
      </View>
    );
  }

  return (
    <View style={[styles.card, { backgroundColor: theme.card, borderColor: theme.border }]}>
      <Text style={[styles.cardTitle, { color: theme.text }]}>Sign In</Text>
      <Text style={[styles.cardSubtitle, { color: theme.subtext }]}>
        Tap to sign in as a demo user
      </Text>
      <TouchableOpacity
        style={[styles.btn, { backgroundColor: theme.accent }]}
        onPress={() => login({ name: 'Jane Developer', email: 'jane@devs.io' })}
      >
        <Text style={styles.btnText}>Sign In as Jane</Text>
      </TouchableOpacity>
    </View>
  );
}

// Main screen — note: it receives ZERO props from App
// All data comes from context
function HomeScreen() {
  const { theme } = useTheme();
  const { isAuthenticated } = useAuth();

  return (
    <ScrollView
      style={{ flex: 1, backgroundColor: theme.bg }}
      contentContainerStyle={styles.screenContent}
    >
      <Text style={[styles.screenTitle, { color: theme.text }]}>
        Context API Demo
      </Text>
      <Text style={[styles.screenSubtitle, { color: theme.subtext }]}>
        All components read from context directly.
        Zero props are passed through intermediate components.
      </Text>

      {/* Context flow visualization */}
      <View style={[styles.flowCard, { backgroundColor: theme.card, borderColor: theme.border }]}>
        <Text style={[styles.flowTitle, { color: theme.accent }]}>
          Context Data Flow
        </Text>
        <Text style={[styles.flowText, { color: theme.subtext }]}>
          {`App\n`}
          {`  ThemeProvider ──── broadcasts theme\n`}
          {`    AuthProvider ─── broadcasts auth\n`}
          {`      HomeScreen (no props needed!)\n`}
          {`        LoginCard ── reads auth ✓\n`}
          {`        UserBadge ── reads auth + theme ✓\n`}
          {`        ThemeSwitcher ─ reads + writes theme ✓\n`}
          {`        AdminPanel ──── reads auth ✓`}
        </Text>
      </View>

      <LoginCard />
      <ThemeSwitcher />
      {isAuthenticated && <AdminPanel />}
      {isAuthenticated && (
        <View style={[styles.card, { backgroundColor: theme.card, borderColor: theme.border }]}>
          <Text style={[styles.cardTitle, { color: theme.text }]}>Profile Badge</Text>
          <Text style={[styles.cardSubtitle, { color: theme.subtext }]}>
            This component is deeply nested and reads from both contexts:
          </Text>
          <UserBadge />
        </View>
      )}
    </ScrollView>
  );
}

// ═══════════════════════════════════════════════
// 5. ROOT — Wrap everything in Providers
// ═══════════════════════════════════════════════

export default function App() {
  // Providers at the top — their value flows down to ALL descendants
  return (
    <ThemeProvider>
      <AuthProvider>
        <HomeScreen />
      </AuthProvider>
    </ThemeProvider>
  );
}
```

## Advanced Patterns — Context Done Right

### Pattern 1: Always Use Custom Hooks

Rather than importing the context object and calling `useContext` in every component, always create a custom hook that wraps the context access. This gives you a clean, readable API, allows you to add validation (throwing an error if the hook is used outside its Provider), and means you can refactor the internal context structure without touching every consumer.

jsx

```jsx
// ❌ Verbose — every component does this
import ThemeContext from './ThemeContext';
const { theme } = useContext(ThemeContext);

// ✅ Clean — every component does this
import { useTheme } from './ThemeProvider';
const { theme } = useTheme();
```

### Pattern 2: Split Context by Update Frequency

One of the most important performance considerations with Context is that every consumer re-renders when the Provider's `value` changes. If you put both your user object and your frequently-changing counter in the same context, every component that reads the user will re-render every time the counter changes — even though they do not care about the counter.

The solution is to **split contexts by how frequently they update**. Rarely-changing global state (user, theme, language) should be in separate contexts from frequently-changing state.

```
┌────────────────────────────────────────────────────────────────────────┐
│              Context Splitting — Update Frequency                      │
│                                                                        │
│  ❌ One big context — everything in one place:                         │
│  { user, theme, counter, filter, selectedId }                         │
│  → Counter changes → user, theme, filter consumers ALL re-render      │
│                                                                        │
│  ✅ Split by update frequency:                                         │
│  UserContext:    { user }           ← changes rarely (login/logout)   │
│  ThemeContext:   { theme }          ← changes occasionally            │
│  UIContext:      { counter, filter }← changes frequently              │
│                                                                        │
│  Counter changes → only UIContext consumers re-render ✅              │
└────────────────────────────────────────────────────────────────────────┘
```

### Pattern 3: Memoize the Context Value

Every time the Provider's parent re-renders, a new `value` object is created (because object literals `{}` produce new references on every execution). This new reference causes all consumers to re-render even if the data inside the object is identical. The fix is to memoize the value object with `useMemo`.

jsx

```jsx
// ❌ New object reference every render → all consumers re-render
<ThemeContext.Provider value={{ theme, setTheme }}>

// ✅ Same reference if theme and setTheme haven't changed
const value = useMemo(() => ({ theme, setTheme }), [theme, setTheme]);
<ThemeContext.Provider value={value}>
```

## Context API — Summary

```
┌────────────────────────────────────────────────────────────────────────┐
│                    Context API — Core Concepts                         │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  Problem it solves: Prop drilling through intermediate components     │
│                                                                        │
│  Three parts:                                                          │
│  1. createContext(defaultValue)   → creates the channel               │
│  2. <Context.Provider value={...}>→ broadcasts data down the tree     │
│  3. useContext(Context)           → reads data from the channel       │
│                                                                        │
│  Best for:                                                             │
│  ✅ Authentication state (user, login, logout)                        │
│  ✅ Theme (colors, dark/light mode)                                   │
│  ✅ Language / locale settings                                        │
│  ✅ App-wide configuration                                            │
│  ✅ Navigation state (in navigators)                                  │
│                                                                        │
│  Not ideal for:                                                        │
│  ❌ Frequently updating values (every consumer re-renders)            │
│  ❌ Complex state with many actions (use useReducer or Redux)         │
│  ❌ Server state / data fetching (use React Query / SWR)             │
│                                                                        │
│  Performance rules:                                                    │
│  1. Always memoize the context value with useMemo                    │
│  2. Always memoize callbacks in context with useCallback              │
│  3. Split contexts by update frequency                                │
│  4. Create custom hooks (useTheme, useAuth) for clean DX              │
└────────────────────────────────────────────────────────────────────────┘
```
