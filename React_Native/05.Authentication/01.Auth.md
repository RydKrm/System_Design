# Authentication & Security in React Native

### JWT, Secure Storage, Refresh Tokens & Biometric Login — A Deep Guide from First Principles

---

## The Foundation — Why Authentication Is Complex on Mobile

Authentication on mobile is fundamentally different from authentication on the web. In a web browser, session cookies are managed by the browser itself, protected by `HttpOnly` flags that prevent JavaScript from ever reading them, and scoped to a domain so they cannot be stolen by other sites. The browser is a sandboxed, well-defined security environment with decades of hardened infrastructure.

A mobile application, by contrast, is a process running on a device that the user physically possesses. The device may be rooted or jailbroken, allowing system-level access to data. The application stores its data in files on disk that could be read if the device is compromised. The network traffic may pass through untrusted Wi-Fi. The user may share the device with family members. These realities make mobile authentication a much richer, more demanding engineering challenge.

Understanding this landscape deeply — not just the "how" but the "why" of every security decision — is what separates a professional mobile developer from one who merely copies authentication boilerplate and hopes for the best.

# Part One: JWT — JSON Web Tokens

---

## What Is a JWT?

A **JSON Web Token** (pronounced "jot") is a compact, self-contained string that encodes a set of claims — assertions about who a user is and what they are permitted to do — in a format that can be cryptographically verified without querying a database. It is the most widely used authentication mechanism in modern REST APIs, and understanding its structure deeply is essential for any developer working with authenticated APIs.

The critical philosophical shift that JWT represents is this: in traditional session-based authentication, the server stores the session state in a database, and the client holds only a session ID. Every request requires a database lookup to verify the session. With JWT, the server stores nothing. The client holds a token that is signed by the server's private key, and the server verifies the token's authenticity by checking the signature — a cryptographic operation that requires no database at all.

This stateless verification is what makes JWT ideal for distributed systems, microservices, and mobile APIs where multiple backend servers must all be able to validate the same tokens without sharing session state.

### The Most Important JWT Misconception

The single most dangerous misunderstanding about JWT is the belief that because the token is a cryptic-looking string, its contents are secret. They are not. Base64URL encoding is not encryption — it is merely a text encoding format. Any person who holds the token can decode the payload and read every claim inside it. The signature does not hide the data; it only proves the data has not been modified.

This is why you must never store sensitive information — passwords, credit card numbers, personal identification numbers, medical data — inside a JWT payload. The payload should contain only what is necessary for authorization decisions: the user's ID, their roles or permissions, and the standard timing claims.

### How Token Verification Works

```
┌──────────────────────────────────────────────────────────────────────────┐
│                    JWT Verification Flow                                 │
│                                                                          │
│  Client sends request:                                                   │
│  GET /api/profile                                                        │
│  Authorization: Bearer eyJhbGci...                                      │
│                          │                                               │
│                          ▼                                               │
│  Server receives token and:                                              │
│                                                                          │
│  Step 1: Split into header.payload.signature                            │
│                                                                          │
│  Step 2: Re-compute the signature:                                       │
│          HMAC_SHA256(header + "." + payload, SECRET_KEY)                 │
│                                                                          │
│  Step 3: Compare computed signature with token's signature               │
│          If they match → token is genuine ✅                             │
│          If they differ → token was tampered with ❌                    │
│                                                                          │
│  Step 4: Check expiry: is exp > current time?                           │
│          Yes → token is still valid ✅                                   │
│          No  → token is expired, return 401 ❌                          │
│                                                                          │
│  Step 5: Read claims from payload                                        │
│          sub = "usr_123" → user is identified                           │
│          role = "admin"  → user has admin privileges                    │
│                                                                          │
│  NO DATABASE QUERY REQUIRED for steps 1-4. That is the power of JWT.   │
└──────────────────────────────────────────────────────────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────
// JWT Utilities — Decoding and Reading Claims on the Client
// ─────────────────────────────────────────────────────────

// ── Decode a JWT payload WITHOUT verifying the signature ──
// This is safe to do on the client because:
// 1. The server already verified the signature when it issued the token
// 2. We use the decoded claims for UI decisions only (e.g., show admin menu)
// 3. We never trust decoded claims for actual security decisions on the server

function decodeJWT(token) {
  try {
    if (!token || typeof token !== 'string') return null;

    const parts = token.split('.');
    if (parts.length !== 3) return null;

    // The payload is the second part
    // Base64URL uses - and _ instead of + and /
    // We need to convert to standard Base64 before decoding
    const base64Payload = parts[1]
      .replace(/-/g, '+')  // URL-safe → standard Base64
      .replace(/_/g, '/'); // URL-safe → standard Base64

    // Pad with = if needed (Base64 requires length % 4 === 0)
    const padded = base64Payload.padEnd(
      base64Payload.length + (4 - (base64Payload.length % 4)) % 4,
      '='
    );

    const decoded = atob(padded);
    return JSON.parse(decoded);

  } catch {
    return null; // malformed token
  }
}

// ── Check if a token is expired ──
function isTokenExpired(token) {
  const payload = decodeJWT(token);
  if (!payload?.exp) return true; // no exp claim = treat as expired

  // exp is in seconds, Date.now() is in milliseconds
  // Add a 30-second buffer to account for clock skew between client and server
  const CLOCK_SKEW_BUFFER_SECONDS = 30;
  const nowInSeconds = Math.floor(Date.now() / 1000);
  return payload.exp < (nowInSeconds + CLOCK_SKEW_BUFFER_SECONDS);
}

// ── Get time until expiry in milliseconds ──
function getTokenTTL(token) {
  const payload = decodeJWT(token);
  if (!payload?.exp) return 0;

  const nowInSeconds = Math.floor(Date.now() / 1000);
  const remainingSeconds = payload.exp - nowInSeconds;
  return Math.max(0, remainingSeconds * 1000); // convert to ms, never negative
}

// ── Extract specific claims ──
function getTokenClaims(token) {
  const payload = decodeJWT(token);
  if (!payload) return null;

  return {
    userId:    payload.sub,
    name:      payload.name,
    email:     payload.email,
    role:      payload.role,
    roles:     payload.roles ?? [],
    issuedAt:  payload.iat ? new Date(payload.iat * 1000) : null,
    expiresAt: payload.exp ? new Date(payload.exp * 1000) : null,
    issuer:    payload.iss,
  };
}

// ── Usage examples ──
const token = 'eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c3JfMTIzIiwicm9sZSI6ImFkbWluIn0.sig';

const claims = getTokenClaims(token);
// { userId: 'usr_123', role: 'admin', ... }

const expired = isTokenExpired(token);
// true or false

const ttl = getTokenTTL(token);
// milliseconds until expiry — schedule a proactive refresh before it hits 0
```

# Part Two: Secure Storage — Protecting Tokens on Device

---

## Why AsyncStorage Is Insufficient for Tokens

`AsyncStorage` — the most common storage solution in React Native — is a simple, unencrypted, persistent key-value store. Its data is written as plain text files on the device's filesystem. While these files are sandboxed by the operating system (other apps cannot read them on a non-rooted device), they are accessible on rooted Android devices and jailbroken iOS devices. An attacker who gains root-level access to the device can read every byte of AsyncStorage data.

Storing authentication tokens in AsyncStorage is like writing your house key on a Post-it note and sticking it under the doormat. It will be fine for most days, but it is not a serious security practice.

The correct storage solution for sensitive data on mobile is the **operating system's dedicated secure storage** — iOS Keychain and Android Keystore. These are not just filesystem locations with different permissions. They are hardware-backed security modules that use the device's dedicated security chip (the Secure Enclave on iOS, StrongBox/TEE on Android) to store cryptographic keys and encrypted data. Even with root access, an attacker cannot extract the underlying keys from these hardware modules.

### The `react-native-keychain` Library

The `react-native-keychain` library provides a unified JavaScript API over iOS Keychain Services and Android Keystore System. It abstracts away the platform differences and exposes a simple interface for securely storing and retrieving credentials.

```bash
npm install react-native-keychain
cd ios && pod install
```

```jsx
// ─────────────────────────────────────────────────────────
// Secure Token Storage — Complete Implementation
// ─────────────────────────────────────────────────────────

import * as Keychain from 'react-native-keychain';

// ── Service identifiers — separate entries per data type ──
const SERVICES = {
  AUTH_TOKENS:    'com.yourapp.auth.tokens',
  USER_PROFILE:   'com.yourapp.auth.profile',
  BIOMETRIC_KEY:  'com.yourapp.biometric.key',
};

// ── The complete secure storage service ──
export const SecureStorage = {

  // ── Store authentication tokens ──
  async saveTokens({ accessToken, refreshToken, userId }) {
    try {
      // Keychain stores a username/password pair per service.
      // We use userId as the "username" and the tokens as the "password".
      // The tokens are serialized to JSON for the password field.
      await Keychain.setGenericPassword(
        userId,                                      // username (user identifier)
        JSON.stringify({ accessToken, refreshToken }), // password (our token payload)
        {
          service: SERVICES.AUTH_TOKENS,
          accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
          // WHEN_UNLOCKED_THIS_DEVICE_ONLY:
          //   - Only readable when device is unlocked
          //   - NOT transferred during iCloud backup
          //   - NOT transferred to a new device (prevents token migration)
        }
      );
    } catch (error) {
      throw new SecureStorageError('Failed to save tokens securely', error);
    }
  },

  // ── Retrieve authentication tokens ──
  async getTokens() {
    try {
      const result = await Keychain.getGenericPassword({
        service: SERVICES.AUTH_TOKENS,
      });

      if (!result || result === false) {
        return null; // no credentials stored
      }

      const { username: userId, password } = result;
      const { accessToken, refreshToken } = JSON.parse(password);

      return { userId, accessToken, refreshToken };

    } catch (error) {
      throw new SecureStorageError('Failed to retrieve tokens', error);
    }
  },

  // ── Delete tokens (on logout) ──
  async clearTokens() {
    try {
      await Keychain.resetGenericPassword({ service: SERVICES.AUTH_TOKENS });
    } catch (error) {
      throw new SecureStorageError('Failed to clear tokens', error);
    }
  },

  // ── Save user profile (less sensitive, but still worth encrypting) ──
  async saveUserProfile(userId, profileData) {
    await Keychain.setGenericPassword(
      userId,
      JSON.stringify(profileData),
      { service: SERVICES.USER_PROFILE }
    );
  },

  // ── Get user profile ──
  async getUserProfile() {
    const result = await Keychain.getGenericPassword({
      service: SERVICES.USER_PROFILE,
    });
    if (!result) return null;
    return JSON.parse(result.password);
  },

  // ── Store credentials requiring biometric access ──
  // This creates a Keychain entry that REQUIRES biometric verification
  // every time it is read — even if the device is unlocked
  async saveBiometricSecret(userId, secret) {
    await Keychain.setGenericPassword(
      userId,
      secret,
      {
        service: SERVICES.BIOMETRIC_KEY,
        accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
        accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET,
        // BIOMETRY_CURRENT_SET:
        //   Requires the same biometric enrolled at the time of storage.
        //   If the user adds a new fingerprint, this entry becomes invalid.
        //   This is MORE secure than BIOMETRY_ANY (which accepts newly enrolled biometrics)
      }
    );
  },

  // ── Read a biometric-protected secret ──
  // This will trigger the Face ID / fingerprint prompt automatically
  async getBiometricSecret(promptMessage = 'Authenticate to continue') {
    try {
      const result = await Keychain.getGenericPassword({
        service: SERVICES.BIOMETRIC_KEY,
        authenticationPrompt: {
          title: promptMessage,
          subtitle: 'Use your registered biometric',
          cancel: 'Use Password Instead',
        },
      });
      return result ? result.password : null;
    } catch (error) {
      if (error.message?.includes('cancel')) {
        throw new BiometricCancelledError('User cancelled biometric authentication');
      }
      throw error;
    }
  },

  async clearAll() {
    await Promise.allSettled([
      Keychain.resetGenericPassword({ service: SERVICES.AUTH_TOKENS }),
      Keychain.resetGenericPassword({ service: SERVICES.USER_PROFILE }),
      Keychain.resetGenericPassword({ service: SERVICES.BIOMETRIC_KEY }),
    ]);
  },
};

class SecureStorageError extends Error {
  constructor(message, originalError) {
    super(message);
    this.name = 'SecureStorageError';
    this.originalError = originalError;
  }
}

class BiometricCancelledError extends Error {
  constructor(message) {
    super(message);
    this.name = 'BiometricCancelledError';
  }
}
```

### Keychain Accessibility Levels — Choosing the Right One

The `accessible` option in Keychain controls when the stored item can be read. Choosing the wrong level is a common security mistake that developers make when they copy examples without understanding the implications.

```
┌──────────────────────────────────────────────────────────────────────────┐
│               Keychain Accessibility Levels                              │
│                                                                          │
│  WHEN_UNLOCKED (default)                                                 │
│  ─ Readable only while device is unlocked                               │
│  ─ Transferred in iCloud Backup → readable on new device                │
│  ─ Use for: Data the user expects to migrate to a new phone             │
│                                                                          │
│  WHEN_UNLOCKED_THIS_DEVICE_ONLY  ← recommended for auth tokens          │
│  ─ Readable only while device is unlocked                               │
│  ─ NOT transferred in backup → a new device starts fresh                │
│  ─ Use for: Authentication tokens, session data                         │
│                                                                          │
│  AFTER_FIRST_UNLOCK                                                      │
│  ─ Readable after first unlock following a restart                      │
│  ─ Allows background access (e.g., background refresh tasks)            │
│  ─ Use for: Refresh tokens needed by background processes               │
│                                                                          │
│  ALWAYS (avoid for sensitive data)                                       │
│  ─ Readable even while device is locked                                 │
│  ─ Suitable only for non-sensitive, background-required data            │
└──────────────────────────────────────────────────────────────────────────┘
```

# Part Three: Refresh Tokens — Session Continuity Without Re-Login

---

## The Token Pair Architecture

The access token + refresh token architecture is a carefully designed two-token system that balances security with user experience. Understanding why both tokens exist — and why they have the properties they do — is essential for implementing them correctly.

An **access token** is deliberately short-lived (typically 15 minutes to 1 hour). Its short lifespan is a security feature: if an access token is stolen, the damage is time-limited. An attacker who intercepts an access token gains unauthorized access to the API for at most one hour before the token expires. Because the server is stateless, there is no way to immediately revoke a valid access token without adding a database lookup (which defeats the purpose of JWT). Short expiry is the practical substitute for revocation.

A **refresh token** is long-lived (typically 7 to 30 days) and serves only one purpose: obtaining new access tokens. It is stored more securely than the access token, sent only to a single dedicated refresh endpoint (never to any other API endpoint), and is typically single-use (each use invalidates the current refresh token and returns a new one — this is called **refresh token rotation**).

```
┌──────────────────────────────────────────────────────────────────────────┐
│                     Token Pair Architecture                              │
│                                                                          │
│  ACCESS TOKEN                     REFRESH TOKEN                          │
│  ─────────────────────────        ──────────────────────────             │
│  Short-lived: 15min - 1hr         Long-lived: 7 - 30 days               │
│  Sent with every API request      Sent ONLY to /auth/refresh             │
│  Stored: Keychain (memory OK)     Stored: Keychain (disk required)      │
│  If stolen: 1hr max damage        If stolen: must invalidate immediately │
│                                                                          │
│  Timeline:                                                               │
│                                                                          │
│  Login ──► [accessToken A] ──► expires after 1hr                       │
│              [refreshToken X]                                            │
│                    │                                                     │
│                    │  (access token expires, call /auth/refresh)        │
│                    ▼                                                     │
│            [accessToken B] ──► expires after 1hr                       │
│            [refreshToken Y]    ← NEW refresh token (rotation)           │
│            refreshToken X is now INVALID                                 │
│                    │                                                     │
│                    │  (repeat)                                           │
│                    ▼                                                     │
│            [accessToken C] + [refreshToken Z]                           │
│                                                                          │
│  Refresh token rotation: if attackers steal refresh token X,           │
│  and legitimate user uses X to get Y, the attacker's copy of X is      │
│  now invalid. The server detects the old token being reused.            │
└──────────────────────────────────────────────────────────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────
// Complete Token Lifecycle Management
// ─────────────────────────────────────────────────────────

import { createContext, useContext, useState, useEffect, useCallback, useRef } from 'react';
import axios from 'axios';

// ── Auth Service — all token operations ──
export const AuthService = {

  // ── Login — exchange credentials for tokens ──
  async login(email, password) {
    const response = await axios.post('https://api.yourapp.com/auth/login', {
      email,
      password,
    });

    const { accessToken, refreshToken, user } = response.data;

    // Store tokens in secure storage (Keychain / Keystore)
    await SecureStorage.saveTokens({
      accessToken,
      refreshToken,
      userId: user.id,
    });

    return { accessToken, refreshToken, user };
  },

  // ── Refresh — exchange refresh token for new token pair ──
  // Uses plain axios (not the interceptor-wrapped client) to avoid infinite loop
  async refresh() {
    const stored = await SecureStorage.getTokens();
    if (!stored?.refreshToken) {
      throw new Error('No refresh token available');
    }

    try {
      const response = await axios.post('https://api.yourapp.com/auth/refresh', {
        refreshToken: stored.refreshToken,
      });

      const { accessToken, refreshToken: newRefreshToken } = response.data;

      // Save the new token pair — the old refresh token is now invalid
      await SecureStorage.saveTokens({
        accessToken,
        refreshToken: newRefreshToken,
        userId: stored.userId,
      });

      return { accessToken, refreshToken: newRefreshToken };

    } catch (error) {
      // Refresh failed (token expired or revoked)
      // Clear all stored tokens and force re-login
      await SecureStorage.clearTokens();
      throw error;
    }
  },

  // ── Logout — revoke tokens on server + clear local storage ──
  async logout() {
    const stored = await SecureStorage.getTokens();

    if (stored?.refreshToken) {
      // Tell the server to invalidate the refresh token
      // (silent failure — we logout locally regardless)
      try {
        await axios.post(
          'https://api.yourapp.com/auth/logout',
          { refreshToken: stored.refreshToken },
          { headers: { Authorization: `Bearer ${stored.accessToken}` } }
        );
      } catch {
        // Ignore server errors — always clear local tokens
      }
    }

    await SecureStorage.clearAll();
  },

  // ── Initialize — restore session from storage on app start ──
  async initialize() {
    try {
      const stored = await SecureStorage.getTokens();
      if (!stored) return null;

      const { accessToken, refreshToken, userId } = stored;

      // Check if access token is still valid
      if (!isTokenExpired(accessToken)) {
        return { accessToken, refreshToken, userId };
      }

      // Access token expired — try to refresh
      if (refreshToken) {
        return await AuthService.refresh();
      }

      return null; // no valid session
    } catch {
      await SecureStorage.clearTokens();
      return null;
    }
  },
};

// ── Auth Context — global session state management ──
const AuthContext = createContext(null);

export function AuthProvider({ children, onUnauthenticated }) {
  const [user, setUser] = useState(null);
  const [accessToken, setAccessToken] = useState(null);
  const [isLoading, setIsLoading] = useState(true); // true while restoring session
  const [isAuthenticated, setIsAuthenticated] = useState(false);

  // Proactive token refresh: schedule refresh 2 minutes before expiry
  const refreshTimerRef = useRef(null);

  const scheduleProactiveRefresh = useCallback((token) => {
    if (refreshTimerRef.current) {
      clearTimeout(refreshTimerRef.current);
    }

    const ttl = getTokenTTL(token);
    const TWO_MINUTES_MS = 2 * 60 * 1000;
    const refreshIn = Math.max(0, ttl - TWO_MINUTES_MS);

    if (refreshIn > 0) {
      console.log(`Proactive token refresh scheduled in ${Math.round(refreshIn / 1000)}s`);
      refreshTimerRef.current = setTimeout(async () => {
        try {
          const { accessToken: newToken } = await AuthService.refresh();
          setAccessToken(newToken);
          scheduleProactiveRefresh(newToken); // schedule next refresh
        } catch {
          // Refresh failed — trigger sign-out
          handleLogout();
        }
      }, refreshIn);
    }
  }, []);

  // ── Restore session on app launch ──
  useEffect(() => {
    const init = async () => {
      try {
        const session = await AuthService.initialize();
        if (session) {
          setAccessToken(session.accessToken);
          setIsAuthenticated(true);
          const claims = getTokenClaims(session.accessToken);
          setUser({ id: session.userId, ...claims });
          scheduleProactiveRefresh(session.accessToken);
        }
      } catch {
        setIsAuthenticated(false);
      } finally {
        setIsLoading(false);
      }
    };
    init();

    return () => {
      if (refreshTimerRef.current) clearTimeout(refreshTimerRef.current);
    };
  }, [scheduleProactiveRefresh]);

  const handleLogin = useCallback(async (email, password) => {
    const result = await AuthService.login(email, password);
    setAccessToken(result.accessToken);
    setIsAuthenticated(true);
    setUser(result.user);
    scheduleProactiveRefresh(result.accessToken);
    return result;
  }, [scheduleProactiveRefresh]);

  const handleLogout = useCallback(async () => {
    if (refreshTimerRef.current) clearTimeout(refreshTimerRef.current);
    await AuthService.logout();
    setUser(null);
    setAccessToken(null);
    setIsAuthenticated(false);
    onUnauthenticated?.();
  }, [onUnauthenticated]);

  return (
    <AuthContext.Provider value={{
      user,
      accessToken,
      isAuthenticated,
      isLoading,
      login: handleLogin,
      logout: handleLogout,
    }}>
      {children}
    </AuthContext.Provider>
  );
}

export function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used inside AuthProvider');
  return ctx;
}
```

# Part Four: Biometric Login — Face ID, Touch ID & Fingerprint

## What Biometric Authentication Is

Biometric authentication uses the user's physical characteristics — their fingerprint, their face geometry, their iris — to verify identity. On iOS this is Face ID or Touch ID. On Android this is the fingerprint scanner, face unlock, or iris scanner, accessed through the BiometricPrompt API. Both platforms expose these capabilities through their respective secure hardware modules, ensuring that biometric data itself never leaves the device's secure enclave and is never accessible to your application code.

It is crucial to understand what biometric authentication in a mobile app is and is not. It is a **local device authentication mechanism** — it proves that the person holding the phone is the person who enrolled their biometric on that device. It does not, by itself, authenticate the user to your server. Biometric authentication is used to unlock locally stored credentials (tokens or passwords), which are then used to authenticate to the server. The biometric is a gate to the secure storage, not a credential transmitted to the backend.

```jsx
// ─────────────────────────────────────────────────────────
// Complete Biometric Authentication Implementation
// ─────────────────────────────────────────────────────────

import * as Keychain from 'react-native-keychain';
import ReactNativeBiometrics, { BiometryTypes } from 'react-native-biometrics';
import { useState, useEffect, useCallback } from 'react';
import {
  View, Text, TouchableOpacity, StyleSheet,
  Alert, ActivityIndicator
} from 'react-native';

const rnBiometrics = new ReactNativeBiometrics({
  allowDeviceCredentials: false, // false = biometric only, no PIN fallback
});

// ── Biometric Service ──
export const BiometricService = {

  // ── Check what biometric is available on this device ──
  async getAvailability() {
    try {
      const { available, biometryType, error } = await rnBiometrics.isSensorAvailable();

      return {
        available,
        type: biometryType,             // 'FaceID', 'TouchID', 'Biometrics'
        typeLabel: getBiometricLabel(biometryType),
        typeIcon: getBiometricIcon(biometryType),
        error: error || null,
      };
    } catch {
      return { available: false, type: null, typeLabel: 'Biometric', typeIcon: '🔐', error: 'Check failed' };
    }
  },

  // ── Enroll — save refresh token behind biometric lock ──
  async enrollBiometric(userId, refreshToken) {
    const { available, error } = await this.getAvailability();
    if (!available) throw new Error(`Biometric unavailable: ${error}`);

    await Keychain.setGenericPassword(
      userId,
      refreshToken,
      {
        service: 'com.yourapp.biometric.session',
        accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
        accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET,
        // BIOMETRY_CURRENT_SET ensures that if the user adds a new fingerprint,
        // this stored item becomes invalid — a security feature
      }
    );
    console.log('[BiometricService] Enrolled successfully');
  },

  // ── Authenticate — trigger biometric prompt and get stored token ──
  async authenticate(promptMessage = 'Verify your identity') {
    try {
      const result = await Keychain.getGenericPassword({
        service: 'com.yourapp.biometric.session',
        authenticationPrompt: {
          title: promptMessage,
          subtitle: 'Use biometric to sign in',
          description: 'Touch the sensor or look at the camera',
          cancel: 'Use Password',
        },
      });

      if (!result) {
        throw new Error('No biometric credentials enrolled');
      }

      return {
        success: true,
        userId: result.username,
        refreshToken: result.password,
      };

    } catch (error) {
      const message = error.message ?? '';
      if (message.includes('cancel') || message.includes('Cancel')) {
        return { success: false, cancelled: true };
      }
      if (message.includes('NOT_ENROLLED') || message.includes('No biometric')) {
        return { success: false, notEnrolled: true };
      }
      if (message.includes('LOCKOUT')) {
        return { success: false, lockedOut: true };
      }
      return { success: false, error: message };
    }
  },

  // ── Check if biometric is enrolled for this app ──
  async isEnrolled() {
    try {
      const result = await Keychain.hasGenericPassword({
        service: 'com.yourapp.biometric.session',
      });
      return !!result;
    } catch {
      return false;
    }
  },

  // ── Remove biometric enrollment ──
  async unenroll() {
    await Keychain.resetGenericPassword({
      service: 'com.yourapp.biometric.session',
    });
  },
};

function getBiometricLabel(type) {
  switch (type) {
    case BiometryTypes.FaceID:      return 'Face ID';
    case BiometryTypes.TouchID:     return 'Touch ID';
    case BiometryTypes.Biometrics:  return 'Fingerprint';
    default:                         return 'Biometric';
  }
}

function getBiometricIcon(type) {
  switch (type) {
    case BiometryTypes.FaceID:      return '🆔';
    case BiometryTypes.TouchID:     return '👆';
    case BiometryTypes.Biometrics:  return '👆';
    default:                         return '🔐';
  }
}

// ── Biometric Login Screen ──
export function LoginScreen({ onSuccess }) {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [loading, setLoading] = useState(false);
  const [bioLoading, setBioLoading] = useState(false);
  const [error, setError] = useState(null);

  const [bioAvailable, setBioAvailable] = useState(false);
  const [bioEnrolled, setBioEnrolled] = useState(false);
  const [bioInfo, setBioInfo] = useState(null);

  // ── Check biometric capability on mount ──
  useEffect(() => {
    const checkBiometric = async () => {
      const availability = await BiometricService.getAvailability();
      const enrolled = await BiometricService.isEnrolled();
      setBioAvailable(availability.available);
      setBioEnrolled(enrolled);
      setBioInfo(availability);
    };
    checkBiometric();
  }, []);

  // ── Auto-trigger biometric if enrolled (optional, can be annoying) ──
  useEffect(() => {
    if (bioAvailable && bioEnrolled) {
      // Small delay to let the screen render before showing the prompt
      const timer = setTimeout(() => handleBiometricLogin(), 500);
      return () => clearTimeout(timer);
    }
  }, [bioAvailable, bioEnrolled]);

  // ── Biometric login flow ──
  const handleBiometricLogin = useCallback(async () => {
    setBioLoading(true);
    setError(null);

    try {
      const result = await BiometricService.authenticate(
        `Sign in with ${bioInfo?.typeLabel ?? 'biometric'}`
      );

      if (result.cancelled) {
        setBioLoading(false);
        return; // User chose password instead
      }

      if (!result.success) {
        if (result.lockedOut) {
          setError('Too many failed attempts. Please use your password.');
        } else if (result.notEnrolled) {
          setBioEnrolled(false);
          setError('Biometric setup has changed. Please sign in with password.');
        } else {
          setError('Biometric authentication failed. Please try again.');
        }
        setBioLoading(false);
        return;
      }

      // Biometric succeeded — use the stored refresh token to get a new access token
      const tokenResponse = await axios.post(
        'https://api.yourapp.com/auth/refresh',
        { refreshToken: result.refreshToken }
      );

      const { accessToken, refreshToken: newRefreshToken } = tokenResponse.data;

      // Update stored refresh token with the rotated one
      await BiometricService.enrollBiometric(result.userId, newRefreshToken);

      onSuccess({ accessToken, userId: result.userId });

    } catch (err) {
      setError('Authentication failed. Please sign in with your password.');
    } finally {
      setBioLoading(false);
    }
  }, [bioInfo, onSuccess]);

  // ── Password login flow ──

          [

                try {
                  await BiometricService.enrollBiometric(result.user.id, result.refreshToken);
                  setBioEnrolled(true);
                } catch {
                  // Non-critical — enrollment failure doesn't block login
                }
              },
            },
            { text: 'Not Now', style: 'cancel' },
          ]
        );
      }

      onSuccess({ accessToken: result.accessToken, userId: result.user.id });

    } catch (err) {
      setError(err.message || 'Login failed. Please check your credentials.');
    } finally {
      setLoading(false);
    }
  }, [email, password, bioAvailable, bioEnrolled, bioInfo, onSuccess]);

  return (
    <View style={loginStyles.container}>
      <Text style={loginStyles.title}>Welcome Back</Text>
      <Text style={loginStyles.subtitle}>Sign in to continue</Text>

      {/* Biometric button — shown only if enrolled */}
      {bioAvailable && bioEnrolled && (
        <TouchableOpacity
          style={loginStyles.bioBtn}
          onPress={handleBiometricLogin}
          disabled={bioLoading}
        >
          {bioLoading ? (
            <ActivityIndicator color="#6C63FF" />
          ) : (
            <>
              <Text style={loginStyles.bioIcon}>{bioInfo?.typeIcon ?? '🔐'}</Text>
              <Text style={loginStyles.bioBtnText}>
                Sign in with {bioInfo?.typeLabel ?? 'Biometric'}
              </Text>
            </>
          )}
        </TouchableOpacity>
      )}

      {/* Divider */}
      {bioAvailable && bioEnrolled && (
        <View style={loginStyles.dividerRow}>
          <View style={loginStyles.dividerLine} />
          <Text style={loginStyles.dividerText}>or use password</Text>
          <View style={loginStyles.dividerLine} />
        </View>
      )}

      {/* Email / Password form */}
      <View style={loginStyles.form}>
        <TextInput
          style={loginStyles.input}
          value={email}
          onChangeText={setEmail}
          placeholder="Email address"
          placeholderTextColor="#aaa"
          keyboardType="email-address"
          autoCapitalize="none"
          autoCorrect={false}
        />
        <TextInput
          style={loginStyles.input}
          value={password}
          onChangeText={setPassword}

          placeholderTextColor="#aaa"
          secureTextEntry
        />

        {error && (

            <Text style={loginStyles.errorText}>⚠ {error}</Text>

  );
}
```

## Security Checklist

```
┌──────────────────────────────────────────────────────────────────────────┐
│                  Production Security Checklist                           │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  JWT                                                                     │
│  ✅ Never store sensitive data in JWT payload (it's readable by anyone) │
│  ✅ Always verify exp claim server-side on every request                 │
│  ✅ Use short expiry for access tokens (15min – 1hr)                    │
│  ✅ Use HTTPS only — JWT in transit must be encrypted                   │
│  ✅ Check iss and aud claims to prevent token confusion attacks         │
│                                                                          │
│  SECURE STORAGE                                                          │
│  ✅ Use Keychain (iOS) / Keystore (Android) for all tokens              │
│  ✅ NEVER use AsyncStorage for access/refresh tokens                    │
│  ✅ Use WHEN_UNLOCKED_THIS_DEVICE_ONLY accessibility level              │
│  ✅ Use BIOMETRY_CURRENT_SET (not BIOMETRY_ANY) for biometric entries   │
│  ✅ Clear all Keychain entries on logout                                 │
│                                                                          │
│  REFRESH TOKENS                                                          │
│  ✅ Implement refresh token rotation (each use = new refresh token)     │
│  ✅ Detect refresh token reuse on server (sign of theft)                │
│  ✅ Use mutex pattern to prevent concurrent refresh requests            │
│  ✅ Schedule proactive refresh before expiry (avoid 401 mid-session)   │
│  ✅ Always revoke refresh token on logout (server-side invalidation)    │
│                                                                          │
│  BIOMETRIC                                                               │
│  ✅ Biometric authenticates locally — never send biometric data to server│
│  ✅ Use BIOMETRY_CURRENT_SET so new enrollments don't inherit access    │
│  ✅ Handle all failure modes: cancelled, lockout, not enrolled          │
│  ✅ Always offer password fallback when biometric is unavailable        │
│  ✅ Unenroll biometric credentials on logout / account deletion        │
└──────────────────────────────────────────────────────────────────────────┘
```

_Authentication is not a feature you bolt on at the end of a project. It is a foundational architectural concern that shapes every layer of your application — from how you store data on device, to how you structure your HTTP client, to how you design your component tree. JWT gives you stateless, verifiable identity. Secure storage gives that identity a safe home on the device. Refresh tokens give your users a seamless, long-lived session without sacrificing security. Biometric authentication gives your users the fastest, most frictionless path to that session. Together, these four pillars form a complete, production-grade mobile authentication system._
