# React Hooks — A Deep & Comprehensive Guide

### Understanding the Five Essential Hooks Every React Native Developer Must Master

---

## What Is a Hook, and Why Does It Exist?

Before hooks were introduced in React 16.8, the only way to use state or lifecycle methods in a React component was to write a **class component**. Class components are verbose, difficult to read, and carry a lot of conceptual overhead — you had to understand `this`, bind methods, and manage complex lifecycle methods like `componentDidMount`, `componentDidUpdate`, and `componentWillUnmount`. Writing a simple counter required more boilerplate code than the actual logic itself.

Hooks were React's answer to this complexity. A hook is simply a **special function** that begins with the word `use` and allows you to "hook into" React's internal machinery — its state system, its side-effect scheduler, its memory — directly from a plain function component. No classes, no `this`, no confusing lifecycle methods. Just functions that read clearly from top to bottom.

```
┌────────────────────────────────────────────────────────────┐
│              The Hook Mental Model                         │
│                                                            │
│  Your Component (a plain JS function)                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  const [count, setCount] = useState(0)              │   │
│  │         │         │                                 │   │
│  │         │         └── Function to UPDATE state      │   │
│  │         └──────────── Current state VALUE           │   │
│  │                              │                      │   │
│  └──────────────────────────────┼──────────────────────┘   │
│                                 │                          │
│                          ┌──────▼──────┐                   │
│                          │  React's    │                   │
│                          │  Internal   │                   │
│                          │  Memory     │                   │
│                          │  (Fiber)    │                   │
│                          └─────────────┘                   │
│                                                            │
│  The hook is the bridge between your function             │
│  and React's internals. It remembers things               │
│  across re-renders so your function doesn't have to.      │
└────────────────────────────────────────────────────────────┘
```

There are two golden rules that govern all hooks. First, hooks must always be called at the **top level** of your component — never inside an `if` block, a loop, or a nested function. This is because React identifies each hook by its call order, and if that order changes between renders, React loses track of which state belongs to which hook. Second, hooks can only be called from **React function components** or from **custom hooks** — never from plain JavaScript functions.

With that foundation established, let us examine each hook in depth.

## 1. `useState` — Remembering Values Across Renders

### The Problem It Solves

A plain JavaScript function has no memory. Every time a function is called, it starts fresh — all variables are created anew and discarded when the function returns. This is normally exactly what you want from a function. But a React component is a function that React calls repeatedly — once on initial render, and again every time something changes. If your component has no memory, it cannot remember that a user has typed their name, pressed a button, or opened a dropdown. It would reset to its initial state every single time.

`useState` gives your component a persistent memory. It creates a **state variable** that survives across renders. When you call `useState`, React stores the value in its own internal memory (not in your function's local scope), and every time the component re-renders, React hands that stored value back to you.

### How It Works

`useState` accepts a single argument — the **initial value** of the state — and returns an array of exactly two items: the current value of the state, and a function to update it. By convention, these are destructured immediately using array destructuring.

```
┌──────────────────────────────────────────────────────────────┐
│                   useState Lifecycle                         │
│                                                              │
│  First Render:                                               │
│  useState(0)  →  React stores 0  →  returns [0, setCount]   │
│                                                              │
│  User calls setCount(1):                                     │
│  React updates stored value to 1                             │
│  React schedules a re-render                                 │
│                                                              │
│  Second Render:                                              │
│  useState(0)  →  React ignores 0  →  returns [1, setCount]  │
│                  (already has stored value)                  │
│                                                              │
│  The initial value (0) is ONLY used on the very first render │
└──────────────────────────────────────────────────────────────┘
```

When you call the updater function (e.g., `setCount`), React does not immediately update the variable — it schedules a re-render. On the next render, React will give you the new value. This is why state updates are called **asynchronous** — the value does not change the instant you call the setter.

For cases where the new state depends on the old state, you should use the **functional update form**: `setCount(prev => prev + 1)`. This ensures you are always working with the most recent value, even if multiple updates are batched together.

### Use Cases

`useState` is appropriate for any value that, when changed, should cause the component to visually update. This includes form field values, toggle states (open/closed, visible/hidden), counters, selected items, loading indicators, error messages, fetched data, and virtually any dynamic value your interface needs to display or react to.

```jsx
import { useState } from 'react';

            style={[styles.btn, styles.btnPrimary]}
            onPress={() => setCount(prev => prev + 1)}
          >
            <Text style={[styles.btnLabel, { color: '#fff' }]}>+</Text>
          </TouchableOpacity>
          <TouchableOpacity
            style={[styles.btn, styles.btnDanger]}
            onPress={() => setCount(0)}
          >
            <Text style={[styles.btnLabel, { color: '#fff' }]}>Reset</Text>
          </TouchableOpacity>

      </View>

          <Text style={styles.greeting}>Hello, {name}! 👋</Text>
        )}
      </View>

        <Text style={styles.cardTitle}>Boolean Toggle</Text>

            🎉 This message is now visible!

            <TouchableOpacity
              key={tag + i}
              style={styles.tag}

          <Text style={{ color: '#fff', fontWeight: '600' }}>+ Add Tag</Text>
        </TouchableOpacity>
      </View>

    </View>
  );
}
```

## 2. `useEffect` — Synchronizing with the Outside World

### The Problem It Solves

React components are meant to be **pure functions** of their props and state — given the same inputs, they should produce the same output every time. But real-world applications need to do things that fall outside this pure model: fetching data from an API, setting up event listeners, starting timers, writing to a database, or subscribing to a real-time stream. These activities are called **side effects** because they reach outside the component and interact with the world beyond React's control.

`useEffect` is the designated place for all of these side effects. It says to React: "After you have finished rendering this component and the screen has been updated, please also run this function." This ensures that side effects never interfere with the rendering process itself.

### How It Works

`useEffect` accepts two arguments: a **callback function** (the effect itself) and a **dependency array** (which controls when the effect runs).

```
┌──────────────────────────────────────────────────────────────────┐
│                   useEffect Dependency Array                     │
│                                                                  │
│  useEffect(() => { ... })                                        │
│  No dependency array → runs after EVERY render                  │
│  ⚠️  Rarely what you want                                        │
│                                                                  │
│  useEffect(() => { ... }, [])                                    │
│  Empty array → runs ONCE after the first render (mount)         │
│  ✅  Perfect for: API calls, subscriptions, one-time setup      │
│                                                                  │
│  useEffect(() => { ... }, [userId, filter])                      │
│  Specific values → runs when userId OR filter changes           │
│  ✅  Perfect for: re-fetching when a dependency changes         │
│                                                                  │
│  Cleanup Function:                                               │
│  useEffect(() => {                                               │
│    const timer = setInterval(fn, 1000);                          │
│    return () => clearInterval(timer);  // ← CLEANUP             │
│  }, []);                                                         │
│  Cleanup runs BEFORE the next effect, or on component unmount   │
└──────────────────────────────────────────────────────────────────┘
```

The **cleanup function** — an optional function you return from the effect — is one of the most important concepts in `useEffect`. If your effect starts something that persists (a timer, a subscription, an event listener), you must stop it in the cleanup function. Otherwise, when the component unmounts, those resources keep running, leading to **memory leaks** and bugs where callbacks fire on components that no longer exist.

### Use Cases

`useEffect` is used for: initial data fetching, refetching when a filter or ID changes, setting up WebSocket subscriptions, starting and clearing timers, listening to device sensors or events, logging analytics, syncing state to AsyncStorage, and updating the document title.

```jsx
import { useState, useEffect } from 'react';
import { View, Text, TextInput, FlatList, ActivityIndicator, StyleSheet } from 'react-native';

// Simulates a data fetch (in a real app, this would be axios or fetch)
const fakeFetch = (query) =>
  new Promise((resolve) =>
    setTimeout(() => {
      const allPosts = [
        { id: '1', title: 'Getting Started with React Native', author: 'Alice' },
        { id: '2', title: 'Understanding useEffect Deeply', author: 'Bob' },
        { id: '3', title: 'Building a REST API with Node.js', author: 'Charlie' },
        { id: '4', title: 'PostgreSQL Performance Tips', author: 'Alice' },
        { id: '5', title: 'Docker for Backend Developers', author: 'David' },
        { id: '6', title: 'Redis Caching Strategies', author: 'Bob' },
      ];
      const results = query
        ? allPosts.filter(p =>
            p.title.toLowerCase().includes(query.toLowerCase()))
        : allPosts;
      resolve(results);
    }, 800)
  );

export default function UseEffectDemo() {
  const [posts, setPosts] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  const [query, setQuery] = useState('');
  const [seconds, setSeconds] = useState(0);

  // ── Effect 1: A timer that runs once on mount and cleans up on unmount ──
  useEffect(() => {
    const interval = setInterval(() => {
      setSeconds(prev => prev + 1);
    }, 1000);

    // CLEANUP: clear the interval when the component unmounts
    // Without this, the interval would keep running even after navigating away
    return () => {
      clearInterval(interval);
      console.log('Timer cleaned up — component unmounted');
    };
  }, []); // ← empty array: run once on mount

  // ── Effect 2: Fetch posts whenever the search query changes ──
  useEffect(() => {
    let cancelled = false; // prevents setting state on unmounted component

    const loadPosts = async () => {
      setLoading(true);
      setError(null);
      try {
        const results = await fakeFetch(query);
        if (!cancelled) {
          setPosts(results);
        }
      } catch (err) {
        if (!cancelled) {
          setError('Failed to load posts. Please try again.');
        }
      } finally {
        if (!cancelled) {
          setLoading(false);
        }
      }
    };

    // Debounce: wait 500ms after user stops typing before fetching
    const debounceTimer = setTimeout(loadPosts, 500);

    // CLEANUP: cancel the debounce timer if query changes before 500ms
    // This prevents firing a fetch for every single keystroke
    return () => {
      cancelled = true;
      clearTimeout(debounceTimer);
    };
  }, [query]); // ← runs whenever 'query' changes

  return (
    <View style={styles.container}>

      {/* Timer — demonstrates mount/unmount cleanup */}
      <View style={styles.timerBadge}>
        <Text style={styles.timerText}>⏱ Component alive for {seconds}s</Text>
      </View>

      {/* Search input */}
      <TextInput
        style={styles.searchInput}
        value={query}
        onChangeText={setQuery}
        placeholder="Search posts..."
        placeholderTextColor="#aaa"
      />

      {/* Results */}
      {loading ? (
        <ActivityIndicator size="large" color="#6C63FF" style={{ marginTop: 40 }} />
      ) : error ? (
        <Text style={styles.errorText}>{error}</Text>
      ) : (
        <FlatList
          data={posts}
          keyExtractor={item => item.id}
          renderItem={({ item }) => (
            <View style={styles.postCard}>
              <Text style={styles.postTitle}>{item.title}</Text>
              <Text style={styles.postAuthor}>by {item.author}</Text>
            </View>
          )}
          ListEmptyComponent={
            <Text style={styles.emptyText}>No posts match your search.</Text>
          }
        />
      )}
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, backgroundColor: '#f2f2f7', padding: 16 },
  timerBadge: {
    backgroundColor: '#fff3cd',
    padding: 10,
    borderRadius: 8,
    marginBottom: 12,
    alignSelf: 'flex-start',
  },
  timerText: { fontSize: 13, color: '#856404', fontWeight: '600' },
  searchInput: {
    backgroundColor: '#fff',
    borderRadius: 10,
    borderWidth: 1.5,
    borderColor: '#ddd',
    paddingHorizontal: 14,
    paddingVertical: 11,
    fontSize: 16,
    color: '#333',
    marginBottom: 16,
  },
  postCard: {
    backgroundColor: '#fff',
    borderRadius: 10,
    padding: 16,
    marginBottom: 10,
    elevation: 1,
  },
  postTitle: { fontSize: 15, fontWeight: '600', color: '#1a1a1a' },
  postAuthor: { fontSize: 13, color: '#888', marginTop: 4 },
  errorText: { color: '#e74c3c', textAlign: 'center', marginTop: 30, fontSize: 15 },
  emptyText: { textAlign: 'center', color: '#aaa', marginTop: 40, fontSize: 15 },
});
```

## 3. `useMemo` — Remembering Expensive Calculations

### The Problem It Solves

Every time a React component re-renders, every line of code inside it runs again from scratch. For simple operations — returning a string, accessing a prop — this is perfectly fine because these operations are instantaneous. But what if your component contains a computationally expensive operation — sorting a list of ten thousand items, filtering a complex dataset, running a mathematical calculation, or building a derived data structure? Running this expensive computation on every single render, even renders where the input data has not changed, is wasteful and can make your application feel sluggish.

`useMemo` solves this by **memoizing** (caching) the result of an expensive computation. It runs the computation once, stores the result, and on subsequent renders, it checks whether the inputs (the dependencies) have changed. If they have not changed, it returns the cached result without re-running the computation. Only when a dependency actually changes does `useMemo` re-run the function and produce a fresh result.

### How It Works

`useMemo` accepts the same two arguments as `useEffect`: a function that returns the computed value, and a dependency array. The critical difference from `useEffect` is that `useMemo` **returns the computed value directly** — it is not about side effects but about returning a value efficiently.

```
┌────────────────────────────────────────────────────────────────────┐
│                    useMemo — Cache Logic                           │
│                                                                    │
│  First render:                                                     │
│  useMemo(() => expensiveCalc(data), [data])                        │
│  → Runs expensiveCalc(data) → stores result → returns result       │
│                                                                    │
│  Re-render (data unchanged):                                       │
│  → Checks: has [data] changed? → NO                               │
│  → Returns CACHED result (expensiveCalc NOT called) ✅ Fast       │
│                                                                    │
│  Re-render (data changed):                                         │
│  → Checks: has [data] changed? → YES                              │
│  → Runs expensiveCalc(data) again → stores new result → returns it │
│                                                                    │
│  ⚠️  Important: useMemo is for DERIVED VALUES only.               │
│     Do not use it to prevent re-renders of child components.      │
│     For that, use React.memo on the child + useCallback for fns.  │
└────────────────────────────────────────────────────────────────────┘
```

### When NOT to Use useMemo

`useMemo` is not free — it costs memory to store the cached value and time to compare dependencies on every render. For simple computations (adding two numbers, concatenating strings), the overhead of `useMemo` can actually be greater than just re-running the computation. Use `useMemo` only when you have a genuinely expensive computation that runs on a large dataset, or when you need referential stability of an object or array to prevent unnecessary re-renders in child components.

### Use Cases

Filtering or sorting large arrays, expensive mathematical calculations, building derived data structures from raw state, computing aggregated statistics, and generating processed data that multiple parts of the component depend upon.

```jsx
import { useState, useMemo } from 'react';
import {
  View, Text, TextInput, FlatList, TouchableOpacity, StyleSheet
} from 'react-native';

// Simulates a large dataset
const generateProducts = () =>
  Array.from({ length: 1000 }, (_, i) => ({
    id: String(i + 1),
    name: `Product ${i + 1}`,
    category: ['Electronics', 'Clothing', 'Food', 'Books', 'Sports'][i % 5],
    price: Math.floor(Math.random() * 5000) + 100,
    rating: (Math.random() * 2 + 3).toFixed(1),
    inStock: Math.random() > 0.3,
  }));

const ALL_PRODUCTS = generateProducts(); // created once outside component

const CATEGORIES = ['All', 'Electronics', 'Clothing', 'Food', 'Books', 'Sports'];

export default function ProductCatalogue() {
  const [search, setSearch] = useState('');
  const [selectedCategory, setSelectedCategory] = useState('All');
  const [sortBy, setSortBy] = useState('name'); // 'name' | 'price' | 'rating'
  const [showInStockOnly, setShowInStockOnly] = useState(false);
  const [unrelatedCounter, setUnrelatedCounter] = useState(0);

  // ── WITHOUT useMemo ──
  // This entire filter+sort runs on EVERY render — including when
  // 'unrelatedCounter' changes, even though it has nothing to do
  // with the product list. With 1000 items, this is wasteful.

  // ── WITH useMemo ──
  // This only re-runs when search, selectedCategory, sortBy, or
  // showInStockOnly actually change. The unrelatedCounter changing
  // does NOT trigger this computation.
  const filteredAndSorted = useMemo(() => {
    console.log('🔄 Running expensive filter+sort...');

    let result = ALL_PRODUCTS;

    // Step 1: Filter by category
    if (selectedCategory !== 'All') {
      result = result.filter(p => p.category === selectedCategory);
    }

    // Step 2: Filter by search term
    if (search.trim()) {
      const lower = search.toLowerCase();
      result = result.filter(p => p.name.toLowerCase().includes(lower));
    }

    // Step 3: Filter in-stock only
    if (showInStockOnly) {
      result = result.filter(p => p.inStock);
    }

    // Step 4: Sort
    result = [...result].sort((a, b) => {
      if (sortBy === 'price') return a.price - b.price;
      if (sortBy === 'rating') return b.rating - a.rating;
      return a.name.localeCompare(b.name);
    });

    return result;
  }, [search, selectedCategory, sortBy, showInStockOnly]);
  //   ↑ Dependencies — only re-run when these change

  // ── Also memoize derived statistics ──
  const stats = useMemo(() => ({
    total: filteredAndSorted.length,
    avgPrice: filteredAndSorted.length
      ? Math.round(
          filteredAndSorted.reduce((sum, p) => sum + p.price, 0) /
          filteredAndSorted.length
        )
      : 0,
    inStockCount: filteredAndSorted.filter(p => p.inStock).length,
  }), [filteredAndSorted]);

  return (
    <View style={styles.container}>

      {/* This counter causes re-renders but should NOT re-run the filter */}
      <TouchableOpacity
        style={styles.counterBtn}
        onPress={() => setUnrelatedCounter(c => c + 1)}
      >
        <Text style={styles.counterText}>
          Unrelated counter: {unrelatedCounter} (tap me — filter won't re-run!)
        </Text>
      </TouchableOpacity>

      <TextInput
        style={styles.searchInput}
        value={search}
        onChangeText={setSearch}
        placeholder="Search products..."
        placeholderTextColor="#aaa"
      />

      {/* Category chips */}
      <View style={styles.chips}>
        {CATEGORIES.map(cat => (
          <TouchableOpacity
            key={cat}
            style={[styles.chip, selectedCategory === cat && styles.chipActive]}
            onPress={() => setSelectedCategory(cat)}
          >
            <Text style={[styles.chipText, selectedCategory === cat && styles.chipTextActive]}>
              {cat}
            </Text>
          </TouchableOpacity>
        ))}
      </View>

      {/* Sort buttons */}
      <View style={styles.sortRow}>
        {['name', 'price', 'rating'].map(s => (
          <TouchableOpacity
            key={s}
            style={[styles.sortBtn, sortBy === s && styles.sortBtnActive]}
            onPress={() => setSortBy(s)}
          >
            <Text style={[styles.sortText, sortBy === s && styles.sortTextActive]}>
              {s.charAt(0).toUpperCase() + s.slice(1)}
            </Text>
          </TouchableOpacity>
        ))}
      </View>

      {/* Stats — also memoized */}
      <View style={styles.statsBar}>
        <Text style={styles.statText}>{stats.total} results</Text>
        <Text style={styles.statText}>Avg: NPR {stats.avgPrice}</Text>
        <Text style={styles.statText}>In Stock: {stats.inStockCount}</Text>
      </View>

      <FlatList
        data={filteredAndSorted}
        keyExtractor={item => item.id}
        renderItem={({ item }) => (
          <View style={styles.productCard}>
            <View style={{ flex: 1 }}>
              <Text style={styles.productName}>{item.name}</Text>
              <Text style={styles.productMeta}>{item.category} • ⭐ {item.rating}</Text>
            </View>
            <View style={styles.priceCol}>
```

## 4. `useCallback` — Remembering Functions Across Renders

### The Problem It Solves

In JavaScript, functions are objects. And a very fundamental property of objects in JavaScript is that two objects with identical content are not considered equal unless they are the exact same object in memory. This means that every time a component re-renders, every function defined inside it is a **brand new function** — even if its code is identical to the previous render.

```
┌────────────────────────────────────────────────────────────────┐
│              Why Functions Change on Every Render              │
│                                                                │
│  Render 1:                                                     │
│  const handlePress = () => doSomething();                      │
│  → handlePress is stored at memory address 0x001              │
│                                                                │
│  Render 2 (some unrelated state changed):                      │
│  const handlePress = () => doSomething();                      │
│  → handlePress is stored at memory address 0x002  ← DIFFERENT!│
│                                                                │
│  0x001 === 0x002  →  false                                     │
│  Even though the code is identical, it's a new function.      │
│                                                                │
│  If this function is passed to a child component,             │
│  the child sees a "new" prop and re-renders needlessly.       │
└────────────────────────────────────────────────────────────────┘
```

This becomes a real performance concern when you pass callback functions as props to child components that are wrapped with `React.memo`. `React.memo` is a higher-order component that tells React to skip re-rendering a child if its props have not changed. But if the parent re-renders and creates a new function every time, `React.memo` always sees new props and always re-renders the child anyway — defeating the entire purpose.

`useCallback` solves this by **memoizing the function itself**. It returns the same function reference across renders, as long as its dependencies have not changed.

### The Relationship Between useMemo and useCallback

`useCallback` and `useMemo` are closely related. In fact, `useCallback(fn, deps)` is equivalent to `useMemo(() => fn, deps)`. The difference is purely syntactic: `useMemo` memoizes **the return value of a function**, while `useCallback` memoizes **the function itself**.

```
┌──────────────────────────────────────────────────────────┐
│  useMemo  vs  useCallback                                │
│                                                          │
│  useMemo(() => computeValue(a, b), [a, b])               │
│  → Returns: the VALUE that computeValue returns          │
│  → Use for: expensive calculations, derived data         │
│                                                          │
│  useCallback((id) => deleteItem(id), [deleteItem])       │
│  → Returns: the FUNCTION itself (stable reference)       │
│  → Use for: event handlers passed to child components    │
└──────────────────────────────────────────────────────────┘
```

### Use Cases

`useCallback` is most valuable when passing event handlers or callbacks to child components wrapped in `React.memo`, when passing callbacks to `useEffect` as dependencies (to prevent the effect from re-running every render), and when passing handlers to `FlatList`'s `renderItem` or similar performance-sensitive APIs.

```jsx
        data={tasks}
        keyExtractor={item => item.id}
        renderItem={({ item }) => (
          <TaskItem
            task={item}
            onToggle={handleToggle}  // stable reference — memo works!
            onDelete={handleDelete}  // stable reference — memo works!
          />
        )}
        ListEmptyComponent={
          <Text style={styles.empty}>All tasks completed! 🎉</Text>
```

## 5. `useRef` — A Box That Persists Without Causing Re-renders

### The Problem It Solves

`useRef` solves two distinct but related problems that neither `useState` nor any other hook addresses.

The first problem is the need for **mutable values that should not trigger re-renders**. Sometimes you need to store a piece of information that your component needs to remember — a timer ID, a previous value, a flag — but updating this information should not cause the component to re-render. With `useState`, every update triggers a re-render. With `useRef`, you can mutate a value freely and React never re-renders because of it.

The second problem is gaining **direct access to a native element**. Sometimes you need to programmatically focus a text input, scroll a list to a specific position, measure an element's dimensions, or call a method on a child component. These are operations that cannot be done through props and state — you need a direct handle to the underlying DOM or native component instance. `useRef` provides this handle.

### How It Works

`useRef` returns a **ref object** — a plain JavaScript object with a single property called `current`. On the first render, `current` is initialized to whatever value you pass to `useRef()`. Unlike state, mutating `current` directly (`ref.current = newValue`) does not trigger a re-render, and the value persists across renders without being reset.

```
┌────────────────────────────────────────────────────────────────┐
│                 useRef — Two Primary Use Cases                 │
│                                                                │
│  USE CASE 1: Storing a mutable value without re-render         │
│                                                                │
│  const timerRef = useRef(null);                                │
│                                                                │
│  timerRef.current = setInterval(fn, 1000);  // set value       │
│  clearInterval(timerRef.current);           // read value      │
│  timerRef.current = null;                   // reset value     │
│  → Component never re-renders because of this                  │
│                                                                │
│  USE CASE 2: Direct access to a native element                 │
│                                                                │
│  const inputRef = useRef(null);                                │
│  <TextInput ref={inputRef} />                                  │
│                                                                │
│  inputRef.current.focus();   // programmatically focus input   │
│  inputRef.current.blur();    // programmatically unfocus       │
│  inputRef.current.clear();   // clear input text               │
│  → inputRef.current is now the actual native TextInput node    │
└────────────────────────────────────────────────────────────────┘
```

### Use Cases

`useRef` is used for: programmatically focusing and blurring `TextInput` components, storing interval or timeout IDs for cleanup, tracking the previous value of a state variable, maintaining a flag (`hasMounted`, `isFetching`) that controls logic without triggering renders, referencing `FlatList` to call `scrollToIndex`, and holding any mutable value that your component's logic depends on but does not need to display.

```jsx
import { useState, useEffect, useRef, useCallback } from 'react';

  View, Text, TextInput, TouchableOpacity,
  FlatList, StyleSheet, Animated

  const [name, setName] = useState('');
  const [email, setEmail] = useState('');
  const [phone, setPhone] = useState('');
  const [stopwatchTime, setStopwatchTime] = useState(0);
  const [isRunning, setIsRunning] = useState(false);
  const [renderCount, setRenderCount] = useState(0);

  // ── Ref Use Case 1: Direct native element access ──
  const nameRef = useRef(null);
  const emailRef = useRef(null);
  const phoneRef = useRef(null);

  // ── Ref Use Case 2: Storing timer ID without triggering re-renders ──
  const intervalRef = useRef(null);

  // ── Ref Use Case 3: Counting renders without triggering a re-render ──
  // If we used useState for this, incrementing it would cause another render,
  // which would increment it again — an infinite loop!
  const renderCountRef = useRef(0);

  // ── Ref Use Case 4: Storing the previous value of state ──
  const prevNameRef = useRef('');

  // Track renders using a ref (does not cause re-render)
  renderCountRef.current += 1;

  // Store previous name before each render
  useEffect(() => {
    prevNameRef.current = name;
  }); // no dependency array — runs after every render

  // Auto-focus the first field on mount
  useEffect(() => {
    const timer = setTimeout(() => nameRef.current?.focus(), 300);
    return () => clearTimeout(timer);
  }, []);

  // Move focus to next field when step changes
  useEffect(() => {
    if (step === 2) emailRef.current?.focus();
    if (step === 3) phoneRef.current?.focus();
  }, [step]);

    if (isRunning) {
      // Stop: clear the interval using the stored ref
      clearInterval(intervalRef.current);
      intervalRef.current = null;
      setIsRunning(false);
    } else {
      // Start: store interval ID in ref (NOT in state — no re-render needed)
      intervalRef.current = setInterval(() => {
        setStopwatchTime(prev => prev + 1);
      }, 100);
      setIsRunning(true);
    }
  }, [isRunning]);

  // Reset stopwatch
  const resetStopwatch = useCallback(() => {
    clearInterval(intervalRef.current);
    intervalRef.current = null;
    setIsRunning(false);
    setStopwatchTime(0);
  }, []);

  useEffect(() => {
    return () => clearInterval(intervalRef.current);
  }, []);

    const s = Math.floor(tenths / 10);

    const m = Math.floor(s / 60);

  };

  return (
    <View style={styles.container}>

      {/* Render counter (via ref — incrementing it doesn't cause re-renders) */}
      <View style={styles.badge}>

        </Text>
      </View>

      {/* Multi-step form with programmatic focus */}
      <View style={styles.card}>
        <Text style={styles.cardTitle}>Multi-Step Form (Auto-Focus)</Text>
        <Text style={styles.helper}>
          Each field auto-focuses when you proceed to the next step.
        </Text>

        <TextInput
          ref={nameRef}
          style={[styles.input, step !== 1 && styles.inputDone]}
          value={name}
          onChangeText={setName}
          placeholder="Step 1: Full Name"
          placeholderTextColor="#aaa"
          editable={step === 1}
          returnKeyType="next"
          onSubmitEditing={() => name && setStep(2)}
        />
        {name.length > 0 && step > 1 && (
          <Text style={styles.prevValueHint}>
            Previous name: "{prevNameRef.current}"
          </Text>
        )}

        <TextInput
          ref={emailRef}
          style={[styles.input, step === 3 && styles.inputDone]}
          value={email}
          onChangeText={setEmail}
          placeholder="Step 2: Email Address"
          placeholderTextColor="#aaa"
          editable={step === 2}
          keyboardType="email-address"
          autoCapitalize="none"
          returnKeyType="next"
          onSubmitEditing={() => email && setStep(3)}
        />

        <TextInput
          ref={phoneRef}
          style={styles.input}
          value={phone}
          onChangeText={setPhone}
          placeholder="Step 3: Phone Number"
          placeholderTextColor="#aaa"
          editable={step === 3}
          keyboardType="phone-pad"
          returnKeyType="done"
        />

        <View style={styles.stepBtns}>
          {step > 1 && (
            <TouchableOpacity style={styles.backBtn} onPress={() => setStep(s => s - 1)}>
```

# Complete Summary & Decision Guide

```
┌───────────────────────────────────────────────────────────────────────┐
│                    HOOKS DECISION GUIDE                               │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  "I need to store a value and re-render when it changes"             │
│   → useState                                                          │
│                                                                       │
│  "I need to run code after render (API call, timer, subscription)"   │
│   → useEffect                                                         │
│     - Once on mount:          useEffect(() => {}, [])                │
│     - When X changes:         useEffect(() => {}, [x])               │
│     - Always:                 useEffect(() => {})  ← rarely needed   │
│     - With cleanup:           return () => cleanup()                  │
│                                                                       │
│  "I have an expensive calculation that depends on some values"       │
│   → useMemo                                                           │
│     const result = useMemo(() => expensiveCalc(a, b), [a, b])        │
│                                                                       │
│  "I pass a function to a memoized child and don't want it to         │
│   re-render unnecessarily"                                           │
│   → useCallback                                                       │
│     const fn = useCallback(() => doThing(id), [id])                  │
│                                                                       │
│  "I need to access a native element (TextInput, FlatList)"           │
│   → useRef (attach to component via ref prop)                        │
│                                                                       │
│  "I need to store a value that should NOT cause re-renders"          │
│   → useRef (read/write via ref.current)                              │
│                                                                       │
├───────────────────────────────────────────────────────────────────────┤
│  useMemo  vs  useCallback                                             │
│  ─────────────────────────────────────────────────────────────────   │
│  useMemo    → memoizes a RETURNED VALUE   (data, objects, arrays)    │
│  useCallback → memoizes the FUNCTION ITSELF (event handlers, fns)    │
│                                                                       │
│  useState vs useRef (for storing values)                             │
│  ─────────────────────────────────────────────────────────────────   │
│  useState  → value displayed in UI → needs re-render on change       │
│  useRef    → value used in logic only → no re-render needed          │
└───────────────────────────────────────────────────────────────────────┘
```
