# React Native Storage: A Complete Deep Dive

> A comprehensive guide to Secure Storage, File Storage, and SQLite in React Native — explained from first principles for absolute beginners, written in the voice of an experienced engineer.

---

## Preface — Why Storage Matters

Every meaningful mobile application needs to remember things. It might need to remember that the user has already logged in, so it does not ask for credentials every time the app is opened. It might need to cache a list of articles so the user can read them even when offline. It might need to store thousands of financial transactions that can be queried, filtered, and sorted at lightning speed. Or it might need to hold a user's profile photo that was taken with the device's camera.

These are fundamentally different kinds of data — a login token, a cached list, a large dataset, and a binary image file — and they each demand a different storage solution. Choosing the wrong tool for the job leads to security vulnerabilities, poor performance, wasted battery life, and frustrated users. This guide will equip you with a thorough understanding of the three most important storage mechanisms available in React Native: **Secure Storage**, **File Storage**, and **SQLite**.

# Part I — Secure Storage

## Chapter 1: What Is Secure Storage and Why Is It Different?

When we talk about "storage" in general, most developers immediately think of saving data to a file or a database. But there is a category of data that is so sensitive — authentication tokens, passwords, private keys, biometric credentials, API secrets — that storing it in an ordinary file or database is simply not acceptable. If a malicious application or an attacker with physical access to the device were to read an ordinary file, they could steal these secrets and cause catastrophic harm.

**Secure Storage** is a completely different class of storage mechanism. Instead of writing data to a file on the filesystem, it uses the hardware-backed security modules built into modern smartphones to store data in an encrypted vault that is tied to the specific device and, optionally, to the user's biometric identity. On iOS, this is called the **Keychain** — a service provided by the operating system that encrypts data using keys stored in the device's Secure Enclave, a dedicated security chip that is physically isolated from the main processor. On Android, it is called the **Keystore** system, which similarly stores cryptographic keys in a tamper-resistant hardware component. The critical property of both systems is that the encryption keys _never leave the secure hardware_ — not even your own application code can read them directly.


The most popular library for Secure Storage in React Native is `expo-secure-store`, which is part of the Expo ecosystem but works in plain React Native projects as well. For projects that do not use Expo at all, `react-native-keychain` is an excellent alternative that provides similar functionality with more granular control over Keychain access groups and biometric authentication.

## Chapter 2: The Three Core Operations — Set, Get, Delete

The API for secure storage is intentionally kept simple, because simplicity reduces the chance of making security mistakes. There are essentially three operations: writing a value (`setItemAsync`), reading a value (`getItemAsync`), and deleting a value (`deleteItemAsync`). All three are asynchronous — they return Promises — because communicating with the secure hardware involves system calls that can take a small but non-trivial amount of time.

It is important to understand that Secure Storage stores data as **string key-value pairs**. This means that if you want to store a complex JavaScript object, you must first serialize it to a JSON string using `JSON.stringify()`, and then parse it back using `JSON.parse()` when you read it. This is a very common pattern and you will use it constantly.

```

│  ┌─────────────────┬──────────────────────────────┐  │

│  ├─────────────────┼──────────────────────────────┤  │
│  │  "auth_token"   │  "eyJhbGciOiJIUzI1NiJ9..."   │  │
│  │  "user_id"      │  "usr_8f3a2b1c"              │  │

```

## Chapter 3: Access Control Levels

One of the most powerful features of Secure Storage — and one that is frequently overlooked by beginners — is the ability to set **access control levels** on stored items. These levels dictate _under what circumstances_ the stored data can be read back. The default level allows the data to be read any time the app is in the foreground. But you can configure it to require the user to authenticate with Face ID, Touch ID, or their device PIN before the data is revealed. This is exactly how banking applications protect your account balance and transaction history — the data is stored in the secure vault, but it cannot be accessed without biometric confirmation.

```
Access Control Levels (from least to most secure)
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  AFTER_FIRST_UNLOCK          ◄── Default. Data accessible   │
│  (least restrictive)              after device is unlocked  │
│         │                         once after reboot.        │
│         │                                                   │
│         ▼                                                   │
│  WHEN_UNLOCKED               ◄── Data only accessible       │
│                                   while device is unlocked. │
│         │                                                   │
│         ▼                                                   │
│  WHEN_PASSCODE_SET_THIS_     ◄── Requires device passcode   │
│  DEVICE_ONLY                      to be set. Data lost if   │
│         │                         passcode removed.         │
│         ▼                                                   │
│  BIOMETRIC_ANY               ◄── Requires Face ID or        │
│  (most restrictive)               Touch ID to read data.    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Chapter 4: Complete Example — Authentication Token Manager

The following example demonstrates a complete, production-ready authentication token management system. This is the exact pattern used in real-world applications to handle JWT (JSON Web Token) authentication securely.

```javascript
// Installation:
// expo install expo-secure-store
// OR for bare React Native:
// npm install react-native-keychain

import * as SecureStore from 'expo-secure-store';

// ─────────────────────────────────────────────────────────────────────────────
// TOKEN MANAGER
// A clean abstraction layer over SecureStore. By wrapping the library in a
// manager class, you avoid scattering raw SecureStore calls throughout your
// codebase, making it easy to swap implementations if needed.
// ─────────────────────────────────────────────────────────────────────────────

const KEYS = {
  ACCESS_TOKEN:  'auth.access_token',
  REFRESH_TOKEN: 'auth.refresh_token',
  USER_PROFILE:  'auth.user_profile',
};

export const TokenManager = {

  // ── Save Tokens After Login ────────────────────────────────────────────────
  // Called right after a successful login API response.
  // We store both the access token and refresh token securely.
  saveTokens: async (accessToken, refreshToken) => {
    try {
      // SecureStore.setItemAsync(key, value, options?)
      // The options object lets us specify access control.
      await SecureStore.setItemAsync(
        KEYS.ACCESS_TOKEN,
        accessToken,
        {
          // WHEN_UNLOCKED means this data is only accessible while
          // the device screen is unlocked — appropriate for access tokens.
          keychainAccessible: SecureStore.WHEN_UNLOCKED,
        }
      );

      await SecureStore.setItemAsync(
        KEYS.REFRESH_TOKEN,
        refreshToken,
        {
          // Refresh tokens are long-lived and more sensitive,
          // so we require biometric authentication to read them.
          requireAuthentication: true,
        }
      );

      console.log('✅ Tokens saved securely.');
    } catch (error) {
      console.error('❌ Failed to save tokens:', error);
      throw error;
    }
  },

  // ── Save User Profile ──────────────────────────────────────────────────────
  // Demonstrates storing a complex object.
  // SecureStore only accepts strings, so we JSON.stringify() the object.
  saveUserProfile: async (userProfile) => {
    try {
      const profileString = JSON.stringify(userProfile);
      await SecureStore.setItemAsync(KEYS.USER_PROFILE, profileString);
    } catch (error) {
      console.error('❌ Failed to save profile:', error);
      throw error;
    }
  },

  // ── Read Access Token ──────────────────────────────────────────────────────
  // Returns null if the token doesn't exist (e.g., user never logged in)
  // or if an error occurs.
  getAccessToken: async () => {
    try {
      const token = await SecureStore.getItemAsync(KEYS.ACCESS_TOKEN);
      return token; // null if not found
    } catch (error) {
      console.error('❌ Failed to read access token:', error);
      return null;
    }
  },

  // ── Read User Profile ──────────────────────────────────────────────────────
  // Demonstrates reading and parsing a stored JSON object.
  getUserProfile: async () => {
    try {
      const profileString = await SecureStore.getItemAsync(KEYS.USER_PROFILE);
      if (!profileString) return null;
      return JSON.parse(profileString); // Parse back to object
    } catch (error) {
      console.error('❌ Failed to read profile:', error);
      return null;
    }
  },

  // ── Check If User Is Logged In ─────────────────────────────────────────────
  // A convenience helper used on app startup to decide whether to show
  // the login screen or the main app.
  isAuthenticated: async () => {
    const token = await TokenManager.getAccessToken();
    return token !== null;
  },

  // ── Clear All Auth Data on Logout ─────────────────────────────────────────
  // ALWAYS clear all stored auth data on logout.
  // Forgetting this step is a serious security vulnerability.
  logout: async () => {
    try {
      await SecureStore.deleteItemAsync(KEYS.ACCESS_TOKEN);
      await SecureStore.deleteItemAsync(KEYS.REFRESH_TOKEN);
      await SecureStore.deleteItemAsync(KEYS.USER_PROFILE);
      console.log('✅ All auth data cleared.');
    } catch (error) {
      console.error('❌ Logout cleanup failed:', error);
    }
  },
};

// ─────────────────────────────────────────────────────────────────────────────
// USAGE IN A REACT COMPONENT
// ─────────────────────────────────────────────────────────────────────────────

import React, { useState, useEffect } from 'react';
import { View, Text, TouchableOpacity, StyleSheet, Alert } from 'react-native';

const AuthScreen = () => {
  const [authStatus, setAuthStatus] = useState('Checking...');

  // On mount: check if a token already exists (returning user)
  useEffect(() => {
    const checkAuth = async () => {
      const isAuth = await TokenManager.isAuthenticated();
      setAuthStatus(isAuth ? '🔐 Logged In' : '🔓 Not Logged In');
    };
    checkAuth();
  }, []);

  const handleLogin = async () => {
    // Simulate a login API call that returns tokens
    const fakeAccessToken  = 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjoiMTIzIn0.abc';
    const fakeRefreshToken = 'dGhpc19pc19hX3JlZnJlc2hfdG9rZW4';
    const fakeProfile      = { id: '123', name: 'Alice', email: 'alice@example.com' };

    await TokenManager.saveTokens(fakeAccessToken, fakeRefreshToken);
    await TokenManager.saveUserProfile(fakeProfile);
    setAuthStatus('🔐 Logged In');
    Alert.alert('Success', 'Tokens saved securely in device Keychain!');
  };

  const handleReadToken = async () => {
    const token = await TokenManager.getAccessToken();
    Alert.alert('Access Token', token || 'No token found.');
  };

  const handleLogout = async () => {
    await TokenManager.logout();
    setAuthStatus('🔓 Not Logged In');
  };

  return (
    <View style={styles.container}>
      <Text style={styles.status}>{authStatus}</Text>
      <TouchableOpacity style={styles.btn} onPress={handleLogin}>
        <Text style={styles.btnText}>Login (Save Tokens)</Text>
      </TouchableOpacity>
      <TouchableOpacity style={styles.btn} onPress={handleReadToken}>
        <Text style={styles.btnText}>Read Access Token</Text>
      </TouchableOpacity>
      <TouchableOpacity style={[styles.btn, styles.danger]} onPress={handleLogout}>
        <Text style={styles.btnText}>Logout (Clear All)</Text>
      </TouchableOpacity>
    </View>
  );
};

const styles = StyleSheet.create({
  container: { flex: 1, justifyContent: 'center', alignItems: 'center', padding: 20 },
  status: { fontSize: 20, fontWeight: 'bold', marginBottom: 30, color: '#333' },
  btn: {
    width: '100%', backgroundColor: '#6200EE', padding: 14,
    borderRadius: 8, alignItems: 'center', marginBottom: 12,
  },
  danger: { backgroundColor: '#B00020' },
  btnText: { color: '#FFF', fontWeight: 'bold', fontSize: 15 },
});

export default AuthScreen;
```

# Part II — File Storage

## Chapter 5: Understanding the Mobile Filesystem

Before writing a single line of file storage code, you need to understand how the mobile operating system organises the filesystem and what areas of it your application is permitted to access. On both iOS and Android, every application runs in a **sandboxed environment**, which means it is confined to its own private directory and cannot read or write files belonging to other applications. This is a fundamental security feature of mobile operating systems.

Within your application's sandbox, there are several distinct directories, each with a different purpose and different persistence characteristics.

```
Application Sandbox Filesystem Layout
┌──────────────────────────────────────────────────────────────────┐
│                   App Sandbox (Private)                          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  documentDirectory                                      │     │
│  │  /Documents/                                            │     │
│  │  ─────────────────────────────────────────────────────  │     │
│  │  ✔ Persistent — survives app restarts & updates         │     │
│  │  ✔ Backed up to iCloud (iOS) / Google Drive (Android)   │     │
│  │  ✔ User-generated files, exports, downloaded content    │     │
│  │  Example: user_report.pdf, downloaded_book.epub         │     │
│  └─────────────────────────────────────────────────────────┘     │
│                                                                  │
```

The primary library for filesystem operations in React Native is `expo-file-system`. It provides a comprehensive API for reading, writing, copying, moving, and deleting files and directories, as well as downloading files from the network with progress reporting.

## Chapter 6: The Core File Operations

Every filesystem API, no matter how complex, ultimately reduces to a small set of fundamental operations: **write**, **read**, **check existence**, **delete**, **copy**, **move**, and **create directories**. Understanding these primitives deeply allows you to build any file-based feature you can imagine.

A file path in `expo-file-system` is always an absolute URI string. You construct file paths by concatenating a base directory constant (like `FileSystem.documentDirectory`) with a filename. For example, `FileSystem.documentDirectory + 'notes.txt'` gives you the full path to a file named `notes.txt` in the documents directory.

```
File Path Construction

│                                                             │

│   {UUID}/Documents/"       ◄── iOS example                  │
│                                                             │
│  + "my_notes.txt"                                           │
│  ══════════════════════════                                 │
```

## Chapter 7: Reading and Writing Text Files

The most fundamental file operation is writing and reading text. `expo-file-system` provides `writeAsStringAsync` and `readAsStringAsync` for working with text content, with support for multiple encodings including UTF-8 (the default, for normal text) and Base64 (for binary data encoded as text).

```javascript
import * as FileSystem from 'expo-file-system';

// ─────────────────────────────────────────────────────────────────────────────
// TEXT FILE OPERATIONS
// ─────────────────────────────────────────────────────────────────────────────

const NotesManager = {

  // ── Write a text file ─────────────────────────────────────────────────────
  // FileSystem.writeAsStringAsync(uri, content, options?)
  // If the file already exists, it is overwritten by default.
  saveNote: async (filename, content) => {
    const path = FileSystem.documentDirectory + filename;
    try {
      await FileSystem.writeAsStringAsync(path, content, {
        encoding: FileSystem.EncodingType.UTF8,
      });
      console.log(`✅ Note saved to: ${path}`);
      return path;
    } catch (error) {
      console.error('❌ Write failed:', error);
      throw error;
    }
  },

  // ── Read a text file ──────────────────────────────────────────────────────
  // Returns the file contents as a string.
  readNote: async (filename) => {
    const path = FileSystem.documentDirectory + filename;
    try {
      // ALWAYS check existence before reading — reading a non-existent
      // file throws an error, and checking first gives a cleaner UX.
      const info = await FileSystem.getInfoAsync(path);
      if (!info.exists) {
        console.warn('File does not exist:', path);
        return null;
      }
      const content = await FileSystem.readAsStringAsync(path, {
        encoding: FileSystem.EncodingType.UTF8,
      });
      return content;
    } catch (error) {
      console.error('❌ Read failed:', error);
      throw error;
    }
  },

  // ── Get file metadata ─────────────────────────────────────────────────────
  // getInfoAsync returns: { exists, uri, size, isDirectory, modificationTime }
  // This is extremely useful for checking freshness of cached data.
  getFileInfo: async (filename) => {
    const path = FileSystem.documentDirectory + filename;
    const info = await FileSystem.getInfoAsync(path);
    /*
      info = {
        exists: true,
        uri: "file:///.../.../notes.txt",
        size: 1024,           // bytes
        isDirectory: false,
        modificationTime: 1712345678.123  // Unix timestamp
      }
    */
    return info;
  },

  // ── Delete a file ─────────────────────────────────────────────────────────
  deleteNote: async (filename) => {
    const path = FileSystem.documentDirectory + filename;
    try {
      const info = await FileSystem.getInfoAsync(path);
      if (info.exists) {
        await FileSystem.deleteAsync(path);
        console.log('✅ File deleted.');
      }
    } catch (error) {
      console.error('❌ Delete failed:', error);
    }
  },

  // ── List all files in the documents directory ─────────────────────────────
  listNotes: async () => {
    try {
      const files = await FileSystem.readDirectoryAsync(
        FileSystem.documentDirectory
      );
      return files; // Array of filenames (strings)
    } catch (error) {
      console.error('❌ List failed:', error);
      return [];
    }
  },
};
```

## Chapter 8: Downloading Files from the Network

One of the most powerful capabilities of `expo-file-system` is the ability to download files from a remote URL and save them directly to the device filesystem. This is how applications implement features like offline reading, downloadable content packs, or local media libraries. The `downloadAsync` function handles the entire download pipeline — making the HTTP request, streaming the response, and writing the bytes to disk — in a single call.

For large files, you can use `FileSystem.createDownloadResumable()`, which gives you fine-grained control: you can track download progress with a callback, pause the download, resume it later, and even survive app restarts by persisting the resumable download state.

```javascript
import * as FileSystem from 'expo-file-system';
import { useState } from 'react';

// ── Simple One-Shot Download ─────────────────────────────────────────────────
const downloadPDF = async (url, filename) => {
  const destination = FileSystem.documentDirectory + filename;
  try {
    const { uri, status } = await FileSystem.downloadAsync(url, destination);
    if (status === 200) {
      console.log(`✅ Downloaded to: ${uri}`);
      return uri;
    } else {
      throw new Error(`Download failed with HTTP status: ${status}`);
    }
  } catch (error) {
    console.error('❌ Download error:', error);
    throw error;
  }
};

// ── Resumable Download with Progress Tracking ────────────────────────────────
// This is the professional way to download large files.
const useResumableDownload = (url, filename) => {
  const [progress, setProgress] = useState(0);
  const [downloadRef, setDownloadRef] = useState(null);

  const startDownload = async () => {
    const destination = FileSystem.cacheDirectory + filename;

    // The progress callback fires periodically during the download.
    // totalBytesWritten is how much has been received so far.
    // totalBytesExpectedToWrite is the total file size.
    const callback = (downloadProgress) => {
      const percent =
        downloadProgress.totalBytesWritten /
        downloadProgress.totalBytesExpectedToWrite;
      setProgress(Math.round(percent * 100));
    };

    // createDownloadResumable prepares the download but does NOT start it yet.
    const downloadResumable = FileSystem.createDownloadResumable(
      url,
      destination,
      {},        // options (headers, etc.)
      callback   // progress callback
    );

    setDownloadRef(downloadResumable);

    try {
      const { uri } = await downloadResumable.downloadAsync();
      console.log('✅ Download complete:', uri);
      return uri;
    } catch (error) {
      console.error('❌ Resumable download error:', error);
    }
  };

  const pauseDownload = async () => {
    if (downloadRef) {
      await downloadRef.pauseAsync();
    }
  };

  const resumeDownload = async () => {
    if (downloadRef) {
      await downloadRef.resumeAsync();
    }
  };

  return { progress, startDownload, pauseDownload, resumeDownload };
};

/*
Download Progress Flow:
┌──────────────────────────────────────────────────────┐
│                                                      │
│  createDownloadResumable(url, dest, {}, callback)    │
│                    │                                 │
│                    ▼                                 │
│             downloadAsync()                          │
│                    │                                 │
│         ┌──────────▼──────────┐                      │
│         │   Streaming bytes   │                      │
│         │   from server...    │                      │
│         └──────────┬──────────┘                      │
│                    │                                 │
│  callback fires → progress: 0% → 10% → ... → 100%    │
│                    │                                 │
│                    ▼                                 │
│          File fully written to disk ✅               │
│                                                      │
└──────────────────────────────────────────────────────┘
*/
```

## Chapter 9: Working with Directories

Real applications often need to organise their files into subdirectories rather than dumping everything in the root of the documents folder. `expo-file-system` provides `makeDirectoryAsync` for creating directories (with an option to create all intermediate parent directories in one call, similar to the Unix `mkdir -p` command), and `readDirectoryAsync` for listing directory contents.

```javascript
import * as FileSystem from 'expo-file-system';

// ── Directory Management ─────────────────────────────────────────────────────

const setupAppDirectories = async () => {
  const dirs = [
    FileSystem.documentDirectory + 'reports/',
    FileSystem.documentDirectory + 'exports/',
    FileSystem.cacheDirectory   + 'thumbnails/',
    FileSystem.cacheDirectory   + 'api_cache/',
  ];

  for (const dir of dirs) {
    const info = await FileSystem.getInfoAsync(dir);
    if (!info.exists) {
      // intermediates: true creates parent dirs if they don't exist
      await FileSystem.makeDirectoryAsync(dir, { intermediates: true });
      console.log(`📁 Created: ${dir}`);
    }
  }
};

// ── Copying and Moving Files ─────────────────────────────────────────────────
// Copy leaves the source intact; move (rename) removes the source.
```

# Part III — SQLite

## Chapter 11: Why SQLite? The Case for a Relational Database on Mobile

Both Secure Storage and File Storage have clear use cases, but they share a fundamental limitation: they are not designed for querying. Secure Storage is excellent for a handful of secrets, but you cannot ask it "give me all tokens that were created in the last 30 days." File Storage is great for large blobs of data, but you cannot ask a folder of JSON files "find all products in the Electronics category with a price below $50, sorted by rating." For these kinds of structured data queries, you need a **relational database**.

**SQLite** is the answer. It is the world's most widely deployed database engine — embedded in every iOS and Android device, every web browser, and most operating systems. Unlike server-side databases such as PostgreSQL or MySQL, SQLite is **serverless** — the entire database is stored in a single file on the local filesystem, and the database engine runs directly inside your application process. There is no network connection, no authentication, no separate process to manage. For a mobile application, this is ideal.

In React Native, the modern standard for SQLite is the `expo-sqlite` library, which in its v11+ version provides a vastly improved synchronous API (`useSQLiteContext` hook) alongside the traditional asynchronous API.


## Chapter 12: Opening a Database and Running Migrations

The very first thing you do with SQLite is **open the database**. If the database file does not yet exist, SQLite creates it automatically. Once opened, you need to create your tables — this is done using **migrations**, which are versioned SQL scripts that describe how the database schema should evolve over time. `expo-sqlite` supports migrations natively through the `migrateDbIfNeededAsync` pattern introduced in newer versions, but the classic approach uses `CREATE TABLE IF NOT EXISTS` statements in a setup function.

```javascript
// Installation:
// expo install expo-sqlite

import * as SQLite from 'expo-sqlite';

// ─────────────────────────────────────────────────────────────────────────────
// DATABASE INITIALISATION
// The database file is stored in the app's documents directory automatically.
// ─────────────────────────────────────────────────────────────────────────────

let db = null; // Module-level singleton

export const getDatabase = () => {
  if (!db) {
    // openDatabaseSync returns a synchronous database connection.
    // The file "app.db" is created if it does not exist.
    db = SQLite.openDatabaseSync('app.db');
  }
  return db;
};

// ── Run Migrations: Create Tables ─────────────────────────────────────────────
// This function should be called ONCE, early in the application's lifecycle
// (e.g., in App.js before rendering the main navigator).
// execSync runs a SQL statement and ignores the result.
export const initDatabase = () => {
  const db = getDatabase();

  // execSync is perfect for DDL (Data Definition Language) statements
  // like CREATE TABLE, CREATE INDEX, etc.
  db.execSync(`
    PRAGMA journal_mode = WAL;
    -- WAL (Write-Ahead Logging) improves write performance significantly.
    -- It allows concurrent reads while a write is in progress.

    CREATE TABLE IF NOT EXISTS users (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      name        TEXT    NOT NULL,
      email       TEXT    NOT NULL UNIQUE,
      created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
      is_active   INTEGER NOT NULL DEFAULT 1
      -- SQLite has no BOOLEAN type; we use INTEGER (0 = false, 1 = true)
    );

    CREATE TABLE IF NOT EXISTS products (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      name        TEXT    NOT NULL,
      description TEXT,
      price       REAL    NOT NULL,
      category    TEXT    NOT NULL,
      stock       INTEGER NOT NULL DEFAULT 0,
      created_at  TEXT    NOT NULL DEFAULT (datetime('now'))
    );

    CREATE TABLE IF NOT EXISTS orders (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      user_id     INTEGER NOT NULL,
      total       REAL    NOT NULL,
      status      TEXT    NOT NULL DEFAULT 'pending',
      created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
      FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
    );

    CREATE TABLE IF NOT EXISTS order_items (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      order_id    INTEGER NOT NULL,
      product_id  INTEGER NOT NULL,
      quantity    INTEGER NOT NULL,
      unit_price  REAL    NOT NULL,
      FOREIGN KEY (order_id)   REFERENCES orders(id)   ON DELETE CASCADE,
      FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
    );

    -- Indexes speed up queries that filter or sort by these columns.
    CREATE INDEX IF NOT EXISTS idx_products_category ON products(category);
    CREATE INDEX IF NOT EXISTS idx_orders_user_id    ON orders(user_id);
  `);

  console.log('✅ Database initialised successfully.');
};
```

## Chapter 13: SQL Statements — CRUD Operations in Depth

**CRUD** stands for **Create, Read, Update, Delete** — the four fundamental operations of any persistent storage system. In SQLite, these map directly to the SQL statements `INSERT`, `SELECT`, `UPDATE`, and `DELETE`. Understanding these operations deeply is the foundation of all database programming.

A critically important concept here is **parameterised queries** (also called prepared statements). Instead of concatenating user input directly into your SQL string — which creates a catastrophic security vulnerability called **SQL Injection** — you use `?` placeholders in your SQL and pass the actual values as a separate array. The database engine safely escapes and binds these values, making injection attacks impossible.

```
SQL Injection Risk (NEVER do this):
┌────────────────────────────────────────────────────────────────┐
│  BAD:                                                          │
│  db.runSync(`SELECT * FROM users WHERE email = '${userInput}'`)│
│                                                                │
│  If userInput = "' OR '1'='1"  ← SQL Injection attack!         │
│  The query becomes:                                            │
│  SELECT * FROM users WHERE email = '' OR '1'='1'               │
│  This returns ALL users — a total security breach! ❌          │
│                                                                │
│  GOOD (Parameterised):                                         │
│  db.runSync(                                                   │
│    'SELECT * FROM users WHERE email = ?',                      │
│    [userInput]   ← Value is safely escaped by SQLite           │
│  )  ✅                                                         │
└────────────────────────────────────────────────────────────────┘
```

```javascript
import * as SQLite from 'expo-sqlite';
import { getDatabase } from './database';

// ─────────────────────────────────────────────────────────────────────────────
// COMPLETE CRUD REPOSITORY
// A Repository is a design pattern that wraps all database operations
// for a specific entity (e.g., users, products) in one place.
// ─────────────────────────────────────────────────────────────────────────────

export const UserRepository = {

  // ── CREATE ─────────────────────────────────────────────────────────────────
  // INSERT INTO adds a new row to the table.
  // runSync is used for INSERT, UPDATE, DELETE — statements that modify data.
  // It returns a result object with `lastInsertRowId` and `changes`.
  insert: (name, email) => {
    const db = getDatabase();
    const result = db.runSync(
      'INSERT INTO users (name, email) VALUES (?, ?)',
      [name, email]
      // ↑ Parameterised! SQLite safely binds name and email.
    );
    console.log(`✅ User inserted with ID: ${result.lastInsertRowId}`);
    return result.lastInsertRowId; // Return the auto-generated primary key
  },

  // ── READ (single row) ──────────────────────────────────────────────────────
  // getFirstSync returns the FIRST matching row as a JavaScript object,
  // or null if no row matches. Perfect for lookups by primary key or unique field.
  findById: (id) => {
    const db = getDatabase();
    const user = db.getFirstSync(
      'SELECT id, name, email, created_at, is_active FROM users WHERE id = ?',
      [id]
    );
    return user; // { id: 1, name: "Alice", email: "...", ... } or null
  },

  findByEmail: (email) => {
    const db = getDatabase();
    return db.getFirstSync(
      'SELECT * FROM users WHERE email = ?',
      [email]
    );
  },

  // ── READ (multiple rows) ──────────────────────────────────────────────────
  // getAllSync returns ALL matching rows as an array of objects.
  findAll: () => {
    const db = getDatabase();
    return db.getAllSync(
      'SELECT * FROM users WHERE is_active = 1 ORDER BY name ASC'
    );
  },

  // ── READ with Search ──────────────────────────────────────────────────────
  // The LIKE operator with % wildcards performs a substring search.
  search: (query) => {
    const db = getDatabase();
    return db.getAllSync(
      'SELECT * FROM users WHERE name LIKE ? OR email LIKE ?',
      [`%${query}%`, `%${query}%`]
      // %query% means "query can appear anywhere in the string"
    );
  },

  // ── UPDATE ────────────────────────────────────────────────────────────────
  // UPDATE modifies existing rows. result.changes tells you how many rows
  // were actually updated (0 means the ID was not found).
  update: (id, name, email) => {
    const db = getDatabase();
    const result = db.runSync(
      'UPDATE users SET name = ?, email = ? WHERE id = ?',
      [name, email, id]
    );
    console.log(`✅ Updated ${result.changes} row(s).`);
    return result.changes > 0; // true if update was successful
  },

  // ── Soft Delete (recommended) ─────────────────────────────────────────────
  // Instead of permanently deleting data, we set is_active = 0.
  // This preserves referential integrity and allows data recovery.
  softDelete: (id) => {
    const db = getDatabase();
    const result = db.runSync(
      'UPDATE users SET is_active = 0 WHERE id = ?',
      [id]
    );
    return result.changes > 0;
  },

  // ── Hard Delete (permanent) ───────────────────────────────────────────────
  hardDelete: (id) => {
    const db = getDatabase();
    const result = db.runSync('DELETE FROM users WHERE id = ?', [id]);
    return result.changes > 0;
  },

  // ── Count ─────────────────────────────────────────────────────────────────
  count: () => {
    const db = getDatabase();
    const row = db.getFirstSync('SELECT COUNT(*) as total FROM users');
    return row.total;
  },
};
```

## Chapter 14: Transactions — The Gold Standard for Data Integrity

A **transaction** is a sequence of database operations that are treated as a single atomic unit. "Atomic" means that either _all_ operations in the transaction succeed, or _none_ of them do. If any step fails — due to a power failure, an application crash, a constraint violation, or any other error — the database is automatically rolled back to the state it was in before the transaction began.

This property, known as **atomicity**, is absolutely critical for maintaining data integrity in any application that performs multi-step writes. Consider the classic example of a bank transfer: you debit $100 from Account A and credit $100 to Account B. Without a transaction, it is possible for the debit to succeed but the crash to occur before the credit, leaving $100 permanently lost. With a transaction, both operations are guaranteed to succeed together or fail together.

```
Transaction: Atomicity Guarantee
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  WITHOUT TRANSACTION (dangerous):                            │
│  ──────────────────────────────────                          │
│  Step 1: Debit Account A  → SUCCESS ✅                       │
│  Step 2: Credit Account B → CRASH 💥  ← $100 LOST FOREVER    │
│                                                              │
│  WITH TRANSACTION (safe):                                    │
│  ────────────────────────                                    │
│  db.withTransactionSync(() => {                              │
│    Step 1: Debit Account A  → SUCCESS ✅                     │
│    Step 2: Credit Account B → CRASH 💥                       │
│  })                                                          │
│  ↓↓↓ ROLLBACK automatically triggered ↓↓↓                    │
│  Account A: unchanged ✅   Account B: unchanged ✅          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```javascript
// ─────────────────────────────────────────────────────────────────────────────
// TRANSACTIONS IN PRACTICE
// ─────────────────────────────────────────────────────────────────────────────

export const OrderRepository = {

  // ── Create an Order with Multiple Line Items ─────────────────────────────
  // This is a multi-step operation:
  //   1. Insert a row into the orders table
  //   2. Insert one row into order_items for EACH product in the cart
  //   3. Decrease the stock of each product
  // If any step fails, ALL steps must be rolled back.
  createOrder: (userId, cartItems) => {
    const db = getDatabase();

    // withTransactionSync wraps the callback in a BEGIN/COMMIT transaction.
    // If the callback throws, it automatically issues ROLLBACK.
    const orderId = db.withTransactionSync(() => {

      // Step 1: Calculate total
      const total = cartItems.reduce(
        (sum, item) => sum + item.price * item.quantity, 0
      );

      // Step 2: Insert the order header
      const orderResult = db.runSync(
        'INSERT INTO orders (user_id, total, status) VALUES (?, ?, ?)',
        [userId, total, 'pending']
      );
      const newOrderId = orderResult.lastInsertRowId;

      // Step 3: Insert each line item and deduct stock
      for (const item of cartItems) {
        // Insert line item
        db.runSync(
          'INSERT INTO order_items (order_id, product_id, quantity, unit_price) VALUES (?, ?, ?, ?)',
          [newOrderId, item.productId, item.quantity, item.price]
        );

        // Deduct stock
        const stockResult = db.runSync(
          'UPDATE products SET stock = stock - ? WHERE id = ? AND stock >= ?',
          [item.quantity, item.productId, item.quantity]
        );

        // If no rows were updated, the stock check failed — throw to trigger rollback
        if (stockResult.changes === 0) {
          throw new Error(
            `Insufficient stock for product ID: ${item.productId}`
          );
        }
      }

      return newOrderId;
    });

    console.log(`✅ Order created with ID: ${orderId}`);
    return orderId;
  },
};
```

## Chapter 15: Advanced Queries — JOINs, Aggregates, and Pagination

The real power of SQLite — the reason to use it over simple key-value storage — lies in its ability to express complex questions about your data in a single, efficient query. Understanding `JOIN`, `GROUP BY`, `HAVING`, `ORDER BY`, and `LIMIT`/`OFFSET` unlocks the full expressiveness of the relational model.

```javascript
export const ReportRepository = {

  // ── JOIN: Combine data from multiple tables ────────────────────────────────
  // This query fetches orders along with the name of the user who placed them.
  // An INNER JOIN returns only rows that have a match in BOTH tables.
  // A LEFT JOIN returns all rows from the left table even if no match exists.
  getOrdersWithUserNames: () => {
    const db = getDatabase();
    return db.getAllSync(`
      SELECT
        o.id          AS order_id,
        o.total,
        o.status,
        o.created_at,
        u.name        AS user_name,
        u.email       AS user_email
      FROM orders o
      INNER JOIN users u ON o.user_id = u.id
      ORDER BY o.created_at DESC
    `);
  },

  // ── Aggregate: GROUP BY + COUNT + SUM ─────────────────────────────────────
  // This gives us a summary per user: how many orders they've placed and
  // how much they've spent in total.
  getUserOrderStats: () => {
    const db = getDatabase();
    return db.getAllSync(`
      SELECT
        u.id,
        u.name,
        COUNT(o.id)   AS total_orders,
        SUM(o.total)  AS lifetime_value,
        AVG(o.total)  AS avg_order_value
      FROM users u
      LEFT JOIN orders o ON u.id = o.user_id
      GROUP BY u.id, u.name
      HAVING COUNT(o.id) > 0
      ORDER BY lifetime_value DESC
    `);
  },

  // ── Pagination: LIMIT + OFFSET ────────────────────────────────────────────
  // Loading 1000 rows at once is slow and wastes memory.
  // Pagination loads a "page" of rows at a time — typically 20 or 50.
  // Page 1: LIMIT 20 OFFSET 0    (rows 1–20)
  // Page 2: LIMIT 20 OFFSET 20   (rows 21–40)
  // Page 3: LIMIT 20 OFFSET 40   (rows 41–60)
  getProductsPaginated: (category, page = 1, pageSize = 20) => {
    const db = getDatabase();
    const offset = (page - 1) * pageSize;

    return db.getAllSync(`
      SELECT id, name, price, category, stock
      FROM products
      WHERE category = ?
        AND stock > 0
      ORDER BY name ASC
      LIMIT ? OFFSET ?
    `, [category, pageSize, offset]);
  },

  // ── Full-Text Search with LIKE ────────────────────────────────────────────
  searchProducts: (searchTerm, maxPrice = null) => {
    const db = getDatabase();

    let query  = 'SELECT * FROM products WHERE name LIKE ?';
    let params = [`%${searchTerm}%`];

    // Dynamically build the query based on optional filters
    if (maxPrice !== null) {
      query  += ' AND price <= ?';
      params.push(maxPrice);
    }

    query += ' ORDER BY price ASC LIMIT 50';

    return db.getAllSync(query, params);
  },
};
```

## Chapter 16: Using SQLite with React Hooks — The useSQLiteContext Pattern

Modern `expo-sqlite` (v11+) provides a `useSQLiteContext` React hook that integrates the database directly into the React component tree through a context provider. This is the recommended pattern for new projects as it handles the database lifecycle cleanly and provides synchronous access to the database object anywhere in the component tree.

```javascript
import React, { useEffect, useState } from 'react';
import {
  View, Text, FlatList, TextInput,
  TouchableOpacity, StyleSheet, Alert,
} from 'react-native';
import * as SQLite from 'expo-sqlite';
import { initDatabase } from './database';

// ─────────────────────────────────────────────────────────────────────────────
// ROOT: Wrap the app with SQLiteProvider
// ─────────────────────────────────────────────────────────────────────────────

export default function App() {
  return (
    // SQLiteProvider opens the database and makes it available via context.
    // onInit is called the FIRST time the database is opened — ideal for
    // running CREATE TABLE migrations.
    <SQLite.SQLiteProvider
      databaseName="app.db"
      onInit={(db) => {
        db.execSync(`
          CREATE TABLE IF NOT EXISTS todos (
            id        INTEGER PRIMARY KEY AUTOINCREMENT,
            text      TEXT    NOT NULL,
            done      INTEGER NOT NULL DEFAULT 0,
            created_at TEXT   NOT NULL DEFAULT (datetime('now'))
          )
        `);
      }}
    >
      <TodoScreen />
    </SQLite.SQLiteProvider>
  );
}

// ─────────────────────────────────────────────────────────────────────────────
// SCREEN: Todo list backed by SQLite
// ─────────────────────────────────────────────────────────────────────────────

const TodoScreen = () => {
  // useSQLiteContext gives synchronous access to the db opened by SQLiteProvider
  const db = SQLite.useSQLiteContext();
  const [todos, setTodos]     = useState([]);
  const [inputText, setInput] = useState('');

  // ── Load todos from SQLite ─────────────────────────────────────────────────
  const loadTodos = () => {
    const rows = db.getAllSync(
      'SELECT * FROM todos ORDER BY done ASC, created_at DESC'
    );
    setTodos(rows);
  };

  useEffect(() => {
    loadTodos();
  }, []);

  // ── Add a new todo ─────────────────────────────────────────────────────────
  const addTodo = () => {
    const text = inputText.trim();
    if (!text) return;
    db.runSync('INSERT INTO todos (text) VALUES (?)', [text]);
    setInput('');
    loadTodos(); // Refresh the list
  };

  // ── Toggle done/undone ────────────────────────────────────────────────────
  const toggleTodo = (id, currentDone) => {
    db.runSync('UPDATE todos SET done = ? WHERE id = ?', [
      currentDone ? 0 : 1,
      id,
    ]);
    loadTodos();
  };

  // ── Delete a todo ─────────────────────────────────────────────────────────
  const deleteTodo = (id) => {
    db.runSync('DELETE FROM todos WHERE id = ?', [id]);
    loadTodos();
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>📋 SQLite Todo</Text>
      <Text style={styles.subtitle}>
        {todos.filter(t => !t.done).length} tasks remaining
      </Text>

      {/* Input */}
      <View style={styles.inputRow}>
        <TextInput
          style={styles.input}
          value={inputText}
          onChangeText={setInput}
          placeholder="Add a new task..."
          onSubmitEditing={addTodo}
          returnKeyType="done"
        />
        <TouchableOpacity style={styles.addBtn} onPress={addTodo}>
          <Text style={styles.addBtnText}>＋</Text>
        </TouchableOpacity>
      </View>

      {/* List */}
      <FlatList
        data={todos}
        keyExtractor={(item) => item.id.toString()}
        renderItem={({ item }) => (
          <View style={[styles.todoItem, item.done && styles.doneItem]}>
            <TouchableOpacity
              style={styles.todoText}
              onPress={() => toggleTodo(item.id, item.done)}
            >
              <Text style={[styles.todoLabel, item.done && styles.doneLabel]}>
                {item.done ? '✅' : '⬜'} {item.text}
              </Text>
              <Text style={styles.todoDate}>{item.created_at}</Text>
            </TouchableOpacity>
            <TouchableOpacity onPress={() => deleteTodo(item.id)}>
              <Text style={styles.deleteBtn}>🗑</Text>
            </TouchableOpacity>
          </View>
        )}
      />
    </View>
  );
};

const styles = StyleSheet.create({
  container: { flex: 1, backgroundColor: '#F5F5F5', padding: 16, paddingTop: 56 },
  title: { fontSize: 28, fontWeight: 'bold', color: '#1A1A2E', marginBottom: 4 },
  subtitle: { fontSize: 14, color: '#888', marginBottom: 20 },
```

# Part IV — Choosing the Right Storage Tool

## Chapter 17: The Decision Framework

After studying all three storage mechanisms in depth, the most important skill is knowing _when_ to use each one. The wrong choice leads to applications that are either insecure, slow, or unnecessarily complex. The following framework distills the decision into a simple, practical guide.

```
│                                                                     │
│  START: What kind of data do you need to store?                     │

│           │              │              │                           │
│           ▼              ▼              ▼                           │
│    Sensitive?       Structured?      Large Files /                  │
│    (Tokens,         (Many records,   Binary Data?                   │
```

## Conclusion

The three storage systems covered in this guide are complementary rather than competing. A well-architected React Native application typically uses all three of them simultaneously: **Secure Storage** guards the authentication credentials that permit every authenticated API call; **SQLite** holds the rich, queryable application data — users, products, messages, transactions — that forms the core of the user experience; and **File Storage** manages the binary assets — images, documents, downloaded content — that give the application its media richness. Mastering these three tools, understanding their trade-offs, and knowing instinctively which to reach for in any given situation is a hallmark of a senior React Native engineer.
