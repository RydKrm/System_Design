# API Integration in React Native

### A Complete Deep Guide to Fetch, Axios, Error Handling, Retry Logic, Interceptors, Token Refresh & Offline Handling

---

## The Foundation — What Is API Integration?

Every serious mobile application communicates with a remote server. Whether you are building a social media feed, an e-commerce catalogue, a banking dashboard, or a delivery tracking system, data must travel from your backend servers to the user's device and back again. This communication happens through **APIs** — Application Programming Interfaces — and the practice of connecting your React Native application to those APIs is called **API integration**.

In React Native, API integration is the bridge between your carefully crafted JavaScript logic and the real world of servers, databases, and live data. Understanding it deeply means understanding not just the happy path — the straightforward "request goes out, response comes back" scenario — but also the complex, messy reality of unreliable networks, expired authentication tokens, server failures, and users who walk into tunnels with no signal.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     API Integration Landscape                           │
│                                                                         │
│  React Native App                        Your Backend Server            │
│  ┌──────────────────────────┐            ┌───────────────────────┐      │
│  │  Component               │            │  REST API             │      │
│  │  (needs data)            │            │  POST /auth/login     │      │
│  │         │                │            │  GET  /users/:id      │      │
│  │         ▼                │            │  POST /orders         │      │
│  │  API Client Layer        │            │  GET  /products       │      │
│  │  (Axios / Fetch)         │  HTTP/S    └───────────────────────┘      │
│  │         │                │◄──────────────────────────►               │
│  │  Interceptors            │  Request  → Headers, Body, Params         │
│  │  Token Refresh           │  Response ← JSON, Status Code             │
│  │  Retry Logic             │  Error    ← Network fail, 4xx, 5xx        │
│  │  Offline Queue           │                                           │
│  └──────────────────────────┘                                           │
└─────────────────────────────────────────────────────────────────────────┘
```

This guide builds a complete, production-quality API layer from the ground up — starting with the basics and culminating in a robust client that handles every real-world scenario gracefully.

## Chapter 1: Fetch — The Native HTTP Client

### What Fetch Is

`fetch` is a built-in JavaScript API available globally in React Native without any installation. It is the most fundamental tool for making HTTP requests. When you call `fetch`, React Native's JavaScript runtime translates your request into a native network call, sends it to the server, waits for the response, and resolves a Promise with the result.

`fetch` is **Promise-based**, which means every request returns a Promise that you can handle with `.then()/.catch()` or, more idiomatically in modern code, with `async/await`.

### The Critical Detail Most Beginners Miss

The most important thing to understand about `fetch` is a behavior that surprises nearly every developer who encounters it for the first time: **`fetch` only rejects its Promise for network failures — not for HTTP error responses like 404 Not Found or 500 Internal Server Error**. If the server responds with a 401 Unauthorized or a 500 Server Error, `fetch` considers that a successful response and resolves the Promise. You must manually check `response.ok` or `response.status` to detect HTTP-level errors.

```
┌────────────────────────────────────────────────────────────────────────┐
│            fetch — What Rejects vs What Resolves                       │
│                                                                        │
│  REJECTS (throws error):                                               │
│  ─ No internet connection                                              │
│  ─ Server is completely unreachable (DNS failure, timeout)            │
│  ─ Request was aborted                                                 │
│                                                                        │
│  RESOLVES (does NOT throw, even though it's an error!):               │
│  ─ 400 Bad Request     ← you must check response.ok                   │
│  ─ 401 Unauthorized    ← you must check response.status               │
│  ─ 403 Forbidden       ← you must check response.status               │
│  ─ 404 Not Found       ← you must check response.ok                   │
│  ─ 500 Server Error    ← you must check response.ok                   │
│                                                                        │
│  if (!response.ok) {                                                   │
│    throw new Error(`HTTP error! status: ${response.status}`);         │
│  }                                                                     │
└────────────────────────────────────────────────────────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────
// Fetch — From Basic to Production-Quality
// ─────────────────────────────────────────────────────────

// ── Level 1: The simplest possible fetch ──
async function basicFetch() {
  const response = await fetch('https://jsonplaceholder.typicode.com/posts/1');
  const data = await response.json();
  console.log(data);
}

// ── Level 2: With proper error handling ──
async function fetchWithErrorHandling(userId) {
  try {
    const response = await fetch(`https://api.example.com/users/${userId}`, {
      method: 'GET',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${getAccessToken()}`,
        'Accept': 'application/json',
        'X-App-Version': '1.0.0',
      },
    });

    // fetch resolves even for 4xx/5xx — we MUST check this
    if (!response.ok) {
      const errorBody = await response.json().catch(() => ({}));
      throw new ApiError(
        errorBody.message || `Request failed with status ${response.status}`,
        response.status,
        errorBody
      );
    }

    const data = await response.json();
    return data;

  } catch (error) {
    if (error instanceof ApiError) {
      throw error; // re-throw our own errors
    }
    // Network failure, CORS, etc.
    throw new NetworkError('Network request failed. Please check your connection.', error);
  }
}

// ── Level 3: POST request with a body ──
async function createOrder(orderData) {
  const response = await fetch('https://api.example.com/orders', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${getAccessToken()}`,
    },
    body: JSON.stringify(orderData), // serialize object to JSON string
  });

  if (!response.ok) {
    throw new ApiError('Failed to create order', response.status);
  }

  return response.json();
}

// ── Level 4: With AbortController for cancellation ──
async function fetchWithCancellation(url) {
  const controller = new AbortController();

  // Set a 10-second timeout — automatically cancels if server is too slow
  const timeoutId = setTimeout(() => controller.abort(), 10000);

  try {
    const response = await fetch(url, { signal: controller.signal });
    clearTimeout(timeoutId);

    if (!response.ok) throw new ApiError('Request failed', response.status);
    return response.json();

  } catch (error) {
    if (error.name === 'AbortError') {
      throw new TimeoutError('Request timed out after 10 seconds');
    }
    throw error;
  }
}

// ── Custom error classes for precise error handling ──
class ApiError extends Error {
  constructor(message, statusCode, body = {}) {
    super(message);
    this.name = 'ApiError';
    this.statusCode = statusCode;
    this.body = body;
  }
}

class NetworkError extends Error {
  constructor(message, originalError) {
    super(message);
    this.name = 'NetworkError';
    this.originalError = originalError;
  }
}

class TimeoutError extends Error {
  constructor(message) {
    super(message);
    this.name = 'TimeoutError';
  }
}

  // In a real app, read from AsyncStorage or a state manager
  return 'your_access_token_here';
}
```

## Chapter 2: Axios — The Feature-Rich HTTP Client

### Why Axios Exists

While `fetch` is perfectly functional, it requires a significant amount of boilerplate for production use — manually checking `response.ok`, manually serializing request bodies, manually parsing response bodies, setting up cancellation. **Axios** is a third-party library that wraps HTTP requests with a much richer, more developer-friendly API. It was built specifically to solve the common frustrations of working with `fetch` in production applications.

The most important differences are that Axios automatically serializes and deserializes JSON (you pass an object as `data`, and it arrives as an object on the other side without any `JSON.stringify`/`response.json()` calls), it automatically throws for any non-2xx status code (solving the `response.ok` problem), it has a built-in interceptor system for modifying requests and responses globally, and it supports request cancellation through `CancelToken` or `AbortController`.

### Fetch vs Axios — The Critical Differences

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Fetch vs Axios — Feature Comparison               │
├───────────────────────────────┬────────────────┬─────────────────────┤
│  Feature                      │  Fetch         │  Axios              │
├───────────────────────────────┼────────────────┼─────────────────────┤
│  Installation                 │  Built-in      │  npm install axios  │
│  JSON auto-serialize          │  ❌ Manual     │  ✅ Automatic       │
│  JSON auto-deserialize        │  ❌ Manual     │  ✅ Automatic       │
│  Throws on 4xx/5xx            │  ❌ Manual     │  ✅ Automatic       │
│  Request interceptors         │  ❌ None       │  ✅ Built-in        │
│  Response interceptors        │  ❌ None       │  ✅ Built-in        │
│  Request cancellation         │  AbortController│  AbortController   │
│  Request timeout              │  ❌ Manual     │  ✅ timeout prop    │
│  Upload progress              │  ❌ None       │  ✅ onUploadProgress│
│  Base URL config              │  ❌ Manual     │  ✅ baseURL         │
│  Default headers              │  ❌ Manual     │  ✅ defaults        │
│  Instance creation            │  ❌ None       │  ✅ axios.create()  │
│  Response shape               │  Raw Response  │  {data,status,...}  │
└───────────────────────────────┴────────────────┴─────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────

import axios from 'axios';

// ── Level 1: Basic Axios request ──
async function basicAxios() {
  // Response is already { data, status, headers, config }
  // data is already parsed JSON — no .json() call needed
  const { data } = await axios.get('https://jsonplaceholder.typicode.com/posts/1');
  console.log(data); // already a JavaScript object
}

// ── Level 2: Axios with all HTTP methods ──
async function axiosMethods() {
  // GET — fetch a resource
  const { data: user } = await axios.get('/users/123', {
    params: { include: 'profile,settings' },  // adds ?include=profile,settings
  });

  // POST — create a resource
  const { data: newPost } = await axios.post('/posts', {
    title: 'Hello World',
    content: 'This is my first post',
    tags: ['react', 'native'],
  }); // object is auto-serialized — no JSON.stringify needed

  // PUT — replace a resource entirely
  const { data: updated } = await axios.put('/users/123', { name: 'Bob', email: 'bob@example.com' });

  // PATCH — partial update of a resource
  const { data: patched } = await axios.patch('/users/123', { name: 'Bob' });

  // DELETE — remove a resource
  await axios.delete('/posts/456');
}

// This is the recommended pattern for real applications.

  baseURL: 'https://api.yourapp.com/v1',  // prepended to all relative URLs
  timeout: 15000,                          // 15 second timeout for all requests
  headers: {
    'Content-Type': 'application/json',
    'Accept': 'application/json',
    'X-Platform': 'react-native',
    'X-App-Version': '2.1.0',
  },
});

// ── Level 4: Using the instance ──

  const { data } = await apiClient.get('/products', {
    params: { categoryId, page, limit: 20 },
  });
  return data; // { products: [...], total: 150, page: 1 }
}

async function updateProfile(userId, profileData) {
  const { data } = await apiClient.patch(`/users/${userId}/profile`, profileData);
  return data;
}
```

## Chapter 3: Error Handling — Every Failure Has a Type

### Errors Are Not All the Same

When an API call fails, it can fail for many different reasons, and each reason demands a different response from your application. A `401 Unauthorized` should trigger a token refresh and retry. A `404 Not Found` should show the user a "not found" message. A `500 Server Error` should show an apology message. A network timeout should prompt the user to check their connection. And a `422 Validation Error` should highlight the specific form fields that were rejected by the server.

Treating all errors identically — showing a generic "Something went wrong" message for every failure — is the mark of an immature API integration. Production applications classify errors precisely and respond to each class appropriately.

```
┌──────────────────────────────────────────────────────────────────────┐
│                    The Error Classification Tree                     │
│                                                                      │
│  All API Errors                                                      │
│  ├── Network Errors (no response received)                           │
│  │   ├── No internet connection                                      │
│  │   ├── Server unreachable (DNS failure)                            │
│  │   ├── Request timed out                                           │
│  │   └── Request cancelled (AbortController)                        │
│  │                                                                   │
│  └── HTTP Errors (response received, but status indicates failure)   │
│      ├── 4xx Client Errors (our fault)                               │
│      │   ├── 400 Bad Request (malformed data sent)                   │
│      │   ├── 401 Unauthorized (not authenticated / token expired)    │
│      │   ├── 403 Forbidden (authenticated but not permitted)         │
│      │   ├── 404 Not Found (resource doesn't exist)                  │
│      │   ├── 409 Conflict (duplicate, race condition)                │
│      │   └── 422 Unprocessable Entity (validation errors)           │
│      └── 5xx Server Errors (their fault)                             │
│          ├── 500 Internal Server Error                               │
│          ├── 502 Bad Gateway                                         │
│          ├── 503 Service Unavailable                                 │
│          └── 504 Gateway Timeout                                     │
└──────────────────────────────────────────────────────────────────────┘
```

```jsx
// ── Custom Error Classes ──
class ApiError extends Error {
  constructor(message, statusCode, data = null) {
    super(message);
    this.name = 'ApiError';
    this.statusCode = statusCode;
    this.data = data;         // server's error response body
    this.isApiError = true;
  }
}

class NetworkError extends Error {
  constructor(message) {
    super(message);
    this.name = 'NetworkError';
    this.isNetworkError = true;
  }
}

class ValidationError extends ApiError {
  constructor(message, errors) {
    super(message, 422, errors);
    this.name = 'ValidationError';
    this.fieldErrors = errors; // { email: 'Invalid format', name: 'Required' }
  }
}

class AuthError extends ApiError {
  constructor(message) {
    super(message, 401);
    this.name = 'AuthError';
  }
}

// ── Central error normalizer — converts Axios errors into our error types ──
function normalizeError(error) {
  // No response — network-level failure
  if (!error.response) {
    if (error.code === 'ECONNABORTED' || error.message.includes('timeout')) {
      return new NetworkError('The request timed out. Please try again.');
    }
    if (error.message === 'Network Error') {
      return new NetworkError('No internet connection. Please check your network.');
    }
    return new NetworkError('An unexpected network error occurred.');
  }

  const { status, data } = error.response;
  const serverMessage = data?.message || data?.error || 'An error occurred';

  switch (true) {
    case status === 400:
      return new ApiError(`Bad request: ${serverMessage}`, 400, data);

    case status === 401:
      return new AuthError('Your session has expired. Please sign in again.');

    case status === 403:
      return new ApiError('You do not have permission to perform this action.', 403, data);

    case status === 404:
      return new ApiError('The requested resource was not found.', 404, data);

    case status === 409:
      return new ApiError(`Conflict: ${serverMessage}`, 409, data);

    case status === 422:
      // 422 usually comes with field-level validation errors
      return new ValidationError(
        'Please correct the highlighted fields.',
        data?.errors || data?.validationErrors || {}
      );

    case status >= 500:
      return new ApiError(
        'Our servers are experiencing issues. Please try again shortly.',
        status,
        data
      );

    default:
      return new ApiError(serverMessage, status, data);
  }
}
```

## Chapter 4: Retry Logic — Trying Again When It Makes Sense

### When to Retry and When Not To

Not all errors should trigger a retry. Retrying a `401 Unauthorized` without first refreshing the token will just fail again immediately. Retrying a `422 Validation Error` is pointless — the data you sent was invalid, and sending the same data again will always get the same rejection. Retrying a `404 Not Found` will never find a resource that does not exist.

However, retrying a `503 Service Unavailable` or a network timeout is highly sensible — these errors are transient. The server might be momentarily overwhelmed, a load balancer might be restarting, or the user's connection might have briefly dropped. A well-implemented retry system attempts the request again with increasing delays between attempts — a strategy called **exponential backoff with jitter**.

**Exponential backoff** means the delay doubles with each attempt — 1 second, then 2 seconds, then 4 seconds. This prevents many clients from overwhelming a struggling server simultaneously. **Jitter** adds a small random component to each delay — instead of all clients retrying at exactly the same moment (2.000s, 2.000s, 2.000s), they retry at slightly different times (1.847s, 2.112s, 1.993s), spreading the load.

```
┌──────────────────────────────────────────────────────────────────────┐
│                   Exponential Backoff with Jitter                    │
│                                                                      │
│  Attempt 1: REQUEST ──► 503 error                                    │
│  Wait: 1000ms + random(0–300ms) = 1,147ms                           │
│                                                                      │
│  Attempt 2: REQUEST ──► 503 error                                    │
│  Wait: 2000ms + random(0–600ms) = 2,412ms                           │
│                                                                      │
│  Attempt 3: REQUEST ──► 503 error                                    │
│  Wait: 4000ms + random(0–1200ms) = 4,893ms                          │
│                                                                      │
│  Attempt 4: REQUEST ──► 200 OK ✅                                    │
│                                                                      │
│  Total time: ~8.5 seconds. Without retry: permanent failure.        │
│                                                                      │
│  Retryable errors:   503, 504, 502, Network timeout, Network error  │
│  Non-retryable:      400, 401, 403, 404, 409, 422                   │
└──────────────────────────────────────────────────────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────
// Retry Logic — Standalone + Axios Plugin Style
// ─────────────────────────────────────────────────────────

// ── Retryable status codes ──
const RETRYABLE_STATUS_CODES = new Set([408, 429, 500, 502, 503, 504]);
const MAX_RETRIES = 3;
const BASE_DELAY_MS = 1000;

// ── Core retry utility ──
async function withRetry(requestFn, options = {}) {
  const {
    maxRetries = MAX_RETRIES,
    baseDelay = BASE_DELAY_MS,
    onRetry = null,
  } = options;

  let lastError;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const result = await requestFn();
      return result; // success — return immediately

    } catch (error) {
      lastError = error;

      const shouldRetry = isRetryableError(error) && attempt < maxRetries;
      if (!shouldRetry) break;

      // Exponential backoff: 1s, 2s, 4s, 8s...
      const exponentialDelay = baseDelay * Math.pow(2, attempt);

      // Jitter: adds 0–30% random variation to spread load
      const jitter = Math.random() * exponentialDelay * 0.3;
      const delay = Math.round(exponentialDelay + jitter);

      // Handle 429 Too Many Requests — respect Retry-After header
      const retryAfter = error.response?.headers?.['retry-after'];
      const finalDelay = retryAfter
        ? parseInt(retryAfter, 10) * 1000  // server tells us when to retry
        : delay;

      console.log(`Attempt ${attempt + 1} failed. Retrying in ${finalDelay}ms...`);
      onRetry?.({ attempt, delay: finalDelay, error });

      await sleep(finalDelay);
    }
  }

  throw lastError; // all retries exhausted
}

function isRetryableError(error) {
  // Network failure (no response) — always retryable
  if (!error.response) return true;
  return RETRYABLE_STATUS_CODES.has(error.response.status);
}

function sleep(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

// ── Usage of withRetry ──
async function fetchProductsWithRetry(categoryId) {
  return withRetry(
    () => apiClient.get('/products', { params: { categoryId } }),
    {
      maxRetries: 3,
      onRetry: ({ attempt, delay }) => {
        console.log(`Retrying products fetch (attempt ${attempt + 1}), waiting ${delay}ms`);
      },
    }
  );
}

// ── Axios-native retry via interceptors (see next chapter) ──
// Install: npm install axios-retry
import axiosRetry from 'axios-retry';

axiosRetry(apiClient, {
  retries: 3,
  retryDelay: (retryCount) => {
    // exponential backoff: 1s, 2s, 4s
    return axiosRetry.exponentialDelay(retryCount);
  },
  retryCondition: (error) => {
    // retry on network errors AND specific server errors
    return axiosRetry.isNetworkError(error)
      || axiosRetry.isRetryableError(error)
      || error?.response?.status === 503;
  },
  onRetry: (retryCount, error, requestConfig) => {
    console.log(`Retrying request to ${requestConfig.url}, attempt ${retryCount}`);
  },
});
```

## Chapter 5: Interceptors — Global Request and Response Middleware

### What Interceptors Are

An interceptor is a function that runs automatically on every request before it is sent, or on every response before it reaches your calling code. Think of them as middleware for your HTTP client — they sit in the pipeline and can inspect, modify, or even short-circuit every request and response that flows through.

Interceptors are one of the most powerful features of Axios and one of the primary reasons developers choose it over raw `fetch`. Without interceptors, you would have to manually add authorization headers to every single request, manually log every response, and manually handle 401 errors in every single catch block across your entire application. With interceptors, you write this logic once and it applies everywhere automatically.

```
┌──────────────────────────────────────────────────────────────────────┐
│               Interceptor Pipeline                                   │
│                                                                      │
│  Your Code                                                           │
│  apiClient.get('/users')                                             │
│       │                                                              │
│       ▼  REQUEST INTERCEPTORS (run before request is sent)           │
│  ┌──────────────────────────────────────────┐                        │
│  │  1. Add Authorization header             │                        │
│  │  2. Add request timestamp                │                        │
│  │  3. Log the outgoing request             │                        │
│  └──────────────────────────────────────────┘                        │
│       │                                                              │
│       ▼  (request goes to server)                                    │
│  ────────────────────────────────────────────── NETWORK              │
│       │                                                              │
│       ▼  (response comes back from server)                           │
│  ┌──────────────────────────────────────────┐                        │
│  │  RESPONSE INTERCEPTORS                   │                        │
│  │  Success path:    Error path:            │                        │
│  │  1. Log response  1. Check if 401        │                        │
│  │  2. Unwrap data   2. Refresh token       │                        │
│  │  3. Track timing  3. Retry request       │                        │
```

## Chapter 6: Token Refresh — Handling Expired Authentication

### The Authentication Token Problem

Modern APIs use short-lived **access tokens** (typically valid for 15 minutes to 1 hour) and long-lived **refresh tokens** (valid for days or weeks). The access token is sent with every request to authenticate the user. When it expires, the server returns `401 Unauthorized`. The client must then use the refresh token to obtain a new access token, and retry the original request — all without the user noticing that anything happened.

This seems straightforward, but in a real mobile application, multiple requests can fail with 401 simultaneously. If three API calls are in-flight when the token expires, and all three try to refresh the token independently, you end up making three refresh requests — three calls to your refresh endpoint when only one is needed. The second and third refresh requests will likely fail because the first one already consumed the old refresh token.

The solution is a **refresh mutex** — a pattern that ensures only one refresh happens at a time, and all waiting requests receive the new token and retry automatically.

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Token Refresh Flow                                │
│                                                                      │
│  Three requests fail with 401 simultaneously                        │
│                                                                      │
│  Request A → 401 → starts refresh, sets isRefreshing = true        │
│  Request B → 401 → isRefreshing is true → queues itself            │
│  Request C → 401 → isRefreshing is true → queues itself            │
│       │                                                              │
│       ▼                                                              │
│  Refresh token request sent → new access token received            │
│  isRefreshing = false                                                │
│       │                                                              │
│       ▼                                                              │
│  Request A retries with new token ─────────────────────► 200 ✅    │
│  Request B dequeued, retries with new token ───────────► 200 ✅    │
│  Request C dequeued, retries with new token ───────────► 200 ✅    │
│                                                                      │
│  Only ONE refresh request was made for all three failures ✅        │
└──────────────────────────────────────────────────────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────
// Complete Token Refresh with Queue / Mutex Pattern
// ─────────────────────────────────────────────────────────

import axios from 'axios';
import AsyncStorage from '@react-native-async-storage/async-storage';

// Token storage helpers
const TokenStorage = {
  async getAccessToken()  { return AsyncStorage.getItem('access_token'); },
  async getRefreshToken() { return AsyncStorage.getItem('refresh_token'); },
  async setTokens(access, refresh) {
    await Promise.all([
      AsyncStorage.setItem('access_token', access),
      AsyncStorage.setItem('refresh_token', refresh),
    ]);
  },
  async clearTokens() {
    await Promise.all([
      AsyncStorage.removeItem('access_token'),
      AsyncStorage.removeItem('refresh_token'),
    ]);
  },
};

// Mutex state
let isRefreshing = false;
let failedQueue = []; // requests waiting for new token

// Process all queued requests with either the new token or an error
function processQueue(error, token = null) {
  failedQueue.forEach(({ resolve, reject }) => {
    if (error) {
      reject(error);
    } else {
      resolve(token);
    }
  });
  failedQueue = [];
}

const apiClient = axios.create({
  baseURL: 'https://api.yourapp.com/v1',
  timeout: 15000,
});

// Request interceptor — attach token
apiClient.interceptors.request.use(async (config) => {
  const token = await TokenStorage.getAccessToken();
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Response interceptor — handle 401 with token refresh
apiClient.interceptors.response.use(
  (response) => response,

  async (error) => {
    const originalRequest = error.config;

    // Only handle 401 errors that haven't already been retried
    if (error.response?.status !== 401 || originalRequest._retry) {
      return Promise.reject(error);
    }

    if (isRefreshing) {
      // Another request is already refreshing.
      // Queue this request — it will be retried when refresh completes.
      return new Promise((resolve, reject) => {
        failedQueue.push({ resolve, reject });
      }).then((newToken) => {
        originalRequest.headers.Authorization = `Bearer ${newToken}`;
        return apiClient(originalRequest); // retry with new token
      });
    }

    // This request is the first to detect the 401
    originalRequest._retry = true;
    isRefreshing = true;

    try {
      const refreshToken = await TokenStorage.getRefreshToken();

      if (!refreshToken) {
        throw new Error('No refresh token available');
      }

      // Call the refresh endpoint (using plain axios, not the instance,
      // to avoid interceptor loops)
      const { data } = await axios.post(
        'https://api.yourapp.com/v1/auth/refresh',
        { refreshToken }
      );

      const { accessToken, refreshToken: newRefreshToken } = data;

      // Persist the new tokens
      await TokenStorage.setTokens(accessToken, newRefreshToken);

      // Update the default header for future requests
      apiClient.defaults.headers.common.Authorization = `Bearer ${accessToken}`;

      // Release all queued requests with the new token
      processQueue(null, accessToken);

      // Retry the original request that triggered the 401
      originalRequest.headers.Authorization = `Bearer ${accessToken}`;
      return apiClient(originalRequest);

    } catch (refreshError) {
      // Refresh failed — user must log in again
      processQueue(refreshError, null);
      await TokenStorage.clearTokens();

      // Emit a global event so the app can navigate to the login screen
      // In a real app, use an event emitter, navigation ref, or context
      console.error('Session expired. User must re-authenticate.');
      // navigationRef.current?.navigate('Login');

      return Promise.reject(refreshError);

    } finally {
      isRefreshing = false;
    }
  }
);

export { apiClient, TokenStorage };
```

## Chapter 7: Offline Handling — Graceful Degradation Without Internet

### The Reality of Mobile Networks

Unlike a web browser running on a fixed broadband connection, a mobile application lives in the physical world. Users ride the subway, walk into buildings, drive through areas with no signal, and travel internationally. Their network connection is a dynamic, unreliable thing that can disappear and reappear at any moment. A production React Native application must handle these conditions gracefully — not crash, not lose data, not silently fail.

Offline handling has several distinct layers. The first layer is **detection** — knowing when the user has lost connectivity. The second layer is **feedback** — showing the user clear, non-alarming information about their connectivity status. The third layer is **queue** — persisting operations that the user attempted while offline and replaying them automatically when connectivity is restored. The fourth layer is **cache** — serving previously loaded data from local storage so the app remains functional even without a connection.

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Offline Handling Architecture                     │
│                                                                      │
│  User performs action while offline                                  │
│         │                                                            │
│         ▼                                                            │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │              Offline Detection Layer                        │     │
│  │  @react-native-community/netinfo                           │     │
│  │  NetInfo.addEventListener('change', handleConnectivity)    │     │
│  └────────────────────────────────────────────────────────────┘     │
│         │                                                            │
│    Is online?                                                        │
│    ├── YES → send request normally                                   │
│    └── NO  → queue request in AsyncStorage                          │
│                │                                                     │
│                ▼                                                     │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │              Offline Queue (persisted to disk)              │     │
│  │  [{ url, method, data, timestamp, id }, ...]               │     │
│  └────────────────────────────────────────────────────────────┘     │
│                │                                                     │
│    Connection restored                                               │
│                │                                                     │
│                ▼                                                     │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │              Queue Processor                                │     │
│  │  Replay all queued requests in order                       │     │
│  │  Handle success, partial failure, conflicts               │     │
│  └────────────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────────┘
```

```jsx
// ─────────────────────────────────────────────────────────
// Complete Offline Handling System
// ─────────────────────────────────────────────────────────

import { useState, useEffect, useCallback, useRef } from 'react';
import { View, Text, StyleSheet, Animated } from 'react-native';
import NetInfo from '@react-native-community/netinfo';
import AsyncStorage from '@react-native-async-storage/async-storage';

// ── Offline Queue Manager ──
const QUEUE_KEY = 'offline_request_queue';

const OfflineQueue = {
  async getQueue() {
    try {
      const raw = await AsyncStorage.getItem(QUEUE_KEY);
      return raw ? JSON.parse(raw) : [];
    } catch {
      return [];
    }
  },

  async enqueue(request) {
    const queue = await this.getQueue();
    const item = {
      id: `req_${Date.now()}_${Math.random().toString(36).slice(2)}`,
      timestamp: Date.now(),
      ...request, // { url, method, data, headers }
    };
    queue.push(item);
    await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(queue));
    console.log(`[OfflineQueue] Queued: ${request.method} ${request.url}`);
    return item.id;
  },

  async dequeue(id) {
    const queue = await this.getQueue();
    const filtered = queue.filter(item => item.id !== id);
    await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(filtered));
  },

  async clearQueue() {
    await AsyncStorage.removeItem(QUEUE_KEY);
  },

  async processQueue(onItemSuccess, onItemFailure) {
    const queue = await this.getQueue();
    if (queue.length === 0) return;

    console.log(`[OfflineQueue] Processing ${queue.length} queued requests`);

    for (const item of queue) {
      try {
        const response = await apiClient({
          method: item.method,
          url: item.url,
          data: item.data,
          headers: item.headers,
        });
        await this.dequeue(item.id);
        onItemSuccess?.(item, response.data);
        console.log(`[OfflineQueue] ✅ Processed: ${item.method} ${item.url}`);
      } catch (error) {
        console.error(`[OfflineQueue] ❌ Failed: ${item.method} ${item.url}`, error.message);
        onItemFailure?.(item, error);
        // Stop processing on network failure (we're still offline)
        if (!error.response) break;
        // For server errors, remove from queue to avoid infinite retries
        if (error.response?.status >= 400 && error.response?.status < 500) {
          await this.dequeue(item.id);
        }
      }
    }
  },
};

// ── useNetworkStatus hook ──
export function useNetworkStatus() {
  const [isOnline, setIsOnline] = useState(true);
  const [connectionType, setConnectionType] = useState('unknown');
  const [queueLength, setQueueLength] = useState(0);
  const wasOffline = useRef(false);

  const refreshQueueLength = useCallback(async () => {
    const queue = await OfflineQueue.getQueue();
    setQueueLength(queue.length);
  }, []);

  useEffect(() => {
    // Subscribe to network state changes
    const unsubscribe = NetInfo.addEventListener(async (state) => {
      const online = !!(state.isConnected && state.isInternetReachable);
      setIsOnline(online);
      setConnectionType(state.type);

      if (online && wasOffline.current) {
        // Just came back online — process the queue
        wasOffline.current = false;
        console.log('[Network] Back online — processing offline queue');
        await OfflineQueue.processQueue(
          (item) => console.log(`Synced: ${item.method} ${item.url}`),
          (item, err) => console.error(`Sync failed: ${item.url}`, err.message)
        );
        refreshQueueLength();
      }

      if (!online) {
        wasOffline.current = true;
      }
    });

    refreshQueueLength();
    return () => unsubscribe();
  }, [refreshQueueLength]);

  // Safe request function that queues when offline
  const safeRequest = useCallback(async (config) => {
    const state = await NetInfo.fetch();
    const online = !!(state.isConnected && state.isInternetReachable);

    if (online) {
      return apiClient(config);
    } else {
      // Queue for later execution
      await OfflineQueue.enqueue(config);
      refreshQueueLength();
      throw new OfflineError('Request queued for when connectivity is restored');
    }
  }, [refreshQueueLength]);

  return { isOnline, connectionType, queueLength, safeRequest, refreshQueueLength };
}

class OfflineError extends Error {
  constructor(message) {
    super(message);
    this.name = 'OfflineError';
    this.isOfflineError = true;
  }
}

// ── Offline Banner Component ──
export function NetworkStatusBanner() {
  const { isOnline, queueLength } = useNetworkStatus();
  const slideAnim = useRef(new Animated.Value(-60)).current;
  const [wasOffline, setWasOffline] = useState(false);
  const [showReconnected, setShowReconnected] = useState(false);

  useEffect(() => {
    if (!isOnline) {
      setWasOffline(true);
      // Slide the banner down into view
      Animated.spring(slideAnim, {
        toValue: 0,
        useNativeDriver: true,
      }).start();
    } else {
      if (wasOffline) {
        // Show "back online" message briefly, then hide
        setShowReconnected(true);
        setTimeout(() => {
          Animated.timing(slideAnim, {
            toValue: -60,
            duration: 400,
            useNativeDriver: true,
          }).start(() => setShowReconnected(false));
        }, 2500);
      } else {
        // Hide immediately if was never offline
        slideAnim.setValue(-60);
      }
    }
  }, [isOnline]);

  if (!wasOffline && isOnline) return null;

  return (
    <Animated.View
      style={[
        bannerStyles.banner,
        isOnline ? bannerStyles.bannerOnline : bannerStyles.bannerOffline,
        { transform: [{ translateY: slideAnim }] },
      ]}
    >
      <Text style={bannerStyles.bannerText}>
        {isOnline
          ? '✅ Back online! Syncing your changes...'
          : queueLength > 0
            ? `📴 Offline — ${queueLength} request${queueLength !== 1 ? 's' : ''} queued`
            : '📴 No internet connection'
        }
      </Text>
    </Animated.View>
  );
}

const bannerStyles = StyleSheet.create({
  banner: {
    position: 'absolute',
    top: 0, left: 0, right: 0,
    paddingVertical: 12, paddingHorizontal: 16,
    zIndex: 9999,
  },
  bannerOffline: { backgroundColor: '#e74c3c' },
  bannerOnline: { backgroundColor: '#27ae60' },
  bannerText: { color: '#fff', fontWeight: '700', fontSize: 14, textAlign: 'center' },
});
```

## Chapter 8: Putting It All Together — The Complete API Client

The following is a complete, production-ready API service that combines every concept from this guide — Axios instance configuration, request and response interceptors, automatic token refresh with queue, retry logic, error normalization, and offline awareness.

```jsx
// ─────────────────────────────────────────────────────────
// api/client.js — The Complete Production API Client
// ─────────────────────────────────────────────────────────

import axios from 'axios';
import AsyncStorage from '@react-native-async-storage/async-storage';

// ── Configuration ──
const API_BASE_URL = __DEV__
  ? 'https://dev-api.yourapp.com/v1'
  : 'https://api.yourapp.com/v1';

const DEFAULT_TIMEOUT = 15000;
const MAX_RETRIES = 3;
const RETRYABLE_CODES = new Set([408, 500, 502, 503, 504]);

// ── Token Storage ──
const TokenStorage = {
  async get(key)         { return AsyncStorage.getItem(key); },
  async set(key, val)    { return AsyncStorage.setItem(key, val); },
  async remove(key)      { return AsyncStorage.removeItem(key); },
  async getAccess()      { return this.get('access_token'); },
  async getRefresh()     { return this.get('refresh_token'); },
  async saveTokens(a, r) { return Promise.all([this.set('access_token', a), this.set('refresh_token', r)]); },
  async clearAll()       { return Promise.all([this.remove('access_token'), this.remove('refresh_token')]); },
};

// ── Refresh Mutex ──
let isRefreshing = false;
let refreshQueue = [];
function drainQueue(err, token) {
  refreshQueue.forEach(({ resolve, reject }) => err ? reject(err) : resolve(token));
  refreshQueue = [];
}

// ── Create Client ──
const client = axios.create({
  baseURL: API_BASE_URL,
  timeout: DEFAULT_TIMEOUT,
  headers: {
    'Content-Type': 'application/json',
    'Accept': 'application/json',
  },
});

// ── Request Interceptor ──
client.interceptors.request.use(async (config) => {
  const token = await TokenStorage.getAccess();
  if (token) config.headers.Authorization = `Bearer ${token}`;
  config.metadata = { startTime: Date.now(), retryCount: config.metadata?.retryCount ?? 0 };
  if (__DEV__) console.log(`→ ${config.method?.toUpperCase()} ${config.url}`);
  return config;
});

// ── Response Interceptor ──
client.interceptors.response.use(
  (response) => {
    if (__DEV__) {
      const ms = Date.now() - (response.config.metadata?.startTime ?? Date.now());
      console.log(`← ${response.status} ${response.config.url} [${ms}ms]`);
    }
    return response;
  },
  async (error) => {
    const config = error.config;
    const status = error.response?.status;

    // ── Token Refresh (401) ──
    if (status === 401 && !config._authRetry) {
      config._authRetry = true;
      if (isRefreshing) {
        return new Promise((resolve, reject) => refreshQueue.push({ resolve, reject }))
          .then(token => {
            config.headers.Authorization = `Bearer ${token}`;
            return client(config);
          });
      }
      isRefreshing = true;
      try {
        const refreshToken = await TokenStorage.getRefresh();
        if (!refreshToken) throw new Error('No refresh token');
        const { data } = await axios.post(`${API_BASE_URL}/auth/refresh`, { refreshToken });
        await TokenStorage.saveTokens(data.accessToken, data.refreshToken);
        client.defaults.headers.common.Authorization = `Bearer ${data.accessToken}`;
        drainQueue(null, data.accessToken);
        config.headers.Authorization = `Bearer ${data.accessToken}`;
        return client(config);
      } catch (refreshErr) {
        drainQueue(refreshErr, null);
        await TokenStorage.clearAll();
        return Promise.reject(new AuthError('Session expired'));
      } finally {
        isRefreshing = false;
      }
    }

    // ── Retry Logic (5xx + network) ──
    const retryCount = config.metadata?.retryCount ?? 0;
    const shouldRetry = retryCount < MAX_RETRIES
      && (RETRYABLE_CODES.has(status) || !error.response);

    if (shouldRetry) {
      const delay = Math.min(1000 * Math.pow(2, retryCount) + Math.random() * 300, 10000);
      config.metadata.retryCount = retryCount + 1;
      console.log(`Retry ${retryCount + 1}/${MAX_RETRIES} for ${config.url} in ${Math.round(delay)}ms`);
      await new Promise(r => setTimeout(r, delay));
      return client(config);
    }

    // ── Normalize Error ──
    return Promise.reject(normalizeError(error));
  }
);

export { client as apiClient, TokenStorage };
```
