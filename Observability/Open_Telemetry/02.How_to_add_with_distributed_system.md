# Implementing OpenTelemetry in a Golang Distributed System

## Understanding OpenTelemetry in the Go Ecosystem

When you embark on the journey of adding observability to a Go distributed system, you are essentially equipping your application with a sophisticated nervous system that can sense and report on everything happening within it. Go, with its emphasis on simplicity, concurrency, and performance, presents both unique opportunities and considerations when implementing OpenTelemetry. Unlike some other ecosystems where automatic instrumentation can magically observe your entire application with minimal code, Go's philosophy leans toward explicitness and clarity. This means you will have more control over your instrumentation, but it also requires a deeper understanding of how to properly integrate OpenTelemetry into your services.

The Go implementation of OpenTelemetry follows the same specification as other language implementations, ensuring that your Go services can seamlessly interoperate with services written in Node.js, Python, Java, or any other supported language. This uniformity is crucial in a polyglot distributed system where different teams might choose different technologies based on their specific needs. Your Go-based authentication service will produce traces that perfectly connect with spans from your Node.js API gateway and your Python machine learning service, creating a complete picture of your system's behavior.

In the Go ecosystem, OpenTelemetry instrumentation is achieved through a combination of explicit SDK initialization, middleware integration for web frameworks, and manual span creation for business logic. While this requires more setup than some automatic instrumentation approaches, it gives you precise control over what gets observed and how. You decide exactly which operations deserve their own spans, what attributes to attach, and how sampling should be configured. This explicitness aligns perfectly with Go's philosophy of making dependencies and behavior clear and visible.

At the foundation of this architecture lies the SDK initialization, which must happen early in your application's lifecycle, typically in your main function before any other services start. This initialization creates the TracerProvider and MeterProvider, which are the factories for creating tracers and meters throughout your application. These providers are configured with exporters that determine where your telemetry data will be sent, samplers that decide which traces to keep, and resource attributes that identify your service.

The context propagation mechanism is particularly important in Go because of how the language handles concurrency. Go's context.Context is used not just for cancellation and deadlines, but also as the carrier for OpenTelemetry's trace context. When you create a span, you pass in a context, and when that span is created, the trace information is stored in that context. This context then flows through your function calls, goroutines, and even across service boundaries when you make HTTP or gRPC calls. Understanding how to properly propagate context is crucial for creating meaningful distributed traces.

The middleware layer provides automatic instrumentation for common operations. When an HTTP request arrives at your server, middleware intercepts it, extracts any trace context from the incoming headers, creates a span for the request, and ensures that context flows to your handler function. Similarly, when your service makes an outbound HTTP call, client middleware injects the trace context into the outgoing request headers. This bidirectional flow of context creates the connected trace that spans your entire distributed system.

Let's walk through a complete implementation, starting with the initialization code that sets up OpenTelemetry in your application. This initialization code is typically placed in a separate package to keep concerns separated and make testing easier.

```go
// File: pkg/telemetry/telemetry.go
package telemetry

import (
    "context"
    "fmt"
    "time"

    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc"
    "go.opentelemetry.io/otel/propagation"
    "go.opentelemetry.io/otel/sdk/resource"
    sdktrace "go.opentelemetry.io/otel/sdk/trace"
    semconv "go.opentelemetry.io/otel/semconv/v1.24.0"
    "google.golang.org/grpc"
    "google.golang.org/grpc/credentials/insecure"
)

// Config holds the configuration for OpenTelemetry initialization
type Config struct {
    ServiceName    string
    ServiceVersion string
    Environment    string
    CollectorURL   string
}

// InitTracer initializes the OpenTelemetry tracer provider
// This function must be called once during application startup
// It returns a shutdown function that should be called during graceful shutdown
func InitTracer(cfg Config) (func(context.Context) error, error) {
    // Create a resource that identifies this service
    // Resource attributes are attached to all telemetry produced by this service
    res, err := resource.New(
        context.Background(),
        resource.WithAttributes(
            // Semantic conventions define standard attribute names
            semconv.ServiceName(cfg.ServiceName),
            semconv.ServiceVersion(cfg.ServiceVersion),
            semconv.DeploymentEnvironment(cfg.Environment),
            
            // Custom attributes specific to your infrastructure
            attribute.String("service.team", "platform-engineering"),
            attribute.String("service.tier", "critical"),
        ),
        // Automatically detect resource attributes from the environment
        // This includes cloud provider info, container ID, host info, etc.
        resource.WithFromEnv(),
        resource.WithProcess(),
        resource.WithOS(),
        resource.WithContainer(),
        resource.WithHost(),
    )
    if err != nil {
        return nil, fmt.Errorf("failed to create resource: %w", err)
    }

    // Create a gRPC connection to the OpenTelemetry Collector
    ctx := context.Background()
    conn, err := grpc.DialContext(
        ctx,
        cfg.CollectorURL,
        // In production, use proper TLS credentials
        grpc.WithTransportCredentials(insecure.NewCredentials()),
        grpc.WithBlock(),
    )
    if err != nil {
        return nil, fmt.Errorf("failed to create gRPC connection: %w", err)
    }

    // Create the OTLP trace exporter that sends data to the Collector
    traceExporter, err := otlptracegrpc.New(
        ctx,
        otlptracegrpc.WithGRPCConn(conn),
    )
    if err != nil {
        return nil, fmt.Errorf("failed to create trace exporter: %w", err)
    }

    // Create a batch span processor
    // Batching improves performance by reducing network calls
    batchSpanProcessor := sdktrace.NewBatchSpanProcessor(
        traceExporter,
        sdktrace.WithBatchTimeout(5*time.Second),
        sdktrace.WithMaxExportBatchSize(512),
    )

    // Create the tracer provider with our configuration
    tracerProvider := sdktrace.NewTracerProvider(
        sdktrace.WithResource(res),
        sdktrace.WithSpanProcessor(batchSpanProcessor),
        
        // Sampling configuration
        // ParentBased means: if parent span is sampled, sample this span too
        // Otherwise, use the root sampler (TraceIDRatioBased samples 10% of traces)
        sdktrace.WithSampler(
            sdktrace.ParentBased(
                sdktrace.TraceIDRatioBased(0.1), // Sample 10% of root spans
            ),
        ),
    )

    // Set the global tracer provider
    // This allows the otel package to be used anywhere in your code
    otel.SetTracerProvider(tracerProvider)

    // Configure context propagation
    // This determines how trace context is encoded in HTTP headers
    otel.SetTextMapPropagator(
        propagation.NewCompositeTextMapPropagator(
            propagation.TraceContext{}, // W3C Trace Context standard
            propagation.Baggage{},      // W3C Baggage for additional context
        ),
    )

    // Return a shutdown function
    // This function should be called when your application shuts down
    // It ensures all pending spans are exported before exit
    return tracerProvider.Shutdown, nil
}
```

This initialization code demonstrates several important concepts. The Resource identifies your service with attributes that follow semantic conventions, ensuring consistency across your entire organization. The batch span processor improves performance by collecting spans in memory and sending them in batches rather than individually. The sampling configuration controls the volume of traces you collect—sampling all traces in a high-throughput system would be prohibitively expensive, so you sample a percentage while ensuring that error traces and slow traces are always captured.

Now let's examine how to structure your main function to properly initialize OpenTelemetry and handle graceful shutdown:

```go
// File: cmd/order-service/main.go
package main

import (
    "context"
    "log"
    "net/http"
    "os"
    "os/signal"
    "syscall"
    "time"

    "github.com/yourcompany/order-service/pkg/telemetry"
    "github.com/yourcompany/order-service/internal/handlers"
    "github.com/gin-gonic/gin"
)

func main() {
    // Initialize OpenTelemetry
    shutdown, err := telemetry.InitTracer(telemetry.Config{
        ServiceName:    "order-service",
        ServiceVersion: "1.2.0",
        Environment:    getEnv("ENVIRONMENT", "development"),
        CollectorURL:   getEnv("OTEL_COLLECTOR_URL", "localhost:4317"),
    })
    if err != nil {
        log.Fatalf("Failed to initialize tracer: %v", err)
    }

    // Ensure proper shutdown of OpenTelemetry on application exit
    // This defer ensures pending spans are flushed even if panic occurs
    defer func() {
        ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        
        if err := shutdown(ctx); err != nil {
            log.Printf("Error shutting down tracer provider: %v", err)
        }
    }()

    // Create your application server
    router := setupRouter()
    
    server := &http.Server{
        Addr:    ":8080",
        Handler: router,
    }

    // Handle graceful shutdown
    go func() {
        if err := server.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatalf("Server failed: %v", err)
        }
    }()

    // Wait for interrupt signal
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit

    log.Println("Shutting down server...")

    // Graceful shutdown with timeout
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    if err := server.Shutdown(ctx); err != nil {
        log.Printf("Server forced to shutdown: %v", err)
    }

    log.Println("Server exited")
}

func getEnv(key, defaultValue string) string {
    if value := os.Getenv(key); value != "" {
        return value
    }
    return defaultValue
}
```

The graceful shutdown handling is critical because OpenTelemetry batches spans for efficiency. If your application terminates abruptly, those batched spans may be lost. The shutdown function ensures all pending telemetry is exported before the process exits.

## Instrumenting HTTP Services with Gin Framework

Now that we have initialized OpenTelemetry, let's examine how to instrument an HTTP service built with the Gin framework, one of the most popular Go web frameworks. The instrumentation happens at multiple levels: automatic middleware for HTTP requests and manual spans for business logic.

```go
// File: internal/server/server.go
package server

import (
    "context"
    "fmt"
    "time"

    "github.com/gin-gonic/gin"
    "go.opentelemetry.io/contrib/instrumentation/github.com/gin-gonic/gin/otelgin"
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/codes"
    "go.opentelemetry.io/otel/trace"
)

// SetupRouter creates and configures the Gin router with OpenTelemetry middleware
func SetupRouter() *gin.Engine {
    router := gin.New()

    // Add OpenTelemetry middleware as the first middleware
    // This ensures all requests are traced
    // The middleware will:
    // 1. Extract trace context from incoming request headers
    // 2. Create a span for the HTTP request
    // 3. Inject trace context into the Gin context
    // 4. Capture request/response metadata (status code, method, route)
    router.Use(otelgin.Middleware("order-service"))

    // Add recovery middleware to capture panics as span errors
    router.Use(gin.Recovery())

    // Define routes
    router.POST("/orders", CreateOrderHandler)
    router.GET("/orders/:id", GetOrderHandler)
    router.GET("/health", HealthCheckHandler)

    return router
}

// Order represents an order in the system
type Order struct {
    ID         string    `json:"id"`
    UserID     string    `json:"user_id"`
    Amount     float64   `json:"amount"`
    Status     string    `json:"status"`
    CreatedAt  time.Time `json:"created_at"`
}

// CreateOrderRequest is the request body for creating orders
type CreateOrderRequest struct {
    UserID string   `json:"user_id" binding:"required"`
    Amount float64  `json:"amount" binding:"required,gt=0"`
    Items  []string `json:"items" binding:"required,min=1"`
}

// CreateOrderHandler handles order creation requests
func CreateOrderHandler(c *gin.Context) {
    // The context from Gin already contains the trace context
    // thanks to the otelgin middleware
    ctx := c.Request.Context()

    // Get the tracer for this service
    // Tracer names should match your service or component name
    tracer := otel.Tracer("order-service")

    // Parse request body
    var req CreateOrderRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(400, gin.H{"error": "Invalid request"})
        return
    }

    // Create a child span for the order creation logic
    // This span will be a child of the HTTP request span created by middleware
    ctx, span := tracer.Start(ctx, "create_order",
        trace.WithSpanKind(trace.SpanKindInternal),
        trace.WithAttributes(
            // Add business-relevant attributes
            attribute.String("order.user_id", req.UserID),
            attribute.Float64("order.amount", req.Amount),
            attribute.Int("order.item_count", len(req.Items)),
        ),
    )
    defer span.End()

    // Simulate order creation with proper error handling
    order, err := createOrder(ctx, req)
    if err != nil {
        // Record the error in the span
        span.RecordError(err)
        span.SetStatus(codes.Error, "Failed to create order")
        
        c.JSON(500, gin.H{"error": "Failed to create order"})
        return
    }

    // Add the order ID to the span now that we have it
    span.SetAttributes(attribute.String("order.id", order.ID))
    span.SetStatus(codes.Ok, "Order created successfully")

    c.JSON(201, order)
}

// createOrder contains the business logic for order creation
func createOrder(ctx context.Context, req CreateOrderRequest) (*Order, error) {
    tracer := otel.Tracer("order-service")
    
    // Create a span for validation
    ctx, validationSpan := tracer.Start(ctx, "validate_order")
    err := validateOrder(ctx, req)
    validationSpan.End()
    
    if err != nil {
        return nil, fmt.Errorf("validation failed: %w", err)
    }

    // Create a span for checking inventory
    // This might call another service
    ctx, inventorySpan := tracer.Start(ctx, "check_inventory",
        trace.WithAttributes(
            attribute.StringSlice("items", req.Items),
        ),
    )
    available, err := checkInventory(ctx, req.Items)
    inventorySpan.End()
    
    if err != nil {
        return nil, fmt.Errorf("inventory check failed: %w", err)
    }
    if !available {
        return nil, fmt.Errorf("items not available")
    }

    // Create a span for payment processing
    ctx, paymentSpan := tracer.Start(ctx, "process_payment",
        trace.WithAttributes(
            attribute.Float64("payment.amount", req.Amount),
            attribute.String("payment.user_id", req.UserID),
        ),
    )
    paymentID, err := processPayment(ctx, req.UserID, req.Amount)
    if err != nil {
        paymentSpan.RecordError(err)
        paymentSpan.SetStatus(codes.Error, "Payment failed")
        paymentSpan.End()
        return nil, fmt.Errorf("payment processing failed: %w", err)
    }
    paymentSpan.SetAttributes(attribute.String("payment.id", paymentID))
    paymentSpan.SetStatus(codes.Ok, "Payment successful")
    paymentSpan.End()

    // Create a span for database insertion
    ctx, dbSpan := tracer.Start(ctx, "save_order_to_database")
    order := &Order{
        ID:        generateOrderID(),
        UserID:    req.UserID,
        Amount:    req.Amount,
        Status:    "confirmed",
        CreatedAt: time.Now(),
    }
    err = saveOrderToDatabase(ctx, order)
    dbSpan.End()
    
    if err != nil {
        return nil, fmt.Errorf("database save failed: %w", err)
    }

    return order, nil
}
```

This example demonstrates the hierarchical nature of spans. The HTTP middleware creates a root span for the request. Inside the handler, we create a span for the overall order creation operation. Within that operation, we create child spans for validation, inventory checking, payment processing, and database operations. Each span captures the timing and outcome of its operation, and any errors are properly recorded. When you view this trace in Jaeger, you'll see a waterfall diagram showing exactly where time was spent.

## Instrumenting Database Operations

Database operations are often the most critical part of your application to observe because they're frequently the source of performance bottlenecks. Let's examine how to instrument PostgreSQL operations using the otelsql library:

```go
// File: internal/database/database.go
package database

import (
    "context"
    "database/sql"
    "fmt"
    "time"

    "github.com/lib/pq"
    "go.opentelemetry.io/contrib/instrumentation/database/sql/otelsql"
    semconv "go.opentelemetry.io/otel/semconv/v1.24.0"
)

// DB wraps the standard database/sql.DB with OpenTelemetry instrumentation
type DB struct {
    *sql.DB
}

// NewDatabase creates a new instrumented database connection
func NewDatabase(connectionString string) (*DB, error) {
    // Register the instrumented driver
    // This wraps the PostgreSQL driver with OpenTelemetry instrumentation
    // Every query will automatically create a span
    driverName, err := otelsql.Register(
        "postgres",
        otelsql.WithAttributes(
            semconv.DBSystemPostgreSQL,
        ),
        // Record each SQL statement executed
        otelsql.WithSQLStatementInSpan(true),
    )
    if err != nil {
        return nil, fmt.Errorf("failed to register driver: %w", err)
    }

    // Open database connection using the instrumented driver
    db, err := sql.Open(driverName, connectionString)
    if err != nil {
        return nil, fmt.Errorf("failed to open database: %w", err)
    }

    // Configure connection pool
    db.SetMaxOpenConns(25)
    db.SetMaxIdleConns(5)
    db.SetConnMaxLifetime(5 * time.Minute)

    // Verify connection
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    
    if err := db.PingContext(ctx); err != nil {
        return nil, fmt.Errorf("failed to ping database: %w", err)
    }

    // Record database metrics
    if err := otelsql.RecordStats(db, otelsql.WithMeterProvider(
        otel.GetMeterProvider(),
    )); err != nil {
        return nil, fmt.Errorf("failed to record stats: %w", err)
    }

    return &DB{DB: db}, nil
}

// SaveOrder saves an order to the database with automatic instrumentation
func (db *DB) SaveOrder(ctx context.Context, order *Order) error {
    // The context already contains trace information from upstream
    // The otelsql instrumentation will create a span for this query
    // and attach it as a child of the current span in the context
    
    query := `
        INSERT INTO orders (id, user_id, amount, status, created_at)
        VALUES ($1, $2, $3, $4, $5)
    `
    
    // The query execution is automatically traced
    // The span will include:
    // - SQL statement (if configured)
    // - Execution time
    // - Database system (PostgreSQL)
    // - Any errors that occur
    _, err := db.ExecContext(
        ctx,
        query,
        order.ID,
        order.UserID,
        order.Amount,
        order.Status,
        order.CreatedAt,
    )
    
    if err != nil {
        // Error is automatically recorded in the database span
        return fmt.Errorf("failed to insert order: %w", err)
    }
    
    return nil
}

// GetOrder retrieves an order by ID with automatic instrumentation
func (db *DB) GetOrder(ctx context.Context, orderID string) (*Order, error) {
    query := `
        SELECT id, user_id, amount, status, created_at
        FROM orders
        WHERE id = $1
    `
    
    var order Order
    
    // QueryRowContext is also automatically instrumented
    err := db.QueryRowContext(ctx, query, orderID).Scan(
        &order.ID,
        &order.UserID,
        &order.Amount,
        &order.Status,
        &order.CreatedAt,
    )
    
    if err == sql.ErrNoRows {
        return nil, fmt.Errorf("order not found")
    }
    
    if err != nil {
        return nil, fmt.Errorf("failed to query order: %w", err)
    }
    
    return &order, nil
}

// GetOrdersByUser retrieves all orders for a user with manual span creation
func (db *DB) GetOrdersByUser(ctx context.Context, userID string) ([]*Order, error) {
    // Sometimes you want to add custom attributes to database operations
    // In those cases, you can create a manual span around the query
    tracer := otel.Tracer("order-service.database")
    ctx, span := tracer.Start(ctx, "get_orders_by_user",
        trace.WithAttributes(
            attribute.String("db.user_id", userID),
        ),
    )
    defer span.End()
    
    query := `
        SELECT id, user_id, amount, status, created_at
        FROM orders
        WHERE user_id = $1
        ORDER BY created_at DESC
    `
    
    rows, err := db.QueryContext(ctx, query, userID)
    if err != nil {
        span.RecordError(err)
        return nil, fmt.Errorf("failed to query orders: %w", err)
    }
    defer rows.Close()
    
    var orders []*Order
    for rows.Next() {
        var order Order
        if err := rows.Scan(
            &order.ID,
            &order.UserID,
            &order.Amount,
            &order.Status,
            &order.CreatedAt,
        ); err != nil {
            span.RecordError(err)
            return nil, fmt.Errorf("failed to scan order: %w", err)
        }
        orders = append(orders, &order)
    }
    
    if err := rows.Err(); err != nil {
        span.RecordError(err)
        return nil, fmt.Errorf("error iterating rows: %w", err)
    }
    
    span.SetAttributes(attribute.Int("orders.count", len(orders)))
    
    return orders, nil
}
```

The database instrumentation creates spans for every query, capturing execution time and the SQL statement itself. This visibility is invaluable when debugging slow queries or understanding which database operations dominate your service's latency. The automatic instrumentation means you don't need to manually create spans for every database call, but you can still add custom spans when you want to capture additional context.

## Making Instrumented HTTP Client Calls

In a distributed system, services communicate with each other over HTTP. Proper instrumentation of these outbound calls is essential for creating end-to-end traces. Let's examine how to instrument an HTTP client that calls other services:

```go
// File: internal/clients/inventory_client.go
package clients

import (
    "bytes"
    "context"
    "encoding/json"
    "fmt"
    "net/http"
    "time"

    "go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp"
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/trace"
)

// InventoryClient handles communication with the inventory service
type InventoryClient struct {
    baseURL string
    client  *http.Client
}

// NewInventoryClient creates a new instrumented inventory client
func NewInventoryClient(baseURL string) *InventoryClient {
    // Create an HTTP client with OpenTelemetry instrumentation
    // This client will automatically:
    // 1. Create a span for each HTTP request
    // 2. Inject trace context into request headers
    // 3. Capture request/response metadata
    client := &http.Client{
        Transport: otelhttp.NewTransport(
            http.DefaultTransport,
            otelhttp.WithSpanNameFormatter(func(operation string, r *http.Request) string {
                // Custom span names for better clarity
                return fmt.Sprintf("HTTP %s %s", r.Method, r.URL.Path)
            }),
        ),
        Timeout: 10 * time.Second,
    }

    return &InventoryClient{
        baseURL: baseURL,
        client:  client,
    }
}

// CheckAvailabilityRequest represents the request to check item availability
type CheckAvailabilityRequest struct {
    Items []string `json:"items"`
}

// CheckAvailabilityResponse represents the inventory service response
type CheckAvailabilityResponse struct {
    Available bool              `json:"available"`
    Stock     map[string]int    `json:"stock"`
}

// CheckAvailability checks if items are available in inventory
func (c *InventoryClient) CheckAvailability(
    ctx context.Context,
    items []string,
) (*CheckAvailabilityResponse, error) {
    // Get tracer for custom span attributes
    tracer := otel.Tracer("order-service.inventory-client")
    
    // Create a span for this business operation
    // The HTTP client will create a child span for the actual HTTP request
    ctx, span := tracer.Start(ctx, "inventory.check_availability",
        trace.WithSpanKind(trace.SpanKindClient),
        trace.WithAttributes(
            attribute.StringSlice("inventory.items", items),
            attribute.Int("inventory.item_count", len(items)),
        ),
    )
    defer span.End()

    // Prepare request payload
    reqBody := CheckAvailabilityRequest{
        Items: items,
    }
    
    bodyBytes, err := json.Marshal(reqBody)
    if err != nil {
        span.RecordError(err)
        return nil, fmt.Errorf("failed to marshal request: %w", err)
    }

    // Create HTTP request with context
    // The context contains the trace information
    req, err := http.NewRequestWithContext(
        ctx,
        "POST",
        c.baseURL+"/api/v1/inventory/check",
        bytes.NewBuffer(bodyBytes),
    )
    if err != nil {
        span.RecordError(err)
        return nil, fmt.Errorf("failed to create request: %w", err)
    }

    req.Header.Set("Content-Type", "application/json")

    // Make the HTTP request
    // The otelhttp transport will:
    // 1. Create a child span for this HTTP call
    // 2. Inject W3C Trace Context headers (traceparent, tracestate)
    // 3. Capture timing and status code
    resp, err := c.client.Do(req)
    if err != nil {
        span.RecordError(err)
        span.SetAttributes(attribute.Bool("inventory.available", false))
        return nil, fmt.Errorf("request failed: %w", err)
    }
    defer resp.Body.Close()

    // Record HTTP status in our business logic span
    span.SetAttributes(
        attribute.Int("http.response.status_code", resp.StatusCode),
    )

    if resp.StatusCode != http.StatusOK {
        span.SetAttributes(attribute.Bool("inventory.available", false))
        return nil, fmt.Errorf("unexpected status code: %d", resp.StatusCode)
    }

    // Parse response
    var result CheckAvailabilityResponse
    if err := json.NewDecoder(resp.Body).Decode(&result); err != nil {
        span.RecordError(err)
        return nil, fmt.Errorf("failed to decode response: %w", err)
    }

    // Add business-relevant attributes to the span
    span.SetAttributes(
        attribute.Bool("inventory.available", result.Available),
    )

    return &result, nil
}
```

The HTTP client instrumentation is sophisticated because it must handle context propagation across network boundaries. When your service makes a call to the inventory service, the OpenTelemetry transport injects the current trace context into HTTP headers using the W3C Trace Context standard. When the inventory service receives this request (assuming it's also instrumented with OpenTelemetry), it extracts the trace context and continues the same distributed trace. This creates a seamless flow of tracing information across your entire system.

## Working with gRPC Services

Many Go-based distributed systems use gRPC for inter-service communication because of its performance and strong typing. OpenTelemetry provides excellent support for gRPC through interceptors:

```go
// File: internal/grpc/server.go
package grpc

import (
    "context"
    "log"
    "net"

    "go.opentelemetry.io/contrib/instrumentation/google.golang.org/grpc/otelgrpc"
    "google.golang.org/grpc"
    
    pb "github.com/yourcompany/order-service/proto"
)

// Server implements the gRPC order service
type Server struct {
    pb.UnimplementedOrderServiceServer
}

// StartGRPCServer starts a gRPC server with OpenTelemetry instrumentation
func StartGRPCServer(port string) error {
    listener, err := net.Listen("tcp", port)
    if err != nil {
        return err
    }

    // Create gRPC server with OpenTelemetry interceptors
    // These interceptors automatically:
    // 1. Extract trace context from incoming gRPC metadata
    // 2. Create spans for RPC calls
    // 3. Inject trace context into outgoing gRPC metadata
    grpcServer := grpc.NewServer(
        grpc.UnaryInterceptor(otelgrpc.UnaryServerInterceptor()),
        grpc.StreamInterceptor(otelgrpc.StreamServerInterceptor()),
    )

    pb.RegisterOrderServiceServer(grpcServer, &Server{})

    log.Printf("gRPC server listening on %s", port)
    return grpcServer.Serve(listener)
}

// CreateOrder implements the CreateOrder RPC method
func (s *Server) CreateOrder(
    ctx context.Context,
    req *pb.CreateOrderRequest,
) (*pb.CreateOrderResponse, error) {
    // The context already contains trace information from the interceptor
    tracer := otel.Tracer("order-service.grpc")
    
    ctx, span := tracer.Start(ctx, "grpc.CreateOrder")
    defer span.End()
    
    // Add request attributes
    span.SetAttributes(
        attribute.String("order.user_id", req.UserId),
        attribute.Float64("order.amount", req.Amount),
    )
    
    // Your business logic here
    // All database calls, HTTP calls, etc. will be child spans
    
    return &pb.CreateOrderResponse{
        OrderId: "order-123",
        Status:  "created",
    }, nil
}
```

```go
        Items: items,
    })
}
```

## Adding Metrics to Your Go Services

Beyond traces, metrics provide aggregated insights into your system's behavior over time. OpenTelemetry's metrics API allows you to create counters, gauges, and histograms that capture operational data:

```go
// File: internal/metrics/metrics.go
package metrics

import (
    "context"

    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/metric"
)

// Metrics holds all application metrics
type Metrics struct {
    OrdersCreated    metric.Int64Counter
    OrdersFailed     metric.Int64Counter
    OrderValue       metric.Float64Histogram
    ActiveOrders     metric.Int64UpDownCounter
    OrderProcessTime metric.Float64Histogram
}

// NewMetrics initializes application metrics
func NewMetrics() (*Metrics, error) {
    meter := otel.Meter("order-service")

    ordersCreated, err := meter.Int64Counter(
        "orders.created",
        metric.WithDescription("Total number of orders created"),
        metric.WithUnit("1"),
    )
    if err != nil {
        return nil, err
    }

    ordersFailed, err := meter.Int64Counter(
        "orders.failed",
        metric.WithDescription("Total number of failed order attempts"),
        metric.WithUnit("1"),
    )
    if err != nil {
        return nil, err
    }

    orderValue, err := meter.Float64Histogram(
        "orders.value",
        metric.WithDescription("Distribution of order values"),
        metric.WithUnit("USD"),
    )
    if err != nil {
        return nil, err
    }

    activeOrders, err := meter.Int64UpDownCounter(
        "orders.active",
        metric.WithDescription("Number of currently active orders"),
        metric.WithUnit("1"),
    )
    if err != nil {
        return nil, err
    }

    orderProcessTime, err := meter.Float64Histogram(
        "orders.process_time",
        metric.WithDescription("Time taken to process orders"),
        metric.WithUnit("ms"),
    )
    if err != nil {
        return nil, err
    }

    return &Metrics{
        OrdersCreated:    ordersCreated,
        OrdersFailed:     ordersFailed,
        OrderValue:       orderValue,
        ActiveOrders:     activeOrders,
        OrderProcessTime: orderProcessTime,
    }, nil
}

// RecordOrderCreated records a successful order creation
func (m *Metrics) RecordOrderCreated(ctx context.Context, userID string, amount float64) {
    attrs := []attribute.KeyValue{
        attribute.String("user.id", userID),
    }
    
    m.OrdersCreated.Add(ctx, 1, metric.WithAttributes(attrs...))
    m.OrderValue.Record(ctx, amount, metric.WithAttributes(attrs...))
}

// RecordOrderFailed records a failed order attempt
func (m *Metrics) RecordOrderFailed(ctx context.Context, reason string) {
    attrs := []attribute.KeyValue{
        attribute.String("failure.reason", reason),
    }
    
    m.OrdersFailed.Add(ctx, 1, metric.WithAttributes(attrs...))
}

// RecordProcessingTime records how long order processing took
func (m *Metrics) RecordProcessingTime(ctx context.Context, durationMs float64, status string) {
    attrs := []attribute.KeyValue{
        attribute.String("order.status", status),
    }
    
    m.OrderProcessTime.Record(ctx, durationMs, metric.WithAttributes(attrs...))
}
```

These metrics can be exported to Prometheus or any other metrics backend, giving you dashboards and alerting capabilities on top of your distributed traces.

Implementing OpenTelemetry in a Go distributed system requires thoughtful integration at multiple layers of your application stack. By following these patterns and understanding the underlying principles, you can build highly observable distributed systems that provide the visibility necessary to maintain reliability, diagnose issues quickly, and optimize performance at scale. The investment in proper instrumentation pays dividends every time you need to debug a production issue or understand the performance characteristics of your system under load.
