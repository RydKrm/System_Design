## 1. Why Logging Matters

Imagine you have deployed your Go application to a production server. Everything runs fine for days, then suddenly — at 3 AM — the server starts throwing errors and users cannot complete their requests. You wake up, connect to the server, and stare at your screen. The question you will inevitably ask is: _What happened?_

Logging is the answer to that question. A log is essentially a timestamped diary that your application keeps about everything it does — every request it receives, every database query it runs, every error it encounters, and every decision it makes. Without logs, debugging a production issue is like solving a crime with no witnesses and no evidence.

In a backend system — particularly one built with Go and backed by databases like PostgreSQL or MongoDB, message brokers like RabbitMQ, or caches like Redis — logs become your first line of investigation. When a user reports that their order was not placed, your logs should tell you exactly where in the chain of operations something went wrong: Was it the HTTP handler? The database transaction? The RabbitMQ message that never got published?

---

## 2. What is the Standard `log` Package — and Why It Falls Short

Go ships with a built-in `log` package that works perfectly for small, simple programs. It is easy to use and gets you started immediately.

```go
package main

import "log"

func main() {
    log.Println("Server started on port 8080")
    log.Printf("User %d logged in", 42)
    log.Fatal("Database connection failed")
}
```

The output looks like this:

```
2024/01/15 10:30:00 Server started on port 8080
2024/01/15 10:30:01 User 42 logged in
2024/01/15 10:30:02 Database connection failed
```

This is perfectly readable to a human, but now consider what happens when your production system processes 10,000 requests per minute. Your logs become a wall of text with no structure. To find all logs related to a specific user or a specific request, you would have to `grep` through thousands of lines hoping the format is consistent enough to filter.

Furthermore, the standard `log` package has no concept of **log levels** (you cannot distinguish between a debug message and a critical error), no concept of **structured fields** (you cannot attach metadata like `user_id`, `request_id`, or `duration` as searchable key-value pairs), and critically, it is **slow**. Every call to `log.Printf` involves string formatting using `fmt.Sprintf` internally, which allocates memory on the heap. In a high-throughput system, this adds up to measurable latency.

---

## 3. Enter Zap — Uber's Gift to the Go Ecosystem

Zap was created by Uber's engineering team to solve the exact problems described above. Uber runs some of the most high-traffic backend systems in the world, and they needed a logger that was blazing fast, produced machine-readable output, and did not get in the way of their services.

The name "Zap" reflects the philosophy: it should be as fast as a lightning bolt. Zap achieves remarkable performance through two primary innovations.

The first innovation is **zero-allocation logging**. Most loggers, including the standard library, format log messages by concatenating strings and using `fmt.Sprintf`. Each of these operations allocates a new string on the heap, which eventually triggers Go's garbage collector. Zap avoids this entirely by writing bytes directly to a buffer without creating intermediate string objects. When the garbage collector runs less often, your service experiences fewer latency spikes.

The second innovation is **structured logging by design**. Rather than asking you to format a human-readable message with embedded data, Zap encourages you to attach data as typed key-value fields. This makes every log entry a structured record that tools like Elasticsearch, Splunk, Datadog, or Grafana Loki can parse, index, and query without any special configuration.

## 4. How Zap Works Internally

Understanding the internals of Zap helps you make better decisions when configuring it. At its core, Zap is built around three major components: the **Core**, the **Encoder**, and the **WriteSyncer**.

```
ASCII Diagram: Zap Internal Architecture

Your Application Code
        │
        ▼
┌───────────────────┐
│   zap.Logger      │  ← The facade you call (logger.Info, logger.Error)
└────────┬──────────┘
         │ passes log entry
         ▼
┌───────────────────┐
│   zapcore.Core    │  ← The engine: checks level, decides to log or discard
└──────┬─────┬──────┘
       │     │
       ▼     ▼
┌──────────┐ ┌──────────────┐
│ Encoder  │ │ WriteSyncer  │
│ (JSON or │ │ (file, stdout│
│ Console) │ │ or both)     │
└──────────┘ └──────────────┘
       │             │
       └──────┬──────┘
              ▼
   Final log bytes written to output
```

The **Core** is the decision-making center. When you call `logger.Info(...)`, the Core first checks whether the `Info` level is enabled. If the level is below the configured minimum (for example, if you have configured the logger to only log `Warn` and above), the Core discards the entry immediately — no allocation, no formatting, no I/O. This is why Zap is so fast.

The **Encoder** converts your structured fields and message into bytes. Zap provides two encoders out of the box: a **JSON encoder** for production (producing clean, machine-readable JSON) and a **Console encoder** for development (producing human-readable, colorized output).

The **WriteSyncer** is the output destination. It is an interface wrapping an `io.Writer` with a `Sync()` method, allowing Zap to flush buffers when the application shuts down. You can write to `os.Stdout`, a file, or multiple destinations simultaneously.

## 6. The Two Flavors of Zap Logger

Zap gives you two different logger types, each designed for a different purpose.

### `zap.Logger` — The Fast, Strict Logger

This is the raw, uncompromising logger. Every field you attach must be a typed `zap.Field` object. There is no variadic string formatting. It is the most performant option and the one you should use in production hot paths.

```go
package main

import "go.uber.org/zap"

func main() {
    // A pre-configured production logger (JSON output, Info level and above)
    logger, _ := zap.NewProduction()
    defer logger.Sync() // flush any buffered log entries

    logger.Info("Server started",
        zap.String("host", "0.0.0.0"),
        zap.Int("port", 8080),
        zap.String("env", "production"),
    )

    logger.Error("Database connection failed",
        zap.String("host", "postgres:5432"),
        zap.Error(err),
    )
}
```

Output:

```json
{"level":"info","ts":1705312201.123,"caller":"main.go:10","msg":"Server started","host":"0.0.0.0","port":8080,"env":"production"}
```

### `zap.SugaredLogger` — The Friendly, Flexible Logger

The SugaredLogger wraps the raw logger and provides a more ergonomic API similar to `fmt.Printf`. It is slightly slower due to reflection-based field handling, but it is far more convenient for non-critical paths and for developers transitioning from `log.Printf`.

```go
package main

import "go.uber.org/zap"

func main() {
    logger, _ := zap.NewProduction()
    sugar := logger.Sugar()
    defer logger.Sync()

    // Printf-style: use format verbs
    sugar.Infof("User %d logged in from %s", 42, "192.168.1.10")

    // Key-value style: alternating keys and values (loosely typed)
    sugar.Infow("Order placed",
        "user_id", 42,
        "order_id", 789,
        "total", 59.99,
    )

    // Simple message only
    sugar.Info("Health check passed")
}
```

## 7. Structured Logging — The Heart of Zap

Structured logging is the practice of attaching data to log entries as typed key-value fields rather than embedding them in a formatted string. This seemingly small distinction has enormous consequences for how you operate a production system.

When logs are structured, your log aggregation tool (whether that is the ELK Stack, Datadog, Splunk, or Grafana Loki) can index every field independently. This means you can query: _"Show me all error logs where user_id is 42 and the response_time_ms is greater than 500."_ With unstructured logs, this query requires a fragile regex; with structured logs, it is a first-class query on indexed data.

Zap provides typed field constructors for every Go primitive and many common types. Using typed fields (rather than `zap.Any`) ensures zero-allocation performance because Zap knows exactly how to encode each type without reflection.

```go
package main

import (
    "time"
    "go.uber.org/zap"
)

func main() {
    logger, _ := zap.NewProduction()
    defer logger.Sync()

    logger.Info("HTTP request completed",
        // String fields
        zap.String("method", "POST"),
        zap.String("path", "/api/v1/orders"),
        zap.String("remote_addr", "203.0.113.42"),

        // Numeric fields
        zap.Int("status_code", 201),
        zap.Int64("user_id", 42),
        zap.Float64("response_time_ms", 12.5),

        // Boolean fields
        zap.Bool("cache_hit", false),

        // Duration (very useful for timing operations)
        zap.Duration("latency", 12*time.Millisecond),

        // Time (timestamps for events)
        zap.Time("requested_at", time.Now()),

        // Error (nil-safe — logs nothing if err is nil)
        zap.Error(nil),

        // Byte string (for binary-safe strings)
        zap.ByteString("trace_id", []byte("abc123")),

        // Arrays and slices
        zap.Strings("tags", []string{"checkout", "payment", "v2"}),
        zap.Ints("retry_attempts", []int{1, 2, 3}),

        // Any (fallback — uses reflection, slower)
        zap.Any("metadata", map[string]string{"region": "us-east-1"}),
    )
}
```

Output:

```json
{
  "level": "info",
  "ts": 1705312201.123,
  "caller": "main.go:10",
  "msg": "HTTP request completed",
  "method": "POST",
  "path": "/api/v1/orders",
  "remote_addr": "203.0.113.42",
  "status_code": 201,
  "user_id": 42,
  "response_time_ms": 12.5,
  "cache_hit": false,
  "latency": 0.012,
  "tags": ["checkout", "payment", "v2"]
}
```

## 8. Log Levels in Zap

Zap supports six log levels, ordered from least to most severe. Understanding when to use each level is one of the most important disciplines in production logging.

```
ASCII Diagram: Zap Log Levels (Least → Most Severe)

  ┌──────────┬─────────────────────────────────────────────────────────────┐
  │  Level   │  When to Use                                                │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  DEBUG   │  Detailed diagnostic info. NEVER in production by default.  │
  │          │  Ex: "Entering function ProcessOrder with args..."          │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  INFO    │  Normal operational events. The business is running.        │
  │          │  Ex: "User logged in", "Order placed", "Server started"     │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  WARN    │  Something unexpected but recoverable. Investigate later.   │
  │          │  Ex: "Retry #2 on DB connection", "Deprecated API used"     │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  ERROR   │  Something failed. A request could not complete.            │
  │          │  Ex: "Payment gateway timeout", "DB query failed"           │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  DPANIC  │  Panic in development mode only. Logs error in production.  │
  │          │  Used for programmer errors you want to catch in tests.     │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  PANIC   │  Logs then calls panic(). For unrecoverable states.         │
  ├──────────┼─────────────────────────────────────────────────────────────┤
  │  FATAL   │  Logs then calls os.Exit(1). Only for startup failures.     │
  │          │  Ex: "Cannot bind to port", "Cannot connect to DB on boot"  │
  └──────────┴─────────────────────────────────────────────────────────────┘

  Production recommendation:  Minimum level = INFO
  Development recommendation: Minimum level = DEBUG
```

```go
logger.Debug("Cache key computed", zap.String("key", "user:42:profile"))
logger.Info("User authenticated", zap.Int64("user_id", 42))
logger.Warn("Rate limit approaching", zap.Int("requests_remaining", 5))
logger.Error("Failed to publish message", zap.Error(err), zap.String("queue", "orders"))
logger.Fatal("Cannot connect to database on startup", zap.Error(err))
```

## 9. Building a Production Logger

The pre-configured `zap.NewProduction()` is a great starting point, but a real production application needs more control. You will want to configure the log level, output format, timestamp format, and caller information. The `zap.Config` struct is your tool for this.

```go
package logger

import (
    "os"
    "go.uber.org/zap"
    "go.uber.org/zap/zapcore"
)

// NewLogger builds a production-grade Zap logger.
// env: "production" outputs JSON; anything else outputs human-readable console format.
func NewLogger(env string, level string) (*zap.Logger, error) {
    // Step 1: Parse the log level from a string (e.g., "info", "debug", "warn")
    var zapLevel zapcore.Level
    if err := zapLevel.UnmarshalText([]byte(level)); err != nil {
        zapLevel = zapcore.InfoLevel // default to Info if parsing fails
    }

    // Step 2: Configure the encoder
    // The encoder controls how each log entry is formatted.
    encoderConfig := zapcore.EncoderConfig{
        TimeKey:        "ts",             // JSON key for timestamp
        LevelKey:       "level",          // JSON key for log level
        NameKey:        "logger",         // JSON key for logger name
        CallerKey:      "caller",         // JSON key for file:line
        FunctionKey:    zapcore.OmitKey,  // skip function name (verbose)
        MessageKey:     "msg",            // JSON key for the log message
        StacktraceKey:  "stacktrace",     // JSON key for stack traces
        LineEnding:     zapcore.DefaultLineEnding,
        EncodeLevel:    zapcore.LowercaseLevelEncoder,     // "info" not "INFO"
        EncodeTime:     zapcore.ISO8601TimeEncoder,        // "2024-01-15T10:30:00.000Z"
        EncodeDuration: zapcore.MillisDurationEncoder,     // 12.5 (milliseconds)
        EncodeCaller:   zapcore.ShortCallerEncoder,        // "main.go:42"
    }

    // Step 3: Choose encoder type based on environment
    var encoder zapcore.Encoder
    if env == "production" {
        encoder = zapcore.NewJSONEncoder(encoderConfig)
    } else {
        // Console encoder adds color and human-readable level names
        encoderConfig.EncodeLevel = zapcore.CapitalColorLevelEncoder
        encoder = zapcore.NewConsoleEncoder(encoderConfig)
    }

    // Step 4: Set up the output destination
    // zapcore.AddSync wraps any io.Writer into a WriteSyncer
    stdout := zapcore.AddSync(os.Stdout)

    // Step 5: Combine into a Core
    core := zapcore.NewCore(encoder, stdout, zapLevel)

    // Step 6: Build the Logger with options
    log := zap.New(core,
        zap.AddCaller(),              // include file and line number
        zap.AddCallerSkip(0),         // how many stack frames to skip
        zap.AddStacktrace(zapcore.ErrorLevel), // add stack traces on Error+
    )

    return log, nil
}
```

Using the logger in `main.go`:

```go
package main

import (
    "os"
    "yourapp/logger"
    "go.uber.org/zap"
)

var log *zap.Logger

func main() {
    env := os.Getenv("APP_ENV")     // "production" or "development"
    level := os.Getenv("LOG_LEVEL") // "debug", "info", "warn", "error"
    if level == "" {
        level = "info"
    }

    var err error
    log, err = logger.NewLogger(env, level)
    if err != nil {
        panic("failed to initialize logger: " + err.Error())
    }
    defer log.Sync()

    log.Info("Application starting",
        zap.String("version", "1.0.0"),
        zap.String("env", env),
    )
}
```

## 10. Writing to Multiple Outputs

In a production environment, you often need to write logs to multiple places simultaneously — to `stdout` (so container orchestration tools like Kubernetes can collect them), and to a log file on disk (for local debugging or short-term retention).

Zap makes this straightforward with `zapcore.NewTee`, which fans out every log entry to multiple Cores simultaneously.

```go
package logger

import (
    "os"
    "go.uber.org/zap"
    "go.uber.org/zap/zapcore"
)

func NewMultiOutputLogger(logFilePath string) (*zap.Logger, error) {
    // Encoder config (shared between both outputs)
    encoderConfig := zap.NewProductionEncoderConfig()
    encoderConfig.EncodeTime = zapcore.ISO8601TimeEncoder

    // Output 1: JSON to stdout (for log collectors like Fluentd, Filebeat)
    jsonEncoder := zapcore.NewJSONEncoder(encoderConfig)
    stdoutSyncer := zapcore.AddSync(os.Stdout)
    stdoutCore := zapcore.NewCore(jsonEncoder, stdoutSyncer, zapcore.InfoLevel)

    // Output 2: Console-readable to a log file (for local debugging)
    logFile, err := os.OpenFile(logFilePath, os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0644)
    if err != nil {
        return nil, err
    }
    consoleEncoder := zapcore.NewConsoleEncoder(encoderConfig)
    fileSyncer := zapcore.AddSync(logFile)
    fileCore := zapcore.NewCore(consoleEncoder, fileSyncer, zapcore.DebugLevel)

    // Combine both cores: every log entry goes to both destinations
    combinedCore := zapcore.NewTee(stdoutCore, fileCore)

    return zap.New(combinedCore, zap.AddCaller(), zap.AddStacktrace(zapcore.ErrorLevel)), nil
}
```

## 11. Log Rotation with Lumberjack

Log files grow. A busy production service can generate gigabytes of logs in a single day. Without log rotation, your disk fills up and your entire server goes down. Lumberjack is a Go library that handles log rotation seamlessly, integrating beautifully with Zap.

```go
package logger

import (
    "go.uber.org/zap"
    "go.uber.org/zap/zapcore"
    "gopkg.in/natefinish/lumberjack.v2"
)

func NewRotatingLogger() *zap.Logger {
    // Lumberjack handles file rotation automatically
    lumberjackLogger := &lumberjack.Logger{
        Filename:   "/var/log/myapp/app.log", // path to the log file
        MaxSize:    100,                        // rotate after 100 MB
        MaxBackups: 7,                          // keep 7 old rotated files
        MaxAge:     30,                         // delete files older than 30 days
        Compress:   true,                       // gzip old log files to save space
    }

    encoderConfig := zap.NewProductionEncoderConfig()
    encoderConfig.EncodeTime = zapcore.ISO8601TimeEncoder

    core := zapcore.NewCore(
        zapcore.NewJSONEncoder(encoderConfig),
        zapcore.AddSync(lumberjackLogger), // Lumberjack implements io.Writer
        zapcore.InfoLevel,
    )

    return zap.New(core, zap.AddCaller())
}
```

```
ASCII Diagram: Log Rotation Lifecycle

  app.log (growing...)
  ┌──────────────────────────────────--────┐
  │ {"ts":"2024-01-15T10:00:00","msg":...} │
```

## 12. Adding Global Context with Fields

Every log entry in your application should carry certain fields automatically — things like the application name, version, environment, and hostname. Rather than adding these to every single log call, you can embed them once using `logger.With(...)`, which creates a new logger with those fields pre-attached to every subsequent entry.

```go
package main

import (
    "os"
    "go.uber.org/zap"
)

func main() {
    baseLogger, _ := zap.NewProduction()
    defer baseLogger.Sync()

    // Create an application-level logger with global context
    // Every log entry from this logger will include these fields automatically
    appLogger := baseLogger.With(
        zap.String("app", "order-service"),
        zap.String("version", "2.3.1"),
        zap.String("env", os.Getenv("APP_ENV")),
        zap.String("host", mustGetHostname()),
    )

    // Now every log call carries those fields for free
    appLogger.Info("Server started", zap.Int("port", 8080))
    // Output: {"level":"info","msg":"Server started","app":"order-service",
    //           "version":"2.3.1","env":"production","host":"web-01","port":8080}

    appLogger.Error("Payment failed", zap.Error(someError))
    // Output: {"level":"error","msg":"Payment failed","app":"order-service",
    //           "version":"2.3.1","env":"production","host":"web-01","error":"timeout"}
}

func mustGetHostname() string {
    h, err := os.Hostname()
    if err != nil {
        return "unknown"
    }
    return h
}
```

## 13. Child Loggers — Contextual Logging Per Module

One of the most powerful patterns in Zap is the **child logger**, sometimes also called a "named logger" or "contextual logger." The idea is that each module, service, or handler in your application gets its own logger derived from the base logger, with additional fields that identify where the log entry originated.

```go
package handlers

import "go.uber.org/zap"

// Each handler gets its own child logger
type OrderHandler struct {
    logger *zap.Logger
    db     *Database
}

func NewOrderHandler(baseLogger *zap.Logger, db *Database) *OrderHandler {
    return &OrderHandler{
        // Child logger adds component context automatically
        logger: baseLogger.With(zap.String("component", "order-handler")),
        db:     db,
    }
}

func (h *OrderHandler) CreateOrder(userID int64, items []Item) error {
    // Further scope: per-request child logger
    reqLogger := h.logger.With(
        zap.Int64("user_id", userID),
        zap.Int("item_count", len(items)),
    )

    reqLogger.Info("Creating order")

    orderID, err := h.db.InsertOrder(userID, items)
    if err != nil {
        reqLogger.Error("Failed to insert order into database",
            zap.Error(err),
        )
        return err
    }

    reqLogger.Info("Order created successfully",
        zap.Int64("order_id", orderID),
    )
    return nil
}
```

Output when an order is created:

```json
{"level":"info","msg":"Creating order","component":"order-handler","user_id":42,"item_count":3}
{"level":"info","msg":"Order created successfully","component":"order-handler","user_id":42,"item_count":3,"order_id":789}
```

Notice how `component`, `user_id`, and `item_count` appear automatically in every log line without being passed around. This is the power of child loggers — they carry context silently through every operation.

## 14. Logging in HTTP Middleware

In any backend API, logging HTTP requests is essential. Every request that enters your system should produce a log entry with the method, path, status code, duration, and client IP. The cleanest way to achieve this is through middleware — a function that wraps every handler.

```go
package middleware

import (
    "net/http"
    "time"
    "go.uber.org/zap"
)

// RequestLogger returns an HTTP middleware that logs every incoming request.
func RequestLogger(logger *zap.Logger) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            start := time.Now()

            // Wrap the ResponseWriter to capture the status code
            wrapped := &responseWriter{ResponseWriter: w, statusCode: http.StatusOK}

            // Process the actual request
            next.ServeHTTP(wrapped, r)

            // After the handler returns, log the completed request
            duration := time.Since(start)

            logger.Info("HTTP request",
                zap.String("method", r.Method),
                zap.String("path", r.URL.Path),
                zap.String("query", r.URL.RawQuery),
                zap.Int("status", wrapped.statusCode),
                zap.Duration("duration", duration),
                zap.String("remote_addr", r.RemoteAddr),
                zap.String("user_agent", r.UserAgent()),
                zap.Int64("content_length", r.ContentLength),
            )
        })
    }
}

// responseWriter is a thin wrapper to capture the HTTP status code
type responseWriter struct {
    http.ResponseWriter
    statusCode int
}

func (rw *responseWriter) WriteHeader(code int) {
    rw.statusCode = code
    rw.ResponseWriter.WriteHeader(code)
}
```

Registering the middleware:

```go
mux := http.NewServeMux()
mux.HandleFunc("/api/v1/orders", orderHandler.CreateOrder)

// Wrap the entire mux with the logging middleware
http.ListenAndServe(":8080", middleware.RequestLogger(logger)(mux))
```

## 15. Logging with Request ID (Tracing)

In a distributed system or even a monolith handling concurrent requests, being able to trace all log entries that belong to a single request is invaluable. This is done by generating a unique `request_id` for each incoming HTTP request and attaching it to a child logger that is passed through the entire request lifecycle via Go's `context.Context`.

```go
package middleware

import (
    "context"
    "net/http"
    "github.com/google/uuid"
    "go.uber.org/zap"
)

// contextKey is an unexported type to avoid collisions with other packages
type contextKey string
const loggerKey contextKey = "logger"

// RequestID injects a unique request ID and a pre-scoped logger into every request context.
func RequestID(baseLogger *zap.Logger) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            // Generate a unique request ID (or read from header if upstream provided one)
            requestID := r.Header.Get("X-Request-ID")
            if requestID == "" {
                requestID = uuid.New().String()
            }

            // Set the request ID in the response header so callers can trace
            w.Header().Set("X-Request-ID", requestID)

            // Create a request-scoped child logger
            reqLogger := baseLogger.With(
                zap.String("request_id", requestID),
                zap.String("method", r.Method),
                zap.String("path", r.URL.Path),
            )

            // Inject the logger into the request context
            ctx := context.WithValue(r.Context(), loggerKey, reqLogger)
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

// FromContext extracts the request-scoped logger from a context.
// Falls back to the base logger if none is found.
func FromContext(ctx context.Context, fallback *zap.Logger) *zap.Logger {
    if l, ok := ctx.Value(loggerKey).(*zap.Logger); ok {
        return l
    }
    return fallback
}
```

Using the logger in a handler:

```go
func (h *OrderHandler) CreateOrder(w http.ResponseWriter, r *http.Request) {
    // Extract the request-scoped logger — it already has request_id, method, path
    log := middleware.FromContext(r.Context(), h.logger)

    log.Info("Processing order creation")

    order, err := h.service.CreateOrder(r.Context(), req)
    if err != nil {
        log.Error("Order creation failed", zap.Error(err))
        http.Error(w, "Internal Server Error", 500)
        return
    }

    log.Info("Order created", zap.Int64("order_id", order.ID))
}
```

All log entries from this single request will share the same `request_id`, making it trivial to reconstruct the full story of what happened during that request by simply filtering on that ID.

## 16. Logging Errors and Stack Traces

Errors are the most important things to log, and they deserve special attention. Zap provides the `zap.Error(err)` field, which safely handles `nil` errors (logs nothing if `err` is nil). For critical errors, Zap can automatically attach a full stack trace.

```go
package service

import (
    "fmt"
    "go.uber.org/zap"
)

func (s *PaymentService) ChargeCard(userID int64, amount float64) error {
    log := s.logger.With(
        zap.Int64("user_id", userID),
        zap.Float64("amount", amount),
    )

    log.Info("Initiating card charge")

    result, err := s.gateway.Charge(amount)
    if err != nil {
        // Log with error — zap.Error attaches the error message
        log.Error("Payment gateway returned error",
            zap.Error(err),
            zap.String("gateway", "stripe"),
        )
        return fmt.Errorf("charge failed: %w", err)
    }

    // Log a warning for partial failures
    if result.Status == "pending" {
        log.Warn("Payment is pending, may require manual review",
            zap.String("transaction_id", result.TxID),
            zap.String("status", result.Status),
        )
    }

    log.Info("Card charged successfully",
        zap.String("transaction_id", result.TxID),
        zap.Float64("charged_amount", result.Amount),
    )
    return nil
}
```

For stack traces on errors, configure your logger with:

```go
logger := zap.New(core,
    zap.AddCaller(),
    zap.AddStacktrace(zapcore.ErrorLevel), // attaches stack trace to all Error+ entries
)
```

This produces:

```json
{
  "level": "error",
  "msg": "Payment gateway returned error",
  "user_id": 42,
  "error": "connection timeout",
  "stacktrace": "main.(*PaymentService).ChargeCard\n\t/app/service/payment.go:28\n..."
}
```

## 17. Logging in Goroutines

Go's concurrency model means your application will run many goroutines simultaneously. Each goroutine should use a logger that already carries context about what it is doing. Since `*zap.Logger` is safe for concurrent use from multiple goroutines, you can share a logger without mutex protection.

```go
package worker

import (
    "context"
    "sync"
    "go.uber.org/zap"
)

type OrderProcessor struct {
    logger *zap.Logger
    jobs   chan Order
}

func (p *OrderProcessor) Start(ctx context.Context, workerCount int) {
    var wg sync.WaitGroup

    for i := 0; i < workerCount; i++ {
        wg.Add(1)
        workerID := i

        go func() {
            defer wg.Done()

            // Each worker gets its own child logger with its worker ID
            workerLog := p.logger.With(zap.Int("worker_id", workerID))
            workerLog.Info("Worker started")

            for {
                select {
                case order, ok := <-p.jobs:
                    if !ok {
                        workerLog.Info("Job channel closed, worker shutting down")
                        return
                    }
                    // Each job gets its own log scope
                    jobLog := workerLog.With(zap.Int64("order_id", order.ID))
                    jobLog.Info("Processing order")

                    if err := p.processOrder(order); err != nil {
                        jobLog.Error("Order processing failed", zap.Error(err))
                        continue
                    }

                    jobLog.Info("Order processed successfully")

                case <-ctx.Done():
                    workerLog.Info("Context cancelled, worker shutting down")
                    return
                }
            }
        }()
    }

    wg.Wait()
    p.logger.Info("All workers stopped")
}
```

## 18. Logging with Database Operations

Database operations are among the most common sources of errors and performance issues in a backend application. Every database call should be logged with the operation type, the table or collection name, the duration, and any errors that occurred.

```go
package repository

import (
    "context"
    "database/sql"
    "time"
    "go.uber.org/zap"
)

type OrderRepository struct {
    db     *sql.DB
    logger *zap.Logger
}

func (r *OrderRepository) FindByID(ctx context.Context, orderID int64) (*Order, error) {
    log := r.logger.With(
        zap.String("operation", "FindByID"),
        zap.String("table", "orders"),
        zap.Int64("order_id", orderID),
    )

    log.Debug("Executing database query")
    start := time.Now()

    var order Order
    err := r.db.QueryRowContext(ctx,
        "SELECT id, user_id, total, status, created_at FROM orders WHERE id = $1",
        orderID,
    ).Scan(&order.ID, &order.UserID, &order.Total, &order.Status, &order.CreatedAt)

    duration := time.Since(start)

    if err == sql.ErrNoRows {
        log.Warn("Order not found",
            zap.Duration("db_duration", duration),
        )
        return nil, ErrNotFound
    }

    if err != nil {
        log.Error("Database query failed",
            zap.Error(err),
            zap.Duration("db_duration", duration),
        )
        return nil, err
    }

    log.Debug("Database query completed",
        zap.Duration("db_duration", duration),
        zap.String("status", order.Status),
    )

    // Warn on slow queries (helps identify N+1 or missing indexes)
    if duration > 100*time.Millisecond {
        log.Warn("Slow database query detected",
            zap.Duration("db_duration", duration),
            zap.String("suggestion", "check for missing index on orders.id"),
        )
    }

    return &order, nil
}
```

## 19. Logging with Redis / RabbitMQ

Cache operations (Redis) and message queue operations (RabbitMQ) follow the same pattern as database logging: you want to capture the operation, the key or queue name, the duration, and any errors.

### Redis Logging

```go
package cache

import (

type UserCache struct {
```

```go
package messaging

import (
```

## 20. Testing Your Logger

In unit tests, you want to capture log output and make assertions against it — for example, verifying that an error log is emitted when a database call fails. Zap provides `zaptest` for exactly this purpose.

```go
package service_test

import (
    "testing"
    "go.uber.org/zap"
    "go.uber.org/zap/zaptest"
    "go.uber.org/zap/zaptest/observer"
)

func TestOrderService_CreateOrder_LogsError(t *testing.T) {
    // zaptest.NewLogger creates a logger that writes to t.Log()
    // (visible in `go test -v` output)
    testLogger := zaptest.NewLogger(t)

    // For assertions, use the observer package to capture log entries
    core, logs := observer.New(zap.ErrorLevel)
    observedLogger := zap.New(core)

    service := NewOrderService(observedLogger, &failingDB{})

    err := service.CreateOrder(42, []Item{{ID: 1}})
    if err == nil {
        t.Fatal("expected error but got nil")
    }

    // Assert that exactly one error log was emitted
    if logs.Len() != 1 {
        t.Fatalf("expected 1 error log, got %d", logs.Len())
    }

    entry := logs.All()[0]
    if entry.Message != "Failed to insert order into database" {
        t.Errorf("unexpected log message: %s", entry.Message)
    }

    // Assert specific fields on the log entry
    if entry.ContextMap()["user_id"] != int64(42) {
        t.Errorf("expected user_id=42 in log entry")
    }

    _ = testLogger // suppress unused variable warning
}
```

## 21. Global Logger — The Zap Sugar Global

Zap provides a package-level global logger that is convenient for small applications or for cases where you need to log before your main logger is initialized. You configure it once, and then any package can use it via `zap.L()` and `zap.S()`.

```go
package main

import "go.uber.org/zap"

func main() {
    logger, _ := zap.NewProduction()
    defer logger.Sync()

    // Set the global logger — now any package can call zap.L()
    zap.ReplaceGlobals(logger)

    // In any package, anywhere:
    zap.L().Info("This uses the global logger")
    zap.S().Infof("User %d joined", 42) // S() returns the global SugaredLogger
}
```

**Note:** For production services with dependency injection, it is better practice to pass loggers explicitly through constructors rather than relying on the global logger. The global logger is best suited for `init()` functions and early startup code where dependency injection is not yet set up.

## 23. Full Production Logger Setup — The Reference Implementation

This is a complete, ready-to-use logger package for a production Go service.

```go
// package logger provides application-wide structured logging built on Zap.
package logger

import (
    "os"
    "go.uber.org/zap"
    "go.uber.org/zap/zapcore"
    "gopkg.in/natefinish/lumberjack.v2"
)

// Config holds all configuration options for the logger.
type Config struct {
    Environment string // "production" | "development"
    Level       string // "debug" | "info" | "warn" | "error"
    LogFilePath string // "/var/log/myapp/app.log" — empty means stdout only
    MaxSizeMB   int    // rotate log file after this many megabytes
    MaxBackups  int    // number of old rotated files to keep
    MaxAgeDays  int    // delete files older than this many days
    Compress    bool   // gzip rotated files
    AppName     string // attached to every log entry as "app"
    AppVersion  string // attached to every log entry as "version"
}

// New builds and returns a fully configured production-grade Zap logger.
// The returned logger has global context fields pre-attached and is safe
// for concurrent use by multiple goroutines.
func New(cfg Config) (*zap.Logger, error) {
    // Parse log level
    var level zapcore.Level
    if err := level.UnmarshalText([]byte(cfg.Level)); err != nil {
        level = zapcore.InfoLevel
    }

    // Build encoder config
    encCfg := zapcore.EncoderConfig{
        TimeKey:        "ts",
        LevelKey:       "level",
        NameKey:        "logger",
        CallerKey:      "caller",
        FunctionKey:    zapcore.OmitKey,
        MessageKey:     "msg",
        StacktraceKey:  "stacktrace",
        LineEnding:     zapcore.DefaultLineEnding,
        EncodeLevel:    zapcore.LowercaseLevelEncoder,
        EncodeTime:     zapcore.ISO8601TimeEncoder,
        EncodeDuration: zapcore.MillisDurationEncoder,
        EncodeCaller:   zapcore.ShortCallerEncoder,
    }

    var cores []zapcore.Core

    // Core 1: Always log to stdout
    stdoutEncoder := zapcore.NewJSONEncoder(encCfg)
    if cfg.Environment != "production" {
        devCfg := encCfg
        devCfg.EncodeLevel = zapcore.CapitalColorLevelEncoder
        stdoutEncoder = zapcore.NewConsoleEncoder(devCfg)
    }
    cores = append(cores, zapcore.NewCore(
        stdoutEncoder,
        zapcore.AddSync(os.Stdout),
        level,
    ))

    // Core 2: Optionally also log to a rotating file
    if cfg.LogFilePath != "" {
        lj := &lumberjack.Logger{
            Filename:   cfg.LogFilePath,
            MaxSize:    cfg.MaxSizeMB,
            MaxBackups: cfg.MaxBackups,
            MaxAge:     cfg.MaxAgeDays,
            Compress:   cfg.Compress,
        }
        cores = append(cores, zapcore.NewCore(
            zapcore.NewJSONEncoder(encCfg),
            zapcore.AddSync(lj),
            level,
        ))
    }

    // Combine all cores
    combinedCore := zapcore.NewTee(cores...)

    // Build logger with caller info and stacktraces on errors
    base := zap.New(combinedCore,
        zap.AddCaller(),
        zap.AddCallerSkip(0),
        zap.AddStacktrace(zapcore.ErrorLevel),
    )

    // Attach global fields that appear on every single log entry
    hostname, _ := os.Hostname()
    global := base.With(
        zap.String("app", cfg.AppName),
        zap.String("version", cfg.AppVersion),
        zap.String("env", cfg.Environment),
        zap.String("host", hostname),
    )

    return global, nil
}
```

Usage in `main.go`:

